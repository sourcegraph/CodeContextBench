## Codebase Patterns
- Suite run scripts live in `configs/*_2config.sh` and typically source `configs/_common.sh` for auth, token refresh, canary, and validation behavior.
- Paired config execution convention is baseline (`BASELINE_MCP_TYPE=none`) vs MCP (`BASELINE_MCP_TYPE=sourcegraph_full`) with deterministic jobs directory naming.
- Reporting and metrics extraction currently assume Claude transcript artifacts in several places; multi-harness support needs transcript discovery abstraction.
- Operational docs under `docs/` are canonical; avoid adding process-critical guidance only in ad hoc notes.
- Run health gates rely on `scripts/validate_task_run.py`, `scripts/generate_manifest.py`, and `scripts/generate_eval_report.py` after batch completion.
- Multi-harness requirements should be centralized in `docs/HARNESS_CONTRACT.md` (fields, transcript artifact expectations, and allowed MCP modes) so runner and registry changes stay aligned.
- Harness registry entries should be keyed by stable harness IDs (`codex`, `cursor`, `gemini`, `copilot`, `openhands`) and constrain `allowed_mcp_modes` to exactly `none` and `sourcegraph_full` for this rollout.
- Non-Claude harness runners should still source `configs/_common.sh` for validation/parallel helpers, but must not depend on Claude OAuth refresh flows.
