# Bug Investigation: Authentication Fails for Browser Cookie-Based Tokens

**Repository:** flipt-io/flipt
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

The authentication system in Flipt only accepts client tokens via the `Authorization` header using Bearer format. This prevents browser-based sessions from working, since browsers typically store authentication tokens in HTTP cookies.

Specifically:

1. **Cookie tokens rejected**: When a valid client token is stored in an HTTP cookie (under the key `flipt_client_token`), the authentication middleware does not extract or validate it. Requests that carry a valid token in a cookie but not in the Authorization header are rejected as unauthenticated.

2. **No server-level authentication bypass**: Certain internal servers (such as those handling delegated authentication flows) need to operate without authentication checks. There is currently no mechanism to configure the middleware to skip authentication for specific server instances, forcing all registered servers through the same authentication path regardless of their role.

```go
    logger.Error("unauthenticated", zap.String("reason", "no authorization provided"))
    return ctx, errUnauthenticated
}
```

### Test Verification
The test demonstrates:
- ✅ **Token in Authorization header**: Tokens with `Authorization: Bearer <token>` ARE extracted
- ❌ **Token in cookie**: Tokens in `flipt_client_token=<token>` cookies ARE NOT extracted
- ❌ **Token in metadata key**: Tokens in custom cookie metadata ARE NOT extracted

### Impact
- **Browser compatibility**: Browsers can't use this authentication system
- **Session management**: No cookie-based session support
- **Standard compliance**: Doesn't follow HTTP authentication conventions

### Fix Required
The middleware needs to:
1. Check for cookies in metadata
2. Extract `flipt_client_token` from cookies
3. Fall back to Authorization header if cookie absent

## Bug #2: No Server Authentication Skip Mechanism

### Root Cause
**File**: `/app/cmd/flipt/main.go`
**Lines**: 485-489 (middleware registration)
**Issue**: Same interceptor applied to ALL services with no bypass mechanism

### Evidence
```go
// Lines 485-489 in main.go

authrpc.RegisterAuthenticationServiceServer(grpcServer, authServer)
```

### Design Problem
The authentication middleware is mandatory for all services with no:
- Per-service configuration options
- Exception lists or bypass mechanisms
- Public/private endpoint distinction

### Test Verification
The test demonstrates:
- ❌ **No public endpoints**: All services require authentication
- ❌ **Auth service requires auth**: Even the AuthenticationService requires auth
- ❌ **Chicken-and-egg problem**: CreateToken needs auth but is needed to create first token
- ❌ **Global enforcement**: All server instances have identical auth requirements

### Impact
- **Bootstrap problem**: Users can't create their first token
- **Delegated auth**: Internal delegation flows are impossible
- **Service isolation**: Can't have public vs. private service tiers
- **Architecture limitation**: Prevents certain deployment scenarios

### Example: Bootstrap Token Problem
1. User configures Flipt with `authentication.required = true`
2. User calls `CreateToken` to create their first token
3. CreateToken RPC requires authentication (same middleware)
4. User has no token to authenticate with
5. **Result**: Stuck in auth loop - can't create first token!

### Fix Required
Need to add:
1. Per-service authentication configuration
2. Exception list for public services
3. Bootstrap/initial setup endpoint
4. Optional server-specific auth bypass
