## Bug Report

**Bug ID**: 53441
**Summary**: hda_intel: sound stops working after suspend to ram
**Regression**: Yes (works on v3.6.11, broken since v3.7.1)

On a fresh boot the sound card works fine. After a few suspends to ram (it doesn't necessarily happen after just one) the sound card stops working well. The sound comes out choppy, as a series of clicks separated by 2 second or so silences.

I see these messages in the kernel log right after resume:
```
hda-intel: azx_get_response timeout, switching to polling mode: last cmd=0x00170503
hda-intel: No response from codec, disabling MSI: last cmd=0x00170503
hda_intel: azx_get_response timeout, switching to single_cmd mode: last cmd=0x00170503
```

The only way to fix the sound seems to be a reboot. I don't have this problem with v3.6.11, it appeared with kernel v3.7.1 (I have not tested v3.7). Right now I'm running 3.7.6 and the problem is still there.

## Fault Localization Summary

I've identified the root cause of the HDA Intel sound driver failure after suspend to RAM:

### **Buggy File**
- `sound/pci/hda/hda_intel.c`

### **Buggy Functions**
1. `azx_resume()` (line 2539) - System suspend/resume handler
2. `azx_runtime_resume()` (line 2580) - Runtime PM resume handler

### **Root Cause**

The bug occurs due to a missing reset of error-handling flags during device resume:

1. **Error Handling Phase**: When codec communication timeouts occur, `azx_rirb_get_response()` progressively enables error-handling modes:
   - Sets `polling_mode = 1` (line 912)
   - Disables MSI (line 923)
   - Sets `single_cmd = 1` (line 951) and calls `azx_free_cmd_io()` to disable CORB/RIRB

2. **Suspend Phase**: `azx_suspend()` calls `azx_stop_chip()` which sets `initialized = 0`

3. **Resume Phase** (THE BUG): `azx_resume()` and `azx_runtime_resume()` call `azx_init_chip()` to re-initialize, BUT:
   - They do NOT reset `single_cmd`, `polling_mode`, or `poll_count` flags
   - In `azx_init_chip()` at line 1222, there's a critical check: `if (!chip->single_cmd)`
   - Since `single_cmd = 1` from error handling, CORB/RIRB is NOT re-initialized
   - Device remains stuck in single_cmd mode, causing codec communication failures

4. **Result**: Choppy audio with 2-second silences after resume. Only solution is reboot (which clears all state).

### **Confidence: 0.95**
