## Key Findings

**There are NO separate v1/v2 resource handlers** in the Kafka Connect REST API. Instead, the implementation uses:

1. **Single Unified API**: All REST endpoints are served by a single set of resource classes (ConnectorsResource, RootResource, etc.) without version-specific variants.

2. **Backward-Compatible Schema Evolution**: 
   - Response schemas use optional fields marked with `@JsonInclude` annotations
   - Old clients safely ignore new fields
   - New clients can handle older responses missing new fields

3. **Storage-Level Versioning**: The only explicit versioning found is in `KafkaConfigBackingStore` with:
   - `TARGET_STATE_V0`: Legacy schema
   - `TARGET_STATE_V1`: Enhanced schema with optional `state.v2` field
   - This dual-field approach maintains compatibility with old workers

4. **Error Response Consistency**: All endpoints return errors in the same `ErrorMessage` format with `error_code` and `message` fields.

5. **Task Status Schema**: Returned via `ConnectorStateInfo` containing state, worker_id, trace, and version fields for both connectors and tasks.
