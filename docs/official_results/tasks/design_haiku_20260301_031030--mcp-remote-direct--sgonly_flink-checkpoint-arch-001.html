## Summary

I've mapped the complete checkpoint coordination flow across 8 phases:

1. **Checkpoint Trigger**: CheckpointCoordinator initiates, creates PendingCheckpoint, and dispatches RPC to sources
2. **Source Barrier Emission**: Sources inject CheckpointBarrier events into their output streams
3. **Barrier Propagation**: Barriers flow through the network DAG between data records
4. **Barrier Alignment**: Downstream tasks' barrier handlers track arrivals and enforce alignment (for exactly-once)
5. **Operator State Snapshot**: Once aligned, operators snapshot their state to the configured backend
6. **Acknowledgment**: TaskManagers send back AcknowledgeCheckpoint with state handles and metrics
7. **Finalization**: When all acks received, coordinator writes checkpoint metadata and creates CompletedCheckpoint
8. **Completion Notification**: Tasks notified to commit external state (Kafka, S3, etc.)

**Key Components**:
- **CheckpointCoordinator**: JobManager orchestrator managing PendingCheckpoint lifecycle
- **PendingCheckpoint**: Intermediate state tracking task acknowledgments
- **CheckpointBarrierHandler**: Enforces alignment semantics (SingleCheckpointBarrierHandler for exactly-once, CheckpointBarrierTracker for at-least-once)
- **Execution/Task**: RPC endpoints bridging JobManager and TaskManager
- **StreamTask**: Task-side checkpoint executor

The architecture implements a distributed two-phase commit with state machine-based barrier alignment, supporting both exactly-once (with aligned barriers + buffering) and at-least-once (no blocking) semantics.
