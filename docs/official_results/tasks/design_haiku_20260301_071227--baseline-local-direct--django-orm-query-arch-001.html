## Summary

I've traced the entire Django ORM query compilation architecture from high-level QuerySet API to SQL execution. The analysis has been saved to `/logs/agent/solution.md`.

### Key Findings:

**Architecture Layers:**
1. **Manager/QuerySet** - Public API layer (`filter()`, `exclude()`, etc.)
2. **Query Object** - Internal structure holding WHERE tree (WhereNode), JOINs, SELECT columns
3. **Compiler** - Generates SQL by visiting expression tree
4. **Backend-Specific** - Database adapters providing vendor dispatch

**Vendor Dispatch Mechanism:**
The system uses an `as_{vendor}()` method dispatch pattern where:
- `Compiler.compile(node)` checks if node has `as_postgresql()`, `as_mysql()`, etc.
- Falls back to generic `as_sql()` if not found
- Backend selection happens via `connection.ops.compiler_module` attribute

**Lazy Evaluation:**
- `QuerySet.filter()` doesn't execute SQL—only accumulates filters in `Query.where` (a WhereNode tree)
- SQL is only generated when iteration occurs (`__iter__()` → `_fetch_all()`)
- Creates immutable QuerySet copies via `_clone()` for method chaining

**Expression Tree Compilation:**
- WhereNode forms a composite tree of Lookup objects connected by AND/OR/NOT
- Compiler recursively compiles each expression via `compile()` dispatch
- Each Lookup combines left-hand side (field) + operator + right-hand side (value)

**10 Core Files Identified:**
manager.py, query.py (QuerySet), sql/query.py (Query), sql/compiler.py (base compiler), sql/where.py (WhereNode), expressions.py (Expression classes), lookups.py (Lookup class), base/operations.py (backend abstraction), and backend-specific compiler overrides.

# Django ORM Architecture Notes

## Key Architectural Insights

### Query Compilation Pipeline Phases

1. **Lazy Construction** (Manager/QuerySet)
   - `Manager.get_queryset()` creates QuerySet with empty Query
   - `QuerySet.filter()` accumulates filters via `Query.add_q()`
   - No SQL generated until iteration triggered

2. **Query Structure** (Query class)
   - Stores WHERE tree in `self.where` (WhereNode)
   - Stores JOINs in `self.alias_map`
   - Stores SELECT expressions in `self.select`
   - Method `get_compiler()` dispatches to backend-specific compiler

3. **SQL Generation** (Compiler.as_sql())
   - Iterates through Query structure
   - Recursively compiles expressions via `compile(node)` dispatch
   - Assembles SELECT/FROM/WHERE/GROUP BY/HAVING/ORDER BY clauses
   - Returns (sql_string, params_tuple)

4. **Vendor Dispatch** (compile() method)
   - Checks for `as_{vendor}()` method on expression (e.g., `as_postgresql()`)
   - Falls back to generic `as_sql()` if not found
   - Each backend defines vendor name via `connection.vendor`

5. **Execution** (execute_sql())
   - Calls `as_sql()` to get SQL and params
   - Gets cursor from connection
   - Calls `cursor.execute(sql, params)`
   - Returns iterator over results
