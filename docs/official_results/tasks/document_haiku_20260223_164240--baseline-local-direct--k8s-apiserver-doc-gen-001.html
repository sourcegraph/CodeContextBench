Conflict Handling: Optimistic Locking vs Server-Side Merge\n\n**Mechanism**: Every object has `metadata.resourceVersion` = etcd's mod_revision\n\n**Conflict Scenario:**\n```\nClient1: GET Pod \u2192 rv=5\nClient2: GET Pod \u2192 rv=5\nClient2: PUT Pod with rv=5 \u2192 Success, new rv=6\nClient1: PUT Pod with rv=5 \u2192 409 Conflict! etcd now has rv=6\nClient1: Must GET again, retry with rv=6\n```\n\n**Failure Mode:** High contention resources (e.g., frequently updated Service) require many retries, latency spikes\n\n**Trade-off:** Simple, non-blocking server code vs client retry logic. Server never needs to merge field conflicts.\n\n**Mitigation:** Server-side Apply (strategic merge patch) for multi-controller ownership:\n- Each controller registers a \"manager\" identity\n- Server tracks `managedFields` per manager\n- Merges non-overlapping field changes\n- File: `pkg/registry/generic/registry/store.go` with structured merge diff\n\n### 2. Admission Plugin Failures: Synchronous vs Asynchronous\n\n**Current:** Admission plugins block the request (synchronous)\n\n**Failure Mode:** Misbehaving webhook (timeout, crash) blocks all requests for that resource\n\n**Timeout Protection:**\n- Webhook timeout configured per policy (typical: 5 seconds)\n- On timeout: return 500 if policy.failurePolicy=Fail, allow if failurePolicy=Ignore\n- File: `pkg/admission/webhook/mutating/dispatcher.go`\n\n**Failure Policy Options:**\n- `Fail`: Reject request if webhook fails (safe for critical policies)\n- `Ignore`: Allow request if webhook fails (risky, only for non-critical)\n\n**Mitigation:**\n- Webhook latency monitored via metrics `apiserver_admission_webhook_admission_duration_seconds`\n- Cluster operators can adjust timeout or disable webhook\n\n### 3. Cacher Staleness: Watch Cache Lag\n\n**Issue:** Reflector may lag behind etcd if:\n- etcd very active (high event rate)\n- Slow watcher processing (backlog accumulates)\n- Network congestion\n\n**Failure Mode:** Client watches for change, but sees stale data\n\n**Symptom:** resourceVersion in watch event lags behind etcd current revision\n\n**Trade-off:**\n- **Memory bounded**: watchCache stores only N recent events (configurable via EventsHistoryWindow)\n- **Fallback to etcd**: If client requests too-old resourceVersion, must list from etcd (expensive)\n- **Bookmarks provide reassurance**: Periodic bookmarks show cache resourceVersion catching up\n\n**File:** `pkg/storage/cacher/cacher.go:1155` (Compact operation)\n\n### 4. Priority and Fairness: Queuing vs Rejection\n\n**Mechanism:** Classification + priority levels + queues\n\n**Failure Modes:**\n\na) **Queue Full:** Request rejected with 429 Too Many Requests\n- **Trade-off:** Protects server from unbounded memory, client must retry exponentially\n- **Risk:** Badly behaving clients retry in tight loops \u2192 thundering herd\n\nb) **Timeout in Queue:** Request queued but times out before execution\n- **Response:** 504 Gateway Timeout\n- **File:** `pkg/server/filters/priority-and-fairness.go:300`\n\nc) **Work Estimation Errors:** Request classified with wrong cost \u2192 priorities inverted\n- **Example:** A simple GET estimated as 100 units causes starvation of actual complex queries\n- **Mitigation:** adjustable estimators, metrics on estimation accuracy\n- **File:** `pkg/util/flowcontrol/request/work_estimator.go`\n\n### 5. Discovery Endpoint Staleness\n\n**Issue:** `/apis` discovery endpoint cached for performance\n\n**Failure Mode:** Newly installed CRD not immediately visible in discovery\n\n**Latency:** ~10-30 seconds before discovery cache refreshes\n\n**Trade-off:**\n- **Cache**: Reduces load on main request handler\n- **Freshness**: Delayed visibility of new API groups\n- **File:** `pkg/endpoints/discovery/discovery.go` with TTL\n\n**Mitigation:** Clients can call `/apis` directly if immediate visibility needed\n\n---\n\n## Extension Points and Customization Hooks\n\n### 1.
