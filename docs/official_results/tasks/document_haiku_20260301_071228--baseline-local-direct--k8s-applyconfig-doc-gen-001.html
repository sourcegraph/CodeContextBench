Typically includes API version and operation type for deduplication.\n\n**Operation**:\n- `Apply`: From PATCH with ApplyYAMLPatchType; idempotent\n- `Update`: From PUT/POST; overrides previous values\n\n**FieldsV1**: Internal representation using structured-merge-diff's fieldpath.Path API. Each field path is tracked separately.\n\n### Conflict Detection Algorithm\n\n**Trigger**: When `FieldManager.Apply()` is called with `force=false`:\n\n```go\n// managedfields/internal/fieldmanager.go:160+\nobj, managed, err = f.fieldManager.Update(liveObj, newObj, managed, manager)\nif conflict, isConflict := err.(*merge.Conflicts); isConflict && !force {\n    return nil, internal.NewConflictError(conflict)\n}\n```\n\n**Detection**: Delegated to structured-merge-diff:\n1. Decode managed fields from live object (knows current ownership)\n2. For each field in patch:\n   - If owned by a different manager AND value differs \u2192 conflict\n3. Return list of conflicting paths and their owners\n\n**Example**:\n- Live: `spec.replicas=3, owned by hpa`\n- Patch: `spec.replicas=5, sent by my-app`\n- Conflict: Path `[spec][replicas]`, manager `hpa`\n\n**Conflict Error**: HTTP 409 with causes per `/workspace/staging/src/k8s.io/apimachinery/pkg/util/managedfields/internal/conflict.go`:\n\n```json\n{\n  \"status\": \"Failure\",\n  \"code\": 409,\n  \"reason\": \"Conflict\",\n  \"details\": {\n    \"causes\": [\n      {\n        \"reason\": \"FieldManagerConflict\",\n        \"message\": \"conflict with \\\"hpa\\\"\",\n        \"field\": \"spec.replicas\"\n      }\n    ]\n  }\n}\n```\n\n### Force Apply Semantics\n\n**PatchOptions.Force = true** means: \"I acknowledge conflicts and claim disputed fields.\"\n\n**Server behavior**:\n1. Ignores conflicts during merge\n2. Removes conflicting fields from other managers' entries\n3. Adds those fields to requesting manager's entry\n4. Returns 200 OK with modified object\n\n**Critical caveat**: Force apply is **not** a three-way merge. The patch completely replaces fields, not merges them. If patch omits a list, entire list is deleted.\n\n### Backward Compatibility: Last-Applied-Configuration\n\n**Purpose**: kubectl apply traditionally stored `kubectl.kubernetes.io/last-applied-configuration` to compute three-way diffs (local \u2192 server \u2192 last-known). SSA renders this obsolete but maintains it for compatibility.\n\n**Manager**: `LastAppliedManager` wrapper in managedfields/internal/fieldmanager.go\n\n**Behavior**:\n- When Apply operation succeeds, annotation is updated to the patch that was applied\n- Non-Apply operations (PUT) don't modify it\n- Useful for clients that haven't migrated to native SSA\n\n---\n\n## Part 7: Extension Points and If invalid, **reverts** to pre-admission state and adds warning.\n\n**Limitation**: Validation only catches structural corruption (invalid JSON in fieldsV1). Webhooks can still add fields that bypass field ownership checks.\n\n#### Unbounded Manager Count\n\n**Problem**: If many different clients apply to same resource, managed fields grows without bound.\n\n**Mitigation** (CapManagersManager): Limit to DefaultMaxUpdateManagers=10. Older Update entries are merged (combined).\n\n**Limitation**: Compressed history loses field ownership details for old managers. After compression, can't extract precise previous configurations.\n\n**Impact**: If >10 controllers touch a resource, some history is lost.\n\n#### Conflict on Apply with Small Patch\n\n**Problem**: User applies a partial patch that conflicts with another controller's full specification.\n\n**Example**:\n```yaml\n# Controller A owns full Pod spec\n# User tries to apply just image: \"v2\"\n# But controller A had image: \"v1\"\n# Conflict on image!\n```\n\n**Solution**: Force-apply, but only if intentional.\n\n**Limitation**: Force-apply is all-or-nothing. Can't force specific fields.\n\n#### Schema Mismatch with CRDs\n\n**Problem**: CRD schema changes (list field becomes map), but old managed fields still reference old structure.\n\n**Mitigation**: If schema can't be found during merge, falls back to scalar merge (no recursion).\n\n**Limitation**: Older managed fields entries for pre-schema-change structure become stale.\n\n#### Lost Updates via GuaranteedUpdate Retry\n\n**Problem**: HTTP request timeout during GuaranteedUpdate retry loop.\n\n**Mitigation** (patch.go:734-736): If timeout error received, server can't guarantee whether object was persisted.\n\n**Behavior**: Returns 504 Gateway Timeout. Client must retry with awareness that object may have been created.\n\n**Limitation**: No way to know if partial application succeeded. Requires idempotent apply semantics.\n\n### 8.2 Limits\n\n| Limit | Value | Location | Note |\n|-------|-------|----------|------|\n| Max JSON Patch operations | 10,000 | patch.go:64 | Per RFC 6902 |\n| Max Update managers | 10 | managedfields/internal/fieldmanager.go:35 | Older entries merged |\n| Max request body size | Configurable (default 3MB) | handlers/patch.go:110 | Via apiserver flag |\n| Max field path depth | Unbounded | structured-merge-diff | Theoretical limit depends on memory |\n| Managed fields JSON size | Unbounded | Storage | Large numbers of managers \u2192 larger objects |\n\n### 8.3 Design Tradeoffs\n\n#### Tradeoff 1: Field-Level vs. Resource-Level Ownership\n\n**Choice**: Kubernetes uses field-level tracking.\n\n**Pro**: Multiple controllers can co-manage same resource at different fields. HPA sets replicas, user manages selector.\n\n**Con**: Adds complexity, requires schema understanding, merge logic more intricate.\n\n**Alternative**: Resource-level (last-write-wins). Simpler but forces exclusive ownership.\n\n#### Tradeoff 2: Three-Way Merge vs. Two-Way\n\n**Choice**: Kubernetes uses two-way merge (current state + patch), reconstructs previous state from managed fields.\n\n**Pro**: Doesn't require client to send three-way state, reduces API complexity.\n\n**Con**: Reconstructed \"previous\" state may be incomplete (other managers may have modified fields).\n\n**Example**:\n```\nLive state:  replicas=5, owned by HPA\nPatch:       replicas=3, sent by user\nReconstructed \"previous\" from managed fields: HPA owns replicas (assumed value is 5)\nMerge result: Conflict (HPA disagrees with user's 3)\n```\n\n#### Tradeoff 3: Force-Apply Atomicity\n\n**Choice**: Force-apply claims all fields in patch, not selective.\n\n**Pro**: Simple to implement, clear semantics.\n\n**Con**: Can't force-apply specific fields, must accept all-or-nothing.\n\n**Workaround**: Extract, modify specific fields, reapply (avoids conflicts).\n\n#### Tradeoff 4: Manager String Format\n\n**Choice**: Opaque JSON string containing { manager, operation, apiVersion }.\n\n**Pro**: Extensible, can add fields for future needs.\n\n**Con**: Not human-readable, requires JSON parsing to inspect.\n\n**Alternative**: Structured tuple (manager, op, version).\n\n---\n\n## Part 9: Operational Implications\n\n### 9.1 Performance Impact\n\n#### Apply Patch Processing\n\n**Cost**: Structured-merge-diff three-way merge is O(n) in live object size and patch size.\n\n**Optimization 1** (ProbabilisticSkipNonAppliedManager): For non-Apply operations, skip merge if fields are likely untouched (configurable probability, default 100% skip).\n\n**Optimization 2** (CapManagersManager): Merge old Update entries to keep manager count low.\n\n**Optimization 3** (patch.go:713-726): If object too large, retry after removing managedFields. Managed fields can be >100KB on large objects with many managers.\n\n**Result**: First few applies are slow (merge), subsequent non-apply updates are fast (skip).\n\n#### Managed Fields Storage Overhead\n\n**Typical**: 0.5-2KB per managed entry, depends on number of fields tracked.\n\n**Large scale**: 10 managers \u00d7 500 fields average = ~5KB managed fields metadata.\n\n**Impact**: Cluster with 10,000 Pods \u00d7 5KB = 50MB total managed fields metadata in etcd.\n\n**Mitigation**: Regularly delete old/unused controllers to keep manager count low.\n\n### 9.2 Debugging and Troubleshooting\n\n#### Inspect Managed Fields\n\n```bash\nkubectl get pod <name> -o yaml | grep -A 50 \"managedFields:\"\n```\n\nShows which manager owns which fields, when last modified.\n\n#### Extract Configuration\n\n```bash\n# Extract what \"my-app\" manager owns\nkubectl get deployment <name> -o yaml | \\\n  jq '.metadata.managedFields[] | select(.manager==\"my-app\")'\n```\n\nHelps understand what a controller thinks it should own.\n\n#### Conflict Resolution\n\n```bash\n# Check conflict details\nkubectl apply -f config.yaml 2>&1 | grep \"conflict with\"\n\n# Force apply to claim fields\nkubectl apply -f config.yaml --field-manager=my-app --force-conflicts\n```\n\n#### Manager Cleanup\n\n```bash\n# List active managers on a resource\nkubectl patch deployment <name> --type='json' -p='[{\"op\":\"test\",\"path\":\"/metadata/managedFields\"}]'\n\n# Manually clear stale manager (CAUTION: only if manager is truly abandoned)\nkubectl patch deployment <name> --type='json' \\\n  -p='[{\"op\":\"remove\",\"path\":\"/metadata/managedFields/0\"}]'\n```\n\n### 9.3 Migration Considerations\n\n#### From Traditional PUT/POST to Apply\n\n**Challenge**: Existing resources have no managed fields. First apply sees all fields as unowned.\n\n**Mitigation**: Initial apply claims all fields:\n```bash\nkubectl apply -f existing-resource.yaml --field-manager=new-manager --force-conflicts\n```\n\n**Result**: Migrates resource to SSA model.\n\n#### From Client-Side Apply to Server-Side Apply\n\n**Tool**: `kubectl apply --server-side` (requires cluster v1.18+).\n\n**Process**:\n1. Read current state (live object + managedFields)\n2. Extract last-applied-configuration annotation\n3. Compute three-way diff (last \u2192 live \u2192 desired)\n4. Send as apply PATCH\n5. Server merges with other manager changes\n\n**Compatibility**: Server-side apply is backward compatible; old resources automatically gain managed fields.\n\n#### Multi-Version Controller Rollout\n\n**Scenario**: Rolling out new version of controller with different field ownership.\n\n**Approach**:\n1. Old and new versions coexist (different manager strings, e.g., \"my-controller:v1\" vs \"my-controller:v2\")\n2. Each manages same fields but with different manager names\n3. On rollout: New version applies same fields, old version's entries become stale\n4. After stabilization: Run cleanup to merge old manager entries\n\n**Caution**: If both versions apply simultaneously, conflicts arise until old version is deleted.\n\n### 9.4 Reliability and Data Consistency\n\n#### Idempotency\n\n**Property**: Applying same configuration multiple times produces same result.\n\n**Guaranteed by**: Apply with same field values always succeeds (no conflict on identical fields).\n\n**Exception**: External changes (e.g., HPA sets replicas differently) cause conflicts on next apply.\n\n**Implication**: Controllers can safely retry apply on transient errors without duplicating side effects.\n\n#### Observability\n\n**What's Tracked**:\n- Each managed field's owner, timestamp, operation\n- **Not tracked**: When fields were set, intermediate values, change # Kubernetes Server-Side Apply and ApplyConfigurations: A Deep Systems Guide

## Executive Summary

Server-Side Apply (SSA) is a declarative configuration management system in Kubernetes that tracks field ownership through **managed fields**, enabling multiple controllers to safely co-manage resources without conflicts. The subsystem comprises three major parts: (1) **applyconfigurations**, auto-generated builder types that provide a fluent API for constructing apply patches; (2) **server-side field management**, which tracks field ownership and detects conflicts; and (3) **structured-merge-diff**, an external library implementing the merge algorithm that respects field ownership boundaries.

This guide explains how these components work together, the control flow from client through server, conflict resolution semantics, and operational considerations.

---

## Part 1: Subsystem Purpose and Boundaries

### Purpose

The Server-Side Apply subsystem solves a fundamental problem in declarative configuration management: **how do multiple parties (controllers, operators, users) safely modify the same resource without overwriting each other's changes?**

Before SSA, Kubernetes had no way to track field ownership. When controller A and user B both modified a Pod, their changes could overwrite each other unpredictably. SSA introduces field-level tracking via `metadata.managedFields`, creating a clear ownership model:

- **Users/Controllers** declare intent: "I want to own these fields"
- **Server tracks** who owns what through managed fields entries
- **Server detects conflicts** when two managers disagree on field values
- **Clients can force-apply** to claim disputed fields, with explicit acknowledgment
