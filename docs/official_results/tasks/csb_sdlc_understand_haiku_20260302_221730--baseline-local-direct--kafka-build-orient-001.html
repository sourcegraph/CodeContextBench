## Key Findings:

   - Root configuration in `/workspace/build.gradle`
   - Multiple subprojects organized by responsibility

   - **ZooKeeper Mode**: `KafkaServer` class for traditional setup
   - **KRaft Mode**: `KafkaRaftServer` for new consensus-based approach
   - Entry point: `kafka.Kafka.main()` loads `server.properties`

### 3. **Module Structure**: Well-organized codebase
   - Core modules: server, controller, log, network, coordinator, cluster, zk, raft
   - Clear separation of concerns with dedicated packages for each functionality
   - Over 15 major modules + supporting modules for clients, streams, connect

### 4. **Topic Creation Flow**: Request → Controller → Metadata Store → Brokers
   - ZK path: Admin request → KafkaApis → Controller → ZooKeeper → Brokers
   - KRaft path: Admin request → ControllerApis → Metadata Log → Brokers
   - Key classes: `ControllerApis`, `KafkaApis`, `ZkAdminManager`

### 5. **Testing Framework**: JUnit 5 + Mockito with integration tests
   - Unit tests in `/core/src/test/scala/unit/`
   - Integration tests in `/core/src/test/scala/integration/`
   - Uses `TestUtils` for creating test configurations

### 6. **Configuration System**: Hierarchical with dynamic updates
   - Defined in `*Configs.java` files (ServerConfigs, ZkConfigs, etc.)
   - Merged via `AbstractKafkaConfig.CONFIG_DEF`
   - Dynamic configs via `DynamicBrokerConfig` with `Reconfigurable` handlers
   - 4-level precedence: dynamic broker → dynamic defaults → static → built-in defaults
