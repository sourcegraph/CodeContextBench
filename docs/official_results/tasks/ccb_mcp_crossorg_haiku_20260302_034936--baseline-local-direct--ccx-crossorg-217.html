**Middleware Stack Construction:**
- Django processes `MIDDLEWARE` setting in **reverse order** (line 41 of base.py)
- Each middleware wraps the previous handler, creating a nested "onion" structure
- The last middleware in the list becomes the innermost wrapper

**Calling Order by Hook Type:**
- **process_request()**: Forward order (first middleware first) - executed as requests flow inward
- **process_view()**: Forward order - executed before the view
- **process_response()**: Reverse order - executed as responses flow back out
- **process_template_response()**: Forward order - executed after view returns
- **process_exception()**: Forward order - executed when exceptions occur

**Django vs WSGI Middleware:**
- Django uses multiple hook methods within each middleware, whereas WSGI uses single `__call__` wrapping
- Django has a MiddlewareMixin that orchestrates the different phases
- Automatic exception-to-response conversion at each middleware layer
- Support for both sync (WSGI) and async (ASGI) execution modes
- Django-specific hooks (process_view, process_template_response) don't exist in standard WSGI
