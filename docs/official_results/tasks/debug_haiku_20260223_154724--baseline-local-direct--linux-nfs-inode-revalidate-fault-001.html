Everything mounts normally, but after mount expiration (during unmount) autofs mount disappears - that is, not only /storage/<volume> unmounts, but /storage is also unmounted. This happens quite often - usually on the very first expiration.

automount logs umount error in such cases.

```
NFS reply getattr: -512

```

tcpdump shows, that server replies to getattr without error, but client doesn't see this reply, and returning error instead.

## Summary

I've successfully localized the NFS mount disappearance bug to the **`nfs3_rpc_wrapper` function in `fs/nfs/nfs3proc.c`** (lines 30-44).

### Root Cause Analysis

**The Bug:** The `nfs3_rpc_wrapper` function has a critical flaw in its JUKEBOX error retry logic:

1. When the NFS server responds with a `JUKEBOX` error (indicating a temporary server-side delay), the wrapper enters a retry loop
2. It calls `freezable_schedule_timeout_killable_unsafe()` to sleep and retry
3. **After the sleep, it unconditionally sets `res = -ERESTARTSYS`**
4. If a fatal signal arrives during this sleep (which happens during autofs mount expiration/unmount), the while loop condition `!fatal_signal_pending(current)` becomes false
5. The function returns **-ERESTARTSYS (-512) without retrying the RPC call**, even though the server may have already sent a valid reply

**Why This Causes the Bug:**
- The getattr RPC call fails with error -512
- This causes `nfs_revalidate_inode()` to fail (as seen in kernel logs)
- `nfs_lookup_revalidate()` marks the `/storage` dentry as invalid
- The `/storage` autofs mount is unmounted, along with all submounts

**Why It's a Regression (3.14.12 → 4.1.15):**
The issue occurs specifically during autofs mount expiration when the kernel sends signals to tasks waiting on the NFS mount to trigger unmount cleanup. In earlier kernels, the signal handling or RPC retry logic likely worked differently.
