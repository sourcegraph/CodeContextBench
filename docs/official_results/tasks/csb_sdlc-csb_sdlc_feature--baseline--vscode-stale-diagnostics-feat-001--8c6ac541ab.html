## Description

Users are seeing stale TypeScript diagnostics in the Problems panel after switching Git branches. Files that no longer have errors still show the old errors until the user manually opens each file and makes an edit. It seems like the diagnostics pipeline only refreshes on in-editor changes and misses file-on-disk changes from Git operations.

## Task

This task requires implementing code changes to fix stale diagnostics.

### Required Implementation

Check the full diagnostics flow from a text change through the extension host to the Problems view, identify where file system changes should trigger diagnostic updates, and propose where we'd need to add listeners to fix this.

2. **Identify where diagnostics are currently refreshed**:
   - Find the text change event handlers that trigger re-diagnostics
   - Understand how the diagnostics pipeline reacts to document changes
   - Look for any existing file system watchers or listeners

3. **Add file system change triggers**:
   - Add listener for file changes on disk (e.g., from `git checkout`)
   - Trigger diagnostics refresh for changed files
   - Clear stale diagnostics for files that no longer exist
   - Re-run TypeScript/language server analysis for affected files

4. **Test the fix**:

# Technical Documentation: Stale Diagnostics Fix

## Problem Analysis

### Issue Description
When users switch Git branches using `git checkout`, files on disk change but the TypeScript Language Server in VS Code continues to show diagnostics based on the old file contents. This is because:

1. The file on disk has new content (from the switched branch)
2. VS Code's text editor in memory may or may not have updated the content
3. The TypeScript Language Server (TS Server) has the old content in memory
4. VS Code's diagnostics pipeline is based on the TS Server's analysis
5. The TS Server only gets re-analyzed when the editor content changes

### Root Cause
The `BufferSyncSupport` class handles synchronizing files between the editor and the TS Server. It triggers re-diagnostics when:
- Files are opened or closed
- File content changes within the editor
- Text editors become visible

**It does NOT trigger re-diagnostics when files change on disk outside the editor.**

## Solution Architecture

### Component: File System Watchers

**Location**: `BufferSyncSupport.listen()` method
**Pattern**: Observer pattern using VS Code's `createFileSystemWatcher`

1. **Multiple Watchers vs. Single Watcher**
   - **Choice**: Multiple watchers for different file extensions
   - **Reason**: More explicit, easier to maintain, clearer intent
   - **Alternative Considered**: Single glob pattern (e.g., `**/*.{ts,tsx,js,jsx}`)
   - **Reason Not Used**: VS Code glob patterns may not support brace expansion consistently

2. **Which Events to Monitor**
   - **`onDidChange`**: Triggered when files are modified
   - **`onDidDelete`**: Triggered when files are deleted
   - **`onDidCreate`**: Not used (new files don't have stale diagnostics)
