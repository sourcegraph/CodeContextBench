### Key Findings:

**Root Cause**: The v38 schema migration contains broken conditional logic that skips processing entirely when `fieldConfig.defaults.custom` doesn't exist in a dashboard's JSON.

**The Specific Problem**:
- **Backend** (`v38.go`, lines 115-137): Uses a type assertion that fails silently when `custom` object is missing, causing the entire panel migration to be skipped
- **Frontend** (`DashboardMigrator.ts`, lines 644-674): Uses optional chaining that skips defaults migration when `custom` is missing

**Why It Only Affects Dashboards Without Explicit `defaults.custom`**:
- When `fieldConfig.defaults.custom` IS explicitly defined, the migration works correctly
- When `fieldConfig.defaults.custom` is missing (common in older dashboards with minimal configurations), the migration skips to the next panel without migrating field overrides
- The default `fieldConfig` structure (defined in `PanelModel.ts`) only initializes `defaults: {}` with no `custom` object

**The Silent Failure**:
The migration doesn't error—it just silently skips processing, leaving `displayMode` un-migrated to `cellOptions`, which causes field override configuration to be lost, resulting in missing column formatting.

**Evidence**:
- All test cases in `v38_test.go` have explicit `fieldConfig.defaults.custom` defined—there's no test for the failing scenario
- Earlier migrations (v30, v35) handle missing objects gracefully; v38 breaks this pattern
- The test gap directly correlates with the regression
