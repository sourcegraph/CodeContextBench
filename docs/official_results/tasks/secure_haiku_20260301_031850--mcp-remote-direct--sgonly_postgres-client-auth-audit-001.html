# big-code-pg-sec-001: PostgreSQL Client Authentication Pipeline Security Analysis

## Task

Trace the PostgreSQL client authentication pipeline from the point a TCP connection is accepted through HBA (Host-Based Authentication) rule matching, password/SCRAM-SHA-256 verification, and role validation to session establishment. Identify all entry points where untrusted client data enters the authentication subsystem, map the complete data flow through each verification stage, and analyze the security properties of the authentication chain.

## Context

- **Repository**: github.com/sg-evals/postgres--5a461dc4 (mirror of github.com/sg-evals)/postgres--5a461dc4 (mirror of postgres/postgres) (C, ~1.5M LOC)
- **Category**: Security Analysis
- **Difficulty**: hard
- **Subsystem Focus**: `src/backend/libpq/` (auth.c, auth-scram.c, auth-sasl.c, crypt.c, hba.c) and `src/backend/utils/init/` (postinit.c, miscinit.c)

## Requirements

1. Identify all entry points where untrusted data enters the authentication subsystem (startup packet, password packets, SASL tokens)
2. Trace data flow from each entry point through transformations to sensitive operations (role lookup, password comparison, session establishment)
3. Map the HBA configuration matching pipeline (`load_hba` -> `check_hba` -> `hba_getauthmethod`)
4. Trace the password verification chain through both plaintext (`CheckPasswordAuth`) and challenge-response (`CheckPWChallengeAuth`) paths
5. Trace the SCRAM-SHA-256 exchange through the SASL framework (`auth-sasl.c`) to the SCRAM implementation (`auth-scram.c`)
6. Identify all files in the attack surface and document existing mitigations

Analysis\n\n### Authentication Architecture\n\nPostgreSQL uses a three-stage authentication model:\n\n1. **Stage 1: Connection Acceptance & HBA Matching**\n   - Postmaster spawns backend for each TCP connection\n   - Backend reads startup packet containing username and database\n   - HBA rules matched against: connection type, IP, database, username\n   - Auth method determined (trust, password, md5, scram, gss, ldap, pam, etc.)\n   - This stage has NO cryptographic verification; only config-based routing\n\n2. **Stage 2: Method-Specific Authentication**\n   - **Trust**: No password required (if HBA says so)\n   - **Password**: Client sends plaintext \u2192 compared against stored hash\n   - **MD5**: Server sends salt \u2192 client hashes password+salt \u2192 server verifies\n   - **SCRAM-SHA-256**: Multi-round challenge-response with cryptographic proof\n   - **GSS/SSPI/LDAP/PAM**: Delegated to external systems\n\n3. **Stage 3: Session Initialization**\n   - After authentication success, authenticated user context set\n   - Backend connects to requested database\n   - Transaction processing begins\n\n### Attack Surface & Entry Points\n\n**Untrusted Data Entry regardless of user existence\n   - Prevents username enumeration via timing side-channels\n\n2. **Password Never Transmitted (SCRAM & MD5)**\n   - SCRAM: Client never sends plaintext; only cryptographic proof\n   - MD5: Client sends hash (not plaintext password)\n   - Plaintext: Only when explicitly configured (legacy)\n\n3. **Cryptographic Verification**\n   - SCRAM: HMAC-SHA256 with 4096+ iterations (PBKDF2)\n   - MD5: Single MD5(MD5(password+user)+salt) (deprecated)\n   - Plaintext: byte-wise comparison (no hashing, highest risk)\n\n4. **HBA-Based Authorization**\n   - IP-based access control before authentication\n   - Database/role-based filtering\n   - Can reject connections before password verification\n\n5. **Channel Binding (SCRAM with TLS)**\n   - Prevents connection hijacking via MITM\n   - Verifies client is on same TLS connection as server\n   - Protects against credential disclosure on compromised network\n\n6. **Configuration-Driven Security**\n   - pg_hba.conf allows fine-grained access control\n   - Can require SSL for certain connections\n   - Can use external auth (LDAP, GSS)\n   - Can enforce strong auth methods (reject MD5)\n\n**Weaknesses & Vulnerabilities:**\n\n1. **Username Field Not Validated (CWE-601: URL Redirection)**\n   - Username from startup packet used directly in:\n     - Catalog lookups: SearchSysCache1(AUTHNAME, ...)\n     - Log messages: errmsg(...user \\\"%s\\\"...)\n   - If username contains special chars/nulls, could affect:\n     - Log parsing (if unescaped)\n     - Error messages displayed to client\n   - **Mitigation Present**: Username truncated to NAMEDATALEN; treated as opaque key\n\n2. **Case Folding in Username Comparison**\n   - Startup packet username compared against HBA role list (case-sensitive by default)\n   - PostgreSQL treats role names as case-insensitive internally\n   - Could allow \"admin\" and \"ADMIN\" to match different HBA rules\n   - **Known behavior**: Case-folding applied consistently in catalog\n\n3. **MD5 Authentication Deprecated**\n   - MD5 is broken for collision resistance (though collision not helpful for auth)\n   - Single-round hashing (no PBKDF2)\n   - Vulnerable to brute-force with rainbow tables\n   - **Mitigation**: SCRAM-SHA-256 strongly preferred; MD5 must be explicitly enabled in hba.conf\n\n4. **Plaintext Password in Transit (if plaintext auth)**\n   - \"password\" auth method sends plaintext over network\n   - **Mitigation**: Requires SSL/TLS in production (enforced via HBA \"hostssl\" type)\n   - **Acceptable for**: Unix domain socket connections (localhost, trusted)\n\n5. **SCRAM-SHA-256 Iteration Count**\n   - Iterations stored in password hash, not configurable per-password\n   - Current default: 4096 (may be low by modern standards)\n   - **Mitigation**: Configurable via password_encryption GUC; 15000+ recommended\n\n6. **Role Lookup Happens Even If User Doesn't Exist**\n   - get_role_oid(role, true) in check_hba [hba.c:2538]\n   - If role doesn't exist, OidIsValid returns false, continue HBA matching\n   - **Intent**: Allow HBA rules to match even for non-existent roles (e.g., reject rule)\n   - **Potential Issue**: Malicious user can probe for role existence via connection attempts\n   - **Mitigation**: Mock authentication for non-existent users ensures same timing\n\n7. **No Rate Limiting on Authentication Attempts**\n   - Multiple failed password attempts not rate-limited by server\n   - Attacker can brute-force weak passwords\n   - **Mitigation**: Authentication timeout (AuthenticationTimeout GUC) limits hanging connections\n   - **Recommendation**: Use pgbouncer or pg_stat_statements to detect brute-force\n\n8. **SASLprep Processing Lenient**\n   - auth-scram.c comments note: \"If password isn't valid UTF-8, raw bytes used\"\n   - Could allow different passwords to normalize to same value\n   - **Mitigation**: Consistent application; no known collision computational load |\n| Plaintext password in transit | SSL/TLS encryption layer; \"hostssl\" in HBA |\n| Weak hashing (MD5) | SCRAM-SHA-256 available; MD5 deprecated |\n| Brute-force password guessing | PBKDF2 iterations (4096+); authentication timeout |\n| Connection hijacking (SCRAM) | Channel binding via TLS server certificate fingerprint |\n| Log injection | Username truncated, but not Attack Scenarios\n\n**Scenario 1: Username Enumeration**\n- Attacker tries login attempts with common usernames\n- For non-existent user: server goes through mock authentication (same timing)\n- For
