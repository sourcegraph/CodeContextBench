# Bug Investigation: Application Crashes When Ad-Blocker Cache Is Corrupted

**Repository:** qutebrowser/qutebrowser
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

When qutebrowser starts up and attempts to load the ad-blocker filter cache, the application crashes if the cache file is corrupted or contains invalid data.

The expected behavior is that the application handles corrupted cache files gracefully — it should log an error, discard the invalid cache, and continue running normally (allowing the user to browse without ad blocking until the cache is rebuilt).

Instead, the deserialization error propagates as an unhandled exception, causing a crash before the browser window even appears. Users who encounter a corrupted cache file cannot use qutebrowser at all until they manually delete the cache file.

This is particularly problematic because cache corruption can happen for reasons outside the user's control, such as a system crash or disk error during a cache write.

## Constraints

- Your test must fail on the current (buggy) code for the RIGHT reason
- The test should simulate a corrupted cache file and verify the crash occurs
- Test timeout: 60 seconds

## Summary

I have successfully completed the bug investigation and created a comprehensive regression test for the qutebrowser ad-blocker cache corruption bug.

### **Root Cause Found**

**Location**: `/app/qutebrowser/components/braveadblock.py` (lines 204-231)

**The Problem**: 
- When the cache file is corrupted, `deserialize_from_file()` raises `adblock.DeserializationError`
- The code attempts to catch `ValueError`, but `DeserializationError` inherits from `BlockerException`, NOT `ValueError`
- This mismatch causes the exception to propagate unhandled, crashing the application

**Exception Hierarchy** (the issue):
```
ValueError          ← Code tries to catch this
  (doesn't match)
adblock.DeserializationError  ← But this is what gets raised
  → adblock.BlockerException
     → Exception
```

The bug is located in `/app/qutebrowser/components/braveadblock.py` in the `read_cache()` method (lines 204-231).

### Code Issue
```python
try:
    self._engine.deserialize_from_file(str(self._cache_path))
except ValueError as e:
    if str(e) != "DeserializationError":
        raise
    message.error("Reading adblock filter data failed (corrupted data?). "
                  "Please run :adblock-update.")
```

### The Problem
1. When `deserialize_from_file()` encounters corrupted cache data, it raises `adblock.DeserializationError`
2. The code attempts to catch `ValueError`, but `adblock.DeserializationError` inherits from `adblock.BlockerException`, NOT `ValueError`
3. Because the exception type doesn't match, it is not caught by the `except ValueError` clause
4. The exception propagates up as unhandled, causing the application to crash

### Root Cause Details
The exception hierarchy is:
```
adblock.DeserializationError -> adblock.BlockerException -> Exception
  (NOT a ValueError)
```

Therefore, the `except ValueError` handler will never catch `adblock.DeserializationError`, regardless of the error message check.

## Expected Behavior
The application should:
1. Detect the corrupted cache file
2. Catch the deserialization error
3. Display an error message to the user
4. Continue running normally (allowing the user to browse without ad-blocking until the cache is rebuilt)

## Actual Behavior
The application crashes with an unhandled `adblock.DeserializationError` exception before the browser window appears.

## Test Case
The regression test in `/workspace/regression_test.py` demonstrates this bug:

1. **test_corrupted_cache_file_raises_unhandled_exception**:

3. **test_valid_cache_file_loads_successfully**:
   - Verifies that valid cache files still load correctly
   - Test PASSES on both buggy and fixed code

## How to Fix
The `except ValueError` clause should be changed to `except adblock.DeserializationError` (or more broadly, `except Exception` to handle all adblock-related exceptions).

After the fix, the exception will be properly caught, the error message will be displayed to the user, and the application will continue running.
