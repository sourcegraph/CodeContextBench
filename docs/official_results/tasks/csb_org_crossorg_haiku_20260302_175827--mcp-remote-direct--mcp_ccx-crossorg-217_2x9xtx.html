## How It Works

**Middleware Stack Construction:**
- Middleware from `settings.MIDDLEWARE` is loaded in **reversed order** (line 41 of base.py)
- This creates a nested callable structure resembling an "onion" where:
  - Each middleware wraps the next one
  - Request flows through outer layers first (top-down)
  - Response flows back through the same layers (bottom-up)

**Request/Response Processing:**
- **Request phase**: Middleware processes in the order defined in MIDDLEWARE (top-down)
- **Response phase**: Responses pass back through the same layers that processed the request
- **Short-circuiting**: If a middleware returns a response without calling `get_response()`, inner layers never execute

**Key Differences from Old MIDDLEWARE_CLASSES:**
1. The new MIDDLEWARE behaves like an "onion" - only layers that saw the request see the response
2. Exceptions from middleware are converted to responses (via `convert_exception_to_response()`)
3. process_response methods are only called for middleware that processed the request
4. More predictable and consistent exception handling
