## Key Findings

### Core Implementation Files:
1. **django/core/handlers/base.py** - Contains `BaseHandler` with the critical `load_middleware()` method
2. **django/core/handlers/wsgi.py** - WSGI interface implementation
3. **django/core/handlers/exception.py** - Exception handling wrapper
4. **django/utils/deprecation.py** - Modern `MiddlewareMixin` pattern

### Middleware Stack Construction:
- Django iterates through `settings.MIDDLEWARE` in **reverse order** (line 41 in base.py)
- Each middleware is instantiated as `middleware(adapted_handler)`, wrapping the previous handler
- Creates a nested callable chain: `A(B(C(get_response)))`
- Each middleware is wrapped with exception handling via `convert_exception_to_response()`

### Request/Response Order:
- **Request**: A → B → C → view (middleware called in config order via process_request hooks)
- **Response**: view → C → B → A (response flows back in reverse)
- Special hooks like `process_view`, `process_template_response`, and `process_exception` are stored in separate lists with their own orderings

### Django vs WSGI Middleware:
Django uses a modern **inversion-of-control pattern** where middleware receives a `get_response` callable and executes code before/after calling it. This is fundamentally different from traditional WSGI middleware that directly wraps the next application. This allows for cleaner request and response processing semantics.
