## Key Findings:

**Files Examined (13 core files):**
- Manager, QuerySet, Query (ORM layer)
- SQLCompiler, WhereNode (SQL generation)
- Lookup, Expression (lookup/expression system)
- Backend operations and vendor-specific compilers

**Dependency Chain:**
Traces the complete flow from `Model.objects.filter()` → lazy QuerySet building → Query.add_q() → Query.get_compiler() (vendor dispatch) → SQLCompiler.as_sql() → expression compilation via Lookup.as_sql() → cursor.execute() → model instantiation

**Design Patterns Identified:**
1. **Lazy Evaluation** - QuerySet defers SQL generation until iteration
2. **Tree-Structured Representation** - WHERE clauses as WhereNode trees with Lookup leaf nodes
3. **Double-Dispatch Compilation** - `compiler.compile()` tries vendor-specific `as_{vendor}()` before falling back to generic `as_sql()`
4. **Vendor Dispatch** - Backend's `compiler_module` attribute selects vendor-specific compilers
5. **Expression Protocol** - All compilable objects implement `as_sql(compiler, connection)` for composability

**Key Architecture Insight:**
The vendor dispatch mechanism works via:
- Each backend (PostgreSQL, MySQL, etc.) overrides `DatabaseOperations.compiler_module`
- `Query.get_compiler()` uses `connection.ops.compiler("SQLCompiler")` to import the right compiler class
- `SQLCompiler.compile(node)` then attempts node-level vendor dispatch via `getattr(node, "as_" + connection.vendor)`

The analysis includes a detailed concrete example showing how `Book.objects.filter(title__startswith="Django", author__name="Alice")` flows through each layer to generate SQL.
