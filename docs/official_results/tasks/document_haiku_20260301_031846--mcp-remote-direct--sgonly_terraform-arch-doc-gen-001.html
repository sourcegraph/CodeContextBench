State Management\n\n### 3.1 State Structure\n\nThe `State` type represents Terraform's persistent state:\n\n```\nState\n\u251c\u2500\u2500 Modules: map[string]*Module\n\u2502   \u2514\u2500\u2500 Module (for each module instance)\n\u2502       \u251c\u2500\u2500 Resources: map[string]*Resource\n\u2502       \u2502   \u2514\u2500\u2500 Resource\n\u2502       \u2502       \u2514\u2500\u2500 Instances: map[addrs.InstanceKey]*ResourceInstance\n\u2502       \u2502           \u2514\u2500\u2500 ResourceInstance\n\u2502       \u2502               \u251c\u2500\u2500 Current: *ResourceInstanceObject\n\u2502       \u2502               \u2514\u2500\u2500 Deposed: map[DeposedKey]*ResourceInstanceObject\n\u2502       \u2514\u2500\u2500 Outputs: map[string]*OutputValue\n\u251c\u2500\u2500 RootOutputValues: map[string]*OutputValue\n\u2514\u2500\u2500 CheckResults: *CheckResults\n```\n\n**Key Properties:**\n- **Hierarchical**: Organized by modules and resources\n- **Instance-Based**: Resources can have multiple instances (count/for_each)\n- **Deposed Objects**: Tracks old objects during create-before-destroy operations\n- **Immutable Interface**: State should only be modified through `SyncState` methods\n\n### 3.2 State Synchronization (SyncState)\n\nFor concurrent graph execution, state is wrapped in `SyncState`:\n\n```go\ntype SyncState struct {\n    state    *State\n    writable bool\n    lock     sync.RWMutex  // Protects concurrent access\n}\n```\n\n**Features:**\n- Atomic operations: Multiple state changes happen together or not at all\n- Read-Write separation: Multiple readers or single writer\n- Deep copying: Returned values are independent copies\n- Automatic pruning: Empty modules are removed (except root)\n\n**High-Level Operations:**\n\n```go\n// Resource instance operations\nSetResourceInstanceCurrent(addr, obj, provider)\nDeposeResourceInstanceObject(addr) DeposedKey\nSetResourceInstanceDeposed(addr, key, obj, provider)\n\n// Output operations\nSetOutputValue(addr, value, sensitive)\nRemoveOutputValue(addr)\n\n// Module operations\nRemoveModule(addr)\n\n// Batch operations for planning\nUpdateResourceInstanceObjects(...)\n```\n\n**Benefits:**\n- Enables concurrent graph node execution without race conditions\n- Provides consistent snapshots for reading\n- Prevents partial state updates during failures\n\n### 3.3 Multiple State Versions During Execution\n\nTerraform maintains multiple state objects simultaneously during a walk:\n\n```\nState: Current state being built\n    \u2191\n    \u2514\u2500\u2500 Updated by Apply/Plan nodes\n\nRefreshState: State after refresh phase\n    \u2191\n    \u2514\u2500\u2500 Populated during refresh walk\n\nPrevRunState: State from previous run\n    \u2191\n    \u2514\u2500\u2500 Loaded at start, used for comparison\n```\n\n**Usage:**\n- `State`: Primary state being modified\n- `RefreshState`: Previous state snapshot (for change detection)\n- `PrevRunState`: Used for comparisons and understanding drift\n\nThis enables:\n- Change detection: Compare current vs previous state\n- Rollback capability: Previous state preserved\n- Consistency checking: Ensure applied changes match plan\n\n### 3.4 State Persistence and Backends\n\nState management is abstracted through the state manager interface in `statemgr` package:\n\n```\nBackend (e.g., S3, Azure Storage)\n    \u2191\nState Manager (implements locking, versioning)\n    \u2191\nLocal State File (JSON format)\n```\n\n**Key Responsibilities:**\n- **Reading**: Load state from backend\n- **Locking**: Prevent concurrent operations\n- **Versioning**: Track state history\n- **Persistence**: Write updated state back to backend\n\nThe Context holds a reference to the state manager for:\n- Initial state loading\n- Atomic write-back after successful apply\n- Lock management during operations\n\n---\n\n## 4. Execution Flow and Hook System\n\n### 4.1 Overall Plan Execution Flow\n\n```\nContext.Plan(config, opts)\n\u2502\n\u251c\u2500 1. Build Plan Graph\n\u2502   \u251c\u2500\u2500 PlanGraphBuilder.Build()\n\u2502   \u251c\u2500\u2500 Apply transformers in sequence\n\u2502   \u2514\u2500\u2500 Return executable Graph\n\u2502\n\u251c\u2500 2. Apply Walk\n\u2502   \u251c\u2500\u2500 Walk graph with ApplyWalker\n\u2502   \u251c\u2500\u2500 For each resource change node:\n\u2502   \u2502   \u251c\u2500\u2500 Call provider resource operation (Create/Update/Delete)\n\u2502   \u2502   \u251c\u2500\u2500 Update state with result\n\u2502   \u2502   \u2514\u2500\u2500 Call hooks (PreApply, PostApply)\n\u2502   \u251c\u2500\u2500 Handle deposed objects cleanup\n\u2502   \u2514\u2500\u2500 Update State atomically\n\u2502\n\u2514\u2500 3. Return Updated State\n    \u2514\u2500\u2500 State persisted to backend\n```\n\n### 4.3 Graph Traversal (Walker)\n\nThe DAG walker executes nodes respecting dependencies:\n\n```\nWalker Algorithm:\n\u2502\n\u251c\u2500 For each vertex (node) in graph:\n\u2502   \u251c\u2500\u2500 Wait for all dependencies to complete\n\u2502   \u251c\u2500\u2500 If dependency failed: Skip or error (per graph semantics)\n\u2502   \u251c\u2500\u2500 Execute node callback (concurrent if possible)\n\u2502   \u251c\u2500\u2500 Collect diagnostics\n\u2502   \u2514\u2500\u2500 Mark vertex complete\n\u2502\n\u2514\u2500 Return aggregated diagnostics from all vertices\n```\n\n**Key Walker Characteristics:**\n- **Parallel Execution**: Independent nodes execute concurrently\n- **Dependency Respect**: Nodes only execute after upstream dependencies complete\n- **Atomic Failures**: Failed nodes block downstream dependents\n- **Cancelable**: Graph walk can be stopped via context cancellation\n\n### 4.4 Execution in ContextGraphWalker\n\nThe `ContextGraphWalker` implements the Terraform-specific walker pattern:\n\n```go\ntype ContextGraphWalker struct {\n    Context Provider configuration inheritance\n\n### 4.5 Hook System\n\nHooks are callback points allowing external code to observe and influence execution.\n\n#### Hook Interface\n\n```go\ntype Hook interface # Terraform Plan/Apply Pipeline Architecture

## Overview

Terraform's plan and apply operations are executed through a sophisticated graph-based execution engine. The architecture decomposes these operations into two major phases:

1. **Graph Construction Phase**: Builds a directed acyclic graph (DAG) representing resources, their configurations, and their dependencies
2. **Graph Execution Phase**: Traverses the graph respecting dependencies, executing operations in parallel where possible

This document details the internal architecture of these components and their interactions.

---

## 1. Graph Builder System

### 1.1 Core Graph Structure

The `Graph` type represents the execution plan as a directed acyclic graph with the following structure:

```
Graph
├── dag.AcyclicGraph (embedded)
└── Path: addrs.ModuleInstance  // Module scope for this graph
```

**Key Responsibilities:**
- Maintains graph vertices (nodes) and edges (dependencies)
- Validates that the graph is acyclic and has a single root
- Provides traversal capabilities via the embedded `AcyclicGraph`
- Supports concurrent graph walks through the `Walk()` method

### 1.2 GraphBuilder Interface

The `GraphBuilder` interface defines how graphs are constructed:

```go
type GraphBuilder interface {
    Build(addrs.ModuleInstance) (*Graph, tfdiags.Diagnostics)
}
```

The standard implementation is `BasicGraphBuilder`, which:
- Maintains an ordered list of `GraphTransformer` steps
- Executes each transformer in sequence, modifying the graph incrementally
- Validates the final graph for cycles and connectivity
- Logs graph state changes after each transformation step
