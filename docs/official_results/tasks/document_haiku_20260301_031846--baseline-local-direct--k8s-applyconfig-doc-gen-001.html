Conflict Handling and Resolution Strategies\n\n### Conflict Anatomy\n\nA conflict occurs when:\n1. The current apply request (manager A) tries to modify field F\n2. Field F is currently owned by a different manager (manager B)\n3. Manager A does not have `force=true`\n\nExample:\n```yaml\n# Current object (after kubectl and operator both applied)\nmetadata:\n  managedFields:\n  - manager: kubectl\n    fields:\n      f:spec:\n        f:replicas: {}\n  - manager: operator\n    fields:\n      f:spec:\n        f:image: {}\n\n# User runs: kubectl apply --server-side\n# New patch sets: spec.replicas=5 (kubectl manager)\n# But operator manager owns spec.replicas \u2192 CONFLICT\n```\n\n### Conflict Detection\n\n**File: `/workspace/staging/src/k8s.io/apimachinery/pkg/util/managedfields/internal/conflict.go`**\n\n```go\nfunc NewConflictError(conflicts merge.Conflicts) *errors.StatusError {\n    causes := []metav1.StatusCause{}\n    for _, conflict := range conflicts {\n        causes = append(causes, metav1.StatusCause{\n            Type:    metav1.CauseTypeFieldManagerConflict,\n            Message: fmt.Sprintf(\"conflict with %v\", printManager(conflict.Manager)),\n            Field:   conflict.Path.String(),\n        })\n    }\n    return errors.NewApplyConflict(causes, getConflictMessage(conflicts))\n}\n```\n\nReturns HTTP 409 with details of conflicting fields and which manager owns each.\n\n### Resolution Strategies\n\n#### Strategy 1: Fail (Default)\n\n```\nRequest: apply with fieldManager=\"kubectl\" (no force)\nResponse: HTTP 409 Conflict\nError: \"Apply failed with 1 conflict: conflict with manager 'operator': spec.replicas\"\n```\n\nUser must either:\n- Manually remove the conflicting manager from object (rare)\n- Use `--force-conflicts` / `force=true` on the new apply request\n\n#### Strategy 2: Force Ownership Transfer\n\n```bash\nkubectl apply --server-side --force-conflicts\n```\n\nSets `force=true` in PatchOptions:\n\n```go\n// Overrides conflict: new manager takes ownership\nMerge(..., force=true) \u2192\n    Removes old manager from managedFields for conflicting fields\n    Adds new manager to managedFields for those fields\n    Result: HTTP 200, object updated\n```\n\n**Risks of force:**\n- Silently removes another tool's field tracking\n- Can cause thrashing if two tools continuously force-apply conflicting changes\n- Should be used only when you're certain you're the intended owner\n\n#### Strategy 3: Three-Way Merge Prevention\n\nBest practice: **Design ownership boundaries.**\n\n- **kubectl** manages user-specified fields (image, replicas, labels)\n- **operator** manages system fields (status, internal configuration)\n\nUse different field sets to avoid conflicts:\n```yaml\n# kubectl manages these:\nmetadata:\n  labels:\n    app: myapp\nspec:\n  replicas: 3\n  template:\n    spec:\n      containers:\n      - name: app\n        image: myapp:1.0\n\n# operator manages these (via separate apply):\nstatus:\n  conditions: [...]\nspec:\n  internalConfig: {...}\n```\n\nBoth can apply without conflict because they never both claim the same field.\n\n#### Strategy 4: Re-Apply After Conflict\n\nSome workflows retry with `force=true`:\n\n```go\nfor attempt := 0; attempt < maxRetries; attempt++ {\n    resp, err := applyWithForce(false) // force=false\n    if err != nil && isConflict(err) && attempt < maxRetries-1 {\n        applyWithForce(true) // Try with force\n    }\n}\n```\n\n### Conflict Prevention in Practice\n\n#### Test Case: `TestUpdateApplyConflict`\n\n**File: Failure Modes, Limits, and Design Tradeoffs\n\n### Failure Mode 1: Corrupted managedFields\n\n**Cause:** Buggy mutating admission webhook modifies managedFields incorrectly\n\n**Detection:**\n```go\nif err := managedfields.ValidateManagedFields(encodedManagedFields); err != nil {\n    // Admission corrupted managedFields\n}\n```\n\n**Recovery:**\n```go\n// NewManagedFieldsValidatingAdmissionController\nif err := managedfields.ValidateManagedFields(managedFieldsAfterAdmission); err != nil {\n    objectMeta.SetManagedFields(managedFieldsBeforeAdmission)  // Reset\n    warning.AddWarning(ctx, \"\", \"managedFields was in an invalid state after admission\")\n}\n```\n\n**User Impact:** Fields marked corrupted are reset; ## Executive Summary

Kubernetes Server-Side Apply (SSA) is a control plane subsystem that implements declarative resource management through field-level ownership tracking and intelligent merge semantics. Unlike client-side apply (which computes diffs locally), SSA shifts conflict resolution and ownership management to the API server, enabling robust multi-manager scenarios and reducing client-side complexity.

This guide explains how the subsystem is architected, how requests flow through the system, where conflicts are detected and resolved, and the operational tradeoffs involved.

### Purpose

Server-Side Apply solves three core problems:

1. **Ownership Tracking**: Who last modified which field? Without this, concurrent edits by different tools (kubectl, Helm, operators) cause data loss or unwanted overwrites.

2. **Conflict Detection and Resolution**: When one manager tries to modify a field controlled by another, SSA detects the conflict and can either (a) fail with a clear error, (b) allow forced takeover, or (c) use structured merge semantics to prevent unintended overwrites.

3. **Declarative State Management**: Clients declare *what they want* (via apply configurations) rather than *how to change it* (via imperative patches). The server reconciles the desired state with the live state and field ownership information.
