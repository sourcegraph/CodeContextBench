## Key Findings

**Root Cause:** The V38 migration has an asymmetry in how it handles table panels without explicit `fieldConfig.defaults.custom`:

1. **The Migration** (v38.go:115-127): Only processes `defaults.custom` if it already exists - it does NOT create it
2. **The Tracking** (frontend_defaults.go:1138-1169): Only sets `_originallyHadFieldConfigCustom = true` if custom exists in the original
3. **The Cleanup** (frontend_defaults.go:606-620): Only preserves the custom structure during cleanup if the tracking flag is set

**The Failure Scenario:**
- A table panel without explicit `defaults.custom` has field overrides with `custom.displayMode`
- V38 migration converts those overrides to `custom.cellOptions` correctly
- BUT `defaults.custom` is NOT created (intentional design)
- The tracking flag is NOT set (because original didn't have custom)
- During cleanup, the preservation logic is skipped
- This leaves a structural inconsistency: overrides reference `custom.*` but defaults lacks the `custom` container

**Why dashboards WITH explicit `defaults.custom` work:** The tracking flag IS set when custom exists, triggering the cleanup preservation logic that recreates the structure.

The report includes exact file references (v38.go, frontend_defaults.go, migrate.go) with line numbers, a test case that documents the exact scenario, and specific recommendations for fixing the asymmetry to ensure structural consistency between field defaults and overrides.
