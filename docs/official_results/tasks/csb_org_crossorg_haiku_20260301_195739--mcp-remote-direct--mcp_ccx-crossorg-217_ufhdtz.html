- `django/core/handlers/asgi.py` - ASGI handler implementation
- `django/utils/deprecation.py` - MiddlewareMixin providing the middleware pattern
- `django/core/handlers/exception.py` - Exception handling wrapper

**Critical Implementation Detail:**
The key to understanding Django's middleware is in `BaseHandler.load_middleware()` at **line 41**: the middleware stack is constructed by iterating through `settings.MIDDLEWARE` in **REVERSE order**. This creates a nested callable structure that naturally implements the "onion" pattern:

- If `MIDDLEWARE = [A, B, C]`, the request flows: **A → B → C → view → C → B → A**

**Middleware Chain Structure:**
1. Each middleware wraps the next one via the `get_response` callable
2. When a middleware calls `get_response()`, it passes control to the next layer
3. The response returns through each layer in reverse order
4. This is achieved through callable nesting, not explicit phase methods

**Comparison with WSGI:**
Unlike traditional WSGI middleware that always calls process_response for every layer, Django's middleware implements true onion semantics: if a middleware short-circuits (returns without calling get_response), inner layers never see the request, and the response only returns through layers that saw the request.
