## Investigation Complete

I've analyzed the Grafana codebase and identified the root cause of the regression. The investigation report has been written to `/logs/agent/investigation.md`.

### Key Findings:

**Root Cause**: The v38 migration correctly converts table panel field overrides from `custom.displayMode` to `custom.cellOptions`. However, when the panel is saved via `PanelModel.getSaveModel()` (line 490 in PanelModel.ts), it filters out **all custom property overrides** using the `isStandardFieldProp` filter. This causes the field override configuration to be silently dropped.

**Why it specifically affects dashboards without explicit `defaults.custom`**:
- Dashboards with explicit `fieldConfig.defaults.custom` objects receive the migrated `cellOptions` in their defaults, providing some configuration
- Dashboards without explicit `defaults.custom` have their overrides filtered out during save, leaving no configuration at all

**The Evidence Chain**:
1. **Backend v38 migration** (`apps/dashboard/pkg/migration/schemaversion/v38.go`): Correctly migrates overrides
2. **Custom override filtering** (`public/app/features/dashboard/state/PanelModel.ts:490`): Filters out all properties starting with `custom.`
3. **Property detection** (`packages/grafana-data/src/panel/getPanelOptionsWithDefaults.ts`): The `isStandardFieldProp()` function treats all custom properties as "non-standard" and removes them during save

The issue highlights a critical mismatch between the migration logic (which preserves custom overrides) and the save logic (which discards them).

# Investigation Report: Dashboard Migration v38 Table Panel Regression

## Summary

The v38 dashboard migration correctly converts table panel field overrides from `custom.displayMode` to `custom.cellOptions`. However, when the panel is persisted, all custom field overrides are filtered out by `PanelModel.getSaveModel()`, causing the field override configuration to be silently dropped during dashboard import/export cycles. The issue manifests specifically for dashboards where `fieldConfig.defaults.custom` was not explicitly set in the original v10.3 dashboard JSON.

## Root Cause
