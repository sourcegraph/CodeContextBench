# Prometheus TSDB Compaction Operational Runbook

## 1. Overview

### Purpose of Compaction

TSDB (Time Series Database) compaction in Prometheus serves three critical functions:

1. **Block Consolidation**: Merges overlapping time blocks to reduce the number of files that need to be queried
2. **Tombstone Cleanup**: Physically removes series marked for deletion by applying tombstone metadata, reducing disk footprint
3. **Query Performance**: Maintains optimal query performance by keeping block sizes and levels balanced according to exponential growth

Without compaction, Prometheus would accumulate small blocks indefinitely, degrading query performance and consuming excessive disk space. Compaction is triggered automatically by Prometheus and typically requires no manual intervention.

### Block Structure and Compaction Levels

**Block Basics**:
- Each block represents a time-ordered chunk of time series data
- Blocks have half-open time intervals: `[MinTime, MaxTime)` where MaxTime is exclusive
- Block structure includes: chunks (samples), index (series metadata), tombstones (deletion markers), meta.json (metadata)
- Block identifiers are ULIDs (Universally Unique Lexicographically Sortable Identifiers)

**Compaction Levels**:
Prometheus uses exponential block ranges to organize blocks hierarchically. Given default configuration:
- **Level 0** (Head blocks): ~2 hours of data from active writes
- **Level 1**: Result of compacting Level 0 blocks (~6 hours = 3 × 2h)
- **Level 2**: Result of compacting Level 1 blocks (~18 hours = 3 × 6h)
- **Level 3+**: Continue exponential growth with factor of 3

This is configured via `DefaultBlockDuration` (2 hours) and exponential step factor (3) in `tsdb/db.go:745-749`.
