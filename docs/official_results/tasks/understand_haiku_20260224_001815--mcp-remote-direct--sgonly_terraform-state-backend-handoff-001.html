Failure Modes\n\n### State Locking Failures\n\n#### Stale Locks\n- **Problem**: Lock exists but is held by a crashed process\n- **Detection**: `LockWithContext()` retries with backoff (1s\u219216s)\n- **Resolution**: `-force-unlock <lock-id>` forces lock release\n- **Mitigation**: Lock timeout mechanisms (if supported by backend)\n\n**Example**: S3 backend doesn't have automatic timeout; DynamoDB entry persists until manually removed.\n\n#### Lock Contention\n- **Problem**: Multiple processes waiting for same lock\n- **Behavior**: `LockWithContext()` exponentially backs off (max 16s delay)\n- **Timeout**: Context timeout determines how long to retry\n- **User Impact**: User sees \"Error acquiring lock\" after timeout\n\n#### Lock ID Mismatch\n- **Problem**: Unlock called with wrong lock ID\n- **Detection**: Backend verifies ID matches before unlocking\n- **Resolution**: Correct ID required (visible in lock error output)\n\n### State Storage Failures\n\n#### Storage Unavailable\n- **Local**: File permission errors, disk full\n  - Error propagates immediately\n  - Transactional write ensures consistency\n- **Remote**: Network timeout, service outage\n  - Backend-specific retry logic\n  - Some backends have consistency checks (e.g., S3 checksum)\n\n#### Concurrent Modifications\n- **Problem**: Two processes write state simultaneously\n- **Prevention**: Locker interface prevents (via locking)\n- **Without Locking**: Backends use serial numbers in state\n  - Higher serial = newer state\n  - Prevents overwriting newer states with old ones\n  - Not reliable without locking!\n\n#### Corruption Detection\n- **Checksums**: S3 backend verifies MD5\n  - Retries with timeout (10s by default)\n  - Returns `ErrBadChecksum` if mismatch persists\n- **Serial Number Mismatch**: State manager detects serial jumps\n  - Indicates concurrent write by another process\n  - User warned about potential corruption\n\n### Configuration Errors\n\n#### Missing Required Config\n- **Detection**: `ConfigSchema()` marks fields as Required\n- **Validation**: `PrepareConfig()` checks for nil/empty values\n- **Error**: Diagnostic error blocks `Configure()`\n\n#### Invalid Config Values\n- **Detection**: `PrepareConfig()` validates types and ranges\n- **Example S3**: Bucket name validation, region validation\n- **Example Local**: Path validation (not empty)\n\n#### Missing Workspace\n- **Error**: `ErrDefaultWorkspaceNotSupported` (some backends)\n- **Meaning**: Backend requires named workspace, doesn't support \"default\"\n- **User Solution**: Create workspace with `terraform workspace new`\n\n#### Workspace Not Supported\n- **Error**: `ErrWorkspacesNotSupported` (simple backends)\n- **Meaning**: Backend doesn't support multiple workspaces\n- **Example**: Some HTTP backends only support single state\n\n### State Lock Error Handling\n\nThe system uses `statemgr.LockError` for specific handling:\n\n```go\ntype LockError struct {\n    Info *LockInfo  // Lock held by other process\n    Err  error      // Underlying error\n}\n```\n\n**Detection**:\n```go\nlockID, err := locker.Lock(info)\nif lockErr, ok := err.(*statemgr.LockError); ok {\n    // Another process holds the lock\n    // lockErr.Info contains lock holder details\n    // Can force unlock with lockErr.Info.ID\n}\n```\n\n**Force Unlock** (command: `terraform force-unlock <lock-id>`):\n```go\n// Extract lock ID from error output\n// Call: locker.Unlock(lockID) with the ID from LockInfo\n// Requires user confirmation in CLI\n```\n\n## 5. # Terraform State Backend Subsystem Handoff Document

## 1. Purpose

The state backend subsystem provides an abstraction layer that allows Terraform to manage state in multiple ways while maintaining a consistent interface. It solves the following problems:

- **Local vs. Remote Operations**: The same CLI can operate both locally (storing state on disk) and remotely (e.g., via HCP Terraform, Terraform Cloud, or cloud storage services)
- **Multiple Storage Backends**: Supports various storage systems (S3, Azure Storage, Google Cloud Storage, Consul, Kubernetes, etc.) without CLI changes
- **Team Collaboration**: Enables state sharing and locking for teams using remote backends
- **Workspace Management**: Provides isolation between different infrastructure environments (default workspace plus named workspaces)
- **State Locking**: Prevents concurrent modifications through distributed locking mechanisms

### Why Multiple Backend Types?

Different environments have different requirements:
- **Local**: Simple single-user development, no network dependency
- **S3**: AWS-native teams, mature ecosystem, cost-effective
- **Azure/GCS**: Native cloud provider solutions
- **Remote/Cloud**: Enterprise features, VCS integration, state history, policy
- **Consul/etcd**: Kubernetes-native, distributed systems teams
- **HTTP**: Custom solutions, on-premises backends
