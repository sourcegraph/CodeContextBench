Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### etcd Unavailability\n- **Impact**: All writes fail immediately (no quorum); reads degrade to stale data from Cacher buffer\n- **Mitigation**: Cacher buffers events for ~90 seconds (default `DefaultEventFreshDuration`); clients can tolerate brief etcd outages if they don't overwhelm the cache\n- **Observable**: `/readyz?verbose` health endpoint; etcd health check in `storageBackend`\n\n#### Cache Invalidation (Stale Reads)\n- **Scenario**: Cacher falls behind (e.g., overloaded apiserver, slow etcd watch stream)\n- **Symptom**: Client reads old version of object\n- **Mitigation**: List requests perform a \"consistency read\"\u2014fetch latest revision from etcd, ensure cache is caught up before serving\n- **Tradeoff**: Consistency read adds latency but prevents stale list responses\n\n#### Admission Webhook Timeout\n- **Scenario**: External webhook takes >30s to respond\n- **Symptom**: Request hangs, then returns error\n- **Mitigation**: Webhook timeout configurable per webhook; typically 30s; timeout failures fail-open (allow request) or fail-closed (reject), per policy\n- **Observable**: Admission webhook latency metrics in Prometheus\n\n#### Request Overload\n- **Scenario**: Client sends 10,000 requests/sec; apiserver capacity is ~2,000 req/sec\n- **Symptom**: Requests queued, then 429 Too Many Requests after queue full\n- **Mitigation**: Flow control (Priority and Fairness) classifies requests; critical operations (e.g., node updates) prioritized over client watches\n- **Tradeoff**: Fair queuing adds latency for low-priority requests\n\n#### Handler Chain Panic\n- **Scenario**: Bug in admission controller or handler causes panic\n- **Impact**: Request fails, but server remains up (panic recovered by `WithPanicRecovery`)\n- **Observable**: Error logged; client receives 500 Internal Server Error\n\n### 4.2 Limits and Constraints\n\n#### Request Size\n- **Limit**: `MaxRequestBodyBytes` (default 3MB in kube-apiserver); configured via `--max-request-body-bytes`\n- **Exceeding**: 413 Payload Too Large\n- **Rationale**: Prevent memory exhaustion; limits CRD instance size, ConfigMap size, etc.\n\n#### In-Flight Request Concurrency\n- **Limit**: `MaxRequestsInFlight` (default 400 for non-mutating) + `MaxMutatingRequestsInFlight` (default 200 for mutating)\n- **Exceeding**: Queued in Priority and Fairness; 429 Too Many Requests if queue full\n- **Rationale**: Prevent OOM; ensure server responsiveness\n\n#### Watch Event Buffer\n- **Limit**: ~1000 events per watch stream (configurable in Cacher); event history kept for ~90 seconds\n- **Exceeding**: Watch stream re-lists from storage (expensive but correct)\n- **Rationale**: Balance memory vs. responsiveness; most applications don't fall behind\n\n#### API Group/Version Count\n- **Limit**: No hard limit; Kubernetes core has ~50 APIs\n- **Impact**: Larger OpenAPI spec, longer discovery time, more memory\n- **Observable**: Metrics in PrometheusHTTP response size, startup time\n\n#### etcd Key Size\n- **Limit**: ~1.5MB per key (etcd3 limit); Kubernetes enforces ~1MB safety margin per object\n- **Rationale**: Prevent pathological cases (e.g., very large ConfigMap)\n\n#### ResourceVersion Lifetime\n- **Guarantee**: ResourceVersion remains valid for ~5 minutes (configurable; depends on etcd compaction)\n- **Implication**: Long-lived watches must refresh if they lag beyond this window\n- **Observable**: Watch restart events; API server logs \"too old revision\" errors\n\n### 4.3 Design Tradeoffs\n\n#### Consistency vs. Performance\n- **Tradeoff**: Cacher serves most reads from stale in-memory state for speed\n- **Resolution**: Consistency reads (list operations) verify freshness; single gets trust cache\n- **Assumption**: Applications tolerate brief staleness (seconds); strong consistency not required for most operations\n\n#### Caching Complexity vs. Simplicity\n- **Tradeoff**: Watch cache adds ~2k LOC but enables 10x throughput improvement; without it, etcd would saturate at ~100s req/sec\n- **Resolution**: Worth the complexity cost; nearly all deployments enable caching\n- **Failure mode**: Bug in caching logic \u2192 stale reads; mitigated by extensive tests\n\n#### Admission Control Latency\n- **Tradeoff**: Webhooks add latency (~50\u2013500ms per request if misconfigured) but enable rich policy\n- **Resolution**: Operator must size webhooks; timeout failures fail-open by default (allow, not block)\n- **Assumption**: Policies are worth the latency cost; if not, can be disabled\n\n#### Write Concurrency and Conflict Resolution\n- **Tradeoff**: Optimistic concurrency (resourceVersion-based) instead of locking\n- **Resolution**: Client retries on conflict (409); scales better than pessimistic locking\n- **Assumption**: Most writes don't conflict; if they do, client should re-read and retry (built into kubectl)\n\n#### Graceful Shutdown Complexity\n- **Tradeoff**: Long-running requests (watch, exec) can outlive shutdown signals; must wait for them to drain\n- **Resolution**: ShutdownWatchTerminationGracePeriod limits wait time; after timeout, forcibly close connections\n- **Observable**: /readyz returns false during shutdown; new requests rejected with 429 Retry-After\n\n---\n\n## 5. Operational Implications: Performance, Reliability, and Maintainability\n\n### 6.1 Performance\n\n#### Cache Effectiveness\n- **Metric**: Cache hit rate for list/watch operations\n- **Observation**: With effective caching, etcd sees ~10\u201320% of actual requests (rest served from cache)\n- **Impact**: ~100x throughput improvement vs. uncached backend\n- **Tuning**: Adjust `CacheSizes` per resource; smaller cache = less memory but more etcd load\n\n#### Watch Event Latency\n- **Typical**: ~10\u201350ms from update to watch event arrival\n- **Bottleneck**: Network latency to etcd, Cacher dispatch, client network\n- **Optimization**: Colocate etcd and apiserver; use fast network; increase buffer size (memory tradeoff)\n\n#### Request Latency Distribution\n- **p50**: 5\u201310ms (cache hit)\n- **p95**: 50\u2013100ms (etcd backend or webhook)\n- **p99**: 100\u2013500ms (slow webhook, etcd under load)\n- **Observable**: apiserver_request_duration_seconds_bucket in Prometheus\n\n#### Memory Footprint\n- **Typical**: 1\u20132GB baseline + ~1KB per object in cache (rough estimate)\n- **Scaling**: 10,000 objects in cache \u2248 10MB overhead (modest); 1M objects \u2248 1GB (concerning)\n- **Mitigation**: Paginate list requests; limit watch connections per client\n\n### 6.2 Reliability\n\n#### Health Probes\n- **Livez** (`/livez`): Process alive and not panicking; grace period for post-start hooks\n- **Readyz** (`/readyz`): Server ready to serve requests; all initialization hooks completed, storage accessible\n- **Healthz** (`/healthz`): Legacy, rarely used\n- **Behavior during shutdown**: `/readyz` returns false immediately when shutdown signal received; `/livez` remains true for `ShutdownDelayDuration` (allows load balancer time to drain traffic)\n\n#### Graceful Shutdown\n- **Signal**: SIGTERM or `POST /shutdown` endpoint (disabled by default for security)\n- **Grace period**: `ShutdownTimeout` (default 30s)\n- **Steps**:\n  1. Stop accepting new requests (return 429 Retry-After)\n  2. Drain in-flight requests (wait up to grace period)\n  3. Drain watch connections (if ShutdownWatchTerminationGracePeriod set)\n  4. Close listeners\n  5. Call PreShutdownHooks (e.g., write logs, notify peers)\n  6. Exit\n- **Observable**: Logs indicate shutdown progress; `kubernetes_build_info` or `etcd_server_has_leader` can detect unavailability\n\n#### Watch Resilience\n- **Client reconnection**: If watch stream breaks, client should re-list and resume with new ResourceVersion\n- **Server backpressure**: If client slow to consume events, server can skip events (BOOKMARK events help client detect staleness)\n- **Fallback**: If cache buffer overflows, Cacher automatically re-lists from etcd (transparent to client, but latency spike)\n\n#### Conflict Resolution\n- **Optimistic concurrency**: Client updates with resourceVersion; if stale, 409 Conflict\n- **Expected behavior**: Client catches error, re-reads object, retries with new resourceVersion\n- **Observable**: apiserver_client_request_conflict_total metric (should be low; indicates application logic issues if high)\n\n### 6.3 Maintainability\n\n#### Code Organization\n- **Clear separation**: Authentication, authorization, admission are orthogonal; storage is pluggable\n- **Interface boundaries**: `rest.Storage` and friends are stable contracts; implementation can change without affecting callers\n- **Testing**: Extensive test suites in parallel; e.g., `/workspace/staging/src/k8s.io/apiserver/pkg/endpoints/apiserver_test.go` (170k LOC) tests integration\n\n#### Configuration Management\n- **Options pattern**: Command-line flags \u2192 `RecommendedOptions` struct \u2192 `Config` \u2192 `GenericAPIServer`\n- **Versionability**: Config fields rarely removed; deprecation warnings added before removal\n- **Documentation**: Each option documented in code comments and CLI help text\n\n#### Observability\n- **Logging**: Structured logging via klog; request-scoped logs include request ID for correlation\n- **Metrics**: Prometheus format; key metrics: request latency, cache stats, admission latency, etcd performance\n- **Tracing**: Distributed tracing via OpenTelemetry; can trace request flow through filters (if enabled)\n- **Example**: Slow request detection: logs requests >1s with full context\n\n#### Backwards Compatibility\n- **API versioning**: Multiple versions supported simultaneously (v1, v1alpha1, etc.); new fields have defaults\n- **Storage migration**: APIResourceVersion tracks what version is stored in etcd; migration controller handles upgrades\n- **Deprecation policy**: Stable APIs removed only after 12+ months of deprecation warnings\n\n#### Extension Point Stability\n- **Stable**: REST storage interfaces, admission interface, authenticator/authorizer interfaces\n- **Unstable**: Internal metrics, handler chain structure (code-level; changes require recompilation)\n- **Risk mitigation**: Prefer interfaces to concrete types; document contracts clearly\n\n---\n\n## 7. # Kubernetes API Server Library: Architecture and Extension Guide

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

The `k8s.io/apiserver` library provides a generic, reusable foundation for building RESTful HTTP API servers compatible with Kubernetes clients. Rather than a single monolithic API server, it enables composition of multiple independent servers (core Kubernetes, extensions, and aggregated services) into a unified HTTP server chain. The library handles cross-cutting concerns—authentication, authorization, admission control, persistence, discovery, and lifecycle management—allowing API authors to focus on domain logic.

### Boundaries

The library operates at the HTTP layer and above. It does not:
- Manage networking infrastructure (listeners, TLS certs, certificate rotation)
- Persist data directly—instead, it abstracts storage through pluggable interfaces
- Implement specific Kubernetes APIs (core/v1, apps/v1) or CRDs
- Control cluster topology, node management, or workload scheduling

It does manage:
- HTTP request/response handling and serialization (Go JSON, Protobuf, etc.)
- Authentication/authorization decision-making and audit logging
- Admission policy enforcement (mutation and validation)
- API registration, discovery, and versioning schemes
- Graceful shutdown and lifecycle signals
- Watch and list performance through caching
