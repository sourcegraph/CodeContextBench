## Key Findings:

**Architecture Overview:** Flink implements a **barrier-based distributed two-phase commit protocol** that elegantly decouples checkpoint synchronization from centralized coordination:

1. **Trigger & Barrier Injection:** CheckpointCoordinator sends an RPC to all tasks, which inject CheckpointBarrier events into the data stream. These barriers flow naturally through the topology like data, creating a logical snapshot line.

2. **Barrier Propagation & Alignment:** 
   - **SingleCheckpointBarrierHandler (Aligned):** Blocks input channels until all sources deliver their barrier, ensuring minimal in-flight data is captured
   - **CheckpointBarrierTracker (Unaligned):** Lets data flow freely while tracking barriers, trading state size for lower latency

3. **State Snapshot:** Upon receiving a barrier, tasks asynchronously snapshot operator state, key-value state, and channel state. This happens without centralized blocking, allowing computation to continue.

4. **Acknowledgment & Completion:** Tasks send TaskStateSnapshot back to the coordinator. Once all tasks acknowledge, the coordinator:
   - Finalizes the checkpoint (creates CompletedCheckpoint)
   - Persists it to the checkpoint store
   - Sends second-phase commit notifications to all tasks
   - Triggers post-commit hooks (e.g., advancing Kafka offsets)

**Key Design Patterns:**
- Two-phase commit (trigger + prepare, then commit)
- Asynchronous snapshot collection with Future composition
- Incremental snapshots with shared state deduplication
- RPC-based distributed messaging with decay-and-discard for expired checkpoints
