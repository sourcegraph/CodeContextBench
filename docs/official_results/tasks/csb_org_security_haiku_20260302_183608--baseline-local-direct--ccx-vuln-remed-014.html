## Summary

I've identified HTTP handlers registered without authentication middleware in both the **Grafana Loki** and **Grafana Mimir** repositories:

### **Key Findings:**

**Mimir** (`grafana/mimir:pkg/api/api.go`):
- The API module explicitly registers endpoints with `auth=false` parameter, bypassing `middleware.AuthenticateUser`
- This includes ~30 endpoints for:
  - Configuration/status: `/config`, `/api/v1/status/*`, `/runtime_config`
  - Debugging: `/debug/fgprof`, `/static/`
  - Operational endpoints: `/distributor/ring`, `/ingester/flush`, `/ingester/prepare-shutdown`, `/ruler/ring`, `/store-gateway/ring`, `/compactor/ring`, etc.
  - Alertmanager: `/multitenant_alertmanager/status`, `/multitenant_alertmanager/configs`, etc.

**Loki** (`grafana/loki`):
- `pkg/loki/modules.go`: Multiple ring status pages and operational endpoints registered without authentication
- `pkg/ingester/ingester.go`: Handlers like `FlushHandler`, `ShutdownHandler`, `PrepareShutdownHandler` for administrative operations
- `pkg/querier/http.go`: Query handlers (`RangeQueryHandler`, `LabelHandler`, `SeriesHandler`, etc.)

### **Design Pattern:**
Both systems deliberately skip handler-level authentication, relying instead on reverse proxy or outer middleware layer for security. The code explicitly shows `auth=false` parameters (Mimir) and lacks middleware wrapping (Loki), indicating this is architectural rather than accidental.

The complete analysis with file paths, function names, and detailed explanations is available in `/workspace/answer.json`.
