## Summary of Findings

**Kubernetes Implementation (Lease-Based):**
- Core files: `leaderelection.go`, `leaselock.go`, and coordinated election controllers
- Uses distributed Lease objects as locks with explicit timing (15s lease duration, 10s renewal deadline)
- Includes coordinated election mode that prefers older/stable nodes
- Failover occurs when lease expires; non-leaders detect expiry and compete to acquire
- Graceful transitions via `PreferredHolder` field and `LeaseCandidate` objects

**etcd Implementation (Raft-Based):**
- Core files: `raft.go`, `server.go`, and transport layer
- Uses Raft consensus algorithm for automatic leader election via quorum
- Faster failover via election timeouts (typically 150-300ms vs Kubernetes's 15s)
- Graceful leadership transfer via `MoveLeader()` using `longestConnected()` strategy
- Client-side election API in `election.go` for application-level leader elections

**Key Differences:**
1. Kubernetes relies on explicit distributed leases; etcd on Raft consensus
2. Kubernetes failover takes ~15s; etcd typically ~150-300ms
3. Kubernetes prefers old nodes for stability; etcd prefers longest-connected
4. Kubernetes is control-plane focused; etcd exposes election as a client API
