## Context

The comment likes feature spans three backend files:

1. **`ghost/core/core/server/services/comments/comments-service.js`** — Service layer: `getCommentLikes()` method that queries the database for likes on a given comment.
2. **`ghost/core/core/server/services/comments/comments-controller.js`** — Controller layer: `getCommentLikes()` method that extracts the comment ID from the request frame and delegates to the service.

## Review Summary

### **1. Critical Bug: Incorrect Frame Property Access** 
**File**: `comments-controller.js` (line 370)
- **Issue**: Uses `frame.data.id` instead of `frame.options.id`
- **Impact**: The `id` parameter is defined in the endpoint's `options` array (query parameters), not in the request body. This causes `commentId` to be undefined, making the API call fail.
- **Fix**: Change to `frame.options.id` (consistent with similar method `getCommentReporters`)

### **2. High Severity Bug: Missing Comment Verification**
**File**: `comments-service.js` (line 261)
- **Issue**: Fetches a comment but doesn't verify it exists (no `require: true`)
- **Impact**: If the comment doesn't exist, the method silently proceeds and returns empty results instead of throwing a proper error. This violates the error handling pattern used elsewhere (e.g., `getCommentByID`).
- **Fix**: Add `{require: true}` to the `findOne` call to validate the comment exists

### **3. Medium Severity: Compliance Violation**
**File**: `comment-likes.js` (line 8)
- **Issue**: Missing `headers` configuration in the browse endpoint
- **Impact**: Violates Ghost's API conventions. All endpoints should explicitly define `headers: { cacheInvalidate: false }` for read-only operations (as seen in `comments.js`).
- **Fix**: Add the `headers` configuration to the browse action definition
