## Executive Summary

Kubernetes Server-Side Apply (SSA) is a declarative resource management system that enables multi-manager coordination of object fields without conflicts. Unlike client-side apply, SSA tracks field ownership through `managedFields` metadata, allowing multiple tools (kubectl, Helm, operators, etc.) to safely manage different parts of the same resource.

The **applyconfigurations** subsystem provides type-safe client-side building blocks for constructing resources compatible with SSA, while the **Server-Side Apply** subsystem handles the merge logic, conflict detection, and field ownership tracking on the API server.

## Part 1: Subsystem Purpose, Boundaries, and Dependencies

### 1.1 What is Server-Side Apply?

Server-Side Apply is a declarative, field-ownership-based update mechanism introduced in Kubernetes v1.16 (alpha) and promoted to stable in v1.21. It addresses the fundamental problem of multi-manager coordination: **when multiple tools manage the same resource, how do you prevent accidental overwrites?**

**Key Principles:**
- **Declarative Intent:** Clients declare desired state, not imperative changes
- **Field Ownership:** The server tracks which manager owns which fields
- **Conflict Detection:** Server rejects conflicting updates from different managers trying to control the same field
- **Multi-Manager Safety:** Multiple independent tools can safely manage non-overlapping fields in a single resource

**Example:** A controller might manage `spec.replicas` while a user interactively edits `spec.template.metadata.labels`. Both operations can coexist because the server tracks that the controller owns `spec.replicas` and the user owns `spec.template.metadata.labels`.
