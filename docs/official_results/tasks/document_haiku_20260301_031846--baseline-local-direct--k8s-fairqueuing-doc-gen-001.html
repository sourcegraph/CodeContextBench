Instead, **shuffle sharding** is used: a probabilistic technique that reduces head-of-line blocking.\n\n```go\nfunc (qs *queueSet) shuffleShardLocked(hashValue uint64, ...) int {\n    var backHand [8]int\n    hand := qs.dealer.DealIntoHand(hashValue, backHand[:])\n    handSize := len(hand)\n    offset := qs.enqueues % handSize\n    qs.enqueues++\n    bestQueueIdx := -1\n    minQueueSeatSeconds := fqrequest.MaxSeatSeconds\n    for i := 0; i < handSize; i++ {\n        queueIdx := hand[(offset+i)%handSize]\n        queue := qs.queues[queueIdx]\n        queueSum := queue.requestsWaiting.QueueSum()\n        thisQueueSeatSeconds := queueSum.TotalWorkSum\n        if thisQueueSeatSeconds < minQueueSeatSeconds {\n            minQueueSeatSeconds = thisQueueSeatSeconds\n            bestQueueIdx = queueIdx\n        }\n    }\n    return bestQueueIdx\n}\n```\n\n**Algorithm**:\n1. Dealer \"deals a hand\" of `HandSize` queue indices (typically 4-8) based on the flow's hash value.\n2. Iterate through the hand in a rotating offset to avoid biasing early indices.\n3. Select the queue with the **minimum total work** (in seat-seconds) waiting.\n\n**Why shuffle sharding?**\n- Without it, a few flows with the same hash can starve each other (head-of-line blocking).\n- With it, conflicts are probabilistically reduced, and if they occur, the least-loaded queue in the hand is chosen.\n- This provides **probabilistic fairness** with low computational overhead.\n\n#### Queue Assignment at Arrival (shuffleShardAndRejectOrEnqueueLocked, lines 552-579)\n\nOnce a queue is selected by shuffle sharding, the request is either enqueued or rejected based on capacity:\n\n```go\nfunc (qs *queueSet) rejectOrEnqueueToBoundLocked(request *request) bool {\n    queue := request.queue\n    curQueueLength := queue.requestsWaiting.Length()\n    if qs.totSeatsInUse >= qs.dCfg.ConcurrencyLimit &&\n        curQueueLength >= qs.qCfg.QueueLengthLimit {\n        return false  // Reject: at concurrency limit AND queue is full\n    }\n    qs.enqueueToBoundLocked(request)\n    return true\n}\n```\n\n**Rejection condition**: A request is rejected if and only if:\n- The queue is already at its length limit, **AND**\n- Total seats in use \u2265 concurrency limit.\n\nThis ensures queuing can still occur even if concurrency is exhausted, as long as the queue is not full.\n\n---\n\n## Part 3: End-to-End Control and Request Flow\n\n### requests to utilize available capacity.\n\n### 3.2 Queued Request Waiting Phase (Request.wait)\n\nAfter `StartRequest()` returns, the APF filter calls `request.Finish(execFn func())`. Finish internally calls `request.wait()` (lines 397-441), which **blocks until a dispatch decision is made**:\n\n```go\nfunc (req *request) wait() (bool, bool) {\n    decisionAny := req.decision.Get()  // BLOCK HERE until decision is set\n    qs.lockAndSyncTime(req.ctx)\n    defer qs.lock.Unlock()\n\n    switch decisionAny {\n    case decisionExecute:\n        return true, false  // Proceed to execution\n    case decisionCancel:\n        return false, ...   // Request was timed out or rejected\n    }\n}\n```\n\nThe **decision promise** is set in two scenarios:\n\n**Scenario 1: Immediate dispatch** (lines 710-713)\nIf a request is dequeued for execution before the client calls Wait(), the decision is pre-set:\n```go\nif !request.decision.Set(decisionExecute) {\n    return true  // Decision was already set\n}\n```\n\n**Scenario 2: Wait deadline exceeded** (lines 414-440)\nThe caller has set a context deadline (derived from request wait limit, see `/workspace/staging/src/k8s.io/apiserver/pkg/server/filters/priority-and-fairness.go:396-426`). When the deadline fires, the context is cancelled. The next call to `wait()` finds `decisionAny != decisionExecute` and removes the request from the queue, incrementing `totRequestsCancelled`.\n\n### 3.3 Dispatch Phase (findDispatchQueueToBoundLocked)\n\nThe **core fairness algorithm** selects which queue to dispatch from. This is where **virtual time fairness** is applied.\n\nLines 776-843 implement the selection:\n\n```go\nfunc (qs *queueSet) findDispatchQueueToBoundLocked() (*queue, *request) {\n    minVirtualFinish := fqrequest.MaxSeatSeconds\n    var minQueue *queue\n    var minIndex int\n    nq := len(qs.queues)\n\n    // Round-robin through queues\n    for range qs.queues {\n        qs.robinIndex = (qs.robinIndex + 1) % nq\n        queue := qs.queues[qs.robinIndex]\n        oldestWaiting, _ := queue.requestsWaiting.Peek()\n        if oldestWaiting != nil {\n            // Compute estimated virtual finish time\n            estimatedWorkInProgress := fqrequest.SeatsTimesDuration(\n                float64(queue.seatsInUse),\n                qs.estimatedServiceDuration)\n            dsMin = min(dsMin, queue.nextDispatchR - estimatedWorkInProgress)\n\n            // The finish R is: next dispatch point + work of this request\n            currentVirtualFinish := queue.nextDispatchR + oldestWaiting.totalWork()\n\n            if currentVirtualFinish < minVirtualFinish {\n                minVirtualFinish = currentVirtualFinish\n                minQueue = queue\n                minIndex = qs.robinIndex\n            }\n        }\n    }\n\n    // Check if selected request can fit\n    if !qs.canAccommodateSeatsLocked(oldestReqFromMinQueue.MaxSeats()) {\n        return nil, nil  // Can't fit, wait for executing requests to complete\n    }\n\n    oldestReqFromMinQueue.removeFromQueueLocked()\n    qs.robinIndex = minIndex  // Update robin for next dispatch\n    return minQueue, oldestReqFromMinQueue\n}\n```\n\n**Fairness semantics**:\n\nFor each queue i, define `finishR_i = nextDispatchR_i + work_i`, where `work_i` is the total work (seat-seconds) of the oldest request in queue i. The queue with **minimum finishR** is selected. This implements classic fair queuing: queues are serviced in order of when they would complete, which ensures no queue gets starved.\n\n**Anti-windup mechanism**:\n\nBefore dispatch, `boundNextDispatchLocked()` adjusts `nextDispatchR` if the queue has been idle to prevent \"windup\" where a queue that was idle accumulates a low virtual finish time and then dominates subsequent dispatch rounds (line 839-841):\n\n```go\nif minQueue.nextDispatchR < oldestReqFromMinQueue.arrivalR {\n    // This shouldn't happen, but log if it does\n    klog.ErrorS(errors.New(\"dispatch before arrival\"), ...)\n}\n```\n\nThe system enforces that `nextDispatchR \u2265 arrivalR` to prevent this anomaly.\n\n### 3.4 Request Execution and Finish Phase\n\nOnce dispatched, the request executes via `execFn()` (line 192 in apf_filter.go). When execution completes:\n\n```go\nfunc (req *request) Finish(execFn func()) bool {\n    exec, idle := req.wait()\n    if !exec {\n        return idle\n    }\n    defer func() {\n        idle = req.qs.finishRequestAndDispatchAsMuchAsPossible(req)\n    }()\n    execFn()  // Actually execute the handler\n    return idle\n}\n```\n\nThe deferred call to `finishRequestAndDispatchAsMuchAsPossible()` (lines 851-858) does the cleanup:\n\n```go\nfunc (qs *queueSet) finishRequestAndDispatchAsMuchAsPossible(req *request) bool {\n    qs.lockAndSyncTime(req.ctx)\n    defer qs.lock.Unlock()\n    qs.finishRequestLocked(req)         // Remove from executing set, free seats\n    qs.dispatchAsMuchAsPossibleLocked() // Try to dispatch waiting requests\n    return qs.isIdleLocked()            // Return idle status\n}\n```\n\n**finishRequestLocked** (lines 863-895) updates accounting:\n- Decrements `totRequestsExecuting`, `totSeatsInUse`, `queue.seatsInUse`.\n- Updates metrics.\n- Handles the distinction between initial and final phases (see Section 3.5).\n\n### 3.5 Work Estimation: Initial and Final Phases\n\nAPF distinguishes between **initial phase work** and **final phase work** to model request lifecycle:\n\n```\nInitial Phase: Request is being processed\n              (executing in RequestsExecuting)\n\nFinal Phase: Request is finishing up (sending response,\n            writing logs, etc.)\n```\n\n**WorkEstimate structure** (`/workspace/staging/src/k8s.io/apiserver/pkg/util/flowcontrol/request/config.go`):\n```go\ntype WorkEstimate struct {\n    InitialSeats      uint64         // Seats during main processing\n    FinalSeats        uint64         // Seats for finalization\n    AdditionalLatency time.Duration  // Time to release FinalSeats\n}\n```\n\n**Total work computation** (types.go:127-146):\n```go\ntotalWork = InitialSeats * estimatedServiceDuration + FinalSeats * AdditionalLatency\n```\n\n**Seat allocation transitions**:\n1. Request dispatched: occupies `InitialSeats`.\n2. Request finishes execution: transitions to `FinalSeats` for `AdditionalLatency` duration.\n3. After `AdditionalLatency` elapses (managed elsewhere), completely releases.\n\nThe QueueSet does not directly implement phase transitions; it assumes the caller manages this. The controller informs the queueset about the total work (both phases combined) for dispatch scheduling.\n\n---\n\n## Part 4: Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Capacity Exhaustion and Rejection\n\nWhen a QueueSet reaches capacity, requests are rejected. The rejection logic (rejectOrEnqueueToBoundLocked, lines 618-629) ensures a \"graceful degradation\" curve:\n\n**Capacity state transitions**:\n\n```\nIdle state (totSeatsInUse = 0)\n    \u2193\n  Requests arrive and execute\n    \u2193\nSeats in use < ConcurrencyLimit\n    \u2192 New requests enqueued into queues\n    \u2193\nSeats in use \u2265 ConcurrencyLimit AND queue not full\n    \u2192 New requests still enqueued (queue backpressure)\n    \u2193\nSeats in use \u2265 ConcurrencyLimit AND queue full\n    \u2192 New requests are REJECTED (HTTP 429)\n```\n\n**Why not reject immediately at concurrency limit?**\n\nThe design allows **some queueing even at capacity**. This reduces rejection spikes and provides better Retry-After signals. It's a tradeoff between:\n- **Rejection ratio**: Queueing reduces rejections but increases latency.\n- **Latency**: Clients waiting in queues experience queueing delay.\n\nThe `QueueLengthLimit` (typically 50-100) bounds the maximum queue depth, preventing unbounded memory growth.\n\n### 4.2 Virtual Time Overflow and Epoch Advance\n\n**Limit**: `currentR` is a `uint64` with fixed-point scaling. Maximum representable value is ~2^64 seat-seconds.\n\n**Risk**: If a queueset runs continuously for years, `currentR` could overflow, causing fairness violations.\n\n**Mitigation**: `advanceEpoch()` resets when `currentR \u2265 highR = (rDecrement + rDecrement/2)`, where `rDecrement = MaxSeatSeconds / 2 \u2248 2^63`. This occurs roughly when the accumulated work reaches ~2.25 \u00d7 10^18 seat-seconds. For a typical server with a concurrency limit of 5000 seats and an average service duration of 3ms, this corresponds to years of continuous operation.\n\n**Design tradeoff**: The epoch advance is O(number of queues \u00d7 requests waiting), which could be expensive. However, in practice, it triggers rarely (years apart). The code includes safety checks (klog.ErrorS) to detect and report underflows during epoch advance, which would indicate a severe bug.\n\n### 4.3 Queue Saturation and Head-of-Line Blocking\n\n**Without shuffle sharding**: If multiple flows hash to the same queue, the oldest request (regardless of flow) is always dispatched. This can cause head-of-line blocking: a large request from Flow A can block small requests from Flow B.\n\n**With shuffle sharding**: Conflicts are reduced probabilistically. Even if flows hash to overlapping queues, the system selects from the least-loaded queue in the hand. Blocking still occurs, but is mitigated.\n\n**Limitation**: Shuffle sharding is probabilistic, not deterministic. Two flows with certain hash patterns could still share most queue selections. The tradeoff is:\n- **Deterministic fairness**: Assign each flow to a dedicated queue (expensive, doesn't scale).\n- **Probabilistic fairness**: Use shuffle sharding (efficient, provides statistical fairness).\n\n**Operational tuning**: HandSize (number of queues sampled during sharding) and DesiredNumQueues control the trade-off. Larger HandSize and more queues reduce conflicts but increase memory and CPU.\n\n### 4.4 Virtual Time Ratio and Fairness Under Variable Load\n\n**Challenge**: How should virtual time advance when the number of active queues changes?\n\n**Current approach** (`getVirtualTimeRatioLocked`):\n```\nratio = min(totalSeatsRequested, concurrencyLimit) / activeQueues\n```\n\nThis ensures:\n- If 5 queues are active and limit is 5 seats, each queue advances by 1 seat-second per real second.\n- If 10 queues become active, each advances by 0.5 seat-seconds per real second.\n- If load decreases back to 5 queues, each advances back to 1.\n\n**Fairness guarantee**: All active queues share the concurrency limit equally (in steady state). However, **bursty workloads** can cause unfairness:\n\nExample: Queue A has 10 large requests; Queue B has 1 small request. Both arrive simultaneously. Queue A will occupy more seats, causing its virtual time to advance slower. By the time Queue B's request executes, Queue A has accumulated a backlog, leading to poor latency for Queue B. This is a **fundamental tradeoff** in fair queuing: fairness is measured in work (seat-seconds), not wall-clock time.\n\n### 4.5 Concurrency Limit Borrowing and Seat Demand\n\nThe queueset itself does not implement **seat borrowing** (allowing a priority level to exceed its nominal concurrency limit). That logic is in the `apf_controller.go` (`seatDemandStats`, lines 255-274, and `borrowingAdjustmentPeriod`, line 79).\n\n**Why separate this?** The queueset operates on the **currentCL** (dynamic concurrency limit set by the controller). The controller periodically adjusts currentCL based on seat demand statistics, allowing priority levels to borrow unused capacity from others. The queueset simply enforces currentCL without knowing about borrowing.\n\n---\n\n## Part 5: Extension Points and Customization Hooks\n\n### 5.1 Work Estimator Interface\n\n**Location**: `/workspace/staging/src/k8s.io/apiserver/pkg/util/flowcontrol/request/list_work_estimator.go`, `mutating_work_estimator.go`, `config.go`\n\n**Purpose**: Estimate the \"work\" (seats) a request will consume.\n\n**Customization point**: The `workEstimator` function passed to `Handle()` (apf_filter.go:156-159) determines seat counts. Each queue adds O(n) overhead to dispatch. 1000 queues would cause noticeable latency in dispatch path (every request triggers a scan).\n- **Queue length**: Typically bounded at 50-100 due to QueueLengthLimit. Unbounded queue length would cause memory issues.\n- **Requests per second**: The queueset itself is O(1) per request (except dispatch). However, the global lock (`qs.lock`) is held during critical sections, limiting parallelism. At very high request rates (> 100k RPS), the lock contention becomes the bottleneck.\n\n### 6.2 Lock Contention and Concurrency\n\n**Critical section**: `lockAndSyncTime()` (lines 456-463) acquires a single mutex protecting the queueset. All updates (enqueue, dequeue, virtual time sync) hold this lock.\n\n**Risk**: Under high request rates, lock contention can serialize the dispatch path, reducing throughput.\n\n**Mitigation in code**:\n- Lock is held for the minimal duration (milliseconds per request).\n- Dispatch happens greedily in a loop (dispatchAsMuchAsPossibleLocked), so multiple requests can be dispatched before releasing the lock.\n- Virtual time sync is piggy-backed onto request arrival/finish, avoiding a separate sync goroutine.\n\n**Operational consideration**: If klog verbosity is set to V(5) or higher, extensive logging occurs while holding the lock (lines 318-319, 686-687, etc.), which can significantly impact performance. Production deployments should use V(2) or V(3).\n\n### 6.3 Memory Usage\n\n**Per-QueueSet overhead**:\n- Array of n queues, each ~100 bytes (mostly pointers) \u2192 ~1KB per 10 queues.\n- FIFO list for each queue, grows with queue length (each element ~200 bytes including request struct).\n- At 100 queues \u00d7 100 request/queue limit \u2192 ~2MB overhead per priority level.\n\n**Per-request overhead**:\n- request struct (~300 bytes): context, work estimate, tracking fields.\n- promise.WriteOnce (~100 bytes): coordination value.\n- ~400 bytes per queued request.\n\n**Practical bounds**:\n- Default: 5-10 priority levels \u00d7 128 queues \u00d7 50 request limit = ~256 MB for APF queues alone (excluding the HTTP handlers themselves).\n- At capacity (429 errors), all queue lengths are at their limit, so memory is bounded.\n\n**Risk**: If QueueLengthLimit is misconfigured too high, memory growth is unbounded until the API server hits the system OOM limit.\n\n### 6.4 Fairness and Starvation Guarantees\n\n**Guarantee 1: No indefinite starvation**\n\nProof sketch: Virtual time is monotonically increasing. For any queue Q with waiting requests, its `nextDispatchR` is finite. Eventually, `currentR \u2265 nextDispatchR`, and Q becomes the minimum virtual finish time queue (or close to it). Thus Q's requests are dispatched.\n\n**Guarantee 2: Fairness under homogeneous load**\n\nIf all queues have requests of equal work, and the concurrency limit is not exhausted, all queues advance their virtual time at the same rate. Each queue's requests are dispatched in FIFO order within that queue. By the fair queuing algorithm, all queues receive equal \"service\" (seat-seconds) over time.\n\n**Caveat: Practical fairness is probabilistic**\n\nDue to shuffle sharding, two flows can conflict and end up in adjacent or overlapping queues. The oldest request in the selected queue is always dispatched, regardless of flow age within that queue. In the worst case, a long-running request can block many short requests from a competing flow.\n\n**Operational advice**:\n- Monitor queue depth per flow (via metrics). If certain flows consistently have long queues, increase DesiredNumQueues or HandSize.\n- Adjust QueueLengthLimit based on acceptable worst-case queueing delay.\n\n### 6.5 Observability and Debugging\n\n**Key metrics exported** (metrics.go):\n\n```\n# Prometheus metrics\n- apiserver_flowcontrol_request_concurrency_limit\n- apiserver_flowcontrol_request_queue_length_limit\n- apiserver_flowcontrol_current_r  // Virtual time value\n- apiserver_flowcontrol_requests_executing  // Per priority level\n- apiserver_flowcontrol_requests_in_queues\n- apiserver_flowcontrol_request_dispatch_no_accommodation\n- apiserver_flowcontrol_queue_age_seconds  // How long requests wait\n```\n\n**Debug information**:\n\nQueueSet.Dump() (lines 148-171 of types.go) provides a snapshot of internal state:\n\n```go\ntype QueueDump struct {\n    NextDispatchR      string\n    Requests           []RequestDump  // Waiting\n    RequestsExecuting  []RequestDump  // Executing\n    ExecutingRequests  int\n    SeatsInUse         int\n    QueueSum           QueueSum\n}\n```\n\nThis is exposed via the `/debug/api_priority_and_fairness` HTTP endpoint (apf_controller_debug.go).\n\n**Logging**:\n- V(4): Queue selections and dispatch decisions.\n- V(5): Request arrivals, rejections, queueing/dequeueing events.\n- V(6): Detailed dispatch metrics and virtual time values.\n- V(7): Shuffle shard selection details.\n- V(11): Per-queue virtual finish time calculations.\n\n**Troubleshooting**:\n1. # Kubernetes API Priority and Fairness (APF) QueueSet: An Algorithmic Deep-Dive

## Executive Summary

The Kubernetes Advanced Priority and Fairness (APF) subsystem implements a multi-queue, fair-queuing based approach to request prioritization and concurrency management at the API server level. The **QueueSet** component—sitting within the fairqueuing subsystem—is the algorithmic core that implements fair queuing for server requests (FQSR), a generalization of the classic fair queuing work that combines priority levels with multiple isolated queues and virtual-time-based dispatch scheduling.

This document explains how APF QueueSet manages request queuing, dispatches requests fairly across flows, estimates work using "seats" (a concurrency model), and maintains fairness guarantees under load.

---

## Part 1: Subsystem Purpose and Boundaries

### 1.1 Purpose

The APF QueueSet subsystem exists to **prevent API server overload while ensuring fair resource sharing across multiple priority levels and distinguishable flows within each level**. Rather than simply rejecting or dropping requests when the API server reaches capacity, APF uses queuing and virtual-time-based scheduling to:

- **Preserve fairness**: Requests from different sources (users, services, system components) are isolated and receive fair scheduling priority based on configurable priority levels.
- **Prevent starvation**: Each queue maintains its own virtual time progress, preventing low-priority flows from blocking high-priority ones indefinitely.
- **Enable graceful degradation**: Requests wait in queues instead of being immediately dropped, giving clients a better experience through predictable Retry-After headers.
- **Support work-aware queueing**: Requests are assigned "seats" (a unit of concurrency)
