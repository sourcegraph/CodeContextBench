`poll()` only\n- **Callback ordering**: `onPartitionsRevoked()` \u2192 `onPartitionsAssigned()` in normal scenarios\n- **Empty subscription**: Calling `subscribe(Collections.emptyList())` is equivalent to `unsubscribe()`\n\n### Assign (Manual Assignment)\n\nDirect partition assignment without group coordination:\n\n```java\nTopicPartition partition0 = new TopicPartition(\"topic\", 0);\nTopicPartition partition1 = new TopicPartition(\"topic\", 1);\nconsumer.assign(Arrays.asList(partition0, partition1));\n```\n\n**Behavioral Notes:**\n\n- **No rebalancing**: Manual assignment bypasses group coordination\n- **No callbacks**: `ConsumerRebalanceListener` is not invoked\n- **Non-incremental**: Replaces previous assignment (manual or subscription)\n- **Use cases**:\n  - Specialized partition assignment logic\n  - Stateful processing (each consumer maintains local state per partition)\n  - Integration with external coordination systems\n- **Unique group ID**: Recommended to use unique `group.id` for each manually-assigned consumer to avoid offset conflicts\n\n### Unsubscribe\n\n```java\nconsumer.unsubscribe();\n```\n\nClears both subscriptions and manual assignments. Consumer has no partitions assigned after this call.\n\n### Subscription State\n\n```java\nSet<String> subscribed = consumer.subscription();      // Current topics (empty if using manual assign)\nSet<TopicPartition> assigned = consumer.assignment();  // Currently assigned partitions\n```\n\n---\n\n## Polling and Data Fetching\n\n### poll(Duration timeout)\n\nFetches records from assigned partitions. **Central operation** in the consumer loop.\n\n```java\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\nfor (ConsumerRecord<String, String> record : records) {\n    System.out.println(\"Topic: \" + record.topic() +\n                      \", Partition: \" + record.partition() +\n                      \", Offset: \" + record.offset() +\n                      \", Key: \" + record.key() +\n                      \", Value: \" + record.value());\n}\n```\n\n**Blocking Behavior:**\n\n- Returns immediately if records are available\n- Returns immediately if position advances past control records or aborted transactions (read_committed)\n- Waits up to timeout if no records available and no immediate triggers\n- **May block longer than timeout** if:\n  - Rebalancing is in progress\n  - Rebalance callbacks are executing\n  - Network metadata refresh is needed\n  - Broker is unavailable\n\n**Behavior Semantics:**\n\n| Scenario | Behavior |\n|----------|----------|\n| Records available | Returns immediately with records |\n| No records, normal case | Waits up to timeout, returns empty |\n| Rebalance triggered | Executes callbacks (may exceed timeout) |\n| Metadata needs refresh | Fetches metadata before returning |\n| Consumer paused | Returns immediately (empty) |\n| Position invalid | Applies `auto.offset.reset` policy |\n\n**Automatic Offset Commits:**\n\nIf `enable.auto.commit=true`:\n- Offsets are committed periodically during `poll()` (interval controlled by `auto.commit.interval.ms`)\n- Commits the last **position** returned from previous `poll()` calls\n- Failures are logged but not thrown\n\n**What Happens During poll():**\n\n1. Update metadata if needed\n2. Check for session timeout / max.poll.interval.ms violations\n3. Execute rebalance callbacks if rebalancing\n4. Refresh committed offsets on assignment\n5. Fetch records from brokers\n6. Deserialize records (may throw `RecordDeserializationException`)\n7. Filter control records and aborted transactional messages (if `read_committed`)\n8. Commit offsets (if auto-commit enabled)\n9. Return `ConsumerRecords`\n\n**Exceptions Thrown:**\n\n- `WakeupException`: If `wakeup()` was called\n- `InterruptException`: If thread was interrupted\n- `InvalidOffsetException`: Position undefined or out of range\n- `RecordDeserializationException`: Deserialization failed\n- `AuthenticationException`: Authentication failed\n- `AuthorizationException`: Not authorized\n- `KafkaException`: Other unrecoverable errors\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\nThree offset concepts are important:\n\n1. **Position**: The offset of the next record to be fetched\n   - Maintained by consumer\n   - Advances on each `poll()` call\n   - Controlled via `seek()` operations\n   - Not persisted; lost on restart\n\n2. **Committed Offset**: The last offset successfully committed\n   - Persisted in Kafka or external store\n   - Used to resume after restart\n   - Must always be the **next offset to read** (position + already-processed)\n   - Accessed via `committed()`\n\n3. **Offset Lag**: Position - Committed\n   - Distance between what's been read and what's been committed\n   - Indicates potential data loss on failure\n\n### Auto-commit\n\nOffsets committed automatically if `enable.auto.commit=true`:\n\n```properties\nenable.auto.commit=true\nauto.commit.interval.ms=5000  # Commit every 5 seconds\n```\n\n**Semantics:**\n- Commits during `poll()` at configured interval\n- Commits the position (next offset to read)\n- Failures are logged, not thrown\n- With `enable.auto.commit=true`, you **must** consume all records returned by `poll()` or risk skipping records\n\n**When NOT to use auto-commit:**\n- Processing time is unpredictable\n- Need exactly-once semantics\n- Processing result must be atomic with offset commit\n- Using external offset storage\n\n### Manual Synchronous Commit\n\n```java\n// Commit all offsets\nconsumer.commitSync();\n\n// Commit with timeout\nconsumer.commitSync(Duration.ofSeconds(5));\n\n// Commit specific offsets\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\nfor (TopicPartition partition : records.partitions()) {\n    offsets.put(partition, records.nextOffsets().get(partition));\n}\nconsumer.commitSync(offsets);\n\n// Commit per-partition\nfor (TopicPartition partition : records.partitions()) {\n    long nextOffset = records.records(partition).isEmpty()\n        ? consumer.position(partition)\n        : records.records(partition).get(records.records(partition).size() - 1).offset() + 1;\n    consumer.commitSync(Collections.singletonMap(\n        partition,\n        new OffsetAndMetadata(nextOffset)\n    ));\n}\n```\n\n**Behavior:**\n\n- **Blocks** until commit completes or timeout\n- **Retries** retriable errors automatically\n- Throws `CommitFailedException` if partitions are lost (group rebalanced)\n- Throws `TimeoutException` if timeout expires\n\n**When to use:**\n- After successfully processing and persisting records\n- In rebalance listener `onPartitionsRevoked()`\n- Before closing consumer\n\n### Manual Asynchronous Commit\n\n```java\n// Fire and forget\nconsumer.commitAsync();\n\n// With callback\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        logger.error(\"Commit failed: \" + exception);\n        // Handle error (may need to retry or shutdown)\n    } else {\n        logger.info(\"Offsets committed: \" + offsets);\n    }\n});\n\n// Commit specific offsets with callback\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\nfor (TopicPartition partition : records.partitions()) {\n    offsets.put(partition, records.nextOffsets().get(partition));\n}\nconsumer.commitAsync(offsets, callback);\n```\n\n**Behavior:**\n\n- **Non-blocking** - returns immediately\n- Callback executes asynchronously\n- Multiple async commits maintain order\n- Async commits complete before subsequent `commitSync()`\n- Failures in callback are handled by callback only (not thrown)\n\n**Common Pattern:**\n```java\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        // Log but continue (retry on next success)\n    }\n});\n```\n\n### Commit Failure Scenarios\n\n**CommitFailedException:**\n- Consumer is no longer part of group\n- Rebalance occurred before commit completed\n- `max.poll.interval.ms` exceeded\n- Session timed out\n\n**Recovery:**\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Partitions likely reassigned; cannot retry\n    // Shutdown cleanly or let next poll() discover new assignment\n    logger.error(\"Commit failed, consumer out of group\", e);\n}\n```\n\n---\n\n## Position and Seek Operations\n\n### Getting Position\n\n```java\n// Next offset that will be fetched\nlong position = consumer.position(partition);\n\n// With timeout\nlong position = consumer.position(partition, Duration.ofSeconds(5));\n```\n\n**Behavior:**\n\n- Returns offset of next record to be fetched\n- May trigger remote call to broker if not yet determined\n- Throws `InvalidOffsetException` if undefined\n- Blocks up to timeout\n\n### Seeking to Offsets\n\n```java\n// Seek to specific offset\nconsumer.seek(partition, 100L);\n\n// Seek to beginning\nconsumer.seekToBeginning(Collections.singleton(partition));\nconsumer.seekToBeginning();  // All assigned partitions\n\n// Seek to end\nconsumer.seekToEnd(Collections.singleton(partition));\nconsumer.seekToEnd();  // All assigned partitions\n\n// Seek with leader epoch (for log truncation detection)\nconsumer.seek(partition, new OffsetAndMetadata(100L, Optional.of(5)));\n```\n\n**Behavioral Notes:**\n\n- **Lazy evaluation**: Seek changes don't take effect until next `poll()`\n- **In-flight requests**: Existing fetch requests complete before new position takes effect\n- **Invalid offset handling**: Controlled by `auto.offset.reset` policy:\n  - `earliest`: Seek to beginning\n  - `latest`: Seek to end\n  - `none`: Throw `OffsetOutOfRangeException`\n- **seekToEnd for read_committed**: Seeks to Last Stable Offset (LSO), not high watermark\n- **Position after seek**: Call `position()` after `poll()` to see actual position\n\n### Seeking by Timestamp\n\n```java\n// Find offsets for given timestamps\nMap<TopicPartition, Long> timestampsToSearch = new HashMap<>();\ntimestampsToSearch.put(partition1, System.currentTimeMillis() - 60000); // 1 min ago\ntimestampsToSearch.put(partition2, System.currentTimeMillis());\n\nMap<TopicPartition, OffsetAndTimestamp> results = consumer.offsetsForTimes(timestampsToSearch);\n\nfor (TopicPartition partition : results.keySet()) {\n    OffsetAndTimestamp result = results.get(partition);\n    if (result != null) {\n        consumer.seek(partition, result.offset());\n    } else {\n        // No message found for timestamp\n    }\n}\n```\n\n**Behavior:**\n\n- Returns earliest offset with timestamp >= requested timestamp\n- Returns `null` if no matching offset or topic has no timestamps\n- Requires message format v0.10+\n- Blocks up to timeout\n\n### Beginning and End Offsets\n\n```java\n// First available offset\nMap<TopicPartition, Long> beginnings = consumer.beginningOffsets(partitions);\n\n// Last offset\nMap<TopicPartition, Long> ends = consumer.endOffsets(partitions);\n```\n\n**Semantics:**\n\n- **Beginning**: First offset in partition (log start offset)\n- **End**: One past last offset\n  - For `read_uncommitted`: High watermark\n  - For `read_committed`: Last Stable Offset (LSO)\n- Does not change consumer position\n\n---\n\n## Consumer Groups and Rebalancing\n\n### Group Coordination\n\nKafka consumer groups provide automatic partition assignment and failure detection:\n\n**How it works:**\n1. Consumer joins group on first `poll()`\n2. Broker detects membership changes\n3. Broker triggers rebalance\n4. Members execute `onPartitionsRevoked()` callbacks\n5. Broker assigns new partitions\n6. Members execute `onPartitionsAssigned()` callbacks\n7. Normal consumption resumes\n\n### Rebalance Triggers\n\nRebalances occur when:\n- New consumer joins group\n- Consumer leaves/crashes\n- Subscribed topics change partition count\n- New topics match subscription pattern\n- Manually triggered via `enforceRebalance()`\n\n### Rebalance Listeners\n\nImplement `ConsumerRebalanceListener` for custom logic:\n\n```java\nclass RebalanceListener implements ConsumerRebalanceListener {\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // Called when losing partitions (before reassignment)\n        // - Commit offsets\n        // - Flush writes\n        // - Release resources\n        consumer.commitSync();\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Called after new assignment (before fetching)\n        // - Initialize position\n        // - Load state\n        // - Open resources\n        for (TopicPartition partition : partitions) {\n            long offset = readOffsetFromExternalStore(partition);\n            consumer.seek(partition, offset);\n        }\n    }\n\n    @Override\n    public void onPartitionsLost(Collection<TopicPartition> partitions) {\n        // Called when partitions unexpectedly lost (e.g., timeout)\n        // Partitions may already be owned by other consumers\n        // - Don't commit (might fail)\n        // - Clean up resources\n        // - Default: delegates to onPartitionsRevoked()\n    }\n}\n\nconsumer.subscribe(Arrays.asList(\"topic\"), new RebalanceListener());\n```\n\n**Callback Execution:**\n- Callbacks execute during `poll()` only\n- On rebalance, `poll()` may block longer than timeout\n- Exceptions in callbacks propagate to caller of `poll()`\n- `WakeupException` or `InterruptException` from callbacks propagate to `poll()`\n\n### Rebalance Protocols\n\nKafka supports two rebalancing protocols:\n\n**Classic (Eager) Rebalancing:**\n- All members stop consuming\n- `onPartitionsRevoked()` called with ALL partitions\n- Broker assigns partitions\n- `onPartitionsAssigned()` called with all partitions (even unchanged)\n- Higher downtime, simpler logic\n\n**Cooperative Rebalancing:**\n- Members retain partitions when possible\n- `onPartitionsRevoked()` called only with revoked partitions\n- Broker assigns partitions\n- `onPartitionsAssigned()` called only with new partitions\n- Lower downtime, more complex\n\nControl via `partition.assignment.strategy`:\n```properties\npartition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor\n```\n\n### Heartbeat and Session Management\n\n```properties\nsession.timeout.ms=10000        # Seconds without heartbeat before removal\nmax.poll.interval.ms=300000     # Max time between polls (liveness check)\nheartbeat.interval.ms=3000      # Heartbeat frequency\n```\n\n**Session Timeout:**\n- If consumer can't send heartbeat for `session.timeout.ms`, broker removes it\n- Other members trigger rebalance\n- Consumer unaware until next `poll()`\n\n**Max Poll Interval:**\n- If consumer doesn't call `poll()` within `max.poll.interval.ms`, proactively leaves group\n- Prevents consumer from holding partitions while hung\n- Next `poll()` triggers rebalance\n- Subsequent `commitSync()` throws `CommitFailedException`\n\n### Force Rebalance\n\n```java\n// Non-blocking; rebalance occurs on next poll()\nconsumer.enforceRebalance();\nconsumer.enforceRebalance(\"user-initiated rebalance\");\n```\n\nUse case: When external state (not tracked by Kafka) changes and requires partition reassignment.\n\n### Group Metadata\n\n```java\nConsumerGroupMetadata metadata = consumer.groupMetadata();\n// Contains: group ID, member ID, generation\n```\n\nUsed when manually coordinating with transactional producers.\n\n---\n\n## Flow Control\n\n### Pause and Resume\n\nControl which partitions are fetched:\n\n```java\n// Pause (no records returned for these partitions)\nconsumer.pause(Arrays.asList(partition1, partition2));\n\n// Resume\nconsumer.resume(Arrays.asList(partition1, partition2));\n\n// Check paused partitions\nSet<TopicPartition> paused = consumer.paused();\n```\n\n**Use Cases:**\n- Slow consumers: Pause fast partitions while processing slow ones\n- Stream join: Pause ahead stream while lagging stream catches up\n- Backpressure: Pause while downstream queue is full\n- Batching: Pause some partitions to batch process others\n\n**Behavioral Notes:**\n- Pause doesn't affect subscription or assignment\n- Does not trigger rebalance\n- Rebalance does NOT preserve pause/resume state\n- Call `pause()` again after rebalance if needed\n\n### Pattern: Controlled Processing\n\n```java\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    // Process with pause/resume for flow control\n    processingQueue.addAll(records);\n\n    if (processingQueue.size() > 10000) {\n        // Too many pending; pause faster partitions\n        Set<TopicPartition> fastPartitions = identifyFastPartitions();\n        consumer.pause(fastPartitions);\n    } else if (processingQueue.size() < 1000) {\n        // Queue emptying; resume\n        consumer.resume(paused);\n    }\n\n    processQueue();\n}\n```\n\n---\n\n## Metadata and Partition Information\n\n### Partition Information\n\n```java\nList<PartitionInfo> partitions = consumer.partitionsFor(\"topic\");\n// Returns: List of PartitionInfo (partition #, leader, replicas, in-sync replicas)\n```\n\n### List All Topics\n\n```java\nMap<String, List<PartitionInfo>> allTopics = consumer.listTopics();\n// All topics consumer is authorized to access\n```\n\n### Current Lag\n\n```java\nOptionalLong lag = consumer.currentLag(partition);\n// Returns: position - end offset (or empty if unknown)\n```\n\n### Metrics\n\n```java\nMap<MetricName, ? extends Metric> metrics = consumer.metrics();\n// Includes: record lag, fetch latency, bytes consumed, etc.\n```\n\n---\n\n## Multi-threaded Processing\n\n### Thread Safety\n\n**KafkaConsumer is NOT thread-safe.**\n\nOnly the `wakeup()` method is thread-safe and can be called from any thread.\n\n### Pattern 1: One Consumer Per Thread\n\nSimplest approach. Each thread has its own consumer:\n\n```java\nExecutorService executor = Executors.newFixedThreadPool(4);\n\nfor (int i = 0; i < 4; i++) {\n    executor.submit(() -> {\n        KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\n        try {\n            consumer.subscribe(Arrays.asList(\"topic\"));\n            while (running.get()) {\n                ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n                for (ConsumerRecord<String, String> record : records) {\n                    processRecord(record);\n                }\n            }\n        } finally {\n            consumer.close();\n        }\n    });\n}\n```\n\n**Pros:**\n- Simplest implementation\n- No inter-thread synchronization\n- In-order processing per partition guaranteed\n\n**Cons:**\n- More TCP connections\n- Limited by partition count\n- Cannot scale beyond partition count\n\n### Pattern 2: Consumer + Processing Thread Pool\n\nOne consumer thread feeds work queue to processors:\n\n```java\nclass ConsumerRunner implements Runnable {\n    private final KafkaConsumer<String, String> consumer;\n    private final BlockingQueue<ConsumerRecord<String, String>> queue;\n    private final AtomicBoolean running;\n\n    @Override\n    public void run() {\n        try {\n            consumer.subscribe(Arrays.asList(\"topic\"));\n            while (running.get()) {\n                ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n                for (ConsumerRecord<String, String> record : records) {\n                    try {\n                        queue.put(record);  // Blocks if queue full\n                    } catch (InterruptedException e) {\n                        Thread.currentThread().interrupt();\n                        break;\n                    }\n                }\n\n                // Pause if queue too full\n                if (queue.size() > 10000) {\n                    consumer.pause(consumer.assignment());\n                } else if (queue.size() < 1000 && paused) {\n                    consumer.resume(consumer.assignment());\n                }\n            }\n        } finally {\n            consumer.close();\n        }\n    }\n}\n\n// Processing threads\nfor (int i = 0; i < 4; i++) {\n    executor.submit(() -> {\n        while (running.get()) {\n            ConsumerRecord<String, String> record = queue.take();\n            processRecord(record);\n            offsetsToCommit.put(record.partition(), record.offset() + 1);\n        }\n    });\n}\n\n// Periodic commit thread\nexecutor.submit(() -> {\n    while (running.get()) {\n        Thread.sleep(5000);\n        Map<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\n        offsetsToCommit.forEach((partition, offset) ->\n            offsets.put(partition, new OffsetAndMetadata(offset))\n        );\n        consumer.commitAsync(offsets);\n    }\n});\n```\n\n**Pros:**\n- Scales beyond partition count\n- Can process independently\n- Flexible throughput tuning\n\n**Cons:**\n- Complex offset management\n- Lost message ordering guarantees\n- Potential for duplicate processing\n\n### Shutting Down\n\n```java\nExecutorService executor = Executors.newFixedThreadPool(2);\n\n// Consumer thread\nFuture<?> consumerFuture = executor.submit(consumerRunnable);\n\n// Shutdown\nrunning.set(false);\nexecutor.awaitTermination(30, TimeUnit.SECONDS);\n\nif (!executor.isTerminated()) {\n    executor.shutdownNow();\n}\n```\n\n### Wakeup Pattern\n\nInterrupt blocking operations from another thread:\n\n```java\n// Consumer thread\ntry {\n    consumer.subscribe(Arrays.asList(\"topic\"));\n    while (running.get()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(10));\n        processRecords(records);\n    }\n} catch (WakeupException e) {\n    if (!running.get()) {\n        // Expected shutdown\n    } else {\n        throw e;\n    }\n} finally {\n    consumer.close();\n}\n\n// Shutdown thread\npublic void shutdown() {\n    running.set(false);\n    consumer.wakeup();  // Safe to call from any thread\n}\n```\n\n---\n\n## Error Handling\n\n### Exception Hierarchy\n\n```\nKafkaException (base)\n\u251c\u2500\u2500 WakeupException (from wakeup())\n\u251c\u2500\u2500 InterruptException (from thread interruption)\n\u251c\u2500\u2500 AuthenticationException (auth failed)\n\u251c\u2500\u2500 AuthorizationException (not authorized)\n\u251c\u2500\u2500 TimeoutException (operation timed out)\n\u251c\u2500\u2500 CommitFailedException (can't commit, out of group)\n\u251c\u2500\u2500 RetriableCommitFailedException (commit may succeed if retried)\n\u251c\u2500\u2500 InvalidOffsetException (position undefined/out of range)\n\u251c\u2500\u2500 NoOffsetForPartitionException (no offset found)\n\u251c\u2500\u2500 OffsetOutOfRangeException (auto.offset.reset=none)\n\u251c\u2500\u2500 LogTruncationException (committed offset lost to log truncation)\n\u251c\u2500\u2500 RecordDeserializationException (deserialization failed)\n\u251c\u2500\u2500 InvalidGroupIdException (bad group ID)\n\u2514\u2500\u2500 ... (others)\n```\n\n### WakeupException\n\nRaised when `wakeup()` was called:\n\n```java\ntry {\n    ConsumerRecords records = consumer.poll(Duration.ofSeconds(10));\n} catch (WakeupException e) {\n    // wakeup() was called; clean up and exit\n    break;\n}\n```\n\n### CommitFailedException\n\nRaised when commit fails due to group rebalancing:\n\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Consumer no longer in group\n    // Cannot commit; partitions reassigned\n    // Solution: Retry on next poll() or shutdown\n    logger.error(\"Commit failed, out of group\", e);\n}\n```\n\n### InvalidOffsetException\n\nPosition is invalid (outside log bounds):\n\n```java\ntry {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n} catch (InvalidOffsetException e) {\n    // Position undefined or out of range\n    // auto.offset.reset policy applies on next poll()\n    // Or seek explicitly: consumer.seek(partition, validOffset);\n}\n```\n\n### RecordDeserializationException\n\nDeserialization failed for a record:\n\n```java\ntry {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n} catch (RecordDeserializationException e) {\n    // Record deserialization failed\n    // Partition: e.topicPartition()\n    // Offset: e.offset()\n    // Solution: skip record or shutdown\n\n    // Skip to next record\n    consumer.seek(e.topicPartition(), e.offset() + 1);\n}\n```\n\n### AuthenticationException / AuthorizationException\n\nAuthentication or authorization failed:\n\n```java\ntry {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n} catch (AuthenticationException e) {\n    // Invalid credentials\n    logger.error(\"Authentication failed\", e);\n} catch (AuthorizationException e) {\n    // Not authorized to read topic/group\n    logger.error(\"Not authorized\", e);\n}\n```\n\n### Common Error Handling Patterns\n\n**Pattern 1: Log and Retry**\n```java\nwhile (true) {\n    try {\n        consumer.commitSync();\n        break;\n    } catch (CommitFailedException e) {\n        logger.warn(\"Commit failed, retrying\", e);\n        Thread.sleep(100);\n    }\n}\n```\n\n**Pattern 2: Log and Fail**\n```java\ntry {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n    processRecords(records);\n} catch (KafkaException e) {\n    logger.error(\"Fatal error\", e);\n    System.exit(1);\n}\n```\n\n**Pattern 3: Skip Bad Records**\n```java\ntry {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord record : records) {\n        try {\n            processRecord(record);\n        } catch (Exception e) {\n            logger.error(\"Failed to process record at offset \" + record.offset(), e);\n            // Continue with next record\n        }\n    }\n} catch (WakeupException e) {\n    // Shutdown\n}\n```\n\n---\n\n## Configuration-Driven Behaviors\n\n### Key Configuration Properties\n\n| Property | Default | Impact |\n|----------|---------|--------|\n| `bootstrap.servers` | (required) | Initial brokers to connect |\n| `group.id` | (required) | Consumer group ID |\n| `key.deserializer` | (required) | Key deserializer class |\n| `value.deserializer` | (required) | Value deserializer class |\n| `enable.auto.commit` | true | Auto-commit offsets |\n| `auto.commit.interval.ms` | 5000 | Commit interval |\n| `session.timeout.ms` | 10000 | Heartbeat timeout |\n| `max.poll.interval.ms` | `enable.auto.commit=true`:\n- Offsets committed automatically during `poll()`\n- Interval controlled by `auto.commit.interval.ms`\n- Commits the latest position\n- Failures logged, not thrown\n\n**Risk:** If `poll()` returns records but they're not processed before the next `poll()` or `close()`, records are skipped.\n\n**Solution:** Disable auto-commit if processing time is significant:\n```properties\nenable.auto.commit=false\n```\n\n### Offset Reset Policy\n\nWhen position is invalid (out of range):\n\n- `earliest`: Seek to beginning of partition\n- `latest`: Seek to ## Overview

### Purpose

`KafkaConsumer<K, V>` is the main client API for consuming records from Apache Kafka. It handles:
- Connection management to Kafka brokers
- Fetching records from assigned partitions
- Offset management (position tracking and commits)
- Consumer group coordination and rebalancing
- Thread-safe wakeup signaling

### Key Characteristics

- **Not thread-safe**: All methods (except `wakeup()`) must be called from a single consumer thread
- **Event-driven**: Uses a polling model; all coordination and network I/O occurs during `poll()`
- **Group-managed**: Supports both automatic partition assignment via consumer groups and manual partition assignment
- **Stateful**: Maintains position (offset) for each assigned partition
