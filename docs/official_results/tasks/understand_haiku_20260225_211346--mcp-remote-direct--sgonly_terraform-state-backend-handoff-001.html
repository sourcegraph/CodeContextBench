Failure Modes\n\n### Configuration Errors\n\n**Detection**: `PrepareConfig()` and `Configure()` methods\n\nCommon issues:\n- Missing required config values (bucket name, table name, etc.)\n- Invalid credential formats\n- Invalid region specifications\n- Deprecated backend configurations\n\n**Handling**: Returned as `tfdiags.Diagnostics` with contextual information\n\n**Example S3 Validation** (`s3/backend.go:PrepareConfig()`):\n```go\n// Checks bucket existence, DynamoDB table, KMS key access, account IDs\n// Returns clear diagnostics for missing/misconfigured resources\n```\n\n### State Locking Failures\n\n**Lock Contention**: When state is already locked\n\n**Mechanism** (`locker.go:LockWithContext()`):\n```go\n// Retry logic with exponential backoff\n// Max delay: 16 seconds\n// Returns LockError with existing lock info\n// Allows force-unlock via CLI\n```\n\n**Timeout Handling**:\n- Context cancellation stops retry attempts\n- Returns last LockError to user\n- User can use `terraform force-unlock <ID>` to break stale locks\n\n**Common Issues**:\n- Process crash with held lock (state remains locked)\n- Stale locks from failed CI/CD runs\n- Network partitions preventing lock release\n\n### Storage Unavailability\n\n**S3 Examples**:\n- Bucket doesn't exist \u2192 RefreshState/PersistState returns error\n- No AWS credentials \u2192 Configure fails with auth error\n- Network timeout \u2192 Persistence operations timeout\n- S3 access denied \u2192 RefreshState/PersistState fails with 403\n\n**Handling**:\n- Operations fail with clear error messages\n- Terraform aborts before state corruption\n- User must resolve underlying issue (credentials, network, permissions)\n\n### State Consistency Issues\n\n**Serial/Lineage Mismatch** (`.stateSnapshotMeta()`):\n- Serial is incremented with each successful persist\n- Lineage tracks state file generations\n- Mismatches indicate concurrent modifications or rollback attempts\n\n**Detection**: In S3 backend's `PersistState()` via DynamoDB conditionally checked against expected serial/lineage\n\n**Corruption Scenarios**:\n1. Direct state file modification (bypassing backends)\n2. Multiple concurrent Terraform runs without locking\n3. Forced state overwrites without lineage/serial preservation\n\n**Prevention**:\n- Always use backends with locking support for team environments\n- `terraform push`/`pull` preserve metadata via `Migrator` interface\n- Versioning in remote storage (S3 versioning, Git, etc.)\n\n### Configuration Change Detection\n\n**File**: `meta_backend.go:BackendConfig()` and `backendConfigNeedsMigration()`\n\nWhen backend type or config changes, Terraform detects and prompts user to:\n1. Review differences\n2. Choose to migrate state between backends\n3. Confirm force migration if serial/lineage checks fail\n\n### Workspace-Specific Failures\n\n- `DeleteWorkspace(\"default\", ...)` always fails - default is mandatory\n- Creating workspaces on single-workspace backends returns `ErrWorkspacesNotSupported`\n- Workspace directory corruption prevents new workspace creation\n\n---\n\n## 5. State Inspection** (`terraform state list/show`):\n```bash\nterraform state list                     # List all resources\nterraform state show resource.id         # Show specific resource\nterraform state pull > state.json        # Export state locally\n```\n\n**2. Lock Status Checking**:\n```bash\nterraform state show -raw output       # See lock metadata in outputs\n# Or inspect storage directly (S3, DynamoDB, Consul, etc.)\n```\n\n**3. Remote State Diagnosis**:\n```bash\n# For S3 backend\naws s3api head-object --bucket BUCKET --key KEY  # Check state file\naws dynamodb scan --table-name TABLE              # Check locks\n\n# For Consul\nconsul kv get terraform/state/default\n\n# For PostgreSQL\nSELECT * FROM states WHERE name = 'default';\n```\n\n**4. Configuration Inspection**:\n```bash\nterraform init -backend=false           # Skip backend init\nterraform backends                      # List configured backends\ncat .terraform/terraform.tfstate         # Backend state metadata\n```\n\n### Common Issues and Resolution\n\n**Issue: State is locked by another process**\n\nDiagnosis:\n1. Check lock metadata: Extract lock ID from error message\n2. Verify process is not actually running: `ps aux | grep terraform`\n3. Check storage lock entry (DynamoDB, Consul, file system)\n\nResolution:\n```bash\nterraform force-unlock <LOCK_ID>\n# or manually delete lock from storage if urgent\n```\n\n**Issue: State file corruption or serial mismatch**\n\nDiagnosis:\n```bash\nterraform state pull | jq '.version, .serial, .lineage'\n# Compare with remote storage\n```\n\nResolution:\n```bash\n# Manual state push (preserves serial/lineage via Migrator)\nterraform state push -force backup.tfstate\n\n# Or export/import via pull/push\nterraform state pull > backup.json\n# ... make manual fixes ...\nterraform state push backup.json\n```\n\n**Issue: Backend configuration conflicts**\n\nDiagnosis:\n```bash\n# When backend config changes\nterraform init  # Shows migration prompts\n```\n\nResolution:\n```bash\nterraform init -migrate-state   # Automatic migration\n# or\nterraform init -reconfigure     # Abandon old backend\n```\n\n**Issue: S3 bucket access denied**\n\nDiagnosis:\n```bash\naws s3api head-object --bucket BUCKET --key KEY\n# or\naws s3api get-bucket-versioning --bucket BUCKET\n```\n\nResolution:\n```bash\n# Verify IAM policy includes:\n# s3:GetObject, s3:PutObject, s3:DeleteObject\n# s3:GetBucketVersioning (if using versioning)\n# dynamodb:DescribeTable, dynamodb:PutItem, etc. (if using DynamoDB locks)\n```\n\n**Issue: Stale lock not releasing**\n\nDiagnosis:\n```bash\n# For S3 + DynamoDB locking\naws dynamodb get-item --table-name LOCKS --key '{\"LockID\":{\"S\":\"path/to/state\"}}'\n\n# Extract lock info\nterraform state show terraform_remote_state  # Won't work if locked\n```\n\nResolution:\n```bash\n# Delete stale lock\nterraform force-unlock <LOCK_ID>\n\n# Or directly from DynamoDB\naws dynamodb delete-item --table-name LOCKS \\\n  --key '{\"LockID\":{\"S\":\"path/to/state\"}}'\n```\n\n---\n\n## 7. # Terraform State Backend Subsystem - Team Handoff Document

## Overview

This document provides a comprehensive guide to the Terraform state backend subsystem, responsible for managing how Terraform stores, retrieves, and locks state files across different storage backends and operational modes. This system enables both local development and large-scale remote state management across teams.

---

## 1. Purpose

### Problem Statement

Terraform's state file is critical—it maps the declared infrastructure in code to real-world resources. A robust backend system solves several key problems:

1. **Shared State Management**: Multiple team members need safe concurrent access to the same state
2. **State Persistence**: State must survive process restarts and failures
3. **Storage Flexibility**: Different organizations need different storage solutions (local files, S3, Terraform Cloud, etc.)
4. **State Integrity**: Concurrent writes must be atomic and prevent corruption
5. **Workspace Isolation**: Teams need multiple independent state files (workspaces) within a single backend

### Why Multiple Backend Types

Different environments have different requirements:

- **Local Backend** (`local`): Development, single-user, filesystem-based
- **S3 Backend** (`s3`): Team collaboration, AWS-native, with optional DynamoDB locking
- **Cloud/Remote Backend** (`cloud`/`remote`): Enterprise, full remote operations, HCP Terraform integration
- **Other Backends** (`azurerm`, `gcs`, `consul`, `pg`, `kubernetes`, etc.): Cloud-provider-specific and specialized solutions
