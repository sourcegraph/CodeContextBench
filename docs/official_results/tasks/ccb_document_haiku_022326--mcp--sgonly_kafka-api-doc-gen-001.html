## Objective

## Scope

Your documentation should cover the **KafkaConsumer API** from the `org.apache.kafka.clients.consumer` package, including:

1. **Core Consumer Lifecycle**
   - Constructor variants and configuration
   - Subscription methods (dynamic and manual assignment)
   - Polling and data fetching semantics
   - Resource cleanup and closing

2. **Offset Management**
   - Synchronous and asynchronous commit strategies
   - Offset queries and position control
   - Seek operations and offset discovery
   - Committed offset semantics

3. **Consumer Group Mechanics**
   - ConsumerRebalanceListener interface and callbacks
   - Rebalance triggers and timing
   - Group membership and heartbeat behavior
   - Partition assignment vs subscription models

4. **Flow Control and Position Management**
   - Pause and resume functionality
   - Position queries and manipulation
   - Offset-to-timestamp lookups
   - Beginning and end offset discovery

5. **Error Handling**
   - Exception types and recovery strategies
   - CommitFailedException and group fencing
   - WakeupException and thread interruption
   - Timeout and authentication errors

## Requirements

### API Methods Documentation (40%)

Document all public methods of the `KafkaConsumer` class with:
- Method signatures including all overloads
- Parameter semantics and validation rules
- Return types and their meanings
- Exception types thrown and conditions

### Behavioral Notes (30%)

Explain critical behavioral semantics:
- **poll() blocking behavior**: when it returns immediately vs when it blocks, timeout handling, rebalance callback execution during poll
- **Offset commit semantics**: difference between sync/async commits, retry behavior, commit failure handling
- **Rebalance coordination**: when rebalances occur (only during poll), callback ordering (revoked then assigned), partition ownership guarantees
- **Thread safety**: which methods are thread-safe, wakeup() special case, event loop model
- **Group membership**: max.poll.interval.ms enforcement, proactive leave behavior, session timeout vs poll timeout
- **Manual vs dynamic assignment**: mutually exclusive nature, use cases for each model
- **Position vs committed offset**: the off-by-one relationship ("committed should be next offset to read")
- **Transactional semantics**: read_committed isolation level, LSO boundary, filtered messages

### Usage Examples (20%)

Provide concrete code examples demonstrating:
- **Basic subscription and polling loop**: subscribe to topics, poll for records, process messages
- **Manual offset commit**: disable auto-commit, explicit commitSync/commitAsync after processing
- **Rebalance listener**: implement ConsumerRebalanceListener, commit offsets on revoke, initialize positions on assign
- **Seek operations**: seekToBeginning, seekToEnd, seek to specific offset, timestamp-based seeking
- **Multi-threaded processing pattern**: single consumer thread with pause/resume coordination and worker pool

### Documentation Structure (10%)

Organize documentation with clear sections:
- Overview and threading model
- Core types (KafkaConsumer, ConsumerRebalanceListener, ConsumerRecords, etc.)
- Subscription and assignment methods
- Polling and data fetching
- Offset management methods
- Flow control and position queries
- Metadata and monitoring
- Lifecycle and resource management
- Exception handling guide
- Configuration-driven behaviors
- Common patterns and best practices

timeout\nConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\n\n// Process records\nfor (ConsumerRecord<String, String> record : records) {\n    System.out.println(record.key() + \": \" + record.value());\n}\n\n// Get records for specific partition\nList<ConsumerRecord<String, String>> tp0Records = records.records(\n    new TopicPartition(\"orders\", 0)\n);\n\n// Get next offsets after consuming\nMap<TopicPartition, OffsetAndMetadata> nextOffsets = records.nextOffsets();\n```\n\n### poll() Blocking Behavior\n\n**When `poll()` returns immediately:**\n- Records are available in internal buffers (returns data with no additional network wait)\n- Timeout expires (returns empty ConsumerRecords)\n- Fetcher has reached end of assigned partitions\n- Metadata refresh not needed\n\n**When `poll()` blocks:**\n- Waiting for broker to accumulate `fetch.min.bytes` (up to `fetch.max.wait.ms`)\n- Rebalance callback executing (can block longer than timeout)\n- Metadata refresh in progress\n- Broker connection establishment\n\n**Critical semantic:** The provided timeout is NOT guaranteed to limit total execution time. If rebalance callbacks (`onPartitionsRevoked()`, `onPartitionsAssigned()`) take time, the actual `poll()` duration can exceed timeout. This is intentional to ensure callbacks complete without interruption.\n\n**Example of extended blocking:**\n```java\n// Timeout is 1 second, but can take longer if rebalance occurs\nConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\n// If rebalance callback takes 3 seconds, this call may take 3+ seconds total\n```\n\n### poll() Frequency and Group Membership\n\nFor group-managed consumers (via `subscribe()`):\n- **Must call `poll()` at least once every `max.poll.interval.ms`** (default 5 minutes)\n- If no `poll()` call within this window, broker removes consumer from group\n- Rebalance is triggered, and partitions are reassigned to other members\n- Useful for detecting consumer deadlock or processing hang\n\nFor standalone consumers (via `assign()`):\n- No polling frequency requirement\n- No heartbeat or session timeout\n\n### Non-blocking Variants\n\n**Why there's no non-blocking variant:**\nKafka consumer design intentionally couples data fetching with background operations (heartbeats, rebalances, metadata refreshes) via `poll()`. This ensures:\n- Rebalance callbacks execute deterministically\n- Position advances atomically with data reception\n- No race conditions with group coordination\n\nFor async processing patterns, use `pause()` to buffer records into application queues.\n\n---\n\n## Offset Management\n\n### Understanding Offsets\n\n**Offset:** A monotonically increasing integer identifying a record's position in a partition.\n\n**Three offset concepts:**\n\n1. **Position**: Offset of next record to fetch\n   - Auto-advances when records returned from `poll()`\n   - Can be queried with `position(TopicPartition)`\n   - Can be set with `seek(TopicPartition, long)`\n\n2. **Committed Offset**: Last offset committed (durably stored in broker)\n   - Retrieved with `committed(Set<TopicPartition>)`\n   - Set via `commitSync()` or `commitAsync()`\n   - Used as restart point on consumer failure\n   - Stored in special `__consumer_offsets` topic\n\n3. **Lag**: Difference between latest offset and committed offset\n   - Indicates how far behind the consumer is\n   - Queryable via `currentLag(TopicPartition)`\n   - Important for monitoring\n\n**Offset invariant:** \"Committed offset should be the NEXT offset to read\"\n```\nConsumed offset 99 \u2192 commit offset 100 \u2192 seek to 100 \u2192 consume record at offset 100\n```\n\n### Automatic Offset Commits\n\n```java\n// Enabled by default with enable.auto.commit=true\n// Commits happen every auto.commit.interval.ms (default 5 seconds)\n\n// Triggered during poll() calls\nConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\n// If 5+ seconds since last commit, auto-commit fires BEFORE returning to user\n\n// At-least-once guarantee is harder (records may process multiple times on failure)\n// Exactly-once requires manual commit AFTER processing\n```\n\n**Auto-commit semantics:**\n- Commits the highest offset consumed so far in each partition\n- Happens BEFORE `poll()` returns records to user (important for timing)\n- No coordination with user processing (records may fail to process)\n- Not suitable for exactly-once semantic requirements\n\n**Risks with auto-commit:**\n- Records returned to user but not yet committed (process crashes)\n- Rebalance can revoke partition before implicit commit fires\n- Committed offset may not reflect actual user consumption\n\n### Manual Offset Commits (Synchronous)\n\n```java\n// Disable auto-commit first\nprops.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\n\ntry {\n    consumer.subscribe(Arrays.asList(\"orders\"));\n\n    while (running) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\n\n        for (ConsumerRecord<String, String> record : records) {\n            // Process record\n            process(record);  // May throw exception\n        }\n\n        // Commit only after successful processing\n        try {\n            consumer.commitSync();\n        } catch (CommitFailedException e) {\n            log.error(\"Commit failed, offset tracking may be lost\", e);\n        }\n    }\n} finally {\n    consumer.close();\n}\n```\n\n**Synchronous commit API:**\n\n```java\n// Commit current position in all assigned partitions\nconsumer.commitSync();\n\n// Commit with timeout\nconsumer.commitSync(Duration.ofSeconds(30));\n\n// Commit specific offsets (batch)\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\noffsets.put(tp0, new OffsetAndMetadata(100));\noffsets.put(tp1, new OffsetAndMetadata(200));\nconsumer.commitSync(offsets);\n\n// Commit with timeout\nconsumer.commitSync(offsets, Duration.ofSeconds(30));\n```\n\n**Behavioral semantics:**\n- Blocks until broker responds or timeout expires\n- Throws `CommitFailedException` if commit fails (not retriable from client perspective)\n- Throws `WakeupException` if consumer woken up during commit\n- Throws `InterruptException` if thread interrupted\n- For group-managed consumers, must be called before `onPartitionsRevoked()` completes\n\n### Manual Offset Commits (Asynchronous)\n\n```java\n// Asynchronous commit with callback\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        log.error(\"Async commit failed for offsets: \" + offsets, exception);\n        // Cannot retry automatically - must handle at application level\n    } else {\n        log.debug(\"Successfully committed offsets: \" + offsets);\n    }\n});\n\n// Continue processing without waiting for response\nConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\nfor (ConsumerRecord<String, String> record : records) {\n    process(record);\n}\n```\n\n**Asynchronous commit API:**\n\n```java\n// Commit without callback\nconsumer.commitAsync();\n\n// Commit with callback\nconsumer.commitAsync(new OffsetCommitCallback() {\n    @Override\n    public void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets,\n                          Exception exception) {\n        // Handle response\n    }\n});\n\n// Commit specific offsets with callback\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\noffsets.put(tp0, new OffsetAndMetadata(100));\nconsumer.commitAsync(offsets, callback);\n```\n\n**Important semantics:**\n- Returns immediately (non-blocking)\n- Callback invoked asynchronously when response received\n- Callback exceptions do NOT propagate to `poll()` caller\n- Failed async commits are NOT automatically retried (application must retry)\n- Multiple pending async commits can race; last one wins\n- Useful for fire-and-forget scenarios but requires careful error handling\n\n### Sync vs. Async Trade-offs\n\n| Aspect | Synchronous | Asynchronous |\n|--------|------------|----------------|\n| Blocking | Yes, until response | No, returns immediately |\n| Error handling | Immediate exception | Callback receives exception |\n| Retries | Framework may retry | Application must retry |\n| Performance | Lower throughput (blocking) | Higher throughput |\n| Reliability | Stronger (confirms before proceeding) | Weaker (fire-and-forget) |\n| Use case | Exactly-once semantics | High-throughput, at-least-once |\n\n### Querying Committed Offsets\n\n```java\n// Query committed offset for specific partitions\nSet<TopicPartition> partitions = consumer.assignment();\nMap<TopicPartition, OffsetAndMetadata> committed = consumer.committed(partitions);\n\nfor (TopicPartition tp : partitions) {\n    OffsetAndMetadata offsetMeta = committed.get(tp);\n    if (offsetMeta != null) {\n        System.out.println(tp + \": offset=\" + offsetMeta.offset() +\n                          \", metadata=\" + offsetMeta.metadata());\n    } else {\n        System.out.println(tp + \": no committed offset\");\n    }\n}\n\n// With timeout\nMap<TopicPartition, OffsetAndMetadata> committed =\n    consumer.committed(partitions, Duration.ofSeconds(10));\n```\n\n**Behavioral notes:**\n- Queries broker's `__consumer_offsets` topic (coordinator)\n- Returns `null` for partitions with no prior commit\n- Includes metadata string and leader epoch if set\n- May block if coordinator discovery needed\n\n---\n\n## Flow Control and Position Management\n\n### Seeking to Specific Offsets\n\n```java\nTopicPartition tp = new TopicPartition(\"orders\", 0);\n\n// Seek to specific offset\nconsumer.seek(tp, 100);  // Next poll() fetches from offset 100\n\n// Seek to beginning\nconsumer.seekToBeginning(Arrays.asList(tp));  // Seek to offset 0\n\n// Seek to end\nconsumer.seekToEnd(Arrays.asList(tp));  // Seek to latest+1 (next to be produced)\n\n// Seek to offset from metadata (with leader epoch)\nOffsetAndMetadata offsetMeta = new OffsetAndMetadata(100, 2);\nconsumer.seek(tp, offsetMeta);\n\n// Validate offset was set\nlong position = consumer.position(tp);\nassert position == 100;\n```\n\n**Key semantic:** `seekToEnd()` seeks to the offset AFTER the last message. Reading from this position returns empty until new messages arrive.\n\n### Timestamp-Based Seeking\n\n```java\n// Search for offset by timestamp\nMap<TopicPartition, Long> timestampMap = new HashMap<>();\ntimestampMap.put(tp0, System.currentTimeMillis() - 3600000);  // 1 hour ago\ntimestampMap.put(tp1, System.currentTimeMillis() - 7200000);  // 2 hours ago\n\nMap<TopicPartition, OffsetAndTimestamp> offsets =\n    consumer.offsetsForTimes(timestampMap);\n\nfor (Map.Entry<TopicPartition, OffsetAndTimestamp> entry : offsets.entrySet()) {\n    TopicPartition tp = entry.getKey();\n    OffsetAndTimestamp offsetAndTimestamp = entry.getValue();\n\n    if (offsetAndTimestamp != null) {\n        consumer.seek(tp, offsetAndTimestamp.offset());\n + \": No offset found for that timestamp\");\n    }\n}\n```\n\n**Timestamp seeking requires:**\n- Broker-side log index retention (default enabled)\n- Messages must have timestamp (enabled by default)\n- Timestamp semantics: CreateTime (message creation) or LogAppendTime (broker append)\n\n### Beginning and End Offsets\n\n```java\n// Get beginning offset (oldest available)\nMap<TopicPartition, Long> beginningOffsets =\n    consumer.beginningOffsets(consumer.assignment());\n\n// Get end offset (offset after latest)\nMap<TopicPartition, Long> endOffsets =\n    consumer.endOffsets(consumer.assignment());\n\n// Example: Log all unread messages\nfor (TopicPartition tp : consumer.assignment()) {\n    long start = beginningOffsets.get(tp);\n    long end = broker\n- End offset: Offset of next message to be produced\n- Available messages: `[beginningOffsets, endOffsets)`\n- Older messages may be deleted due to retention policy\n\n### Position Queries\n\n```java\n// Get current position (offset of next record to fetch)\nlong position = consumer.position(tp);\nSystem.out.println(\"Next fetch will read from offset: \" + position);\n\n// After processing records, position advances\nConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\nlong newPosition = consumer.position(tp);\nSystem.out.println(\"Position than highest offset consumed\n  - After consuming offset 99, position is 100\n  - Seek to 100 means next record fetched is offset 100\n\n### Pause and Resume\n\n```java\n// Pause specific partitions for monitoring consumer health and throughput\n\n---\n\n## Consumer Group Mechanics\n\n### Group Coordination Protocol\n\nKafka uses a group coordination protocol to manage distributed consumption:\n\n1. **Join**: Consumer joins group (initial `poll()` or `subscribe()` change)\n2. **Sync**: Partitions assigned to consumers based on assignor strategy\n3. **Assignment**: Broker sends assignment to each consumer\n4. **Stable**: Members consume in parallel\n5. **Rebalance**: Triggered by group change, partition count change, timeout\n6. **Revoke**: Partitions revoked from old owners\n7. **Assign**: Partitions assigned to new owners\n\n### Group ID\n\n```java\nprops.put(ConsumerConfig.GROUP_ID_CONFIG, \"order-processing-group\");\n```\n\n**Semantics:**\n- Identifies the consumer group for coordination\n- Multiple consumers with same group ID form a group\n- Partitions distributed across group members\n- Each partition assigned to exactly one member at any time\n- Required for group-managed subscription (via `subscribe()`)\n- Not used for manually assigned consumers (via `assign()`)\n\n### Instance ID (Static Membership)\n\n```java\nprops.put(ConsumerConfig.GROUP_ID_CONFIG, \"order-processing-group\");\nprops.put(ConsumerConfig.GROUP_INSTANCE_ID_CONFIG, \"consumer-1\");\n```\n\n**Benefits:**\n- Avoids rebalance during rolling restart\n- Consumer rejoins with same instance ID, keeps same partitions\n- Useful for avoiding processing interruptions\n- Requires `session.timeout.ms` to be large enough for restart\n\n### Group Metadata\n\n```java\n// Get current group membership metadata\nConsumerGroupMetadata groupMetadata = consumer.groupMetadata();\n\n// Use in transactional produce\n// (example with producer API)\nproducer.initTransactions();\nproducer.beginTransaction();\n\n// Process records and send results\nList<ProducerRecord<String, String>> results = process(records);\nfor (ProducerRecord<String, String> result : results) {\n    producer.send(result);\n}\n\n// Commit offsets transactionally\nproducer.sendOffsetsToTransaction(offsets, groupMetadata);\nproducer.commitTransaction();\n```\n\n**Use case:** Exactly-once semantics with transactional produce (coordinating consumer offsets with producer output)\n\n### Rebalancing Triggers\n\nRebalance occurs when:\n1. Consumer joins group (first `poll()` after `subscribe()`)\n2. Consumer leaves group (`close()`, `unsubscribe()`)\n3. Consumer times out (`max.poll.interval.ms` exceeded)\n4. Group membership changes (other members join/leave)\n5. Subscribed topic partitions change (count increases/decreases)\n6. User calls `enforceRebalance()`\n\n### Enforcing Rebalance\n\n```java\n// Request rebalance on next poll()\nconsumer.enforceRebalance();  // Default reason\n\n// With custom reason (for logging)\nconsumer.enforceRebalance(\"User requested rebalance due to external state change\");\n```\n\n**Use cases:**\n- Coordinating with external systems\n- Recovering from detected inconsistencies\n- Testing rebalance behavior\n\n---\n\n## Rebalancing and Callbacks\n\n### ConsumerRebalanceListener\n\nImplement to handle partition assignment changes.\n\n```java\npublic class MyRebalanceListener implements ConsumerRebalanceListener {\n    private KafkaConsumer<String, String> consumer;\n    private DatabaseConnection db;\n\n    public MyRebalanceListener(KafkaConsumer<String, String> consumer,\n                               DatabaseConnection db) {\n        this.consumer = consumer;\n        this.db = db;\n    }\n\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // Called when about to lose partitions\n        // Safe to commit offsets here\n        System.out.println(\"Revoked: \" + partitions);\n\n        try {\n            consumer.commitSync();  // Commit pending offsets\n        } catch (CommitFailedException e) {\n            log.error(\"Commit failed during revocation\", e);\n        }\n\n        // Save any state/close resources\n        for (TopicPartition tp : partitions) {\n            db.savePartitionState(tp);\n        }\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Called when partitions assigned after rebalance\n        // Safe to initialize positions here\n        System.out.println(\"Assigned: \" + partitions);\n\n        // Restore saved state\n        for (TopicPartition tp : partitions) {\n            long savedOffset = db.getSavedOffset(tp);\n            if (savedOffset >= 0) {\n                consumer.seek(tp, savedOffset);\n            } else {\n                consumer.seekToBeginning(Arrays.asList(tp));\n            }\n        }\n    }\n\n    @Override\n    public void onPartitionsLost(Collection<TopicPartition> partitions) {\n        // Called when partitions lost without revocation\n        // (e.g., session timeout)\n        System.out.println(\"Lost: \" + partitions);\n        // Do NOT commit offsets - they're already owned by other consumers\n        // Clean up resources only\n    }\n}\n\n// Register listener\nconsumer.subscribe(Arrays.asList(\"orders\"), new MyRebalanceListener(consumer, db));\n```\n\n### Callback Execution Details\n\n**Execution context:**\n- Callbacks execute on consumer thread during `poll()` call\n- NO special threading; they block the polling thread\n- Exceptions propagate to `poll()` caller\n- Safe to call consumer methods inside callbacks (e.g., `commitSync()`, `seek()`)\n\n**Callback order (normal rebalance):**\n1. `onPartitionsRevoked()` - revoke old partitions\n2. Behavior\n\n```java\nprops.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, \"true\");\nprops.put(ConsumerConfig.AUTO_COMMIT_INTERVAL_MS_CONFIG, \"5000\");  // 5 seconds\n```\n\n**Semantics:**\n- Auto-commit fires during `poll()` call\n- Commits highest offset consumed so far (in each partition)\n- Interval is minimum time between commits\n- Last commit may not fire if consumer closes abruptly\n- NOT triggered on `close()` alone\n\n**Example timeline:**\n```\nT=0:   poll() - no auto-commit (first poll)\nT=1:   poll() - no auto-commit (< 5 sec)\nT=5.1: poll() - auto-commit fires BEFORE returning records\nT=10:  close() - does NOT trigger auto-commit\n```\n\n### Offset Reset Policy\n\n```java\nprops.put(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, \"earliest\");  // or \"latest\", \"none\"\n```\n\n**\"earliest\":** Start from beginning of partition\n- Useful for initial data load\n- May read old data\n\n**\"latest\":** Start from end of partition (skip historical data)\n- Useful for tailing new events\n- Misses historical data\n\n**\"none\":** Throw `NoOffsetForPartitionException` if no offset found\n- Useful for explicit offset management\n- Fails fast if offset undefined\n\n### Session Timeout\n\n```java\nprops.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG, \"45000\");  // 45 seconds\n```\n\n**Semantics:**\n- Broker removes consumer from group if no heartbeat for this duration\n- Triggers rebalance\n- For manual assign (via `assign()`), not used\n\n**When exceeded:**\n- Broker considers consumer dead\n- Other members may begin processing its partitions\n- `onPartitionsLost()` may be called\n- Next `poll()` may throw exception\n\n### Max Poll Interval\n\n```java\nprops.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, \"300000\");  // 5 minutes\n```\n\n**Semantics:**\n- Broker requires `poll()` call at least once per this duration\n- Longer than `session.timeout.ms`\n- For group-managed consumers (via `subscribe()`)\n\n**When exceeded:**\n- Broker removes consumer from group\n- Triggers rebalance\n- Consumer loses partition ownership\n- Next `poll()` may throw `IllegalStateException`\n\n**Use case:** Detecting consumer hangs/deadlocks\n\n### Max Poll Records\n\n```java\nprops.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, \"500\");\n```\n\n**Semantics:**\n- `poll()` returns at most this many records across all partitions\n- Applies to a single `poll()` call, not cumulative\n- Useful for bounding processing batches\n- Affects `poll()` throughput vs latency tradeoff\n\n### Fetch Size Configuration\n\n```java\nprops.put(ConsumerConfig.FETCH_MIN_BYTES_CONFIG, \"1024\");  // 1 KB\nprops.put(ConsumerConfig.FETCH_MAX_WAIT_MS_CONFIG, \"500\");  // 500 ms\n```\n\n**Semantics:**\n- Broker waits up to `FETCH_MAX_WAIT_MS` to accumulate `FETCH_MIN_BYTES`\n- Larger `FETCH_MIN_BYTES` = fewer requests, higher latency\n- Smaller `FETCH_MIN_BYTES` = more requests, lower latency\n- Trade-off: throughput vs latency\n\n### Isolation Level\n\n```java\nprops.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG, \"read_committed\");  // or \"read_uncommitted\"\n```\n\n**\"read_uncommitted\" (default):**\n- Returns all messages (committed and in-progress transactions)\n- Highest throughput\n- May read aborted transactions\n\n**\"read_committed\":**\n- Filters out in-progress transaction messages\n- Stops at LSO (Last Stable Offset)\n- Ensures only committed messages consumed\n- Lower throughput due to filtering\n\n**Use case:** Reading committed transaction output.\n\n---\n\n## Usage ### What is KafkaConsumer?

- **Cluster membership**: Automatically maintains TCP connections to broker nodes

- **Consumer group coordination**: Works with the Kafka broker's group coordination protocol for distributed, fault-tolerant consumption
- **Offset management**: Tracks read position and supports automatic or manual offset commits
- **Load balancing**: Partitions are automatically distributed across group members
