## Key Findings:

**API Architecture:**
- The Kafka Connect REST API does **NOT use URL-based versioning** (no /v1 or /v2 path prefixes)
- Versioning is implemented through response entity schemas and backward-compatible data structures

**REST Resource Handlers:**
- **ConnectorsResource.java** - Main handler for `/connectors` endpoints (status, config, tasks)
- **RootResource.java** - Root endpoint and health checks
- **ConnectorPluginsResource.java** - Plugin management
- **InternalConnectResource.java** & **InternalClusterResource.java** - Inter-worker communication
- **ConnectRestServer.java** - Registers resources (no explicit version routing)

**Task Status Schema (v1/current):**
- Defined in **ConnectorStateInfo.java**
- Contains: name, connector (state, worker_id, trace, version), tasks list
- Version field uses custom **PluginInfo.NoVersionFilter** for conditional JSON inclusion

**Error Response Schema:**
- Defined in **ErrorMessage.java**
- Contains: error_code (int), message (String)
- Consistent across all versions

**Internal State Versioning:**
- **KafkaConfigBackingStore.java** defines schema versions:
  - **TARGET_STATE_V0** - Original schema with 'state' field
  - **TARGET_STATE_V1** - Extended with 'state.v2' field for new STOPPED state while maintaining backward compatibility

**Implicit v1/v2 Differences:**
- v1: Uses 'state' field for target state
- v2: Uses 'state.v2' field (internally) while maintaining backward compatibility
