## Summary

I've identified **three distinct approaches** to Prometheus metric registration:

### **Kubernetes Pattern: Global Registry with Wrapper**
- Uses `k8s.io/component-base/metrics` and `legacyregistry`
- Metrics defined as package-level variables
- Registration via `legacyregistry.MustRegister()` with `sync.Once` pattern
- Example files: `pkg/controller/certificates/clustertrustbundlepublisher/metrics.go`, `pkg/controller/daemon/metrics/metrics.go`

### **Grafana Pattern: Flexible Registerer Injection**
- Uses `prometheus.Registerer` interface as dependency
- Two sub-patterns: global metrics + separate `RegisterMetrics()` functions, OR factory functions that handle registration
- Example files: `apps/dashboard/pkg/migration/metrics.go`, `apps/plugins/pkg/app/metrics/metrics.go`, `apps/iam/pkg/reconcilers/metrics.go`

### **Prometheus Pattern: Factory Functions with Encapsulation**
- Metrics created inside factory functions that take registerer as parameter
- Handles nil registerers gracefully
- Proper error handling for failed registrations
- Example files: `rules/group.go` (NewGroupMetrics), `notifier/metric.go` (newAlertMetrics), `discovery/metrics.go` (NewManagerMetrics)

- **14 representative files** showing metric registration
