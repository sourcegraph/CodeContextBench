Clients had to parse and compare this.\n- **Why SSA Chose Server-Side:** A buggy or malicious client could corrupt the annotation, breaking for all future operations. Server-side state is enforceable and auditable.\n\n#### 4.2.3 Dual Patch Type Support (YAML and CBOR)\n**Tradeoff:** Wire format compatibility vs. efficiency.\n\n- **YAML (Default):** Human-readable, easy to debug, larger over the wire.\n- **CBOR (New):** Binary, more compact, more efficient for large objects.\n- **Feature Gate:** `ClientsAllowCBOR` and `ClientsPreferCBOR` control rollout, allowing gradual migration.\n- **Location:** `staging/src/k8s.io/client-go/util/apply/apply.go` (lines 38-41).\n\n#### 4.2.4 Force Flag Semantics\n**Tradeoff:** Safety vs. pragmatism.\n\n- **Without Force (default):** SSA is \"safe\" but can get stuck if fields are owned by other managers.\n- **With Force=true:** Forcefully takes ownership, breaking coexistence guarantees for those specific fields.\n- **Risk:** If an operator uses `Force: true` constantly, it defeats the purpose of managed fields.\n- **Recommendation:** Force should be rare; it signals \"I'm taking over this field and I know the risks.\"\n\n### 4.3 Performance Characteristics\n\n| Operation | Cost | Notes |\n|-----------|------|-------|\n| Conflict Detection | O(N) in field count | Compares field sets in managedFields; usually small (< 100 fields) |\n| Merge/Apply | O(N) in object size | Creates a new merged object in memory |\n| Field Tracking Serialization | O(N log N) | Encodes field paths into protobuf; scales with object complexity |\n| Optimistic Concurrency Check | O(1) | Single etcd version comparison; no locking |\n\n**Scaling Considerations:**\n- Objects with thousands of fields in managedFields (very rare in practice) may see degraded performance.\n- Rapid concurrent applies from many managers benefit from SSA's lock-free design.\n- SSA is faster than read-modify-write loops because it doesn't require the client to fetch the current state.\n\n---\n\n## 5. ## 1. Subsystem Overview and Purpose

### 1.1 What is Server-Side Apply?

Server-Side Apply (SSA) is a declarative, field-ownership-aware mechanism for updating Kubernetes objects. Unlike the traditional read-modify-write pattern (used by `PUT` and `PATCH` with optimistic concurrency), Server-Side Apply allows multiple actors—controllers, operators, and users—to safely co-manage different fields of the same object without overwriting each other's changes.

The core innovation is **managed fields**: the API server maintains metadata (`metadata.managedFields[]`) for each object, tracking which field manager owns each field and in which API version. This enables:
- **Conflict Detection:** When multiple managers try to claim the same field, SSA can detect and report the conflict
- **Field Takeover:** Conflicts can be forcefully resolved by setting `Force: true` in `ApplyOptions`
- **Partial Updates:** Managers can update only the fields they own, leaving other fields untouched
