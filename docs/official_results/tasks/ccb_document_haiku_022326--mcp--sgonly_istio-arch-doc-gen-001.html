# Istio Pilot Discovery Service Architecture

## Overview

Pilot's Discovery Service is the control plane component that serves Envoy proxies with xDS (Envoy Discovery Service) configuration. It translates high-level Istio networking resources (VirtualServices, DestinationRules, Gateways) into Envoy-compatible configurations (Listeners, Routes, Clusters, Endpoints) and pushes them to connected proxies via gRPC.

The discovery service architecture comprises four tightly integrated subsystems:

1. **DiscoveryServer** — The xDS gRPC server coordinating all communication
2. **ADS (Aggregated Discovery Service)** — The stream handler managing individual proxy connections
3. **PushContext** — An immutable configuration snapshot regenerated on each push
4. **ServiceRegistry** — The service discovery layer feeding endpoints and services

---

## 1. DiscoveryServer — The xDS gRPC Server

**File:** `pilot/pkg/xds/discovery.go`

### Responsibilities

The `DiscoveryServer` is Pilot's core gRPC implementation for Envoy's xDS APIs. It is responsible for:

- **Accepting gRPC streams** from Envoy proxies via the `StreamAggregatedResources()` RPC
- **Tracking active connections** in the `adsClients` map, keyed by connection ID
- **Debouncing configuration changes** to avoid thundering herd pushes
- **Limiting concurrency** of active push operations to prevent CPU/memory overload
- **Managing push queues** that serialize updates to individual proxies
- **Rate limiting incoming requests** to prevent resource exhaustion
- **Caching xDS resources** to avoid re-computation for unchanged configurations
- **Coordinating updates** between configuration watchers (Istio CRDs, K8s services) and proxy pushes
