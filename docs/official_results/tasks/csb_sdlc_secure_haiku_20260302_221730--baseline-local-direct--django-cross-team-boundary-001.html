## Incident Report

**Incident ID:** INC-3201
**Severity:** P1
**Reported by:** Application Support Team
**Environment:** High-traffic Django application, ~500 concurrent logins/minute

Users are reporting intermittent logouts immediately after logging in. The issue is rare under normal load (estimated 0.1% of logins) but increases proportionally with traffic. During peak events, the failure rate spiked to ~2% of logins.

**Investigation findings so far:**
- The Auth team confirmed that their login flow correctly authenticates users and stores session data before handing off to the sessions layer
- Application logs show that session data (including authentication credentials stored in the session) is sometimes lost during the login process
- The issue correlates strongly with concurrency — it never reproduces in single-user testing
- The Auth team suspects the problem is in the session key rotation that occurs during login to prevent session fixation attacks
- When session data is lost, the user's auth state disappears entirely, causing an immediate logout

**Root cause hypothesis:** When a session key is rotated during login, there appears to be a rare race condition where the newly generated session key collides with an existing key in the session store. Under high concurrency, this collision probability increases. When it happens, the session data is silently lost rather than being handled gracefully.

## Task

Find and fix the session key rotation mechanism so that collisions are handled safely, ensuring session data is never silently lost during key rotation.

## Summary

I've successfully fixed the intermittent user logout issue under high concurrency. Here's what was done:

### Problem
Under high-traffic conditions (~500 concurrent logins/minute), users experienced rare logouts immediately after login (0.1% to 2% failure rate). The issue was caused by a race condition in the session key rotation mechanism during login, where newly generated session keys could collide with existing keys, causing session data to be silently lost.

### Root Cause
The `cycle_key()` method in `/workspace/django/contrib/sessions/backends/base.py` had a critical race condition:
1. It created a new session with a new key
2. Restored the session data to the cache
3. Deleted the old key
4. **But never explicitly saved the data with the new key to persistent storage**

This created a window where the data could be lost if an error occurred or another request interfered before the middleware called `save()`.

### Solution
I added explicit `save()` calls in both `cycle_key()` and `acycle_key()` methods to ensure session data is immediately persisted with the new key **before** the old key is deleted. This eliminates the race condition.

### Changes Made
**File Modified:** `django/contrib/sessions/backends/base.py`

# Session Key Rotation Collision Fix

## Problem
Under high concurrency, users experienced intermittent logouts immediately after login (estimated 0.1% of logins, up to 2% during peak events). The root cause was a race condition in the session key rotation mechanism (cycle_key()) where:

1. When a new session key was rotated during login, there was a rare collision with existing keys
2. Under high concurrency, collision probability increased
3. When collisions occurred, session data was silently lost instead of being handled gracefully

## Root Cause Analysis
The `cycle_key()` method in `/workspace/django/contrib/sessions/backends/base.py` had a race condition:

```python
def cycle_key(self):
    data = self._session           # Load data (in cache)
    key = self.session_key         # Get old key
    self.create()                  # Create new session with new key
    self._session_cache = data     # Restore data to cache (but NOT persisted!)
    if key:                        # Delete old key
        self.delete(key)
```

**The Issue**: After `create()` creates a new empty session and sets `modified=True`, the session data is only restored to the cache (`_session_cache = data`), not persisted to storage. If any error or race condition occurs before the middleware calls `save()`, the data is lost.
