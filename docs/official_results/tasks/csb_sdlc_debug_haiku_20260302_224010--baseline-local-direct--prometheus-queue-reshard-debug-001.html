## Key Findings:

**Root Cause:** The `prometheus_remote_storage_samples_pending` metric is unconditionally reset to 0 during shard initialization (line 1241 in `queue_manager.go`), which can mask samples that were dropped or lost during the hard shutdown phase of resharding.

**The Mechanism:**
1. When resharding is triggered (e.g., due to target discovery changes), old shards attempt to gracefully flush their queues
2. If flushing exceeds the timeout, a hard shutdown occurs that cancels the context
3. Partial batches in queues may be abandoned without being properly sent
4. The subsequent call to `start()` resets `pendingSamples.Set(0)`, overwriting any Sub() operations that recorded dropped samples
5. This creates a metric inconsistency where pending samples appear to be 0 despite data loss

**Why It's Intermittent:**
- Only manifests when hard shutdown timeout occurs
- Depends on load, network latency, and specific target discovery patterns
- More likely with slow or unresponsive remote endpoints

**Affected Components:**
- `storage/remote/queue_manager.go` (queue_manager, reshardLoop, start/stop functions)
- Prometheus Gauge metrics (Set() vs Sub() operations)
- Queue flushing mechanism
