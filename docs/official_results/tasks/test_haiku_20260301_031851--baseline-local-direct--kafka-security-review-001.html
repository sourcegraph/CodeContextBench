Looking at the code, I've identified critical security defects:

## 1. **ScramSaslServer.java (Line 135)** - Inverted Iteration Count Check

The condition is inverted. It checks `if (scramCredential.iterations() > mechanism.minIterations())` and throws an exception saying iterations are "less than" the minimum. This allows weak credentials with fewer iterations than required.

## 2. **SaslServerAuthenticator.java (Line 698)** - Session Expiration Never Calculated

The session expiration calculation is commented out, so `sessionExpirationTimeNanos` remains `null`. This means authenticated sessions never expire, allowing indefinite access.

## 3. **CredentialCache.java (Lines 36-38)** - Type Check Removed

The type validation has been removed, allowing credentials of one type to be retrieved as a different type. This enables credential type confusion attacks.

## 4. **AclAuthorizer.scala (Line 520)** - Inverted Authorization Logic

The authorization logic `(denyAclExists(acls) && allowAclExists(acls))` is backwards. It allows access when BOTH deny and allow rules exist, instead of allowing when NO deny rule exists AND an allow rule exists.
