Without the `resource.` prefix, `local.config` could be interpreted as a reference to a local value (`local.config`) rather than a resource (`resource \"local\" \"config\"`).\n\n#### Migration Instructions\n\n1. **Identify affected resources** - Search your codebase for resource types matching the reserved keywords list above\n2. **Find `moved` blocks** referencing these resources - Search for `moved` blocks with `from` or `to` attributes\n3. **Add `resource.` prefix** - Update the references to use `resource.<type>.<name>` format\n4. **Validate configuration** - Run `terraform validate` to ensure all references are correct\n\n#### Step-by-Step Example\n\nIf you have a custom provider with a resource type `template`:\n\n**Before migration:**\n```terraform\nresource \"template\" \"example\" {\n  content = \"...\"\n}\n\nmoved {\n  from = template.example\n  to   = template.v2\n}\n```\n\n**After migration:**\n```terraform\nresource \"template\" \"example\" {\n  content = \"...\"\n}\n\nmoved {\n  from = resource.template.example\n  to   = resource.template.v2\n}\n```\n\n#### Common Pitfalls\n\n- **Forgetting module references** - If using `module.X.local` in a moved block, you don't need to prefix; only resource type names matching keywords need the prefix\n- **Partial updates** - Update both `from` and `to` addresses if both reference the problematic resource type\n- **Testing after migration** - Always run `terraform plan` after updating moved blocks to verify the references are correct\n\n---\n\n## Sensitive Value Handling\n\n### What Changed\n\nIn v1.10.0, Terraform improved how sensitive marks are propagated through conditional expressions and for expressions. Previously, marks could be **lost** in certain complex expressions. ## Overview

Terraform v1.10.0 is a minor release in the stable Terraform v1.0 series and honors the [Terraform v1.0 Compatibility Promises](https://developer.hashicorp.com/terraform/language/v1-compatibility-promises).
