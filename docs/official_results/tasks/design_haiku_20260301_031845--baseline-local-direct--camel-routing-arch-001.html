## Key Findings

I've traced the complete message flow through Apache Camel's architecture:

### **Component→Endpoint→Consumer/Producer Hierarchy**
- **Component** acts as a factory for Endpoints from URI strings
- **Endpoint** creates Protocol-specific Consumer (receive) and Producer (send) implementations
- **Consumer/Producer** integrate with external systems while implementing the Processor interface

### **DSL Model to Runtime (Reifier Bridge)**
The RouteReifier converts RouteDefinition (Java/XML DSL) into executable runtime by:
1. Resolving the source Endpoint from URI
2. Creating a DefaultRoute instance
3. For each EIP definition (outputs), creating ProcessorReifiers
4. Each reifier creates a Processor and wraps it in a **Channel**

### **Channel as Interceptor Pattern**
Each processor is wrapped in a Channel that:
- Applies interceptor strategies (tracing, debugging, management)
- Wraps error handling (retry, redelivery logic)
- Controls message flow and MEP (Message Exchange Pattern)
- Manages message history and routing metadata

### **Pipeline for Sequential Processing**
All event-driven processors are assembled into a RoutePipeline that:
- Chains multiple processors sequentially
- Reuses the same Exchange object between steps
- Handles OUT→IN conversion between processors
- Uses AsyncProcessor for reactive/event-driven execution

### **Message Flow Path**
External Message → Consumer → RoutePipeline → Channel→Processor chain → Producer → Destination
