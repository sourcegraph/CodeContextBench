Runs admission (deletion admission controllers), marks for deletion or immediate deletion.\n- **`watch.go`**: Handles GET with ?watch=true. Streams events from the watch cache.\n\nEach handler receives a **`RequestScope`** (`handlers/rest.go`), which bundles:\n- Serialization context (codecs, media types).\n- Resource metadata (GVR, GVK, namespaced vs. cluster-scoped).\n- Admission, authorization, field managers.\n- Storage backend (via `rest.Storage` interface).\n\n### 2.5 Admission Control (`pkg/admission/`)\n\nThe admission system enforces policies before objects are persisted. It is a **chain** of plugins, each handling specific operations:\n\n- **`MutationInterface`**: Modifies objects (e.g., setting defaults, injecting sidecars).\n- **`ValidationInterface`**: Rejects invalid objects (e.g., enforcing pod security policies).\n\n**Key files:**\n- `interfaces.go`: Defines `Attributes` (request info for policy decisions), `Interface`, and `ObjectInterfaces`.\n- `chain.go`: Chains plugins; fails immediately on first error.\n- `handler.go`: Base handler for plugins with readiness checks.\n- `plugin/webhook/`: Validating and mutating webhooks (external policy enforcement).\n- `plugin/policy/`: CEL-based validation and mutation policies.\n\n**Critical property**: Deserialization into objects happens **only after authorization** and **before admission control**. This ensures that large payloads are not deserialized for unauthenticated users.\n\n### 2.6 Authentication (`pkg/authentication/`)\n\nIdentifies the user making the request. The system is **pluggable** and composed of multiple authenticators:\n\n- **`authenticator/`**: Interfaces for request authenticators.\n- **`request/`**: Implementations (client cert, bearer token, OIDC, etc.).\n- **`token/`**: Token validation (ServiceAccount, OIDC providers).\n- **`user/`**: User and group information structures.\n\nThe authenticator chain stops at the **first successful authenticator**; if all fail, the request is unauthenticated.\n\n### 2.7 Authorization (`pkg/authorization/`)\n\nDetermines if an authenticated user is permitted to perform the requested action. Pluggable:\n\n- **`authorizer/`**: Core interfaces (`Authorizer`, `RuleResolver`).\n- **`union/`**: Chains multiple authorizers; uses **\"majority wins\"** logic (deny overrides allow/no-opinion).\n- **`path/`**: Path-based rules (e.g., exempt `/healthz` from authorization).\n- **`cel/`**: CEL-based policy engine for custom authorization rules.\n\n### 2.8 API Installation and Routing (`pkg/endpoints/installer.go`)\n\nThe **`APIInstaller`** dynamically registers REST handlers for resources. For each API group version:\n\n1. **Retrieve resource metadata** from `APIGroupInfo.VersionedResourcesStorageMap[resourceName]`.\n2. **Extract storage backend** (`rest.Storage` interface) for each resource.\n3. **Enumerate actions** (GET, POST, PUT, PATCH, DELETE, WATCH) based on storage capabilities.\n4. **Register webservice routes** with the gorestful container, one per action.\n5. **Install sub-resources** (e.g., `/pods/{name}/logs`, `/pods/{name}/status`).\n6. **Generate discovery resources** (list of resources, verbs, short names, categories).\n\n**Storage interface** (`pkg/registry/rest/`):\nThe `Storage` type must implement a subset of CRUD interfaces:\n- `Getter`: Fetch a single object.\n- `Lister`: Fetch multiple objects matching criteria.\n- `Creator`: Create new objects.\n- `Updater`: Modify existing objects.\n- `Deleter`: Remove objects.\n- `Watcher`: Stream object changes.\n- `TableConvertor`: Convert objects to table format.\n\nMost builtin resources use **`genericregistry.Store`**, which wraps an etcd client and applies a `Strategy` for validation and mutation.\n\n### 2.9 Watch Cache (`pkg/storage/cacher/`)\n\nTo prevent etcd from being overwhelmed by watch requests from hundreds of controllers, the apiserver maintains an in-memory watch cache:\n\n- **Initialization**: On first list or watch request for a resource, performs a full LIST and captures the resource version, then starts a WATCH from that point.\n- **In-memory buffer**: Stores recent events (typically the last few hundred per resource).\n- **Consistent reads**: For watch requests with a specific `resourceVersion`, ensures the cache has caught up to that version before serving.\n- **Bookmarks**: Periodically emits bookmark events to prevent the cache's version from falling too far behind.\n- **Fallback**: If a watch request asks for a version older than the cache buffer, the request falls through to etcd for a fresh watch.\n\n### 2.10 Storage Backend (`pkg/storage/etcd3/`)\n\nThe **etcd3** storage backend handles all persistence:\n\n- **Key-value encoding**: Marshals objects into etcd key-value pairs, applying optional encryption and compression.\n- **Resource versioning**: Maps etcd's `mod_revision` to Kubernetes `resourceVersion`.\n- **Transaction support**: Uses etcd transactions for multi-object operations.\n- **Lease-based TTL**: Leverages etcd leases for object expiration.\n\n---\n\n## 3. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Request Timeout and Long-Running Requests\n\n**Tradeoff**: Ensure API server responsiveness without killing legitimate long-running operations.\n\n- **Default**: Non-long-running requests timeout after `RequestTimeout` (e.g., 30s).\n- **Long-running exemption**: Requests matching `LongRunningRequestCheck()` (e.g., watch, exec, portforward) are exempt.\n- **Watch cache lag**: If a watch request specifies a `resourceVersion` the cache hasn't reached yet, it blocks (with an upper timeout of `requestTimeoutUpperBound = 34s`). If the cache falls too far behind, etcd is queried directly.\n\n**Failure consequence**: If a single slow list operation takes too long to reach a requested `resourceVersion`, subsequent watches may timeout, forcing clients to reconnect.\n\n### 4.2 Optimistic Concurrency and Conflict Resolution\n\n**Tradeoff**: Avoid locking (unscalable) but ensure consistency.\n\n- **Mechanism**: Every object has a `resourceVersion` field mapped to etcd's `mod_revision`.\n- **Update conflict**: If a client PUTs or PATCHes with a stale `resourceVersion`, etcd rejects the transaction, and the API server returns **409 Conflict**.\n- **Client responsibility**: Re-read the object, merge changes, and retry with the new `resourceVersion`.\n\n**Failure consequence**: High-contention resources (e.g., `ConfigMap` accessed by many controllers) may experience high conflict rates, forcing clients into exponential backoff loops.\n\n### 4.3 Admission Webhook Timeout and Cascading Failures\n\n**Problem**: An external admission webhook may be slow or hung.\n\n**Design**:\n- Each webhook has a timeout (default ~30s) configured in `ValidatingWebhookConfiguration`.\n- If a webhook times out, the `failurePolicy` determines the outcome (Fail or Ignore).\n- No cascading failure protection: If all webhooks for a resource hang, the API server blocks all writes to that resource.\n\n**Mitigation**: Webhook timeouts and failure policies must be carefully tuned; excessively permissive settings (failurePolicy: Ignore) reduce security.\n\n### 4.4 Watch Cache Consistency Windows\n\n**Problem**: The watch cache is eventually consistent, not immediately consistent.\n\n- **Scenario**: A controller updates object A at time T1, then immediately reads object B which has a reference to A.\n- **Risk**: Object B might be served from the cache before the watch stream has notified it of A's update.\n\n**Mitigation**: The apiserver uses \"consistent reads\" for most operations. When a write completes, the etcd revision is known; subsequent reads for objects at that revision or later wait for the cache to catch up (or fall through to etcd).\n\n### 4.5 Memory Pressure and List Operations\n\n**Tradeoff**: In-memory watch cache improves scalability but consumes memory.\n\n- **Problem**: A LIST operation requesting 10,000 objects without pagination must deserialize and hold all in memory.\n- **Mitigations**:\n  - `MaxRequestBodyBytes` limit on writes prevents unbounded request body sizes.\n  - Pagination via `limit` and `continue` tokens forces paginated lists (though clients can disable this).\n  - Watch cache size is bounded but unbounded growth is possible during list operations.\n\n**Failure consequence**: A runaway list operation (e.g., without limit) can exhaust memory, triggering OOM killer.\n\n### 4.6 Authorization Denial Not Logged in Audit\n\n**Tradeoff**: Log all requests vs. log volume.\n\n- **Design**: If authorization denies a request, the request is not logged to the audit backend (only to the logs and metrics).\n- **Consequence**: Attacks probing for accessible resources are not recorded in the audit trail, only in server logs.\n\n### 4.7 Graceful Shutdown with Complex Drain Sequencing\n\n**Problem**: API server must cleanly shut down without dropping in-flight requests or leaving data inconsistent.\n\n**Design** (from `pkg/server/genericapiserver.go:RunWithContext()`):\n\n1. **Pre-shutdown hooks** run first (e.g., lease reconcilers deregistering themselves).\n2. **Non-long-running requests wait group**: After hooks complete, the server stops accepting new requests and waits for in-flight requests to complete (bounded by `RequestTimeout`).\n3. **Watch request wait group**: Separately waits for watch streams to drain (bounded by `ShutdownWatchTerminationGracePeriod`).\n4. **Audit backend shutdown**: After all requests drain, audit events are flushed and backend is shut down.\n5. **HTTP server shutdown**: Finally, the HTTP server is shut down.\n\n**Failure consequence**: If a single in-flight request hangs, the entire graceful shutdown sequence stalls, forcing a hard timeout.\n\n### 4.8 Admission Control Ordering and Side Effects\n\n**Problem**: Admission plugins may have side effects (e.g., webhooks calling external systems).\n\n**Design**:\n- Mutating plugins run first, then validating plugins.\n- Within each phase, plugins run in the order specified in their configuration.\n- No rollback mechanism: If a later plugin rejects an object, earlier plugins' side effects are not undone.\n\n**Consequence**: Admission plugin ordering is critical and fragile; misconfiguration can cause half-applied policies.\n\n---\n\n## 5. Extension Points, Customization Hooks, and Associated Risks\n\n### 5.1 Filter Chain Customization\n\n**Hook**: `Config.BuildHandlerChainFunc`\n\n**Mechanism**: Before the server starts, callers can provide a custom function to wrap the API handler with additional filters.\n\n```go\ntype Config struct {\n    BuildHandlerChainFunc func(apiHandler http.Handler, c *Config) (secure http.Handler)\n}\n```\n\n**Example use cases**:\n- Add custom authentication schemes (e.g., OAuth2 proxy).\n- Inject custom request metadata (e.g., tenant ID from headers).\n- Implement custom rate limiting (though Priority/Fairness is preferred).\n\n**Risk**: Custom filters run before admission control and can leak sensitive data. Ensure custom filters do not log request bodies or headers.\n\n### 5.2 Admission Control Plugins\n\n**Hook**: `admission.Interface` implementations registered via `admission.Plugins` registry.\n\n**Mechanism**: Plugins are loaded dynamically, configured via `--enable-admission-plugins` and `--disable-admission-plugins` flags, and can read external configuration files.\n\n**Built-in plugins** (from `pkg/server/plugins.go`):\n- **Namespace lifecycle**: Prevents deletion of non-empty namespaces.\n- **Validating webhooks**: Call external services to validate objects.\n- **Mutating webhooks**: Call external services to mutate objects.\n- **Validating admission policies**: CEL-based policies (no external calls).\n- **Mutating admission policies**: CEL-based mutations.\n\n**Custom plugin example**:\n```go\ntype MyPlugin struct{}\nfunc (p *MyPlugin) Admit(ctx context.Context, a admission.Attributes, o admission.ObjectInterfaces) error {\n    // Custom validation or mutation\n    return nil\n}\nfunc (p *MyPlugin) Validate(ctx context.Context, a admission.Attributes, o admission.ObjectInterfaces) error {\n    // Custom validation\n    return nil\n}\n```\n\n**Risks**:\n- **Denial of service**: A slow or hanging webhook blocks all writes.\n- **Inconsistent state**: A plugin rejecting a DELETE may leave objects in limbo.\n- **Privilege escalation**: A plugin with write access to RBAC roles can elevate privileges.\n\n### 5.3 Authenticator Chain\n\n**Hook**: `Config.Authentication.Authenticator`\n\n**Mechanism**: The authenticator is a `authenticator.Request` implementation. Multiple authenticators can be chained via `union.New()`.\n\n**Built-in authenticators**:\n- **Client certificate** (x509 subject \u2192 user).\n- **Bearer token** (ServiceAccount token validation).\n- **OIDC** (external OIDC provider).\n- **Request header** (proxy trusts X-Remote-User headers).\n\n**Custom authenticator example**:\n```go\ntype MyAuth struct{}\nfunc (a *MyAuth) AuthenticateRequest(req *http.Request) (*authenticator.Response, bool, error) {\n    // Extract user from custom header\n    user := &user.DefaultInfo{Name: \"custom-user\"}\n    return &authenticator.Response{User: user}, true, nil\n}\n```\n\n**Risks**:\n- **Privilege escalation**: A custom authenticator incorrectly parsing tokens may grant unintended access.\n- **Denial of service**: Slow external authentication (e.g., LDAP) can block the API server.\n\n### 5.4 Authorizer Chain\n\n**Hook**: `Config.Authorization.Authorizer`\n\n**Mechanism**: Multiple authorizers are chained with \"majority wins\" logic (deny overrides allow/no-opinion).\n\n**Built-in authorizers**:\n- **RBAC** (role-based access control, most common).\n- **Node** (kubelet-specific authorization).\n- **Webhook** (call external authz service).\n- **CEL-based** (policy engine).\n\n**Custom authorizer example**:\n```go\ntype MyAuthz struct{}\nfunc (a *MyAuthz) Authorize(ctx context.Context, attributes authorizer.Attributes) (authorizer.Decision, string, error) {\n    // Custom allow/deny logic\n    return authorizer.DecisionAllow, \"allowed\", nil\n}\n```\n\n**Risks**:\n- **Incorrect deny logic**: A custom authorizer may accidentally deny legitimate requests.\n- **Cascade of denials**: Multiple authorizers in the chain may cause unexpected cumulative denials.\n\n### 5.5 Storage Backend Customization\n\n**Hook**: `Config.RESTOptionsGetter`\n\n**Mechanism**: When an API group is installed, the RESTOptionsGetter factory creates storage backends for each resource. Custom implementations can use alternative backends (e.g., CRDs stored in a separate database).\n\n**Example**:\n```go\ntype Config struct {\n    RESTOptionsGetter genericregistry.RESTOptionsGetter\n}\n```\n\n**Use case**: Store certain resources in a different etcd cluster or database.\n\n**Risk**: Custom storage backends must implement the `rest.Storage` interface correctly, including support for watch streams, conflict detection, and consistent reads. Incorrect implementations can cause data loss or inconsistency.\n\n### 5.6 API Group Installation Hooks\n\n**Hook**: `GenericAPIServer.InstallAPIGroup()` and `GenericAPIServer.InstallLegacyAPIGroup()`\n\n**Mechanism**: After instantiating a `GenericAPIServer`, callers invoke these methods to register API resources.\n\n**Example** (from kube-apiserver):\n```go\n// Install core v1 API group\nvar coreGroupInfo = genericapiserver.APIGroupInfo{\n    PrioritizedVersions: []schema.GroupVersion{{Group: \"\", Version: \"v1\"}},\n    VersionedResourcesStorageMap: map[string]map[string]rest.Storage{\n        \"v1\": {\n            \"pods\": podStorage,\n            \"services\": serviceStorage,\n            // ...\n        },\n    },\n    Scheme: legacyscheme.Scheme,\n    // ...\n}\ns.InstallAPIGroup(&coreGroupInfo)\n```\n\n**Risk**: Incorrect API group info (e.g., missing resources, wrong versions) silently results in 404s for those resources. No validation at install time.\n\n### 5.7 Post-Start Hooks\n\n**Hook**: `Config.PostStartHooks` and `GenericAPIServer.AddPostStartHook()`\n\n**Mechanism**: Hooks registered here run after the server begins listening but before the API is fully operational.\n\n**Example use cases**:\n- Initialize external system integrations (e.g., cloud provider sync).\n- Perform health checks (e.g., verify etcd connectivity).\n- Start background reconcilers.\n\n**Risk**: A post-start hook that fails causes the server to mark itself as not ready, potentially causing restarts and cascading failures in dependent systems.\n\n### 5.8 Pre-Shutdown Hooks\n\n**Hook**: `GenericAPIServer.AddPreShutdownHook()`\n\n**Mechanism**: Hooks registered here run when shutdown is initiated but before request draining.\n\n**Example use cases**:\n- Deregister from service discovery (e.g., Kubernetes endpoints).\n- Flush pending state (e.g., lease reconciliation).\n\n**Risk**: A pre-shutdown hook that hangs causes the entire graceful shutdown to stall, forcing a hard timeout and unclean exit.\n\n---\n\n## 6. Operational Implications: Performance, Reliability, and Maintainability\n\n### 6.1 Watch Cache Scalability\n\n**Impact**: The watch cache in-memory storage is critical for scalability. Controllers use watches to monitor resource changes; without the cache, every controller would directly stress etcd.\n\n**Operational concern**: Watch cache size grows with the number of objects in a resource type. For a cluster with 10,000 pods, the apiserver's watch cache for pods consumes significant memory. Monitor memory usage and ensure adequate heap allocation.\n\n**Tuning**: The `--list-watch-cache-size` flag controls the number of events cached per resource (default ~100). Increase for high-churn resources.\n\n### 6.2 Admission Webhook Timeouts and Latency\n\n**Impact**: Admission webhooks are in the critical path for all writes. A single slow webhook slows all writes cluster-wide.\n\n**Operational concern**: Monitor webhook latency and timeout rates. Misconfigured webhooks (e.g., pointing to a non-existent endpoint) can cause cascading failures.\n\n**Tuning**:\n- Set webhook timeouts conservatively (e.g., 5s vs. the default 30s).\n- Use `failurePolicy: Fail` to catch webhook failures, not `Ignore`.\n- Implement retry logic in webhooks themselves, not relying on the apiserver's retry (there is none).\n\n### 6.3 Priority and Fairness (Flow Control)\n\n**Impact**: The flow control system prevents one \"bursty\" client from starving others. Misconfiguration can cause all requests to be rejected.\n\n**Operational concern**: Default flow control is configured for typical clusters, but atypical workloads (e.g., bulk list operations) may trigger rate limiting.\n\n**Tuning**: The `--flow-control-config` flag points to a `FlowControlConfiguration` YAML file. Adjust flow schemas and priority levels based on actual traffic patterns.\n\n### 6.4 Audit Logging and I/O\n\n**Impact**: Audit logging is always enabled (if an audit backend is configured). High-frequency, verbose audit policies can overwhelm the audit backend.\n\n**Operational concern**: If the audit backend is slow (e.g., file I/O on a slow disk), the apiserver's response latency degrades. The audit backend runs asynchronously but is backpressure-limited.\n\n**Tuning**: Use a batch audit backend (not the simple file backend) to buffer events. Monitor audit backend latency via metrics.\n\n### 6.5 Resource Version Staleness and Etcd Lag\n\n**Impact**: The apiserver's view of the cluster state lags behind etcd due to watch cache propagation delays.\n\n**Operational concern**: Clients that require strongly consistent reads (e.g., kubectl apply) must explicitly request `resourceVersion=0` to bypass the cache. Default is eventually consistent.\n\n**Mitigation**: The apiserver uses \"consistent reads\" for write operations (updates and deletes), ensuring the cache is up-to-date before responding.\n\n### 6.6 Graceful Shutdown Drain Time\n\n**Impact**: During shutdown, the apiserver must drain in-flight requests without dropping them. The drain time is bounded by `RequestTimeout` (usually ~30s).\n\n**Operational concern**: If a single request takes longer than `RequestTimeout`, the graceful shutdown stalls, eventually timing out and forcing a hard exit.\n\n**Tuning**: Set `--shutdown-delay-duration` to allow controllers to discover the shutdown and reconnect to other apiservers (in HA deployments).\n\n### 6.7 Error Handling and Observability\n\n**Impact**: The apiserver logs errors to stdout/stderr and emits metrics. Debugging API server issues requires logs and metrics analysis.\n\n**Operational concern**: Large request bodies in error messages can bloat logs. The apiserver truncates request bodies in logs to prevent log explosion.\n\n**Mitigation**: Use structured logging (JSON) and aggregation tools (e.g., ELK) to search and correlate logs.\n\n---\n\n## 7. # Kubernetes API Server Library: Architecture and Extension Guide

## 1. Subsystem Purpose and Boundaries

The **apiserver library** (`staging/src/k8s.io/apiserver/pkg/`) is a foundational framework for building RESTful API servers with enterprise-grade operational features. It is not a standalone server but rather a pluggable, composable subsystem designed to:

- **Accept HTTP requests** through a chain of filters that handle authentication, authorization, admission control, and rate limiting.
- **Dispatch requests** to resource handlers that implement CRUD operations (Create, Read, Update, Delete, Watch).
- **Persist state** to etcd through pluggable storage backends with in-memory caching for watch streams.
- **Serve discovery endpoints** that describe available APIs and OpenAPI specifications.
- **Manage lifecycle events** through hooks (post-start, pre-shutdown) and graceful termination.

### Upstream Dependencies
- **etcd**: The persistent storage backend (via `staging/src/k8s.io/apiserver/pkg/storage/etcd3/`).
- **Runtime & Apimachinery**: Type marshaling, conversion, and serialization (`k8s.io/apimachinery`).
- **Component-base**: Shared utilities for logging, metrics, and signal handling.

### Downstream Dependencies
- **kube-apiserver** (`cmd/kube-apiserver/app/`): Instantiates three servers in a delegation chain (aggregator, core, extensions).
- **Custom API servers**: Third-party APIs can embed and customize this library.
- **Controllers and clients**: Consume the APIs exposed by servers using this library.
