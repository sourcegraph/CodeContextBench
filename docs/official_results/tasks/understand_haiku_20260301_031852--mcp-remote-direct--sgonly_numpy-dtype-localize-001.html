I've thoroughly traced the data flow when pandas nullable integers flow into scikit-learn's preprocessing and identified the root cause:

### **Root Cause Location**
- **File**: `pandas/core/arrays/_utils.py`
- **Function**: `to_numpy_dtype_inference()`
- **Lines**: 41-42

```python
if arr.dtype.kind in "iu":
    result_dtype = np.dtype(np.float64)  # <-- PROBLEM: Converts integers to float64
```

### **The Issue**
When a pandas DataFrame with nullable integer types (Int64Dtype, etc.) is passed to scikit-learn's `StandardScaler` or `MinMaxScaler`:

1. **scikit-learn's `validate_data()`** (sklearn/utils/validation.py:961-968) calls `check_array()`
2. **`_pandas_dtype_needs_early_conversion()`** (sklearn/utils/validation.py:681-717) identifies nullable integers as needing conversion
3. **`check_array()`** (sklearn/utils/validation.py:935-942) calls `.astype()` on the DataFrame
4. **Pandas' masked array `__array__()` method** (pandas/core/arrays/masked.py:694-716) delegates to `to_numpy()`
5. **`to_numpy_dtype_inference()`** detects missing values and **unconditionally converts Int64 → float64**
6. This causes:
   - **Precision loss** for large integers (>2^53)
   - **Dtype mismatch** with NumPy operations expecting integers
   - **Data loss** when precision is lost
