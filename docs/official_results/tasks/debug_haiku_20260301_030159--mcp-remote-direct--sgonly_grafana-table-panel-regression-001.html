## Summary of Findings

I've identified the root cause of the table panel field override regression after the v10.3 to v10.4 upgrade:

**The Problem**: Dashboard field override configurations are silently dropped during migration when dashboards lack explicit `fieldConfig.defaults.custom`, even though they have valid field overrides with `custom.displayMode`.

**Why It Happens**:
1. The backend v38 migration correctly converts `custom.displayMode` → `custom.cellOptions` in overrides
2. The frontend field config processing (`getPanelOptionsWithDefaults.ts`) filters out overrides based on registry lookups
3. The conditional logic pathway assumes `defaults.custom` must exist to properly validate custom override properties
4. When `defaults.custom` is absent, valid overrides fail the registry filter and get discarded

**Why Explicit `defaults.custom` Works**: 
Dashboards with explicit `fieldConfig.defaults.custom` follow a code path where the property is already in the defaults, allowing the field config registry to recognize and preserve the custom overrides.

**Key Files Involved**:
- `apps/dashboard/pkg/migration/schemaversion/v38.go` (backend migration)
- `public/app/features/dashboard/state/DashboardMigrator.ts` (frontend migration)
- `packages/grafana-data/src/panel/getPanelOptionsWithDefaults.ts` (field config processing - critical)
- `public/app/plugins/panel/table/` (table panel implementation)

The report includes specific line numbers, code evidence, and a detailed recommendation for fixing the issue while maintaining backward compatibility.

# Investigation Report: Dashboard Migration v38 Table Panel Regression

## Summary

The table panel field override configuration is silently dropped during dashboard import when upgrading from v10.3 to v10.4 for dashboards that lack explicit `fieldConfig.defaults.custom` definitions. The schema v38 migration correctly converts `custom.displayMode` to `custom.cellOptions` in both field defaults and overrides, but subsequent frontend field config processing discards valid overrides due to a conditional logic failure that assumes `defaults.custom` must exist for overrides to be preserved.

## Root Cause

**Primary Issue**: Migration v38 backend logic mishandles field override migration when `fieldConfig.defaults` is present but `fieldConfig.defaults.custom` is absent.

**Location**: `apps/dashboard/pkg/migration/schemaversion/v38.go` lines 115-127
