No group coordination or rebalancing occurs.\n\n#### `assign(Collection<TopicPartition> partitions)`\n```java\nvoid assign(Collection<TopicPartition> partitions)\n```\nManually assign a list of partitions to this consumer.\n\n**Behavior**:\n- Replaces previous assignment\n- Empty assignment treated as `unsubscribe()`\n- No consumer group coordination (no rebalancing)\n- Offsets committed to group using configured `group.id`, but group membership not maintained\n- Each consumer acts independently; ensure unique `group.id` to avoid offset conflicts\n\n**Advantages**:\n- Deterministic partition assignment\n- No rebalancing overhead\n- Suitable for applications managing own state per partition\n\n**Disadvantages**:\n- No automatic failure recovery\n- Manual handling of topology changes\n- Difficult with dynamic partition counts\n\n**Exceptions**:\n- `IllegalArgumentException`: If partitions contain null or empty topics\n- `IllegalStateException`: If `subscribe()` is active\n\n### 2.3 Assignment and Subscription Queries\n\n#### `assignment()`\n```java\nSet<TopicPartition> assignment()\n```\nReturns currently assigned partitions.\n\n**Return Value**:\n- For manual assignment: exact partitions assigned via `assign()`\n- For subscription: current assigned partitions (may be empty if no rebalance yet, or reassignment in progress)\n\n#### `subscription()`\n```java\nSet<String> subscription()\n```\nReturns currently subscribed topics.\n\n**Return Value**:\n- Empty set if no subscription active\n- Set of topics from last `subscribe()` call\n- Empty set after `unsubscribe()`\n\n---\n\n## 3. Polling and Data Fetching\n\nThe `poll()` method is the primary API for retrieving records and driving consumer group operations. It is **not a simple fetch operation**\u2014it also executes rebalance callbacks, heartbeats, and other background work.\n\n### 3.1 Poll Method\n\n#### `poll(Duration timeout)`\n```java\nConsumerRecords<K, V> poll(final Duration timeout)\n```\nFetches records for subscribed topics or assigned partitions.\n\n**Timeout Semantics** (Important):\n- If data is immediately available, returns immediately (timeout is ignored)\n- If position advances past control records/aborted transactions in `read_committed` isolation, returns immediately\n- Otherwise, blocks up to `timeout` waiting for data\n- **Rebalances block longer than timeout**: If a rebalance starts, `poll()` will block for the full rebalance duration even if exceeds timeout\n- Poll timeout is separate from broker operation timeouts (controlled by `default.api.timeout.ms`)\n\n**Return Value**:\n- `ConsumerRecords<K, V>`: May be empty (timeout expired, no data)\n- Records organized by partition with offsets, timestamps, headers\n- `ConsumerRecords.nextOffsets()` provides next positions for all partitions with records\n\n**Preconditions**:\n- At least one topic/partition must be subscribed/assigned\n- Subscription/assignment must be valid (topics must exist for group protocol users)\n\n**Poll Loop Requirements** (Essential):\n- Call `poll()` frequently to:\n  - Send heartbeats (prevents session timeout)\n  - Detect group membership changes\n  - Execute rebalance callbacks\n  - Enforce liveness check (`max.poll.interval.ms`)\n- Failure to poll frequently triggers:\n  - Session timeout (default 10 seconds) \u2192 rebalance\n  - Liveness timeout (`max.poll.interval.ms`, default 5 minutes) \u2192 proactive group leave\n\n**Record Processing**:\n- Records may have gaps in offsets (transaction markers, aborted messages filtered in `read_committed`)\n- Transaction markers and aborted messages never returned to application\n- Key and value may be null\n- Headers always present (possibly empty)\n\n**Exceptions**:\n- `InvalidOffsetException`: Offset undefined/out-of-range with no auto-reset policy\n- `WakeupException`: If `wakeup()` called before/during poll\n- `InterruptException`: If thread interrupted\n- `AuthenticationException`: Authentication failed\n- `AuthorizationException`: Insufficient permissions\n- `KafkaException`: Unrecoverable errors (invalid group ID, session timeout, deserialization failures, rebalance callback exceptions)\n- `InvalidTopicException`: Invalid topic in current subscription\n- `UnsupportedVersionException`: Broker doesn't support requested features\n- `FencedInstanceIdException`: Consumer instance fenced by broker\n\n**Important Pattern**:\n```java\ntry {\n    while (!closed.get()) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        for (ConsumerRecord<String, String> record : records) {\n            process(record);\n            // DO NOT commit here - wait until batch complete\n        }\n        // Commit after processing batch\n        consumer.commitSync();\n    }\n} finally {\n    consumer.close();\n}\n```\n\n### 3.2 Record Container: ConsumerRecords\n\n#### Iterating Records\n```java\n// Iterate all records across partitions\nfor (ConsumerRecord<K, V> record : records) {\n    // Process record\n}\n\n// Iterate records for specific topic\nfor (ConsumerRecord<K, V> record : records.records(\"topic-name\")) {\n    // Process\n}\n\n// Get records for specific partition\nList<ConsumerRecord<K, V>> partitionRecords = records.records(partition);\n\n// Get all partitions with data\nSet<TopicPartition> partitions = records.partitions();\n```\n\n#### Next Offsets\n```java\nMap<TopicPartition, OffsetAndMetadata> nextOffsets = records.nextOffsets();\n// Use for commitSync/commitAsync to commit next positions\n```\n\n---\n\n## 4. Offset Management\n\nOffsets represent positions in partitions. Kafka supports several offset concepts:\n- **Position (Current)**: Next offset to fetch (`position()`)\n- **Committed**: Last offset persisted to offset store (`committed()`)\n- **Log Start**: Earliest available offset (`beginningOffsets()`)\n- **Log End (High Water Mark)**: Latest available offset (`endOffsets()`)\n- **Last Stable Offset (LSO)**: Highest offset readable in `read_committed` mode\n\n**Key Principle**: Committed offset should always be the offset of the next message to consume, not the last consumed message.\n\n### 4.1 Automatic Offset Commits\n\nWhen `enable.auto.commit=true` (default):\n- Commits current position to offset storage periodically\n- Frequency: `auto.commit.interval.ms` (default 5 seconds)\n- Timing: After `poll()` returns (processed records may not be persisted yet!)\n- No callback or error indication\n\n**Risk**: If processing a `poll()` result fails, auto-commit proceeds anyway \u2192 message loss\n\n**When to Use**: Simple scenarios with fault-tolerant downstream or acceptable loss.\n\n### 4.2 Manual Synchronous Commits\n\n#### `commitSync()`\n```java\nvoid commitSync()\n```\nCommits the current position for all subscribed/assigned partitions.\n\n**Behavior**:\n- Synchronous: blocks until success or unrecoverable error\n- Timeout: `default.api.timeout.ms`\n- Commits **current position** (next offset to fetch)\n- Suitable for at-least-once semantics\n\n**Usage**:\n```java\nconsumer.poll(Duration.ofMillis(100));\n// Process records...\nconsumer.commitSync(); // Blocks until committed\n```\n\n#### `commitSync(Duration timeout)`\n```java\nvoid commitSync(Duration timeout)\n```\nCommits current position with explicit timeout.\n\n#### `commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)`\n```java\nvoid commitSync(final Map<TopicPartition, OffsetAndMetadata> offsets)\n```\nCommits specified offsets for specific partitions.\n\n**Parameters**:\n- `offsets`: Map from partition to offset and metadata\n- Offset should be next record to consume (e.g., `lastConsumed.offset() + 1`)\n- Can obtain from `ConsumerRecords.nextOffsets()`\n- Metadata may include leader epoch\n\n**Usage Example**:\n```java\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\nfor (TopicPartition partition : records.partitions()) {\n    List<ConsumerRecord<String, String>> partitionRecords = records.records(partition);\n    for (ConsumerRecord<String, String> record : partitionRecords) {\n        process(record);\n    }\n    // Commit offset of next record to read\n    long nextOffset = records.records(partition).get(0).offset() + partitionRecords.size();\n    consumer.commitSync(Collections.singletonMap(\n        partition,\n        new OffsetAndMetadata(nextOffset, records.nextOffsets().get(partition).metadata())\n    ));\n}\n```\n\n#### `commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)`\n```java\nvoid commitSync(final Map<TopicPartition, OffsetAndMetadata> offsets, final Duration timeout)\n```\nCommits specified offsets with explicit timeout.\n\n**Exceptions from Commit**:\n- `CommitFailedException`: Partitions no longer assigned (group protocol user fenced)\n- `RebalanceInProgressException`: Rebalance in progress, positions may change\n- `WakeupException`: `wakeup()` called\n- `InterruptException`: Thread interrupted\n- `TimeoutException`: Timeout expired\n- `AuthenticationException`: Auth failed\n- `AuthorizationException`: Insufficient permissions\n\n### 4.3 Manual Asynchronous Commits\n\n#### `commitAsync()`\n```java\nvoid commitAsync()\n```\nAsynchronously commits current position. Equivalent to `commitAsync(null)`.\n\n**Behavior**:\n- Non-blocking: returns immediately\n- Errors not reported (discarded unless callback provided)\n- Ordered: multiple commits sent in order\n- Guaranteed to complete before next `commitSync()` call\n\n#### `commitAsync(OffsetCommitCallback callback)`\n```java\nvoid commitAsync(OffsetCommitCallback callback)\n```\nAsynchronously commits with callback notification.\n\n**Callback Execution**:\n- Invoked when commit completes (success or failure)\n- Order: callbacks invoked in same order as `commitAsync()` calls\n- Execution: in background without blocking poll thread\n- Exception handling: callback receives exception, not thrown to caller\n\n**Callback Interface**:\n```java\n@FunctionalInterface\npublic interface OffsetCommitCallback {\n    void onComplete(Map<TopicPartition, OffsetAndMetadata> offsets, Exception exception);\n}\n```\n\n**Usage**:\n```java\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception != null) {\n        logger.error(\"Async commit failed\", exception);\n    }\n});\n```\n\n#### `commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)`\n```java\nvoid commitAsync(final Map<TopicPartition, OffsetAndMetadata> offsets, OffsetCommitCallback callback)\n```\nAsynchronously commits specified offsets with callback.\n\n### 4.4 Querying Offsets\n\n#### `position(TopicPartition partition)`\n```java\nlong position(TopicPartition partition)\n```\nReturns the **current position** (next offset to fetch) for a partition.\n\n**Behavior**:\n- Synchronous blocking call\n- May issue remote call if position undefined\n- Timeout: `default.api.timeout.ms`\n- Returns offset of next record to be fetched\n\n**Postcondition**:\n- `position(partition) == committed(partition).offset() + 1` (after no consume)\n\n**Exceptions**:\n- `IllegalStateException`: Partition not assigned\n- `InvalidOffsetException`: No position defined\n- `TimeoutException`: Timeout expired\n- `WakeupException`: Wakeup called\n\n#### `position(TopicPartition partition, Duration timeout)`\n```java\nlong position(TopicPartition partition, final Duration timeout)\n```\nPosition query with explicit timeout.\n\n#### `committed(Set<TopicPartition> partitions)`\n```java\nMap<TopicPartition, OffsetAndMetadata> committed(final Set<TopicPartition> partitions)\n```\nReturns **last committed offsets** for specified partitions.\n\n**Behavior**:\n- Synchronous remote call to offset coordinator\n- Timeout: `default.api.timeout.ms`\n- Returns only partitions with commits; others map to null\n- Offset is \"next to read\" (last consumed + 1)\n- Metadata includes leader epoch if available\n\n**Return Value**:\n- Map with entries only for partitions with commits\n- null for partitions with no commit history\n\n**Usage**:\n```java\nMap<TopicPartition, OffsetAndMetadata> committed = consumer.committed(partitions);\nfor (TopicPartition tp : partitions) {\n    OffsetAndMetadata om = committed.get(tp);\n    if (om != null) {\n        System.out.println(\"Committed offset: \" + om.offset());\n    }\n}\n```\n\n#### `committed(Set<TopicPartition> partitions, Duration timeout)`\n```java\nMap<TopicPartition, OffsetAndMetadata> committed(final Set<TopicPartition> partitions, final Duration timeout)\n```\nCommitted offsets with explicit timeout.\n\n#### `beginningOffsets(Collection<TopicPartition> partitions)`\n```java\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions)\n```\nReturns the **earliest available offsets** for specified partitions.\n\n**Behavior**:\n- Does not change consumer position\n- Remote call required; blocks until complete\n- Timeout: `default.api.timeout.ms`\n- Partitions not required to be assigned\n\n**Return Value**:\n- Map of earliest available offsets per partition\n- Empty map if timeout is zero\n\n**Usage**:\n```java\nMap<TopicPartition, Long> offsets = consumer.beginningOffsets(partitions);\nfor (Map.Entry<TopicPartition, Long> entry : offsets.entrySet()) {\n    System.out.println(entry.getKey() + \" starts at \" + entry.getValue());\n}\n```\n\n#### `beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)`\n```java\nMap<TopicPartition, Long> beginningOffsets(Collection<TopicPartition> partitions, Duration timeout)\n```\nBeginning offsets with explicit timeout.\n\n#### `endOffsets(Collection<TopicPartition> partitions)`\n```java\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions)\n```\nReturns the **latest available offsets** for specified partitions.\n\n**Behavior**:\n- In `read_uncommitted`: high watermark (HWM)\n- In `read_committed`: Last Stable Offset (LSO) = min(HWM, first offset in open transaction)\n- Does not change position\n- Remote call; blocks until complete\n- Timeout: `default.api.timeout.ms`\n\n**Semantics**:\n- Offset X returned means: record at offset X-1 is readable, offset X not readable yet\n- Empty partition has end offset 0\n\n#### `endOffsets(Collection<TopicPartition> partitions, Duration timeout)`\n```java\nMap<TopicPartition, Long> endOffsets(Collection<TopicPartition> partitions, Duration timeout)\n```\nEnd offsets with explicit timeout.\n\n---\n\n## 5. Position Control and Seeking\n\n### 5.1 Seeking\n\nSeeking changes the position to a specific offset. Useful for replaying messages, skipping to latest, or restoring from external state.\n\n#### `seek(TopicPartition partition, long offset)`\n```java\nvoid seek(TopicPartition partition, long offset)\n```\nSeeks to specified offset.\n\n**Behavior**:\n- Sets position for next `poll()`; does not affect in-flight fetches\n- Valid offsets: 0 to log end offset (or last offset for empty partitions)\n- Invalid offsets (out of range): behavior depends on `auto.offset.reset`\n  - \"earliest\": seek to start\n  - \"latest\": seek to end\n  - \"none\": `OffsetOutOfRangeException` on next poll\n\n**Important**:\n- Seek won't affect fetches already in flight\n- Consumer may wait up to `fetch.max.wait.ms` for next fetch\n- Position advances through poll; seek only changes fetch offset\n\n**Exceptions**:\n- `IllegalArgumentException`: Offset < 0\n- `IllegalStateException`: Partition not assigned\n\n#### `seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)`\n```java\nvoid seek(TopicPartition partition, OffsetAndMetadata offsetAndMetadata)\n```\nSeeks to specified offset with metadata (including leader epoch).\n\n### 5.2 Seek-to-Position Helpers\n\n#### `seekToBeginning(Collection<TopicPartition> partitions)`\n```java\nvoid seekToBeginning(Collection<TopicPartition> partitions)\n```\nSeeks to earliest available offset for each partition.\n\n**Behavior**:\n- Lazy evaluation: actual seek happens on next `poll()` or `position()` call\n- Empty partition set: seeks all currently assigned partitions\n- Equivalent to `seek(partition, 0)` if earliest offset is 0\n\n**Exceptions**:\n- `IllegalArgumentException`: partitions is null\n- `IllegalStateException`: Any partition not assigned\n\n#### `seekToEnd(Collection<TopicPartition> partitions)`\n```java\nvoid seekToEnd(Collection<TopicPartition> partitions)\n```\nSeeks to latest offset for each partition.\n\n**Behavior**:\n- `read_uncommitted`: seeks to high watermark\n- `read_committed`: seeks to Last Stable Offset (LSO)\n- Lazy evaluation: actual seek on next `poll()` or `position()`\n- Empty partition set: seeks all currently assigned partitions\n\n**Use Cases**:\n- Skip historical data on startup\n- Process only new messages\n\n### 5.3 Timestamp-based Seeking\n\n#### `offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)`\n```java\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch)\n```\nFinds offsets for specified timestamps.\n\n**Behavior**:\n- Returns earliest offset with timestamp \u2265 target timestamp\n- Remote call; blocks until complete\n- Timeout: `default.api.timeout.ms`\n- Partitions not required to be assigned\n- Old message formats without timestamps return null\n\n**Return Value**:\n- Map of partition \u2192 OffsetAndTimestamp\n- null for partitions without result (no timestamp, old format)\n\n**Usage**:\n```java\nMap<TopicPartition, Long> timestamps = partitions.stream()\n    .collect(Collectors.toMap(tp -> tp, tp -> System.currentTimeMillis() - 3600000)); // 1 hour ago\nMap<TopicPartition, OffsetAndTimestamp> offsets = consumer.offsetsForTimes(timestamps);\nfor (Map.Entry<TopicPartition, OffsetAndTimestamp> entry : offsets.entrySet()) {\n    if (entry.getValue() != null) {\n        consumer.seek(entry.getKey(), entry.getValue().offset());\n    }\n}\n```\n\n#### `offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch, Duration timeout)`\n```java\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(Map<TopicPartition, Long> timestampsToSearch, Duration timeout)\n```\nOffset lookup with explicit timeout.\n\n---\n\n## 6. Flow Control and Pause/Resume\n\n### 6.1 Pause and Resume\n\nControl which partitions are fetched without unsubscribing.\n\n#### `pause(Collection<TopicPartition> partitions)`\n```java\nvoid pause(Collection<TopicPartition> partitions)\n```\nPauses fetching from specified partitions.\n\n**Behavior**:\n- Paused partitions not returned from future `poll()` calls\n- Subscription/assignment unchanged\n- No rebalance triggered\n- Idempotent: pausing already-paused partition is no-op\n\n**Use Cases**:\n- Decouple consumption from processing (pause until workers ready)\n- Flow control in multi-partition scenarios\n- Temporarily stop processing from slow partitions\n\n**Important**: Pause state is **not preserved across rebalances**. Must re-pause in `onPartitionsAssigned()` if needed.\n\n**Exceptions**:\n- `IllegalStateException`: Any partition not assigned\n\n#### `resume(Collection<TopicPartition> partitions)`\n```java\nvoid resume(Collection<TopicPartition> partitions)\n```\nResumes fetching from paused partitions.\n\n**Behavior**:\n- Paused partitions will be fetched in next `poll()`\n- Idempotent: resuming non-paused partition is no-op\n- Subscription unchanged\n\n#### `paused()`\n```java\nSet<TopicPartition> paused()\n```\nReturns currently paused partitions.\n\n**Return Value**: Set of TopicPartition objects; empty set if none paused\n\n### 6.2 Common Pause/Resume Pattern\n\n**Multi-threaded Processing with Worker Pool**:\n```java\nExecutorService executor = Executors.newFixedThreadPool(10);\nQueue<ConsumerRecord<String, String>> queue = new LinkedBlockingQueue<>(100);\n\n// Consumption thread\nThread consumeThread = new Thread(() -> {\n    try {\n        consumer.subscribe(Arrays.asList(\"topic\"));\n        while (running) {\n            // If queue full, pause to avoid unbounded memory\n            if (queue.size() >= 100) {\n                consumer.pause(consumer.assignment());\n            } else {\n                consumer.resume(consumer.assignment());\n            }\n\n            ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n            for (ConsumerRecord<String, String> record : records) {\n                queue.put(record);\n            }\n        }\n    } finally {\n        consumer.close();\n    }\n});\n\n// Worker threads process from queue\nfor (int i = 0; i < 10; i++) {\n    executor.execute(() -> {\n        try {\n            while (running) {\n                ConsumerRecord<String, String> record = queue.poll(1, TimeUnit.SECONDS);\n                if (record != null) {\n                    process(record);\n                }\n            }\n        } catch (InterruptedException e) {\n            Thread.currentThread().interrupt();\n        }\n    });\n}\n```\n\n---\n\n## 7. Consumer Group Mechanics and Rebalancing\n\nConsumer groups enable horizontal scaling by distributing partitions across multiple consumers. The broker coordinates group membership and rebalancing.\n\n### 7.1 Group Membership and Heartbeats\n\n**Session Timeout** (`session.timeout.ms`, default 10s):\n- Broker expects heartbeat at least every `session.timeout.ms`\n- If heartbeat not received, consumer considered dead\n- Partitions reassigned to other consumers\n- Triggered when `poll()` not called for duration\n\n**Poll Interval Timeout** (`max.poll.interval.ms`, default 5 minutes):\n- Consumer must call `poll()` at least every `max.poll.interval.ms`\n- If exceeded, consumer **proactively leaves group**\n- Next commit will fail with `CommitFailedException`\n- Safety mechanism against livelock (consumer processing but making no progress)\n\n**Poll frequency**: Call `poll()` more frequently than `max.poll.interval.ms` to stay in group.\n\n### 7.2 ConsumerRebalanceListener Interface\n\nThe rebalance listener allows custom handling of partition assignment changes.\n\n#### Interface Definition\n```java\npublic interface ConsumerRebalanceListener {\n    void onPartitionsRevoked(Collection<TopicPartition> partitions);\n    void onPartitionsAssigned(Collection<TopicPartition> partitions);\n    default void onPartitionsLost(Collection<TopicPartition> partitions) {\n        onPartitionsRevoked(partitions);\n    }\n}\n```\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled when partitions are being revoked before reassignment.\n\n**When Called**:\n- Graceful rebalance (normal group changes)\n- Closing consumer\n- Unsubscribing\n\n**Behavior**:\n- Called BEFORE partitions reassigned\n- Single-threaded: executes in poll() thread\n- Exceptions propagate to `poll()` call\n\n**Classic Protocol with Eager Rebalancing**: All partitions revoked even if no change\n**Classic/Cooperative or Consumer Protocol**: Only changed partitions revoked\n\n**Typical Use**: Commit offsets, flush caches\n```java\n@Override\npublic void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n    // Commit offsets for partitions being revoked\n    consumer.commitSync();\n    // Flush any cached state\n    cache.flush();\n}\n```\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled when new partitions assigned after rebalance completes.\n\n**When Called**:\n- After each rebalance (even if no new partitions)\n- Before fetching from assigned partitions\n- Single-threaded: in poll() thread\n\n**Behavior**:\n- Called with newly assigned partitions (not previously owned)\n- Guaranteed after all previous members' `onPartitionsRevoked()` completes\n- Can query/modify consumer state before fetching starts\n\n**Typical Use**: Initialize position, load state\n```java\n@Override\npublic void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n    for (TopicPartition partition : partitions) {\n        // Load last processed offset from external store\n        long offset = stateStore.getOffset(partition);\n        consumer.seek(partition, offset);\n    }\n}\n```\n\n#### `onPartitionsLost(Collection<TopicPartition> partitions)`\n\nCalled when partitions lost unexpectedly (not gracefully revoked).\n\n**When Called**:\n- Session timeout expired\n- Fencing by broker\n- Other fatal group errors\n- Partitions already owned by other consumers (cannot commit)\n\n**Default Behavior**: Calls `onPartitionsRevoked()`\n\n**Typical Use**: Skip expensive operations (don't commit, already owned)\n```java\n@Override\npublic void onPartitionsLost(Collection<TopicPartition> partitions) {\n    // Don't commit - partitions owned by others\n    // Just cleanup local state\n    for (TopicPartition partition : partitions) {\n        stateStore.clear(partition);\n    }\n}\n```\n\n### 7.3 Rebalance Protocols\n\nKafka supports multiple group coordination protocols:\n\n**CLASSIC (Eager Rebalancing)**:\n- All members revoke all partitions\n- Broker reassigns all partitions\n- Higher coordination overhead\n- Simple to reason about\n\n**COOPERATIVE**:\n- Only revoke partitions actually moving\n- Reduces stop-the-world effect\n- Requires stateless partition assignment\n- Modern default\n\n**CONSUMER**:\n- New group coordination protocol (4.0+)\n- More efficient rebalancing\n- Improved performance\n\n### 7.4 Enforce Rebalance\n\n#### `enforceRebalance()`\n```java\nvoid enforceRebalance()\nvoid enforceRebalance(final String reason)\n```\n\nForces a rebalance on next `poll()`.\n\n**Behavior**:\n- Non-blocking; actual rebalance on next `poll()`\n- Useful when external state changes affect partition assignment\n- Assignor can use `userdata` to transmit state in rebalance\n\n**Use Case**: Stream processor detects topology change\n```java\nif (topologyChanged()) {\n    consumer.enforceRebalance(\"Topology change detected\");\n}\n```\n\n---\n\n## 8. Transactional Message Consumption\n\n### 8.1 Isolation Levels\n\n**read_uncommitted** (default):\n- Reads all messages regardless of transaction status\n- Includes aborted transaction messages\n- Faster; low latency\n- Suitable for non-critical data\n\n**read_committed**:\n- Reads only committed transaction messages\n- Aborted messages filtered out\n- Control records (markers) filtered out\n- End offset is Last Stable Offset (LSO)\n\n### 8.2 Last Stable Offset (LSO)\n\nThe LSO is the highest offset safe to read in `read_committed` mode. It's the minimum of:\n- High watermark (last replicated message + 1)\n- First offset in any open transaction\n\n**Implications**:\n- `endOffsets()` returns LSO, not HWM\n- `seekToEnd()` seeks to LSO\n- Fetch may return fewer records than requested\n- Gaps in offsets due to filtered aborted messages\n\n### 8.3 Transactional Semantics\n\nWith `isolation.level=read_committed`:\n- Consumer never sees uncommitted messages from ongoing transactions\n- Consumer never sees aborted messages\n- Consistent application-level reads across partitions\n\n**Usage**:\n```java\nProperties props = new Properties();\nprops.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG, \"read_committed\");\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\n```\n\n---\n\n## 9. Thread Safety and Multi-threaded Consumption\n\n### 9.1 Thread Safety Model\n\n**NOT thread-safe**:\n- All operations except `wakeup()` must be called from single thread\n- Concurrent calls cause `ConcurrentModificationException`\n- No synchronization provided by consumer\n\n**Exception**: `wakeup()` is thread-safe and can be called from any thread.\n\n### 9.2 Wakeup and Interruption\n\n#### `wakeup()`\n```java\nvoid wakeup()\n```\nWakes up consumer thread from blocking operation.\n\n**Behavior**:\n- Thread-safe; ## Overview

The `KafkaConsumer` is a client that consumes records from a Kafka cluster. It provides a flexible and powerful API for consuming messages from one or more topics, with support for both group-managed and manual partition assignment, automatic and manual offset management, and rich options for controlling consumption flow.

### Key Characteristics

- **Not Thread-Safe**: The consumer is designed for single-threaded use. All operations except `wakeup()` must be called from the same thread.
- **Automatic Failure Handling**: Transparently handles broker failures and adapts as topic partitions migrate within the cluster.
- **Consumer Group Support**: Enables horizontal scaling through consumer groups with automatic rebalancing.
- **Flexible Offset Management**: Supports both automatic and manual offset commits, with optional external offset storage.
- **TCP Connection Management**: Maintains connections to brokers; must be closed to avoid resource leaks.

### Version Compatibility

The KafkaConsumer can communicate with brokers version 0.10.0 and newer. Some features may not be available on older broker versions and will raise `UnsupportedVersionException`.
