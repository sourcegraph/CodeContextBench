# Bug Investigation: Authentication Fails for Browser Cookie-Based Tokens

**Repository:** flipt-io/flipt
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

The authentication system in Flipt only accepts client tokens via the `Authorization` header using Bearer format. This prevents browser-based sessions from working, since browsers typically store authentication tokens in HTTP cookies.

Specifically:

1. **Cookie tokens rejected**: When a valid client token is stored in an HTTP cookie (under the key `flipt_client_token`), the authentication middleware does not extract or validate it. Requests that carry a valid token in a cookie but not in the Authorization header are rejected as unauthenticated.

2. **No server-level authentication bypass**: Certain internal servers (such as those handling delegated authentication flows) need to operate without authentication checks. There is currently no mechanism to configure the middleware to skip authentication for specific server instances, forcing all registered servers through the same authentication path regardless of their role.

# Flipt Authentication Regression Test

This regression test demonstrates two critical authentication issues in the Flipt codebase:

## Issues Demonstrated

### 1. Cookie-Based Token Extraction Failure (BUG #1)
**Location**: `/app/internal/server/auth/middleware.go`

The authentication middleware only accepts client tokens via the `Authorization` header using the "Bearer " format. Browser-based sessions that store tokens in HTTP cookies are rejected.

**Test Cases**:
- `BearerTokenSucceeds`: ✅ PASSES - Valid Bearer token authentication works
- `CookieTokenFails`: ❌ FAILS - Cookie-based tokens are rejected (demonstrates the bug)
- `AlternativeTokenHeaderSucceeds`: ❌ FAILS - Alternative header formats (like `x-client-token`) are not supported
- `TokenWithoutBearerPrefixFails`: ✅ PASSES - Correctly rejects tokens without Bearer prefix

**Root Cause**:
```go
// Line 51-54 in middleware.go
authenticationHeader := md.Get(authenticationHeaderKey)
if len(authenticationHeader) < 1 {
    logger.Error("unauthenticated", zap.String("reason", "no authorization provided"))
    return ctx, errUnauthenticated
}
```

The middleware only checks for the "authorization" header key. It doesn't attempt to:
- Extract tokens from cookies (cookie header)
- Check alternative header names (x-client-token, etc.)
- Fall back to cookies if Authorization header is missing

### 2. Server-Level Authentication Bypass Unavailable (BUG #2)
**Location**: `/app/cmd/flipt/main.go`

There is no mechanism to configure certain servers (such as internal/delegated authentication services) to operate without authentication enforcement. All services registered on the gRPC server are equally protected.

**Test Cases**:
- `UnauthenticatedAuthServiceCallFails`: ✅ PASSES - Correctly rejects unauthenticated calls
- `NoWayToBypassAuthenticationForSpecificService`: ❌ FAILS - No way to skip auth for internal services
- `MiddlewareHasNoBypassMechanism`: ✅ PASSES - Documents the architectural limitation

**Root Cause**:
```go
// main.go - the middleware is registered globally
if cfg.Authentication.Required {
    interceptors = append(interceptors, auth.UnaryInterceptor(logger, authenticationStore))
}

// All services use the same interceptor chain
grpcOpts := []grpc.ServerOption{grpc_middleware.WithUnaryServerChain(interceptors...)}
srv := grpc.NewServer(grpcOpts...)

// Register all services on the same server with the same interceptors
pb.RegisterFliptServer(grpcServer, srv)
authrpc.RegisterAuthenticationServiceServer(grpcServer, authServer)
authrpc.RegisterAuthenticationMethodTokenServiceServer(grpcServer, tokenServer)
```

The middleware doesn't have any logic to check:
- Which service is being called
- Which method is being called
- Any configuration to exempt specific services/methods from authentication

This creates an architectural problem: you need a token to authenticate, but you need to be authenticated to create a token.

## How to Fix These Bugs

### Fix #1: Support Cookie-Based Token Extraction
Modify `/app/internal/server/auth/middleware.go` to:
1. Check for Authorization header first (backward compatibility)
2. Fall back to cookie extraction if not found
3. Parse the cookie value as a direct token (not "Bearer " prefixed)
4. Support alternative header names as needed

### Fix #2: Support Service-Level Authentication Bypass
Modify the authentication middleware and server initialization to:
1. Allow configuring services/methods that should bypass authentication
2. Check the method being called and skip authentication if configured
3. Support marking servers as "internal" or "delegated"
4. Provide a configuration option to specify which services require authentication

This would allow:
- Browser-based sessions with cookie authentication
- Internal services to operate without authentication
- Proper bootstrapping of the authentication system
