## Summary of Analysis

I've mapped the complete journey from a high-level QuerySet API call through to SQL execution:

**Key Findings:**

1. **Entry Point**: `Model.objects.filter(...)` creates a QuerySet wrapping a Query object (intermediate SQL representation)

2. **Three-Layer Architecture**:
   - **QuerySet API** — Lazy query building (filter, exclude, values, etc.)
   - **Query Object** — Stores WHERE/ORDER BY/GROUP BY structure as a tree
   - **SQLCompiler** — Generates vendor-specific SQL via as_sql() protocol

3. **WhereNode Tree Structure**: WHERE conditions form a tree where each node is either a Lookup (comparison operation) or another WhereNode (grouped conditions), composing complex filters

4. **Vendor Dispatch Mechanism** (the compile pattern):
   - `compiler.compile(node)` checks for `as_<vendor>()` method
   - Falls back to generic `as_sql()` if none exists
   - Applies to all Expression and Lookup types
   - Allows backends (PostgreSQL, MySQL, SQLite, Oracle) to override dialect-specific SQL generation

5. **Design Patterns**: Lazy evaluation, visitor/traversal for tree processing, compiler pattern for separation of concerns, and strategy pattern for vendor dispatch

6. **Execution Flow**: Query → Compiler → as_sql() generates (sql, params) → cursor.execute() → results iterated and converted to model instances
