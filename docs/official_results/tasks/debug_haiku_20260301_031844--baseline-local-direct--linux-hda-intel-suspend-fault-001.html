**Bug ID**: 53441
**Summary**: hda_intel: sound stops working after suspend to ram
**Regression**: Yes (works on v3.6.11, broken since v3.7.1)

On a fresh boot the sound card works fine. After a few suspends to ram (it doesn't necessarily happen after just one) the sound card stops working well. The sound comes out choppy, as a series of clicks separated by 2 second or so silences.

I see these messages in the kernel log right after resume:
```

hda_intel: azx_get_response timeout, switching to single_cmd mode: last cmd=0x00170503
```

Perfect! I've completed the fault localization for this HDA Intel suspend/resume bug. Here's a summary of my findings:

## Summary

I've identified the source of the HDA Intel sound failure after suspend to RAM in kernel v3.7.6:

**Root Cause**: The HDA Intel driver sets `chip->single_cmd=1` and `chip->polling_mode=1` when codec communication timeouts occur (fallback to degraded communication modes), but these flags are **never reset**. During suspend/resume cycles, these persistent flags prevent proper re-initialization of the CORB/RIRB communication channel, leaving the codec in a degraded state.

**Buggy File**: `sound/pci/hda/hda_intel.c`

**Buggy Functions**:
1. **`azx_resume`** (line 2539) - The resume function fails to reset the degradation flags before reinitializing the chip
2. **`azx_rirb_get_response`** (line 863) - This function sets the degradation flags but provides no mechanism to recover/reset them

**The Problem Flow**:
1. Fresh boot → normal operation (flags = 0)
2. If any timeout occurs → `chip->single_cmd=1` and/or `chip->polling_mode=1` set
3. During suspend → `chip->initialized=0` is reset, but the degradation flags persist
4. During resume → `azx_init_chip()` checks `chip->single_cmd` (line 1222) and skips CORB/RIRB re-initialization if it's 1
5. Result → codec stays in degraded mode, causing choppy audio and continued timeouts
6. Only reboot (which re-initializes the chip) fixes it
