Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### 4.1.1 Watch Connection Loss and ResourceVersion Staleness\n\n**Symptom:** API server returns `410 Gone` when reflector attempts to watch from stale resourceVersion.\n\n**Cause:**\n- Watch events older than the API server's history window (~5-10 min, configurable) are garbage-collected\n- Network disruption causes watcher to lose connection; by the time it reconnects, resourceVersion is too old\n\n**Handling:**\n- Reflector detects 410 and immediately re-lists (ListAndWatch loop retries)\n- BOOKMARK events allow efficient recovery without full relist\n- Watch-list feature (useWatchList=true) further optimizes by streaming initial state alongside bookmarks\n\n**Mitigation:**\n- Increase API server history window if frequent 410s occur\n- Use watch bookmarks (default in k8s 1.20+)\n- Monitor reflector error metrics for patterns\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/reflector.go:612 \u2013 list() detects 410\n- staging/src/k8s.io/client-go/tools/cache/reflector.go:499 \u2013 watch() error handling\n\n#### 4.1.2 Informer Event Handler Blocking\n\n**Symptom:** All event notifications blocked; informer's event channel fills up; new events dropped or delayed.\n\n**Cause:**\n- Event handler (OnAdd/OnUpdate/OnDelete) performs slow operations (API calls, disk I/O)\n- Block delays all other handlers and new events\n\n**Design Choice:**\n- SharedInformer intentionally does NOT make handlers concurrent or queue-based\n- Handlers are called sequentially in the controller's main loop thread\n\n**Mitigation:**\n- **Critical Rule:** Event handlers must return immediately (milliseconds).\n- **Correct Pattern:**\n  ```go\n  // Handler: just enqueue the key\n  OnAdd: func(obj interface{}) {\n    key, _ := cache.MetaNamespaceKeyFunc(obj)\n    workqueue.Add(key)  // ~1 microsecond\n  }\n  ```\n- **Wrong Pattern:**\n  ```go\n  // Handler: do the work here\n  OnAdd: func(obj interface{}) {\n    pod := obj.(*corev1.Pod)\n    reconcile(pod)  // WRONG: blocks all event processing\n  }\n  ```\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/shared_informer.go:200 \u2013 Event handler contract\n\n#### 4.1.3 Controller Requeue Loop\n\n**Symptom:** Work queue never drains; controller loops infinitely retrying the same key.\n\n**Cause:**\n- reconcile() function always returns error (permanent failure condition)\n- Controller calls workqueue.AddRateLimited(key) on every failure\n- Backoff maxes out at 15 minutes; key keeps getting re-added\n\n**Handling:**\n- ReconcileErr must eventually succeed or be explicitly removed from queue\n- Bad pattern: retrying without any side effect (idempotence check)\n\n**Mitigation:**\n- Add idempotence check: reconcile() should detect if work is already done and succeed\n- Implement `reconcile(key) error` to return nil once the desired state is achieved\n- Use finalizers to ensure cleanup on deletion\n- Log persistent failures with context for debugging\n\n**File References:**\n- staging/src/k8s.io/client-go/util/workqueue/rate_limiting_queue.go \u2013 Retry logic\n\n#### 4.1.4 Memory Exhaustion Due to Unbounded Cache\n\n**Symptom:** client-go process OOM-kills after running for hours or days.\n\n**Cause:**\n- Informer caches all objects it watches (no eviction by default)\n- If watching resources with large payloads or high cardinality (millions of objects), memory grows linearly\n- Example: Watching all Pods in cluster with large volumes' metadata\n\n**Handling:**\n- Transform function can drop fields before caching (staging/src/k8s.io/client-go/tools/cache/delta_fifo.go:154\u2013164)\n  ```go\n  // Example: Keep only metadata, drop spec/status\n  factory := NewSharedInformerFactoryWithOptions(\n    clientset, resync,\n    WithTransform(func(obj interface{}) (interface{}, error) {\n      pod := obj.(*corev1.Pod)\n      // Clear large fields\n      pod.Spec.Volumes = nil\n      return pod, nil\n    }),\n  )\n  ```\n- Alternative: Use metadata-only client for discovery, DynamicClient for lazy fetching\n- Monitor metrics: informer cache size, GC pressure\n\n**Mitigation:**\n- Set resource limits (memory requests/limits in Kubernetes)\n- Use transform function to drop unnecessary fields\n- Shard watches across multiple informers/replicas\n- Monitor metrics: informer_cache_size_items, go_memstats_alloc_bytes\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/delta_fifo.go:154\u2013164 \u2013 TransformFunc\n- staging/src/k8s.io/client-go/informers/factory.go:103 \u2013 WithTransform option\n\n#### 4.1.5 Race Between Cache Access and Update\n\n**Symptom:** Controller reads stale data from cache; business logic makes incorrect decisions.\n\n**Cause:**\n- Indexer updates and handler notifications are serialized, but lister queries are not locked to the same transaction\n- Example:\n  ```\n  1. Informer event: Pod \"foo\" updated (new resourceVersion)\n  2. Lister query: cache.Get(\"foo\") might return old or new version\n  3. Handler callback: Pod{\"foo\", oldVersion}\n  4. Controller reconciles using different state than what handler saw\n  ```\n\n**Handling:**\n- Indexer provides **read-consistency**: List() and Get() are atomic snapshots\n- However, between two separate queries (even milliseconds apart), state can change\n- No SERIALIZABLE isolation across multiple queries\n\n**Mitigation:**\n- Rely on **eventual consistency**: reconcile() is idempotent; if it runs again with newer data, it corrects itself\n- For strong consistency needs, refetch from API server: `clientset.CoreV1().Pods(ns).Get(ctx, name, opts)`\n- Use object UID to detect object identity changes (namespace reuse)\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/thread_safe_store.go:70\u2013120 \u2013 Lock scopes\n\n#### 4.1.6 Dual-Write Race (API Server vs. Local State)\n\n**Symptom:** Two controller replicas (before leader election kicks in) write conflicting updates to the same object.\n\n**Cause:**\n- Leader election lock acquired, but grace period means old leader hasn't shut down yet\n- Both issue writes concurrently\n- No versioning; last-write-wins\n\n**Handling:**\n- **Proper solution:** Use server-side apply (ApplyConfiguration) or field managers\n  ```go\n  applyConfig := &appsv1ac.DeploymentApplyConfiguration{\n    ObjectMetaApplyConfiguration: &metav1ac.ObjectMetaApplyConfiguration{\n      Name:      \"my-deploy\",\n      Namespace: \"default\",\n    },\n    Spec: &appsv1ac.DeploymentSpecApplyConfiguration{...},\n  }\n  clientset.AppsV1().Deployments(\"default\").Apply(ctx, applyConfig, opts)\n  ```\n- Server-side apply safely merges field ownership; multiple writers can coexist\n\n**Mitigation:**\n- Use leader election (lease-based) for single-writer semantics\n- Or: Use server-side apply for declarative, multi-writer-safe updates\n- Monitor conflicts: APIserver tracks field ownership per writer\n\n**File References:**\n- staging/src/k8s.io/client-go/applyconfigurations/ \u2013 Generated apply builders\n- staging/src/k8s.io/client-go/tools/leaderelection/ \u2013 Leader election\n\n---\n\n### 4.2 Design Limits\n\n#### 4.2.1 Informer Pagination and List/Watch Timeouts\n\n**Limit:** LIST requests are paginated in chunks (default: 500 items).\n\n**Implication:**\n- Initial LIST takes longer for clusters with many objects\n- Paginated LIST less efficient than streaming watch-list (Kubernetes 1.27+)\n- Watch is unbounded; streamed one event at a time\n\n**Trade-off:**\n- **Pagination:** Predictable memory usage per request; compatible with older servers\n- **Watch-list (streaming):** Fewer round-trips; higher server load initially; requires k8s 1.27+\n\n**Configuration:**\n- `Reflector.WatchListPageSize`: Control chunk size (0 = disable pagination, use stream)\n- `minWatchTimeout`: Prevent connection storms (default: 5 min + jitter)\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/reflector.go:127\u2013134 \u2013 WatchListPageSize, useWatchList\n\n#### 4.2.2 Event Handler Registration Timing\n\n**Limit:** Handlers added after informer.Run() is called get an initial batch of \"current\" objects, not a full resync.\n\n**Implication:**\n- If controller starts handlers very late (after cache is populated), those handlers see a snapshot, not a guaranteed initial listing\n- Handlers added before Run() get every object via OnAdd during initial LIST\n\n**Trade-off:**\n- **Early registration (before Run()):** Guaranteed to see initial state as OnAdd\n- **Late registration:** Efficient (no re-list), but may miss some state\n\n**Mitigation:**\n- Register all handlers before calling factory.Start()\n- If adding handlers dynamically, refetch from API: `lister.List(selector)`\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/shared_informer.go:150\u2013200 \u2013 AddEventHandler timing\n\n#### 4.2.3 Resync Churn\n\n**Limit:** Default resync period (12 hours) can cause spike in processing when triggered.\n\n**Implication:**\n- Resync generates Sync delta for every cached object\n- If millions of pods cached, resync enqueues millions of work items\n- CPU/API load spike every resync period\n\n**Trade-off:**\n- **Resync:** Recovers from controller crashes or missed events; ensures reconciliation happens periodically\n- **No resync:** Lean on watch events only; risk missing changes if reconciler crashes between events\n\n**Configuration:**\n```go\nfactory := NewSharedInformerFactoryWithOptions(\n  clientset, 30*time.Second,  // Shorter resync = more frequent spikes\n  WithCustomResyncConfig(map[runtime.Object]time.Duration{\n    &appsv1.Deployment{}: 5 * time.Minute,  // Custom per type\n    &corev1.Pod{}: 1 * time.Hour,\n  }),\n)\n```\n\n**Mitigation:**\n- Tune resync period based on acceptable reconciliation lag\n- Monitor: `num_requeues` histogram for controller-specific metrics\n- Use multiple controller replicas to spread load\n\n**File References:**\n- staging/src/k8s.io/client-go/informers/factory.go:78\u201384 \u2013 WithCustomResyncConfig\n\n---\n\n### 4.3 Design Tradeoffs\n\n#### 4.3.1 Type Safety vs. Flexibility\n\n**Typed Client (kubernetes.Clientset)**\n- **Pros:** Compile-time type checking, IDE autocomplete, no reflection overhead\n- **Cons:** Code generation required on every API change; CRDs must be separately imported\n- **Use Case:** Core Kubernetes APIs, well-defined API schemas\n\n**Dynamic Client (dynamic.DynamicClient)**\n- **Pros:** No code generation; works with any resource (CRDs, new APIs without restart)\n- **Cons:** map[string]interface{} manipulation; runtime errors only; slower (reflection)\n- **Use Case:** Operators handling diverse CRDs, admission webhooks, multi-cluster tooling\n\n**Trade-off Decision:**\nclient-go provides both; users choose per-resource or per-application. Typical pattern: typed for core, dynamic for CRDs.\n\n**File References:**\n- staging/src/k8s.io/client-go/kubernetes/ \u2013 Typed clientset\n- staging/src/k8s.io/client-go/dynamic/ \u2013 Dynamic client\n\n#### 4.3.2 Strong Consistency vs. Low Latency\n\n**Cache-Based (Informer)**\n- **Pros:** Sub-millisecond query latency; no API server load; local filtering\n- **Cons:** Eventual consistency; cache can be stale by minutes (network lag, processing delay)\n- **Use Case:** Listing, filtering, latency-sensitive queries\n\n**Direct API Calls**\n- **Pros:** Strongly consistent; latest truth\n- **Cons:** Network latency (50ms+); API server load; no local filtering\n- **Use Case:** Critical decisions (deletion, mutation), rare queries\n\n**Common Pattern:**\n- Query cache for fast reads: `podLister.Pods(\"ns\").List(selector)`\n- On write, refetch to verify: `clientset.CoreV1().Pods(ns).Get(ctx, name, opts)`\n\n**File References:**\n- staging/src/k8s.io/client-go/tools/cache/store.go \u2013 Cache interface\n- staging/src/k8s.io/client-go/rest/request.go \u2013 Direct API calls\n\n#### 4.3.3 Single Watch vs. Multiplexed Watches\n\n**Multiplexed (SharedInformer)**\n- **Pros:** One watch per resource type; efficient for multiple handlers; automatic deduplication\n- **Cons:** Shared event channel; one slow handler blocks others\n- **Use Case:** Multiple unrelated controllers/handlers for same resource type\n\n**Independent Watches**\n- **Pros:** Isolation; one slow handler doesn't affect others\n- **Cons:** Duplicate watch streams; higher API server/network load\n- **Use Case:** Very different handlers with conflicting priorities\n\n**client-go Design Choice:**\n- Default: Multiplexed (SharedInformer factory)\n- Advanced: Can create independent Reflectors if needed\n\n**File References:**\n- staging/src/k8s.io/client-go/informers/factory.go:57\u201370 \u2013 Deduplication\n\n#### 4.3.4 Optimistic Concurrency vs. Distributed Locks\n\n**Optimistic (ResourceVersion Compare)**\n- **Pros:** No extra lock object; every API object is a mini-lock via resourceVersion\n- **Cons:** Conflicts when concurrent writers; requires retry/backoff logic\n- **Use Case:** Default for Kubernetes objects; server-side apply\n\n**Pessimistic (LeaderElection/Lease)**\n- **Pros:** Guaranteed exclusion; only one writer can proceed\n- **Cons:** Extra coordination object; clock skew risks; failover latency\n- **Use Case:** Controller leadership; data-critical sections\n\n**client-go Provides Both:**\n- Optimistic: PATCH/Update with resourceVersion\n- Pessimistic: LeaderElection with Lease\n\n**File References:**\n- staging/src/k8s.io/client-go/rest/request.go:500\u2013600 \u2013 PATCH/Update semantics\n- staging/src/k8s.io/client-go/tools/leaderelection/ \u2013 Leader election\n\n---\n\n## 5. # Kubernetes client-go: Advanced Systems Guide

## Executive Summary

client-go is the canonical Go client library for accessing Kubernetes clusters. It provides three distinct client models (typed, dynamic, and metadata clients) layered on top of a foundational REST client, combined with sophisticated caching, watch-based informer machinery, and controller infrastructure. The library decouples API interaction from business logic through strategic architectural boundaries: the REST transport layer handles HTTP concerns; the Reflector + DeltaFIFO + Indexer stack converts raw watch events into a queryable, eventually-consistent cache; and the SharedInformer facade multiplexes this cache to multiple event handlers. Controllers decouple from informers via workqueues, enabling resilient reconciliation patterns.
