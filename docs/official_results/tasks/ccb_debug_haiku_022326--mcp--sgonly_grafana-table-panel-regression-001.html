## Summary of Findings

**The regression is in the v38 migration function** (`apps/dashboard/pkg/migration/schemaversion/v38.go`, lines 115-131) which introduces a **structural inconsistency** in table panel field configurations.

### The Core Issue

The migration code only processes `fieldConfig.defaults` **if** `defaults.custom` already exists:

```go
if custom, ok := defaults["custom"].(map[string]interface{}); ok {  // ← CONDITION FAILS
    // Migration skipped if custom doesn't exist
}
migrateOverrides(fieldConfig)  // ← BUT THIS ALWAYS RUNS
```

### Why Dashboards Without Explicit `defaults.custom` Break

1. **During defaults application:** `applyPanelDefaults()` creates `fieldConfig.defaults` but NOT `fieldConfig.defaults.custom`
2. **During v38 migration:** 
   - Dashboards WITH explicit custom → both defaults and overrides are migrated ✓
   - Dashboards WITHOUT explicit custom → only overrides are migrated, creating structural mismatch ✗
3. **Result:** Overrides reference `custom.cellOptions` but `defaults.custom` never exists, causing the field configuration to be silently dropped during rendering

### Key Evidence

- **Migration code:** Lines 115-131 in `v38.go` - conditional check on line 117 fails for panels without explicit custom
- **Test case:** `v38_test.go` lines 444-596 demonstrating the exact regression scenario
- **Tracking logic:** `trackOriginalFieldConfigCustom()` in `frontend_defaults.go` marks only originally-saved custom objects, leaving missing ones unprotected

The investigation report provides the complete architectural analysis with file paths and line numbers for implementation review.
