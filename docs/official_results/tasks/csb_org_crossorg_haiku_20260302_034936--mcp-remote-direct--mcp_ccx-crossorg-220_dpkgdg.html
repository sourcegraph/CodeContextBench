## Key Findings:

**Service Lifecycle Pattern:**
Apache Kafka implements a hierarchical, coordinated lifecycle management system where:

1. **Forward Initialization Order** (startup):
   - Worker → Status Backing Store → Config Backing Store
   - Each service depends on previous ones being initialized

2. **Reverse Shutdown Order** (graceful shutdown):
   - Config Backing Store → Status Backing Store → Worker
   - Services are stopped in reverse dependency order

3. **Main Components:**
   - **Herder interface** - defines start()/stop() contract
   - **AbstractHerder** - provides startServices()/stopServices() template methods
   - **DistributedHerder** - implements distributed coordination with group membership
   - **Connect class** - top-level orchestrator managing Herder, REST server, and shutdown hooks

4. **Backing Stores** - all follow the same lifecycle pattern:
   - OffsetBackingStore (Kafka/File/Memory implementations)
   - ConfigBackingStore (Kafka/Memory implementations)
   - StatusBackingStore (Kafka/Memory implementations)

5. **Coordinated Shutdown:**
   - Distributed nodes coordinate via consumer group protocol
   - Graceful shutdown with configurable timeouts (60+ seconds)
   - Extension points for custom resource cleanup via `uponShutdown` list
