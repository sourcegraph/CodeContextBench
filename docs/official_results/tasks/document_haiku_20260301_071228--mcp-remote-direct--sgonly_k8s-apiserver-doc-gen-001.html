Subsystem Purpose, Boundaries, and Dependencies\n\n### Purpose\nThe `k8s.io/apiserver` library provides a generic, pluggable framework for building Kubernetes-compatible API servers. It encapsulates the core logic needed to:\n- **Serve REST APIs** over HTTP/HTTPS with OpenAPI documentation\n- **Authenticate and authorize** incoming requests\n- **Validate and admit** resources via pluggable admission controllers\n- **Store and retrieve** resources durably in a backend (etcd by default)\n- **Stream changes** via the watch mechanism\n- **Manage API versioning** and schema evolution\n- **Provide observability** through audit logs, metrics, and tracing\n\nThe library is designed to support three primary deployment patterns:\n1. **Core API Server**: Kubernetes' built-in API server\n2. **API Extensions**: Custom resource definitions (CRDs) via the `apiextensions-apiserver`\n3. **Aggregated APIs**: Third-party API servers proxied behind the aggregation layer (`kube-aggregator`)\n\n### Boundaries\nThe apiserver library sits at the intersection of several architectural layers:\n\n**Upstream Dependencies:**\n- `k8s.io/apimachinery`: Provides `runtime.Scheme`, `TypeMeta`, `ObjectMeta`, and generic serialization/deserialization\n- `k8s.io/client-go`: Supplies clients and informers for controller communication\n- `etcd3/clientv3`: Backend storage communication\n- `k8s.io/component-base`: Common utilities (logging, metrics, feature gates)\n\n**Downstream Consumers:**\n- API server processes that embed the library (kube-apiserver, kube-aggregator, extension API servers)\n- Client tools (kubectl) that consume the served APIs and discovery endpoints\n- Controllers and webhooks that extend API server behavior\n\n**Responsibility Limits:**\n- The library does NOT implement the core Kubernetes business logic (that lives in kube-apiserver)\n- It does NOT manage cluster infrastructure (kubelet, kube-controller-manager are separate)\n- It does NOT provide the storage backend itself; it defines interfaces that backends must implement\n\n---\n\n## 2. Key Components and Responsibilities\n\n### 2.1 GenericAPIServer\n**Location**: `pkg/server/genericapiserver.go`\n\nThe central orchestrator that owns the HTTP server lifecycle, handler chain assembly, and API group installation. Key responsibilities:\n\n- **Server Lifecycle**: Manages startup, shutdown gracefully, health checking\n- **Request Routing**: Directs incoming HTTP requests through a pluggable filter chain\n- **API Registration**: Installs API groups via the `InstallAPIGroup()` method\n- **Discovery Serving**: Publishes `/apis` and `/openapi/v2|v3` endpoints\n\n**Key Fields:**\n```go\ntype GenericAPIServer struct {\n    Handler *APIServerHandler             // HTTP request dispatcher\n    SecureServingInfo *SecureServingInfo   // TLS configuration\n    admissionControl admission.Interface    // Pluggable admission chain\n    LoopbackClientConfig *restclient.Config // Privileged internal client\n}\n```\n\nThe `GenericAPIServer` does not directly handle individual requests. Instead, it assembles a middleware stack and delegates to the `APIServerHandler`, which manages routing between go-restful and custom mux.\n\n### 2.2 Handler Chain (Request Filtering)\n**Location**: `pkg/server/config.go` (DefaultBuildHandlerChain), `pkg/endpoints/filters/`, `pkg/server/filters/`\n\nThe handler chain is a middleware stack that processes every HTTP request. It is constructed in `DefaultBuildHandlerChain()` and consists of ~20+ filters applied in a specific order.\n\n**Filter Chain (top to bottom, outermost to innermost):**\n\n1. **Panic Recovery** (`WithPanicRecovery`): Catches panics and returns HTTP 500\n2. **Audit Initialization** (`WithAuditInit`): Establishes the audit event context\n3. **Request Info & Tracing** (`WithRequestInfo`, Headers** (`WithHSTS`): Security headers for HTTPS\n6. **Shutdown Handling** (`WithRetryAfter`, `WithWatchTerminationDuringShutdown`): Signals during graceful shutdown\n7. **Request Timeout** (`WithTimeoutForNonLongRunningRequests`, `WithRequestDeadline`): Enforces timeouts\n8. **Wait Groups** (`WithWaitGroup`): Tracks request lifecycle for graceful shutdown\n9. **Warning Recorder** (`WithWarningRecorder`): Collects deprecation warnings\n10. **CORS** (`WithCORS`): Handles cross-origin requests\n11. **Authentication** (`WithAuthentication`): Identifies the user (x509 cert, bearer Custom servers can replace `DefaultBuildHandlerChain` with their own stack\n\n### 2.3 Authentication & Authorization\n**Location**: `pkg/authentication/`, `pkg/authorization/`\n\nAuthentication and BEFORE an object is persisted. They have two phases:\n\n**Mutation Phase** (`MutationInterface.Admit`):\n- Plugins can modify the incoming object (e.g., set defaults, inject sidecar containers)\n- Executes in order; first error is fatal\n- Has access to the deserialized object\n\n**Validation Phase** (`ValidationInterface.Validate`):\n- Plugins validate the object but CANNOT modify it\n- Also first-error-fatal\n- Commonly used for policy enforcement (Pod Security Policies, CEL-based rules)\n\n**Key Interfaces** (`pkg/admission/interfaces.go`):\n- `Attributes`: Encapsulates the request context (user, resource, operation, object)\n- `ObjectInterfaces`: Provides converters, defaulters, etc. needed to manipulate objects\n- `MutationInterface` and `ValidationInterface`: Plugin contracts\n\n**Built-in Examples**:\n- `PodSecurityAdmission`: Enforces Pod Security Standards per namespace\n- Webhook admission controllers: External policy evaluation\n\n**Design Tradeoff**: Admission runs after deserialization, so malformed JSON is rejected before any plugin runs, reducing attack surface.\n\n### 2.5 REST Storage and Endpoints\n**Location**: `pkg/registry/generic/`, `pkg/endpoints/installer.go`, `pkg/storage/`\n\n**REST Storage** (`rest.Storage` interface):\n- Abstracts the backend (etcd, memory, etc.)\n- Implements CRUD operations: Create, Update, Delete, Get, List, DeleteCollection, Watch\n- Each resource type has its own storage implementation\n\n**Key Storage Implementations**:\n- `genericregistry.Store`: Generic REST storage backed by etcd; used for most resources\n- Custom implementations: Some resources need special handling (e.g., `endpoints` uses a ServiceRegistry)\n\n**Endpoints Installer** (`APIInstaller`):\n- Takes an `APIGroupInfo` and dynamically registers HTTP handlers via go-restful\n- Generates paths like `/api/v1/pods`, `/apis/apps/v1/deployments`\n- Registers subresources (e.g., `/pods/{name}/logs`, `/pods/{name}/exec`)\n- Installs discovery endpoints for client introspection\n\n**Flow**:\n```\nAPIGroupInfo \u2192 APIInstaller.Install() \u2192 go-restful WebService\n  \u2192 HTTP Handler \u2192 REST Storage \u2192 etcd\n```\n\n### 2.6 Watch Cache\n**Location**: `pkg/storage/cacher/`\n\nTo handle high watch load without overwhelming etcd, the apiserver caches all objects in memory and streams changes.\n\n**Key Classes**:\n- `Cacher`: Wraps the underlying storage; maintains in-memory cache\n- `watchCache`: Ring buffer of recent events indexed by resource version\n- `cacheWatcher`: Per-client watcher that pulls from the cache\n\n**Initialization**:\n1. List all objects from storage to establish baseline state and resource version\n2. Start a WATCH from that resource version\n3. Stream events through the cache to clients\n\n**Optimization**:\n- **Bookmarks**: Cache sends periodic \"bookmark\" events with the latest resource version even for unchanged objects, preventing clients from \"falling off\" the cache (requiring a relist)\n- **Consistent Reads**: For non-watch requests, the cache checks etcd's current revision and waits for the cache to catch up before serving\n\n**Tradeoff**: Uses significant memory (typically 1-10 GB for a large cluster) to reduce etcd load by 100x or more.\n\n### 2.7 API Group and Resource Registration\n**Location**: `pkg/server/config.go` (APIGroupInfo), `pkg/runtime/schema`\n\n**APIGroupInfo** struct:\n```go\ntype APIGroupInfo struct {\n    PrioritizedVersions []schema.GroupVersion\n    VersionedResourcesStorageMap map[string]map[string]rest.Storage\n    Scheme *runtime.Scheme\n    NegotiatedSerializer runtime.NegotiatedSerializer\n    ParameterCodec runtime.ParameterCodec\n}\n```\n\nThis bundles:\n- **Scheme**: Type registry mapping `GroupVersionKind \u2194 Go type`; handles conversions and defaulting\n- **Storage**: Map of version \u2192 resource \u2192 backend storage\n- **Version Priority**: Order in which versions are preferred (v1 before v1beta1, etc.)\n\n**Registration Flow**:\n1. Define Go types (structs with `TypeMeta`, `ObjectMeta`)\n2. Generate boilerplate (deepcopy, conversion, defaulting) via `codegen`\n3. Populate `Scheme` with type info and conversion functions\n4. Create storage implementations (typically `genericregistry.Store`)\n5. Assemble `APIGroupInfo` and call `GenericAPIServer.InstallAPIGroup()`\n\n**Design**: The `Scheme` is the single source of truth for type information; it enables polymorphic request handling.\n\n---\n\n## 3. If found and version matches etcd, return immediately (very fast)\n     - If version stale, wait for cache to catch up\n   - **etcd Lookup**: Fall through to `etcd3/clientv3.Get()` if cache miss\n   - Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n**Authentication Failure**:\n- **Symptom**: 401 Unauthorized on every request\n- **Root Causes**:\n  - Cert/token invalid or expired\n  - Authenticator not configured (all fall through)\n  - Mismatch between client identity and server's trust anchors\n- **Recovery**: Regenerate credentials, reconfigure server\n\n**Authorization Failure**:\n- **Symptom**: 403 Forbidden\n- **Root Causes**:\n  - RBAC role missing or permission denied\n  - Webhook authorizer down or rejecting\n  - Audit shows \"verb=get, resource=pods, namespace=default\" but user lacks that permission\n- **Recovery**: Update RBAC roles or webhook logic\n\n**Admission Failure**:\n- **Symptom**: 400 Bad Request (validation) or mutation errors\n- **Root Causes**:\n  - Pod Security Policy violation\n  - Webhook admission controller rejects\n  - Object fails schema validation\n- **Recovery**: Fix object spec or relax policies\n\n**Watch Cache Staleness**:\n- **Symptom**: Watch clients see stale data or `ResourceVersion` falls off cache\n- **Conditions**:\n  - Cache ring buffer too small for event volume\n  - etcd writes very fast; cache can't keep up\n- **Mitigation**: Cache has a compactor that periodically discards old events; clients are redirected to relist if they fall off\n- **Code**: `pkg/storage/cacher/compactor.go`\n\n**etcd Unavailability**:\n- **Symptom**: All requests timeout or fail with \"etcd server unavailable\"\n- **Root Causes**: etcd cluster down, network partition, or load too high\n- **Watch Cache Behavior**:\n  - Reads: Cache continues serving stale data until timeout\n  - Writes: Fail immediately\n- **Recovery**: Restore etcd; apiserver caches help mitigate brief outages\n\n**Storage Conflict (409 Conflict)**:\n- **Symptom**: Client gets 409 on a PUT or PATCH\n- **Root Cause**: `resourceVersion` mismatch; another actor modified the object\n- **Required Action**: Client must re-read, resolve conflict, resubmit\n- **Code**: `pkg/storage/interfaces.go`, Preconditions check\n\n### 4.2 Limits and Capacity\n\n**Request Concurrency**:\n- **Default**: 400 read requests, 200 write requests in flight (legacy `maxInflight`)\n- **Modern**: Priority and Fairness system classifies requests into `PriorityLevel`s with seat counts\n- **Tradeoff**: Too low \u2192 requests queue/timeout; too high \u2192 server memory/etcd overload\n- **Configuration**: `--max-requests-inflight`, `--max-mutating-requests-inflight` flags\n- **Code**: `pkg/server/filters/priority-and-fairness.go`, `pkg/util/flowcontrol/`\n\n**Watch Cache Size**:\n- **Memory**: Entire cluster state in RAM; typical 1-5 GB for large clusters\n- **Ring Buffer**: Default ~100,000 events; older events are compacted/discarded\n- **Tradeoff**: Larger = better cache hit rate; smaller = less memory, more etcd load\n- **Configuration**: Per-resource via `--default-watch-cache-size`, `--watch-cache-sizes`\n- **Code**: `pkg/storage/cacher/watch_cache.go`\n\n**List Pagination**:\n- **Default Limit**: 500 items per list request (without `limit` param)\n- **Max Limit**: No hard max, but very large lists (10k+ items) degrade performance\n- **Tradeoff**: Smaller limit \u2192 more requests, lower memory; larger \u2192 fewer requests, higher memory\n- **Code**: `pkg/endpoints/handlers/list.go`, `pageSize` logic\n\n**API Version Support**:\n- **Concurrent Versions**: A server can serve multiple versions of the same API (e.g., v1, v1beta1, v1alpha1)\n- **Version Priority**: The `PrioritizedVersions` list determines which version is preferred for storage and discovery\n- **Tradeoff**: More versions \u2192 more conversion code, more testing; fewer \u2192 less flexibility for clients\n- **Code**: `pkg/server/config.go`, APIGroupInfo.PrioritizedVersions\n\n### 4.3 Design Tradeoffs\n\n**Lazy Deserialization**:\n- **Decision**: Request body is NOT parsed until after auth/authz pass\n- **Benefit**: Reduces exposure to malicious JSON; auth/authz are fast\n- **Cost**: Slightly delayed error reporting (malformed JSON caught by admission, not before)\n- **Code**: `pkg/server/config.go`, DefaultBuildHandlerChain order (auth before admission)\n\n**In-Memory Watch Cache**:\n- **Decision**: Every object is cached in RAM; no query goes directly to etcd for watches\n- **Benefit**: Massive reduction in etcd load (100x improvement common)\n- **Cost**: High memory usage; consistency challenges (cache may lag)\n- **Alternative**: Skip cache (slower watches, etcd overload)\n- **Code**: `pkg/storage/cacher/cacher.go`\n\n**Optimistic Concurrency Control**:\n- **Decision**: No locks; clients use `resourceVersion` for compare-and-set\n- **Benefit**: Highly scalable; no locking contention\n- **Cost**: Clients must handle conflicts (409 Conflict errors)\n- **Code**: `pkg/storage/interfaces.go`, Preconditions\n\n**Server-Side Apply**:\n- **Decision**: Clients can POST a declarative patch; server tracks field ownership via `managedFields`\n- **Benefit**: Multiple actors can manage different fields without overwrites\n- **Cost**: Significant complexity; special conflict resolution rules\n- **Code**: `pkg/endpoints/handlers/fieldmanager/`, `managedFields` logic\n\n**Pluggable Admission**:\n- **Decision**: Admission controllers are external plugins, not built-in\n- **Benefit**: Extensibility; operators can add custom policies\n- **Cost**: Ordering is critical; a buggy plugin can break the whole server\n- **Mitigations**: Webhook timeouts, dry-run mode\n- **Code**: `pkg/admission/`, `pkg/admission/plugin/`\n\n---\n\n## 5. Extension Points and Customization Hooks\n\n### 5.1 Extension Point: Custom Authenticators\n\n**Hook**: `Config.Authentication.Authenticator`\n\n**Interface** (`pkg/authentication/authenticator/interfaces.go`):\n```go\ntype Request interface {\n    AuthenticateRequest(req *http.Request) (*Response, bool, error)\n}\n\ntype Response struct {\n    User user.Info\n    // ...\n}\n```\n\n**Usage**:\n- Implement `Request.AuthenticateRequest()` to extract identity from the HTTP request\n- Return `user.Info` with username, UID, groups, and extra claims\n- Return `false` if your authenticator doesn't recognize the request (pass to next)\n- Return `error` if recognition failed (e.g., invalid token format)\n\n**Example**: A custom OIDC authenticator that validates JWTs from a private issuer.\n\n**Risk**: A broken authenticator can lock out all users. Test thoroughly in dry-run mode first.\n\n### 5.2 Extension Point: Custom Authorizers\n\n**Hook**: `Config.Authorization.Authorizer`\n\n**Interface** (`pkg/authorization/authorizer/interfaces.go`):\n```go\ntype Authorizer interface {\n    Authorize(ctx context.Context, a Attributes) (Decision, Reason, Error)\n}\n\ntype Attributes interface {\n    GetUser() user.Info\n    GetVerb() string\n    GetResource() string\n    GetNamespace() string\n    // ... more fields\n}\n\ntype Decision int\nconst (\n    DecisionAllow Decision = iota\n    DecisionDeny\n    DecisionNoOpinion\n)\n```\n\n**Usage**:\n- Implement `Authorize()` to check if a (user, action) pair is allowed\n- Return `DecisionAllow`, `DecisionDeny`, or `DecisionNoOpinion`\n- If you return `NoOpinion`, the next authorizer in the chain is consulted\n- First `Deny` wins; if all abstain, request is denied\n\n**Example**: A custom authorizer that enforces resource quotas based on department tags.\n\n**Risk**: A broken authorizer may deny everything or allow everything. Always test with a fail-safe mode.\n\n### 5.3 Extension Point: Admission Webhooks\n\n**Hook**: Admission webhook configuration (not a direct code hook, but via API)\n\n**Flow**:\n1. Define an HTTP server that implements the `AdmissionReview` API\n2. Register it via `ValidatingWebhookConfiguration` or `MutatingWebhookConfiguration` resources\n3. When requests arrive, apiserver makes HTTP POST to the webhook\n4. Webhook returns accept/reject and optionally mutates the object\n\n**Advantages over Built-in Plugins**:\n- Can be deployed independently (not in-process)\n- Can be updated without restarting apiserver\n- Multiple webhooks can run concurrently\n\n**Risks**:\n- Network dependency; webhook outage = admission outage\n- Admission timeout (default 30s) can cause slow requests\n- Complex serialization/deserialization of `AdmissionReview` messages\n\n**Code Reference**: Webhooks are invoked in `pkg/admission/plugin/` via webhook admission controller.\n\n### 5.4 Extension Point: Custom REST Storage\n\n**Hook**: `APIGroupInfo.VersionedResourcesStorageMap[version][resource]`\n\n**Interface** (`pkg/registry/rest/rest.go`):\n```go\ntype Storage interface {\n    New() runtime.Object\n    Destroy()\n}\n\ntype Creater interface {\n    Create(ctx context.Context, obj runtime.Object, options *CreateOptions) (runtime.Object, error)\n}\n\ntype Getter interface {\n    Get(ctx context.Context, name string, options *GetOptions) (runtime.Object, error)\n}\n\ntype Lister interface {\n    List(ctx context.Context, options *ListOptions) (runtime.Object, error)\n}\n\ntype Watcher interface {\n    Watch(ctx context.Context, options *ListOptions) (watch.Interface, error)\n}\n// ... and other interfaces\n```\n\n**Usage**:\n- Implement `Storage` and any combination of `Creater`, `Getter`, `Lister`, etc.\n- Return objects serialized as you wish; apiserver handles HTTP wrapping\n- Handle conflict resolution, resource versioning, etc.\n\n**Example**: A custom storage backend that reads from a file system instead of etcd, or one that delegates to an external service.\n\n**Typical Implementation**: `genericregistry.Store` wraps etcd3 storage and adds caching, versioning, and precondition checking. Most resources use this; custom implementations are rare.\n\n**Risks**:\n- Responsible for correctness (no double-writes, proper versioning)\n- Must handle concurrency (optimistic locking)\n- If broken, the resource becomes unusable\n\n### 5.5 Extension Point: Custom Handler Chain\n\n**Hook**: `Config.BuildHandlerChainFunc`\n\n**Signature**:\n```go\nBuildHandlerChainFunc func(apiHandler http.Handler, c *Config) (secure http.Handler)\n```\n\n**Usage**:\n- Replace the default `DefaultBuildHandlerChain` function\n- Insert custom filters before/after authentication, authorization, admission, etc.\n- Example: inject a rate-limiting filter per-namespace, or add a custom security header\n\n**Risks**:\n- Easy to break authentication/authorization order\n- Filters must be idempotent and non-blocking\n- High-level middleware errors can crash the server\n\n**Example in Code**: `pkg/server/config.go`, lines 1015-1096 show `DefaultBuildHandlerChain`; you can copy and modify this.\n\n### 5.6 Extension Point: Informers and Controllers\n\n**Not** a direct apiserver hook, but apiserver is designed to be controlled by external controllers.\n\n**Pattern**:\n1. Create an informer from a client: `informers.NewSharedInformerFactory(clientset, resyncDuration)`\n2. Install event handlers: `informer.Core().V1().Pods().Informer().AddEventHandler(...)`\n3. Run the informer: `stopCh := make(chan struct{}); go informer.Start(stopCh)`\n4. Controllers react to events (Create, Update, Delete)\n\n**apiserver Support**:\n- Watch caching is optimized for controller efficiency\n- Bookmarks prevent cache falloff\n- API versioning allows controllers to use preferred versions\n\n---\n\n## 6. Operational Implications\n\n### 6.1 Performance Considerations\n\n**Watch Cache Hit Rate**:\n- **Metric**: Percentage of list/get requests served from cache vs. etcd\n- **Impact**: 99%+ cache hit rate = ~100x etcd load reduction\n- **Tuning**: Increase `--watch-cache-size` (default 100k events) if hit rate drops\n- **Monitoring**: Prometheus metric `apiserver_cache_list_fetched_objects_*`\n\n**Admission Webhook Latency**:\n- **Impact**: Each webhook adds HTTP round-trip latency (10-100ms typical)\n- **Tuning**: Set reasonable timeouts (`--admission-control-config-file`); fail-open or fail-closed per webhook\n- **Monitoring**: Prometheus metric `apiserver_admission_webhook_admission_duration_seconds_*`\n\n**Serialization Overhead**:\n- **Impact**: JSON/protobuf serialization is CPU-bound; large objects (GBs) can be slow\n- **Optimization**: Protobuf is faster than JSON; use CBOR for even denser encoding\n- **Monitoring**: `apiserver_request_duration_seconds` broken down by verb, resource, code\n\n### 6.2 Reliability and Failover\n\n**Graceful Shutdown**:\n- **Trigger**: SIGTERM signal\n- **Process**:\n  1. Stop accepting new requests (`lifecycleSignals.NotAcceptingNewRequest`)\n  2. Send `429 Too Many Requests` for new requests\n  3. Wait for existing requests to complete (configurable timeout, default 3s per phase)\n  4. Close connections gracefully\n- **Code**: `pkg/server/signal.go`, `pkg/server/secure_serving.go`\n- **Tuning**: `--shutdown-delay-duration`, `--shutdown-send-retry-after`\n\n**Health Checks**:\n- **Liveness** (`/livez`): Server is running; failing = crash restart\n- **Readiness** (`/readyz`): Server is ready to serve traffic; failing = remove from load balancer\n- **Default Checks**: Storage ready, dependency services ready\n- **Extension**: `Config.HealthzChecks` allows adding custom checks\n- **Code**: `pkg/server/healthz.go`, `pkg/server/routes/healthz.go`\n\n**High Availability**:\n- Multiple apiserver replicas behind a load balancer\n- etcd cluster (3+ nodes) for durability\n- Watch cache acts as shock absorber during etcd hiccups\n- Watch cache is per-apiserver; clients may be redirected to another replica on failover\n\n### 6.3 Security Implications\n\n**Admission Webhook Security**:\n- Webhooks can read all objects passing through the server\n- Risk: Data exfiltration if webhook endpoint compromised\n- Mitigation: Use TLS, mutual auth; keep webhook logic simple; audit webhook calls\n\n**Custom Authenticator Security**:\n- Authenticators have access to raw HTTP requests and auth credentials\n- Risk: Credential logging, token theft\n- Mitigation: No logging of sensitive data; use HTTPS only; rotate keys\n\n**Audit Logging**:\n- Every request can be logged (configurable verbosity)\n- Includes request/response bodies (potential PII exposure)\n- Mitigation: Use audit policy to limit what's logged; encrypt audit logs at rest\n- **Code**: `pkg/audit/`, define audit policy in `--audit-policy-file`\n\n**RBAC Enforcement**:\n- RBAC is the default authorization mechanism\n- Risk: Misconfigured roles grant too much access\n- Mitigation: Principle of least privilege; regular audit of roles\n- **Code**: `pkg/authorization/` handles RBAC delegation (actual rules in kube-apiserver)\n\n### 6.4 Observability\n\n**Metrics** (Prometheus):\n- `apiserver_request_total`: Total requests by method, path, status\n- `apiserver_request_duration_seconds`: Request latency distribution\n- `apiserver_storage_db_total_size_bytes`: etcd database size\n- `apiserver_cache_list_fetched_objects_*`: Watch cache efficiency\n- `apiserver_audit_event_total`: Audit events logged\n\n**Logs**:\n- Structured logging with `klog` (V(2) = debug, V(4) = verbose)\n- Enable with `--v=2` flag\n- Key events: request arrival, admission decisions, storage operations\n\n**Tracing** (Optional, feature-gated):\n- Distributed tracing via OpenTelemetry\n- Trace follows request through auth, admission, storage layers\n- Enable with `--tracing-config-file`\n- **Code**: `pkg/endpoints/filterlatency/`, `pkg/server/config.go` (WithTracing)\n\n**Audit Logs**:\n- Policy-driven; can log full request/response bodies\n- Outputs to file, webhook, log stream, or custom backend\n- **Code**: `pkg/audit/`\n- Example audit policy (from ARCHITECTURE.md): Different verbosity levels for different resources\n\n---\n\n## 7. # Kubernetes apiserver Library: Architecture and Extension Guide

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose
The `k8s.io/apiserver` library provides a generic, pluggable framework for building Kubernetes-compatible API servers. It encapsulates the core logic needed to:
- **Serve REST APIs** over HTTP/HTTPS with OpenAPI documentation
- **Authenticate and authorize** incoming requests
- **Validate and admit** resources via pluggable admission controllers
- **Store and retrieve** resources durably in a backend (etcd by default)
- **Stream changes** via the watch mechanism
- **Manage API versioning** and schema evolution
- **Provide observability** through audit logs, metrics, and tracing

The library is designed to support three primary deployment patterns:
1. **Core API Server**: Kubernetes' built-in API server
2. **API Extensions**: Custom resource definitions (CRDs) via the `apiextensions-apiserver`
3. **Aggregated APIs**: Third-party API servers proxied behind the aggregation layer (`kube-aggregator`)

- `k8s.io/component-base`: Common utilities (logging, metrics, feature gates)
