## Security Finding

**Finding ID:** SEC-0472
**Severity:** Medium
**Category:** Information Leakage / Context Isolation Bypass
**Reporter:** External Security Researcher (via bug bounty)

Django's template system provides a mechanism to include sub-templates with isolated context — the intent is that the included template can ONLY access variables that are explicitly passed to it. However, the researcher demonstrated that certain special variables from the parent template "leak" into included templates even when context isolation is requested.

**Proof of concept:**

Parent template:
```html
{% for item in items %}
  {% include "child.html" only %}
{% endfor %}
```

Child template (`child.html`):
```html
{{ forloop.counter }}  <!-- Should be undefined with "only" but it renders! -->
```

Despite using the `only` keyword (which should create a completely clean, isolated context), the child template can access loop iteration variables from the parent's `{% for %}` tag. This violates the isolation contract and could lead to information leakage in multi-tenant template rendering scenarios.

## Task

Find and fix the template inclusion mechanism so that context isolation properly prevents ALL parent variables — including special variables injected by template tags — from leaking into isolated included templates.

**YOU MUST IMPLEMENT CODE CHANGES AND PRODUCE AN AUDIT LOG.**

## Summary

I have successfully completed the security audit and fixed the template context isolation vulnerability (SEC-0472) in Django. Here's what was accomplished:

### **The Vulnerability**
Django's `{% include "template.html" only %}` syntax was supposed to provide complete context isolation, but special variables like `forloop` from parent templates were leaking into included templates.

### **Root Cause**
The `context.new()` method was using a shallow copy of `render_context`, which could still reference parent context state.

### **The Fix**
I made three key changes:

1. **Modified `django/template/context.py`**:
   - Added `isolated=False` parameter to `Context.new()` and `RequestContext.new()`
   - When `isolated=True`, creates a fresh `RenderContext()` instead of shallow copying

2. **Modified `django/template/loader_tags.py`**:
   - Updated `IncludeNode.render()` to call `context.new(values, isolated=True)` for includes with the `only` keyword

3. **Added test in `tests/template_tests/test_context.py`**:
   - New test `test_include_isolated_forloop_context()` verifies the vulnerability is fixed

# Security Fix Summary: Template Context Isolation Vulnerability (SEC-0472)

## Vulnerability Description

Django's template system provides the `{% include "template.html" only %}` syntax to include sub-templates with an isolated context. The intent is that the included template can ONLY access variables explicitly passed to it via the `with` keyword.

However, a security researcher discovered that certain special variables from the parent template "leak" into included templates even when context isolation is requested.

### Proof of Concept

```django
{# Parent Template #}
{% for item in items %}
  {% include "child.html" only %}
{% endfor %}

{# Child Template (child.html) #}
{{ forloop.counter }}  {# Should be undefined with "only" but it renders! #}
```

Despite using the `only` keyword, the child template can access `forloop.counter` from the parent's for loop, violating the isolation contract.

## Root Cause Analysis

The issue originates in how `context.new()` creates isolated contexts:

1. **IncludeNode.render()** calls `context.new(values)` when `isolated_context=True`
2. **Context.new()** creates a new context by:
