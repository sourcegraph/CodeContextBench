## Key Findings

**Root Cause:** A metric state loss bug in `storage/remote/queue_manager.go:shards.start()` at line 1241, where `pendingSamples.Set(0)` unconditionally resets the counter without synchronizing with concurrent sample sends and metric updates.

**The Race Condition:**
1. During resharding, `stop()` closes the `softShutdown` channel and waits for samples to flush
2. There's a synchronization window where samples can still be enqueued before the write lock is taken
3. When hard shutdown occurs (timeout), pending samples are dropped and metrics are decremented asynchronously
4. `start()` is called immediately after, which forcefully resets the counter to 0, losing any in-flight metric updates
5. Samples become "stuck" in the pending metric when distribution changes due to new shard count

**Why It's Intermittent:**
- Only occurs when target discovery changes align with resharding triggers
- Requires the flush deadline to expire (forcing hard shutdown)
- Timing must align so `updateMetrics()` calls are in-flight when `start()` resets counters
- Series ref reassignment to different shards during target changes compounds the problem
