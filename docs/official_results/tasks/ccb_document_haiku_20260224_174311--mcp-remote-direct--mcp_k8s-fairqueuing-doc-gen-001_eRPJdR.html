The 3ms default is a heuristic for read-only requests; write estimators use different logic.\n\n#### Queue Depth Limit and Rejection Logic\n\n`QueueLengthLimit` is **per-queue**, not global.\n- With Q queues, max waiting = Q * QueueLengthLimit.\n- **Tradeoff:** Allows tuning of acceptable queue depth without global synchronization, but doesn't directly cap total resource consumption (memory for request objects).\n\n### 5.3 Edge Cases and Anti-Patterns\n\n1. **Empty Queues and Virtual Time Stall**\n   If all queues are empty, `getVirtualTimeRatio()` returns 0, so virtual time does not advance.\n   - Consequence: when the first request arrives, it gets `arrivalR = currentR`, and if the queue was idle, `nextDispatchR` is reset to `currentR`, so the request is dispatched immediately.\n   - **Not a problem:** the algorithm is designed to handle this.\n\n2. **SeatSeconds Overflow**\n   Despite uint64 being huge (~585 years at 1 ns/tick), `advanceEpoch()` is called whenever `currentR >= highR` to prevent overflow.\n   - Metric: `apiserver_flowcontrol_epoch_advance_total` tracks how many times this happens.\n   - **Operational concern:** A very long-lived apiserver with high concurrency could theoretically advance epochs frequently, but this is expected and non-disruptive.\n\n3. **Request Cancellation vs. Timeout**\n   Both map to `decisionCancel`. # API Priority and Fairness QueueSet: Algorithmic Deep-Dive

## Executive Summary

The Kubernetes API Priority and Fairness (APF) QueueSet is a sophisticated fair queuing subsystem that implements **fair queuing for server requests**, adapting networking fair queuing algorithms to API server concurrency management. It enforces per-priority-level concurrency limits while ensuring that no flow of requests can starve others, even under load. The system uses virtual time tracking, shuffle sharding for queue assignment, and seat-based resource accounting to provide both isolation and fairness.

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose
The QueueSet subsystem serves one core responsibility: **manage queuing and dispatch of requests for a single Priority Level Configuration** in the APF system. Given a concurrency limit (seat count) and a desired number of queues, QueueSet decides:
- Which requests get seats to execute immediately
- Which requests enter a queue to wait
- Which requests are rejected (queue full, no available seats)
- When queued requests are dispatched in what order (fair order based on virtual finish time)
