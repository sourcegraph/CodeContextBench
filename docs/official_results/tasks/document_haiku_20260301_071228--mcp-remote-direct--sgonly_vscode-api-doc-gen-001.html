# VS Code Diagnostic API Reference

## Overview

The Diagnostic API allows VS Code extensions to report code issues to users. Diagnostics represent problems such as compilation errors, linting violations, or other code quality concerns. Each diagnostic is associated with a specific location in a document and includes a severity level, message, and optional metadata.

The API is accessed through the `vscode.languages` namespace and provides:
- **Diagnostic**: A class representing a single diagnostic issue
- **DiagnosticCollection**: A container for managing diagnostics across documents
- **DiagnosticSeverity**: An enum for severity levels
- **DiagnosticRelatedInformation**: Additional context for diagnostics
- **DiagnosticTag**: Metadata tags for categorizing diagnostics
- **Functions**: Factory functions and query methods
- **Events**: Notifications when diagnostics change

---

## Core Types

### DiagnosticSeverity

An enumeration that defines the severity levels for diagnostics.

**Signature:**
```typescript
export enum DiagnosticSeverity {
    Error = 0,
    Warning = 1,
    Information = 2,
    Hint = 3
}
```

**Values:**

- **Error** (0): Something not allowed by the rules of a language or other means. Displayed prominently in red.
- **Warning** (1): Something suspicious but allowed. Displayed in yellow/orange.
- **Information** (2): Something to inform about but not a problem. Displayed in blue.
- **Hint** (3): Something to hint at a better way of doing it, like proposing a refactoring. Displayed subtly.

**Usage Notes:**
- The default severity for a diagnostic is `Error` if not specified in the constructor
- Severity affects visual styling in the editor (colors, icons, gutter decorations)
- Severity is also used to organize diagnostics in the Problems panel
