It abstracts away HTTP mechanics, authentication complexity, and API conventions, providing both strongly-typed and dynamic interfaces to the API server.\n\n### Boundaries\n\n**What client-go owns:**\n- HTTP transport configuration and request/response handling (`rest/` package)\n- Configuration loading from kubeconfig files (`tools/clientcmd/`)\n- API discovery (determining what resources exist and their versions)\n- Typed, generated clients for built-in API groups (`kubernetes/` package)\n- Dynamic clients for unstructured data and Custom Resources (`dynamic/` package)\n- Controller infrastructure: list/watch sources, caching, event distribution (`tools/cache/`)\n- Work queue patterns for asynchronous reconciliation (`util/workqueue/`)\n- Leader election for high-availability controllers (`tools/leaderelection/`)\n- Server-side apply mechanics (`applyconfigurations/` package)\n\n**What client-go does NOT own:**\n- Controller reconciliation logic (user code responsibility)\n- Admission control or API server logic\n- Kubernetes cluster operations or scheduler behavior\n- Resource quotas or limit enforcement\n- Authentication/authorization policy definition\n\n### Upstream Dependencies\n\n- **`k8s.io/apimachinery`**: Core Kubernetes types, watch mechanisms, serialization, runtime object handling\n- **`k8s.io/api`**: Typed definitions for core API groups (apps, batch, core, etc.)\n- **`k8s.io/code-generator`**: Code generation tools for typed clients, listers, and informers\n- **`k8s.io/klog`**: Structured logging\n\n### Downstream Dependents\n\n- **Controllers**: Any reconciliation loop (Kubernetes controllers, operators, webhooks)\n- **CLI tools**: kubectl, kube-proxy, kubelet all use client-go\n- **Operators**: Custom controllers managing application deployments\n\n---\n\n## 2. Core Components\n\n#### 2.1 Configuration Management (`rest/config.go`, `tools/clientcmd/`)\n\n**Responsibility**: Load and merge kubeconfig files, apply command-line overrides, configure authentication and TLS.\n\n**Key types**:\n- `rest.Config`: In-memory representation of client configuration (host, auth credentials, TLS settings, QPS, burst)\n- `clientcmd.ClientConfig`: Interface for loading and resolving kubeconfig contexts\n\n**Critical fields in `rest.Config`**:\n- `Host`: API server endpoint\n- `BearerToken`, `BearerTokenFile`: Token authentication\n- `Username`, `Password`: Basic auth\n- `ExecProvider`, `AuthProvider`: Plugin-based authentication\n- `QPS`, `Burst`: Client-side rate limiting parameters\n- `TLSClientConfig`: Certificate-based authentication\n\n**Design decision**: Configuration loading is separate from client initialization. This allows multiple clients to share a single `Config` object, reducing resource overhead.\n\n#### 2.2 REST Client (`rest/client.go`, `rest/request.go`)\n\n**Responsibility**: Build and execute HTTP requests against the API server with proper serialization, error handling, and rate limiting.\n\n**Key types**:\n- `RESTClient`: Low-level HTTP client that enforces Kubernetes API conventions\n- `Request`: Builder pattern object for constructing requests\n\n**Request builder chain** (`rest/request.go:96-131`):\n```go\ntype Request struct {\n    verb           string          // GET, POST, PUT, DELETE, PATCH, WATCH\n    pathPrefix     string          // /api or /apis\n    resource       string          // pods, deployments, etc.\n    resourceName   string          // specific object name\n    namespace      string          // target namespace\n    subresource    string          // status, scale, log, exec\n    params         url.Values      // query parameters\n    headers        http.Header     // custom headers\n    body           io.Reader       // request body\n    // ... rate limiter, backoff, timeout, max retries\n}\n```\n\n**Design pattern**: Deferred execution. The builder is constructed incrementally, but the HTTP request is not sent until a terminal method like `.Into(&obj)` or `.Do()` is called. This allows different client types (typed, dynamic, discovery) to be built on a common HTTP foundation.\n\n**Example flow**:\n```\nclient.CoreV1().Pods(\"default\").Get(ctx, \"my-pod\", metav1.GetOptions{})\n\u2193\ntranslateGetOptions() constructs Query and Headers\n\u2193\nclient.Verb(\"GET\").Resource(\"pods\").Namespace(\"default\").Name(\"my-pod\").Do(ctx)\n\u2193\nBuildHTTPRequest() constructs *http.Request\n\u2193\nDoRaw() executes via http.Client.Do() and deserializes response\n```\n\n#### 2.3 HTTP Transport and Authentication (`rest/transport.go`, `transport/` package)\n\n**Responsibility**: Wire up HTTP transport with authentication middleware, TLS configuration, and request/response hooks.\n\n**Middleware chain** (roundtrippers are composed):\n1. Custom user-provided roundtrippers (via `WrapTransport`)\n2. Bearer token injection\n3. Basic auth or Impersonation\n4. Client certificates / TLS\n5. User-Agent header injection\n6. Compression handling\n7. Retry/backoff for transient failures\n\n**Key design**: Each authentication method is implemented as an `http.RoundTripper` that wraps the underlying transport. This keeps concerns modular and composable.\n\n#### 2.4 Discovery Client (`discovery/discovery_client.go`)\n\n**Responsibility**: Query the API server to determine what API groups, versions, and resources exist.\n\n**Key interfaces**:\n- `DiscoveryInterface`: Methods to fetch API groups (`ServerGroups()`) and resources (`ServerResources()`)\n- `CachedDiscoveryClient`: Wraps DiscoveryClient and caches results to disk (typically `~/.kube/cache/`)\n\n**How discovery works** (`discovery/discovery_client.go:49-71`):\n- `/api/v1` (legacy core API)\n- `/apis` (aggregated API group discovery)\n- `/apis/{group}/{version}` (resources in a specific version)\n- `/openapi/v3` (OpenAPI schema for structural information)\n\n**Design rationale**: Caching discovery results dramatically reduces API server load. Clients cache on disk and only refresh on explicit invalidation or TTL expiry.\n\n#### 2.5 Typed Clients (`kubernetes/clientset.go`, `kubernetes/typed/`)\n\n**Responsibility**: Provide compile-time type-safe access to core API resources (Pods, Deployments, Services, etc.).\n\n**Structure** (`kubernetes/clientset.go:140-150`):\n```go\ntype Clientset struct {\n    *discovery.DiscoveryClient\n    appsV1         *appsv1.AppsV1Client\n    batchV1        *batchv1.BatchV1Client\n    corev1         *corev1.CoreV1Client\n    // ... one client per API group version\n}\n```\n\n**Generated interfaces** (via `k8s.io/code-generator`):\n```go\n// Example: corev1.PodsGetter generates:\ntype PodsGetter interface {\n    Pods(namespace string) PodInterface\n}\n\ntype PodInterface interface {\n    Create(ctx context.Context, pod *v1.Pod, ...) (*v1.Pod, error)\n    Update(ctx context.Context, pod *v1.Pod, ...) (*v1.Pod, error)\n    Delete(ctx context.Context, name string, ...) error\n    Get(ctx context.Context, name string, ...) (*v1.Pod, error)\n    List(ctx context.Context, opts metav1.ListOptions) (*v1.PodList, error)\n    Watch(ctx context.Context, opts metav1.ListOptions) (watch.Interface, error)\n    // ... Patch, UpdateStatus, etc.\n}\n```\n\n**Code generation**: These clients are not hand-written. Running `k8s.io/code-generator` on API type definitions automatically produces:\n- CRUD client methods\n- Listers (cached, read-only accessors)\n- Informers (watch + cache machinery)\n- Apply configuration builders (server-side apply)\n\nThis ensures type safety while avoiding massive boilerplate.\n\n#### 2.6 Dynamic Client (`dynamic/interface.go`, `dynamic/simple.go`)\n\n**Responsibility**: Provide untyped, runtime-safe access to any API resource (core, CRDs, aggregated APIs).\n\n**Key interface** (`dynamic/interface.go:29-50`):\n```go\ntype Interface interface {\n    Resource(resource schema.GroupVersionResource) NamespaceableResourceInterface\n}\n\ntype ResourceInterface interface {\n    Create(ctx context.Context, obj *unstructured.Unstructured, ...) (*unstructured.Unstructured, error)\n    Update(ctx context.Context, obj *unstructured.Unstructured, ...) (*unstructured.Unstructured, error)\n    Delete(ctx context.Context, name string, ...) error\n    List(ctx context.Context, opts metav1.ListOptions) (*unstructured.UnstructuredList, error)\n    Watch(ctx context.Context, opts metav1.ListOptions) (watch.Interface, error)\n    // ... Patch, Apply, etc.\n}\n```\n\n**Data representation**: All objects are `*unstructured.Unstructured`, which is a generic map-like structure containing raw JSON/CBOR data. The dynamic client relies on discovery to determine what API paths exist.\n\n**When to use**:\n- Working with CustomResourceDefinitions\n- Building generic tools (e.g., kubectl)\n- Handling resources not known at compile time\n\n**Tradeoff**: No compile-time type checking, but maximum flexibility.\n\n#### 2.7 Informers and Caching (`tools/cache/`, `informers/`)\n\n**Purpose**: Maintain an in-memory cache of API objects and notify handlers when objects change.\n\n**Core components**:\n\n##### Reflector (`tools/cache/reflector.go`)\n\n**Responsibility**: Perform a LIST to get initial state, then WATCH for incremental updates.\n\n**Algorithm** (`tools/cache/reflector.go:200-350`):\n1. Execute `LIST` with `limit` and get objects + `resourceVersion`\n2. Start `WATCH` from that `resourceVersion`\n3. Stream watch events into the store\n4. On `WATCH` error or \"too old resourceVersion\":\n   - If error was \"resourceVersion too old\": use watch bookmarks to get fresher resourceVersion\n   - Exponential backoff (max 30s) before retry\n   - Exponential backoff resets if no error for 2 minutes\n\n**Key parameters** (`tools/cache/reflector.go:228-251`):\n```go\ntype ReflectorOptions struct {\n    Name            string              // For logging\n    ResyncPeriod    time.Duration       // Force refresh all cached objects\n    MinWatchTimeout time.Duration       // Minimum timeout for WATCH requests\n    Clock           clock.Clock         // For testing\n}\n```\n\n**Why the resync period?** Even if the WATCH stream is healthy, the cache may have drifted due to bugs or missed events. Periodic resync forces re-processing of all cached objects.\n\n##### DeltaFIFO (`tools/cache/delta_fifo.go`)\n\n**Responsibility**: Queue and accumulate changes (deltas) for each object.\n\n**Data model** (`tools/cache/delta_fifo.go:63-99`):\n```go\ntype Deltas []Delta\ntype Delta struct {\n    Type   DeltaType  // Added, Updated, Deleted, Sync, Replaced\n    Object interface{} // The actual Kubernetes object\n}\n```\n\n**Why deltas?** Controllers need to see the full sequence of changes to an object, not just its current state. DeltaFIFO queues are keyed by object name, so multiple changes to the same object before it's processed are coalesced into a Deltas slice.\n\n**Example**:\n- Object created: Queue contains `[{Added, Pod(v1)}]`\n- Object modified 3 times: Queue still contains key once, but Deltas = `[{Added, Pod(v1)}, {Updated, Pod(v2)}, {Updated, Pod(v3)}]`\n- Object deleted: Queue contains `[{Deleted, Pod(v3)}]`\n\n##### Indexer and ThreadSafeStore (`tools/cache/index.go`, `tools/cache/thread_safe_store.go`)\n\n**Responsibility**: Store cached objects, provide thread-safe access, and support secondary indexes for fast lookups.\n\n**Indexing example**:\n```go\n// NamespaceIndex: index objects by namespace\nIndexers{\n    NamespaceIndex: func(obj interface{}) ([]string, error) {\n        meta, _ := meta.Accessor(obj)\n        return []string{meta.GetNamespace()}, nil\n    },\n}\n```\n\n**Why indexing?** Controllers often need to list all objects in a namespace or with a certain label. Indexes provide O(1) lookup instead of O(n) full scans.\n\n##### SharedInformer (`tools/cache/shared_informer.go`)\n\n**Responsibility**: Orchestrate Reflector, DeltaFIFO, and Indexer; dispatch events to multiple handlers.\n\n**Key contract** (`tools/cache/shared_informer.go:41-102`):\n- Cache is **eventually consistent** with the API server\n- Notifications are ordered per-object but not cross-object\n- Handlers are called **after** cache updates\n- Handlers must process quickly (use workqueue for heavy lifting)\n\n**Handler registration**:\n```go\ninformer.AddEventHandler(cache.ResourceEventHandlerFuncs{\n    AddFunc: func(obj interface{}) { /* new object */ },\n    UpdateFunc: func(oldObj, newObj interface{}) { /* changed */ },\n    DeleteFunc: func(obj interface{}) { /* removed */ },\n})\n```\n\n**Design pattern**: Multiple controllers can register handlers on the same informer. The informer deduplicates LIST/WATCH requests to the API server, reducing load.\n\n#### 2.8 Informer Factory (`informers/factory.go`)\n\n**Responsibility**: Create and manage a set of SharedInformers for all API types, with a unified lifecycle.\n\n**Key options** (`informers/factory.go:54-109`):\n```go\ntype SharedInformerOption func(*sharedInformerFactory)\n\n// WithCustomResyncConfig: different resync periods for different types\nWithCustomResyncConfig(map[v1.Object]time.Duration)\n\n// WithTweakListOptions: filter cached objects (e.g., single namespace)\nWithTweakListOptions(func(*metav1.ListOptions))\n\n// WithNamespace: limit to one namespace\nWithNamespace(namespace string)\n\n// WithTransform: transform cached objects before storing (e.g., strip secrets)\nWithTransform(cache.TransformFunc)\n```\n\n**Lifecycle**:\n```go\nfactory := informers.NewSharedInformerFactoryWithOptions(clientset, 10*time.Second)\npodInformer := factory.Core().V1().Pods()\nfactory.Start(stopCh)        // Start all informers\nfactory.WaitForCacheSync(stopCh) // Wait for initial LIST\n// Now podInformer.Lister().Pods(ns).List() is safe\nfactory.Shutdown()           // Stop all informers\n```\n\n#### 2.9 Work Queue (`util/workqueue/`)\n\n**Responsibility**: Decouple event detection (informers) from reconciliation (controller logic).\n\n**Queue properties** (`util/workqueue/doc.go`):\n- **Fair**: Items processed in order\n- **Stingy**: No concurrent processing of the same item\n- **Durable**: Item stays in queue until explicitly marked done\n- **Retryable**: Failed items can be re-queued with backoff\n\n**Rate limiting strategies** (`util/workqueue/default_rate_limiters.go`):\n```go\n// Linear backoff: 1x, 2x, 3x, ..., capped at maxRetries\nNewItemExponentialFailureRateLimiter(1*time.Second, 1*time.Hour)\n\n// Exponential backoff: 2^n\nNewMaxOfRateLimiter(exponential, linear, baseDelay)\n```\n\n**Typical controller loop**:\n```go\nfor {\n    item, shutdown := queue.Get()\n    if shutdown { break }\n\n    err := r.reconcile(item.(string))\n    if err != nil {\n        queue.AddRateLimited(item) // Retry with backoff\n    } else {\n        queue.Forget(item) // Reset backoff counter\n    }\n    queue.Done(item)\n}\n```\n\n#### 2.10 Leader Election (`tools/leaderelection/`)\n\n**Responsibility**: Coordinate multiple replicas of a controller, ensuring only one is active.\n\n**Mechanism**:\n- Replicas compete to acquire a `Lease` object\n- Holder periodically updates the lease\n- Other replicas watch the lease, take over if holder dies\n- Graceful shutdown releases the lease\n\n**Configuration**:\n```go\nleaderelection.LeaderElect(context.Background(), leaderelection.LeaderElectionConfig{\n    Lock: &resourcelock.LeaseLock{\n        Client: clientset.CoordinationV1(),\n        LockConfig: resourcelock.ResourceLockConfig{\n            Identity: hostname,\n        },\n    },\n    LeaseDuration: 15 * time.Second,\n    RenewDeadline: 10 * time.Second,\n    RetryPeriod:   2 * time.Second,\n    Callbacks: leaderelection.LeaderCallbacks{\n        OnStartedLeading: func(ctx context.Context) { /* become active */ },\n        OnStoppedLeading: func() { /* stop reconciling */ },\n    },\n})\n```\n\n---\n\n## 3.
