  "content": "# Terraform Plan/Apply Pipeline Architecture\n\n## Overview\n\nTerraform's plan/apply pipeline is a sophisticated execution framework that transforms human-readable infrastructure configuration into executable operations. The pipeline orchestrates resource graph construction, provider interactions, state management, and execution sequencing to safely and reliably manage infrastructure changes.\n\nThe core execution model is **graph-based**: Terraform constructs a directed acyclic graph (DAG) of resources, data sources, and providers, then traverses this graph to execute operations while respecting dependency constraints.\n\n---\n\n## 1. Graph Builder System\n\nThe Graph Builder System is responsible for constructing the execution DAG from Terraform configuration. It transforms raw configuration into a graph where nodes represent infrastructure entities and edges represent dependencies.\n\n### 1.1 Graph Architecture\n\n**Core Components:**\n\n- **DAG (Directed Acyclic Graph)**: The fundamental data structure representing all infrastructure elements and their relationships. Nodes are execution units; edges encode both explicit and implicit dependencies.\n- **Graph Nodes**: Abstract representations of infrastructure elements including resources, data sources, modules, providers, and variable references.\n- **Graph Edges**: Directed connections representing dependencies between nodes.\n\n**Node Types:**\n\n- **ResourceNode**: Represents a managed resource requiring create, read, update, or delete operations.\n- **DataSourceNode**: Represents a data source requiring evaluation to fetch external data.\n- **ProviderNode**: Represents provider configuration and initialization.\n- **ModuleNode**: Represents module invocations with their own subgraphs.\n- **OutputNode**: Represents output values that depend on resource outputs.\n- **VariableNode**: Represents variable references and their dependencies.\n- **RootNode**: Represents the configuration root and implicit dependencies on all resources.\n\n### 1.2 Graph Construction Pipeline\n\nThe graph building process occurs through sequential transformations:\n\n```\nConfiguration Parsing\n    \u2193\nModule Expansion\n    \u2193\nResource Graph Construction\n    \u2193\nDependency Analysis\n    \u2193\nGraph Transformation Pipeline\n    \u2193\nFinal Execution DAG\n```\n\n**Phase 1: Configuration Parsing**\n- Configuration is parsed from HCL into an intermediate representation\n- Variables, modules, resources, data sources, outputs are identified\n- Locals and computed values are registered\n\n**Phase 2: Module Expansion**\n- Module bodies are recursively expanded\n- Module variables are bound to their sources\n- Module outputs are registered as dependencies for parent references\n- Subgraphs are created for each module instance\n- Count/for_each module expansion creates multiple module instances\n\n**Phase 3: Resource Graph Construction**\n- For each resource block, a ResourceNode is created\n- For each data source block, a DataSourceNode is created\n- Provider configuration blocks are registered\n- The graph initially contains explicit resource definitions without dynamic expansions\n\n**Phase 4: Dependency Analysis**\nTerraform's dependency discovery is multi-faceted:\n\n- **Explicit Dependencies**: `depends_on` blocks create direct edges in the graph\n- **Reference Dependencies**: Variable, local, and resource references in configuration trigger edge creation\n  - `resource.type.name.attribute` creates dependency on that resource\n  - `data.type.name.attribute` creates dependency on that data source\n  - `var.name` creates dependency on variable definition\n  - `module.name.output` creates dependency on module output\n- **Implicit Dependencies**: Provider blocks implicitly depend on their variable references\n- **Module Dependencies**: Resources within modules depend on module variables and locals\n- **Count/For_Each Dependencies**: Dynamic resource expansion creates additional dependency nodes\n\n**Phase 5: Graph Transformation Pipeline**\nThe graph undergoes a series of transformations to prepare for execution:\n\n1. **Provider Transform**: Adds provider configuration nodes and ensures resources depend on appropriate provider instances\n   - Extracts provider alias from resource configuration\n   - Creates dependency edge from resource to provider node\n   - Handles provider inheritance (default provider if none specified)\n\n2. **Root Transform**: Adds synthetic root node that all resources transitively depend on\n   - Enables clean graph traversal starting from root\n   - Handles implicit dependencies on all root-level resources\n\n3. **Count/For_Each Expansion Transform**: Expands resources with count or for_each into multiple nodes\n   - `count.index` and `each.key`/`each.value` become available\n   - Creates edges for dependencies on other instances of the same resource\n   - Handles conditional resource creation based on count/for_each expressions\n\n4. **Module Orphan Transform**: Identifies and marks resources no longer in configuration\n   - Enables safe destruction of removed resources\n   - Tracks resources that exist in state but not in configuration\n\n5. **Targeting Transform**: Filters the graph based on `-target` flags\n   - Removes non-targeted resources and their dependents\n   - Preserves resource dependencies for correct execution order\n   - Used for surgical infrastructure changes\n\n6. **Conditional Destroy Transform**: Modifies destroy actions based on configuration changes\n   - Determines which resources should be destroyed vs. updated\n   - Handles replacement of resources with incompatible changes\n\n7. **Output Transform**: Adds output nodes and their dependencies\n   - Outputs depend on their referenced resources/data sources\n   - Output evaluation may trigger resource evaluation\n\n8. **Local Values Transform**: Processes local value references\n   - Locals depend on their value expressions\n   - Resources referencing locals depend on local values\n   - Enables lazy evaluation of expensive computations\n\n### 1.3 Dependency Edge Types\n\nTerraform distinguishes between different categories of dependencies:\n\n- **Hard Dependencies**: Must be completed before dependent can start\n  - Standard resource-to-resource dependencies\n  - Variable dependencies\n  - Provider dependencies\n\n- **Soft Dependencies**: May be reordered for optimization\n  - Some output dependencies\n  - Informational logging dependencies\n\n- **Conditional Dependencies**: Activated based on resource state\n  - Lifecycle rule dependencies (create_before_destroy)\n  - Provider migration dependencies\n\n### 1.4 Dynamic Graph Expansion\n\nFor resources using `count` or `for_each`:\n\n1. **Planning Phase**: Count/for_each expressions are evaluated\n   - `count.index` is known for iteration\n   - `each.key` and `each.value` are known from map/list\n   - The number of resource instances is determined\n\n2. **Graph Expansion**: For each iteration:\n   - A separate graph node is created (e.g., `aws_instance.web[0]`, `aws_instance.web[1]`)\n   - Reference expressions are evaluated within iteration context\n   - Dependencies between instances are discovered\n   - Splat expressions create dependencies on all instances\n\n3. **Cross-Instance Dependencies**:\n   - Resources can reference other instances: `aws_instance.web[*].id`\n   - Creates edges from referencing resource to all instances\n   - Enables coordinated changes across instance groups\n\n---\n\n## 2. Provider Interface and Lifecycle\n\nProviders are plugins that implement resource operations. The provider architecture decouples Terraform core from provider-specific logic, enabling a plugin ecosystem while maintaining consistent interfaces.\n\n### 2.1 Provider Architecture\n\n**Plugin Model:**\n\nTerraform uses a plugin-based architecture where:\n- Each provider is a separate executable (binary)\n- Providers communicate with Terraform core via gRPC (protocol v6/v5)\n- Providers are loaded on-demand and cached during execution\n- Provider versions are managed via locks and version constraints\n\n**Provider Registry:**\n\n- Providers are declared in configuration: `terraform { required_providers { aws = {...} } }`\n- Terraform downloads providers from the Registry (registry.terraform.io) or custom registries\n- Lock files (`terraform.lock.hcl`) pin provider versions for reproducibility\n- Providers are stored in `.terraform/providers/` directory\n\n### 2.2 Provider Initialization and Management\n\n**Initialization Sequence:**\n\n1. **Discovery Phase**: Terraform identifies required providers from configuration\n   - Parses `terraform` block for `required_providers`\n   - Parses resource and data source types to infer providers\n   - Resolves provider versions from lock file or constraints\n\n2. **Download Phase**: Providers are downloaded if not cached\n   - Queries provider registry for available versions\n   - Downloads provider binary for host platform\n   - Verifies cryptographic signatures\n   - Extracts and caches provider executable\n\n3. **Launch Phase**: Provider process is started\n   - `terraform` spawns provider executable as subprocess\n   - Provider advertises supported schemas via gRPC\n   - Connection is maintained for the duration of plan/apply\n\n4. **Schema Discovery**: Terraform queries provider capabilities\n   - Provider returns schemas for all resource types\n   - Schemas include attribute descriptions, types, and validation rules\n   - Enables configuration validation and IDE support\n\n5. **Configuration Binding**: Provider configuration is bound to instances\n   - Provider configuration block is evaluated (variables resolved)\n   - Configuration is prepared for passing to resources\n   - Multiple provider aliases can exist for same provider type\n\n**Provider Instance Management:**\n\n- **Instance Per Alias**: Each provider alias gets a separate instance\n  - `provider \"aws\" { region = \"us-east-1\" }` - default instance\n  - `provider \"aws\" { alias = \"west\", region = \"us-west-2\" }` - aliased instance\n  - Resources specify which provider via `provider = aws.west`\n\n- **Provider Inheritance**: Resources inherit from default provider if not specified\n  - `provider = \"aws\"` uses configuration from default aws provider\n  - Explicit `provider = aws.west` uses specific aliased instance\n\n- **Lazy Loading**: Providers are not loaded until needed\n  - Only providers referenced in resources are loaded\n  - Reduces startup time and resource consumption\n  - Enables disconnected operations (plan without provider access)\n\n### 2.3 Resource Operation Delegation\n\n**Operation Types:**\n\nTerraform delegates four primary operations to providers:\n\n1. **ValidateResourceConfig**: Validates resource configuration\n   - Called during planning phase\n   - Checks attribute constraints and required fields\n   - Provider-specific validation (e.g., conflicting options)\n   - Errors prevent plan from proceeding\n\n2. **PlanResourceChange**: Computes desired state and change type\n   - Input: prior state, proposed new state, provider configuration\n   - Output: planned new state, change type, diagnostics\n   - Determines whether resource will be created, updated, deleted, or unchanged\n   - Handles computed fields and default values\n\n3. **ApplyResourceChange**: Executes the planned change\n   - Input: planned new state, provider configuration\n   - Output: new state after operation, diagnostics\n   - Makes API calls to cloud provider\n   - Returns actual state as observed after operation\n   - May differ slightly from planned state (computed fields)\n\n4. **ReadDataSourceValues**: Fetches data source values\n   - Input: data source configuration, provider configuration\n   - Output: data source state with fetched values, diagnostics\n   - No-op operation (no resource created)\n   - Used for querying existing infrastructure\n\n5. **ImportResourceState**: Maps existing resource to state\n   - Input: resource ID, provider configuration\n   - Output: resource state reconstructed from cloud provider\n   - Used for `terraform import` command\n   - Optional per provider\n\n**Call Sequence During Plan:**\n\n```\nFor each resource node in execution order:\n  1. Provider.ValidateResourceConfig(resource_config)\n  2. Provider.PlanResourceChange(prior_state, proposed_state, config)\n  3. Store planned change in plan file\n```\n\n**Call Sequence During Apply:**\n\n```\nFor each resource node in execution order:\n  1. Provider.ApplyResourceChange(planned_state, config)\n  2. Update state with new actual state\n  3. If error: handle according to lifecycle rules\n```\n\n### 2.4 Error Handling and Recovery\n\n**Provider-Level Errors:**\n\n- **Transient Errors**: Network timeouts, rate limiting\n  - Provider may retry automatically\n  - Terraform may suggest manual retry\n\n- **Validation Errors**: Invalid configuration, unsupported attributes\n  - Reported during validation phase\n  - Prevent plan from completing\n\n- **Operational Errors**: API failures, permission issues\n  - During apply, recorded but don't necessarily stop execution\n  - Can trigger lifecycle actions (e.g., destroy on error)\n\n**Provider Crash Handling:**\n\n- Provider crashes are detected via gRPC connection loss\n- Terraform propagates crash as plan/apply error\n- State is preserved to enable recovery\n- User can retry or investigate provider logs\n\n### 2.5 Provider Configuration Evaluation\n\nProvider blocks are evaluated as part of graph execution:\n\n```hcl\nprovider \"aws\" {\n  region = var.aws_region\n\n  assume_role {\n    role_arn = aws_iam_role.terraform_role.arn\n  }\n}\n```\n\n**Evaluation Process:**\n\n1. Provider block configuration is identified\n2. All variable references are resolved\n3. All resource/data source references are evaluated (if provider depends on resources)\n4. Final configuration is passed to provider plugin\n5. Provider validates configuration and stores for use with resources\n\n**Dynamic Credentials:**\n\n- Providers may assume roles, fetch tokens, or perform other dynamic operations\n- Provider configuration can reference resources (e.g., assume role from created resource)\n- Creates potential dependency loops - Terraform detects and reports these\n\n---\n\n## 3. State Management\n\nTerraform state is the source of truth for actual infrastructure. State management ensures consistency between configuration and infrastructure across multiple operations and users.\n\n### 3.1 State Structure and Content\n\n**State File Format:**\n\nTerraform state is stored as JSON in `.terraform/tfstate`:\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.x.x\",\n  \"serial\": 42,\n  \"lineage\": \"uuid-string\",\n  \"outputs\": {\n    \"instance_ip\": {\n      \"value\": \"10.0.0.1\",\n      \"type\": \"string\"\n    }\n  },\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"web\",\n      \"instances\": [\n        {\n          \"index_key\": 0,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"ami\": \"ami-0c55b159cbfafe1f0\",\n            \"instance_type\": \"t2.micro\",\n            \"private_ip\": \"10.0.0.1\"\n          },\n          \"sensitive_attributes\": [],\n          \"private\": \"base64-encoded-data\"\n        }\n      ]\n    }\n  ]\n}\n```\n\n**Key Elements:**\n\n- **version**: State format version (currently 4)\n- **terraform_version**: Version of Terraform that wrote state\n- **serial**: Monotonically increasing counter for change detection\n- **lineage**: UUID maintained across state versions, detects state replacement\n- **outputs**: Computed output values\n- **resources**: Array of managed resources with instances\n- **attributes**: Resource attributes as returned by provider\n- **sensitive_attributes**: Paths to attributes containing sensitive values\n- **private**: Provider-specific data (encrypted in remote state)\n\n**State Metadata:**\n\nEach resource instance includes:\n- Resource type and name\n- Instance index (for count/for_each)\n- All attributes returned by provider\n- Sensitive attribute paths\n- Provider-specific data\n- Dependency information (implicit from graph)\n\n### 3.2 State Reading and Modification\n\n**State Loading:**\n\n1. **Local State**: Loaded from `.terraform/tfstate`\n   - Parsed as JSON\n   - Validated for consistency\n   - In-memory representation created\n\n2. **Remote State**: Loaded from configured backend\n   - HTTP request to backend API\n   - State decrypted and decompressed\n   - Merged with local state cache if `backend` block specifies it\n   - Enables team collaboration and state safety\n\n3. **State Locking**:\n   - Before modification, lock is acquired\n   - Prevents concurrent operations\n   - Lock timeout and force-unlock available for recovery\n   - Different backends implement locking differently (DynamoDB for S3, etc.)\n\n**State Modification:**\n\nState is modified through a **delta application** process:\n\n1. **Planned Changes**: Plan file contains:\n   - Resources to be created, updated, deleted, or replaced\n   - Proposed attribute values\n   - Metadata about change operations\n\n2. **Atomic Application**: During apply:\n   - State is locked to prevent concurrent modifications\n   - For each resource in planned order:\n     - Provider executes change\n     - Returned actual state is written to state\n     - State serial increments\n   - Final state is persisted\n\n3. **Partial Application Recovery**:\n   - If apply fails midway, state reflects completed operations\n   - Incomplete resource operations leave state unchanged\n   - User can retry apply (may require manual intervention)\n\n### 3.3 State Snapshots and Concurrent Evaluation\n\n**State Snapshots for Graph Traversal:**\n\nThe state snapshot is a critical mechanism enabling safe concurrent evaluation:\n\n```\nState Snapshot (Point-in-time)\n    \u2193\n[Passed to all graph node evaluations]\n    \u2193\n[Enables safe planning without locking for the entire plan]\n```\n\n**How Snapshots Enable Concurrency:**\n\n1. **Read Lock Acquired**: Full state is read and locked briefly\n2. **Snapshot Created**: In-memory copy of entire state taken\n3. **Lock Released**: State lock is released immediately\n4. **Snapshot Used**: All graph nodes use snapshot for reference dependencies\n   - No need to re-read from state during planning\n   - Reference resolution is consistent (same snapshot)\n   - No conflicting updates possible\n\n5. **Validation**: Before apply, snapshot is compared with current state\n   - Detects concurrent modifications by other users\n   - Prevents applying stale plan to modified state\n   - Forces user to re-plan if state changed\n\n**Benefits:**\n\n- **Planning Lock Duration**: Locks held only for milliseconds during snapshot creation\n- **Concurrent Operations**: Multiple users can plan simultaneously\n- **Consistency**: All references within a plan see consistent state\n- **Merge Conflict Detection**: Concurrent state changes are detected before apply\n\n### 3.4 State Synchronization\n\n**Refresh Operation:**\n\nBefore planning, Terraform can refresh state to match actual infrastructure:\n\n```\nFor each resource in state:\n  1. Call provider ReadState\n  2. Compare actual state with state file\n  3. Update state file with actual values\n  4. Mark if actual state differs from configuration\n```\n\n**Scenarios:**\n\n- Out-of-band changes: Manual API calls modifying infrastructure\n- Attribute drift: Resources modified outside Terraform\n- Computed value changes: Time-sensitive attributes (certificates, etc.)\n\n**Refresh Modes:**\n\n- `-refresh=true` (default for plan): Query all resources\n- `-refresh=false`: Skip refresh, use cached state\n- Useful for speed when state is known current\n- Risk: Plans based on stale state\n\n**State Refreshing During Plan:**\n\n1. **Implicit Refresh**: Plan phase refreshes state (configurable)\n2. **Selective Refresh**: Only non-targeted resources refreshed\n3. **Output of Refresh**: Plan shows drift before any changes\n\n### 3.5 Remote State and Backends\n\n**Backend Configuration:**\n\n```hcl\nterraform {\n  backend \"s3\" {\n    bucket         = \"terraform-state\"\n    key            = \"prod/terraform.tfstate\"\n    region         = \"us-east-1\"\n    encrypt        = true\n    dynamodb_table = \"terraform-locks\"\n  }\n}\n```\n\n**Backend Responsibilities:**\n\n1. **Storage**: Persist state file\n2. **Retrieval**: Provide state on demand\n3. **Locking**: Coordinate concurrent access (if supported)\n4. **Versioning**: Maintain state history (if supported)\n5. **Encryption**: Protect sensitive state data\n\n**Common Backends:**\n\n- **S3**: AWS S3 with DynamoDB locking\n- **Azure Blob Storage**: Azure Storage with blob lease locking\n- **Google Cloud Storage**: GCS with state versioning\n- **Terraform Cloud**: Managed state with built-in locking\n- **Consul**: Distributed state with Consul locking\n- **etcd**: Kubernetes-native state backend\n- **HTTP**: Generic HTTP endpoint for custom backends\n\n**Local vs. Remote State Trade-offs:**\n\n| Aspect | Local | Remote |\n|--------|-------|--------|\n| Team Collaboration | Manual sync/merge | Automatic |\n| Locking | File system | Backend-dependent |\n| Versioning | None | Backend-dependent |\n| Encryption | Unencrypted on disk | Backend-dependent |\n| Latency | Local access | Network dependent |\n\n### 3.6 State Operations and Safety\n\n**State Listing:**\n\n```\nterraform state list\n  \u2193\nOutputs: All managed resource addresses\n  \u2193\nFormat: aws_instance.web, aws_instance.web[0], module.vpc.aws_subnet.public\n```\n\n**State Inspection:**\n\n```\nterraform state show aws_instance.web[0]\n  \u2193\nOutputs: All attributes for specific resource\n  \u2193\nShows actual resource state from state file\n```\n\n**State Manipulation (Advanced):**\n\n- `terraform state rm`: Remove resource from state (without destroying)\n- `terraform state mv`: Move resource address (rename)\n- `terraform state replace-provider`: Change provider for resource\n- **Use with caution**: These operations bypass safety checks\n\n---\n\n## 4. Execution Flow and Hook System\n\nThe execution flow orchestrates graph traversal, node evaluation, and hook invocation to execute plan or apply operations.\n\n### 4.1 Overall Command Flow\n\n**Complete Terraform Execution Lifecycle:**\n\n```\nterraform plan [options] [dir]\n    \u2193\nLoad Configuration & Parse\n    \u2193\nInitialize Providers\n    \u2193\nBuild Resource Graph\n    \u2193\nValidate Graph\n    \u2193\nRefresh State (optional)\n    \u2193\nPlan Graph Evaluation\n    \u2193\nDisplay Plan to User\n    \u2193\nWrite Plan File\n    \u2193\nPlan Complete\n\n\nterraform apply [options] [plan-file]\n    \u2193\nLoad Configuration & Plan File\n    \u2193\nInitialize Providers (if not from plan)\n    \u2193\nRequest User Confirmation\n    \u2193\nApply Graph Evaluation\n    \u2193\nUpdate State File\n    \u2193\nPersist Remote State\n    \u2193\nApply Complete\n```\n\n### 4.2 Graph Node Evaluation\n\n**Execution Semantics:**\n\nNodes are evaluated respecting:\n1. **Dependency Order**: Edges create ordering constraints\n2. **Parallelization**: Nodes with no blocking dependencies run concurrently\n3. **Determinism**: Same configuration produces same plan (modulo external state)\n4. **Idempotence**: Repeated applies on unchanged resources are no-ops\n\n**Planning Phase Evaluation:**\n\nFor each resource node:\n\n```go\n1. Refresh (if enabled):\n   provider.ReadResourceState(state, config)\n   \u2192 updates state with actual values\n\n2. Validate:\n   provider.ValidateResourceConfig(config)\n   \u2192 errors prevent plan\n\n3. Plan:\n   provider.PlanResourceChange(prior_state, proposed_state, config)\n   \u2192 determines change type and planned state\n\n4. Store Planned Change:\n   plan_file[resource] = planned_change\n```\n\n**Change Types:**\n\n- **NoOp**: Resource unchanged\n- **Create**: Resource doesn't exist in state, configuration has it\n- **Update**: Resource exists, configuration specifies different values\n- **Delete**: Resource in state, not in configuration\n- **Replace**: Update requires resource destruction and recreation\n- **Read**: Data source read (no state change)\n\n**Apply Phase Evaluation:**\n\nFor each resource node in plan order:\n\n```go\n1. Verify Plan Still Valid:\n   if current_state != plan.prior_state:\n       error \"state changed since plan\"\n\n2. Apply Change:\n   if change == Create:\n       new_state = provider.ApplyResourceChange(null, planned_state, config)\n   else if change == Update:\n       new_state = provider.ApplyResourceChange(prior_state, planned_state, config)\n   else if change == Delete:\n       provider.ApplyResourceChange(prior_state, null, config)\n       remove from state\n   else if change == NoOp:\n       skip provider call, state unchanged\n\n3. Update State:\n   state[resource] = new_state\n   persist state\n\n4. Handle Errors:\n   if provider.error:\n       check lifecycle.on_failure\n       potentially delete resource\n       report error\n```\n\n### 4.3 Walker Pattern for Graph Traversal\n\nThe **Walker** is the core execution engine that traverses the dependency graph.\n\n**Walker Responsibilities:**\n\n1. **Dependency Tracking**: Maintains which nodes are ready to execute\n2. **Concurrency Control**: Limits parallel execution (default: 10 concurrent)\n3. **Ordering**: Executes nodes respecting dependency constraints\n4. **Hook Invocation**: Calls hooks at appropriate execution points\n5. **Error Handling**: Handles and reports errors from node evaluation\n\n**Walker Algorithm:**\n\n```\n1. # Terraform Plan/Apply Pipeline Architecture

## Overview

Terraform's plan/apply pipeline is a sophisticated execution framework that transforms human-readable infrastructure configuration into executable operations. The pipeline orchestrates resource graph construction, provider interactions, state management, and execution sequencing to safely and reliably manage infrastructure changes.

The core execution model is **graph-based**: Terraform constructs a directed acyclic graph (DAG) of resources, data sources, and providers, then traverses this graph to execute operations while respecting dependency constraints.

---
