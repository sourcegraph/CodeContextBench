Locks acquired from both sources\n  }\n}\n```\n\n#### Migration Steps (Optional)\n\nTo adopt S3 native locking:\n\n1. **Update your S3 bucket policy** to include lock file permissions:\n   ```json\n   {\n     \"Version\": \"2012-10-17\",\n     \"Statement\": [\n       {\n         \"Effect\": \"Allow\",\n         \"Action\": [\n           \"s3:GetObject\",\n           \"s3:PutObject\",\n           \"s3:DeleteObject\"\n         ],\n         \"Resource\": \"arn:aws:s3:::my-terraform-state/path/to/*.tflock\"\n       }\n     ]\n   }\n   ```\n\n2. **Update IAM policy** for your Terraform user/role:\n   ```json\n   {\n     \"Version\": \"2012-10-17\",\n     \"Statement\": [\n       {\n         \"Effect\": \"Allow\",\n         \"Action\": [\n           \"s3:GetObject\",\n           \"s3:PutObject\",\n           \"s3:DeleteObject\"\n         ],\n         \"Resource\": \"arn:aws:s3:::my-terraform-state/path/to/*.tflock\"\n       }\n     ]\n   }\n   ```\n\n3. **Enable in your backend configuration**:\n   ```hcl\n   terraform {\n     backend \"s3\" {\n       # Moved Blocks: Syntax Changes for Reserved Keywords\n\n#### What Changed\n\nTerraform v1.10.0 enforces stricter parsing of resource addresses in `moved` blocks. Resources with type names that match reserved keywords (like `local`, `each`, `self`, `data`, `resource`, `module`, `variable`, `output`, `terraform`, `provider`) must now be prefixed with `resource.` to disambiguate them from the reserved keywords.\n\n#### Why This Matters\n\nPreviously, you could write:\n```hcl\nmoved {\n  from = local.my_resource\n  to   = aws_instance.web\n}\n```\n\nHowever, this creates ambiguity: Is `local` referring to a local value or a resource type named `local`? `local` - Most common\n   - `data` - Data source type\n   - `resource` - Resource keyword\n   - `module` - Module keyword\n   - `variable`, `output` - Variable and output keywords\n   - `terraform`, `provider` - Meta-block keywords\n   - `each`, `self` - Built-in variables\n\n#### Example Scenario\n\nIf your codebase had custom providers with these type names:\n\n```hcl\n# Before migration - breaks in v1.10.0\nmoved {\n  from = local.my_resource\n  to   = aws_instance.new_resource\n}\n\nmoved {\n  from = data.old_config\n  to   = local_file.new_config\n}\n\n# After migration - works in v1.10.0\nmoved {\n  from = resource.local.my_resource\n  to   = aws_instance.new_resource\n}\n\nmoved {\n  from = resource.data.old_config\n  to   = local_file.new_config\n}\n```\n\n---\n\n## New Features with Migration Impact\n\n### 4. Ephemeral Resources and Values\n\n#### What Are Ephemeral Values?\n\nEphemeral values are data that exists during Terraform operations but is **never stored** in plan files or state files. This is ideal for secrets, API tokens, and temporary credentials that should not persist beyond the current operation.\n\n#### Key Characteristics\n\n- **Not persisted**: Never written to `.tfstate` or plan files\n- **Temporary**: Available only during the current operation\n- **Contextual**: Can only be referenced in specific \"ephemeral contexts\"\n- **Secure by design**: Secrets are never at rest\n\n#### New Features in v1.10.0\n\n**1. Ephemeral Input Variables**\n```hcl\nvariable \"api_token\" {\n  type        = string\n  ephemeral   = true\n  description = \"Temporary API token that won't be stored in state\"\n}\n```\n\n**2. Ephemeral Outputs (in child modules)**\n```hcl\noutput \"temporary_credentials\" {\n  value       = aws_iam_access_key.temp.secret_access_key\n  ephemeral   = true\n  description = \"Temporary AWS secret key not stored in state\"\n}\n```\n\n**3. Ephemeral Resources**\n```hcl\nephemeral \"random_password\" \"temp\" {\n  length  = 32\n  special = true\n}\n```\n\n**4. `ephemeralasnull()` Function**\n```hcl\n# Convert ephemeral values to null for contexts where ephemeral isn't allowed\nlocal_file.config = {\n  content = ephemeralasnull(var.sensitive_config)\n}\n```\n\n#### Valid Contexts for Ephemeral Values\n\nEphemeral variables can only be referenced in:\n- Other ephemeral variables\n- Ephemeral local values\n- Ephemeral resources\n- Ephemeral outputs\n- `provider` blocks (for configuring providers)\n- `provisioner` and `connection` blocks\n\n#### Migration Considerations\n\n**If you currently use `sensitive` for secrets:**\n```hcl\n# v1.9.0 approach - secrets stored in state (marked as sensitive in output)\nvariable \"db_password\" {\n  type      = string\n  sensitive = true  # Hidden in output, but still in state\n}\n```\n\n**Consider migrating to ephemeral if the secret never needs to be stored:**\n```hcl\n# v1.10.0 approach - secrets never stored at all\nvariable \"db_password\" {\n  type        = string\n  ephemeral   = true  # Not stored in state or plan\n}\n```\n\n#### Migration Steps for Ephemeral Values\n\n1. **Identify secrets that don't need persistence**: Look for resources/variables that only need to exist during execution\n2. **Mark them as ephemeral** where appropriate:\n   ```hcl\n   # Before\n   variable \"temporary_token\" {\n     type      = string\n     sensitive = true\n   }\n\n   # After\n   variable \"temporary_token\" {\n     type      = string\n     ephemeral = true\n   }\n   ```\n\n3. **Update references to respect ephemeral contexts**: If you reference ephemeral values outside of valid contexts, use `ephemeralasnull()`:\n   ```hcl\n   resource \"aws_instance\" \"web\" {\n     # This would fail - can't use ephemeral variable directly here\n     # user_data = var.temp_token\n\n     # Instead, handle gracefully\n     user_data = base64encode(ephemeralasnull(var.temp_token) ?? \"\")\n   }\n   ```\n\n#### Important Notes\n\n- **Ephemeral is in public beta**: Workflows are subject to change\n- **Root module limitation**: You cannot mark outputs as ephemeral in the root module\n- **Ephemeral outputs are child-module only**: Can only be used to pass values between modules, not persisted to root module state\n\n---\n\n### 5. Sensitive Value Handling in Conditional Expressions\n\n#### What Changed\n\nTerraform v1.10.0 **improves the consistency** of how sensitive marks propagate through conditional expressions. Previously, sensitive marks could be lost in certain combinations of conditional and for expressions with unknown values.\n\n#### The Problem (v1.9.0)\n\n```hcl\n# In v1.9.0, sensitive marks could be lost in conditional expressions\nvariable \"is_prod\" {\n  type = bool\n}\n\nvariable \"database_password\" {\n  type      = string\n  sensitive = true\n}\n\n# Bug: The sensitive mark might be lost in this conditional\noutput \"password\" {\n  value = var.is_prod ? var.database_password : null\n  # Result might not be marked as sensitive!\n}\n```\n\n#### The Solution (v1.10.0)\n\nTerraform v1.10.0 combines marks from all values in conditional expressions to ensure consistency:\n\n```hcl\n# In v1.10.0, this now correctly preserves sensitive marks\noutput \"password\" {\n  value = var.is_prod ? var.database_password : null\n  # Result is correctly marked as sensitive regardless of the branch\n}\n```\n\n#### Migration Impact\n\nThis is mostly **transparent** for users, but you might notice:\n\n1. **More values marked as sensitive**: If you were relying on the previous behavior where marks were lost, output may now show more `(sensitive value)` placeholders\n\n2. **Clearer intent**: Use `nonsensitive()` when you intentionally want to reveal a derived value:\n   ```hcl\n   # If you've derived a value that's safe to show\n   output \"hashed_password\" {\n     value = nonsensitive(bcrypt(var.database_password))\n   }\n   ```\n\n3. **New conditional expression behavior**:\n   ```hcl\n   # Before: Might not be marked sensitive\n   output \"config\" {\n     value = var.include_secrets ? {\n       password = var.db_password\n     } : {}\n   }\n\n   # After: Correctly marked sensitive\n   # If var.db_password is sensitive, the entire map is sensitive\n   output \"config\" {\n     value = var.include_secrets ? {\n       password = var.db_password\n     } : {}\n     sensitive = true  # May need to add this now\n   }\n   ```\n\n#### Migration Steps\n\n1. **Review sensitive outputs**: Run `terraform plan` and look for outputs showing `(sensitive value)` where they didn't before\n\n2. **Add explicit `sensitive = true`** if needed:\n   ```hcl\n   output \"api_key\" {\n     value     = var.is_dev ? var.dev_key : var.prod_key\n     sensitive = true  # Both branches contain sensitive values\n   }\n   ```\n\n3. **Use `nonsensitive()` carefully** when a derived value is safe:\n   ```hcl\n   # Only use nonsensitive() when you're absolutely sure\n   output \"password_hash\" {\n     value = bcrypt(var.password)  # bcrypt result might not be sensitive\n     # Even though input was sensitive\n   }\n   ```\n\n4. **Test conditional branches**: Ensure both true and false branches are handled:\n   ```hcl\n   # Good: Both branches handle sensitivity\n   output \"value\" {\n     value = condition ? sensitive_value : null\n     sensitive = true  # Covers both cases\n   }\n\n   # Problem: False branch might not be sensitive\n   output \"value\" {\n     value = condition ? sensitive_value : \"default\"\n     sensitive = true  # Required now\n   }\n   ```\n\n---\n\n## Testing Your Migration\n\n### Pre-Migration Checklist\n\n```bash\n# 1. # Terraform v1.9.0 to v1.10.0 Migration Guide

**Release Date:** November 27, 2024

This guide provides step-by-step instructions for upgrading from Terraform v1.9.0 to v1.10.0. While v1.10.0 honors the [Terraform v1.0 Compatibility Promises](https://developer.hashicorp.com/terraform/language/v1-compatibility-promises), there are several breaking changes and behavioral modifications that require migration steps.

## Overview

Terraform v1.10.0 introduces two major new features with migration implications:
- **Ephemeral Resources and Values**: A new mechanism for handling secrets and temporary data that isn't persisted to state files
- **Enhanced Sensitive Value Handling**: Improvements to how sensitive marks propagate through conditional and for expressions

It also includes breaking changes to the S3 backend and moved block syntax that require configuration updates.
