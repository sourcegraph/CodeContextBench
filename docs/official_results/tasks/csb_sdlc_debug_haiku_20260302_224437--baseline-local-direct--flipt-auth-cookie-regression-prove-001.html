# Bug Investigation: Authentication Fails for Browser Cookie-Based Tokens

**Repository:** flipt-io/flipt
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

The authentication system in Flipt only accepts client tokens via the `Authorization` header using Bearer format. This prevents browser-based sessions from working, since browsers typically store authentication tokens in HTTP cookies.

Specifically:

1. **Cookie tokens rejected**: When a valid client token is stored in an HTTP cookie (under the key `flipt_client_token`), the authentication middleware does not extract or validate it. Requests that carry a valid token in a cookie but not in the Authorization header are rejected as unauthenticated.

2. **No server-level authentication bypass**: Certain internal servers (such as those handling delegated authentication flows) need to operate without authentication checks. There is currently no mechanism to configure the middleware to skip authentication for specific server instances, forcing all registered servers through the same authentication path regardless of their role.

## Investigation Summary

### **Bug #1: Cookie Token Extraction Failure**
Located in `/app/internal/server/auth/middleware.go`, the `UnaryInterceptor` function only checks the `Authorization` header for tokens. It never extracts tokens from:
- HTTP cookies (passed as gRPC metadata)
- Custom metadata fields
- Other authentication mechanisms

The middleware logic at lines 51-55 only looks for the "authorization" header key, rejecting any request that doesn't have it, regardless of whether a valid token exists in cookies.

### **Bug #2: No Server-Level Authentication Bypass**
Located in `/app/cmd/flipt/main.go` (lines 485-489), when authentication is required, the interceptor is applied globally to all gRPC services, including:
- `AuthenticationService`
- `AuthenticationMethodTokenService` (which contains `CreateToken`)

This creates a bootstrap problem: the `CreateToken` endpoint requires authentication to call it, but you have no token to authenticate with, preventing bootstrap of new Flipt instances.

1. **`CookieTokenNotExtracted`** - Verifies that valid tokens work via Authorization header but fail when passed as cookies or metadata
2. **`CreateTokenBootstrapProblem`** - Demonstrates that CreateToken cannot be called without authentication
3. **`AuthorizationHeaderStillWorks`** - Baseline test confirming the normal auth flow works

# Regression Test Findings: Authentication Issues in Flipt

## Summary

The regression test at `/workspace/regression_test.go` successfully demonstrates two critical authentication bugs in the Flipt codebase:

1. **Cookie-based tokens are not extracted** - The authentication middleware only checks the `Authorization` header
2. **No mechanism to skip authentication for specific servers** - Creates a bootstrap problem for token creation

## Root Cause Analysis

### Bug #1: Cookie Token Extraction Failure

**Location:** `/app/internal/server/auth/middleware.go` (lines 16-82)

**Issue:** The `UnaryInterceptor` function only looks for tokens in the `Authorization` header:

```go
const authenticationHeaderKey = "authorization"

func UnaryInterceptor(logger *zap.Logger, authenticator Authenticator) grpc.UnaryServerInterceptor {
    return func(ctx context.Context, req interface{}, info *grpc.UnaryServerInfo, handler grpc.UnaryHandler) (interface{}, error) {
        md, ok := metadata.FromIncomingContext(ctx)
        if !ok {
            logger.Error("unauthenticated", zap.String("reason", "metadata not found on context"))
            return ctx, errUnauthenticated
        }

        authenticationHeader := md.Get(authenticationHeaderKey)  // <-- ONLY checks Authorization header
        if len(authenticationHeader) < 1 {
            logger.Error("unauthenticated", zap.String("reason", "no authorization provided"))
            return ctx, errUnauthenticated
        }
        // ... rest of token extraction
    }
}
```

The middleware never checks for tokens in:
- HTTP cookies (which would be in metadata)
- Custom metadata keys
- Other standard authentication mechanisms
