## Key Findings

**Architecture Overview:**
- The scheduler implements a **two-phase lock-free design**: 
  1. **Scheduling Cycle** (sync): Pod flows through filter/score plugins synchronously, gets assumed in cache
  2. **Binding Cycle** (async): Binding happens in separate goroutine, doesn't block scheduler

**12 Extension Points:**
PreEnqueue → QueueSort → PreFilter → Filter → PostFilter → PreScore → Score → Reserve → Permit → PreBind → Bind → PostBind

**Core Components:**
1. **SchedulingQueue** - Three-tier queue (activeQ, backoffQ, unschedulablePods) with queueing hints for efficient requeuing
2. **Cache** - Maintains "assumed" pods (pessimistic locking), generates snapshots at cycle start
3. **CycleState** - Thread-safe sync.Map for sharing state between plugins within a cycle
4. **Framework** - Plugin registry and executor; instantiates plugins and runs them at extension points
5. **Profiles** - Multiple scheduler configurations can run simultaneously

**Data Flow:**
Pod enters queue → ScheduleOne pops it → Plugins filter/score nodes → Best node selected → Pod assumed in cache → Reserve/Permit plugins verify → Binding runs async → PreBind/Bind/PostBind execute → Done
