## Summary of Findings

I've identified all Go source files in Kubernetes that define and enforce RBAC verb constraints:

### Key Discoveries:

**1. Core Verb Definitions:**
- `pkg/apis/rbac/types.go` and staging API types define `VerbAll = "*"` constant
- Standard verb sets defined in `plugin/pkg/auth/authorizer/rbac/bootstrappolicy/policy.go`:
  - **Read** = ["get", "list", "watch"]
  - **Write** = ["create", "update", "patch", "delete", "deletecollection"]
  - **ReadWrite** = all of the above
  - **ReadUpdate** = ["get", "list", "watch", "update", "patch"]

**2. Verb Matching & Enforcement:**
- `pkg/apis/rbac/v1/evaluation_helpers.go`: Contains `VerbMatches()` function that enforces verb matching (wildcard or exact)
- `pkg/apis/rbac/helpers.go`: Enforces critical constraint - verbs "list", "watch", "create", "deletecollection" **cannot** be used with ResourceNames

**3. Authorization Chain:**
- `plugin/pkg/auth/authorizer/rbac/rbac.go`: Main RBAC authorizer uses `RuleAllows()` which calls `VerbMatches()` to check all requests
- All API requests flow through this single authorization point

**4. Special Verbs:**
- `pkg/registry/rbac/escalation_check.go`: Defines special verbs "escalate" and "bind" for role/binding modifications

**5. Verb Coverage Validation:**
- `staging/src/k8s.io/component-helpers/auth/rbac/validation/policy_comparator.go`: Implements rule coverage analysis including verb coverage checks

The audit confirms that all API handlers consistently enforce verb constraints through the centralized RBAC authorization engine with well-defined verb validation rules.
