## Summary

I analyzed **three major open-source projects** and identified how they handle Prometheus metric registration:

### **Kubernetes/Kubernetes**
- **Core Pattern**: Centralized abstraction layer via `staging/src/k8s.io/component-base/metrics/`
- **Key Files**: KubeRegistry wrapper, Counter/Gauge/Histogram factories, and 40+ component-specific metrics files
- **Registration**: Uses both global `legacyregistry.MustRegister()` and injected `KubeRegistry.MustRegister()`
- **Approach**: Heavy abstraction with validation and deprecation tracking

### **Grafana/Grafana**
- **Core Pattern**: Distributed dependency injection approach
- **Key Files**: 50+ metrics files across services (alerting, dashboards, auth, storage, etc.)
- **Registration**: Uses `prometheus.MustRegister()`, `registerer.MustRegister()`, and `promauto.With(registerer)`
- **Approach**: Each service receives injected `prometheus.Registerer` instances with optional namespace wrapping

### **Prometheus/Prometheus**
- **Core Pattern**: Direct Prometheus client library integration with no abstraction layer
- **Key Files**: 45+ files across discovery, storage, TSDB, scrape, rules, API, and notification components
- **Registration**: Direct `prometheus.MustRegister()` or `registry.MustRegister()` calls
- **Approach**: Simplest pattern with component-specific metrics initialized at startup
