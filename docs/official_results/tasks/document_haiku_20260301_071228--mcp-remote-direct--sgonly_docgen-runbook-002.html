# Envoy Connection Pool Management Troubleshooting Runbook

## 1. Architecture Overview

### Connection Pool Organization

Envoy manages upstream connections through **connection pools**—one pool per cluster, per thread. Each pool maintains connections to upstream hosts and queues requests when connections are unavailable.

**Pool Structure:**
- **Ready Clients**: Connections available to accept new streams
- **Busy Clients**: Connections at maximum concurrent stream capacity
- **Connecting Clients**: Connections being established
- **Pending Streams**: Requests waiting for a connection to become available

**Connection Lifecycle:**

```
Connecting → ReadyForEarlyData → Ready ↔ Busy → Draining → Closed
                                    ↓
                        (exceeds capacity)
```

### HTTP/1.1 vs HTTP/2 Pool Differences

| Aspect | HTTP/1.1 | HTTP/2 |
|--------|----------|--------|
| **Streams per Connection** | 1 (serial) | Multiple (multiplexed) |
| **Concurrent Stream Limit** | 1 | Configured per connection; negotiated via SETTINGS frame |
| **Max Requests Per Connection** | Enforced immediately | Enforced on stream assignment |
| **Busy State Transition** | When 1 request active | When concurrent stream limit reached |
| **GOAWAY Handling** | N/A (HTTP/1.0 close) | Connection transitions to Draining; blocks new streams |
| **Preconnect Benefit** | Limited (serial processing) | High (multiplexing allows burst absorption) |

### TCP Pools

TCP connection pools manage raw TCP connections (no HTTP semantics). Key differences from HTTP pools:

- Fixed 1:1 mapping between connection and stream
- No multiplexing or stream reset recovery
- Idle timeout managed at the TCP protocol level
- Connection failures are fatal to the associated downstream connection
