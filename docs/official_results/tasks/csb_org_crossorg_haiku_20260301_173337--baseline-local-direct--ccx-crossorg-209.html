I've successfully researched and compiled the health check endpoint implementations across the three Go projects. Here's a summary of my findings:

## Key Findings

**Kubernetes** (`staging/src/k8s.io/apiserver/pkg/server/healthz/`):
- Implements `/healthz` (deprecated), `/livez` (liveness), and `/readyz` (readiness) endpoints
- Pluggable `HealthChecker` interface with functions like `InstallHandler()`, `InstallReadyzHandler()`
- Performs checks on: etcd connectivity, informer synchronization, logging, boot hooks, database locks

**Grafana** (`pkg/api/api.go`):
- Implements lightweight `/healthz` (liveness) endpoint without database requirements
- Separate `/api/health` endpoint for full health checks including database
- Health endpoints bypass middleware processing for independence

**Prometheus** (`web/web.go`, `cmd/prometheus/main.go`):
- Implements `/-/healthy` (liveness) and `/-/ready` (readiness) endpoints
- Uses atomic `ready` field in Handler struct to manage readiness state
- Checks TSDB initialization, WAL replay completion, and storage readiness
- Ready middleware enforces readiness on all routes except `/-/healthy`

## Implementation Patterns

All three projects distinguish between **liveness** (is the process running?) and **readiness** (is it ready for traffic?). Kubernetes provides the most comprehensive framework with pluggable health checkers, while Prometheus uses atomic state management for explicit readiness tracking, and Grafana separates database-dependent checks from lightweight liveness probes.
