## Key Findings

I identified the Go source files and symbols in prometheus/prometheus that enforce head block series limits and handle out-of-order/out-of-bounds samples:

### Core Files:
1. **tsdb/head.go** - Manages the in-memory head block with series creation control
2. **tsdb/head_append.go** - Contains the append logic with out-of-bounds validation
3. **tsdb/head_append_v2.go** - Updated AppenderV2 implementation 
4. **tsdb/ooo_head.go** - Handles out-of-order sample storage
5. **storage/interface.go** - Defines error types

### Key Functions:
- **Head.getOrCreateWithOptionalID()** - Creates new series with `PreCreation` callback that can enforce cardinality limits
- **Head.appendableMinValidTime()** - Calculates the minimum valid timestamp threshold
- **headAppender.Append()** / **headAppenderV2.Append()** - Fast path checks for out-of-bounds samples (when `t < minValidTime`)
- **memSeries.appendable()** - Core validation logic that:
  - Checks if sample can be appended in-order
  - Falls back to OOO chunk if within time window
  - Returns `ErrOutOfBounds` when sample is too old

The crash with "out of bound samples" when cardinality is high occurs because excessive series creation combined with out-of-order samples that exceed the time window causes memory pressure and sample rejection cascades.
