## Summary of Changes

**Issue**: Form widget attributes with special characters (quotes, ampersands, angle brackets) were not being properly HTML-escaped when rendered, causing malformed HTML.

**Root Cause**: The Django template filter `stringformat` is marked with `is_safe=True`, which prevents Django's auto-escaping system from escaping values. This meant attribute values were being passed through without proper HTML escaping.

**Solution**: I replaced `{{ value|stringformat:'s' }}` with `{{ value|escape }}` in 4 template files within `django/forms/`:

1. **django/forms/templates/django/forms/attrs.html** - Used for generic form attributes
2. **django/forms/templates/django/forms/widgets/attrs.html** - Used for widget-specific attributes
3. **django/forms/templates/django/forms/widgets/input.html** - Used for input widget values
4. **django/forms/templates/django/forms/widgets/select_option.html** - Used for option widget values

**How the Fix Works**:
- The `escape` filter uses Django's `conditional_escape()` function
- It properly escapes HTML special characters: `&` → `&amp;`, `"` → `&quot;`, `'` → `&#x27;`, `<` → `&lt;`, `>` → `&gt;`
- It does NOT double-escape values already marked as safe (SafeString)
- It converts values to strings before escaping (maintaining the behavior of `stringformat`)

**Impact**:
- Widget attributes with special characters are now properly HTML-escaped
- All form widgets that render these templates benefit from the fix
- Existing functionality is preserved; SafeString attributes still work correctly
- No breaking changes to existing widget rendering

# Widget Attribute Escaping Fix

## Problem
Widget attributes containing special characters (quotes, ampersands, angle brackets) were not being properly HTML-escaped when rendered, potentially causing malformed HTML or security issues.

## Root Cause
The Django template filter `stringformat` is marked with `is_safe=True`, which prevents Django's auto-escaping system from escaping the output. In the widget rendering templates, `{{ value|stringformat:'s' }}` was being used to render attribute values and input values, which resulted in unescaped HTML special characters appearing in the output.

## Solution
Replaced all occurrences of `{{ value|stringformat:'s' }}` with `{{ value|escape }}` in the following template files:

1. django/forms/templates/django/forms/attrs.html
2. django/forms/templates/django/forms/widgets/attrs.html
3. django/forms/templates/django/forms/widgets/input.html
4. django/forms/templates/django/forms/widgets/select_option.html

The `escape` filter uses `conditional_escape()` which:
- Properly escapes HTML special characters: &, ", ', <, >
- Does NOT double-escape values already marked as safe (SafeString)
- Converts values to strings before escaping (like stringformat did)

## Impact
- Widget attributes with special characters are now properly escaped
- Existing functionality is preserved (SafeString values still work)
- All form widgets that include these templates will benefit from the fix
EOF
