## Summary

I've traced how LLVM optimization passes are registered and executed in the standard O2/O3 pipelines. The system consists of **8 key components**:

### Core Components:

1. **PassManager Template** - Manages sequences of passes with `addPass()` and `run()` methods
2. **PassConcept/PassModel** - Type-erased polymorphic dispatch system for storing different pass types
3. **PassRegistry** - Legacy registry for pass information and dependencies
4. **PassRegistry.def** - Macro-based registry of all available passes
5. **PassBuilder** - Central class for constructing optimization pipelines
6. **PassBuilderPipelines.cpp** - Implements the pipeline construction logic

### Execution Chain:

The **buildPerModuleDefaultPipeline()** function is the main entry point that:
1. Calls `buildModuleSimplificationPipeline()` to add early passes (SimplifyCFG, SROA, GlobalOpt, etc.)
2. Calls `buildModuleOptimizationPipeline()` to add main optimizations (inlining, loop optimizations, vectorization)

### Key Insight for O2/O3:

The passes are added in a specific order using `MPM.addPass()`, with O3 enabling additional passes like CallSiteSplitting. The PassManager executes them sequentially, wrapped in the polymorphic PassModel system that allows type-erased storage of different pass types.
