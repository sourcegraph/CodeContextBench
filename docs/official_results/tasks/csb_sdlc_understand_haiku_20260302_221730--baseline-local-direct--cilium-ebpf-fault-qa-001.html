# Debug Q&A: Cilium eBPF Fault Isolation

**Repository:** cilium/cilium
**Task Type:** Debug Q&A (investigation only — no code changes)

## Background

Cilium is a Kubernetes CNI (Container Network Interface) plugin that uses eBPF (Extended Berkeley Packet Filter) programs to enforce network policies and route traffic. In a multi-node Kubernetes cluster, each node runs its own Cilium agent which compiles, loads, and attaches eBPF programs to enforce policies.

## Behavior Observation

**What happens:** When an eBPF program fails to compile or load on one Kubernetes node (e.g., due to a kernel verifier rejection, compilation error, or incompatible kernel features), the other nodes in the cluster continue to enforce their network policies normally. The failing node may log errors or degrade to a fallback mode, but the failure doesn't propagate cluster-wide.

**Why is this notable?** Many distributed systems that rely on shared state or centralized configuration can fail cluster-wide when one node encounters a problem. Cilium's architecture ensures that eBPF datapath failures are isolated to the affected node.

## Key Findings

**Q1: Per-Node eBPF Lifecycle** - Each node independently:
- Probes its kernel at runtime to detect CPU capabilities (BPF instruction set versions)
- Generates node-specific configuration headers with local device information

**Q2: Deployment Architecture** - Cilium uses DaemonSet model:
- One Cilium agent pod per node (via DaemonSet)
- Each daemon instance has independent Loader, PolicyRepository, and EndpointManager
- Compilation lock is per-node, not cluster-wide
- Failures in one agent's eBPF datapath don't block policy distribution from the control plane

**Q3: Policy Distribution vs. Enforcement**:
- Policies are distributed cluster-wide but compiled on-demand per-endpoint
- Each endpoint's eBPF program encodes only the policies needed for its identity
- Policy compilation failures affect only that endpoint on that node
- Other nodes independently resolve and compile their own endpoint policies

**Q4: eBPF Map Scoping**:
- Maps are pinned to each node's independent BPF filesystem mount
- Maps use endpoint-specific naming (e.g., `cilium_ep_policy_00001`)
- Global maps (like connection tracking) are per-node, not shared across the cluster
- Map creation/update failures affect only local endpoints

The architecture's fault isolation comes from the **per-node granularity** of the Cilium agent architecture—each node independently handles eBPF compilation, loading, and map management, making failures inherently local.
