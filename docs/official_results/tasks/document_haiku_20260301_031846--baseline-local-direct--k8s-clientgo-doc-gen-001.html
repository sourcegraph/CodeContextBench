Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Watch Failure Recovery\n\n**Problem:** Network interruption or API server crash closes watch stream.\n\n**Solution (tools/cache/reflector.go, lines 499-607):**\n\n```go\nfunc (r *Reflector) watch(ctx context.Context, w watch.Interface, ...) error {\n  retry := NewRetryWithDeadline(\n    r.MaxInternalErrorRetryDuration,  // Default 1 minute\n    time.Minute,\n    apierrors.IsInternalError,\n    r.clock,\n  )\n\n  for {\n    if err := handleWatch(...); err != nil {\n      // Expired error: too-old resourceVersion\n      if isExpiredError(err) {\n        logger.V(4).Info(\"Watch closed - re-listing\")\n        return nil  // Trigger re-list in ListAndWatch\n      }\n\n      // Internal server error: retry with backoff\n      if apierrors.IsInternalError(err) && retry.ShouldRetry() {\n        continue\n      }\n\n      // Other errors: return and let caller retry\n      return err\n    }\n  }\n}\n```\n\n**Limits:**\n- Maximum backoff: 1 minute (hardcoded, line 503)\n- Internal error retry deadline: 1 minute default, configurable via `MaxInternalErrorRetryDuration`\n- If watch disconnects and doesn't recover within deadline, error is returned to controller\n\n**Tradeoff:** Prevents infinite reconnection loops on persistent failures, but may cause informer to stop.\n\n### 4.2 ResourceVersion Expiry\n\n**Problem:** If informer is paused for too long, its `lastSyncResourceVersion` becomes invalid (\"too old\").\n\n**Symptoms (rest/request.go):**\n```\nerror: \"the object has been deleted and there is no furthering information about its state\"\ncode: 410 (Gone)\n```\n\n**Recovery mechanism (tools/cache/reflector.go, lines 585-589):**\n\n```go\ncase isExpiredError(err):\n  // Don't set LastSyncResourceVersionUnavailable\n  // LIST call with ResourceVersion=RV already ensures we get fresh data\n  logger.V(4).Info(\"Watch closed - expired resourceVersion\")\n\n  // Return from watch() to trigger re-list in ListAndWatch\n  return nil\n```\n\n**Limit:** `etcd` server keeps ~1GB revision history by default. Rapid object changes can exhaust this in seconds.\n\n**Tradeoff:** Client must tolerate re-listing on high-churn clusters. Bookmark support (line 535) reduces impact by providing fresh resourceVersion.\n\n### 4.3 Rate Limiting and Backoff\n\n**Client-side rate limiting** (rest/config.go, lines 117-129):\n\n```go\ntype Config struct {\n  QPS float32   // Default: 5.0\n  Burst int     // Default: 10\n}\n\n// Token bucket rate limiter: allows Burst requests immediately,\n// then refills at QPS rate. # client-go Systems Guide: Advanced Architecture and Operations

## Executive Summary

`client-go` is the Kubernetes client library that bridges user code to the Kubernetes API server. It provides three primary abstractions:

1. **RESTClient** — low-level HTTP client for generic API requests
2. **Typed Clients** — code-generated, type-safe interfaces (via `kubernetes.Clientset`)
3. **Informers** — event-driven caching infrastructure for efficient server monitoring

This guide explains the internal architecture, end-to-end request flow, extension mechanisms, and operational implications of the client-go subsystem.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

client-go is the canonical Go client library for Kubernetes. Its role is to:

- Abstract HTTP transport, authentication, and error handling from application code
- Provide compile-time type safety for core Kubernetes API objects via code generation
- Implement efficient server-side object caching and watch streams for controllers
- Enable polymorphic API access via dynamic clients for CRDs and extension APIs
- Handle authentication provider plugins (OIDC, exec-based auth, custom providers)

### Subsystem Boundaries

**What client-go owns:**
- HTTP request construction and dispatch to the Kubernetes API server
- Configuration management (kubeconfig parsing, authentication setup)
- Object serialization/deserialization (via apimachinery)
- Cache coherency for watched resources
- Event handler dispatch to user code
- Rate limiting and backoff retry logic

**What client-go does NOT own:**
- API server behavior (owned by `k8s.io/apiserver`)
- API type definitions (owned by `k8s.io/api`)
- Serialization formats beyond negotiation (owned by `k8s.io/apimachinery`)
- User code and business logic
