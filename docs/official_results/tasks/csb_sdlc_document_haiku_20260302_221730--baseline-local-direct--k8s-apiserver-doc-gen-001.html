Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Authentication Failure\n- **Symptom**: 401 Unauthorized response\n- **Root causes**: Invalid token, expired certificate, unknown authenticator\n- **Impact**: User cannot access API\n- **Recovery**: User obtains new credentials\n- **Design choice**: Fail-closed for security; no request proceeds without authentication\n\n#### Authorization Failure\n- **Symptom**: 403 Forbidden response\n- **Root causes**: RBAC rule doesn't permit action, webhook denies request\n- **Impact**: User cannot perform action\n- **Recovery**: Admin updates RBAC rules\n- **Design choice**: Fails fast before admission plugins run; prevents wasted work\n\n#### Admission Webhook Timeout\n- **Symptom**: Request eventually fails (default 30s timeout); webhook never responds\n- **Root causes**: Webhook server crash, network partition, slow webhook\n- **Impact**: Request is rejected and user receives 500 error\n- **Recovery**: Fix webhook, retry request\n- **Design choice**: Webhooks have timeouts to prevent unbounded request latency\n- **Evidence**: `handlers/create.go:95` - admission is called within request timeout\n\n#### Storage Failure (etcd unavailable)\n- **Symptom**: 500 Internal Server Error response; readiness check fails\n- **Root causes**: etcd cluster down, network partition, disk full\n- **Impact**: Cannot persist data; all mutations fail\n- **Recovery**: Restore etcd cluster\n- **Mitigations**: Caching layer provides read availability; watch buffering masks brief outages\n- **Evidence**: `storage/cacher/cacher.go` - cacher wraps storage for high availability\n\n#### Request Timeout\n- **Symptom**: Request context exceeds deadline; connection reset\n- **Root causes**: Long-running operation, slow storage, overloaded system\n- **Impact**: Client receives error or connection closes\n- **Recovery**: Automatic retry (if client implements it)\n- **Design choice**: Non-long-running requests get strict timeout; long-running (watch) get graceful timeout\n- **Evidence**: `config.go:228` - RequestTimeout parameter; `filters/timeout.go` - timeout enforcement\n\n#### Flow Control Rejection\n- **Symptom**: 429 Too Many Requests response\n- **Root causes**: Request queue full, priority level saturated\n- **Impact**: Client must retry with backoff\n- **Recovery**: Automatic retry by client\n- **Design choice**: Protects system from overload; fair queuing prevents thundering herd\n- **Evidence**: `config.go:254-259` - flow control configuration; `server/filters/priorityandfairness.go`\n\n#### Panic in Handler\n- **Symptom**: 500 Internal Server Error response\n- **Root causes**: Bug in handler code, admission plugin, storage, or business logic\n- **Impact**: Request fails but server continues running\n- **Recovery**: Fix bug, redeploy\n- **Design choice**: Panic recovery prevents cascading failures\n- **Evidence**: `filters/panic_recovery.go` - wraps handler chain\n\n### 4.2 Design Limits\n\n#### MaxRequestBodyBytes\n- **Parameter**: `config.go:253`\n- **Default**: Typically 1.5MB for kube-apiserver\n- **Purpose**: Prevent unbounded memory usage from large request bodies\n- **Impact**: Large objects (e.g., big ConfigMaps) can hit this limit\n- **Mitigation**: Clients can split large objects; etcd has its own 1.5GB limit\n\n#### MaxRequestsInFlight / MaxMutatingRequestsInFlight\n- **Parameters**: `config.go:256-259`\n- **Default**: 200 read, 100 write for kube-apiserver\n- **Purpose**: Bound concurrency to prevent resource exhaustion\n- **Impact**: Beyond limit, requests queue and receive 429 if queue full\n- **Modern alternative**: Priority and Fairness (`config.go:140`) is more sophisticated\n\n#### RequestTimeout / MinRequestTimeout\n- **Parameters**: `config.go:228-231`\n- **Default**: 1 minute read, 1 hour watch\n- **Purpose**: Prevent unbounded connections\n- **Tradeoff**: Too short and legitimate operations fail; too long and broken clients hold connections\n- **Evidence**: `filters/timeout.go` - timeout is applied per RequestScope\n\n#### Storage Watch Buffer\n- **Location**: `storage/cacher/cacher.go`\n- **Default**: Small bounded buffer for watch events\n- **Purpose**: Decouple event production from consumption\n- **Tradeoff**: Clients watching many objects can fall behind; buffer overflow causes watch restart\n\n### 4.3 Design Tradeoffs\n\n#### GoRestful vs. Custom Routing\n- **Choice**: Uses go-restful library for REST endpoint registration\n- **Advantages**: Structured OpenAPI documentation, parameter marshaling, content negotiation\n- **Disadvantages**: Added latency, less flexible than custom routing\n- **Alternative considered**: Custom HTTP mux\n- **Evidence**: `handler.go:73-100` - NewAPIServerHandler uses restful.Container\n\n#### Request Filtering Chain vs. Middleware\n- **Choice**: Sequential filter chain (each filter wraps next)\n- **Advantages**: Order is explicit and debuggable; filters can short-circuit (e.g., auth failure)\n- **Disadvantages**: Deeply nested function calls; filter order is critical and easy to get wrong\n- **Evidence**: `config.go:1015-1097` - DefaultBuildHandlerChain shows exact order\n\n#### Admission Webhooks in-process\n- **Choice**: Webhooks are called synchronously during request handling\n- **Advantages**: Strongly ordered; failure during mutation is clear\n- **Disadvantages**: Webhook latency directly impacts request latency; webhook crash can bring down API server\n- **Alternative considered**: Async webhook queuing\n- **Evidence**: `handlers/create.go:40-75` - admission is called before storage\n\n#### Separated Mutating and Validating Admission\n- **Choice**: Two phases (mutating first, then validating)\n- **Advantages**: Plugins can assume objects are validated before seeing them (in validate phase)\n- **Disadvantages**: Complexity; must document which phase to use\n- **Evidence**: `admission/interfaces.go:129-144` - separate interfaces\n\n#### In-Memory Cacher Wrapping Storage\n- **Choice**: Cacher is optional but on by default\n- **Advantages**: Dramatically improves list/watch performance; provides high availability during brief etcd outages\n- **Disadvantages**: Added complexity; consistency window (watch lag); memory overhead\n- **Evidence**: `storage/cacher/cacher.go:40` - cacher wraps storage.Interface\n\n#### Authentication Separate from Authorization\n- **Choice**: Authentication identifies who; authorization determines what they can do\n- **Advantages**: Clear separation of concerns; allows different auth mechanisms (cert, token, proxy header)\n- **Disadvantages**: Auth information must flow through context; two separate plugins must agree\n- **Evidence**: `filters/authentication.go`, `filters/authorization.go` - separate stages\n\n#### Audit Wrapped Around Response\n- **Choice**: Response writer is wrapped during audit filter to capture body\n- **Advantages**: Audit sees actual response sent to client\n- **Disadvantages**: Response is buffered in memory; large responses consume memory\n- **Evidence**: `filters/audit.go:80` - response is wrapped\n\n---\n\n## 5.
