## Key Findings:

**Root Cause:** A race condition between the old and new shard instances during resharding where shared metric counters (`enqueuedSamples`, `pendingSamples`) are reset in `start()` before the hardshutdown cleanup code in old `runShard` goroutines can read them.

**The Problem Flow:**
1. During a timeout-triggered hardshutdown in `stop()`, the context is canceled
2. Old `runShard` goroutines detect `ctx.Done()` and need to read `enqueuedSamples` to count dropped samples
3. But `start()` is called immediately after `stop()` and resets `enqueuedSamples.Store(0)`
4. Old goroutines read the reset value (0) instead of the actual count
5. Samples are lost without being properly decremented from the metric

**Why It's Intermittent:**
- Requires exact timing: flush timeout must occur, and metrics reset must happen before old shards read them
- More likely under high load, slow remote endpoints, or frequent target discovery changes
- Depends on goroutine scheduling and network latency

**Evidence:**
- File: `storage/remote/queue_manager.go`
- Lines 1241, 1256: Metric resets in `start()`
- Lines 1566-1569: Hardshutdown path reading the reset values
- Lines 1189-1194: Sequential `stop()` then `start()` without proper synchronization

The report includes specific code references, detailed timelines, diagnostic steps, and three recommended fix strategies with varying complexity trade-offs.

The issue occurs when the shared sample-counting metrics (`pendingSamples`, `enqueuedSamples`) are reset during shard startup before the hardshutdown cleanup from the previous shard instance completes, causing samples dropped during timeout-triggered hardshutdowns to be incorrectly accounted.

**Functions:** `reshardLoop()` (line 1184), `stop()` (line 1269), `start()` (line 1237), `runShard()` (line 1491)
