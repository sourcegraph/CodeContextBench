## Summary of Findings

### **Kubernetes - Lease-Based Leader Election**
Kubernetes uses two complementary mechanisms:
1. **Basic Lease Election** (`staging/src/k8s.io/client-go/tools/leaderelection/`) - Time-based leases with RenewDeadline and LeaseDuration
2. **Coordinated Election** (`pkg/controlplane/controller/leaderelection/`) - Version-aware candidate selection using LeaseCandidate objects

Key failover mechanism: When a leader fails, its lease expires after `LeaseDuration`, allowing other candidates to acquire it. The system uses timestamp-based expiration with clock skew tolerance.

### **etcd - Raft-Based Leader Election**  
etcd uses the Raft consensus algorithm with three layers:
1. **Raft Core** (`server/etcdserver/raft.go`) - Leader detection through SoftState.Lead field
2. **Leadership Transfer** (`server/etcdserver/server.go`) - Explicit transfer via `MoveLeader()` for graceful shutdown
3. **Client API** (`client/v3/concurrency/election.go`) - Campaign/Proclaim/Resign with lease-based expiration

Key failover mechanism: When a leader's lease expires (after its session closes), other candidates' Campaign() calls unblock, triggering automatic re-election.

### **Key Differences**
- **Failover speed**: Kubernetes (15-60 seconds) vs etcd (~150-300ms)
- **Split-brain risk**: Kubernetes (higher, time-dependent) vs etcd (none, Raft-based)
- **Fencing strength**: Kubernetes (weak, clock-dependent) vs etcd (strong, quorum-based)
