## Summary of Findings

The scheduler binding decision reaches the kubelet through this chain of components:

### **1. Scheduler Binding** (`pkg/scheduler/`)
- **DefaultBinder.Bind()** in `defaultbinder/default_binder.go` creates a Binding object and sends it to the API server via the Kubernetes client

### **2. API Server Binding Handler** (`pkg/registry/core/pod/storage/`)
- **BindingREST.Create()** receives the Binding API request
- **assignPod()** and **setPodHostAndAnnotations()** update the pod in etcd by setting `spec.nodeName`

### **3. Kubelet Pod Watch** (`pkg/kubelet/config/`)
- **NewSourceApiserver()** and **newSourceApiserverFromLW()** establish a watch on the API server for pod changes filtered by `spec.nodeName`
- A Reflector continuously watches and sends PodUpdate events to the kubelet

### **4. Kubelet Sync Loop** (`pkg/kubelet/kubelet.go`)
- **syncLoopIteration()** receives pod updates from the API server
- **HandlePodAdditions()** processes ADD events for new pods

### **5. Pod Admission Gates** (`pkg/kubelet/`)
- **canAdmitPod()** checks admission by iterating through registered **PodAdmitHandlers**
- Admission handlers (from `pkg/kubelet/lifecycle/`) decide whether the pod can start based on resource availability, AppArmor policies, etc.

### **6. Pod Execution**
- **SyncPod()** handles resource setup and delegates to the container runtime to start containers
