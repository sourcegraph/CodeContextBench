## Incident Report

**Filed by:** Platform Reliability
**Severity:** P1
**Incident ID:** INC-7823
**Affected Service:** Content rendering pipeline

### Symptoms

When a template uses three or more levels of inheritance (e.g., `base.html` -> `layout.html` -> `page.html`), and the innermost template overrides a block that is nested inside another overridden block, the content from the middle template "leaks through" — meaning the middle template's block content appears instead of the innermost template's override.

This regression was introduced sometime recently. The template engine's block resolution logic handles two-level inheritance correctly but fails at three or more levels.

### Expected Behavior

The innermost template's block content should always take precedence, regardless of how many inheritance levels exist. This is standard template inheritance semantics.

### What We Know

- The issue is in the template compilation or rendering phase, not in template loading
- Two-level inheritance works correctly
- The bug appears when blocks are nested inside other blocks (not top-level blocks)
- No tests exist for this specific three-level nested scenario

### The Solution  
**File Changed:** `django/template/loader_tags.py` (Lines 150-157)

**The Fix:** Modified the `ExtendsNode.render()` method to only extract top-level blocks from parent templates, rather than all blocks recursively. This prevents nested blocks from parent templates from interfering with nested block overrides from child templates.

**Before:**
```python
blocks = {
    n.name: n
    for n in compiled_parent.nodelist.get_nodes_by_type(BlockNode)
}
```

**After:**
```python
# Only get top-level blocks from the parent's nodelist,
# not nested blocks. Nested blocks will be encountered
# during rendering of their parent blocks.
blocks = {
    n.name: n
    for n in compiled_parent.nodelist
    if isinstance(n, BlockNode)
}
```

### Why This Works
1. Child templates can still override blocks at any nesting level (they use `get_nodes_by_type`)
2. Parent templates only have top-level blocks added to block_context
3. Nested blocks remain in their parent block's structure
4. When a parent block is rendered, nested blocks are encountered and properly use overrides from block_context
5. This works correctly for any depth of template inheritance
