Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n**Authentication Failure (401 Unauthorized)**\n- Invalid or missing credentials\n- Authenticator returns error\n- Response includes audit event with `user.anonymous`\n- Connection remains open (client can retry with valid credentials)\n- Impact: Client must resubmit with valid auth\n\n**Authorization Failure (403 Forbidden)**\n- User authenticated but lacks permission for verb/resource\n- Request fully processed (authenticated, admitted) before authz check\n- Impact: Confirms existence of resource (information leak - design tradeoff)\n\n**Admission Rejection (400 Bad Request or Custom Status)**\n- Mutating webhook rejects due to policy violation\n- Validating webhook rejects after mutation or storage write\n- Audit event includes rejection reason\n- Impact: Object may exist in storage but client sees rejection\n\n**Storage Failure (500 Internal Server Error)**\n- Etcd unreachable or timeout\n- Serialization error\n- Quota exceeded\n- Request marked as failed in audit\n- Impact: Request incomplete, may be partially persisted (depends on storage semantics)\n\n**Rate Limiting (429 Too Many Requests)**\n- Request queue full in Priority & Fairness controller\n- Response includes `Retry-After` header\n- Request NOT processed, no admission or storage mutation\n- Impact: Request is dropped at gateway before reaching handlers\n\n**Timeout (504 Gateway Timeout or 408 Request Timeout)**\n- Request context deadline exceeded\n- Non-long-running requests: timeout after `RequestTimeout` config\n- Watch requests: can run indefinitely\n- If mutating operation: may partially complete\n- Impact: Connection closed, client must retry (potential duplication)\n\n**Panic (500 Internal Server Error)**\n- Handler panic caught by `WithPanicRecovery`\n- Stack trace logged, response set to 500\n- Recovery prevents cascade failures\n- Impact: Minimal - single request fails, server continues\n\n### 4.2 Resource Limits\n\n**Request Body Size**\n- `maxRequestBodyBytes` config limits request body\n- Default: unlimited (0)\n- Exceeded: 413 Request Entity Too Large\n- Applied at handler level (before unmarshaling)\n\n**In-Flight Request Limits**\n- `MaxRequestsInFlight`: max concurrent requests\n- `MaxMutatingRequestsInFlight`: max concurrent mutating requests\n- Exceeded: 429 Too Many Requests (new requests queued)\n- Design: Prevents resource exhaustion\n\n**Timeout Limits**\n- `minRequestTimeout`: minimum allowed client timeout\n- `requestTimeoutUpperBound`: max timeout for context\n- Set: 34 seconds default (unusual number to catch errors)\n- Design: Prevents clients hanging indefinitely, but allows long operations\n\n**Watch Streaming**\n- Watches are long-running (exempt from timeouts)\n- Streaming responses sent incremental to client\n- Client can cancel with TCP FIN or timeout\n\n**Memory Limits**\n- No built-in limits on number of objects\n- Relies on etcd/storage backend limits\n- No garbage collection by apiserver\n\n### 4.3 Design Tradeoffs\n\n**Tradeoff: Admission Ordering vs. Flexibility**\n\n*Decision*: Admission controllers execute in fixed order (mutating \u2192 validating)\n\n*Benefit*: Predictable behavior, validators see final state\n*Cost*: Can't inject conditional validators, can't skip controllers\n\n*Evidence*: `pkg/admission/chain.go` hardcodes mutating vs validating phases\n\n**Tradeoff: Handler Chain Wrapping vs. Plugin System**\n\n*Decision*: Filter middleware wraps HTTP handler (not plugin system)\n\n*Benefit*: Type-safe composition, no reflection, clear execution order\n*Cost*: Must recompile to add filters, can't dynamically load middleware\n\n*Evidence*: `pkg/server/config.go` DefaultBuildHandlerChain returns wrapped handler\n\n**Tradeoff: Synchronous Admission vs. Async Webhooks**\n\n*Decision*: Admission runs synchronously in request handler\n\n*Benefit*: Client receives immediate feedback (success/failure)\n*Cost*: Webhook latency directly impacts API latency (many orgs add load balancer timeout)\n*Evidence*: Many deployments disable validating webhooks on high-volume resources\n\n**Tradeoff: Storage Interface Flexibility vs. Consistency**\n\n*Decision*: Storage interface minimal, no transaction support\n\n*Benefit*: Simple, works with any backend\n*Cost*: No ACID transactions across multiple resources (leads to eventual consistency patterns)\n*Evidence*: `pkg/registry/rest/rest.go` defines simple Create/Update/Delete\n\n**Tradeoff: Audit After Storage vs. Before**\n\n*Decision*: Audit runs after handlers complete\n\n*Benefit*: Audit logs reflect actual stored state\n*Cost*: Audit lag (asynchronous in many backends), audit webhook latency impacts request latency\n*Evidence*: `pkg/endpoints/filters/audit.go` wraps handlers, not called before\n\n**Tradeoff: RequestInfo Parsing Overhead**\n\n*Decision*: All requests parsed to extract namespace/resource/verb\n\n*Benefit*: Consistent across all handlers, enables RBAC\n*Cost*: CPU overhead on every request\n*Evidence*: `WithRequestInfo` (line 1091) called for all requests including 404s\n\n**Tradeoff: Director Pattern (GoRestful + NonGoRestfulMux)**\n\n*Decision*: Dual-handler architecture with director dispatch\n\n*Benefit*: Enables OpenAPI generation (go-restful) while supporting delegation\n*Cost*: Complexity, potential for routing bugs, go-restful performance overhead\n*Evidence*: `pkg/server/handler.go` director type, lines 121-154\n*Comment*: This is widely recognized as a wart; maintainers acknowledge go-restful dependency should be removed\n\n**Tradeoff: Imperative vs. Declarative Request Processing**\n\n*Decision*: Handlers are imperative (hand-written, not declarative schemas)\n\n*Benefit*: Flexibility (can implement non-CRUD operations), precise control\n*Cost*: High complexity for standard CRUD, duplicated logic across handlers\n*Evidence*: `pkg/endpoints/handlers/create.go` (10KB) vs. similar in other frameworks\n\n---\n\n## 5. Extension Points, Customization Hooks, and Risks\n\n### 5.1 Primary Extension Points\n\n#### 5.1.1 Handler Chain Customization\n\n**Extension Point**: `Config.BuildHandlerChainFunc`\n\n```go\ntype MyAuthorizer struct {}\n\nfunc (a *MyAuthorizer) Authorize(ctx context.Context, attrs authorization.Attributes) (decision authorizer.Decision, reason string, err error) {\n    // Allow cluster-admin Operational Implications\n\n### 6.1 Performance Characteristics\n\n**Request Latency Breakdown** (typical values):\n\n1. **Handler chain overhead**: 1-2ms\n   - WithPanicRecovery, WithRequestInfo, context setup\n2. **Authentication**: 5-50ms\n   - Token validation, webhook calls\n3. **Authorization**: 10-100ms\n   - RBAC policy evaluation (worse with large role sets)\n4. **Admission**: 50-500ms\n   - Webhook roundtrips (worst offender in most deployments)\n5. **Storage read**: 5-50ms (etcd 100ms+ over WAN)\n6. **Encoding/serialization**: 1-10ms\n\n**Total (median create request)**: 75-300ms\n**P99 (with webhook retries, lock contention)**: 1-5s\n\n**Optimization Opportunities**:\n- Move webhook validation to separate component\n- Cache authorization decisions (10s TTL typical)\n- Batch etcd requests for list operations\n- Use protobuf instead of JSON (3-5x faster)\n\n### 6.2 Memory Usage\n\n**Per-Server Baseline**: 100-300MB\n- go-restful reflection for registered routes: ~50MB\n- OpenAPI schema: ~50MB\n- Admission controller state: variable\n\n**Per-Request Memory**: 1-10MB\n- Request deserialization buffer\n- Audit event structures\n- Watch connection metadata (long-lived)\n\n**Watch Connections**: 10KB-1MB each\n- 10,000 watch connections = 100-1000MB RAM\n- Long-lived TCP connections from clients\n\n**Total Capacity**: Typically 1000-10,000 concurrent requests on small cluster, scales with memory limit\n\n### 6.3 Network I/O\n\n**Request Size Limits**:\n- Body: configurable (default unlimited)\n- Large requests (>100MB) not recommended\n- Streaming watch responses sent incrementally\n\n**Backend Communication**:\n- etcd: Roundtrip 50-200ms over LAN\n- Webhook: Roundtrip 100-1000ms common\n- WAN deployments: add 50-500ms # Kubernetes API Server Architecture and Extension Guide

## Executive Summary

The Kubernetes apiserver library (`staging/src/k8s.io/apiserver/`) is a foundational subsystem that implements the core request handling pipeline for API servers. It provides the HTTP handler chain, resource installation, authentication/authorization, admission control, and lifecycle management. The architecture emphasizes layered responsibility separation, with each layer addressing distinct concerns (security, rate limiting, request validation, storage) while maintaining extension points for customization.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose
The apiserver library provides a generic, reusable foundation for building Kubernetes-compatible REST API servers. Its primary responsibilities are:
- **Request routing and handling** via an HTTP handler chain
- **Resource lifecycle management** (CRUD operations + watch)
- **Security enforcement** (authentication, authorization, admission control)
- **Request management** (rate limiting, timeout, request tracking)
- **Lifecycle hooks** (startup, shutdown, health checks)
- **API discovery and OpenAPI generation**

### Boundaries
- **Upstream dependencies**: Uses apimachinery (type system, serialization, runtime schema) and client-go (client configuration)
- **Downstream consumers**: Used by kube-apiserver, kube-aggregator, and custom API servers built on top
- **Storage abstraction**: The library defines `rest.Storage` interface but does NOT implement storage; consumers provide storage implementations
- **Configuration**: Does NOT manage configuration files; consumers provide `Config` struct with pre-configured components
