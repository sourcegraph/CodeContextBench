**Critical Finding - Middleware Order:**
Django processes `settings.MIDDLEWARE` in **REVERSE order** using `reversed(settings.MIDDLEWARE)` (line 41 in base.py). This creates an onion-like middleware structure where:
- Middleware listed **first** in settings becomes the **outermost layer**
- Middleware listed **first** processes requests **first**
- This is opposite to traditional WSGI middleware stacking

**Middleware Stack Construction:**
- Each middleware is instantiated with the previous handler as `get_response`
- Creates a nested callable chain: `MW1(MW2(MW3(...(view_handler))))`
- The final `_middleware_chain` is stored and called for each request

**Request/Response Flow:**
- Requests enter through the outermost middleware first
- Responses flow back through the same layers in reverse
- Special middleware hooks: `process_view()` (called just before view), `process_template_response()`, and `process_exception()` are applied in specific orders
