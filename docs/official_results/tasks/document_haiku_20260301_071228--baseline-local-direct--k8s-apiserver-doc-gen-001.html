Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Graceful Degradation Under Failure\n\n| Failure Mode | Effect | Recovery |\n|---|---|---|\n| **Storage backend (etcd) down** | All write operations fail with 500 Internal Server Error; reads may work from cache | Automatic retry via client; server continues accepting requests |\n| **Admission webhook timeout** | Request times out (408 Request Timeout or context deadline); transaction fails | Webhook retried by admission chain; request fails; client can retry |\n| **Authentication service unavailable** | Requests using that authenticator fail (typically webhook-based) | Other authenticators may succeed (token, cert); token auth always available if configured |\n| **Authorization webhook unhealthy** | Requests queued/delayed pending webhook recovery; May eventually timeout | Server implements denial-of-service (DoS) protection via MaxInFlightLimit; requests rejected with 429 |\n| **Out of memory** | Server panics (caught by PanicRecovery filter); returns 500 error | Kubernetes restarts pod; requests in flight may be lost; persistent data safe in etcd |\n| **Goroutine leak in handler** | Memory grows; eventually OOM; affects responsiveness | Mitigated by request timeout enforcement; goroutines are backgrounded and will eventually be killed |\n| **Listener socket error** | HTTP server fails to bind port; StartFailed hook called; server cannot run | Must fix socket issue (port in use, permissions) and restart server |\n\n#### Request-Level Failures\n\n| Failure Type | Status Code | Effect |\n|---|---|---|\n| **Malformed URL** | 400 Bad Request | RequestInfo parsing fails; request rejected early (cannot route) |\n| **Unsupported API version** | 404 Not Found | Resource not registered; director finds no route |\n| **Authentication failed** | 401 Unauthorized | Authentication filter rejects; no user identity established |\n| **Authorization failed** | 403 Forbidden | Authorization filter rejects; user lacks permission |\n| **Admission rejected** | 400/422 Bad Request or Unprocessable Entity | Plugin policy violation; object fails validation/mutation |\n| **Optimistic concurrency conflict** | 409 Conflict | Update request includes stale resourceVersion; storage refuses write |\n| **Request timeout** | 408 Request Timeout | Handler did not complete before deadline; This prevents authorization bypass but may cause false negatives.\n\n**Rationale**: Kubernetes prioritizes security over availability. Ambiguous authorization decisions must reject requests. This prevents privilege escalation.\n\n**Implication**: Misconfigured admission/authz policies can cause service outages. Requires careful policy development and testing.\n\n**Code location**: `pkg/admission/chain.go:70` - returns error immediately on first handler error.\n\n#### Request Filtering Order\n\n**Tradeoff**: Authentication happens before authorization. This means unauthenticated requests are rejected before authorization is checked.\n\n**Rationale**: Cannot make authorization decision without knowing who is requesting. Unauthenticated requests are always unauthorized anyway.\n\n**Implication**: Some authorization failures are masked by earlier auth failures. Cannot distinguish \"unauthorized\" from \"not authenticated\" without examining logs.\n\n#### Admission Plugins are Synchronous\n\n**Tradeoff**: Admission plugins execute synchronously in the request path. This means slow plugins delay all requests.\n\n**Rationale**: Admission must provide timely feedback to clients. Asynchronous admission would complicate retry logic.\n\n**Implication**: Slow admission webhooks impact API server latency for all users. Requires webhook timeout tuning and performance monitoring.\n\n#### Object Versioning is Implicit (Resource Version)\n\n**Tradeoff**: Objects track versions internally (resourceVersion field). Clients must manage versions for optimistic concurrency.\n\n**Rationale**: Implicit versioning simplifies API; clients see versions in responses. Explicit version objects would require separate APIs.\n\n**Implication**: Clients that ignore versions may cause lost updates. Requires client-side discipline and retry logic.\n\n#### Single Authorization Chain (Union)\n\n**Tradeoff**: Multiple authorizers are combined in a union (any \"Allow\" grants access; explicit \"Deny\" prevents access).\n\n**Rationale**: Allows composing multiple auth mechanisms (RBAC + ABAC + webhook). Flexible for varied deployment environments.\n\n**Implication**: Auth bugs in one component may leak access from another. Requires careful understanding of combined policies.\n\n#### Graceful Shutdown is Soft-Limited\n\n**Tradeoff**: Graceful shutdown waits for in-flight requests with timeout. Requests exceeding timeout are forcibly terminated.\n\n**Rationale**: Prevents indefinite hangs during shutdown. Balances politeness with responsiveness.\n\n**Implication**: Long-running requests may be killed during shutdown. Requires clients to handle connection drops and implement retry logic.\n\n#### Watch Streams Use HTTP Chunked Transfer Encoding\n\n**Tradeoff**: Watches use standard HTTP chunks instead of WebSockets or gRPC.\n\n**Rationale**: HTTP compatibility; works through most firewalls/proxies. Simpler protocol.\n\n**Implication**: Watch streams vulnerable to connection timeouts. Load balancers may terminate idle connections. Requires periodic keep-alive events.\n\n---\n\n## 5. ## Executive Summary

The Kubernetes apiserver library (`k8s.io/apiserver`) is a generic framework for building Kubernetes aggregated API servers. It provides a complete HTTP server implementation with integrated authentication, authorization, admission control, audit logging, storage abstraction, and lifecycle management. The architecture follows a layered, composable design where HTTP requests pass through a middleware chain before reaching REST handlers backed by configurable storage.

This document provides an in-depth analysis of the apiserver subsystem, including its architecture, component responsibilities, request-path flow, extension points, and operational considerations.
