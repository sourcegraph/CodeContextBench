## Key Findings:

### **Files Analyzed (30+ critical files)**
- Traffic cop entry point and orchestration logic
- Parser (lexer + grammar), semantic analyzer, rewriter

- Executor with Volcano-style iterator dispatch

### **Query Execution Flow**
```
SQL String
  ↓ [raw_parser]
RawStmt (unanalyzed parse tree)
  ↓ [parse_analyze]
Query (semantically valid)
  ↓ [QueryRewrite]
Query (rewritten, may be multiple)
  ↓ [planner / standard_planner]
  ├─ Phase A: Path generation (allpaths.c) - explores execution alternatives
  └─ Phase B: Plan creation (createplan.c) - converts best Path to Plan
PlannedStmt (executable specification)
  ↓ [Executor: ExecInitNode]
PlanState tree (initialized, with function pointers)
  ↓ [ExecutorRun: ExecProcNode loop]
Tuples (result)
```

### **Key Architectural Insights**

1. **Two-Phase Optimization**: Path objects separate cost estimation from Plan instantiation, enabling sophisticated planning without plan tree manipulation
   
2. **Volcano Iterator Model**: Pull-based execution via function pointers enables:
   - Memory efficiency (no full materialization)
   - Early termination (LIMIT stops pulling)
   - Pipeline parallelism
   
3. **Clean Layered Architecture**: Each stage (parser→analyzer→rewriter→optimizer→executor) has well-defined input/output contracts

4. **Extensibility**: Custom executor nodes (ForeignScan, CustomScan) plug in via function pointers without core modifications
