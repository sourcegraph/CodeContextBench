## Investigation Summary

**Root Cause Identified:** The v38 table panel migration has a critical conditional logic flaw in `/workspace/apps/dashboard/pkg/migration/schemaversion/v38.go` (lines 110-137).

**The Bug:**
When processing table panels, the code checks if `fieldConfig.defaults.custom` exists. If it doesn't, the entire panel is skipped with an early `continue` statement at line 123. This prevents the `migrateOverrides()` function from being called, which means field override properties that reference `custom.displayMode` never get migrated to `custom.cellOptions`.

**Why It Fails:**
- Dashboards saved in v10.3 without explicit `fieldConfig.defaults.custom` are not processed
- Field override configurations that apply column-level formatting are left unmigrated
- When v10.4 renders these tables, it looks for `custom.cellOptions` but finds the old `custom.displayMode` format, which it ignores

**Why Some Dashboards Work:**
- Dashboards with explicit `fieldConfig.defaults.custom` in their JSON pass the type assertion and are processed correctly
- All test fixtures only cover this working case, missing the regression scenario

**Evidence:**
- Backend: `v38.go` skips processing if `custom` doesn't exist (line 120-123)
- Frontend: `DashboardMigrator.ts` (lines 644-674) handles this more gracefully by still processing overrides even when defaults lack displayMode

The full detailed report with all files, functions, and recommendations has been written to `/logs/agent/investigation.md`.
