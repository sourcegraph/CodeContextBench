Policy, No Available CPUs**\n\nIf a pod requests exclusive CPUs but all reserved CPUs are exhausted:\n- Static policy cannot find contiguous CPU set\n- `AddContainer()` returns error\n- Pod container startup fails with `CPUManagerError`\n- **Pod remains in Pending state** until CPUs become available\n\n**Mitigation**:\n- Resize pod request to non-exclusive CPUs\n- Scale down other exclusive pods\n- Requires manual intervention; no auto-remediation\n\n**Design Tradeoff**:\n- Static policy guarantees CPU isolation but can starve pods\n- `none` policy avoids starvation but no CPU affinity\n\n---\n\n**Scenario 2: Reconciliation Lag**\n\nIf a container escapes its assigned cpuset (due to container runtime bug or direct cgroup manipulation):\n- Reconciliation runs every 60s (default)\n- Container consumes CPUs for up to 60 seconds before move\n- **Fairness impact**: One pod can exceed its fair share\n\n**Design Decision**: Trade monitoring frequency for reduced lock contention on container manager\n\n---\n\n### 4.2 Memory Manager Failures\n\n**Scenario 1: NUMA Allocation Fails**\n\nIf memory manager cannot allocate memory on preferred NUMA nodes:\n- Static policy rejects the allocation\n- `AddContainer()` returns error\n- Pod cannot start\n\n**Scenario 2: Memory Overcommit**\n\nIf actual pod memory usage exceeds assigned limit:\n- Cgroup OOM killer terminates container\n- **Observed in practice**: BestEffort pods killed first due to OOM score (-999)\n- **Log evidence**: Container killed by OOM killer signal\n\n**Design Tradeoff**:\n- Cgroup v2 adds `memory.min` to protect high-priority tiers\n- Adds complexity but improves reliability\n\n---\n\n### 4.3 Topology Manager Admission Failures\n\n**Scenario: Topology Affinity Cannot Be Met**\n\nWith `restricted` policy, if CPU/Memory/Device hints have no intersection:\n\n```\nExample:\n  CPU Manager hints: NUMA node 0\n  Memory Manager hints: NUMA node 1\n  Device Manager hints: NUMA node 2\n\n  Intersection: empty\n  Policy: restricted\n  \u2192 Pod REJECTED with TopologyAffinityError\n```\n\n**Impact**: Valid resource requests rejected due to topology constraint mismatch\n\n**Mitigation**:\n- Use `best-effort` policy to prefer aligned but accept misaligned\n- Adjust pod requests to align with available topology\n- Modify device plugin topology reporting\n\n**Design Tradeoff**:\n- `restricted` prevents performance-hurting misalignment (good for latency-sensitive workloads)\n- `best-effort` allows more schedules but degrades tail latency\n\n---\n\n### 4.4 Resource Quota Enforcement Limits\n\n**Hard Limits** (via CFS quota, memory.max):\n- CPU: Hard cap enforced by kernel CFS scheduler\n- Memory: Hard cap enforced by cgroup OOM killer\n- Cannot be violated; violations cause container termination\n\n**Soft Limits** (via CPU shares, memory.high):\n- CPU shares: Kernel guarantees proportional time, but doesn't hard-cap\n- Memory.high: Kernel reclaims memory via throttling, but allows temporary overage\n\n**Design Limit**: No intermediate guarantees between soft and hard; either pod gets CPU or doesn't\n\n---\n\n### 4.5 Cgroup Version Incompatibility\n\n**Cgroup v1 Issues** (Deprecated):\n- Multiple independent subsystems (cpu, memory, cpuset, pids)\n- More memory overhead per cgroup\n- Less precise memory accounting\n- **Warning logged** at startup: \"cgroup v1 detected. cgroup v1 support is deprecated...\"\n\n**Cgroup v2 Requirements**:\n- Kernel 5.4+ required for basic support\n- Kernel 5.8+ required for root `cpu.stat` (required for accurate CPU accounting)\n- **Code path**: Lines 52-59 in `container_manager.go` with kernel version checks\n\n**Design Decision**: Support both for backwards compatibility, but strongly encourage v2 migration\n\n---\n\n### 4.6 Platform-Specific Limits\n\n**Linux** (Full support):\n- All managers enabled: CPU, Memory, Topology, Device, DRA\n- Cgroup enforcement via cgroupfs\n\n**Windows** (Minimal support):\n- CPU Manager: Stub (no affinity)\n- Memory Manager: Stub\n- Topology Manager: Stub\n- Device Manager: Basic support only\n- Cgroup: Not available; relies on container runtime limits\n\n**Other Platforms** (Stub):\n- All managers return no-op stubs\n- No resource isolation beyond runtime defaults\n\n---\n\n### 4.7 Node Allocatable Enforcement Tradeoffs\n\n**Enforcement Goal**: Ensure `Node Allocatable = Node Capacity - Reserved`\n\n**Mechanism**:\n- Create cgroup at `kubepods` with resource limits\n- Force all pod cgroups under `kubepods`\n- Periodically enforce limits via cgroup manager updates\n\n**Challenge: Memory Overcommit** (lines 550-558):\n```\nIf actual pod memory usage > allocatable memory:\n  \u251c\u2500 Reduce burstable tier limit (to free memory for guaranteed)\n  \u251c\u2500 If still overcommitted, retry with increased frequency\n  \u2514\u2500 If unresolvable, log error but allow operation\n```\n\n**Design Tradeoff**:\n- Strict enforcement can crash workloads (OOM kill)\n- Permissive enforcement allows overcommit (performance degradation)\n- Kubernetes chose permissive: Log but don't kill\n\n---\n\n## Part 5: Extension Points, Customization Hooks, and Associated Risks\n\n### 5.1 CPU Manager Policy Customization\n\n**How to Add New Policy**:\n\n1. **Create policy file**:
