## Summary of Analysis

The document traces the complete query processing pipeline from the traffic cop entry point (`exec_simple_query()`) through all major stages:

1. **Parser Stage** — `raw_parser()` uses flex lexer (scan.l) and bison grammar (gram.y) to produce `RawStmt` nodes

2. **Semantic Analyzer** — `parse_analyze_*()` functions transform `RawStmt` into fully resolved `Query` nodes with type checking and name resolution

3. **Query Rewriter** — `QueryRewrite()` applies stored rules and view expansions, potentially producing multiple queries from one input

4. **Two-Phase Optimization**:
   - **Phase 1 (allpaths.c)**: `set_base_rel_pathlists()` and friends enumerate candidate execution Paths with cost estimates
   - **Phase 2 (createplan.c)**: `create_plan()` converts the lowest-cost Path into an executable Plan tree

5. **Executor** — Implements the classic Volcano pull-based model:
   - `ExecutorStart()` → `ExecInitNode()` builds a PlanState tree (mirrors Plan structure)
   - `ExecutorRun()` → recursive `ExecProcNode()` dispatch via function pointers
   - Each plan node type (SeqScan, NestLoop, Sort, Agg, etc.) implements its own tuple-pulling logic

The analysis identifies key design patterns (Compiler Pipeline, Cost-Based Search, Iterator Model, Method Dispatch), documents data structure transformations between stages, and explains the interface contracts between components.
