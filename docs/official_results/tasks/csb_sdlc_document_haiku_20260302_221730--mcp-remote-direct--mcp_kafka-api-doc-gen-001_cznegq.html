 // Commit before losing partitions\n    }\n\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Optionally seek to specific offsets\n    }\n});\n```\n\n**Subscription Semantics**:\n- **Not incremental**: New `subscribe()` call replaces previous subscription\n- Empty list is equivalent to `unsubscribe()`\n- Triggers rebalance after next `poll()` if membership/topics changed\n- Cannot be mixed with `assign()`\n\n**Pattern Subscriptions**:\n```java\n// Classic regex patterns\nconsumer.subscribe(Pattern.compile(\"topic-*\"), listener);\n\n// Google RE2/J patterns (new consumer protocol)\nconsumer.subscribe(new SubscriptionPattern(\"topic-[0-9]+\"), listener);\n```\n\n### Manual Partition Assignment\n\nFor fine-grained control without group coordination:\n\n```java\nTopicPartition partition0 = new TopicPartition(\"topic\", 0);\nTopicPartition partition1 = new TopicPartition(\"topic\", 1);\nconsumer.assign(Arrays.asList(partition0, partition1));\n```\n\n**Assignment Semantics**:\n- No group coordination or rebalancing\n- No automatic partition reassignment on failures\n- Manual offset management required\n- Can change assignment at any time with another `assign()` call\n- Cannot be mixed with `subscribe()`\n\n**Use Cases for Manual Assignment**:\n- Standalone consumers without group semantics\n- Guaranteed processing of specific partitions (e.g., each instance processes subset)\n- External assignment coordination (custom leader election)\n\n### State Queries\n\n```java\nSet<String> subscription()              // Current subscribed topics\nSet<TopicPartition> assignment()        // Currently assigned partitions\n\n// Subscription and assignment are mutually exclusive:\n// - subscription() returns topics -> wait for assignment to complete\n// - assignment() returns partitions -> rebalance may change assignment\n```\n\n### Unsubscribing\n\n```java\nconsumer.unsubscribe();  // Clears both subscriptions and assignments\n```\n\n---\n\n## Polling and Data Fetching\n\n### The Poll Loop\n\nThe `poll()` method is the heartbeat of consumption. It:\n1. Sends heartbeats to broker\n2. Handles rebalancing\n3. Fetches new records\n4. Executes auto-commits if enabled\n\n```java\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord<String, String> record : records) {\n        System.out.println(\"Offset: \" + record.offset() +\n                         \", Key: \" + record.key() +\n                         \", Value: \" + record.value());\n    }\n}\n```\n\n### Poll Timeout Semantics\n\nThe `timeout` parameter controls **maximum wait for new data**, but `poll()` may block longer:\n\n```java\nConsumerRecords<K, V> poll(Duration timeout)\n```\n\n**Blocking Conditions**:\n1. **Data available** \u2192 Returns immediately with records\n2. **No data, rebalance needed** \u2192 Blocks for rebalance (may exceed timeout)\n3. **No data, no rebalance** \u2192 Blocks up to `timeout`\n4. **Timeout expires** \u2192 Returns empty records\n\n**Behavioral Gotchas**:\n- Rebalance callbacks execute during `poll()` and are **not counted** against timeout\n- If rebalance takes 5 seconds and timeout is 100ms, `poll()` blocks for ~5 seconds\n- Position may advance past `timeout` when filtering transactions (`isolation.level=read_committed`)\n- `max.poll.records` limits records returned (default 500), allowing frequent calls\n\n### Return Value\n\n```java\nConsumerRecords<K, V> records = consumer.poll(Duration.ofMillis(100));\n\n// Empty records are normal:\nif (records.isEmpty()) {\n    // No data available, continue polling\n    continue;\n}\n\n// Process records by partition\nfor (TopicPartition partition : records.partitions()) {\n    List<ConsumerRecord<String, String>> partitionRecords =\n        records.records(partition);\n    for (ConsumerRecord<String, String> record : partitionRecords) {\n        // Process record\n    }\n}\n\n// Get next offsets for committing\nMap<TopicPartition, OffsetAndMetadata> nextOffsets = records.nextOffsets();\n```\n\n### Position Tracking\n\nThe consumer tracks two positions:\n\n1. **Current Position**: Next offset to fetch\n   - Advanced by each `poll()` that returns records\n   - Can be set via `seek()`\n   - Returned by `position()`\n\n2. **Committed Position**: Last durably committed offset\n   - Used on restart/rebalance to resume consumption\n   - Set via `commitSync()`/`commitAsync()`\n   - Returned by `committed()`\n\n**Position Relationship**:\n```\nInitial state: position == committed\nAfter polling 10 records: position = committed + 10\nAfter commitSync(): committed == position\n```\n\n---\n\n## Offset Management\n\n### Automatic Offset Commits (Default)\n\nThe consumer automatically commits offsets periodically:\n\n```java\n// Enable automatic commits (default behavior)\nprops.setProperty(\"enable.auto.commit\", \"true\");\nprops.setProperty(\"auto.commit.interval.ms\", \"1000\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(Arrays.asList(\"topic\"));\n\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord<String, String> record : records) {\n        // Process record - offset automatically committed after interval\n    }\n}\n```\n\n**Auto-Commit Semantics**:\n- Commits happen **in the background** at `auto.commit.interval.ms` intervals\n- Only offsets from fully processed batches are committed\n- **Exactly-once semantics are NOT guaranteed** if processing fails between poll and commit\n\n### Manual Offset Commits\n\nFor precise control over when records are considered consumed:\n\n```java\nprops.setProperty(\"enable.auto.commit\", \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(Arrays.asList(\"topic\"));\n\nwhile (running) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    try {\n        for (ConsumerRecord<String, String> record : records) {\n            processRecord(record);\n        }\n        // Commit only after processing completes\n        consumer.commitSync();\n    } catch (ProcessingException e) {\n        // Don't commit - record will be reprocessed on restart\n        handleError(e);\n    }\n}\n```\n\n#### Synchronous Commits\n\n```java\n// Commit all records from last poll()\nvoid commitSync()\n\n// Commit with custom timeout\nvoid commitSync(Duration timeout)\n\n// Commit specific offsets\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\n**Semantics**:\n- **Blocking**: Waits for broker response or timeout\n- **Atomic**: All offsets commit together\n- **Safe**: Retried internally for transient failures\n- **Exceptions**: `CommitFailedException` if group rebalances before commit completes\n\n**Timeout Behavior**:\n```java\n// Use default timeout from config (default.api.timeout.ms)\nconsumer.commitSync();\n\n// Custom timeout\nconsumer.commitSync(Duration.ofSeconds(30));\n```\n\n#### Asynchronous Commits\n\n```java\n// Fire-and-forget commit\nvoid commitAsync()\n\n// Commit with callback for success/failure\nvoid commitAsync(OffsetCommitCallback callback)\n\n// Commit specific offsets\nvoid commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets,\n                 OffsetCommitCallback callback)\n```\n\n**Semantics**:\n- **Non-blocking**: Returns immediately\n- **Ordered**: Multiple async commits maintain order\n- **Callback guarantee**: Previous callbacks invoked before `commitSync()` completes\n- **Exceptions**: Passed to callback or silently discarded if no callback\n\n**Callback Pattern**:\n```java\nconsumer.commitAsync((offsets, exception) -> {\n    if (exception Format\n\n**Critical Semantic**: The committed offset should be the offset of the **next record to read**, not the last record read.\n\n```java\n// WRONG - commits last offset read, will skip next record\nfor (ConsumerRecord<String, String> record : records) {\n    process(record);\n    consumer.commitSync(Collections.singletonMap(\n        record.topic(),\n        new OffsetAndMetadata(record.offset())  // WRONG!\n    ));\n}\n\n// CORRECT - use nextOffsets() or offset + 1\nconsumer.commitSync(records.nextOffsets());\n\n// ALSO CORRECT - explicit offset + 1 per partition\nfor (TopicPartition partition : records.partitions()) {\n    long lastOffset = records.records(partition).get(\n        records.records(partition).size() - 1\n    ).offset();\n partitions\nMap<TopicPartition, OffsetAndMetadata> committed(Set<TopicPartition> partitions)\n\n// With custom timeout\nMap<TopicPartition, OffsetAndMetadata> committed(\n    Set<TopicPartition> partitions,\n    Duration timeout)\n```\n\n**Return Value**:\n- Maps partition to `OffsetAndMetadata` (offset + metadata)\n- Returns `null` for partitions with no prior commit\n- Includes leader epoch for log truncation detection\n\n---\n\n## Flow Control and Position Management\n\n### Position Queries\n\n```java\n// Current position (next offset to fetch)\nlong position(TopicPartition partition)\nlong position(TopicPartition partition, Duration timeout)\n\n// Throws InvalidOffsetException if position undefined\n```\n\n**Position Semantics**:\n- Offset of the next record that will be returned by `poll()`\n- Advances each time `poll()` returns records\n- Uninitialized until first `poll()` or explicit `seek()`\n- May be ahead of committed offset if records haven't been committed Map<TopicPartition, Long> timestampsToSearch)\n\nMap<TopicPartition, OffsetAndTimestamp> offsetsForTimes(\n    Map<TopicPartition, Long> timestampsToSearch,\n    Duration timeout)\n```\n\n**Example**:\n```java\n// Seek to messages from 1 hour ago\nlong oneHourAgo = System.currentTimeMillis() - (60 * 60 * 1000);\nMap<TopicPartition, Long> timestampMap = changes**:\n   - New consumer joins group\n   - Consumer leaves group gracefully\n   - Consumer fails (detected via heartbeat timeout)\n\n2. **Topic/partition changes**:\n   - New partitions added to subscribed topics\n   - Subscribed topic deleted\n   - Pattern subscription matches new topics\n\n3. **Manual triggers**:\n   - `enforceRebalance()` called\n\n### Rebalance Protocol Variants\n\n#### Classic (Eager) Protocol\n```\n1. All consumers pause consumption\n2. Leader collects subscriptions and assigns partitions\n3. Revoke callback: ALL partitions revoked at once\n4. Assign callback: NEW partitions assigned\n5. All consumers resume\n```\n\n#### Cooperative Protocol\n```\n1. Leader computes optimal assignment\n2. Only revoke partitions that need reassignment\n3. Revoke callback: only CHANGED partitions\n4. Assign callback: only NEW partitions\n5. Other partitions continue consuming (no pause)\n```\n\n**Impact on Code**:\n- **Eager** requires expecting full assignment changes\n- **Cooperative** allows incremental state updates\n- Default changed from eager to cooperative in Kafka 3.0+\n\n### Rebalance Listener Callbacks\n\n```java\npublic class MyRebalanceListener implements ConsumerRebalanceListener {\n\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // ALWAYS called at start of rebalance (eager) or when partitions change (cooperative)\n        // Execute in the poll() thread\n\n        // Typical actions:\n        // 1. Commit current offsets\n        consumer.commitSync();\n\n        // 2. Flush buffered state\n        database.flush();\n\n        // 3. Close temporary resources\n        cache.clear();\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Called AFTER rebalance completes and partitions are assigned\n        // With eager: all partitions passed\n        // With cooperative: only NEW partitions passed\n\n        // Typical actions:\n        // 1. Load state for newly assigned partitions\n        for (TopicPartition partition : partitions) {\n            long offset = loadOffsetFromDatabase(partition);\n            consumer.seek(partition, offset);\n        }\n\n        // 2. Initialize per-partition state\n        // 3. Warm up caches\n    }\n\n    @Override\n    public void onPartitionsLost(Collection<TopicPartition> partitions) {\n        // Called when partitions are LOST (not revoked gracefully)\n        // Example: session timeout, fatal group error\n\n        // Key difference from onPartitionsRevoked:\n        // - These partitions may already be assigned to other consumers\n        // - Cannot commit offsets for lost partitions\n        // - Should NOT attempt cleanup that requires offsets\n\n        // Typical action: cleanup without committing\n        cache.invalidate(partitions);\n\n        // Default implementation: calls onPartitionsRevoked()\n        // Override if different handling needed\n    }\n}\n\nconsumer.subscribe(Arrays.asList(\"topic\"), new MyRebalanceListener());\n```\n\n### Rebalance Timing Considerations\n\n**Heartbeat-Based Detection**:\n- Consumer sends heartbeats every `heartbeat.interval.ms` (default: 3s)\n- If no heartbeats for `session.timeout.ms` (default: 10s), consumer presumed dead\n- Can only change assignment during `poll()`\n\n**Max Poll Interval Enforcement**:\n```\nif (time_since_last_poll > max.poll.interval.ms) {\n    consumer proactively leaves group\n    next poll() throws CommitFailedException\n}\n```\n\n**Configuration Tuning**:\n```properties\n# Faster failure detection (trade-off: more network overhead)\nheartbeat.interval.ms=1000        # Default: 3000\nsession.timeout.ms=6000            # Default: 10000\n\n# Longer processing time allowed\nmax.poll.interval.ms=600000        # Default: 300000 (5 minutes)\n```\n\n---\n\n## Error Handling\n\n### Exception Hierarchy\n\n```\nKafkaException (root)\n\u251c\u2500\u2500 CommitFailedException\n\u2502   \u2514\u2500\u2500 Unrecoverable: group rebalanced before commit\n\u2502\n\u251c\u2500\u2500 OffsetOutOfRangeException\n\u2502   \u2514\u2500\u2500 Position outside valid range: seek failed\n\u2502\n\u251c\u2500\u2500 InvalidOffsetException\n\u2502   \u2514\u2500\u2500 No offset defined for partition\n\u2502\n\u251c\u2500\u2500 LogTruncationException\n\u2502   \u2514\u2500\u2500 Log was truncated: leader epoch mismatch\n\u2502\n\u2514\u2500\u2500 WakeupException\n    \u2514\u2500\u2500 Consumer interrupted by wakeup() call\n```\n\n### Common Exceptions and Recovery\n\n#### CommitFailedException\n\n```\nOccurs: During commitSync() when group has rebalanced\nCause: Processing took longer than max.poll.interval.ms\n        or consumer failed and rejoined with new assignment\n\nRecovery:\n- Catch exception\n- Not retriable: partitions already reassigned\n- Prevent by: call poll() frequently enough (see max.poll.interval.ms)\n```\n\n**Example**:\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Partition likely reassigned\n    log.error(\"Commit failed - group rebalanced: \" + e.getMessage());\n\n    // Get new assignment and continue\n    Set<TopicPartition> assignment = consumer.assignment();\n    if (assignment.isEmpty()) {\n        log.error(\"No partitions assigned - waiting for rebalance\");\n    }\n    // Next poll() will handle reassignment\n}\n```\n\n#### OffsetOutOfRangeException\n\n```\nOccurs: When seeking to or polling from invalid offset\nCause: Offset < log start or > log end\n       Log compaction removed messages\n       Requested offset no longer exists\n\nRecovery:\n- Seek to valid offset: seekToBeginning() or seekToEnd()\n- Or set auto.offset.reset=earliest/latest\n- Or commitSync() to reset to last committed offset\n```\n\n**Example**:\n```java\ntry {\n    consumer.poll(Duration.ofSeconds(1));\n} catch (OffsetOutOfRangeException e) {\n    log.warn(\"Offset out of range, seeking to beginning: \" + e.getMessage());\n    consumer.seekToBeginning(consumer.assignment());\n}\n```\n\n#### InvalidOffsetException\n\n```\nOccurs: When calling position() with undefined offset\nCause: Partition assigned but no position set yet\n       No committed offset and auto.offset.reset=none\n\nRecovery:\n- Seek explicitly\n- Or set auto.offset.reset=earliest/latest\n- Or call poll() to initialize position\n```\n\n#### WakeupException\n\n```\nOccurs: When wakeup() called during blocking operation\nCause: External thread called consumer.wakeup()\nAction: Propagates to calling thread's poll() or other blocking operation\n\nRecovery:\n- Catch and exit loop\n- Call close() to cleanup\n- Signal shutdown to other threads\n```\n\n**Example** (See [Multi-threaded Processing](#multi-threaded-processing) for full example):\n```java\ntry {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofSeconds(1));\n    // process records\n} catch (WakeupException e) {\n    // Shutdown gracefully\n    log.info(\"Consumer woken up, shutting down\");\n} finally {\n    consumer.close();\n}\n```\n\n#### LogTruncationException\n\n```\nOccurs: When reading transactional messages with read_committed\nCause: Producer's transaction was rolled back\n       Log was truncated on partition leader\n       Committed offset is no longer valid\n\nRecovery:\n- Seek to valid offset\n- Reset position to earliest/latest\n- Investigate broker/producer for issues\n```\n\n---\n\n## Multi-threaded Processing\n\n### Single Consumer Thread Pattern (Recommended)\n\nUse one thread for consumption, another pool for processing:\n\n```java\nclass ConsumerThread extends Thread {\n    private KafkaConsumer<String, String> consumer;\n    private BlockingQueue<ConsumerRecord<String, String>> queue;\n    private volatile boolean running = true;\n\n    public void run() {\n        try {\n            consumer.subscribe(Arrays.asList(\"topic\"));\n\n            while (running) {\n                ConsumerRecords<String, String> records =\n                    consumer.poll(Duration.ofMillis(100));\n\n                for (ConsumerRecord<String, String> record : records) {\n                    queue.put(record);  // Non-blocking add to queue\n                }\n            }\n        } catch (WakeupException e) {\n            // Clean shutdown\n        } finally {\n            consumer.close();\n        }\n    }\n\n    public void shutdown() {\n        running = false;\n        consumer.wakeup();  // ONLY thread-safe method!\n    }\n}\n\n// Start consumer thread\nConsumerThread consumerThread = new ConsumerThread(consumer, queue);\nconsumerThread.start();\n\n// Process records in worker thread pool\nExecutorService executor = Executors.newFixedThreadPool(10);\nwhile (true) {\n    ConsumerRecord<String, String> record = queue.take();\n    executor.submit(() -> processRecord(record));\n}\n\n// Shutdown\nexecutor.shutdown();\nconsumerThread.shutdown();\nexecutor.awaitTermination(60, TimeUnit.SECONDS);\n```\n\n**Key Points**:\n- Consumer polling happens in dedicated thread\n- Processing happens in separate worker threads\n- Only `wakeup()` is thread-safe\n- Queue ensures no concurrent access to records\n\n### Manual Offset Commit with Thread Pool\n\nFor exactly-once semantics with worker threads:\n\n```java\nclass ProcessedRecord {\n    ConsumerRecord<String, String> record;\n    Exception error;\n}\n\nMap<TopicPartition, Long> processedOffsets = Collections.synchronizedMap(\n    new HashMap<>()\n);\n\n// Worker thread processing\nexecutor.submit(() -> {\n    try {\n        String result = processRecord(record);\n        processedOffsets.put(\n            new TopicPartition(record.topic(), record.partition()),\n            record.offset() + 1  // Next offset to consume\n offsets\nwhile (running) {\n    ConsumerRecords<String, String> records =\n        consumer.poll(Duration.ofMillis(100));\n\n    for (ConsumerRecord<String, String> record : records) {\n        queue.put(record);\n    }\n\n    // Commit offsets processed by workers\n    if (!processedOffsets.isEmpty()) {\n        Map<TopicPartition, OffsetAndMetadata> toCommit =\n            processedOffsets.entrySet().stream()\n            .collect(Collectors.toMap(\n                Map.Entry::getKey,\n                e -> new OffsetAndMetadata(e.getValue())\n            ));\n        consumer.commitAsync(toCommit, (offsets, exception) -> {\n            if (exception == null) {\n                processedOffsets.clear();  // Remove committed offsets\n            }\n        });\n    }\n}\n```\n\n### Using Pause/Resume for Backpressure\n\n```java\nclass ConsumerThread extends Thread {\n    private BlockingQueue<ConsumerRecord<String, String>> queue;\n    private static final int ## Overview

The `KafkaConsumer` is a Java client for consuming records from a Kafka cluster. It provides a flexible consumption model supporting:

- **Group-managed consumption**: Automatic partition assignment across consumer group members
- **Manual assignment**: Direct partition assignment with no group coordination
- **Dynamic subscriptions**: Pattern-based topic subscriptions
- **Flexible offset management**: Automatic or manual commit strategies
- **Metadata queries**: Partition discovery and offset inspection
- **Flow control**: Pause/resume consumption on specific partitions

### Key Principles

1. **Not Thread-Safe**: The consumer is designed for single-threaded use. All methods except `wakeup()` require exclusive access.
2. **Event-Driven Model**: Rebalancing and callbacks occur during `poll()` calls only.
3. **Offset-Based Positioning**: Consumption is controlled via offsets within partitions.
4. **Blocking Semantics**: Various operations may block (networking, broker communication, rebalancing).
