# PostgreSQL Architecture - Key Insights

## Query Execution Pipeline

PostgreSQL implements a **5-stage pipeline** with clear separation of concerns:

1. **Parser** (gram.y + scan.l): Syntactic analysis → RawStmt (no DB access)
2. **Analyzer** (analyze.c): Name/type resolution → Query
3. **Rewriter** (rewriteHandler.c): Rule expansion, RLS → Query list
4. **Optimizer** (planner.c): Two-phase cost-based optimization → PlannedStmt
5. **Executor** (execProcnode.c): Volcano-style iterators → tuples

## Key Architecture Decisions

### Two-Phase Optimization
- **Phase 1** (allpaths.c): Generate all feasible paths with cost estimates
- **Phase 2** (createplan.c): Convert cheapest path → executable Plan tree
- Separation allows cheap path exploration without plan generation overhead

### Volcano-Style Execution
- Each executor node implements pull-based `ExecProcNode()` interface
- Function pointer dispatch in PlanState enables:
  - Transparent instrumentation wrapping
  - Lazy stack depth checking
  - Future JIT optimization
- Switch dispatch in ExecInitNode() creates PlanState tree mirroring Plan tree

### Expression Evaluation
- Compiled to ExprState during node initialization
- Rapid interpretation at execution time
- Can be replaced with JIT code
