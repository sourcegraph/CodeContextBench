## Description

You are reviewing a recently merged pull request that adds scope configuration for the feature opt-in system. This PR allows features to be restricted to specific scopes (org, team, user) instead of being available at all levels. The changes span the service layer, configuration module, type definitions, and the tRPC router. However, several defects were introduced during the merge — both functional bugs and compliance violations.

Your task is to **find the defects and produce a structured review report with proposed fixes**.

## Context

The feature opt-in scope configuration spans three key TypeScript source files:

1. **`packages/features/feature-opt-in/services/FeatureOptInService.ts`** — Service class: resolves feature states across teams with precedence rules, lists features for user/team scopes, and validates scope before setting states.
2. **`packages/features/feature-opt-in/config.ts`** — Configuration module: defines the opt-in feature allowlist with per-feature scope arrays, provides helper functions for scope filtering and validation.
3. **`packages/trpc/server/routers/viewer/featureOptIn/_router.ts`** — tRPC router: exposes endpoints for listing and setting feature states at user/team/org levels, validates slugs against the allowlist before mutations.

## Task

## Summary

### 1. **CRITICAL - Missing null check in isFeatureAllowedForScope** (config.ts:76)
The function assumes `config.scope` is always defined, but the interface marks it as optional. This will throw a TypeError when a feature has no scope field. According to the documentation, features without a scope field should be allowed for all scopes.

### 2. **CRITICAL - Hardcoded policy instead of using config** (FeatureOptInService.ts:161)
The `resolveFeatureState` method hardcodes the policy to "permissive" instead of retrieving it from the feature configuration. This causes all features to be evaluated with the wrong policy, breaking the compliance pattern where each feature defines its own policy.

### 3. **MEDIUM - Comment contradicts implementation** (FeatureOptInService.ts:187)
The JSDoc comment claims the method returns "globally enabled" features, but the actual filter returns "globally disabled" features (opt-in features). While the logic is correct for an opt-in system, the misleading comment violates documentation compliance.

### 4. **HIGH - Missing slug validation in setUserState** (_router.ts:88)
The `setUserState` endpoint doesn't validate the input slug against the opt-in allowlist, unlike `setTeamState` and `setOrganizationState`. This creates inconsistent validation patterns across mutation endpoints.

All defects have been documented with unified diff patches in `/workspace/review.json` ready for application by the evaluation system.
