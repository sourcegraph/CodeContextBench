Failure Modes and Design Tradeoffs\n\n### 4.1 Common Failure Modes\n\n#### 4.1.1 Resource Version Too Old\n**Symptom:** Informer suddenly fails with \"resource version X is too old\"\n\n**Root Cause:**\n- Reflector issued WATCH at resourceVersion RV=100\n- API server's watch cache evicts older versions (typically 5-10 min)\n- New WATCH request with RV=100 rejected\n\n**Current Mitigation (reflector.go:191-204):**\n- Watch bookmarks: API server sends BOOKMARK events with current RV\n- Reflector extracts RV from bookmarks, updates `lastSyncResourceVersion`\n- On error: use bookmark RV instead of last confirmed RV\n- Re-list to get up-to-date snapshot\n\n**Tradeoff:**\n- Bookmarks reduce re-list frequency but increase watch stream size (one extra event per bookmark period)\n- Bookmark period is configurable per API server\n\n#### 4.1.2 Informer Handler Panic\n**Symptom:** Informer stops notifying handlers; no error message\n\n**Root Cause:**\n- User-provided handler panics during OnAdd/OnUpdate/OnDelete\n- Panic is caught by `util/runtime.HandleCrash()` and logged\n- Handler goroutine exits; no further notifications\n\n**Mitigation (tools/cache/shared_informer.go):**\n- Each handler runs in a dedicated goroutine\n- Crashes are logged but don't affect other handlers or the informer itself\n- However, the offending handler is permanently disabled\n\n**Best Practice:**\nHandlers must be careful to avoid panics; use defer/recover if necessary.\n\n#### 4.1.3 Work Queue Starvation (Slow Handler)\n**Symptom:** Controller lags behind API changes; queue keeps growing\n\n**Root Cause:**\n- Business logic is slow (e.g., issuing multiple API calls per item)\n- Workers spend most time in `syncToStdout()`, not processing queue\n- New items accumulate faster than they're processed\n\n**Mitigation:**\n- Increase worker count (more concurrent processing)\n- Optimize business logic (batch operations, cache reads from lister)\n- Use rate limiting to backoff failures, not just generic slowness\n\n**Design Decision:**\nWorkqueue does NOT prioritize items; all items are FIFO. If certain items are critical, that logic must be in the controller.\n\n#### 4.1.4 Cache Inconsistency with API Server\n**Symptom:** Controller reads stale pod via lister; pod was deleted 30 seconds ago\n\n**Root Cause:**\n- Watch delivery lag: Events take time to propagate from API server\n- Network delay, watch stream buffering, handler processing delay\n- Informer is **eventually consistent**, not strongly consistent\n\n**Mitigation:**\n- Controller can call `clientset.CoreV1().Pods(ns).Get(name)` to verify state before mutating\n- Use optimistic locking (check ResourceVersion before patching)\n- Accept eventual consistency in the design (most controllers do)\n\n**Design Principle:**\nInformers are for **observation and triggering**; critical decisions should be verified via direct API call.\n\n#### 4.1.5 API Server Restart / Connectivity Loss\n**Symptom:** Informer stuck; no updates for 5 minutes\n\n**Root Cause:**\n- API server restarts; watch connection drops\n- Network partition; TCP RST or timeout\n- Reflector detects error, backoffs, and retries\n\n**Recovery (reflector.go:~280-350):**\n- Error handler logs failure\n- Exponential backoff: 1s \u2192 2s \u2192 4s \u2192 ... \u2192 capped at several minutes\n- Retry LIST at last known resourceVersion\n- Resume WATCH\n\n**Concern:**\nDuring extended outage, the controller falls behind. Upon recovery, the Reflector may need to re-list (if RV is too old), which is expensive.\n\n### 4.2 Performance Limits and Tradeoffs\n\n#### 4.2.1 Memory Consumption\n**Observation:** Informer caches all objects; large clusters consume significant memory.\n\n**Scale Example:**\n- 10,000 pods \u00d7 ~2 KB per pod = ~20 MB\n- 100,000 pods \u00d7 ~2 KB = ~200 MB\n- Add indexes (by namespace, by label selectors) \u2192 ~300 MB+\n\n**Mitigations:**\n1. **Namespace-scoped informers:** `informers.WithNamespace(\"default\")` only watches one namespace\n2. **Label/field selectors:** `WithTweakListOptions(func(opts *metav1.ListOptions) { opts.LabelSelector = \"app=myapp\" })` pre-filters\n3. **Resync period tuning:** Longer resync (60s instead of 0s) reduces periodic re-list overhead\n4. **Object transformers:** `WithTransform(cache.TransformFunc)` strip unnecessary fields before caching\n\n#### 4.2.2 API Server Load\n**Observation:** LIST requests are expensive; watch streams consume connections.\n\n**Numbers:**\n- Initial LIST: serial request, full response body\n- Subsequent WATCH: streaming, incremental (only deltas)\n- Re-list on error: full response again (often avoided with bookmarks)\n\n**Tradeoff (reflector.go:127-147):**\n- Pagination (`WatchListPageSize`): LIST returns chunks (e.g., 500 items per page). **Pro:** Reduce memory spike. **Con:** Multiple serial requests, slower startup.\n- Watch List mode: Replaces LIST + WATCH with a single streaming LIST. **Pro:** Fewer connections, better resource utilization. **Con:** Requires server-side support (KEP-3157).\n\n#### 4.2.3 Reconciliation Latency\n**Observation:** Lag from API change to handler invocation.\n\n**Sources of Latency:**\n- Network RTT (watch stream delivery): ~10-100ms\n- Server-side queuing (if API server watch cache is overloaded): 100ms - seconds\n- Informer event handler processing: ~1-10ms\n- Work queue item processing (awaiting worker availability): 1-100ms\n- Business logic execution: 10ms - seconds\n\n**Total Typical Case:** 50-200ms.
