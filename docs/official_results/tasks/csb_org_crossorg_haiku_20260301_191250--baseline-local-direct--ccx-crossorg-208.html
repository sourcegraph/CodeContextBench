## Summary of Findings

I've created `/workspace/answer.json` with detailed findings on how each repository handles gRPC error status codes:

### **Key Insights:**

1. **kubernetes/kubernetes** - Uses inline `status.Error(codes.*, message)` creation in service handlers with custom HTTP translation logic (e.g., `WriteError()` function in streaming/errors.go that maps codes.NotFound → 404, codes.ResourceExhausted → 429)

2. **etcd-io/etcd** - Centralizes all ~50 pre-defined gRPC errors in `api/v3rpc/rpctypes/error.go`, then maps internal errors to these gRPC errors via `toGRPCErrorMap` in `server/etcdserver/api/v3rpc/util.go`

3. **grafana/grafana** - Creates gRPC status errors inline during authentication/authorization phases (especially in interceptors and authenticator files) and relies on the standard grpc-gateway `HTTPStatusFromCode()` mapping for HTTP translation

### **Core Files Found:**
- **etcd**: api/v3rpc/rpctypes/error.go (centralized definitions), server/etcdserver/api/v3rpc/util.go (translation logic)
- **Kubernetes**: staging/src/k8s.io/kubelet/pkg/cri/streaming/errors.go, test/e2e/storage CSI mock files
- **Grafana**: pkg/storage/unified/resource/errors.go (HTTP translation), pkg/services/grpcserver/interceptors/auth.go (auth errors), pkg/services/authz files

All three repos ultimately use the shared `github.com/grpc-ecosystem/grpc-gateway/v2/runtime` library's `HTTPStatusFromCode()` function as the standard bridge between gRPC codes and HTTP status codes.
