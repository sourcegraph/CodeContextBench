Users who have registered multiple U2F hardware tokens with Teleport report that during login, only one of their registered devices is recognized. The authentication flow presents a challenge for only a single device, preventing users from selecting which hardware token to use.

Specifically:

1. **Single-device lock-in**: After registering multiple U2F tokens via the MFA management flow, attempting to log in only issues a challenge for one device. The system should present challenges for all registered tokens so the user can tap whichever device is available.

### 🔍 **Bug Investigation Results**

**Root Cause**: The `U2FAuthenticateChallenge` struct in `lib/auth/auth.go` uses an **embedded pointer** that gets flattened during JSON marshaling:

```go
type U2FAuthenticateChallenge struct {
    *u2f.AuthenticateChallenge  // ← Flattened to top level in JSON
    Challenges []u2f.AuthenticateChallenge `json:"challenges"`
}
```

**The Problem**:
1. When 2+ U2F devices are registered, the server creates challenges for all devices
2. During JSON marshaling, the embedded pointer's fields (Challenge, KeyHandle, AppID) are flattened to the top level
3. Only the **first device's challenge** appears at the top level
4. **Old clients (pre-6.0)** only read top-level fields and miss other devices
5. Even **new clients** might fail if they don't properly unmarshal the `Challenges` array

The regression test includes **4 comprehensive test scenarios**:

1. **JSON Marshaling Preservation** - Verifies all challenges survive JSON serialization
2. **Embedded Pointer Flattening** - Demonstrates Go's JSON marshaling behavior
3. **Challenge Field Preservation** - Confirms both embedded and array fields work correctly
4. **Backwards Compatibility** - Tests old and new client behavior

## Bug Description

### Problem Statement
Users who register multiple U2F hardware tokens with Teleport report that during login, only one of their registered devices is recognized. The authentication flow presents a challenge for only a single device, preventing users from selecting which hardware token to use.

### Root Cause Analysis
The issue stems from the `U2FAuthenticateChallenge` struct in `lib/auth/auth.go`:

```go
type U2FAuthenticateChallenge struct {
    *u2f.AuthenticateChallenge  // Embedded, flattened to top-level JSON
    Challenges []u2f.AuthenticateChallenge `json:"challenges"`
}
```

**The JSON marshaling behavior:**
1. Server correctly populates both challenges in the `Challenges` field
2. Server sets `AuthenticateChallenge` to the first device (for backwards compatibility)
3. During JSON marshaling, embedded fields get **flattened** to the top level
4. Result: Top-level fields contain **only the FIRST device's data**
5. Old clients (pre-6.0) see only the top-level fields and miss the other devices
6. Even newer clients might have issues if they don't properly read the `Challenges` array

### Multi-Device Challenge Generation Flow
```
Server:
1. User has 2 registered U2F devices
2. `U2FSignRequest()` loops through all devices
3. Creates unique challenge for each device
4. Populates both challenges in response.Challenges array
5. Sets first device challenge in embedded field for backwards compat
6. Returns response

JSON Marshaling:
{
  "version": "U2F_V2",           // From embedded first device
  "challenge": "device1-...",    // From embedded first device
  "keyHandle": "device1-...",    // From embedded first device
  "appId": "...",                // From embedded first device
  "challenges": [                // Array with ALL devices
    { device 1 challenge },
    { device 2 challenge }
  ]
}

Old Client Behavior (pre-6.0):
- Only reads top-level fields
- Only sees device 1
- Presents challenge for device 1 only
- User cannot tap device 2

New Client Behavior (6.0+):
- Should read challenges array
- Should see both devices
- Should present both as options
- But might fail if Challenges field is lost or not properly populated
```

## Impact of the Bug

**User-Facing Issues:**
1. Only one device is presented during login authentication
2. Users cannot switch to alternate devices if the primary one is unavailable
3. Users with multiple devices experience inconsistent behavior
4. Potential account lockout if only the unavailable device is presented

**Version Compatibility Issues:**
1. Older clients (pre-6.0) connecting to newer servers might not see all devices
2. Newer clients connecting to older servers might not handle the single-challenge format
3. Version mismatch could cause authentication failures
