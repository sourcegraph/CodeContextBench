# Architecture Document: Istio Pilot Discovery Service

**Repository:** istio/istio
**Output:** Write your document to `/workspace/documentation.md`

## Task

Produce an architecture document for **Istio Pilot's Discovery Service** subsystem. The document must explain how Pilot serves xDS (Envoy Discovery Service) configuration to proxies, translates Istio's high-level configuration into Envoy-specific resources, integrates with service registries, and supports multi-cluster deployments. Do not simply list APIs — explain the **design**, **data flow**, and **extension points**.

## Scope

Your document must cover these four components and how they work together:

### 1. DiscoveryServer — The xDS gRPC Server
The central component implementing Pilot's gRPC interface for Envoy's xDS APIs. Explain:
- How it implements Envoy's xDS v3 protocol (ADS, incremental/state-of-the-world modes)
- The role of `adsClients` in tracking active proxy connections
- How configuration pushes are coordinated: debouncing, rate limiting, concurrent push limits
- Cache management for xDS resources (CDS, LDS, RDS, EDS)
- The lifecycle: initialization, handling new connections, shutdown

### 2. ADS and the Connection Model
The Aggregated Discovery Service stream handler. Explain:
- How `Stream()` establishes a bidirectional gRPC stream with an Envoy proxy
- The `Connection` type: wrapping the stream with proxy metadata and state
- Request processing: how `processRequest()` determines what configuration to send
- Push coordination: how `pushConnection()` transmits updates to a single proxy
- The `StartPush()` broadcast mechanism for fleet-wide updates

### 3. PushContext — Configuration Snapshot and Translation
The immutable configuration snapshot used during a push. Explain:
- What `PushContext` holds: services, destination rules, virtual services, sidecars, gateways
- How `InitContext()` rebuilds the push context when Istio config changes (CRD watches)
- Config translation: how Pilot converts Istio's `VirtualService` and `DestinationRule` into Envoy's `RouteConfiguration` and `Cluster` resources
- The relationship between push triggers (config changes, service registry updates) and push context regeneration
- Metrics and error tracking during push generation

### 4. Service Registry and Multi-Cluster Support
The service discovery layer feeding the DiscoveryServer. Explain:
- How `ServiceRegistry` (Kubernetes controller) watches K8s Services, Endpoints, and Pods
- The aggregate controller pattern for multi-registry support (K8s services + `ServiceEntry` resources)
- Multi-cluster support: how Pilot aggregates services from multiple clusters
- How endpoint updates (pod scale-up, health changes) trigger EDS pushes
- The relationship between the service registry and xDS: when does a service change trigger CDS vs. EDS?

## Document Requirements

1. **Component Responsibilities** — what each component owns
2. **Data Flow** — the path from Istio config change → DiscoveryServer → xDS push to Envoy proxies
3. **Extension Points** — where users extend Istio (custom config translation, external service registries, telemetry plugins)
4. **Error Handling** — how errors at each stage (config validation, push failures, proxy disconnects) are handled
5. **relevant source files** — reference the actual source files in the istio/istio repository (e.g., `pilot/pkg/xds/discovery.go`, `pilot/pkg/model/push_context.go`)

## Anti-Requirements

- Do NOT fabricate type names or file paths that don't exist in the repository
- Do NOT cover Istio components outside the Pilot discovery path (e.g., Citadel, Galley, sidecar injection)

The request enters `pushChannel` for debouncing in `handleUpdates()` (`pilot/pkg/xds/discovery.go:326`), which:\n   - Waits for a quiet period (`DebounceAfter`) without new changes\n   - Coalesces multiple changes into a single push\n   - Enforces maximum wait time (`debounceMax`)\n\n3. **`Push()`** (`pilot/pkg/xds/discovery.go:258`) is called with the debounced request:\n   - If full push, `initPushContext()` recomputes the entire mesh state\n   - Updates `CommittedUpdates` counter\n   - Enqueues the request in `pushQueue` for distribution to proxies\n\n#### 3. **Cache Management**\n\nThe `Cache` field implements the `model.XdsCache` interface, storing computed resources keyed by:\n- Proxy metadata (region, cluster, node type)\n- Resource type (Cluster, Listener, Route, etc.)\n- Configuration version\n\n**Cache Invalidation** (`pilot/pkg/xds/discovery.go:246`):\n- `dropCacheForRequest()` clears the cache when configuration changes\n- If a push is `Full`, the entire cache is cleared\n- If incremental, only affected resource types are invalidated\n\n---\n\n## 2. ADS and the Connection Model\n\n### Overview\n\nThe Aggregated Discovery Service (ADS) in `ads.go` implements the bidirectional gRPC streaming protocol that connects Pilot to Envoy proxies. A single `Stream()` handler manages the lifecycle of each proxy connection and coordinates configuration push/pull interactions.\n\n### Connection Lifecycle\n\n#### The `Connection` Type (`pilot/pkg/xds/ads.go:71-88`)\n\nEach connected proxy is represented by a `Connection` struct:\n\n```go\ntype Connection struct {\n    xds.Connection           // Embedded base with stream and peer address\n    node *core.Node          // Original Envoy node metadata\n    proxy *model.Proxy       // Parsed proxy metadata (cluster, namespace, etc.)\n    deltaStream ...          // For Delta XDS (incremental)\n    s *DiscoveryServer       // Reference to parent server\n    ids []string             // Identity certificates from mTLS\n}\n```\n\nKey responsibilities:\n- **Wraps the gRPC stream** \u2014 Holds the `DiscoveryStream` for bidirectional communication\n- **Stores proxy metadata** \u2014 Extracts and caches information from Envoy's node metadata (cluster ID, pod name, namespace, etc.)\n- **Tracks state** \u2014 Maintains resource versions, watched resources, and last push time\n\n#### Establishing a Connection: `Stream()` (`pilot/pkg/xds/ads.go:188-235`)\n\n1. **Server Readiness Check** \u2014 Returns `Unavailable` if caches not synced; prevents connecting proxies from getting empty config\n2. **Authentication** (`ads.go:220-228`) \u2014 Validates mTLS/authentication credentials\n3. **Rate Limiting** (`ads.go:215`) \u2014 Enforces per-request rate limit using token bucket\n4. **Context Initialization** (`ads.go:231`):\n   ```go\n   s.globalPushContext().InitContext(s.Env, nil, nil)\n   ```\n   This ensures the global push context is populated. Happens once; subsequent calls return immediately.\n5. **Connection Creation** (`ads.go:232-234`):\n   - Creates a new `Connection` struct wrapping the stream\n   - Sets parent reference `con.s = s`\n   - Delegates to `xds.Stream()` for stream handling\n6. **Proxy Initialization** in `initConnection()` (`ads.go:240`):\n   - Parses node metadata into a `Proxy` object\n   - Registers the connection in `adsClients` map\n   - Initializes proxy-specific state (sidecar scope, watched resources)\n\n### Request Processing: `processRequest()` (`ads.go:139-181`)\n\nWhen an Envoy proxy sends a `DiscoveryRequest`:\n\n1. **Deduplication Check** \u2014 `xds.ShouldRespond()` determines if a response is needed based on:\n   - Previous version (already sent to this proxy)\n   - Nonce (preventing duplicate ACKs)\n\n2. **Configuration Preparation** \u2014 Constructs a `PushRequest` with:\n   - The proxy's last known `PushContext` (to avoid re-initializing)\n   - `Full: true` for immediate response (not waiting for debouncing)\n   - Last push time (used by cache for staleness detection)\n\n3. **Sidecar Scope Validation** (`ads.go:173-179`) \u2014 If the proxy's sidecar scope version doesn't match the push context version, `computeProxyState()` recomputes it\n\n4. **Response Generation** \u2014 Calls `pushXds()` to generate and send the response via the appropriate generator\n\n### Push Coordination: `pushConnection()` (`ads.go`, line not shown but referenced)\n\nWhen Pilot needs to push configuration to a specific connection:\n\n1. Gets the `Generator` for the proxy type (sidecar, gateway, waypoint)\n2. Calls `Generator.Generate()` to produce xDS resources for the proxy\n3. Sends a `DiscoveryResponse` via the stream\n4. Updates metrics and error tracking\n\n### Delta XDS: `DeltaAggregatedResources()` (`ads.go:467`)\n\nFor incremental updates, Envoy can use Delta XDS to receive only changed resources:\n\n1. **Incremental Resources** \u2014 Only resources that changed are sent\n2. **Deleted Resources** \u2014 Resource names that should be removed are sent\n3. **State Tracking** \u2014 The client tracks subscribed resources and version numbers\n\nThis reduces bandwidth for large clusters with frequent endpoint updates.\n\n---\n\n## 3. PushContext \u2014 Configuration Snapshot and Translation\n\n### Purpose\n\nA `PushContext` is an **immutable snapshot of the entire mesh configuration** used during a single push. It's created when configuration changes and shared across all proxies during that push, ensuring consistency. It also serves as the translation engine that converts Istio's high-level configuration into Envoy-specific resources.\n\nLocated in: `pilot/pkg/model/push_context.go:207-280`\n\n### Architecture\n\n#### Core Indexes\n\nA `PushContext` maintains several in-memory indexes for fast lookups:\n\n1. **`ServiceIndex serviceIndex`** (`pilot/pkg/model/push_context.go:65-91`) \u2014 Indexes all services by visibility:\n   - `public` \u2014 Services with `exportTo: \"*\"` (visible across the mesh)\n   - `privateByNamespace` \u2014 Services with `exportTo: \".\"` (visible only within namespace)\n   - `exportedToNamespace` \u2014 Services exported to specific namespaces\n   - `HostnameAndNamespace` \u2014 Services keyed by hostname and namespace\n   - `instancesByPort` \u2014 Cached endpoint lookups by port\n\n   **Usage**: When generating CDS (Cluster Discovery Service), Pilot looks up services this index determines which proxies can see which services.\n\n2. **`virtualServiceIndex virtualServiceIndex`** (`pilot/pkg/model/push_context.go:100-116`) \u2014 Indexes VirtualServices by visibility:\n   - `publicByGateway` \u2014 Public VirtualServices by gateway\n   - `privateByNamespaceAndGateway` \u2014 Namespace-local VirtualServices\n   - `delegates` \u2014 Delegates in VirtualService hierarchies\n   - `referencedDestinations` \u2014 Services referenced by VirtualServices\n\n   **Usage**: When generating RDS (Route Discovery Service), Pilot looks up VirtualServices to build HTTP routes.\n\n3. **`destinationRuleIndex destinationRuleIndex`** (`pilot/pkg/model/push_context.go:132-146`) \u2014 Indexes DestinationRules:\n   - `namespaceLocal` \u2014 DestinationRules per namespace\n   - `exportedByNamespace` \u2014 Exported DestinationRules\n   - `specificDestRules` and `wildcardDestRules` \u2014 Organized by target host\n\n   **Usage**: When generating clusters, DestinationRules are consulted for load balancing policies, connection pools, and traffic policies.\n\n4. **`sidecarIndex sidecarIndex`** (`pilot/pkg/model/push_context.go:148-180`) \u2014 Stores Sidecar configurations:\n   - `sidecarsByNamespace` \u2014 User-defined sidecars per namespace\n   - `meshRootSidecarConfig` \u2014 Root namespace sidecar (defaults for whole mesh)\n   - `defaultSidecarsByNamespace` \u2014 Computed defaults if no sidecar defined\n\n   **Usage**: Sidecars define which services a proxy can see. When computing routes and clusters for a proxy, the sidecar scope limits what's included.\n\n5. **`gatewayIndex gatewayIndex`** (`pilot/pkg/model/push_context.go:182-195`) \u2014 Indexes Gateway resources by namespace and globally.\n\n6. **`virtualServiceIndex.delegates`** \u2014 Maps root VirtualServices to their delegates, used for hierarchical route computation.\n\n#### Metrics and Status Tracking\n\n- **`ProxyStatus map[string]map[string]ProxyPushStatus`** (`pilot/pkg/model/push_context.go:211`) \u2014 Tracks push errors per proxy per error code, used for debugging and metrics\n- **`serviceAccounts map[serviceAccountKey][]string`** \u2014 Maps services to their service accounts for RBAC\n\n#### Configuration Objects\n\nA `PushContext` also holds:\n\n- **`Mesh *meshconfig.MeshConfig`** \u2014 Global mesh configuration\n- **`JwtKeyResolver *model.JwksResolver`** \u2014 JWT public key resolver for authentication policies\n- **`AuthnPolicies *AuthenticationPolicies`** \u2014 Authentication policies in the mesh\n- **`AuthzPolicies *AuthorizationPolicies`** \u2014 Authorization (RBAC) policies\n- **`Telemetry *Telemetries`** \u2014 Telemetry policies\n- **`ProxyConfigs *ProxyConfigs`** \u2014 Per-proxy configuration overrides\n- **`Networks *meshconfig.MeshNetworks`** \u2014 Multi-cluster network topology\n\n### PushContext Lifecycle\n\n#### Creation: `NewPushContext()` (`pilot/pkg/model/push_context.go:735`)\n\nCreates a new, empty `PushContext` with initialized but empty indexes.\n\n#### Initialization: `InitContext()` (`pilot/pkg/model/push_context.go`, method)\n\nCalled when configuration changes to populate the indexes:\n\n1. **Rebuilds all indexes** from the current `ConfigStore`:\n   - Scans all Services and builds `ServiceIndex`\n   - Scans all VirtualServices and builds `virtualServiceIndex`\n   - Scans all DestinationRules and indexes them\n   - Indexes Sidecars, Gateways, EnvoyFilters, etc.\n\n2. **Applies visibility rules** based on `exportTo`:\n   - Services/routing rules with `exportTo: \"*\"` go into `public`\n   - `exportTo: \".\"` goes into `privateByNamespace`\n   - Explicit namespace exports are indexed separately\n\n3. **Computes derived state**:\n   - Merges DestinationRules for the same host\n   - Validates configurations and logs errors\n   - Caches endpoint lookups for frequently accessed services\n\n4. **Thread-safety**: Protected by `initializeMutex`; multiple concurrent calls return immediately after first initialization\n\n**Performance Note**: `InitContext()` is expensive (scans all CRDs, indexes them) and happens only when configuration changes (debounced). This is why it's debounced; calling it on every request would be prohibitively expensive.\n\n### Configuration Translation: From Istio to Envoy\n\nThe `PushContext` serves as the input to resource generators that translate Istio's configuration into Envoy resources. The translation happens in `pilot/pkg/networking/core/` and uses the indexes in `PushContext`.\n\n#### Example: VirtualService \u2192 Route\n\n1. **Input**: A `VirtualService` specifying:\n   ```yaml\n   apiVersion: networking.istio.io/v1alpha3\n   kind: VirtualService\n   metadata:\n     name: bookinfo\n   spec:\n     hosts:\n     - bookinfo.example.com\n     http:\n     - match:\n       - uri:\n           prefix: /reviews\n       route:\n       - destination:\n           host: reviews\n           port:\n             number: 8080\n           subset: v1\n   ```\n\n2. **Generator Lookup**:\n   - RDS generator (`pilot/pkg/xds/rds.go`) is invoked for the requesting proxy\n   - Looks up `bookinfo.example.com` in `virtualServiceIndex.publicByGateway`\n   - Retrieves the `VirtualService` spec\n\n3. **Route Computation** (`pilot/pkg/networking/core/route/`):\n   - Builds an Envoy `RouteConfiguration` with:\n     - Route match conditions (URI prefix)\n     - Route actions (forward to `reviews` cluster)\n     - Weights, retries, timeouts from the VirtualService spec\n\n4. **DestinationRule Application**:\n   - Looks up `reviews` in `destinationRuleIndex`\n   - If a `DestinationRule` exists for `reviews`, applies:\n     - Load balancing policy (round-robin, least request, etc.)\n     - Connection pool settings\n     - Outlier detection\n\n5. **Output**: An Envoy `RouteConfiguration` with fully populated route actions\n\n#### Example: Service + DestinationRule \u2192 Cluster\n\n1. **Input**: A Kubernetes Service `reviews.default.svc.cluster.local` and optional `DestinationRule`\n\n2. **Cluster Generation** (`pilot/pkg/xds/cds.go`, `pilot/pkg/networking/core/cluster_builder.go`):\n   - Looks up the service in `ServiceIndex.public` or `privateByNamespace`\n   - Creates an Envoy Cluster with:\n     - Service name as cluster name\n     - Virtual IP (from ClusterVIPs) or DNS resolution mode\n     - Service ports\n\n3. **DestinationRule Enhancements**:\n   - Looks up `DestinationRule` for the service in `destinationRuleIndex`\n   - If subsets are defined, creates subset-specific clusters (e.g., `reviews-v1`, `reviews-v2`)\n   - Applies load balancing, connection pools, outlier detection\n\n4. **Endpoint Resolution** (EDS \u2014 Endpoint Discovery Service):\n   - **Full CDS**: Cluster points to EDS (dynamic endpoint discovery)\n   - **EDS**: Endpoints are fetched from service registry, filtered by:\n     - Health status (Ready pods only)\n     - Sidecar scope visibility\n     - Network locality (for multi-cluster routing)\n\n### Shared State During Push\n\nDuring a single push:\n- All proxies see the same `PushContext` snapshot\n- All resource generators use the same indexes\n- If config changes mid-push, they're debounced and handled in the next push\n\nThis guarantees **consistency** \u2014 a proxy won't receive partially updated configuration that doesn't match other proxies' views.\n\n---\n\n## 4. Service Registry and Multi-Cluster Support\n\n### Purpose\n\nThe service registry feeds the discovery service with information about available services and their endpoints. It's abstracted via the `ServiceDiscovery` interface (`pilot/pkg/model/service.go:899`), allowing multiple implementations for Kubernetes, ServiceEntry, external registries, and aggregation across clusters.\n\n### ServiceDiscovery Interface (`pilot/pkg/model/service.go:899`)\n\nThe interface defines:\n\n```go\ntype ServiceDiscovery interface {\n    Services() []*Service\n    GetService(hostname host.Name) *Service\n    GetProxyServiceTargets(proxy *model.Proxy) []model.ServiceTarget\n    InstancesByPort(svc *Service, port int, labels labels.Collection) []*model.IstioEndpoint\n    GetNetworkGateway(network string) *model.NetworkGateway\n    // ... and other methods\n}\n```\n\n**Key Methods**:\n- **`Services()`** \u2014 Returns all services in the registry\n- **`GetService(hostname)`** \u2014 Looks up a service by FQDN\n- **`InstancesByPort()`** \u2014 Gets endpoints for a service on a specific port, optionally filtered by labels\n- **`GetProxyServiceTargets()`** \u2014 Returns services co-located with a proxy (for multi-cluster routing)\n\n### Kubernetes Service Registry\n\nThe Kubernetes implementation in `pilot/pkg/serviceregistry/kube/` watches Kubernetes API objects:\n\n#### Watched Resources:\n\n1. **Services** (`pilot/pkg/serviceregistry/kube/controller/controller.go`):\n   - Each Kubernetes Service becomes an Istio Service\n   - Hostname: `<name>.<namespace>.svc.cluster.local`\n   - Ports, ClusterIP, and labels are extracted\n\n2. **EndpointSlices** (`pilot/pkg/serviceregistry/kube/controller/endpointslice.go`):\n   - Kubernetes 1.19+ uses EndpointSlices instead of Endpoints\n   - Each slice contains a subset of endpoints for a service\n   - Pilot aggregates slices and watches for changes\n\n3. **Pods** (`pilot/pkg/serviceregistry/kube/controller/pod.go`):\n   - Watches pod creation/deletion to track endpoint health\n   - Used for determining which endpoints are \"Ready\"\n   - Feeds into the EndpointIndex for EDS\n\n4. **Nodes** (`pilot/pkg/serviceregistry/kube/controller/network.go`):\n   - Watches node labels for region/zone topology\n   - Used for locality-aware load balancing\n\n#### Service Registry Flow:\n\n```\nKubernetes API\n    \u2193\nService/Endpoint Watchers\n    \u2193\nEndpointIndex (pilot/pkg/model/endpointshards.go)\n    \u2193\nProxyUpdate() [notifies xDS server]\n    \u2193\nDiscoveryServer.ConfigUpdate()\n    \u2193\nPush to all proxies\n```\n\nWhen a pod is added/removed:\n1. EndpointSlice changes \u2192 detected by watcher\n2. `ProxyUpdate()` is called \u2192 triggers EDS push\n3. Pilot computes endpoint list and sends EDS update to proxies\n4. Proxies update their load balancers\n\n### ServiceEntry Registry\n\n`ServiceEntry` CRDs allow declaring external services:\n\n```yaml\napiVersion: networking.istio.io/v1alpha3\nkind: ServiceEntry\nmetadata:\n  name: external-db\nspec:\n  hosts:\n  - db.external.example.com\n  ports:\n  - port: 5432\n    protocol: TCP\n  resolution: STATIC\n  endpoints:\n  - address: 192.168.1.1\n```\n\nThe `serviceentry` registry provider converts these into Istio Service objects, allowing them to be treated like Kubernetes services for routing purposes.\n\n### Aggregate Controller (Multi-Cluster)\n\nThe aggregate controller in `pilot/pkg/serviceregistry/aggregate/` combines multiple service registries:\n\n```\nKubernetes Cluster 1\n    \u2193\nKubernetes Registry\n    \u2193\n        \u2192 Aggregate Controller \u2192 Combined view\n        \u2193\nKubernetes Cluster 2\n    \u2193\nKubernetes Registry + ServiceEntry\n```\n\n**Multi-Cluster Logic**:\n\n1. **Local Services** \u2014 From the local cluster's Kubernetes registry\n2. **Remote Services** \u2014 From other clusters (via `ServiceImport` CRDs or direct watches)\n3. **ServiceEntry Services** \u2014 Defined via Istio CRDs\n\nAll are aggregated into a single `ServiceDiscovery` interface. A proxy in Cluster 1 can access services from Cluster 2 if they're exported via:\n- Kubernetes `ServiceExport` CRD\n- Istio `exportTo` rules\n- Multi-cluster networking configuration\n\n#### Cross-Cluster Routing:\n\nWhen a proxy in Cluster 1 needs to reach a service in Cluster 2:\n\n1. Service discovered via aggregate registry (Cluster 2's service)\n2. CDS includes endpoints from Cluster 2\n3. Endpoints include the cross-cluster gateway IP (from `NetworkManager`)\n4. Traffic is routed through the gateway (via mTLS or HBONE in ambient mode)\n\n### Endpoint Index: Efficient Endpoint Lookups\n\nThe `EndpointIndex` in `pilot/pkg/model/endpointshards.go` maintains efficient endpoint lookups:\n\n- **Organized by Service Key** \u2014 `namespace/hostname`\n- **Sharded by Endpoint IP** \u2014 Allows for incremental updates (when pod scales, only affected shard updates)\n- **Cached InstancesByPort Results** \u2014 Frequently accessed endpoint lookups are cached\n\n**Usage in EDS**:\n\n```\nEDS Request (what endpoints for service X?)\n    \u2193\nEndpointIndex.InstancesByPort(service X, port 8080)\n    \u2193\nReturns endpoints filtered by:\n   - Health status (Ready pods only)\n   - Labels (if specified in DestinationRule subset)\n   - Network locality (for locality-aware load balancing)\n```\n\n### Triggering Pushes from Service Registry\n\nWhen service state changes:\n\n1. **Endpoint Change** (pod added/removed):\n   - Triggers EDS (Endpoint Discovery Service) push (not full push)\n   - Reason: `EndpointUpdate`\n   - Only clusters referencing that service are updated\n\n2. **Service Change** (service created/deleted):\n   - Triggers full CDS (Cluster Discovery Service) push\n   - Also triggers LDS/RDS (listeners and routes may reference new service)\n   - Reason: `ServiceUpdate`\n\n3. **Debouncing**:\n   - Multiple endpoint changes are debounced (if `EnableEDSDebounce` is true)\n   - Avoids flapping when many pods scale simultaneously\n\n### Multi-Cluster Failure Handling\n\nIf a remote cluster becomes unavailable:\n\n1. **Service discovery removes remote services** from the aggregate view\n2. **Proxies are updated** via a config push\n3. **Routes referencing removed services** are pruned\n4. **Clients get connection errors** (graceful degradation)\n\n---\n\n## Data Flow: From Configuration Change to Proxy Update\n\n### Complete Example: VirtualService Creation\n\n```\n1. User creates VirtualService CRD in Kubernetes\n   \u2193\n2. Kubernetes API notifies ConfigStore watcher\n   \u2193\n3. ConfigStore calls ConfigUpdate() with:\n   {Full: true, ConfigsUpdated: {VirtualService/namespace/name}, Reason: ConfigUpdate}\n   \u2193\n4. ConfigUpdate() sends to pushChannel\n   \u2193\n5. handleUpdates() debounces (waits for quiet period or max delay)\n   \u2193\n6. Push() is called:\n   a. initPushContext() rescans all CRDs, rebuilds indexes\n   b. Calls computeProxyState() for each connected proxy (if needed)\n   c. Enqueues PushRequest in pushQueue\n   \u2193\n7. sendPushes() distributes:\n   a. Acquires concurrentPushLimit semaphore\n   b. Iterates over adsClients (all connected proxies)\n   c. For each proxy, calls pushConnection()\n   \u2193\n8. For each proxy, resource generators create Envoy config:\n   a. RDS generator uses virtualServiceIndex to build routes\n   b. CDS generator uses ServiceIndex to build clusters\n   c. Generators apply DestinationRules, EnvoyFilters, etc.\n   \u2193\n9. xDS responses are sent to proxies via gRPC stream\n   \u2193\n10. Proxies acknowledge with DiscoveryRequest (ACK/NACK)\n    \u2193\n11. Errors tracked in ProxyStatus for debugging\n```\n\n### Complete Example: Pod Scale Event\n\n```\n1. Pod is added in Kubernetes\n   \u2193\n2. EndpointSlice watcher detects change\n   \u2193\n3. EndpointIndex is updated with new endpoint IP\n   \u2193\n4. (Same debounce \u2192 push \u2192 xDS flow)\n   \u2193\n7. EDS generator reads EndpointIndex for affected service\n   \u2193\n8. Custom Resource Generators\n\nUsers can extend Istio by registering custom generators for any xDS resource type:\n\n```go\ngenerator := &myCustomGenerator{}\ndiscoveryServer.Generators[\"my-generator-type\"] = generator\n```\n\nGenerators implement the `XdsResourceGenerator` interface:\n- `Generate()` \u2014 Called to produce resources for a proxy\n- `GenerateDelta()` \u2014 Called for delta/incremental updates\n\n**Use Cases**:\n- Custom cluster discovery from external systems\n- Custom route generation based on external rules\n- Custom secret generation from external PKI\n\n### 2. Custom Configuration Types\n\nVia the `ConfigStore` interface, custom CRD types can be added. EnvoyFilter\n\nEnvoyFilter CRDs in `pilot/pkg/networking/core/envoyfilter/` allow patching generated Envoy config:\n\n```yaml\napiVersion: networking.istio.io/v1alpha3\nkind: EnvoyFilter\nmetadata:\n  name: add-custom-header\nspec:\n  workloadSelector:\n    labels:\n      app: httpbin\n  configPatches:\n  - applyTo: HTTP_FILTER\n    patch:\n      operation: INSERT_BEFORE\n      value:\n        name: custom-header-filter\n        typed_config:\n          \"@type\": ...\n```\n\nEnvoyFilters are indexed in `PushContext.envoyFiltersByNamespace` and applied during resource generation.\n\n### 4. Configuration Validation Errors\n\nWhen `InitContext()` processes configurations:\n- Invalid CRDs are logged, not propagated\n- `PushContext.ProxyStatus` tracks errors per proxy\n- Proxies continue with last-known-good configuration\n\n**Recovery**: Fix the CRD and push again.\n\n### 2. Push Failures\n\nIf an error occurs while pushing to a proxy:\n- Tracked in `ProxyStatus[proxyID][errorCode]`\n- Exposed via `/debug/push_status` endpoint\n- Next `ConfigUpdate()` will retry\n\n**Common causes**:\n- Network errors (transient)\n- Stream closed (proxy disconnected)\n- Resource generation failure (bug)\n\n### 3. Proxy Connection Failures\n\nIf a proxy disconnects:\n- Connection is removed from `adsClients`\n- Next connection reconnects and gets full config\n- No special recovery needed\n\n### 4. Resource Generation Panics\n\nGenerators that panic are caught:\n- Error logged and returned to proxy\n- Other proxies unaffected\n- Investigation via logs\n\n---\n\n## Key Design Decisions\n\n### 1. Immutable PushContext\n\nEach push uses a snapshot of configuration. Prevents races where:\n- Generator reads a value\n- Configuration changes\n- Generator reads different value mid-operation\n\n### 2. Debounced Pushes\n\nMultiple rapid changes are debounced to:\n- Reduce computational load\n- Reduce network overhead\n- Ensure configuration converges before pushing\n\nTunable via `DebounceAfter` and `debounceMax` flags.\n\n### 3. Per-Proxy Generators\n\nEach proxy can use a different generator (sidecar vs. gateway vs. waypoint). Allows:\n- Customized config per proxy type\n- External systems to provide custom generators\n\n### 4. Cache-Aware Push Times\n\nPushes include `LastPushTime` to help the xDS cache distinguish stale entries from fresh ones. Ensures cache coherency across pushes.\n\n### 5. Incremental EDS\n\nEDS is incremental (only changed endpoints) while other resources are full. Reduces bandwidth for frequently-changing endpoints without complexity.\n\n---\n\n## Metrics and Observability\n\n### Key Metrics\n\n- **`pilot_xds_pushed_and_acked_configs`** \u2014 Configs # Istio Pilot Discovery Service Architecture

## Executive Summary

Istio Pilot's Discovery Service is the control plane component that translates high-level mesh configuration (VirtualServices, DestinationRules, etc.) into Envoy-specific xDS (Envoy Discovery Service) resources. It serves a bidirectional gRPC stream to each connected proxy, allowing Pilot to push configuration updates in real-time while proxies request specific resources. The service integrates with multiple service registries (Kubernetes, ServiceEntry, external services), supports multi-cluster deployments, and efficiently manages thousands of concurrent proxy connections.

This document explains the four primary subsystems that work together to implement this functionality:

1. **DiscoveryServer** — The gRPC server implementing Envoy's xDS v3 protocol
2. **ADS (Aggregated Discovery Service)** — The bidirectional stream handler and connection model
3. **PushContext** — The immutable configuration snapshot used during each push
4. **ServiceRegistry** — The service discovery layer feeding the DiscoveryServer
