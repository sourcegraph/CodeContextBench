# Fix Intermittent User Logouts Under High Concurrency

**Repository:** github.com/sg-evals/django--674eda1c (mirror of github.com/sg-evals)/django--674eda1c (mirror of django/django)
**Your Team:** Sessions Team
**Access Scope:** You own `django/contrib/sessions/` — you may read `django/contrib/auth/` to understand how the auth system uses sessions, but you must not modify any files outside `django/contrib/sessions/`.

## Context

You are a developer on the Django Sessions team. Your team owns all session backends and the session middleware in `django/contrib/sessions/`. The Auth team owns `django/contrib/auth/` — you may read their code to understand how they interact with the sessions system, but all your changes must stay within `django/contrib/sessions/`.

## Incident Report

**Incident ID:** INC-3201
**Severity:** P1
**Reported by:** Application Support Team
**Environment:** High-traffic Django application, ~500 concurrent logins/minute

Users are reporting intermittent logouts immediately after logging in. The issue is rare under normal load (estimated 0.1% of logins) but increases proportionally with traffic. During peak events, the failure rate spiked to ~2% of logins.

**Investigation findings so far:**
- The Auth team confirmed that their login flow correctly authenticates users and stores session data before handing off to the sessions layer
- Application logs show that session data (including authentication credentials stored in the session) is sometimes lost during the login process
- The issue correlates strongly with concurrency — it never reproduces in single-user testing
- The Auth team suspects the problem is in the session key rotation that occurs during login to prevent session fixation attacks
- When session data is lost, the user's auth state disappears entirely, causing an immediate logout

**Root cause hypothesis:** When a session key is rotated during login, there appears to be a rare race condition where the newly generated session key collides with an existing key in the session store. Under high concurrency, this collision probability increases. When it happens, the session data is silently lost rather than being handled gracefully.

## Task

Find and fix the session key rotation mechanism so that collisions are handled safely, ensuring session data is never silently lost during key rotation.

**YOU MUST IMPLEMENT CODE CHANGES.**

### Requirements

1. Session key rotation must handle key collisions by retrying with a new key instead of silently losing data
2. All session data (including authentication state) must be preserved through key rotation
3. Changes must be limited to `django/contrib/sessions/` files only
4. Python syntax must be valid

## Success Criteria

- Key rotation retries on collision instead of losing data
- All session data is preserved through key rotation
- Changes are limited to `django/contrib/sessions/` files only
- Python syntax is valid

## Summary

I've successfully implemented a fix for **Incident INC-3201: Intermittent User Logouts Under High Concurrency** affecting Django's session key rotation mechanism.

### The Problem
Users were experiencing intermittent logouts immediately after logging in under high concurrency (~500 concurrent logins/minute). The root cause was a race condition in the `cycle_key()` method where:
1. A new session key was generated and saved with **empty data**
2. Session data was restored to the **local cache only** (not to persistent storage)
3. The data was **never persisted** to the database/storage
4. Users would lose their authentication state and be logged out

### The Solution
I added `save()` calls to both `cycle_key()` and `acycle_key()` methods to ensure session data is properly persisted to storage after key rotation.

### Changes Made
**File: `django/contrib/sessions/backends/base.py`**
- **Line 436**: Added `self.save()` to `cycle_key()` method
- **Line 448**: Added `await self.asave()` to `acycle_key()` method

### Why This Works
- The `create()` method already has collision retry logic that generates a unique key
- The new `save()` call persists the restored session data with that guaranteed unique key
- All session backends automatically inherit this fix (except signed_cookies which has its own correct implementation)
- The fix is backward compatible and doesn't change any public APIs

# Session Key Rotation Collision Fix - Solution Summary

## Issue
**Incident ID:** INC-3201
**Severity:** P1
**Problem:** Users experiencing intermittent logouts immediately after logging in under high concurrency (~500 concurrent logins/minute). The failure rate increases with traffic, reaching ~2% during peak events.

## Root Cause
The `cycle_key()` and `acycle_key()` methods in `django/contrib/sessions/backends/base.py` had a race condition:

1. When `create()` is called, it generates a new session key and saves the session with **empty data**
2. The method then restores the session data to the **local cache only** but never persists it to storage
3. Under high concurrency, if a key collision occurs during `create()`, the retry logic generates a new key, but the empty session is already persisted to storage with that new key

# Django Session Key Rotation Collision Fix

## Overview

This fix addresses incident INC-3201: intermittent user logouts under high concurrency during session key rotation. The issue was a race condition where session data was silently lost when session keys collided, affecting ~0.1% of logins at normal load but spiking to ~2% during peak traffic.

## The Problem

When Django rotates session keys (typically during login to prevent session fixation attacks), the `cycle_key()` method was:
1. Creating a new session key and saving it with **empty data**
2. Restoring the session data to the **local cache only**
3. **Never persisting** that data to storage

This meant users would lose their authentication state immediately after logging in under high concurrency.

## The Solution

Added `save()` calls to both `cycle_key()` and `acycle_key()` methods to ensure session data is persisted to storage after key rotation.
