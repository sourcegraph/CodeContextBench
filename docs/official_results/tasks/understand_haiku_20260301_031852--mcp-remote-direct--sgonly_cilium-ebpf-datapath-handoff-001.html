Failure Modes\n\n### eBPF Verifier Failures\n\n**When it happens**: Program loading fails, kernel rejects the compiled eBPF bytecode\n\n**Common causes**:\n- **Unsafe memory access**: Accessing memory outside of BPF-allowed regions\n- **Unbounded loops**: Verifier detects potential infinite loops\n- **Out-of-range jumps**: Branch targets don't exist\n- **Missing null checks**: Map lookups not checked before dereference\n- **Invalid helper calls**: Calling BPF helper with wrong argument types\n\n**How Cilium handles it**:\n```go\n// pkg/datapath/loader/netlink.go\nvar ve *ebpf.VerifierError\nif errors.As(err, &ve) {\n    // Write full verifier log to disk for debugging\n    fmt.Fprintf(os.Stderr, \"Verifier error: %s\\nVerifier log: %+v\\n\", err, ve)\n}\n```\n\n**Recovery**:\n- Verifier log is written to `<endpoint-dir>/verifier.log`\n- Endpoint regeneration fails; endpoint status set to \"Failure\"\n- Operator must fix the C code causing the issue\n\n### Map Incompatibility Errors\n\n**When it happens**: Existing pinned map doesn't match program expectations (different key/value types, sizes, etc.)\n\n**Common causes**:\n- **Program update changes map layout**: e.g., adding fields to `PolicyKey`\n- **Size mismatch**: Old map max_entries < new program's needs\n- **Type change**: Hash map \u2192 array map, etc.\n\n**Error indication**:\n```\nebpf.ErrMapIncompatible: map specs differ\n```\n\n**How Cilium handles it**:\n```go\n// pkg/bpf/bpf_linux.go\nif errors.Is(err, ebpf.ErrMapIncompatible) {\n    // Found incompatible map. Must be removed before re-loading.\n    // Cilium pins maps and must clean them up manually.\n}\n```\n\n**Recovery**:\n- Remove the incompatible map: `rm /sys/fs/bpf/cilium/cilium_policy_<epid>`\n- Retry endpoint regeneration\n- Automatic cleanup happens in template cache revalidation\n\n### Kernel Compatibility Issues\n\n**How Cilium detects them**:\n```go\n// pkg/datapath/linux/probes/probes.go\nprobes.HaveProgramHelper(ebpf.SchedCLS, asm.FnFibLookup)  // Returns error if not supported\nprobes.HaveAttachType(ebpf.SchedCLS, ebpf.AttachTCEgress) // Check attach type support\n```\n\n**Common incompatibilities**:\n- **BPF JIT not available**: Kernel compiled without CONFIG_BPF_JIT\n- **Missing kernel helpers**: Feature added in later kernel (e.g., `bpf_fib_lookup()` added 4.17)\n- **Attach type not supported**: Newer attach types (e.g., TCX) need kernel 6.6+\n- **BPF type format (BTF) issues**: Kernel doesn't have BTF support\n\n**Examples**:\n```go\n// daemon/cmd/kube_proxy_replacement.go\nif probes.HaveProgramHelper(ebpf.SchedCLS, asm.FnFibLookup) != nil {\n    return fmt.Errorf(\"BPF NodePort services needs kernel 4.17.0 or newer\")\n}\n```\n\n**Recovery**:\n- Upgrade kernel or disable features requiring newer kernel versions\n- Cilium logs which features are disabled due to kernel limitations\n\n### Map Full / Full Conditions\n\n**When it happens**: BPF map reaches max_entries; new entries cannot be inserted\n\n**Maps that can fill**:\n- `cilium_ct_*` (connection tracking): Each connection = 1 entry\n- `cilium_nat_*` (NAT): Each NAT translation = 1 entry\n- `cilium_policy_<epid>`: Each peer identity \u00d7 protocol \u00d7 port = 1 entry\n\n**How Cilium handles it**:\n- **CT/NAT maps**: Garbage collection runs periodically to evict idle connections\n- **Policy maps**: Pressure metrics reported; operator warned via logs\n- **Recovery**: Resize maps via agent configuration (`--bpf-ct-tcp-max`, `--bpf-nat-max`, etc.)\n\n**Metrics**:\n```\ncilium_policy_endpoint_enforcement_status{enforcement=\"denied\"} # Policy map pressure\n```\n\n### Program Loading Failures (Permission, Resource Limits)\n\n**When it happens**: `BPF_PROG_LOAD` syscall fails\n\n**Common causes**:\n- **Insufficient privileges**: Not running as root\n- **BPF disabled in kernel**: CONFIG_BPF not set\n- **Memory limit**: Not enough memory to allocate program/maps\n- **Locked memory limit**: RLIMIT_MEMLOCK too small\n- **BPF syscall disabled**: Disabled via seccomp or AppArmor\n\n**How Cilium handles it**:\n```go\n// pkg/datapath/loader/netlink.go\nif err := bpf.LoadCollection(spec, opts); err != nil {\n    return nil, nil, fmt.Errorf(\"loading eBPF collection into kernel: %w\", err)\n}\n```\n\n**Recovery**:\n- Run Cilium with proper privileges\n- Increase locked memory limit: `ulimit -l unlimited`\n- Enable BPF in kernel config and recompile kernel\n\n### eBPF Program Crashes (Kernel Panic)\n\n**How it's prevented**: The BPF verifier guarantees programs cannot crash the kernel.\n\n**What CAN happen instead**:\n- Program returns `DROP` \u2192 Packet silently dropped\n- Program returns invalid action \u2192 Packet dropped with log\n- Program accesses invalid memory \u2192 Verifier rejects program\n\n**Monitoring**:\n- Monitor `cilium-dbg monitor` for `DROP` events\n- Check kernel logs for BPF-related warnings (dmesg)\n- Policy enforcement failures logged via monitor subsystem\n\n---\n\n## 5.
