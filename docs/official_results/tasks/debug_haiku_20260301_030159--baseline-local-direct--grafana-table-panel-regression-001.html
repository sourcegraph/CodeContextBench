I've completed a thorough investigation of the dashboard migration v38 regression in Grafana. The report has been written to `/logs/agent/investigation.md`.

### Key Findings:

**The Bug:** The V38 migration function (`processPanelsV38` in `/workspace/apps/dashboard/pkg/migration/schemaversion/v38.go` at lines 120-123) contains a critical conditional logic flaw:

```go
custom, ok := defaults["custom"].(map[string]interface{})
if !ok {
    continue  // <-- SKIPS ENTIRE PANEL
}
```

This causes panels to be completely skipped from processing if `fieldConfig.defaults.custom` doesn't explicitly exist, preventing the migration of field overrides from `custom.displayMode` to `custom.cellOptions`.

**Why It Happens:**
1. Frontend defaults creation (line 102-117 in `frontend_defaults.go`) creates `fieldConfig.defaults = {}` but does NOT create `fieldConfig.defaults.custom`
2. For dashboards without explicit custom configuration, the V38 migration's type assertion fails
3. The `migrateOverrides()` function (line 135) never executes, leaving field overrides unmigrated
4. Table formatting is lost because the frontend expects `custom.cellOptions` format (v38+)

**Why Explicit Defaults Work:**
Dashboards that explicitly include `fieldConfig.defaults.custom` in their JSON pass the type assertion and allow both defaults and overrides to be properly migrated.

**Root Cause:** The frontend DashboardMigrator.ts correctly processes overrides unconditionally (lines 659-669), but the backend V38 implementation incorrectly depends on defaults.custom existing first. Additionally, the test suite (`v38_test.go`) lacks coverage for this edge case.
