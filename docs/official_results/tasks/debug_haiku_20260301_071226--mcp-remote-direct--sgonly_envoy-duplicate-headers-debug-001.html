# Investigation: Duplicate Response Headers in Envoy Filter Pipeline

**Repository:** github.com/sg-evals/envoy--25f893b4 (mirror of envoyproxy/envoy)
**Task Type:** Deep Causal Chain (investigation only — no code fixes)

## Scenario

An Envoy proxy operator reports that response headers configured via `response_headers_to_add` in route configuration are being **duplicated** in certain responses. Specifically, when the router filter generates a local reply (e.g., upstream timeout, connection failure, or request too large), custom response headers like `x-request-id-echo` appear twice in the HTTP response.

The issue is intermittent — it only affects responses where the router itself generates the reply (local replies) rather than forwarding an upstream response. Normal proxied responses have headers added exactly once.

Access log snippet showing the problem (using `%RESPONSE_CODE_DETAILS%` formatter):

```
[2025-08-15T10:23:45.001Z] "GET /api/v1/data HTTP/1.1" 504 UT
response_code_details=upstream_response_timeout
x-custom-trace: abc123
x-custom-trace: abc123
```

The header `x-custom-trace` appears twice. The route config has:

```yaml
response_headers_to_add:
  - header:
      key: "x-custom-trace"
      value: "%REQ(x-request-id)%"
    append_action: OVERWRITE_IF_EXISTS_OR_ADD
```

Despite using `OVERWRITE_IF_EXISTS_OR_ADD`, the header is duplicated on local replies but not on proxied upstream responses.

## Summary

The investigation reveals that **response headers are duplicated in local replies due to double invocation of `finalizeResponseHeaders()`**:

### Root Cause
The FilterManager's local reply methods (`prepareLocalReplyViaFilterChain`, `sendLocalReplyViaFilterChain`, `sendDirectLocalReply`) unconditionally wrap the caller's `modify_headers` callback to call `finalizeResponseHeaders()` FIRST, then invoke the user's callback. When the router filter generates a local reply for a direct response (lines 486-503 in router.cc), it passes a callback that ALSO calls `finalizeResponseHeaders()`, resulting in two consecutive invocations.

### Key Evidence
1. **Router layer** (router.cc:486-503): Direct response callback includes explicit `finalizeResponseHeaders()` call
2. **FilterManager layer** (filter_manager.cc:1072-1078): Wraps all callbacks with another `finalizeResponseHeaders()` call  
3. **Header evaluation** (header_parser.cc:210-212): Uses `addReferenceKey()` which appends headers, so two calls = two copies
4. **Utility layer** (utility.cc:718-720): Executes the wrapped lambda, triggering both calls

### Causal Chain
Route config → Router local reply generation → FilterManager wrapping → Utility execution → Header evaluation (called twice) → Duplicate headers in response

# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` in route configuration are duplicated in local replies generated by the router filter (e.g., upstream timeout, connection failure, 404 errors) due to double invocation of `finalizeResponseHeaders()`. This occurs because the router's `sendLocalReply()` callback explicitly calls `finalizeResponseHeaders()`, and the FilterManager's local reply path unconditionally wraps the callback to call `finalizeResponseHeaders()` again before invoking the user-supplied modifier function.

## Root Cause

**File:** `source/common/http/filter_manager.cc` lines 1072-1078, 1123-1129, 1156-1162
**Mechanism:** The FilterManager's three local reply methods (`prepareLocalReplyViaFilterChain`, `sendLocalReplyViaFilterChain`, `sendDirectLocalReply`) create a wrapper lambda that unconditionally calls `finalizeResponseHeaders()` before executing the caller-supplied `modify_headers` callback.

When the router's `sendLocalReply()` is invoked with a callback that also calls `finalizeResponseHeaders()` (e.g., for direct responses), this results in the function being called twice in the same response header processing pipeline.
