# Prometheus TSDB Compaction Operational Runbook

## 1. Overview

### What is Compaction?

TSDB compaction is the process of merging multiple time-series blocks into larger, more efficient blocks. It serves three primary purposes:

1. **Space Efficiency**: Consolidates data from smaller 2-hour blocks into larger blocks (up to 31 days or 10% of retention), reducing metadata overhead
2. **Query Performance**: Fewer blocks to scan during queries, reducing I/O and improving query speed
3. **Tombstone Cleanup**: Removes series marked for deletion (via tombstones), recovering disk space

### Block Structure

Prometheus stores time-series data in a hierarchical block structure with exponential block durations:

- **Level 0 (Head Block)**: In-memory head block, ~2 hours of data
- **Level 1-10**: Persisted blocks with exponentially increasing durations
  - Each level is ~3x larger than the previous level
  - Example: 2h → 6h → 18h → 54h → ... → 31 days
  - Blocks containing tombstones (>5% deleted series) are also compacted

### When Compaction is Triggered

Compaction is triggered through two mechanisms:

| Trigger Type | Frequency | Condition |
|---|---|---|
| **Head Block Compaction** | On append commit | `head.compactable()` returns true when head block age > `chunkRange / 2` |
| **Scheduled Compaction** | Every 1 minute | Regular background loop in `db.run()` checks for blocks ready to compact |
| **Block Compaction** | On-demand | Multiple level-0/1 blocks fitting within a time range are merged upward |

**Code References:**
- Head compaction trigger: `dbAppender.Commit()` (tsdb/db.go:1115-1127)
- Scheduled trigger: `db.run()` (tsdb/db.go:982-1027)
- Compaction logic: `db.Compact()` (tsdb/db.go:1133-1212)
