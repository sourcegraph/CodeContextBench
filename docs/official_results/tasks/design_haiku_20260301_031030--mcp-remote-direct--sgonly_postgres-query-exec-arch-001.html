subsystems:\n- **Parser**: Only lexical/grammatical structure (no semantics)\n- **Analyzer**: Semantic resolution (no optimization)\n- **Rewriter**: Query transformation (no cost-based decisions)\n- **Planner**: Cost-based decision making (no execution)\n- **Executor**: Tuple processing (no planning decisions)\n\nThis separation allows each component to be independently tested, maintained, and improved.\n\n#### 2. **Visitor/Dispatch Pattern**\nUsed throughout:\n- **Parser**: Grammar rules dispatch to reduction functions\n- **Analyzer**: transformTopLevelStmt() dispatches to statement-specific analyzers\n- **Executor**: ExecProcNode function pointers dispatch to node-specific execution functions\n\n#### 3. **Tree Traversal**\nEach stage produces a tree structure:\n- Parser: RawStmt tree\n- Analyzer: Query expression trees\n- Planner: RelOptInfo and Path trees\n- Executor: PlanState tree\n\nEach tree is recursively traversed by its corresponding stage.\n\n#### 4. **Pull-Based Execution (Volcano Model)**\nThe executor uses pull-based execution:\n- Each node asks its children for tuples via ExecProcNode()\n- Tuples flow \"up\" the plan tree\n- Enables pipelining without intermediate materialization\n- Some nodes (Sort, HashJoin) still buffer internally but don't expose it\n\n#### 5. **Generate-and-Test Optimization**\nThe planner generates multiple candidate execution paths, estimates costs for each, and selects the cheapest:\n- Path generation (allpaths.c) creates all feasible paths\n- Cost estimation assigns cost to each path\n- Selection phase (createplan.c) chooses best path\n- Extensible through hooks for custom paths\n\n### Component Responsibilities\n\n| Component | Input | Output | Responsibility planner implements a sophisticated two-phase optimization process:\n\n**Phase 1: Path Generation (allpaths.c)**\n- Explores the space of feasible execution strategies\n- For each base relation: SeqScan, IndexScan, BitmapScan paths\n- For each join combination: NestLoop, HashJoin, MergeJoin paths\n- For each upper operation: Aggregate, Sort, Limit, Window paths\n- Assigns cost estimates to each path\n- Maintains list of candidate paths at each level of the RelOptInfo tree\n\n**Phase 2: Plan Creation (createplan.c)**\n- Starting from the best path at the top level (UPPERREL_FINAL)\n- Recursively converts each best Path to a corresponding Plan node\n- Resolves execution details:\n  - Which index to use and in what order\n  - Hash function for hash joins\n  - Sort key ordering\n  - Projection and filtering details\n- Assigns unique node IDs for plan identification\n- Handles special cases like subplans and parameters\n\nThis two-phase approach allows the planner to:\n1. Efficiently explore large solution spaces (Phase 1)\n2. Make detailed execution decisions only for the selected plan (Phase 2)\n3. Be extended with new path types without disrupting the overall algorithm\n\n### Volcano-Style Executor Dispatch\n\nThe executor implements the Volcano model through function pointers in PlanState nodes:\n\n**Dispatch Mechanism**:\n```c\n// In execProcnode.c:\n// ExecProcNode is a macro:\n#define ExecProcNode(ps) ((ps)->ExecProcNode(ps))\n\n// Each PlanState has:\nstruct PlanState {\n    ...\n    ExecProcNodeMtd ExecProcNode;  // Function pointer\n    ...\n};\n\n// ExecProcNodeMtd is defined as:\ntypedef TupleTableSlot *(*ExecProcNodeMtd)(struct PlanState *pstate);\n```\n\n**Dispatch Flow**:\n1.
