### Key Findings

**Root Cause:** The V38 migration function `processPanelsV38()` in `/workspace/apps/dashboard/pkg/migration/schemaversion/v38.go` (lines 120-123) contains a defensive check that causes it to skip **entire panels** if they don't have an explicit `fieldConfig.defaults.custom` object.

**The Bug:**
```go
custom, ok := defaults["custom"].(map[string]interface{})
if !ok {
    continue  // Skips processing entirely if custom doesn't exist
}
```

**What This Breaks:**
- Field overrides with `custom.displayMode` references are never migrated to `custom.cellOptions`
- These orphaned override references cause table panels to lose column formatting (widths, alignment, display modes)
- The data corruption is silent - no error is raised

**Why Explicit `defaults.custom` Escapes the Bug:**
- Dashboards that explicitly define `fieldConfig.defaults.custom` pass the check
- The `migrateOverrides()` function is then called
- Both defaults and overrides get properly migrated

**Migration Pipeline Context:**
1. `migrate.go:69-71` - Tracks which panels had original `custom` object
2. `migrate.go:75` - Applies frontend defaults (which **don't** create `custom`)
3. `migrate.go:119-126` - Runs V38 migration which **expects** `custom` to exist
4. Schema versions affected: v37 → v38

**Affected Components:**
- Package: `github.com/grafana/grafana/apps/dashboard/pkg/migration`
- Key files: `v38.go`, `migrate.go`, `frontend_defaults.go`
- Scope: All table panels without explicit `defaults.custom` when upgrading to v10.4+
