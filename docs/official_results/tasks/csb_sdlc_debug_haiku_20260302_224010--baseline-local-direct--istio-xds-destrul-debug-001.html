# Investigation: Stale Envoy Route Configuration After DestinationRule Update in Istio

**Repository:** istio/istio
**Task Type:** Deep Causal Chain (investigation only — no code fixes)

## Scenario

An Istio service mesh operator reports that updating a DestinationRule in their cluster **silently fails to propagate** to Envoy sidecars. Specifically, when they have multiple DestinationRules for the same host (e.g., one defining subsets, another defining traffic policy), updating one of them does not trigger an xDS push — the Envoy sidecar continues serving the **stale** cluster and route configuration.

The operator has two DestinationRules for `reviews.default.svc.cluster.local`:

**DR-1 (traffic policy):**
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: reviews-traffic-policy
  namespace: default
spec:
  host: reviews.default.svc.cluster.local
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: UPGRADE
```

**DR-2 (subsets):**
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: reviews-subsets
  namespace: default
spec:
  host: reviews.default.svc.cluster.local
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
```

When the operator updates `reviews-subsets` (e.g., adding a `v3` subset), the Envoy sidecar does **not** receive updated CDS/RDS config. Only restarting the pod picks up the change. Updating `reviews-traffic-policy` also has no effect.

Istio debug endpoint `GET /debug/config_dump` on the sidecar shows the old configuration even after the DestinationRule has been updated in the Kubernetes API server.

## Your Task

Investigate the root cause of why updating a merged DestinationRule fails to trigger an xDS push, and produce a report at `/logs/agent/investigation.md`.

Your report MUST cover:

1. **How Istio merges multiple DestinationRules for the same host** — specifically the `mergeDestinationRule()` function in `pilot/pkg/model/destination_rule.go` and how it combines subsets and traffic policies from multiple DRs into a single merged config
2. **How the merged DestinationRule loses identity metadata** — when two DRs are merged, only one DR's `config.Config` metadata (name/namespace) survives in the `consolidatedDestRules` structure. The metadata of contributing DRs is lost.
3. **How SidecarScope builds config dependencies** — the `AddConfigDependencies()` calls in `pilot/pkg/model/sidecar.go` that register which config resources a proxy depends on. Specifically, how `DefaultSidecarScopeForNamespace()` and `ConvertToSidecarScope()` iterate over `destinationRules` and add `ConfigKey{Kind: DestinationRule, Name: dr.Name, Namespace: dr.Namespace}` for each DR
4. **The specific gap: only one DR's ConfigKey is registered** — because the merged DR has only one name/namespace, `AddConfigDependencies` registers only that one DR. The other contributing DR(s) are not tracked.
5. **How the xDS push filter uses DependsOnConfig** — when a config change event occurs, `pilot/pkg/xds/` calls `SidecarScope.DependsOnConfig(configKey)` to decide if the proxy needs a push. If the updated DR is the one whose metadata was lost during merging, `DependsOnConfig` returns `false` and the push is **incorrectly skipped**.
6. **The relationship between PushContext.destinationRule() and the merge pipeline** — how `pilot/pkg/model/push_context.go` stores the `destinationRuleIndex` and how `destinationRule()` returns the merged config, obscuring the fact that multiple DRs contributed to it

The root cause is a **metadata loss during DestinationRule merging** that cascades through the dependency tracking and xDS push filtering systems.

## Key Findings

### The Core Issue

When Istio merges multiple DestinationRules for the same host (e.g., one with subsets, another with traffic policy), the `mergeDestinationRule()` function consolidates them into a **single merged configuration** using the **first DR's metadata** (name/namespace). The contributing DRs' metadata is permanently discarded.

### The Causal Chain

1. **Merging Phase** (`destination_rule.go:38-109`): Multiple DRs for `reviews.default.svc.cluster.local` are processed. The first DR gets all subsets and traffic policies merged into it in-place. The second DR's metadata is lost.

2. **Dependency Registration** (`sidecar.go:219-227`): When building SidecarScope dependencies, only the **merged DR's ConfigKey** is registered. The ConfigKey for the second DR (e.g., `reviews-subsets`) is never added to `configDependencies`.

3. **Update Event** (`proxy_dependencies.go:32-74`): When the operator updates `reviews-subsets`, a ConfigKey is created with its actual name/namespace. But the proxy's `configDependencies` only contains the hash for the merged DR's metadata.

5. **Result**: Envoy sidecar continues using stale configuration. Only pod restart picks up the change.

## Summary

When multiple DestinationRules target the same host (e.g., one defining subsets, another defining traffic policy), updating one of them fails to trigger an xDS push to affected Envoy sidecars. # Istio Investigation Memory

## DestinationRule Merging Bug

**Status:** Investigation Complete

### Key Findings

When multiple DestinationRules target the same host:
- `mergeDestinationRule()` (destination_rule.go:38-109) merges DRs in-place into the first DR
- Only the first DR's name/namespace metadata survives
- Contributing DRs' ConfigKeys are never registered as dependencies
- When a contributed DR is updated, its ConfigKey is not in proxy's configDependencies
- xDS push filtering correctly rejects the push because DependsOnConfig() returns false
- Result: Envoy sidecar gets stale configuration

### Root Cause Location

**Primary:** `pilot/pkg/model/destination_rule.go:38-109` - `mergeDestinationRule()`
- Merges multiple DRs for same host into first DR
- Loses metadata of contributing DRs permanently

**Secondary Gap:** `pilot/pkg/model/sidecar.go:219-227` - Dependency registration
- Only registers merged DR's ConfigKey, not contributing DRs'

**Push Filter:** `pilot/pkg/xds/proxy_dependencies.go:60-74` - `checkProxyDependencies()`
- Uses `SidecarScope.DependsOnConfig()` which does hash lookup
- Hash lookup fails for updated DR if it's not the "first" DR

### Fix Strategy

Need to track ALL contributing DRs when consolidating multiple DRs:
1. Modify `consolidatedDestRules` to store contributing DR ConfigKeys
2. Update `mergeDestinationRule()` to populate this list
3. Update `sidecar.go` dependency registration to include all contributing DRs
4. Option: Create `MergedDestinationRule` wrapper type

- `pilot/pkg/model/` - merging and dependency tracking
- `pilot/pkg/xds/` - push filtering logic
- `pilot/pkg/networking/core/` - route and cluster building
