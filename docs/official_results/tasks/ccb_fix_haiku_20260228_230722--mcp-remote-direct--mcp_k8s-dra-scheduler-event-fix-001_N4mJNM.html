# big-code-k8s-bug-001: Missing ResourceSlice Event Handler in Kubernetes Scheduler

## Task

Investigate a bug in the Kubernetes scheduler where pods requesting Dynamic Resource Allocation (DRA) devices become permanently stuck in an Unschedulable state when the DRA driver starts after the pod is created. Trace the execution path through the scheduler's event-driven re-queuing architecture to identify why ResourceSlice creation/update events are silently dropped.

## Context

- **Repository**: github.com/sg-evals/kubernetes--v1.30.0 (mirror of github.com/sg-evals)/kubernetes--v1.30.0 (mirror of kubernetes/kubernetes) (Go, ~3.5M LOC)
- **Category**: Bug Investigation
- **Difficulty**: hard
- **Entry Point**: `pkg/scheduler/schedule_one.go` — `ScheduleOne()` method

## Symptom

When a pod requests a DRA device and the DRA driver starts *after* the pod is created, the pod becomes permanently stuck in `Pending` with reason `Unschedulable`. The scheduler logs `"cannot allocate all claims"` because no ResourceSlices exist at scheduling time. After the driver starts and publishes ResourceSlice objects, the pod should be rescheduled — but it never is.

The bug is a race condition: if the driver starts first, scheduling succeeds immediately. But if the pod arrives before the driver, the scheduler never retries because ResourceSlice events are not wired into the scheduling queue's re-evaluation mechanism.

## Summary

I've traced the Kubernetes scheduler bug through the event-driven re-queuing architecture and identified the root cause. The analysis has been saved to `/logs/agent/solution.md`.

### The Bug
Pods requesting Dynamic Resource Allocation (DRA) devices become permanently stuck in an `Unschedulable` state when the DRA driver starts after the pod is created. The scheduler should re-evaluate the pod when ResourceSlice objects (which advertise available resources) are created, but it never does.

### Root Cause
**ResourceSlice events are silently dropped** due to missing event handler wiring across three layers:

1. **`pkg/scheduler/framework/types.go`** — ResourceSlice is not defined as a GVK constant and is missing from the `UnrollWildCardResource()` function

2. **`pkg/scheduler/eventhandlers.go`** — The `addAllEventHandlers()` function has no case for `framework.ResourceSlice`, so no informer handler is registered

3. **`pkg/scheduler/framework/plugins/dynamicresources/dynamicresources.go`** — The `EventsToRegister()` method doesn't include ResourceSlice events, so the plugin never signals the queue to retry pods

### The Fix
Add ResourceSlice support at all three layers following existing patterns for other DRA resources (ResourceClaim, ResourceClass, etc.):
- Define `ResourceSlice GVK` constant
- Add ResourceSlice to `UnrollWildCardResource()` 
- Register event handler in eventhandlers.go
- Add ResourceSlice event registration in the DRA plugin
