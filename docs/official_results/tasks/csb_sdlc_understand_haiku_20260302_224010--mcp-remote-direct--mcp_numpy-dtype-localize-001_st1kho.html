## Key Findings

**Root Cause:** The incompatibility stems from a chain of dtype conversions:

1. **Scikit-Learn Input Validation** (`sklearn/utils/validation.py:681-717`)
   - `_pandas_dtype_needs_early_conversion()` flags ALL integer dtypes (including nullable `Int64`) as requiring early conversion
   - `check_array()` (lines 935-942) calls `array.astype(new_dtype)` on these arrays

2. **Pandas Masked Array Handling** (`pandas/core/arrays/_utils.py:27-75`)
   - `to_numpy_dtype_inference()` is the actual culprit
   - When converting Int64 arrays with NA values, it automatically promotes `int → float64` (lines 41-42)
   - This happens because NumPy integers cannot natively represent NA/missing values

3. **Conversion Chain:**
   - Pandas `__array__()` protocol (lines 694-716 in `masked.py`)
   - Calls `to_numpy()` (lines 512-600 in `masked.py`)
   - Which invokes `to_numpy_dtype_inference()`
   - Results in silent conversion to float64

## Problems This Causes

- **Silent Type Conversion:** No warning when Int64 → float64
- **Precision Loss:** Large integers (> 2^53) lose precision
- **Semantic Mismatch:** Integer data becomes float data
- **Unexpected NaN Values:** NA values become `np.nan`

# NumPy Dtype Compatibility Issue: Pandas Nullable Integers in Scikit-Learn Preprocessing

## Summary

When a pandas DataFrame with nullable integer types (e.g., `pd.Int64Dtype`) is passed to scikit-learn's `StandardScaler` or `MinMaxScaler`, the conversion to NumPy arrays triggers an unexpected dtype promotion from `int64` to `float64`. This happens because of an interaction between pandas' masked array handling and scikit-learn's input validation, causing data type mismatches that can lead to silent precision loss or errors in subsequent NumPy operations.

## Root Cause

The issue stems from a chain of dtype conversion logic spanning three libraries:
