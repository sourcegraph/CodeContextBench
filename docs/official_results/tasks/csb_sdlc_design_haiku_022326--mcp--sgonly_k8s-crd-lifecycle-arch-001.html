## Architecture Overview

**Four Sub-Project Integration:**

1. **apimachinery** — Foundation layer providing:
   - `Scheme`: Type registry mapping Go structs ↔ GVK identifiers
   - `Unstructured`: Generic map-based representation for dynamic access
   - `ObjectMeta` & `TypeMeta`: Standard metadata on all objects
   - `GVK/GVR`: Universal resource identifiers

2. **apiextensions-apiserver** — Server-side with:
   - CRD type definitions (internal hub + external v1)
   - Validation against OpenAPI v3 schemas and CEL rules
   - Dynamic HTTP handler that serves CRD requests
   - etcd storage using Unstructured JSON codec

3. **client-go** — Client-side providing:
   - `DynamicClient`: Untyped access to any CRD via `Unstructured`
   - `SharedIndexInformer`: Watch-based local cache with eventual consistency
   - `Listers`: Read-only cache views for controllers
   - `SharedInformerFactory`: Deduplicates watches across resources

4. **api** — Well-known types (Pod, Service, etc.) sharing the same infrastructure

## Key Insights

**The Unstructured Checkpoint**: The bridge between server-side storage (JSON in etcd) and client-side access (Go maps). This enables:
- Runtime CRD registration without code generation
- Loose coupling between components
- Dynamic discovery and access

**Scheme's Central Role**: Acts as the semantic glue, mapping Go types → GVK identifiers used consistently across all projects for registration, serialization, and discovery.

**Client Cache Pattern**: SharedIndexInformer maintains eventual consistency by watching API server events and periodic resyncs, enabling controllers to query locally without network calls.
