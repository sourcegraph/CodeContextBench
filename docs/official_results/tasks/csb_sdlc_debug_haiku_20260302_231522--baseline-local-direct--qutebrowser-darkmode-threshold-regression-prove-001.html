## Summary

I have successfully created a comprehensive regression test at `/workspace/regression_test.py` that demonstrates the dark mode text threshold setting bug on Qt 6.4.

### Bug Investigation Results

**Root Cause Identified:**
The bug occurs because Chromium 99.0.4785.0 (Qt 6.4+) renamed the dark mode text brightness threshold setting from `TextBrightnessThreshold` to `ForegroundBrightnessThreshold`. However, the current qutebrowser code doesn't have a separate variant for Qt 6.4—it uses the same `Variant.qt_63` for all Qt >= 6.3, which still uses the old Chromium setting name.

**Evidence:**
- The `/app/qutebrowser/browser/webengine/darkmode.py` has only three variants: `qt_515_2`, `qt_515_3`, and `qt_63`
- Line 318 shows: `if versions.webengine >= utils.VersionNumber(6, 3): return Variant.qt_63`
- This means Qt 6.4, 6.5, etc. all use the same `Variant.qt_63` definition which specifies `TextBrightnessThreshold`
- For Qt 6.4+, Chromium expects `ForegroundBrightnessThreshold` instead

### Regression Test

The test file contains **3 test cases**:

1. **`test_qt64_text_threshold_setting_name`** (FAILS - demonstrates the bug)
   - Tests that Qt 6.4 generates the correct Chromium setting name
   - **Fails** because the current code uses `TextBrightnessThreshold` instead of `ForegroundBrightnessThreshold`
   - Provides a clear error message showing the old name is being used

2. **`test_qt63_still_uses_text_brightness_threshold`** (PASSES)
   - Verifies Qt 6.3 still correctly uses `TextBrightnessThreshold`
   - Ensures the fix won't break existing Qt 6.3 compatibility

3. **`test_qt64_text_threshold_not_in_blink_settings`** (PASSES)
   - Verifies the setting is in the correct `dark-mode-settings` switch, not `blink-settings`
