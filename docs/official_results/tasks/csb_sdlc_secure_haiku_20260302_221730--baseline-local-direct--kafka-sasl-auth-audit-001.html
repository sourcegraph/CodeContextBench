# big-code-kafka-sec-001: Kafka SASL Authentication Flow Security Analysis

## Task

Trace the Apache Kafka SASL authentication flow from channel creation through mechanism negotiation (PLAIN, SCRAM-SHA-256, OAUTHBEARER, GSSAPI), the challenge-response exchange, principal extraction, to ACL-based authorization. Identify all entry points where untrusted client data enters the SASL subsystem, map the data flow through each authentication mechanism, and analyze the security properties of the authentication and authorization chain.

## Context

- **Repository**: apache/kafka (Java, ~1.2M LOC)
- **Category**: Security Analysis
- **Difficulty**: hard
- **Subsystem Focus**: `clients/src/main/java/org/apache/kafka/common/security/` (authenticator, plain, scram, oauthbearer packages) and `clients/src/main/java/org/apache/kafka/common/network/` (channel builders)

## Requirements

1. Identify all entry points where untrusted data enters the SASL subsystem (client connections, SASL tokens, handshake requests)
2. Trace the channel builder architecture: how `ChannelBuilders` creates `SaslChannelBuilder` which wires up `SaslServerAuthenticator`
3. Map the SASL handshake protocol: `ApiVersionsRequest` -> `SaslHandshakeRequest` -> mechanism-specific exchange -> authenticated principal
4. Trace at least two mechanism implementations (PLAIN and SCRAM) through their callback handlers to credential verification
5. Identify how `KafkaPrincipal` is extracted post-authentication and how `Authorizer.authorize()` enforces ACLs
6. Document existing mitigations and where they may be insufficient

Network Reception (NetworkReceive)\n- **Vulnerability class**: Denial of Service (memory exhaustion)\n- **Untrusted input**: 4-byte size field read from socket\n- **Existing mitigation**:\n  - Size validated at line 92-95: must be >= 0 and <= maxSize\n  - maxSize configured via `BrokerSecurityConfigs.SASL_SERVER_MAX_RECEIVE_SIZE_CONFIG` (default 256KB)\n  - Memory allocated from memoryPool, which can be exhausted\n- **Gaps**:\n  - No rate limiting on invalid size attempts\n  - No per-connection size tracking (client can keep sending max-size buffers)\n  - Memory pressure could cause broker unresponsiveness\n\n#### 2. RequestHeader Parsing (Initial SASL Request)\n- **Vulnerability class**: Protocol confusion, invalid API key handling\n- **Untrusted input**: Kafka RequestHeader (API key, version, correlation ID, client ID)\n- **Existing mitigation**:\n  - Line 515: Validates API key is only API_VERSIONS or SASL_HANDSHAKE during handshake\n  - InvalidRequestException thrown for unexpected API keys\n- **Gaps**:\n  - No validation of correlation ID format (could be used for request confusion if proxy in path)\n  - Client ID is untrusted string, stored in logs/metrics (could contain injection payloads for log analysis)\n  - RequestHeader parsing assumes well-formed Kafka protocol (could throw ParseException)\n\n#### 3. SASL Mechanism Negotiation (SaslHandshakeRequest)\n- **Vulnerability class**: Authentication bypass (if custom handlers improperly implement interface)\n- **Untrusted input**: SASL mechanism name from client\n- **Existing mitigation**:\n  - Line 554: Validates mechanism is in enabledMechanisms list\n  - Mechanism set in broker config, not client-controlled\n  - Default mechanisms: PLAIN, SCRAM-SHA-256, SCRAM-SHA-512, GSSAPI, OAUTHBEARER\n- **Gaps**:\n  - If custom AuthenticateCallbackHandler implementation is provided, could have validation bugs\n  - No audit logging of mechanism selection attempt (would help detect attacks)\n\n#### 4. PLAIN Authentication Mechanism\n- **Vulnerability class**: Plaintext credential handling, username enumeration\n- **Untrusted input**: UTF-8 encoded string with null separators\n- **Parsing vulnerability**:\n  - Line 85: `new String(responseBytes, StandardCharsets.UTF_8)` \u2014 can decode arbitrary UTF-8\n  - No length validation before UTF-8 decoding (could process very long strings)\n  - No validation that null separators appear in expected positions\n- **Credential lookup vulnerability**:\n  - Line 65-67: Username used directly as JAAS config key (\"user_\" + username)\n  - No length limits on username (could be millions of characters)\n  - JAAS config lookup is O(n) or dictionary access O(1) depending on implementation\n  - **Username enumeration**: If credential lookup raises different exception for missing vs wrong password, attacker learns valid usernames\n    - Mitigation at line 103: Catches all Throwable and throws generic \"credentials for user could not be verified\"\n    - Mitigation at line 105-106: Throws generic \"Invalid username or password\" (no distinction)\n  - **Constant-time comparison**: Line 68 uses `Utils.isEqualConstantTime()` \u2014 good\n- **Gaps**:\n  - No rate limiting on failed authentication attempts\n  - No account lockout mechanism\n  - No audit logging of PLAIN authentication attempts\n  - UTF-8 decoding exception not caught \u2014 could reveal information\n\n#### 5. SCRAM Authentication Mechanism\n- **Vulnerability class**: Partial hash disclosure, nonce guessing\n- **Untrusted input**: SCRAM messages (RFC 5802 format)\n- **ClientFirstMessage parsing**:\n  - Line 100: `new ClientFirstMessage(response)` \u2014 parses untrusted bytes\n  - Username extracted without bounds checking\n  - **No nonce validation**: Line 106 generates nonce but doesn't validate client's nonce format/entropy\n  - Client's nonce passed through to ServerFirstMessage\n  - Attacker could send predictable/repeating nonces (doesn't break protocol but could leak timing info)\n- **Credential verification**:\n  - Line 126: Credential is null-checked and raises generic exception\n  - No username enumeration (same null exception for missing user and missing credential)\n- **Cryptographic verification**:\n  - Line 150-180: Proof verification uses HMAC comparison\n  - **No constant-time HMAC comparison visible** \u2014 need to verify Utils class\n  - Iterations validated at line 133-134: must be >= mechanism.minIterations()\n  - Salt is base64-decoded from credential (trusted source)\n- **Gaps**:\n  - No rate limiting on failed authentication attempts\n  - No protection against dictionary attacks (iterations count is critical)\n  - Nonce could theoretically be exhausted in repeated authentication attempts (2^64 possible)\n\n#### 6. Principal Extraction\n- **Vulnerability class**: Authorization bypass (if principal builder has bugs)\n- **Untrusted source**: SaslServer.getAuthorizationID() (set by mechanism)\n- **PLAIN principal extraction**:\n  - Line 84 (DefaultKafkaPrincipalBuilder): `return new KafkaPrincipal(KafkaPrincipal.USER_TYPE, saslServer.getAuthorizationID())`\n  - Username from PLAIN response becomes principal name directly\n  - **No validation** of principal name format/length\n  - Could contain special characters used by authorization system (e.g., wildcards, spaces)\n- **SCRAM principal extraction**:\n  - Same as PLAIN \u2014 username from SCRAM message becomes principal name\n  - **Risk**: If authorization system uses `String.contains()` or regex without anchoring, could have injection\n- **GSSAPI principal extraction**:\n  - Line 82: Applies KerberosShortNamer if GSSAPI\n  - KerberosShortNamer could have regex injection vulnerabilities (depends on rules)\n- **Gaps**:\n  - No validation that principal name matches allowed character set\n  - Custom PrincipalBuilder implementations could bypass all validation\n  - No serialization validation before storing principal in credential cache\n\n#### 7. Authorization\n- **Vulnerability class**: Principal spoofing, privilege escalation\n- **Untrusted principal**: Passed to Authorizer.authorize()\n- **Existing mitigation**:\n  - Principal comes from stored, but principal name is untrusted in format/content\n  - If Authorizer uses principal name in security decisions without escaping, could have injection\n\n---\n\n### Authentication Security Properties\n\n#### PLAIN Authentication\n- **Security level**: Weak (credentials transmitted in plaintext over SASL_PLAINTEXT)\n- **Assumed transport security**: SASL_SSL (TLS encryption) or SASL_PLAINTEXT with network isolation\n- **Vulnerability without SSL**:\n  - Credentials captured on network by passive observer\n  - No authentication of server to client (man-in-the-middle possible)\n  - No perfect forward secrecy\n- **Replay protection**: None (tokens can be replayed if network is not secure)\n- **Mitigation dependency**: Must use SASL_SSL (PLAIN + TLS), not SASL_PLAINTEXT\n- **Key derivation**: No key derivation (plaintext comparison)\n- **Constant-time comparison**: Yes, `Utils.isEqualConstantTime()`\n\n#### SCRAM Authentication\n- **Security level**: Stronger than PLAIN (hash-based, no plaintext credentials on wire)\n- **Protocol compliance**: RFC 5802 SCRAM-SHA-256 / SCRAM-SHA-512\n- **Credential derivation**:\n  - Server stores salted, iterated HMAC-SHA-256/512 of password\n  - Client proves knowledge of password without revealing it\n- **Replay protection**: Nonce included in each exchange, server-side nonce prevents replay\n- **Mitigation dependency**: Still uses TLS (SASL_SSL) for confidentiality and server authentication\n- **Iterations**: Configurable (default likely 4096+), increases security against offline attacks\n- **Key derivation**: PBKDF2-like with configurable iterations\n- **Constant-time comparison**: Should be used but not verified in visible code\n\n#### GSSAPI/Kerberos\n- **Security level**: Strongest (mutual authentication, keying material, ticket-based)\n- **Protocol compliance**: RFC 1964 (GSS-API), Kerberos v5\n- **Ticket authentication**: Client possesses TGT from KDC, proves identity\n- **Server authentication**: Client verifies server's ticket with KDC\n- **Forward secrecy**: Session key derived per connection\n- **Mitigation dependency**: Requires KDC (Kerberos infrastructure), not suitable for all deployments\n- **Principal name transformation**: KerberosShortNamer could have regex bugs\n- **GSSCredential injection**: maybeAddNativeGssapiCredentials() at line 374-402 in SaslChannelBuilder\n\n#### OAUTHBEARER\n- **Security level**: Depends on token issuer and validation implementation\n- **Credential bearer**: OAuth 2.0 bearer token (JWT or opaque)\n- **Token validation**: OAuthBearerUnsecuredValidatorCallbackHandler (name suggests \"unsecured\")\n- **Mitigation dependency**: Token issuer security (offline or online validation)\n\n---\n\n### Data Validation Vulnerabilities\n\n#### Length-based DoS\n1. **PLAIN username length**:\n   - Line 85 in PlainSaslServer: UTF-8 string decoded entirely\n   - No length check before lookup\n   - JAAS config lookup could be slow for very long keys\n   - **Mitigation**: None visible\n   - **Attack**: Send 1MB username string \u2192 broker spends CPU on string operations and config lookup\n\n2. **SCRAM username length**:\n   - Similar to PLAIN\n   - **Additional**: ClientFirstMessage parsing could fail on malformed SCRAM\n   - **Mitigation**: Exceptions caught and treated as auth failure\n\n3. **SASL token size**:\n   - NetworkReceive limits overall size (256KB default)\n   - **Mitigation**: Adequate\n\n#### UTF-8 Decoding\n- **PLAIN mechanism**: Line 85 calls `new String(responseBytes, StandardCharsets.UTF_8)`\n- **Risk**: Malformed UTF-8 could throw exception, but exception is handled generically\n- **No risk**: Java's UTF-8 decoder is well-tested, won't cause code execution\n\n#### Nonce Handling\n- **SCRAM server nonce**: Generated at line 106 with `formatter.secureRandomString()`\n- **Client nonce**: Accepted as-is from untrusted source\n- **Risk**: If RNG is weak, nonces could be guessable\n- **Mitigation**: Java's SecureRandom is cryptographically secure\n\n---\n\n### Configuration Security\n\n#### JAAS Configuration\n- **PLAIN passwords stored in**: JAAS config entries (in-memory)\n- **Loaded from**: Files or dynamic configuration\n- **Plaintext storage**: Passwords visible in config (no encryption at rest)\n- **Mitigation**: File permissions, OS-level access control\n- **Gap**: No encrypted password support for PLAIN\n\n#### Credential Cache\n- **SCRAM credentials stored in**: CredentialCache (in-memory HashMap)\n- **Loaded from**: Zookeeper or other metadata store\n- **Plaintext storage**: Salted hashes visible in memory\n- **Risk**: If JVM is compromised, all credentials accessible\n- **Mitigation**: JVM memory protection, secure erasure of cleared credentials\n\n#### Mechanism Selection\n- **Enabled mechanisms**: Set in broker config\n- **Immutable per broker start**: No dynamic mechanism enable/disable\n- **Risk**: If admin mistakenly enables PLAIN over SASL_PLAINTEXT, credentials exposed\n\n---\n\n### Re-authentication Attack Surface\n\n#### Mechanism Switch Prevention\n- **Line 532 in SaslServerAuthenticator**: `reauthInfo.saslMechanismUnchanged(clientMechanism)`\n- **Behavior**: If mechanism changes, sets REAUTH_BAD_MECHANISM state\n- **Mitigation**: Prevents downgrade (e.g., SCRAM \u2192 PLAIN)\n- **Configuration**: `CONNECTIONS_MAX_REAUTH_MS_CONFIG` per mechanism\n- **Behavior**: Broker forces re-auth after timeout\n- **Mitigation**: Limits window for stolen session token\n- **Gap**: If token/session is stolen, re-auth required before timeout\n\n---\n\n### Authorization Attack Surface\n\n#### Principal Serialization\n- **File**: `DefaultKafkaPrincipalBuilder.serialize()`\n- **Line 115-120**: Encodes principal to byte[] with type, name, and tokenAuthenticated flag\n- **Untrusted deserialization**: Broker could deserialize from untrusted source\n- **Risk**: If custom principal serializer is used, could have deserialization attacks\n- **Mitigation**: Default implementation is safe (manual serialization, no object deserialization)\n\n#### ACL Matching\n- **Authorizer.authorize()** uses principal name to match ACLs\n- **Risk**: If principal name contains wildcards, could match unexpected ACLs\n- **Example**: If principal name is \"user/*\", could grant \u2713\n   - Generic error messages prevent timing attacks\n   - Both mechanisms throw same exception for missing user/wrong password\n\n2. **Constant-time password comparison for PLAIN** \u2713\n   - `Utils.isEqualConstantTime()` prevents timing attacks on passwords\n\n3. **Strong nonce-based protocol for SCRAM** \u2713\n   - RFC 5802 SCRAM-SHA-256/512 prevents replay attacks\n   - Proof verification uses HMAC (cryptographically secure)\n\n4. **Re-authentication mechanism switching protection** \u2713\n   - Prevents downgrade attacks (SCRAM \u2192 PLAIN)\n   - Prevents principal switching during re-auth\n\n5. **Mechanism selection locked to broker configuration** \u2713\n   - Client cannot request unsupported mechanism\n   - Prevents mechanism downgrade attacks\n\n### Vulnerabilities & Gaps\n\n1. **No rate limiting on failed authentication**\n   - Attackers can brute-force passwords indefinitely\n   - No account lockout mechanism\n   - **Impact**: High (password guessing attacks)\n   - **Mitigation**: Implement rate limiting in broker or load balancer\n\n2. **No audit logging of authentication attempts**\n   - Cannot detect attack patterns\n   - Compliance requirement often unsatisfied\n   - **Impact**: Medium (forensics/compliance)\n   - **Mitigation**: Add audit logging to authenticator\n\n3. **PLAIN credentials sent in plaintext if not using SASL_SSL**\n   - Configuration error could expose credentials\n   - **Impact**: Critical if misconfigured\n   - **Mitigation**: Enforce SASL_SSL mode, disable SASL_PLAINTEXT in production\n\n4. **Principal name has no format validation**\n   - Could contain special characters used by authorization system\n   - Depends on Authorizer implementation for safety\n   - **Impact**: Low to Medium (depends on Authorizer)\n   - **Mitigation**: Validate principal name format in DefaultKafkaPrincipalBuilder\n\n5. **No DoS protection for large SASL tokens**\n   - 256KB max size is set, but no rate limiting on number of attempts\n   - Attacker could fill broker memory with pending authentications\n   - **Impact**: Medium (DoS)\n   - **Mitigation**: Implement per-connection request rate limiting\n\n6. **PLAIN UTF-8 decoding has no length limits per field**\n   - Could process multi-MB usernames or passwords\n   - JAAS config lookup could be slow\n   - **Impact**: Low (caught by max size validation, but inefficient)\n   - **Mitigation**: Add field-level length checks in token extraction\n\n7. **RequestHeader parsing assumes Kafka protocol format**\n   - Malformed request could throw exception (caught generically)\n   - Legacy GSSAPI clients (pre-KIP-43) not properly handled\n   - **Impact**: Low (handled with InvalidRequestException)\n   - **Mitigation**: Already mitigated by error handling\n\n8. **Custom callback handlers could bypass validation**\n   - SaslChannelBuilder allows custom `SASL_SERVER_CALLBACK_HANDLER_CLASS_CONFIG`\n   - Buggy custom handler could fail to authenticate properly\n   - **Impact**: Medium (configuration-dependent)\n   - **Mitigation**: Enforce strict callback handler interface contract, add validation tests\n\n---\n\n### Positive Security Properties\n\n1. **Multi-mechanism support** with fallback to SCRAM for most deployments (strong)\n2. **SCRAM RFC 5802 compliance** ensures industry-standard security\n3. **Kerberos/GSSAPI support** for enterprise environments with KDC\n4. **Principal extraction is mechanism-agnostic** (good architecture)\n5. **Metadata registry tracks client information** for debugging/monitoring\n6. **Extensible principal builder** allows custom authentication rules\n7. **Separation of concerns**: Transport layer (TLS), SASL mechanism, callback handlers, principal extraction, authorization\n8. **Pluggable authorization** allows custom ACL systems\n\n---\n\n### Recommendations\n\n1. **Implement per-connection/IP rate limiting** on failed SASL authentication attempts\n2. **Add field-level length validation** in PLAIN token extraction (username, password max lengths)\n3. **Add audit logging** of all SASL authentication attempts (success/failure, mechanism, principal, client IP)\n4. **Validate principal name format** in DefaultKafkaPrincipalBuilder (reject names with special characters)\n5. **Add configuration validation** to prevent enabling PLAIN with SASL_PLAINTEXT (enforce SASL_SSL)\n6. **Document custom callback handler requirements** and add validation tests\n7. **Implement per-connection timeout** on incomplete SASL handshakes (default 10 seconds)\n8. **Add metrics** for authentication success/failure rates by mechanism\n9. **Add explicit nonce format validation** in SCRAM implementation\n10. **Use constant-time HMAC comparison** in SCRAM proof verification
