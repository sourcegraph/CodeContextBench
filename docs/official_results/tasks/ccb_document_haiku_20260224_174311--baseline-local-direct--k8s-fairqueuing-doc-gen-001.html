Fairness Guarantees and Tradeoffs\n\n### 4.1 Fairness Model\n\nThe QueueSet provides **work-conserving fair queuing** with the following properties:\n\n1. **Weighted Fairness**: Each queue receives service proportional to its share of active demand\n   - Fair share = `ConcurrencyLimit / num_active_queues`\n   - A queue with more requests waiting gets no preferential treatment; it's enqueued in the same queues as low-demand flows\n\n2. **Virtual Time Ordering**: Dispatch is strictly ordered by virtual finish time\n   - If two queues are both active and have the same concurrency limit share, the queue with the smaller virtual finish time is served first\n   - This prevents a queue from \"jumping ahead\" by having broader requests\n\n3. **Shuffle Sharding Balancing**: Requests within the same flow are scattered across multiple queues\n   - A single bursty flow occupies only `handSize` queues instead of monopolizing one\n   - This reduces head-of-line blocking (slow requests don't block the entire flow)\n\n### 4.2 Known Limitations and Tradeoffs\n\n#### 4.2.1 Single Global Lock\n\n**Tradeoff**: All mutable state is protected by a single `sync.Mutex`.\n\n**Impact**:\n- Ensures correctness and consistency (no race conditions between queue selection and dispatch)\n- Can become a bottleneck if `StartRequest()`, `Finish()`, and dispatch loops contend heavily\n- Latency of virtual time updates is bounded by lock-holding time\n\n**Mitigation**: Lock-holding periods are kept small; expensive operations (like metrics recording) happen outside the lock.\n\n#### 4.2.2 Shuffle Sharding Overhead\n\n**Tradeoff**: Shuffle sharding requires `Dealer.Deal()` to generate a hand of queue indices via modular arithmetic.\n\n**Cost**: O(handSize) bit operations per enqueue (negligible; typically handSize \u2264 15).\n\n**Benefit**: Prevents a single bursty flow from monopolizing one queue.\n\n#### 4.2.3 Virtual Time Estimation Errors\n\n**Tradeoff**: Service time is estimated via `estimatedServiceDuration` (default 3ms); actual times may vary.\n\n**Impact**:\n- If actual service time < estimate, queue's `nextDispatchR` is incremented (punishing the queue for being faster)\n- If actual service time > estimate, queue's `nextDispatchR` is decremented (rewarding the queue for being slower)\n- This dampens oscillations but introduces bias if estimates are consistently wrong\n\n**Mitigation**: The estimate is conservative (3ms is a typical list operation); most requests are faster, so queues are rarely punished. Configurable via code, not via API objects.\n\n#### 4.2.4 Queue Removal by Attrition\n\n**Tradeoff**: Reducing `DesiredNumQueues` doesn't immediately remove excess queues; they drain by attrition (when they become empty).\n\n**Impact**:\n- If a queue has long-lived requests, the queue persists even after desired count drops\n- During configuration updates, total queue count may temporarily exceed desired count\n\n**Benefit**: Avoids disrupting in-flight requests; maintains isolation guarantees.\n\n#### 4.2.5 Seat-Based Work Model Limitations\n\n**Tradeoff**: Work is modeled as \"seats \u00d7 duration\"; actual CPU, memory, and I/O costs are not distinguished.\n\n**Impact**:\n- A 100-seat request that does I/O-only work monopolizes concurrency but uses little CPU\n- A 1-seat request with heavy CPU computation is starved\n- Seat allocation is per-priority-level, so no cross-priority admission decisions are made here\n\n**Mitigation**: Work estimation is delegable (via `workEstimator` parameter); custom estimators can cap seats per request type (e.g., watch operations vs. list operations).\n\n#### 4.2.6 No Preemption\n\n**Tradeoff**: Once a request is executing, it cannot be preempted; dispatch is only at the boundary (when a request calls `Finish()`).\n\n**Impact**:\n- A low-priority request that is currently executing blocks dispatch of high-priority requests\n- Wait times are bounded by the longest executing request\n\n**Rationale**: Preemption in HTTP request processing is complex (state cleanup, partial response handling); not feasible for a general-purpose API server.\n\n### 4.3 Fairness in Practice\n\n**Example scenario**: Two flows A and B, each with a queue, and ConcurrencyLimit = 10 seats.\n\n1. Flow A sends 5 requests, each requiring 4 seats (total 20 seats demanded)\n2. Flow B sends 1 request requiring 2 seats\n3. Initial dispatch:\n   - Both flows are active, so virtual time ratio = min(20+2, 10) / 2 = 5 seat-seconds per real second per queue\n   - If all 5 of A's requests start, they occupy 20 seats > 10 limit, so only 2.5 on average can run\n   - Request 1 of A gets dispatched, occupies 4 seats (6 remain)\n   - Request 1 of B gets dispatched, occupies 2 seats (4 remain)\n   - Request 2 of A gets dispatched, occupies 4 seats (0 remain)\n   - Dispatch loop exits: no more seats\n\n4. Virtual time advances as requests execute:\n   - If A's request 1 takes 1 second at 4 seats, its virtual finish time = arrivalR + 4 * 1 = arrivalR + 4\n   - If B's request takes 1 second at 2 seats, its virtual finish time = arrivalR + 2 * 1 = arrivalR + 2\n   - B's request finishes first (lower virtual finish time)\n   - When B's request completes, B has released 2 seats; A's queue 3 can now be dispatched\n\n---\n\n## 5. Failure Modes and Design Limits\n\n### 5.1 Virtual Time Overflow\n\n**Condition**: `currentR` exceeds `MaxSeatSeconds` (float64 ~= 9.22e18 seat-seconds)\n\n**Effect**: Epoch advancement is triggered, subtracting `rDecrement` from all progress meters\n\n**Manifestation**: Logs show \"Advancing epoch\" message (line 495)\n\n**Limits**: With ConcurrencyLimit = 5000 and continuous load, overflow occurs after ~60 million years (not a practical concern)\n\n### 5.2 Queue Underflow\n\n**Condition**: During epoch advancement, `queue.nextDispatchR` or `request.arrivalR` would underflow (become larger after subtraction)\n\n**Effect**: Logged as error: \"Underflow\" (line 507-508)\n\n**Causes**: Rare; # API Priority and Fairness (APF) QueueSet Subsystem: Deep-Dive Analysis

## Executive Summary

The APF QueueSet subsystem implements Fair Queuing for Server Requests (FQSR), a classical fair queuing algorithm adapted for distributed API server concurrency management. It provides differentiated service through a multi-queue dispatch mechanism that ensures proportional fairness across flows while respecting per-priority-level concurrency limits. The implementation uses virtual time progress metering, shuffle sharding for load balancing, and a seat-based work model to quantify request complexity.

---

## 1. Subsystem Purpose and Boundaries

### 1.1 Purpose

The QueueSet solves the problem of fair and predictable resource allocation when multiple competing flows arrive at a Kubernetes API server with a limited concurrency budget. Rather than rejecting requests when overloaded, QueueSet queues them and dispatches in a fairness-respecting order, preventing low-frequency or critical flows from being starved by high-frequency bulk operations.

Key capabilities:
- **Flow isolation**: Each flow (e.g., one client's requests) is assigned to a queue via shuffle sharding
- **Fair dispatch**: Requests are served in virtual time order, ensuring each active flow gets proportional progress
- **Concurrency limits**: Respects both per-priority-level seat limits and queue length constraints
- **Dynamic reconfiguration**: Handles changes to queue count, queue length limits, and concurrency allocations
