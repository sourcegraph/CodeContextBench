## Summary

The analysis document maps the complete checkpoint coordination architecture across **20+ critical files**, organized into six phases:

1. **Checkpoint Trigger** (JobManager → TaskManagers): CheckpointCoordinator initiates via RPC dispatch through Execution objects
2. **Barrier Injection** (Sources → Operators): Source tasks emit CheckpointBarrier events that serialize through the network
3. **Barrier Reception & Alignment** (TaskManager side): CheckpointBarrierHandler processes barriers—either blocking (aligned for exactly-once) or tracking (unaligned for at-least-once)
4. **State Snapshot** (Operator Execution): SubtaskCheckpointCoordinator collects operator and channel states

6. **Completion** (Two-Phase Commit Finalization): CheckpointCoordinator persists metadata, creates CompletedCheckpoint, and notifies tasks

**Key architectural patterns identified**:
- **Two-phase distributed commit protocol**: Prepare (trigger + collect state) → Commit (persist + notify)

- **Pluggable handler strategy**: SingleCheckpointBarrierHandler (aligned/exactly-once) vs CheckpointBarrierTracker (unaligned/at-least-once)
- **Asynchronous RPC coordination** with CompletableFuture aggregation
- **Idempotent acknowledgment protocol** tolerating out-of-order and duplicate messages
