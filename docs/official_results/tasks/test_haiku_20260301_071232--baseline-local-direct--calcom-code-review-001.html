## Description

You are reviewing a recently merged pull request that adds scope configuration for the feature opt-in system. This PR allows features to be restricted to specific scopes (org, team, user) instead of being available at all levels. ## Context

The feature opt-in scope configuration spans three key TypeScript source files:

1. **`packages/features/feature-opt-in/services/FeatureOptInService.ts`** — Service class: resolves feature states across teams with precedence rules, lists features for user/team scopes, and validates scope before setting states.

## Analysis

### Defect 1: `config.ts` line 76 - Missing null check for undefined scope
The `isFeatureAllowedForScope` function doesn't handle the case where `config.scope` is undefined. According to the documentation (lines 70-71), "Features without a scope field are allowed for all scopes." But when `config.scope` is undefined, calling `.includes()` on it will crash.

### Defect 2: `FeatureOptInService.ts` line 205 - Inverted filter in listFeaturesForUser
The comment says "Only returns features that are in the allowlist, **globally enabled**, and scoped to 'user'." But the filter uses `!state.globalEnabled`, which is the opposite. This returns features that are globally **disabled** instead of enabled. The `listFeaturesForTeam` method at line 240 correctly filters for `globalEnabled` without negation.

### Defect 3: `FeatureOptInService.ts` line 161 - Hard-coded policy instead of using config
The policy is hard-coded to "permissive" instead of being read from the feature configuration. The `OptInFeatureConfig` interface (config.ts line 8) defines a `policy` field, but the service ignores it. This is a compliance violation - the code should use the configured policy, not a hard-coded default.

### Defect 4: `_router.ts` line 95 - Missing slug validation in setUserState
The `setUserState` mutation (lines 88-104) doesn't validate that the slug is in the opt-in allowlist before calling the service. In contrast, `setTeamState` (line 117) and `setOrganizationState` (line 146) both validate with `isOptInFeature(input.slug)`. This is a compliance violation - inconsistent validation pattern.
