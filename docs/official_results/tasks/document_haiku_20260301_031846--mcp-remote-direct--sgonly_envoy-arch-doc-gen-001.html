Your document must cover these four components and how they work together:

The central network filter that owns the connection lifecycle. Explain:

The terminal decoder filter that forwards requests upstream. Explain:

The upstream connection pool and cluster management layer. Explain:
- How `ClusterManagerImpl` provides connection pools per cluster

# Envoy HTTP Connection Manager Architecture

## Overview

The HTTP Connection Manager (HCM) is Envoy's core subsystem for processing HTTP requests from downstream clients through to upstream servers. It acts as a **network-level filter** that owns the full lifecycle of downstream TCP/QUIC connections, creating and managing per-stream HTTP processing pipelines.

The HCM architecture centers on four primary components working together:

1. **ConnectionManagerImpl** — Connection lifecycle manager and codec orchestrator
2. **FilterManager and HTTP Filter Chain** — Per-stream request/response processing engine
3. **Router Filter** — Terminal decoder filter that performs route selection and upstream dispatch
4. **Cluster Manager and Upstream Connectivity** — Upstream connection pooling and load balancing

### Key Design Principles

- **Protocol Abstraction**: Codecs (HTTP/1.1, HTTP/2, HTTP/3) are abstracted behind a common interface, allowing HCM to process streams uniformly regardless of protocol.
- **Lazy Codec Instantiation**: Codecs are created on-demand (H1/H2 on first `onData()`, H3 on `onNewConnection()`), avoiding unnecessary overhead for idle connections.
- **Streaming Request/Response Processing**: The filter chain processes data as it arrives via callbacks (`decodeHeaders`, `decodeData`, `decodeTrailers`), enabling low-latency proxying.
- **Filter Chain Iteration Control**: Filters can pause iteration, buffer data, or abort processing, allowing sophisticated request/response manipulation without blocking the event loop.
