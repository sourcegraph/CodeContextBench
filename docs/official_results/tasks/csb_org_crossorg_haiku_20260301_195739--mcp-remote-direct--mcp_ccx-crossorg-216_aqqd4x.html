## Summary

I analyzed how Prometheus metrics are registered and exposed in **kubernetes/kubernetes**, **grafana/grafana**, and **prometheus/prometheus**, identifying three distinct architectural approaches:

### **Kubernetes** - Custom KubeRegistry Wrapper

- Uses a global `legacyregistry` module for centralized registration

- Uses standard Prometheus client library without wrappers

- Registration happens via `prometheus.MustRegister()` at init time or through registerer parameters
- Examples: `template/template.go`, `scrape/metrics.go`, `storage/remote/queue_manager.go`

- Uses modern `promauto.With(registerer)` for automatic metric registration
- Most concise approach with metrics created and registered in one step
- Metrics organized in struct types within functions like `NewSchedulerMetrics()`
