## Summary of Findings

I identified **5 key Python source files** in django/django (sg-evals/django--674eda1c) that implement Django's middleware processing:

### Core Files:
1. **django/core/handlers/base.py** - Contains `load_middleware()` which constructs the middleware stack by iterating through `settings.MIDDLEWARE` in **reversed order**, building an "onion" structure
2. **django/utils/deprecation.py** - Contains `MiddlewareMixin` class with `__call__()` and `__acall__()` methods that implement the request/response cycle
3. **django/core/handlers/exception.py** - Wraps middleware with `convert_exception_to_response()` for exception handling
4. **django/core/handlers/wsgi.py** - WSGI-specific handler that initializes synchronous middleware
5. **django/core/handlers/asgi.py** - ASGI-specific handler that initializes asynchronous middleware

### Key Architectural Insights:

- **Middleware Stack Construction**: Built in reversed order, creating a nesting of callables from outermost to innermost
- **Onion Model**: Requests go down through middleware in order, responses come back through the same middleware in reverse - only middleware that saw the request will process the response
- **Short-Circuiting**: If a middleware's `process_request()` returns a response, subsequent middleware are bypassed, preventing the "all middleware see all responses" behavior of the old MIDDLEWARE_CLASSES
- **Exception Handling**: Each middleware is wrapped with exception conversion to prevent exceptions from escaping the chain
- **Async/Sync Adaptation**: The `adapt_method_mode()` method automatically converts between sync and async contexts
