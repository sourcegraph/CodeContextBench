## Key Findings

**Root Cause**: Samples in partial batches are permanently lost when hard shutdown timeout expires during resharding queue transition, without being tracked in metrics or logs.

**The Stall Mechanism**:
1. During resharding, old queue workers attempt to flush partial batches
2. If the downstream `batchQueue` channel fills up, the retry loop sleeps
3. If the hard shutdown timeout (`flushDeadline`) expires before the batch flushes
4. The code receives the `done` signal and exits the retry loop
5. Then `q.batch = nil` discards any remaining samples in the partial batch
6. These samples never reach `runShard()` so they're not counted in failure metrics

**Why It's Intermittent**: Only manifests when resharding occurs while partial batches exist in queues AND the `batchQueue` channel fills up AND the hard shutdown timeout expires before flush completes. This is highly timing-dependent on network conditions.

**Why Metrics Show Stall**: The lost samples bypass the normal failure tracking mechanisms because they never reach the shard workers. The `pendingSamples` metric is reset to 0 when new shards start, masking the data loss.

- Exact file/line number references for the bug (`storage/remote/queue_manager.go:1447-1455`)

- Timing sensitivity analysis
- Diagnostic steps to confirm the root cause
