## Summary of Analysis

I traced the complete CRD lifecycle through four logically separate sub-projects:

### **Foundation Layer (apimachinery)**
- **Scheme**: Central type registry mapping Go types to GVK with serialization/deserialization
- **GVK/GVR**: Identifiers for types and resources
- **TypeMeta/ObjectMeta**: Metadata present on all objects
- **Unstructured**: Universal map[string]interface{} container for dynamic resources

### **Type Definition Layer (apiextensions-apiserver)**
- **Internal Hub Type**: `apiextensions.CustomResourceDefinition` (never serialized)
- **External v1 Type**: `apiextensions.v1.CustomResourceDefinition` (JSON-serializable)
- **Auto-generated Conversion**: Bidirectional conversion functions registered to Scheme
- **Schema Installation**: `install.Install()` registers all versions to Scheme with priority (v1 > v1beta1)

### **Server-Side Layer (apiextensions-apiserver)**
- **Scheme Initialization**: Sets up CRD types with proper versioning
- **CRD Storage**: Generic registry storing CRDs in etcd as internal hub types
- **Dynamic HTTP Handler**: `crdHandler` routes requests to custom resource endpoints, using Unstructured for serialization
- **UnstructuredObjectTyper**: Extracts GVK from Unstructured at runtime

### **Client-Side Layer (client-go)**
- **Typed Clients**: Type-safe access to CRDs (generated via code-gen)
- **Dynamic Client**: Generic `ResourceInterface` working with `*unstructured.Unstructured`
- **SharedIndexInformer**: Local cache that watches API server for changes
- **Dynamic Lister**: Queries cache without network calls
