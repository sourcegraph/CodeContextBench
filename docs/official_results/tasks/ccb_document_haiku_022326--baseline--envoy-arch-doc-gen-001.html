# Architecture Document: Envoy HTTP Connection Manager

**Repository:** envoyproxy/envoy
**Output:** Write your document to `/workspace/documentation.md`

## Task

Produce an architecture document for Envoy's **HTTP Connection Manager** (HCM) subsystem. The document must explain how the major components interact to process an HTTP request from arrival on a downstream connection through to upstream dispatch. Do not simply list APIs — explain the **design**, **data flow**, and **extension points**.

## Scope

Your document must cover these four components and how they work together:

### 1. ConnectionManagerImpl
The central network filter that owns the connection lifecycle. Explain:
- How it implements `Network::ReadFilter` to receive raw bytes
- How it creates and manages `ActiveStream` objects for each HTTP request
- Its role in codec creation (lazy codec instantiation for H1/H2 vs. H3)
- Connection-level concerns: drain decisions, overload management, watermark-based flow control

### 2. FilterManager and the HTTP Filter Chain
The per-stream filter chain execution engine. Explain:
- The decoder filter chain (request path) and encoder filter chain (response path)
- How `FilterChainFactory` creates filters for each stream
- Filter iteration: how `decodeHeaders`/`decodeData`/`decodeTrailers` propagate through decoder filters
- How filters can stop iteration, modify headers, or send local replies
- The distinction between `StreamDecoderFilter`, `StreamEncoderFilter`, and `StreamFilter` (dual)

### 3. Router Filter
The terminal decoder filter that forwards requests upstream. Explain:
- Route selection: how the router uses `RouteConfiguration` to pick a cluster
- How it obtains an HTTP connection pool from `ClusterManager`
- Upstream request lifecycle: `UpstreamRequest` creation, retry logic, timeout handling
- Shadow routing and hedged request support (if applicable)

### 4. Cluster Manager and Upstream Connectivity
The upstream connection pool and cluster management layer. Explain:
- How `ClusterManagerImpl` provides connection pools per cluster
- Load balancing: how the router obtains a host from the cluster's load balancer
- Connection pool mechanics: the relationship between logical connection pools and physical connections
- How health checking and outlier detection feed back into the load balancer

## Document Requirements

1. **Component Responsibilities** — what each component owns
2. **Data Flow** — the path of an HTTP request from downstream bytes to upstream dispatch, and the response path back
3. **Extension Points** — where users extend Envoy (HTTP filters, cluster extensions, load balancers, access loggers)
4. **Error Handling** — how errors at each stage (codec errors, filter errors, upstream failures) are handled
5. **relevant source files** — reference the actual source files in the envoyproxy/envoy repository

## Anti-Requirements

- Do NOT generate a simple API listing or header-file dump
- Do NOT fabricate class names or file paths that don't exist in the repository
- Do NOT cover Envoy components outside the HCM request path (e.g., xDS config delivery, listener management)

It is a complex orchestration of multiple interdependent components that work together to:\n\n1. Accept raw network bytes from downstream connections\n2. Decode HTTP frames using protocol-specific codecs (HTTP/1.1, HTTP/2, HTTP/3)\n3. Execute a chain of filters that can inspect, modify, or intercept the request\n4. Route the request to an appropriate upstream cluster\n5. Manage upstream connections and retry logic\n6. Encode the upstream response back to the downstream client\n\nThis document explains how the four primary components of the HCM interact to achieve this goal.\n\n---\n\n## 1. ConnectionManagerImpl: The Central Request Orchestrator\n\n### Responsibilities\n\n`ConnectionManagerImpl` (source/common/http/conn_manager_impl.h/cc) is the top-level filter in Envoy's network filter stack. It implements:\n\n- **Network::ReadFilter** \u2014 receives raw bytes from the downstream connection\n- **ServerConnectionCallbacks** \u2014 receives callbacks from the HTTP codec when new streams are created\n- **Network::ConnectionCallbacks** \u2014 manages the lifecycle of the underlying TCP/TLS connection\n- **Http::ApiListener** \u2014 allows direct HTTP request injection (for API listeners, not socket-based)\n\n### Key Responsibilities\n\n1. **Codec Management**: Creates and maintains an HTTP codec (HTTP/1.1, HTTP/2, or HTTP/3) lazily or on-demand. The codec is protocol-agnostic internally; ConnectionManagerImpl treats it as a black box that decodes bytes into HTTP frames and encodes HTTP frames back to bytes.\n\n2. **Stream Lifecycle**: For each incoming HTTP request, creates an `ActiveStream` object that wraps the stream and manages its state from arrival through completion. These streams are organized in a linked list for efficient management during connection shutdown or drain.\n\n3. **Filter Chain Creation**: When a new stream arrives, delegates to `FilterChainFactory` to create the per-stream filter chain (detailed below).\n\n4. **Connection-Level Concerns**:\n   - **Drain Decisions**: Monitors connection drain signals and prevents new streams on draining connections (graceful shutdown)\n   - **Overload Management**: Checks with `OverloadManager` to decide whether to accept new connections\n   - **Watermark-Based Flow Control**: Propagates buffer watermark events from the underlying connection to all active streams to manage memory usage\n\n5. **Stream Completion**: When a stream finishes, ensures proper cleanup, logging (via access loggers), and deferred deletion to avoid lifetime issues.\n\n### ActiveStream: Per-Stream State Wrapper\n\n`ActiveStream` (source/common/http/conn_manager_impl.h, lines 141\u2013end of private section) is a critical internal struct that:\n\n- Wraps a single HTTP request/response pair\n- Owns the `FilterManager` that executes the filter chain for this specific stream\n- Implements `RequestDecoder` to receive decoded HTTP frames (headers, data, trailers)\n- Implements `FilterManagerCallbacks` to encode responses back downstream\n- Maintains stream-specific state: headers, trailers, reset reasons, tracing spans, stream info\n\nThe lifecycle of ActiveStream:\n\n1. **Creation**: When the HTTP codec detects a new request (via `ServerConnectionCallbacks::newStream`), ConnectionManagerImpl creates an ActiveStream.\n2. **Filter Chain Execution**: The stream's FilterManager iterates through decoder filters as request data arrives (headers \u2192 data \u2192 trailers).\n3. **Response Path**: Once upstream sends a response, it flows back through encoder filters in reverse order.\n4. **Completion**: After response is fully encoded, the stream calls `completeRequest()`, logs access logs, and is marked for deferred deletion.\n\n### Codec Instantiation Strategy\n\nThe codec is created lazily:\n- **For HTTP/1.1 and HTTP/2**: On the first call to `onData()` (arrival of first byte from client)\n- **For HTTP/3**: On `onConnection()` (connection initialization)\n- **For API Listeners**: Caller must explicitly call `createCodec()`\n\nThis deferred instantiation reduces memory overhead for connections that never receive data.\n\n### Connection Shutdown and Drain\n\nConnectionManagerImpl maintains a `NetworkReadFilter` contract to receive `onNewConnection()` when a connection arrives and `onData()` when bytes arrive. During drain:\n\n1. New streams are not created (request returns 503 Service Unavailable)\n2. Existing streams are allowed to complete gracefully\n3. The underlying TCP connection remains open until all streams finish\n\n---\n\n## 2. FilterManager and the HTTP Filter Chain: Request Processing Pipeline\n\n### Responsibilities\n\n`FilterManager` (source/common/http/filter_manager.h/cc) is the per-stream filter chain execution engine. # Envoy HTTP Connection Manager Architecture

## Overview

The HTTP Connection Manager (HCM) is the core subsystem in Envoy that processes HTTP requests from downstream clients through upstream dispatch, response delivery, and back to clients. It is a complex orchestration of multiple interdependent components that work together to:

1. Accept raw network bytes from downstream connections
2. Decode HTTP frames using protocol-specific codecs (HTTP/1.1, HTTP/2, HTTP/3)
3. Execute a chain of filters that can inspect, modify, or intercept the request
4. Route the request to an appropriate upstream cluster
5. Manage upstream connections and retry logic
6. Encode the upstream response back to the downstream client

This document explains how the four primary components of the HCM interact to achieve this goal.

---
