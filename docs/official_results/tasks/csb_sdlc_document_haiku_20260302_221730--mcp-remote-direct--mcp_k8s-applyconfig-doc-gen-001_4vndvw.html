Algorithm\n\nStructured-merge-diff implements a three-way merge:\n\n1. **Live object** (current state in etcd)\n2. **Applied object** (what the user is applying)\n3. **Last known state** manager |\n| X | Y | Y | X (apply wins) | manager |\n| X | Y | Z | **CONFLICT** | manager vs other |\n| absent | Y | Z | Y (not applying, preserve) | other |\n| absent | Y | Y | removed (apply removes it) | manager |\n| absent | Y | absent | Y (unchanged) | \u2014 |\n\nThe key insight: if another manager owns the field (indicated by Z \u2260 Y), applying X creates a conflict unless `force=true`.\n\n### 4.3 Conflict Resolution Scenarios\n\n**Scenario 1: No conflict**\n```\nappliedObj wants spec.replicas=3\nliveObj has spec.replicas=2 (last applied by myapp-controller: 2)\n\u2192 Failure Modes, Limits, and Design Tradeoffs\n\n### 5.1 Failure Modes\n\n#### 5.1.1 Conflict Without Force\n\n**Trigger**: Apply patch to field owned by another manager, `force=false`\n\n**Error**: HTTP 409 Conflict with field-level conflict details\n\n**Recovery**: Client must choose between:\n- Retry with `force=true` to take ownership\n- Read current state, merge manually, re-apply\n- Wait for other manager to release ownership\n\n**Example**:\n```bash\n$ kubectl apply -f deployment.yaml --server-side\n# Error: Apply failed with 1 conflict: conflict with {manager: \"rolling-update-controller\"}\n# .spec.replicas\n\n$ kubectl apply -f deployment.yaml --server-side --force-conflicts\n# Success (takes ownership from rolling-update-controller)\n```\n\n#### 5.1.2 Invalid Patch Syntax\n\n**Trigger**: Malformed YAML/JSON in apply patch\n\n**Error**: HTTP 400 Bad Request with decode error\n\n**Recovery**: Fix patch syntax\n\n#### 5.1.3 Field Validation Failures\n\n**Trigger**: Apply patch with unknown fields (when `fieldValidation=Strict`)\n\n**Error**: HTTP 400 Bad Request with field details\n\n**Recovery**: Remove unknown fields or add CRD/schema definition\n\n#### 5.1.4 Admission Denial\n\n**Trigger**: Patch violates webhook validation rules\n\n**Error**: HTTP 403 Forbidden (or 400 depending on webhook)\n\n**Recovery**: Modify patch to satisfy admission policy\n\n#### 5.1.5 Concurrent Modifications (Lost Conflicts)\n\n**Trigger**:\n1. Manager A reads object with `managed.myapp={\"spec.replicas\"}`\n2. Manager B updates `spec.replicas` (now owned by B)\n3. Manager A applies patch claiming `spec.replicas` ownership\n4. Manager A gets 409 Conflict\n\n**Not a failure**, but expected behavior\u2014forces coordination.\n\n### 5.2 Hard Limits\n\n| Limit | Value | Source | Impact |\n|-------|-------|--------|--------|\n| Max request body | 3 MB (configurable) | `--max-request-body-bytes` | Prevents OOME attacks |\n| Max JSON Patch ops | 10,000 | `maxJSONPatchOperations` | Applies only to JSON Patch, not Apply |\n| Max managed fields entries | 10 (update managers) | `DefaultMaxUpdateManagers` | Merges oldest update managers to cap growth |\n| Max field path depth | Limited by JSON path | N/A | Unlikely to hit in practice |\n| Field cardinality | Schema-defined | OpenAPI schema | CRD schema constrains list size, etc. |\n\n**Soft limits**:\n- **Manager name string length**: No hard limit, but stored in every object\u2014avoid very long field manager names\n- **FieldSet encoding size**: Larger as more fields owned\u2014mature objects with many managers accumulate large managedFields\n\n### 5.3 Design Tradeoffs\n\n#### 5.3.1 Server-Side vs. Client-Side Apply\n\n| Aspect | Server-Side | Client-Side (legacy) |\n|--------|-------------|---------------------|\n| Conflict awareness | Yes (rich semantics) | No (annotation-based) |\n| Multi-manager support | Yes (field-level ownership) | No (all-or-nothing) |\n| Annotation bloat | No | Yes (stores config in `kubectl.kubernetes.io/last-applied-configuration`) |\n| Client complexity | Lower | Higher (read-modify-write) |\n| Server load | Slightly higher (merge logic) | Lower |\n| Backward compatibility | Can migrate from CSA | Requires upgrade path |\n\n**Tradeoff**: SSA pushes complexity to server (where it's shared across clients), making client code simpler.\n\n#### 5.3.2 ApplyConfiguration Struct Design\n\nWhy are all fields pointers in `*ApplyConfiguration` types?\n\n```go\n// Why NOT this:\ntype PodApplyConfiguration struct {\n    Name string\n    Labels map[string]string\n    // Problem: Can't distinguish \"user provided empty string\" from \"user didn't provide it\"\n}\n\n// Instead, we use:\ntype PodApplyConfiguration struct {\n    Name *string\n    Labels *map[string]string\n    // Solution: nil = omitted, &value = provided\n}\n```\n\n**Tradeoff**: Slightly verbose API (must use pointers, With*() methods) but prevents accidental field zeroing.\n\n#### 5.3.3 Managed Fields Encoding\n\nManaged fields stored in `FieldsV1` (FieldSet trie) rather than list of path strings:\n\n```go\n// Why NOT this:\nmanagedFields:\n  - manager: myapp\n    fields:\n    - spec.replicas\n    - spec.selector\n    - spec.template\n    - ...\n\n// Instead, we use trie encoding (FieldsV1):\nmanagedFields:\n  - manager: myapp\n    fieldsV1:\n      f:spec:\n        f:replicas: {}\n        f:selector: {}\n        f:template: {}\n```\n\n**Tradeoff**: More compact in etcd (trie compresses common prefixes), but less human-readable and requires trie library (sigs.k8s.io/structured-merge-diff).\n\n#### 5.3.4 OpenAPI Schema Requirement\n\nMerge algorithm requires OpenAPI schema to know:\n- List merge strategy (`set`, `map`, `replace`)\n- Map key for # Kubernetes Server-Side Apply and ApplyConfigurations: A Deep Technical Guide

## Executive Summary

Server-Side Apply (SSA) is Kubernetes' declarative object management system that enables safe, conflict-aware field ownership tracking through managed fields. The applyconfigurations subsystem provides typesafe Go client bindings to construct SSA requests. Together, they form a declarative alternative to imperative patch-based updates, with built-in multi-manager coordination via structured merge semantics.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### 1.1 Core Purpose

**Server-Side Apply** solves the fundamental problem of declarative resource management: how do multiple actors (controllers, operators, users) safely update the same Kubernetes object without clobbering each other's changes?

Traditional approaches fail because:
- **Client-side apply** stores configuration in an annotation, vulnerable to loss and difficult to migrate
- **Imperative patching** provides no field ownership semantics—any actor can change any field
- **PUT updates** are all-or-nothing, requiring clients to read-modify-write and risking lost updates

SSA shifts conflict resolution from the client to the server, where the complete picture of field ownership is available.

**ApplyConfigurations** solves the companion problem: Go applications need a typesafe, convenient way to construct apply request payloads without inadvertently including zero values that would be interpreted as field assignments.
