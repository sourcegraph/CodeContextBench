Virtual Time Mechanics and the Fair Queuing Model\n\n### Virtual Time as a Progress Meter\n\n**Definition** (`queueset.go:doc.go`):\n- Virtual time `R(t)` tracks seat-seconds allocated since process startup\n- In real-world fair queuing: R(t) = (bits transmitted at time t) / (transmission rate)\n- In APF: R(t) = (aggregate seats \u00d7 duration used at time t) / (fair share per queue)\n\n### Computing Virtual Time Advancement (`queueset.go:465-479`)\n\nAt each lock acquisition:\n\n```go\ntimeSinceLast := realNow.Sub(qs.lastRealTime)\nincrR := SeatsTimesDuration(qs.getVirtualTimeRatioLocked(), timeSinceLast)\nqs.currentR = prevR + incrR\n```\n\n**Virtual time ratio** (`queueset.go:526-541`):\n\n```go\nfunc (qs *queueSet) getVirtualTimeRatioLocked() float64 {\n    activeQueues := 0\n    seatsRequested := 0  // seatsInUse + max seats of waiting requests\n    for _, queue := range qs.queues {\n        if queue has waiting or executing: activeQueues++\n        seatsRequested += queue.seatsInUse + queue.requestsWaiting.MaxSeatsSum\n    }\n    if activeQueues == 0 { return 0 }\n    return math.Min(float64(seatsRequested), float64(qs.dCfg.ConcurrencyLimit)) / float64(activeQueues)\n}\n```\n\n**Interpretation**:\n- If 3 queues are active and concurrency limit is 30 seats:\n  - Each queue gets a \"fair share\" of `30 / 3 = 10` seats worth of virtual time per real second\n  - Virtual time advances at rate 10 seat-seconds/second per queue\n- If total seats requested is only 15, ratio is `min(15, 30) / 3 = 5` seat-seconds/second per queue\n- If only 1 queue is active, it advances at the full `min(seatsRequested, limit)` rate (but won't exceed limit)\n\n### Epoch Advancement (Overflow Prevention) (`queueset.go:485-522`)\n\nBecause `SeatSeconds` is a `uint64` and can overflow, the system periodically resets:\n\n```go\nconst (\n    highR = rDecrement + rDecrement/2         // \u2248 3/4 of max uint64\n    rDecrement = SeatSeconds(math.MaxUint64/2)\n)\n```\n\nWhen `currentR >= highR`, the system:\n1. Decrements `currentR` by `rDecrement`\n2. Decrements all `queue.nextDispatchR` by `rDecrement`\n3. Decrements all `request.arrivalR` by `rDecrement`\n4. Emits a metric `AddEpochAdvance()`\n\nThis is sound because:\n- All values are decremented equally, so relative orderings are preserved\n- Empty queues are reset to 0 (safe because they have no active requests)\n- Very long-running systems won't overflow for centuries at typical request rates\n\n---\n\n## V. Failure Modes, Limits, and Design Tradeoffs\n\n### A. Failure Mode: Queue Full (Rejection)\n\n**When it happens** (`queueset.go:622-624`):\n```go\nif qs.totSeatsInUse >= qs.dCfg.ConcurrencyLimit &&\n   curQueueLength >= qs.qCfg.QueueLengthLimit {\n    return false  // Reject\n}\n```\n\n**Why this design**: Protects server from unbounded queue growth while under load.\n\n**Consequence**: Client gets HTTP 429 (Too Many Requests) with `X-RateLimit-*` headers.\n\n**Mitigation strategies**:\n- Increase `QueueLengthLimit` if queue fills predictably\n- Increase `ConcurrencyLimit` if the server has spare capacity\n- Adjust FlowSchema rules to distribute load more evenly\n- Implement backoff/retry logic on the client\n\n### B. Failure Mode: Wide Requests Block Dispatch\n\n**When it happens** (`queueset.go:811-820`):\n```go\nif !qs.canAccommodateSeatsLocked(oldestReqFromMinQueue.MaxSeats()) {\n    // Selected queue has a request, but it can't fit in concurrency limit\n    // Must wait for currently executing requests to finish\n    return nil, nil  // Block dispatch of entire queue\n}\n```\n\n**Why**: A request requiring 50 seats can't execute if only 5 concurrency slots are available.\n\n**Impact**: If a queue contains only wide requests and other queues have narrow requests:\n- The wide request waits\n- But other queues continue to dispatch and execute\n- The blocked queue appears to be starved, but it's actually waiting for capacity\n\n**Mitigation**:\n- Design work estimators to avoid excessively wide requests\n- Set `ConcurrencyLimit` high enough for typical wide request width\n- Use `FinalSeats` for requests that release seats early (e.g., watches)\n\n### C. Design Tradeoff: Estimated vs. Actual Service Time\n\n**Trade-off**: All requests use the same default service time estimate (`estimatedServiceDuration = 3ms`).\n\n**Rationale**:\n- Simple: avoids per-request profiling\n- Reasonable: 3ms is typical for fast API operations\n- Self-correcting: anti-windup adjustment fixes systematic errors over time\n\n**Downside**:\n- A batch job that takes 10 seconds will have been overestimated; its queue gets credit\n- But the system adapts: repeated slow operations from the same queue get re-credited, maintaining fairness\n- The bound on divergence is C requests (concurrency limit): at most C requests ahead of the fair share\n\n**Example flow**:\n1. Queue A: 100 requests \u00d7 3ms estimate = 300ms virtual time per request\n2. Queue A actually takes 5 seconds per request\n3. Anti-windup: We deduct `(3ms - 5000ms) = -4997ms per seat`\n4. Queue A's `nextDispatchR` decreases (gets credited)\n5. Next requests from Queue A are dispatched sooner (fair correction)\n\n### D. Design Tradeoff: Shuffle Sharding Entropy\n\n**Hand size and deck size** (`queueset.go:583-587`):\n- `HandSize` is configured per-priority level (default ~8)\n- `DesiredNumQueues` is the \"deck size\" (total queues)\n- Dealer produces a hand: a random subset of queues from the deck\n\n**Tradeoff**:\n- **Larger hand**: Better load balancing, but less isolation (more collisions between flows)\n- **Smaller hand**: Better isolation, but risk of unbalanced hand if hand size is large relative to deck\n- Example: If hand size = 10 and deck size = 5, you're sampling WITH replacement\n\n**Mitigation**:\n- Dealer validates that `HandSize \u2264 DesiredNumQueues` (`queueset.go:203-207`)\n- Default hand size is small relative to default queue count\n- Shuffle sharding is probabilistic: collisions don't break fairness, just reduce isolation\n\n---\n\n## VI. # Kubernetes APF QueueSet: Algorithmic Deep-Dive

## Executive Summary

The Kubernetes Advanced Priority and Fairness (APF) QueueSet subsystem is a **fair queuing implementation for server requests** that enforces both priority-based fairness and concurrency limits across API request traffic. Unlike simple FIFO queuing, QueueSet employs a weighted fair queuing algorithm inspired by networking literature, adapted for server-side execution where multiple requests execute concurrently with unknown service times. The system maintains multiple queues per priority level, uses shuffle sharding for load balancing across queues, and implements virtual time tracking to ensure all flows receive proportional progress regardless of their service duration.

---

## I. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

The QueueSet subsystem addresses a fundamental problem in multitenant Kubernetes clusters: **preventing high-bandwidth flows from starving low-bandwidth flows**. A simple concurrency limit treats all requests equally, but in practice:

- A single user's large LIST request can occupy all concurrency slots indefinitely
- Legitimate interactive requests from other users get queued indefinitely
- Resource utilization becomes unpredictable and unfair

QueueSet solves this by:
1. **Assigning requests to multiple queues** based on shuffle sharding (probabilistic load balancing)
2. **Dispatching requests fairly** across queues using weighted fair queuing, where each queue's dispatch rate is proportional to the number of active queues
3. **Adjusting for actual service time** once requests complete, so queues serving slow operations don't get penalized
