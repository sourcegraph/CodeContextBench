Filters are instantiated in order and may be:\n- **Decoder-only**: Implement `StreamDecoderFilter`\n- **Encoder-only**: Implement `StreamEncoderFilter`\n- **Dual filters**: Implement `StreamFilter` (both decoder and encoder)\n\n#### Local Reply Mechanism\n\nIf a filter (or any upstream component) detects an error, it can issue a local reply:\n\n```cpp\nfilter_manager_.sendLocalReply(\n  Http::Code::BadRequest,\n  \"Invalid header\",\n  [](ResponseHeaderMap& headers) { /* modify headers */ },\n  absl::nullopt,  // grpc_status\n  \"filter_validation_failed\"\n);\n```\n\nThe FilterManager:\n1. Its responsibility is to:\n1. Select a route based on request properties\n2. Determine the target cluster/upstream service\n3. A new upstream request is started\n// 2. The first response to complete is used\n// 3. It:\n1. Owns a registry of Cluster objects (one per configured cluster)\n2. Provides HTTP and TCP connection pools for each cluster\n3. Coordinates health checking and outlier detection\n4. router requests a connection pool, the flow is:\n\n```\nRouter::getPool(cluster_name, load_balancing_context)\n  \u2193\nClusterManager::getPool()\n  \u2193\nCluster::getConnPool(dispatcher, connection pool manages:\n- **Active connections**: Connections with in-flight requests\n- **Idle connections**: Connections available for reuse\n- **Pending requests**: Requests waiting for an available connection\n- **Watermark buffers**: Per-connection write buffers with high/low water marks\n\n#### Host Selection (Load Balancing)\n\nThe load balancer selects a specific host (endpoint) for each request:\n\n```cpp\nclass LoadBalancer {\n  HostConstSharedPtr chooseHost(LoadBalancerContext* context);\n};\n```\n\nThe router provides a `LoadBalancerContext` containing:\n- Request # Envoy HTTP Connection Manager Architecture

## Overview

Envoy's HTTP Connection Manager (HCM) is a sophisticated subsystem that coordinates the processing of HTTP requests from downstream clients, through a configurable filter chain, to upstream services. This document explains the architecture, data flow, and interaction patterns between the key components that make up this system.

The HCM operates at the network layer (as a `Network::ReadFilter`) and is protocol-agnostic—it transparently handles HTTP/1.1, HTTP/2, and HTTP/3 through pluggable codec implementations. The request processing follows a well-defined pipeline: connection establishment → codec instantiation → per-stream filter chain execution → route selection → upstream forwarding.
