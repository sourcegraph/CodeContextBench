## Bug Report

**Bug ID**: 117651
**Summary**: Root NFS and autofs - mount disappears due to inode revalidate failed
**Regression**: Yes (works on 3.14.12, broken on 4.1.15)

After upgrading from 3.14.12 to 4.1.15 I've got strange behaviour with autofs.

It happens on diskless machine with root on NFS. There is autofs indirect mount on /storage, where NFS volumes are mounted on demand.

Everything mounts normally, but after mount expiration (during unmount) autofs mount disappears - that is, not only /storage/<volume> unmounts, but /storage is also unmounted. This happens quite often - usually on the very first expiration.

automount logs umount error in such cases.

Without autofs, mount/umount of the same NFS volume works correctly.

With NFS debug enabled, I've got following errors:

```
NFS reply getattr: -512
nfs_revalidate_inode: (0:15/6291480) getattr failed, error=-512
NFS: nfs_lookup_revalidate(/storage) is invalid
```

After inode revalidate failure all mounts on this inode are unmounted and so autofs mount disappears, while automount daemon itself continue to run.

tcpdump shows, that server replies to getattr without error, but client doesn't see this reply, and returning error instead.

I am ready to supply additional information.

## Summary

I have successfully localized the NFS inode revalidation bug in the Linux kernel 4.1.15. The analysis involved:

### Methodology:
1. **Analyzed the bug report**: Root NFS with autofs mounts disappearing after mount expiration due to inode revalidation failures
2. **Traced error messages**: Found error path from `fs/nfs/proc.c` and `fs/nfs/nfs3proc.c` reporting "NFS reply getattr: -512"
3. **Followed error propagation**: Traced error through `__nfs_revalidate_inode` → `nfs_lookup_verify_inode` → `nfs_lookup_revalidate`
4. **Identified the root cause**: Improper error handling in `nfs_lookup_revalidate` function

### Key Findings:

**Buggy File**: `fs/nfs/dir.c`
**Buggy Function**: `nfs_lookup_revalidate` (lines 1158-1161)

**The Bug**: 
When `nfs_lookup_verify_inode()` returns any non-zero error (including RPC errors like -512), the code unconditionally goes to `out_zap_parent`, which invalidates the dentry by returning 0 to the VFS layer. This causes the VFS to drop the dentry from cache.

**The Problem**: 
RPC errors (like -512) are transient network failures, not indications that the dentry is invalid. The current code treats them identically to parent directory change errors, causing:
- The dentry to be marked invalid
- The VFS to drop it from cache
- For autofs mount points, this unmounts the entire mount tree

**Why It's a Regression**: 
The code doesn't distinguish between errors that indicate parent directory changes (which require cache invalidation) and RPC/network errors (which should be propagated or retried).

The result has been written to `/workspace/fault_localization_result.json` with a confidence level of 0.85.
