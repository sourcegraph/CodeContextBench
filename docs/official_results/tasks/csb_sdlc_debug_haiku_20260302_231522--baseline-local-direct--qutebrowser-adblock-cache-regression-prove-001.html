# Bug Investigation: Application Crashes When Ad-Blocker Cache Is Corrupted

**Repository:** qutebrowser/qutebrowser
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

When qutebrowser starts up and attempts to load the ad-blocker filter cache, the application crashes if the cache file is corrupted or contains invalid data.

The expected behavior is that the application handles corrupted cache files gracefully — it should log an error, discard the invalid cache, and continue running normally (allowing the user to browse without ad blocking until the cache is rebuilt).

Instead, the deserialization error propagates as an unhandled exception, causing a crash before the browser window even appears. Users who encounter a corrupted cache file cannot use qutebrowser at all until they manually delete the cache file.

This is particularly problematic because cache corruption can happen for reasons outside the user's control, such as a system crash or disk error during a cache write.

## Summary

### Bug Investigation Results

I discovered the root cause of the application crash when the ad-blocker cache is corrupted:

**Location:** `/app/qutebrowser/components/braveadblock.py`, lines 204-230 in the `read_cache()` method

**Root Cause:** The code catches `ValueError` exceptions, but the actual exception raised by the adblock library when encountering corrupted cache is `adblock.DeserializationError`, which is **NOT** a subclass of `ValueError`. It inherits from `adblock.AdblockException` instead.

Exception inheritance chain:
```
adblock.DeserializationError
  → adblock.BlockerException
    → adblock.AdblockException
      → Exception
```

Since the exception type doesn't match, it's not caught and propagates as an unhandled exception, crashing the application.

### Regression Test Created

I've created a comprehensive regression test at `/workspace/regression_test.py` with:

1. **test_corrupted_adblock_cache_crashes** - Demonstrates the bug by:
   - Creating a corrupted cache file with invalid binary data
   - Calling `read_cache()` which attempts to deserialize it
   - Verifying that `adblock.DeserializationError` is raised (unhandled)
   - Confirming no error message was logged (proving it wasn't caught gracefully)

2. **test_valid_adblock_cache_loads** - Control test that:
   - Creates a valid cache file using the adblock library
   - Verifies it loads successfully without crashes
   - Confirms no error messages are logged

### Test Execution

The test is fully runnable with the specified command:
```bash
python3 -m pytest -c /dev/null --timeout=60 /workspace/regression_test.py
```

is corrupted. The expected behavior is that the app should handle corrupted
cache files gracefully, but instead it crashes with an unhandled exception.

====================
The bug is in qutebrowser/components/braveadblock.py, in the read_cache() method
(lines 204-230). The code catches ValueError exceptions:

    try:
        self._engine.deserialize_from_file(str(self._cache_path))
    except ValueError as e:
        if str(e) != "DeserializationError":
            raise
        message.error("Reading adblock filter data failed (corrupted data?). "
                      "Please run :adblock-update.")

adblock.DeserializationError, which is NOT a subclass of ValueError. It's a
subclass of adblock.AdblockException instead. This means the exception is NOT
caught by the code, and propagates as an unhandled exception, crashing the app.

The exception inheritance is:
  adblock.DeserializationError
    -> adblock.BlockerException
       -> adblock.AdblockException
          -> Exception

NOT a subclass of ValueError.

EXPECTED FIX:
=============
