## Key Findings

I've identified the root cause of the dashboard migration regression where table panels lose field override configurations in v10.4 when they don't explicitly contain `fieldConfig.defaults.custom` in the original dashboard.

### The Problem

The bug occurs at the intersection of **three components**:

1. **Backend v38 Migration** (`v38.go`): Correctly converts override properties from `custom.displayMode` → `custom.cellOptions`

2. **Field Config Tracking** (`frontend_defaults.go`): Before migrations run, the code marks panels that originally contained `fieldConfig.defaults.custom` with a `_originallyHadFieldConfigCustom` marker

3. **Cleanup Phase** (`frontend_defaults.go` lines 606-626): During the cleanup phase, the code deletes `fieldConfig` if:
   - It matches the default structure (empty defaults/overrides)
   - **AND** the `_originallyHadFieldConfigCustom` marker is FALSE

### Why It Fails

- **Original dashboard:** Has NO `fieldConfig.defaults.custom`, but HAS overrides with `custom.displayMode`
- **Marker:** Not set (because custom doesn't exist initially)
- **After v38 migration:** Overrides are properly converted to `custom.cellOptions`
- **Cleanup:** Deletes the entire `fieldConfig` because the marker is false, even though overrides exist

### Why Other Dashboards Work

Dashboards with **explicit** `fieldConfig.defaults.custom` get the marker set to `true`, which prevents deletion during cleanup, allowing overrides to survive.
