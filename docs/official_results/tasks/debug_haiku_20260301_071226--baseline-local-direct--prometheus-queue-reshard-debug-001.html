### **Root Cause Found**
The issue is a **metric accounting bug during resharding** where:

1. **Partial batches get lost** - During the flush phase, if the remote endpoint is slow or unresponsive, partial batches can be discarded when `FlushAndShutdown()` closes the queue channel without sending them.

2. **Metrics aren't properly reconciled** - When `shards.start()` initializes new shards, it resets `enqueuedSamples` counter to zero using `.Store(0)` instead of properly decrementing the old counter from `pendingSamples` metric.

3. **Result**: The `prometheus_remote_storage_samples_pending` metric stays stuck showing samples that were actually lost, creating the illusion of a stall.

### **Why It's Intermittent**
Three conditions must align:
- **Slow/unresponsive remote endpoint** → timeout triggers hard shutdown
- **Target discovery changes** → generates sample spike during reshard transition  
- **Partial batch exists** → not yet full when resharding initiated

- **Line 1689**: Only decremented on successful send
