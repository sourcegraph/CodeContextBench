# Envoy Connection Pool Management Troubleshooting Runbook

## 1. Architecture Overview

### Connection Pool Organization

Envoy connection pools manage upstream connections on a **per-cluster, per-thread basis**. Each Envoy worker thread maintains its own pool of connections for each upstream cluster, ensuring thread-safe operation without locks.

#### Pool Structure

Connection pools maintain four lists of active clients:
- **`ready_clients_`**: Connections in `Ready` state (can accept new streams immediately)
- **`busy_clients_`**: Connections at concurrent stream limit or draining
- **`connecting_clients_`**: Connections still establishing TCP/TLS handshake
- **`early_data_clients_`** (HTTP/3): Connections with 0-RTT credentials ready for early data

Pending streams waiting for available connections are held in:
- **`pending_streams_`**: Queue of requests waiting for a connection to become available

**Source**: `source/common/conn_pool/conn_pool_base.h:376-388`

### Connection Lifecycle

A connection progresses through the following states:

```
Connecting → Ready ──┬─→ Busy (at concurrent stream limit)
                      │
```
