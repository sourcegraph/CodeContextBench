## Architecture Overview

Kubernetes uses a staging directory structure where four logically separate projects coexist in one monorepo:

1. **apiextensions-apiserver** — CRD type definitions, validation, etcd storage, dynamic HTTP handler
2. **client-go** — Generated typed clients, informers, listers, and the dynamic client
3. **apimachinery** — Runtime scheme, GVK/GVR types, ObjectMeta, Unstructured representation
4. **api** — Internal hub types and group registration

## Requirements

1. Trace the CRD type definition layer: internal hub types, external v1 types, and Scheme registration
2. Map the server-side lifecycle: validation, etcd storage, and the dynamic customresource_handler
3. Trace the client-side access layer: typed clientset, SharedIndexInformer, Lister, and DynamicClient
4. Identify the foundational types in apimachinery that everything depends on: Scheme, GVK/GVR, TypeMeta/ObjectMeta, Unstructured

**apiextensions** registers CRD types at startup via `install.Install(scheme)`\n- **api** registers all core API types (Pod, Service, etc.) via group-specific `AddToScheme()`\n- **apimachinery** provides the Scheme implementation and core Object/Unstructured interfaces\n- **client-go** uses Scheme to encode/decode objects for REST operations\n\n```go\n// Every generated typed client carries a Scheme:\n// clientset.Typed().CoreV1().Pods(namespace).Get(name)\n// \u2192 encodes GroupVersionKind via Scheme\n// \u2192 makes REST call\n// \u2192 decodes response via Scheme\n```\n\n### 2. **Unstructured as the Bridge**\nThe `Unstructured` interface and `unstructured.Unstructured` struct are the critical bridge:\n- **Server-side**: CRDs stored as Unstructured in etcd (no type stubs needed)\n- **Wire format**: JSON serialization of Unstructured (same as any K8s object)\n- **Client-side**: Dynamic client and informers work entirely with Unstructured\n\n```go\ntype Unstructured struct {\n    Object map[string]interface{} // Raw YAML/JSON structure\n}\n// Implements runtime.Object interface \u2192 can be used anywhere in Kubernetes\n// Implements runtime.Unstructured interface \u2192 provides UnstructuredContent() accessor\n```\n\n### 3. **Request Scope as the Checkpoint**\nThe `handlers.RequestScope` object created per CRD version is the checkpoint between:\n- **Server-side validation/storage** (CustomResourceStorage)\n- **Client-side discovery/access** (dynamic client/informers)\n\n```go\ntype RequestScope struct {\n    Serializer runtime.NegotiatedSerializer // Unstructured codec\n    Mapper     meta.RESTMapper // GVR \u2194 GVK mapping\n    Admission  admission.Interface // Webhook validation\n    ...\n}\n// Used by customresource_handler to process HTTP requests\n// Serializer determines how Unstructured objects are encoded/decoded\n```\n\n### 4. **TypeMeta Preservation in etcd**\nCritical for dynamic resources:\n```go\n// Custom resource in etcd looks like:\n{\n  \"apiVersion\": \"example.com/v1\",     // From Unstructured.Object[\"apiVersion\"]\n  \"kind\": \"MyResource\",               // From
