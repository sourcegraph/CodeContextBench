## Fault Localization Summary

**Bug:** HDA Intel sound stops working (becomes choppy with clicks/silences) after suspend to RAM in kernel v3.7.1+ (regression from v3.6.11)

**Root Cause:** The `azx_init_chip()` function in `sound/pci/hda/hda_intel.c` skips CORB/RIRB (Command/Response ring buffer) initialization when `chip->single_cmd` flag is set, even during a full hardware reset after suspend/resume.

**How it happens:**
1. During normal operation, if a codec communication timeout occurs, the driver enters error recovery and sets `chip->single_cmd = 1` (lines 951 in `azx_rirb_get_response()`)
2. System suspends - `azx_suspend()` calls `azx_stop_chip()` which resets `chip->initialized` but **NOT** the error recovery flags
3. System resumes - `azx_resume()` calls `azx_init_chip(chip, 1)` for full reset
4. In `azx_init_chip()`, lines 1222-1223 check `if (!chip->single_cmd)` before calling `azx_init_cmd_io()`
5. Since `single_cmd` is still 1 from before suspend, the CORB/RIRB are never re-initialized
6. The codec communication fails because hardware is reset but driver is in incompatible single_cmd mode

**Buggy file:** `sound/pci/hda/hda_intel.c`
**Buggy function:** `azx_init_chip()` (lines 1222-1223)
**Confidence:** 95% (based on clear evidence from code analysis, error messages in bug report, and suspend/resume flow)
