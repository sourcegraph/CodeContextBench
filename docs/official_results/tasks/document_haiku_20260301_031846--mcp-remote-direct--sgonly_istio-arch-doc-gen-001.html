# Istio Pilot's Discovery Service Architecture

## Overview

Istio Pilot's Discovery Service is the core subsystem that translates Istio's high-level configuration (VirtualServices, DestinationRules, ServiceEntries) into Envoy Data Plane API (xDS) configuration. It serves as a gRPC-based xDS server that maintains long-lived streams with Envoy proxies in the mesh, pushing configuration changes in near real-time. The service bridges the gap between Istio's control plane abstractions and Envoy's low-level proxy semantics.

This document explores the four interrelated components that comprise this system and explains how they work together to manage the complete lifecycle of configuration distribution.

---

## 1. DiscoveryServer: The xDS gRPC Server

**Source:** `pilot/pkg/xds/discovery.go`

### Purpose

`DiscoveryServer` is the central component implementing Pilot's gRPC interface for Envoy's xDS APIs. It serves as the main entry point for all Envoy proxy connections and orchestrates the push of configuration updates to connected proxies.

### Key Responsibilities

1. **gRPC Server Registration**: Registers the ADS (Aggregated Discovery Service) handlers with the gRPC server, implementing the v3 xDS protocol specified in the Envoy data plane API.

2. **Connection Tracking**: Maintains a map (`adsClients`) of all active proxy connections, keyed by connection ID. Each connection represents a bidirectional gRPC stream with an Envoy proxy.

3. **Configuration Push Orchestration**:
   - Accepts configuration update requests via `ConfigUpdate()`
   - Manages push debouncing and rate limiting to avoid overwhelming proxies with frequent updates
   - Coordinates the creation of new `PushContext` snapshots when configuration changes
   - Distributes updates to connected proxies
