## Summary of Findings

I traced the CRD (Custom Resource Definition) lifecycle through:

### **1. Foundation: apimachinery**
- **Scheme**: Central type registry mapping GVK ↔ Go types
- **TypeMeta & ObjectMeta**: Standardized metadata for all objects
- **Unstructured**: Map-based dynamic object representation (the critical bridge)

### **2. CRD Definition: apiextensions-apiserver**
- **Internal Hub Types** (types.go): Core CustomResourceDefinition definitions
- **External v1 Types** (v1/types.go): Wire protocol representation
- **Dynamic HTTP Handler** (customresource_handler.go): Routes requests to per-CRD storage
- **etcd Storage** (etcd.go): Stores CRs as Unstructured objects
- **Schema Validation**: Enforces CRD constraints on incoming data

### **3. Client-Side Access: client-go**
- **Typed Clientset & Informers**: For built-in APIs (strongly-typed code generation)
- **Dynamic Client**: For CRDs (operates on Unstructured without Go types)
- **SharedIndexInformer & Listers**: Local in-memory caching with watch-based updates
- **DynamicSharedInformerFactory**: Creates informers for any GVR at runtime

### **4. The Bridge: Unstructured + Scheme**
- Unstructured enables schema-less storage and dynamic client access
- Scheme maintains serialization/deserialization without compiled types
- GVK drives type detection and routing throughout the stack
