Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### (a) Conflict on Apply\n\n**Scenario:** Manager tries to apply a field already owned by another manager.\n\n**Detection:** StructuredMergeManager.update() compares incoming changes against current managed fields.\n\n**Behavior:**\n- If `force=false`: Return 409 Conflict\n- If `force=true`: Steal ownership (idempotent if repeated)\n\n**Mitigation Strategies:**\n- Clients use exponential backoff + retry\n- Coordinate manager responsibilities via design or CRD field annotations\n- Use ApplySet feature (KEP-3659) to declare declarative resource sets and auto-prune\n\n#### (b) Large FieldsV1 Serialization\n\n**Scenario:** Objects with 1000s of fields generate large FieldsV1.Raw binaries.\n\n**Observed:** FieldsV1 grows with field count but is more compact than JSON. In practice, metadata.managedFields can reach 100KB+ on complex CRD instances.\n\n**Mitigation:**\n- `managedFieldstest.SetupFieldCapManager()` caps managers to avoid unbounded growth\n- Older managers are auto-pruned if > 16 exist (AtMostEveryFewMutations strategy)\n- Comparison-based field detection falls back to heuristics if encoding fails\n\n#### (c) Type Conversion Errors\n\n**Scenario:** TypeConverter can't convert object to TypedValue (missing schema, version mismatch, unstructured data).\n\n**Behavior:**\n- For typed objects: Schema lookup fails \u2192 Error during Apply\n- For CRDs: Comparison-based field detection used (field tracking without structured merge)\n- For unstructured: Fallback to JSON comparison\n\n**Mitigation:**\n- CRDs use NewDefaultCRDFieldManager() which handles unstructured safely\n- Errors are returned to client; retrying won't help without schema changes\n\n#### (d) Admission Webhook Mutations\n\n**Scenario:** Admission webhook mutates object fields. Unclear if mutation is part of declared intent.\n\n**Behavior:**\n- If webhook is a registered manager: Its mutations are merged into that manager's field set\n- If webhook is NOT a manager: Mutations are attributed to \"system:apiserver\" or owner's manager\n- Comparison tracks which fields changed; reconciliation decides ownership\n\n**Risk:** Webhook can inadvertently claim fields intended for another manager.\n\n**Mitigation:**\n- Document webhook manager identity in admission plugin config\n- Use `metadata.generation` to detect unintended mutations\n- Test with multiple field managers in staging first\n\n#### (e) Three-Way Merge Loops\n\n**Scenario:** Old, new, and current objects all differ, but the merge result is identical to current.\n\n**Behavior:** Comparison detects no effective change; storage update is skipped (optimization).\n\n**Risk:** If comparison is incorrect, changes get silently dropped.\n\n**Mitigation:**\n- Extensive unit tests cover three-way merge edge cases\n- Admitters can compare before/after; webhooks see intended mutations\n- Audit logs record patch intent\n\n### 4.2 Design Limits\n\n| Limit | Enforced By | Consequence |\n|-------|-------------|-------------|\n| Max patch size | API server (etcd proto) | Requests > ~2MB rejected |\n| Max managers per object | `CapManagers()` (default 16) | Oldest auto-pruned when exceeded |\n| Max field count | None explicit | FieldsV1 grows with cardinality |\n| Max FieldsV1.Raw size | ~None (soft via manager cap) | Large objects \u2192 larger API responses |\n| Patch retry limit | Client (kubectl: 5 retries) | Conflicts after retries \u2192 user error |\n| Field path depth | Structured merge diff | Practical limit ~100 nested levels |\n\n### 4.3 Design Tradeoffs\n\n#### (a) Dense FieldsV1 Encoding vs. Human Readability\n\n**Tradeoff:**\n- **Chosen:** Dense binary encoding in FieldsV1.Raw (bitset)\n- **Alternative:** Human-readable JSON representation\n\n**Rationale:**\n- Comparison and merge operations need fast bitset operations (union, intersection)\n- Etcd storage savings (hundreds of bytes \u2192 tens of bytes for medium objects)\n- Reduces network traffic in API responses\n- Humans view via `kubectl get -o jsonpath='{.metadata.managedFields}'`, not raw\n\n**Cost:**\n- Debugging requires knowledge of fieldpath package\n- Inspection tools (like `fieldpath.Set.String()`) needed for readability\n\n#### (b) Manager-Based Ownership vs. Field-Level Locking\n\n**Tradeoff:**\n- **Chosen:** Manager claims sets of fields (coarse-grained)\n- **Alternative:** Per-field locks with fine-grained reader/writer tracking\n\n**Rationale:**\n- Simpler model: \"manager X owns Y\" instead of \"field F has readers R, writer W\"\n- Matches organizational structure (kubectl, controller, operator = managers)\n- Conflicts are rare in practice; granularity sufficient for most workloads\n- Storage overhead linear in manager count, not field count\n\n**Cost:**\n- One \"rogue\" field in a manager's set can block other managers\n- Mitigation: Clear manager semantics and documentation\n\n#### (c) Force=true Overwrites vs. Transactional Safety\n\n**Tradeoff:**\n- **Chosen:** force=true silently overwrites other managers' claims\n- **Alternative:** Require explicit manager acknowledgment for ownership transfer\n\n**Rationale:**\n- Operators need escape hatch for stuck states (e.g., deleted manager lingering in metadata)\n- In practice, force=true used when admin decides to take control\n- Safe if users understand implications; documented in kubectl help\n\n**Cost:**\n- Aggressive force=true can cause data loss if multiple managers race\n- Mitigation: Audit logs, RBAC restrictions on force flag\n\n#### (d) Server-Side Merge vs. Client-Side Library\n\n**Tradeoff:**\n- **Chosen:** Core merge logic on server; clients invoke via HTTP PATCH\n- **Alternative:** Client-side SDKs compute merge before sending\n\n**Rationale:**\n- Server is single source of truth for conflict detection\n- Prevents race conditions (client can't know server state between request construction and send)\n- Stateless API server (no session management needed)\n- Enables kubectl, Go client, Python client, etc. to share same semantics\n\n**Cost:**\n- Extra round-trip for conflict retries\n- Server carries heavier load (merge CPU)\n- Mitigation: Conflict retries are rare; CPU acceptable at scale\n\n---\n\n## 5. # Kubernetes Server-Side Apply and ApplyConfigurations: A Deep Guide

## Executive Summary

The Kubernetes Server-Side Apply (SSA) system enables safe, declarative configuration management with built-in conflict resolution and field ownership tracking. ApplyConfigurations are type-safe, fluent builders generated for each Kubernetes API type, bridging client applications to the SSA workflow. Together, these components solve the classic three-way merge problem while handling concurrent mutations from multiple agents (users, controllers, operators) without manual conflict resolution.

---

## 1. Subsystem Purpose, Boundaries, and Upstream/Downstream Dependencies

### Purpose

The Server-Side Apply subsystem tracks **field ownership** across multiple managers (kubectl users, controllers, webhooks) and provides **conflict detection and resolution**. Rather than keeping a "last applied configuration" annotation on the client side, SSA maintains structured metadata in the resource's `metadata.managedFields` array. This enables:

- **Declarative intent separation**: Different managers can own different fields without clobbering each other.
- **Safe concurrent updates**: Conflict detection alerts users when multiple managers try to claim the same field.
- **Gradual adoption**: Controllers can transition from imperative updates to declarative management without disruption.

ApplyConfigurations provide the **client-side counterpart**: generated, type-safe helper builders that construct partial resource snapshots to send as apply patches (instead of raw JSON manifests).
