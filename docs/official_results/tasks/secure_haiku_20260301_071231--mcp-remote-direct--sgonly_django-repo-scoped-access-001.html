## Bug Report

**Reported by:** QA Team
**Severity:** Medium
**Environment:** Django admin site with models using ForeignKey relationships

When a model's admin page is configured to let users filter by a related model (via a ForeignKey), the filter widget appears in the sidebar even when there are zero related objects in the database. This creates a confusing user experience:

1. The filter section title appears in the sidebar with no useful choices
2. Only a generic "All" option is shown — no actual items to filter by
3. Clicking "All" triggers an unnecessary page reload
4. Users have filed support tickets asking why an empty filter exists

**Steps to reproduce:**
1. Register a `ModelAdmin` with `list_filter` set to include a ForeignKey field
2. Ensure the related model's database table has no rows
3. Visit the admin changelist page
4. Observe the filter sidebar — the empty filter is still rendered

**Expected behavior:** When there are no meaningful choices available for a filter, the filter should not appear in the sidebar at all.

## Summary

I have successfully fixed the Django admin filter sidebar issue where empty filters were still appearing in the sidebar even when there were no choices available.

### The Problem
When a model's admin page configured a filter for a field with no values in the database (e.g., a ForeignKey to an empty related model), the filter widget would still appear in the sidebar, but only with an "All" option, which provided no meaningful filtering capability.

### Root Cause
The `AllValuesFieldListFilter` class (the fallback filter applied to any field without a more specific filter type) inherited `has_output()` from its parent `FieldListFilter`, which unconditionally returned `True`. This prevented the admin's filter rendering logic from hiding the empty filter.

### The Fix
I added a `has_output()` method override to `AllValuesFieldListFilter`:

```python
def has_output(self):
    return len(self.lookup_choices) > 0
```

This method checks if there are any actual choices available. When `lookup_choices` is empty (no values in the database), the filter won't be rendered.

### Changes Made
- **File**: `django/contrib/admin/filters.py`
- **Location**: Lines 584-585 in the `AllValuesFieldListFilter` class
- **Change**: Added 3 lines (the method definition and return statement)

✅ Follows existing code patterns (matches `SimpleListFilter.has_output()`)

Empty related-field filters (when a field has no values in the database) were still appearing in the admin sidebar, creating a poor user experience. Users would see filter options with only an "All" choice, even when there were zero meaningful filter choices.

The `AllValuesFieldListFilter` class (the fallback filter used for any field without a more specific filter) inherited the `has_output()` method from its parent class `FieldListFilter`, which unconditionally returns `True`.
