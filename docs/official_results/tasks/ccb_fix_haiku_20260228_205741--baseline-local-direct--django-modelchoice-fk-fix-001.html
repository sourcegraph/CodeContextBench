## Bug Report

**Reported by:** Django Forum user
**Severity:** High
**Django version:** Current main branch

When a model has a ForeignKey that references a non-primary-key field (using the `to_field` parameter), forms generated from that model display the wrong selected option in dropdown (`<select>`) widgets.

**Steps to reproduce:**

```python
# Models
class Country(models.Model):
    code = models.CharField(max_length=2, unique=True)  # e.g., "US", "GB"
    name = models.CharField(max_length=100)

class City(models.Model):
    name = models.CharField(max_length=100)
    country = models.ForeignKey(Country, to_field="code", on_delete=models.CASCADE)

# Form for City — the country dropdown shows wrong selection
form = CityForm(instance=existing_city)
```

The `<option>` elements in the dropdown are generated with `value="US"`, `value="GB"`, etc. (using the `code` field as expected). But when rendering a form with an existing City instance, the initial selected value resolves to the Country's numeric primary key (integer ID) instead of the `code` value. This causes a mismatch — the form renders with no option selected, or the wrong option highlighted.

**Expected behavior:** When a ForeignKey uses `to_field="code"`, the form field should use the `code` field consistently — both for generating `<option>` values AND for determining which option is currently selected.

**Observed behavior:** Option values use `code`, but the selected value comparison uses the primary key, causing a mismatch.

# Bug Fix: Form Dropdown Showing Wrong Selection for Non-PK Foreign Keys

## Problem Description

When a Django ModelForm has a ForeignKey field that references a non-primary-key field (using the `to_field` parameter), the form dropdown shows no selected option or the wrong option when rendering an existing instance.

## Root Cause

The issue is in the `model_to_dict()` function in `django/forms/models.py`.

When extracting the initial value for a ForeignKey field:
1. `model_to_dict()` calls `f.value_from_object(instance)` for each field
2. For ForeignKey fields, `value_from_object()` returns `getattr(obj, self.attname)` which is the primary key ID
3. This primary key value is passed to the form as initial data
4. When rendering, the ModelChoiceField receives this primary key value
5. The `prepare_value()` method receives a primary key (not a model instance), so it returns it as-is
6. The widget tries to find a choice with value="<primary_key>", but all choices use values from the to_field
7. No option matches, so nothing appears selected

## Solution

Modify the `model_to_dict()` function to return the value of the `to_field` instead of the primary key when a ForeignKey has a non-default `to_field`.

### Changes to django/forms/models.py

In the `model_to_dict()` function (around line 99-121), add logic to check if a field is a ForeignKey/OneToOneField with a non-PK `to_field`, and if so, extract the `to_field` value from the related object:

```python
def model_to_dict(instance, fields=None, exclude=None):
    """
    Return a dict containing the data in ``instance`` suitable for passing as
    a Form's ``initial`` keyword argument.

    ``fields`` is an optional list of field names. If provided, return only the
    named.

    ``exclude`` is an optional list of field names. If provided, exclude the
    named from the returned dict, even if they are listed in the ``fields``
    argument.

        if not getattr(f, "editable", False):
            continue
        if fields is not None and f.name not in fields:
            continue
        if exclude and f.name in exclude:
            continue

        # value of the to_field instead of the primary key to match what
```

# Refined Solution: Fix Form Dropdown for Non-PK Foreign Keys

## Key Change: Move Logic to ForeignKey.value_from_object()

Instead of modifying `model_to_dict()`, override the field's `value_from_object()` method in ForeignKey/OneToOneField to return the to_field value directly.

### Location
**File**: `django/db/models/fields/related.py`
**Class**: ForeignKey
**Method**: `value_from_object()`

### Implementation

In `/workspace/django/db/models/fields/related.py`, modify the ForeignKey class to override `value_from_object()`:

```python
class ForeignKey(ForeignObject):
    # ... existing code ...
    
    def value_from_object(self, obj):
        """
        Return the value of this field in the given model instance.
        For fields with a non-PK to_field, returns the to_field value
        instead of the primary key.
        """
        # If this FK points to a non-PK field, return that field's value
        if self.remote_field.field_name != self.remote_field.model._meta.pk.name:
            related_obj = getattr(obj, self.name, None)
            if related_obj is None:
                return None
            return getattr(related_obj, self.remote_field.field_name, None)
        
        # Default behavior: return the FK's database value (the PK)
        return super().value_from_object(obj)
```

### Advantages of This Approach

1. **Single Source of Truth**: Field encapsulates its own value extraction logic
2. **Auto-fixes model_to_dict()**: No changes needed to `model_to_dict()` - it uses `value_from_object()` 
3. **Works for all consumers**: Any code calling `value_from_object()` gets correct behavior
4. **Follows existing patterns**: Consistent with Django's design (field methods handle field logic)
5. **No N+1 Query Impact**: The related object was already being accessed; we're just extracting a different field value
6. **Better Encapsulation**: Logic is in the field class, not in a generic utility

### Why This Avoids the Agent Concerns

**Agent 1 (Code Reuse)**: ✅ 
- Centralizes the to_field extraction logic in one place
- Can be reused by all callers of `value_from_object()`

**Agent 2 (Code Quality)**: ✅

- Overrides `value_from_object()` properly in subclass
