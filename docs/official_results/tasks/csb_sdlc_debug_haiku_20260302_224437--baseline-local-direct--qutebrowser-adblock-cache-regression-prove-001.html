# Bug Investigation: Application Crashes When Ad-Blocker Cache Is Corrupted

**Repository:** qutebrowser/qutebrowser
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

When qutebrowser starts up and attempts to load the ad-blocker filter cache, the application crashes if the cache file is corrupted or contains invalid data.

The expected behavior is that the application handles corrupted cache files gracefully — it should log an error, discard the invalid cache, and continue running normally (allowing the user to browse without ad blocking until the cache is rebuilt).

Instead, the deserialization error propagates as an unhandled exception, causing a crash before the browser window even appears. Users who encounter a corrupted cache file cannot use qutebrowser at all until they manually delete the cache file.

This is particularly problematic because cache corruption can happen for reasons outside the user's control, such as a system crash or disk error during a cache write.

- The test should simulate a corrupted cache file and verify the crash occurs

## Summary

I have successfully investigated the qutebrowser ad-blocker cache corruption bug and created a regression test. Here's what I found:

### **Root Cause: Exception Type Mismatch**

**Location:** `/app/qutebrowser/components/braveadblock.py` (lines 214-222)

The `read_cache()` method attempts to catch `ValueError`:
```python
try:
    self._engine.deserialize_from_file(str(self._cache_path))
except ValueError as e:
    if str(e) != "DeserializationError":
        raise
```

**However**, the adblock library actually raises `adblock.DeserializationError`, which is **NOT** a subclass of `ValueError`. Therefore:

1. When cache is corrupted, `deserialize_from_file()` raises `adblock.DeserializationError`
2. The `except ValueError` clause does NOT catch it
3. The exception propagates as unhandled
4. The application crashes during startup before showing the browser window
5. Users cannot use qutebrowser until they manually delete the cache file

### **Regression Test**

I've created a comprehensive regression test at `/workspace/regression_test.py` that:

✅ **Test 1:** Verifies that `adblock.DeserializationError` is raised (not ValueError)
✅ **Test 2:** Demonstrates that `BraveAdBlocker.read_cache()` crashes on corrupted cache
✅ **Test 3:** Tests empty cache file handling  
✅ **Test 4:** Tests random binary data in cache file

# Ad-Blocker Cache Corruption Bug Investigation

## Summary

I have successfully identified and documented the root cause of the qutebrowser ad-blocker cache corruption crash, and created a comprehensive regression test.

## Root Cause

**Location:** `/app/qutebrowser/components/braveadblock.py`, line 216-220 in the `read_cache()` method

**The Bug:**
```python
try:
    self._engine.deserialize_from_file(str(self._cache_path))
except ValueError as e:
    if str(e) != "DeserializationError":
        raise
    message.error("Reading adblock filter data failed (corrupted data?). "
                  "Please run :adblock-update.")
```

**Why it's broken:**
1. When the cache file is corrupted, `adblock.Engine.deserialize_from_file()` raises `adblock.DeserializationError`
2. `adblock.DeserializationError` is **NOT** a subclass of `ValueError`
3. Therefore, the `except ValueError` clause does NOT catch it
4. The exception propagates as unhandled, crashing the application during startup
5. The user cannot use qutebrowser until they manually delete the cache file

**Proof:**
```python
>>> import adblock
>>> issubclass(adblock.DeserializationError, ValueError)
False
```

## Expected vs Actual Behavior

**Expected Behavior:**
- Application handles corrupted cache gracefully
- Logs an error message
- Continues running with ad-blocker disabled until cache is rebuilt
- User can still browse the web

**Actual Behavior:**
- Application crashes with unhandled `DeserializationError`
- User cannot use qutebrowser at all
- Must manually delete cache file: `~/.config/qutebrowser/adblock-cache.dat`
