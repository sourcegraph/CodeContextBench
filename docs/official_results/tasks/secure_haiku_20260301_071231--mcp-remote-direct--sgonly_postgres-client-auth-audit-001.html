# big-code-pg-sec-001: PostgreSQL Client Authentication Pipeline Security Analysis

## Task

Trace the PostgreSQL client authentication pipeline from the point a TCP connection is accepted through HBA (Host-Based Authentication) rule matching, password/SCRAM-SHA-256 verification, and role validation to session establishment. Identify all entry points where untrusted client data enters the authentication subsystem, map the complete data flow through each verification stage, and analyze the security properties of the authentication chain.

## Context

- **Repository**: github.com/sg-evals/postgres--5a461dc4 (mirror of github.com/sg-evals)/postgres--5a461dc4 (mirror of postgres/postgres) (C, ~1.5M LOC)
- **Category**: Security Analysis
- **Difficulty**: hard
- **Subsystem Focus**: `src/backend/libpq/` (auth.c, auth-scram.c, auth-sasl.c, crypt.c, hba.c) and `src/backend/utils/init/` (postinit.c, miscinit.c)

## Requirements

1. Identify all entry points where untrusted data enters the authentication subsystem (startup packet, password packets, SASL tokens)
2. Trace data flow from each entry point through transformations to sensitive operations (role lookup, password comparison, session establishment)
3. Map the HBA configuration matching pipeline (`load_hba` -> `check_hba` -> `hba_getauthmethod`)
4. Trace the password verification chain through both plaintext (`CheckPasswordAuth`) and challenge-response (`CheckPWChallengeAuth`) paths
5. Trace the SCRAM-SHA-256 exchange through the SASL framework (`auth-sasl.c`) to the SCRAM implementation (`auth-scram.c`)
6. Identify all files in the attack surface and document existing mitigations

Authentication Architecture\n\nPostgreSQL implements a modular authentication system where the authentication method is determined by the pg_hba.conf configuration file matching against the client's IP address, port, database, and user. The authentication pipeline has multiple entry points where untrusted client data enters:\n\n1. **Startup packet parsing**: User name, database name, application name\n2. **Password reception**: Plaintext password for `password` method\n3. **Challenge response**: MD5-hashed password for `md5` method\n4. **SASL/SCRAM tokens**: Binary SASL protocol messages for `scram-sha-256` method\n\nEach entry point has different validation and processing:\n- **Startup packet**: Validated for length and structure, names truncated to NAMEDATALEN\n- **Plaintext password**: No structural validation (arbitrary string)\n- **MD5 response**: Expected to be MD5 hash (32 hex chars)\n- **SCRAM tokens**: Binary SASL protocol with format validation in SCRAM implementation\n\n### 2. Role Validation\n\nThe role lookup happens via `get_role_oid(port->user_name, true)` in `check_hba()` with `missing_ok=true`. This is critical because:\n\n1. It allows the user_name to be validated against the pg_authid catalog\n2. However, the `missing_ok=true` flag means **non-existent users do not cause immediate failure**\n3. This is intentional to support user enumeration prevention through dummy authentication\n\nThe security model relies on never revealing whether a user exists or whether a password is correct for a non-existent user.\n\n### 3. Password Verification Mechanisms\n\n#### Plaintext Authentication (uaPassword)\n- **Flow**: Client sends password in plaintext \u2192 recv_password_packet \u2192 plain_crypt_verify\n- **Verification**:\n  - For SCRAM-SHA-256 passwords: Client password is hashed using stored salt and iterations, compared with StoredKey\n  - For MD5 passwords: Client password is hashed with user name, compared with stored hash\n- **Risk**: Password sent in plaintext over network (only safe if SSL/TLS encrypted)\n\n#### MD5 Challenge-Response (uaMD5)\n- **Flow**: Server sends random salt \u2192 Client responds with MD5(MD5(password + username) + salt) \u2192 md5_crypt_verify\n- **Verification**: Server computes same hash and compares\n- **Advantage**: Password not sent in plaintext (but salt is public)\n- **Disadvantage**: MD5 is cryptographically weak; vulnerability in 1-iteration MD5\n\n#### SCRAM-SHA-256 Challenge-Response (uaSCRAM)\n- **Flow**: Multi-message SASL exchange with HMAC-SHA-256 based proof\n- **Verification**:\n  - Server sends iterations, salt, ServerSignature\n  - Client computes SaltedPassword = PBKDF2-SHA-256(password, salt, iterations)\n  - Client sends ClientProof = ClientKey XOR ClientSignature\n  - Server verifies ClientProof using stored ClientKey\n- **Advantages**:\n  - Password not exposed (uses key derivation)\n  - Salted and iterated hashing (PBKDF2)\n  - Two-way authentication (server provides ServerSignature)\n  - Channel binding support (SCRAM-SHA-256-PLUS)\n\n### 4. User Enumeration Prevention\n\nPostgreSQL implements \"dummy authentication\" to prevent user enumeration attacks:\n\n**SCRAM Implementation** (auth-scram.c:scram_init lines 323-329):\n```c\nif (!got_secret) {\n    mock_scram_secret(state->port->user_name, &state->hash_type,\n                      &state->iterations, &state->key_length,\n                      &state->salt,\n                      state->StoredKey, state->ServerKey);\n    state->doomed = true;\n}\n```\n\n**MD5/SCRAM Selection** (auth.c:CheckPWChallengeAuth lines 844-845):\n```c\nif (!shadow_pass)\n    pwtype = Password_encryption;  // Use configured default\nelse\n    pwtype = get_password_type(shadow_pass);\n```\n\n**Effect**:\n- Authentication always proceeds with valid-looking (but incorrect) credentials\n- Authentication fails identically for non-existent users and wrong passwords\n- No timing differences reveal user existence\n- An attacker cannot enumerate valid usernames\n\n### 5. HBA Rule Matching\n\nThe HBA matching pipeline (hba.c:check_hba):\n\n1. **Connection type matching**: Check if connection is local (Unix socket), host, hostssl, or hostnossl\n2. **IP address matching**: If remote, verify client IP matches HBA rule's address/netmask using check_ip()\n3. **Hostname matching**: Optional reverse DNS lookup and hostname matching\n4. **Database matching**: Check if client's requested database matches HBA rule (or wildcard)\n5. **Role matching**: Check if authenticated role matches HBA rule (or wildcard)\n6. **First match wins**: Rules are processed in order; first match determines auth method\n\n**Security properties**:\n- Rules are configuration-based and static (loaded at startup or via SIGHUP reload)\n- Rules are evaluated in file order, allowing defaults and overrides\n- Implicit reject if no rule matches (uaImplicitReject)\n- Explicit reject allows security rules to be clear\n\n### 6. Data Validation at Entry Points\n\n#### Startup Packet (ProcessStartupPacket)\n- **Validation**: Length check (536 <= len <= MAX_STARTUP_PACKET_LENGTH), protocol version validation\n- **Truncation**: User name and database name truncated to NAMEDATALEN (63 bytes)\n- **Application name**: Cleaned via `pg_clean_ascii()` to prevent log injection\n- **GUC options**: Stored for later processing but not yet validated\n\n#### Password Packet (recv_password_packet)\n- **Validation**: Message type check, packet length validation, empty password rejection\n- **No encoding validation**: Passwords are treated as raw bytes (encoding determined later)\n- **Length limit**: PG_MAX_AUTH_TOKEN_LENGTH (typically 1GB or large value)\n\n#### SASL Tokens (CheckSASLAuth)\n- **Length limit**: mech->max_message_length (SCRAM uses 262144 bytes)\n- **Format validation**: Delegated to SASL mechanism implementation\n\n### 7. Vulnerability Classes and Mitigations\n\n#### User Enumeration (Timing-Based)\n- **Attack**: Measure response time differences to infer user existence\n- **Mitigation**: Dummy authentication - invalid users always proceed through full authentication exchange with mock credentials\n- **Residual risk**: SCRAM iteration count and salt generation timing could leak user existence with extreme precision\n- **Impact**: Network timing is dominated by random variation; differential timing < 1ms is hard to exploit\n\n#### User Enumeration (Information Leakage)\n- **Attack**: Inspect error messages to determine if user exists\n- **Mitigation**: Error messages don't differentiate between \"user doesn't exist\" and \"wrong password\"\n- **Implementation**: Errors logged to server log (not sent to client) via logdetail parameter\n\n#### Password Exposure in Transit\n- **Attack**: Plaintext password captured on network\n- **Mitigation**: SSL/TLS encryption required (uaPassword method highly discouraged without SSL)\n- **Better alternatives**: SCRAM-SHA-256 (challenge-response, password never sent)\n\n#### Dictionary Attacks / Brute Force\n- **Attack**: Attacker tries many passwords\n- **Mitigation**: SCRAM-SHA-256 uses PBKDF2 with iteration count (server-side rate limiting not implemented)\n- **Residual risk**: PostgreSQL doesn't implement connection rate limiting or attempt counting at authentication stage\n- **Recommended**: Configure firewall/proxy to rate-limit authentication attempts\n\n#### MITM Attack (Password Exposure)\n- **Attack**: Attacker intercepts MD5 challenge-response exchange\n- **Risk**: MD5 is weak; reversing password from hash may be feasible for weak passwords\n- **Mitigation**: SCRAM-SHA-256 uses salted PBKDF2 (iteration count \u00d7 password strength required)\n- **Better mitigation**: Enforce SSL/TLS for all connections\n\n#### Channel Binding (TLS Hijacking)\n- **Attack**: Attacker hijacks TLS session after authentication\n- **Mitigation**: SCRAM-SHA-256-PLUS includes channel binding (tls-server-end-point)\n- **Implementation**: Binds SCRAM authentication to TLS certificate (prevents MitM)\n- **Requirement**: Must use SCRAM-SHA-256-PLUS variant (not default)\n\n#### Role Privilege Escalation\n- **Attack**: Authenticated user tries to access another role's data/privileges\n- **Mitigation**: Session role is set to authenticated user (set_authn_id); authorization checks use this role\n- **Scope**: Authentication only verifies user identity; authorization is separate (GRANT/REVOKE)\n\n### 8. Existing Mitigations Summary\n\n| Vulnerability | Mitigation | Strength |\n|---|---|---|\n| User enumeration | Dummy authentication (SCRAM, MD5) | Strong |\n| Password in transit | SSL/TLS encryption + SCRAM | Strong if TLS enforced |\n| Weak password hashing | PBKDF2 + salt + iterations (SCRAM) | Strong |\n| MD5 weakness | Deprecated but still supported | Weak (legacy) |\n| Plaintext password method | Deprecated but configurable | Weak (legacy) |\n| MITM / channel hijacking | SCRAM-SHA-256-PLUS + channel binding | Strong if PLUS variant used |\n| Timing attacks | Constant-time HMAC comparison | Strong (implementation detail) |\n| Log injection via password | Passwords not logged | Strong |\n| Log injection via application_name | Sanitized with pg_clean_ascii() | Strong |\n| Invalid SCRAM secret | Graceful fallback to dummy | Strong |\n| User doesn't exist | Dummy authentication (no error) | Strong |\n\n### 9. Attack Scenarios\n\n#### Scenario 1: User Enumeration Attack\n**Attacker goal**: Determine valid usernames\n\n**Attack attempt**:\n1. Connect to PostgreSQL with user \"admin\"\n2. Send random password\n3. Check if error message differs from connecting with \"nonexistent_user\"\n\n**Defense**:\n- Error messages don't differentiate: both receive \"authentication failed\"\n- Authentication always proceeds with valid-looking (but incorrect) credentials\n- Timing is identical (within network variation)\n- **Result**: Attack fails; attacker cannot enumerate users\n\n#### Scenario 2: Brute Force Password Attack\n**Attacker goal**: Guess password for known user\n\n**Attack attempt**:\n1. Connect with user \"postgres\"\n2. Try 10,000 passwords rapidly\n\n**Defense**:\n- PostgreSQL doesn't implement built-in rate limiting\n- Each password attempt requires full SCRAM exchange (computational cost on server)\n- SCRAM uses PBKDF2 with ~4096 iterations (default)\n- Network latency dominates\n- **Recommended mitigation**: Firewall/proxy implements rate limiting, fail2ban, or PgBouncer\n\n#### Scenario 3: MITM Password Hijacking\n**Attacker goal**: Capture password for offline cracking\n\n**Attack attempt**:\n1. Position MITM between client and server\n2. Intercept authentication exchange\n\n**Defense**:\n- SCRAM-SHA-256: Password not transmitted; hash of password transmitted\n- SCRAM-SHA-256-PLUS: Binds to TLS certificate; MITM causes different channel binding value\n- **If SSL/TLS not enforced**: Plaintext MD5 challenge-response vulnerable\n- **Recommended mitigation**: Require SSL/TLS for all connections (force_ssl=on)\n\n#### Scenario 4: Timing-Based User Enumeration\n**Attacker goal**: Detect user existence via response timing\n\n**Attack attempt**:\n1.
