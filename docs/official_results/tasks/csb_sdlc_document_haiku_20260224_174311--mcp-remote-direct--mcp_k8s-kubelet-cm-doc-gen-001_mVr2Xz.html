Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### CPU Allocation Conflicts\n- **Scenario**: Pod with `requests.cpu` exceeds allocatable non-reserved CPUs\n- **Detection**: cpuManager.Allocate() fails during pod admission\n- **Behavior**: Pod rejected; user sees \"Insufficient cpu\" error\n- **Recovery**: User must reduce pod CPU request or add more nodes\n\n#### Memory Allocation Failures\n- **Scenario**: Pod with NUMA memory affinity but all NUMA nodes exhausted\n- **Detection**: memoryManager.Allocate() fails during pod admission\n- **Behavior**: Pod rejected; user sees \"Insufficient memory\" error\n- **Recovery**: Similar to CPU; add capacity or reduce requests\n\n#### Device Plugin Failures\n- **Scenario**: Device plugin crashes or becomes unresponsive\n- **Detection**: Health check timeout in device manager\n- **Behavior**: Device manager marks plugin as unhealthy; stops allocating resources\n- **Recovery**: Plugin must be restarted (usually by container orchestration)\n\n#### Cgroup v2 Kernel Incompatibility\n- **Scenario**: Kernel doesn't support required cgroup v2 features\n- **Example**: root `cpu.stat` only in kernel 5.8+\n- **Detection**: At container manager startup\n- **Behavior**: Warning issued (see `container_manager.go:57-58`); some enforcement may not work\n- **Impact**: CPU limits may not be enforced correctly\n\n#### State File Corruption\n- **Scenario**: `cpu_manager_state` or `memory_manager_state` corrupted\n- **Detection**: JSON unmarshal failure during manager startup\n- **Behavior**: Manager fails to start; kubelet refuses to start (safe fail)\n- **Recovery**: Admin must delete corrupted state files; kubelet will rebuild state\n\n#### Node Reboot with Live Containers\n- **Scenario**: Kubelet restarted but containers still running\n- **Recovery**: `buildContainerMapAndRunningSetFromRuntime()` rebuilds state from runtime\n- **Limitation**: If state file missing, kubelet may re-allocate same resources \u2192 conflicts\n- **Mitigation**: State persisted to disk; kubelet recovers from it\n\n### 4.2 Limits and Constraints\n\n#### Maximum NUMA Nodes\n- **Limit**: 8 NUMA nodes (topology manager hard limit)\n- **Reason**: Exponential state explosion in topology hint generation\n- **Code**: `topologymanager.go:41`\n- **Impact**: Machines with >8 NUMA nodes cannot use topology manager\n\n#### Maximum Concurrent Device Plugins\n- **Limit**: No explicit limit; each plugin requires dedicated goroutine\n- **Risk**: Resource exhaustion if hundreds of plugins registered\n- **Mitigation**: Practical limit ~20-30 plugins before CPU overhead significant\n\n#### PID Limits per Pod\n- **Enforced via**: `--pod-pids-limit` flag (default: none)\n- **Mechanism**: cgroup v1 pids.max or cgroup v2 pids.max\n- **Behavior**: Once hit, pod cannot spawn new processes; containers become unresponsive\n\n#### Memory Overcommit\n- **By Design**: cgroups allow setting memory.limit > node capacity\n- **Risk**: OOM killer may evict any container regardless of QoS\n- **Mitigation**: Memory manager and admissioncontroller prevent over-allocation\n- **Failure Mode**: If limits set incorrectly, random containers killed\n\n#### CPU CFS Quota Precision\n- **Granularity**: ~1 millisecond (cfs_period_us / cfs_quota_us)\n- **Issue**: Very small CPU requests (<100m) may round to 0 quota\n- **Workaround**: `cpu-cfs-quota-period` flag (default 100ms); larger = coarser\n\n### 4.3 Design Tradeoffs\n\n#### Tradeoff 1: Pre-allocation vs Post-allocation\n- **Pre-allocation** (used): CPU & memory allocated at pod admission\n  - **Pro**: Guaranteed availability; prevents overcommit\n  - **Con**: Fragmentation; reserved but unused resources\n  - **Alternative**: Post-allocation on container start would reduce fragmentation but lose guarantees\n\n#### Tradeoff 2: State Persistence\n- **Current**: State files (`cpu_manager_state`) persisted to disk\n  - **Pro**: Survives kubelet restarts; no CPU leaks\n  - **Con**: Disk I/O overhead; state drift if runtime inconsistent\n  - **Alternative**: In-memory only (faster) but loses recovery capability\n\n#### Tradeoff 3: Topology Manager Policy Strictness\n- **Policies Available**:\n  - **none**: No enforcement; allows suboptimal affinity\n  - **best-effort**: Tries optimal but allows suboptimal as fallback\n  - **restricted**: Rejects suboptimal allocations \u2192 fails pod admission\n  - **single-numa-node**: Forces all resources on one node or reject\n- **Tradeoff**: Stricter = better latency but more pod rejections\n\n#### Tradeoff 4: Cgroup Version Complexity\n- **v1**: Multiple hierarchies; per-resource config; complex but proven\n- **v2**: Unified hierarchy; simpler config; but newer kernel requirement\n- **Decision**: Support both; auto-detect; v1 being deprecated (warnings issued)\n\n#### Tradeoff 5: Dynamic Resource Allocation (DRA)\n- **Traditional**: Static requests in pod spec; known at admission time\n- **DRA**: Dynamic claims; allocated at runtime; more flexible\n- **Tradeoff**: Flexibility vs complexity; DRA adds separate manager and checkpoint overhead\n\n---\n\n## 5. Operational Implications\n\n### 6.1 Performance Considerations\n\n#### CPU Manager Overhead\n- **Reconciliation Frequency**: Default 5 seconds (configurable)\n- **Cost**: Per-container UpdateContainerResources() RPC to runtime\n- **Impact**: With 1000 containers, reconciliation every 5s = 200 RPCs/sec to runtime\n- **Optimization**: Increase reconciliation period for large clusters; skip if no changes\n\n#### Memory Manager State Growth\n- **State File Size**: O(pods \u00d7 containers)\n- **Typical**: 1KB per container; 1000 pods = ~10MB state file\n- **Disk I/O**: Persisted on every allocate/remove; can stress storage on frequent pod churn\n- **Mitigation**: Use fast storage; consider in-memory caches\n\n#### Device Manager Checkpointing\n- **Checkpoint Frequency**: On every device allocation/deallocation\n- **Risk**: With many device plugins, checkpoint overhead becomes significant\n- **Monitoring**: Watch kubelet CPU usage during pod creation surge\n\n#### Topology Manager Hint Complexity\n- **Exponential in NUMA nodes**: 2^NUMA_NODES possibilities to evaluate\n- **Impact**: With 8 NUMA nodes, generating hints for 3-pod resource types (CPU, memory, device) can take 10-50ms\n- **Mitigation**: Reduce NUMA nodes or use \"none\" policy in small topologies\n\n### 6.2 Reliability and Recovery\n\n#### Graceful Degradation on Missing State Files\n- **Behavior**: If CPU/memory manager state corrupted, kubelet refuses to start (safe fail)\n- **Recovery**: Admin must delete state files (`~/.kubelet/cpu_manager_state`, `~/.kubelet/memory_manager_state`)\n- **Consequence**: Next kubelet start rebuilds state from running containers; some allocations may be lost\n- **Best Practice**: Back up state files before kubelet upgrades\n\n#### Device Plugin Failure Handling\n- **Detection**: Health checker timeout (default 30s)\n- **Action**: Device plugin marked unhealthy; stops accepting allocations\n- **User Impact**: New pods cannot request that device; stuck in pending\n- **Recovery**: Restart device plugin or restart kubelet (to clear stale allocations)\n\n#### OOM Killer Behavior\n- **Without QoS classes**: Random container killed when node memory full\n- **With Guaranteed QoS**: Guaranteed pods protected; Burstable/BestEffort killed first\n- **With Memory Manager**: NUMA node local OOM; kills containers on that node only\n- **Visibility**: OOM events appear in pod status and node events\n\n#### Cgroup Enforcement Failures\n- **Symptom**: Container running with >memory.limit (indicates bug or cgroup bug)\n- **Cause**: Kernel may allow short-lived page cache bursts beyond hard limit\n- **Detection**: cAdvisor reports actual vs limit; monitor for discrepancies\n- **Resolution**: Usually requires waiting for OOM killer; no user action\n\n### 6.3 Debugging and Observability\n\n#### Key Metrics\n\n**cAdvisor Metrics** (collected by kubelet):\n```\ncontainer_cpu_cfs_periods_total     // CPU quota cycles\ncontainer_cpu_cfs_throttled_periods_total  // Over-limit throttles\ncontainer_memory_usage_bytes        // Current memory\ncontainer_memory_max_usage_bytes    // Peak memory\n```\n\n**kubelet Metrics** (from `pkg/kubelet/metrics/`):\n```\nkubelet_cpu_manager_cpu_allocation      // CPU allocation status\nkubelet_memory_manager_memory_allocation // Memory allocation\nkubelet_device_manager_allocation_duration // Device allocation latency\nkubelet_topology_manager_admission_duration  // Topology coordination time\n```\n\n#### Logs to Monitor\n\n**CPU Manager**:\n```\n\"Container has pinned cpus\" \u2192 Container got exclusive CPUs\n\"Failed to set CPU affinity\" \u2192 CPU binding failed; check kernel support\n```\n\n**Memory Manager**:\n```\n\"Allocating memory on NUMA node X\" \u2192 Memory allocation info\n\"Memory allocation failed\" \u2192 No available memory on preferred NUMA\n```\n\n**Device Manager**:\n```\n\"Device plugin disconnected\" \u2192 Device plugin process died\n\"Allocate RPC failed\" \u2192 Device plugin returned error\n```\n\n**Topology Manager**:\n```\n\"TopologyAffinityError\" \u2192 Cannot find affinity satisfying all providers\n```\n\n#### PodResources API (Debugging Aide)\n\n**Endpoint**: `/var/lib/kubelet/pod-resources/kubelet.sock`\n\n**Usage**: Query allocated resources for pods:\n```bash\n# List CPUs and devices allocated to pod\ncurl --unix-socket /var/lib/kubelet/pod-resources/kubelet.sock \\\n  'http://localhost/v1/pods' | protoc --decode k8s.io.kubelet.v1.PodResourcesResponse -\n```\n\n**Value**: Verify CPU/memory/device allocations match expectations; identify leaks.\n\n### 6.4 Upgrade and Rollout Considerations\n\n#### State File Format Changes\n- **Risk**: Upgrading kubelet to version with different state format\n- **Mitigation**: Kubelet version must handle old format or fail safe\n- **Best Practice**: Drain node before kubelet upgrade; recreate pods on upgrade\n\n#### cgroup v1 to v2 Migration\n- **Cost**: Must redistribute pods across both cgroup types during transition\n- **Timeline**: Gradual rollout; upgrade kubelet incrementally\n- **Monitoring**: Watch for cgroup version mismatch errors\n\n#### Device Plugin API Version Negotiation\n- **Current**: v1beta1 only; no backward compatibility layer\n- **Risk**: Upgrading kubelet while device plugins are old\n- **Mitigation**: Update device plugins before kubelet (or simultaneously)\n\n#### Pod Eviction During Upgrades\n- **Scenario**: Node marked unschedulable; pods evicted for kubelet upgrade\n- **Impact**: CPU/memory allocations freed; container runtime still enforces old limits\n- **Recovery**: Graceful termination (30s grace period) gives containers time to exit\n- **Failure Mode**: Stuck container \u2192 SIGKILL after grace period\n\n---\n\n## 7. # Kubelet Container Manager Architecture Guide

## 1. Subsystem Purpose and Scope

The **Container Manager (CM)** subsystem is responsible for managing containerized workloads on a Kubernetes node. It serves as the authoritative interface between the kubelet and the underlying container runtime infrastructure, enforcing resource constraints, managing cgroup hierarchies, and coordinating resource allocation across CPUs, memory, and specialized devices.

### Core Responsibilities

- **Container Lifecycle Management**: Coordinate container creation, starting, stopping, and removal through the container runtime
- **Resource Enforcement**: Apply CPU, memory, and device limits via cgroups (v1 and v2)
- **QoS Hierarchy**: Implement Kubernetes QoS classes (Guaranteed, Burstable, BestEffort) using cgroup hierarchies
- **Node Allocatable**: Reserve resources for system daemons and kubelet itself
- **Resource Coordination**: Orchestrate allocation decisions across CPU manager, memory manager, device manager, and topology manager
- **Health Checking**: Monitor device plugins and report resource health status

### Boundaries and Upstream/Downstream Dependencies

**Upstream Dependencies (Callers):**
- **Kubelet (`pkg/kubelet/kubelet.go`)**: Calls CM to initialize, manage pod lifecycle, and enforce node allocatable
- **Pod Workers (`pkg/kubelet/pod_workers.go`)**: Requests container runtime operations via CM
- **Lifecycle Hooks**: Pod admission handlers and resource allocation triggers
- **API Server**: Provides pod and node specifications

**Downstream Dependencies (Consumers):**
- **Container Runtime Interface (CRI)**: gRPC service for container/sandbox operations
- **cgroup Filesystem**: Linux kernel interface for resource enforcement
- **Device Plugins**: Kubernetes device plugin
