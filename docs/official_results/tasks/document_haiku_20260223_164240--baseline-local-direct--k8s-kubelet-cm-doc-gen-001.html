Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### 1. Topology Affinity Failures (Restricted Policy)\n\n**Scenario:** Pod requests resources that cannot be satisfied from single/overlapping NUMA nodes.\n\n**Behavior:**\n- Topology manager rejects pod admission with `TopologyAffinityError`\n- Pod remains `Unschedulable` in API server\n- No resources are allocated; pod never starts\n\n**Recovery:** User must modify pod resource requests to be compatible with available topology, or change policy to `best-effort`.\n\n**Code Reference:** `pkg/kubelet/cm/topologymanager/topology_manager.go` (Admit implementations in policies)\n\n#### 2. Cgroup Creation Failures\n\n**Scenario:** Required cgroup subsystems not mounted; invalid cgroup paths; permission denied.\n\n**Behavior:**\n- Kubelet fails to start; node goes `NotReady`\n- Error logged: \"failed to create top level QOS cgroup\"\n\n**Recovery:** Ensure cgroup subsystems are mounted; check kernel version supports required cgroups; verify kubelet runs as root or with CAP_SYS_ADMIN.\n\n**Code Reference:** `pkg/kubelet/cm/container_manager_linux.go:278` (QOS manager creation)\n\n#### 3. Device Plugin Allocation Failures\n\n**Scenario:** Device plugin unregisters mid-operation; allocation request fails; plugin becomes unhealthy.\n\n**Behavior:**\n- Pod admission is rejected if devices cannot be allocated\n- If device plugin fails after allocation, kubelet logs error but container still starts (resources already configured)\n- Node capacity for that device type drops; scheduler won't schedule more pods requesting that device\n\n**Recovery:**\n- Device plugin must restart and re-register\n- Existing pods using devices may continue running\n- New pods can be scheduled once plugin re-registers\n\n**Code Reference:** `pkg/kubelet/cm/devicemanager/manager.go` (Allocate method and error handling)\n\n#### 4. CPU Manager State Corruption\n\n**Scenario:** Checkpoint file corrupted or inaccessible; inconsistent state from previous run.\n\n**Behavior:**\n- CPU manager falls back to default (none) policy if state unrecoverable\n- Container starts without CPU affinity\n- Potential double-allocation of CPUs in next restart\n\n**Recovery:** Kubelet auto-recovery; state file is best-effort. If corruption persists, delete state file and accept temporary loss of CPU pinning guarantees.\n\n**Code Reference:** `pkg/kubelet/cm/cpumanager/state/state_checkpoint.go` (checkpoint loading)\n\n#### 5. NUMA Topology Change\n\n**Scenario:** Hardware topology changes at runtime (rare; hot-add NUMA node); cAdvisor reports different topology.\n\n**Behavior:**\n- Topology manager continues using old topology until kubelet restarts\n- Misalignment between hints and actual NUMA layout possible\n- No runtime revalidation\n\n**Impact:** Low risk; requires hardware-level changes.\n\n### 4.2 Resource Limits and Constraints\n\n| Limit | Default | Configurable | Impact |\n|-------|---------|--------------|--------|\n| Max NUMA nodes | 8 | No (hardcoded) | Prevents state explosion in topology manager; larger systems require work-around |\n| Cgroup nesting depth | Unlimited | No | Deep pod cgroups consume memory; no enforced limit |\n| CPUs for allocation | # Physical CPUs | Yes (reserved CPUs) | Cannot allocate more than available; excess reserved CPUs reduce capacity |\n| Memory for allocation | Total - reserved | Yes (reserved memory) | Cannot allocate more than available |\n| Device plugin count | Unlimited | No | More plugins = slower admission; no built-in limit |\n| Pod PID limit | 0 (disabled) | Yes (`--pod-pids-limit`) | When set, kubelet enforces per-pod PID cgroup limit |\n\n**Code References:**\n- NUMA node limit: `pkg/kubelet/cm/topologymanager/topology_manager.go:41` (defaultMaxAllowableNUMANodes)\n- PID limits: `pkg/kubelet/cm/types.go` (PidsLimit in ResourceConfig)\n\n### 4.3 Design Tradeoffs\n\n#### 1. Cooperative Allocation vs. Enforcement\n\n**Tradeoff:** Container manager _allocates_ resources at admission time (CPU, memory, devices) via hints, but does not strictly _enforce_ allocation for all resources.\n\n**Design Choice:** Cooperative approach.\n\n**Rationale:**\n- CPU affinity (cgroup cpuset) is enforced; container must run on assigned CPUs\n- Memory NUMA binding is enforced; memory allocated from specified nodes\n- Device allocation is enforced; devices not accessible outside container\n- BUT: CPU shares (cgroup.shares) and memory limits (cgroup.memory.limit) may be overridden by QoS manager later\n- This allows dynamic adjustment of limits during pod lifetime\n\n**Implication:** Pre-admission reservation is approximate; not absolute guarantee.\n\n**Code Reference:** `pkg/kubelet/cm/qos_container_manager_linux.go:141` (periodic UpdateCgroups)\n\n#### 2. Per-Container vs. Per-Pod Topology Scope\n\n**Tradeoff:** Topology manager can operate at container scope (each container gets optimal hints independently) or pod scope (all containers must use same NUMA nodes).\n\n**Design Choice:** Container scope is default; pod scope available as option.\n\n**Rationale:**\n- Container scope allows heterogeneous resource needs within single pod\n- Pod scope is stricter; useful for tightly-coupled workloads\n- Pod scope simpler to reason about; smaller state space\n\n**Implication:** Default behavior allows containers in same pod to be on different NUMA nodes.\n\n**Code Reference:** `pkg/kubelet/cm/topologymanager/topology_manager.go:192` (scope selection)\n\n#### 3. Hint Aggregation Policy\n\n**Tradeoff:** How to choose among multiple valid topology hints (CPU from node 0, memory from node 1, device from node 2)?\n\n**Design Choice:** Pluggable policies allow different strategies.\n\n**Strategies:**\n- `none` - Collect hints but don't enforce; allows unaligned allocation\n- `best-effort` - Prefer aligned; allow unaligned if necessary\n- `restricted` - Require alignment; reject if unaligned possible\n- `single-numa-node` - Strictest; single node required\n\n**Implication:** Different policies suit different workloads; no one-size-fits-all.\n\n**Code Reference:** `pkg/kubelet/cm/topologymanager/policy*.go` (policy implementations)\n\n#### 4. Checkpoint vs. Discovery\n\n**Tradeoff:** Should device plugin state be persisted across restarts (checkpoint) or rediscovered (discovery)?\n\n**Design Choice:** Hybrid.\n\n**Approach:**\n- CPU/memory managers checkpoint state to survive restarts\n- Device managers discover plugins at startup (no checkpoint)\n- DRA manager checkpoints claim state (for recovery)\n\n**Rationale:**\n- CPU/memory state requires persistent stable allocation (cores are pinned)\n- Devices are inherently volatile; plugins re-register on startup\n- Consistent with existing device plugin model\n\n**Implication:** CPU/memory allocations survive kubelet restarts; device allocations may change if plugins register differently.\n\n**Code References:**\n- CPU checkpointing: `pkg/kubelet/cm/cpumanager/state/state_checkpoint.go`\n- Device discovery: `pkg/kubelet/cm/devicemanager/manager.go:Start()`\n\n#### 5. QoS Class Enforcement via Cgroup Shares vs. Hard Limits\n\n**Tradeoff:** How to enforce QoS class guarantees?\n\n**Design Choice:** CPU shares for soft enforcement; memory hard limits for strict enforcement.\n\n**Approach:**\n- Guaranteed pods get high CPU shares (1024/total cpus per container)\n- Burstable pods get medium CPU shares (relative to requests)\n- BestEffort pods get minimum CPU shares (MinShares constant)\n- Memory limits are hard cgroup limits (OOMKilled if exceeded)\n\n**Rationale:**\n- CPU shares allow elasticity (unused cycles available to other pods)\n- Memory hard limits prevent runaway memory consumption\n- Matches Kubernetes QoS semantics\n\n**Implication:** CPU allocation is soft (can burst); memory is hard (strict).\n\n**Code Reference:** `pkg/kubelet/cm/qos_container_manager_linux.go` (CPU shares logic)\n\n---\n\n## 5. Extension Points, Customization Hooks, and Risks\n\n### 5.1 CPU Manager Policies\n\n**Extension Point:** Pluggable CPU allocation strategies\n\n**Location:** `pkg/kubelet/cm/cpumanager/policy.go`\n\n**Interface:**\n```go\ntype Policy interface {\n    Start(state state.State) error\n    GetTopologyHints(state state.State, pod *v1.Pod, container *v1.Container) map[string][]TopologyHint\n    Allocate(state state.State, pod *v1.Pod, container *v1.Container) error\n    RemoveContainer(state state.State, containerID string) error\n}\n```\n\n**Built-in Policies:**\n- `none` - No pinning (shared pools)\n- `static` - Exclusive CPU allocation per container\n\n**Risks:**\n- Custom policy may cause CPU over-subscription or starvation\n- Policy must be stateless or maintain consistent checkpoint\n- Adding policies requires kernel rebuild; not dynamically loadable\n\n**Extension Example:** To implement a `numa-pinning` policy that forces CPU allocation to respect NUMA boundaries:\n1. Operational Implications and Considerations\n\n### 6.1 Performance and Overhead\n\n#### Admission Latency\n\n**Impact:** Topology hint aggregation adds latency to pod admission (only during `Admit()` phase).\n\n**Typical Overhead:**\n- `none` policy: < 1ms (no topology checks)\n- `best-effort` / `restricted`: 10-50ms depending on number of hint providers\n- `single-numa-node`: 5-20ms (narrower search space)\n\n**Bottleneck:** Merging hints from multiple providers (CPU, memory, device) is O(n^m) where n = number of NUMA nodes, m = number of providers.\n\n**Mitigation:**\n- Use `container` scope (default) instead of `pod` scope for smaller state space\n- Limit number of NUMA nodes (max 8 supported)\n- Use `none` policy if topology awareness not needed\n\n**Code Reference:** `pkg/kubelet/cm/topologymanager/topology_manager.go` (Admit)\n\n#### Checkpoint/Recovery Overhead\n\n**CPU Manager:** Reads `cpu_manager_state` file on startup; creates entries for each container with exclusive CPUs. Typically negligible (<100ms for 100 containers).\n\n**Memory Manager:** Similar; reads memory allocation state. Negligible overhead.\n\n**DRA Manager:** Reads claim state; may validate with API server. Can be slow if many claims or API server latency.\n\n#### Cgroup Operations\n\n**Cgroup Creation/Update:** Each pod cgroup creation requires multiple syscalls:\n- `mkdir()` for cgroup directory\n- `write()` to set resource limits\n- `stat()` to verify creation\n\n**Typical Time:** 10-50ms per pod cgroup.\n\n**Overhead at Scale:** With 1000 pods, ~10-50 seconds for initial cgroup setup; amortized over time.\n\n**Mitigation:** Kubelet caches cgroup existence checks; only creates if missing.\n\n### 6.2 Reliability and State Management\n\n#### State Recovery\n\n**Checkpoint Resilience:**\n- CPU manager checkpoints state to disk; survives graceful shutdown and restart\n- Memory manager same\n- DRA manager checkpoints claim state\n- Device allocations not persisted; rediscovered at startup\n\n**Implication:** Restarting kubelet should preserve CPU/memory pinning but may change device assignments.\n\n**Risk:** If checkpoint file corrupted, kubelet may lose track of allocations. Mitigation: auto-fall-back to default policy if unrecoverable.\n\n**Code Reference:** `pkg/kubelet/cm/cpumanager/state/state_checkpoint.go:Restore()`\n\n#### State Synchronization\n\n**Period:** CPU and memory managers periodically reconcile runtime state with checkpoint state (~5-minute reconciliation loop).\n\n**Purpose:** Detect and recover from runtime inconsistencies (e.g., container created outside kubelet).\n\n**Implication:** If containers are manually manipulated, managers may not detect changes immediately.\n\n**Code Reference:** `pkg/kubelet/cm/cpumanager/cpu_manager.go` (reconcileState)\n\n#### Partial State Loss\n\n**Scenario:** DRA plugin crashes during resource prepare; kubelet has inconsistent state.\n\n**Behavior:** Kubelet logs error; pod may fail to start. On kubelet restart, DRA manager tries to unprepare and recover.\n\n**Risk:** Resources may be allocated but not tracked; leaks.\n\n**Mitigation:** DRA plugins should be idempotent; prepare/unprepare operations should be safe to retry.\n\n### 6.3 Troubleshooting Guidance\n\n#### Topology Affinity Errors\n\n**Symptom:** Pod stuck in `Unschedulable` state; events show \"TopologyAffinityError\".\n\n**Diagnosis:**\n1. Check topology manager policy: `kubectl describe node <node>` (no direct way; check kubelet logs)\n2. Check pod resource requests: `kubectl describe pod <pod>`\n3. Check available NUMA topology: `numactl -H` on node\n\n**Resolution:**\n- Increase requested resources if allocation too constrained\n- Change policy to `best-effort` if `restricted` too strict\n- Add more NUMA nodes if available\n- Reduce CPU affinity requests (e.g., remove exclusive CPU request)\n\n#### Device Not Allocated\n\n**Symptom:** Pod requesting device (e.g., `nvidia.com/gpu: 1`) fails to start or misses device.\n\n**Diagnosis:**\n1. Check device plugin registered: `kubectl get nodes -o json | jq '.items[0].status.allocatable'` (see if device is reported)\n2. Check device plugin logs: `kubectl logs -n kube-system <device-plugin-pod>`\n3. Verify device socket exists: `ls -la /var/lib/kubelet/plugins/`\n\n**Resolution:**\n- Restart device plugin if crashed\n- Check plugin logs for allocation errors\n- Verify device physically available: `lspci` on node\n- Check device plugin version compatibility\n\n#### High Memory Usage During Pod Admission\n\n**Symptom:** Kubelet memory grows rapidly during pod creation; especially with large topology or many devices.\n\n**Diagnosis:**\n1. Check number of NUMA nodes: `numactl -H` (> 8 = unsupported)\n2. Check number of registered devices: `nvidia-smi -L` or equivalent\n3. # Kubelet Container Manager Subsystem: Architecture Guide

## Executive Summary

The kubelet container manager is a core subsystem responsible for enforcing resource constraints, coordinating resource allocation across containers, and managing the cgroup hierarchy on nodes. It acts as a mediator between the Kubernetes API, the container runtime, and the host kernel's resource management facilities (cgroups), enabling fine-grained CPU, memory, and device allocation with support for NUMA topology awareness.

---

## 1. Subsystem Purpose and Scope

### Primary Responsibilities

The container manager subsystem is responsible for:

1. **Resource Constraint Enforcement** - Translating Kubernetes resource requests/limits into kernel cgroup limits
2. **Resource Allocation Coordination** - Ensuring that CPU affinity, memory NUMA locality, and device assignments are coordinated across multiple allocation strategies
3. **Cgroup Hierarchy Management** - Creating and maintaining the cgroup directory tree reflecting QoS classes and pod/container boundaries
4. **Policy-Driven Allocation** - Supporting pluggable policies for CPU and memory allocation strategies
5. **Device Plugin Integration** - Allowing external device managers to allocate specialized hardware
6. **Dynamic Resource Allocation** - Managing vendor-specific resource claims via DRA plugins

**Within Scope:**
- Cgroup creation, configuration, and destruction (both cgroup v1 and v2)
- CPU affinity assignment (static CPU manager policy)
- NUMA-aware memory allocation and pinning

- NUMA topology hint aggregation and consensus
- System container isolation (kubelet, docker daemon)
