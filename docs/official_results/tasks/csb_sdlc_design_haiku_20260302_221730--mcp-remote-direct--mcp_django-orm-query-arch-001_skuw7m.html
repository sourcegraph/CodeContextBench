## Summary

The analysis covers:

### **Files Examined** (12 key files)
- Entry points: Manager, QuerySet
- Query representation: Query, WhereNode, datastructures
- Compilation: SQLCompiler (base) and backend-specific variants
- Expression system: Expressions, Lookups (the as_sql protocol)

### **Dependency Chain**
Traced the complete flow with code references:
1. Manager.get_queryset() → QuerySet creation
2. QuerySet.filter() → Query.add_q() (lazy)
3. Iteration triggers → Query.get_compiler() (vendor dispatch!)
4. SQLCompiler.execute_sql() → SQLCompiler.as_sql() orchestrates SQL building
5. compiler.compile(node) → vendor dispatch to as_{vendor}() or as_sql()
6. Expression/Lookup/WhereNode.as_sql() → recursive SQL generation
7. cursor.execute() → results

### **Architectural Insights**

**Key Design Patterns:**
- **Lazy evaluation** — QuerySet doesn't compile SQL until executed
- **Visitor pattern** — Compiler.compile() traverses expression trees
- **Vendor dispatch** — `as_{vendor}()` method naming allows backend-specific SQL
- **Expression protocol** — Unified `as_sql(compiler, connection)` interface
- **Tree-based query** — WhereNode/joins/selections all tree structures

**Vendor Dispatch Mechanism (Critical!):**
```python
vendor_impl = getattr(node, "as_" + self.connection.vendor, None)
if vendor_impl:
    sql, params = vendor_impl(self, self.connection)
else:
    sql, params = node.as_sql(self, self.connection)
```

This elegant pattern allows PostgreSQL/MySQL/SQLite/Oracle-specific optimizations without branching core logic.
