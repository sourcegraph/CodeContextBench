## Review Summary

I identified **3 critical defects** in the tRPC router that violate the service's established type safety patterns:

### Defects Found

All three defects follow the same pattern: the router mutations always pass `assignedBy` even when the state is `"inherit"`, but the service's discriminated union type explicitly requires that `assignedBy` is only included when state is `"enabled"` or `"disabled"`.

1. **setUserState** (line 103): Always passes `assignedBy` when calling `setUserFeatureState`
2. **setTeamState** (line 131): Always passes `assignedBy` when calling `setTeamFeatureState`  
3. **setOrganizationState** (line 161): Always passes `assignedBy` when calling `setTeamFeatureState`

### Root Cause

The service methods use a discriminated union pattern:
```typescript
// ✓ When state is "inherit", assignedBy should NOT be passed
| { userId: number; featureId: FeatureId; state: "inherit" }

// ✓ When state is "enabled"|"disabled", assignedBy MUST be passed
| { userId: number; featureId: FeatureId; state: "enabled" | "disabled"; assignedBy: number }
```

But the router mutations always include `assignedBy` regardless of state, breaking this contract.

### Fixes

Each fix conditionally includes `assignedBy` based on the state value, respecting the service's type signature. The fixes use type-safe patterns with proper type casting to ensure correct behavior at runtime.
