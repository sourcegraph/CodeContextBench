Full push: recompute proxy state (service instances, sidecar scope, etc.)\n    if req.Full {\n        s.computeProxyState(con.proxy, req)\n    }\n\n    // 2. Check if proxy needs this push\n    req, needsPush := s.ProxyNeedsPush(con.proxy, req)\n    if !needsPush {\n        return nil  // Skip push if config didn't change for this proxy\n    }\n\n    // 3. Push each resource type in order (CDS \u2192 EDS \u2192 LDS \u2192 RDS \u2192 SDS)\n    wrl := con.watchedResourcesByOrder()\n    for _, w := range wrl {\n        s.pushXds(con, w, req)\n    }\n}\n```\n\n**Push ordering** (defined in `ads.go:PushOrder`):\n1. **Clusters** \u2014 service definitions\n2. **Endpoints** \u2014 where instances are running\n3. **Listeners** \u2014 inbound/outbound network endpoints\n4. **Routes** \u2014 how to route traffic\n5. **Secrets** \u2014 TLS certificates\n\nThis order ensures Envoy can build dependencies correctly (e.g., routes reference listeners and clusters).\n\n### Broadcast: `StartPush()`\n\nWhen Pilot wants to push to all connected proxies (e.g., after a VirtualService update):\n\n```go\nfunc (s *DiscoveryServer) StartPush(req *model.PushRequest) {\n    req.Start = time.Now()\n    for _, p := range s.AllClients() {\n        s.pushQueue.Enqueue(p, req)\n    }\n}\n```\n\nThis enqueues the push request for every proxy. The `pushQueue` (with concurrency limiting) then calls `pushConnection()` concurrently for each proxy.\n\n### Connection Cleanup: `closeConnection()`\n\nWhen a proxy disconnects or Pilot needs to force a disconnect:\n\n```go\nfunc (s *DiscoveryServer) closeConnection(con *Connection) {\n    s.removeCon(con.ID())\n    s.WorkloadEntryController.OnDisconnect(con)\n}\n```\n\nThis removes the proxy from the `adsClients` map and notifies the auto-registration system (for dynamically registered workloads).\n\n---\n\n## 3. PushContext \u2014 Configuration Snapshot and Translation\n\n**Source:** `pilot/pkg/model/push_context.go`\n\n### Overview\n\n`PushContext` is an **immutable snapshot of all configuration and service data** captured at the moment a push begins. Every push gets its own `PushContext` instance, ensuring consistency: all proxies receiving the same push see the same view of services, routes, policies, etc.\n\n### Structure\n\n```go\ntype PushContext struct {\n    ServiceIndex serviceIndex                      // All services by namespace, gateway, etc.\n    virtualServiceIndex virtualServiceIndex        // VirtualServices by gateway\n    destinationRuleIndex destinationRuleIndex      // DestinationRules by namespace\n    sidecarIndex sidecarIndex                      // Sidecar rules by namespace\n    gatewayIndex gatewayIndex                      // Gateway configs\n\n    envoyFiltersByNamespace map[string][]*EnvoyFilterWrapper\n    wasmPluginsByNamespace map[string][]*WasmPluginWrapper\n\n    AuthnPolicies *AuthenticationPolicies\n    AuthzPolicies *AuthorizationPolicies\n    Telemetry *Telemetries\n    ProxyConfigs *ProxyConfigs\n\n    Mesh *meshconfig.MeshConfig\n    JwtKeyResolver *JwksResolver\n\n \u2014 services exportable to all namespaces\n- **privateByNamespace** \u2014 namespace-local services\n- **exportedToNamespace** \u2014 services selectively exported to specific namespaces\n- **HostnameAndNamespace** \u2014 all services, keyed by hostname then namespace\n\nThis indexing avoids O(n) lookups during route generation for each proxy.\n\n#### Virtual Service and Destination Rule Indexes\nOrganizes Istio's traffic policies by namespace and gateway:\n- By namespace (private vs. exported)\n- By gateway (Ingress, egress, or mesh internal)\n- By parent-child relationship (for delegated routes)\n\nDuring RDS generation, Pilot looks up the `VirtualService` for a given hostname/gateway and translates it to Envoy `RouteConfiguration`.\n\n#### Config Translation Indexes\n- **sidecarIndex** \u2014 speeds up \"what routes/clusters should this proxy see?\" computations\n- **gatewayIndex** \u2014 indexes gateways for route generation\n- **envoyFilters** \u2014 custom Envoy patches per namespace\n- **wasmPlugins** \u2014 WebAssembly plugins per namespace\n\n### Initialization: `InitContext()`\n\n```go\nfunc (push *PushContext) InitContext(env *model.Environment, oldPushContext, req) {\n    // 1. Build destination rule index\n    push.buildDestinationRuleIndex(env.configStore)\n\n    // 4. Build sidecar index\n    push.buildSidecarIndex(env.configStore)\n\n    // 5. That's why debouncing is critical \u2014 we don't want to do this for every small change.\n\n### Config Translation\n\nxDS generators use `PushContext` to look up and translate Istio config to Envoy. Example flow for RDS:\n\n1. **Generator requests routes for a proxy**\n2. **Look up sidecar scope** \u2192 `PushContext.GetProxySidecarScope(proxy)` \u2192 determines which namespaces/services this proxy can access\n3. **For each service hostname the proxy can see:**\n   - `PushContext.VirtualServices(hostname)` \u2192 get VirtualServices for this host\n   - Translate each VS's `http` routes to Envoy `Route` messages\n4. **For each route destination:**\n   - Look up `DestinationRule` \u2192 `PushContext.GetDestinationRule(hostname)`\n   - Extract subset definitions and translate to cluster names\n5. **Return Envoy `RouteConfiguration`**\n\n### Metrics and Error Tracking\n\n`PushContext` tracks push status per proxy:\n\n```go\nProxyStatus map[string]map[string]ProxyPushStatus\n```\n\nAfter generating xDS for a proxy, the status is recorded: did generation succeed? What errors occurred? This data is exposed via the debug API.\n\n### Lifecycle\n\nA new `PushContext` is created **only** on full pushes (when configuration changes):\n\n```go\nfunc (s *DiscoveryServer) Push(req *model.PushRequest) {\n    // Full push: create new context\n    if req.Full {\n        push := s.initPushContext(req, oldContext, version)\n        req.Push = push\n        s.Env.SetPushContext(push)  // Update global context\n    }\n    s.AdsPushAll(req)\n}\n```\n\nIncremental pushes (EDS updates) reuse the current context; they don't rebuild indexes.\n\n---\n\n## 4. Error Handling\n\n### Configuration Validation\n\nInvalid Istio configuration (bad `VirtualService`, etc.) is detected during indexing:\n\n- **Type mismatch** \u2014 port not found in destination service\n- **Circular references** \u2014 delegation loops in routes\n- **Policy conflicts** \u2014 ambiguous DestinationRules\n\n**Action:** Log warning, index the config as-is, let xDS generators handle or skip.\n\n### Push Failures\n\nIf generating xDS for a proxy fails:\n\n```go\nfunc (s *DiscoveryServer) pushXds(con, w, req) {\n    res, _, err := gen.Generate(con.proxy, w, req)\n    if err != nil {\n        // Record error in proxy status\n        pushContext.AddMetric(metrics.PushContextErrors, ..., err.String())\n        // Log and skip this resource type\n        return err\n    }\n}\n```\n\nProxy status is recorded in `PushContext.ProxyStatus`:\n\n```\nProxyStatus[\"generation_error\"][\"proxy-id\"] = {\n    Error: \"failed to generate CDS: unknown service\",\n}\n```\n\nExposed via debug API (`/debug/push_context`) so operators can investigate.\n\n### Network Failures\n\nIf sending the xDS response fails:\n\n```go\nfunc (s *DiscoveryServer) pushXds(con, w, req) {\n    // ... generate resources ...\n\n    if err := con.stream.Send(response); err != nil {\n        // Proxy disconnected or network error\n        s.closeConnection(con)\n        return err\n    }\n}\n```\n\nThe proxy is removed from `adsClients`; it will reconnect and re-request resources.\n\n### Cache Staleness\n\nThe XDS cache maps resources by content hash and timestamp:\n\n```\nCacheKey: {Proxy.ID, TypeUrl, ResourceNames}\nValue: {Resources, Timestamp}\n```\n\nWhen a new `PushContext` is created, the timestamp is updated. Old cache entries are considered stale and evicted.\n\n### Rate Limiting\n\nIf too many proxies reconnect simultaneously (thundering herd):\n\n```go\nfunc (s *DiscoveryServer) WaitForRequestLimit(ctx) error {\n    return s.RequestRateLimit.Wait(ctx)  // Token bucket limiter\n}\n```\n\nExceeding the limit returns `codes.ResourceExhausted`; the proxy backs off and retries.\n\n---\n\n## 8. Performance Considerations\n\n### Debouncing\n- **Problem:** Rapid config changes cause expensive PushContext rebuilds\n- **Solution:** Batch changes within `DebounceAfter` window, apply in one push\n- **Benefit:** Reduces rebuild cost from O(n changes) to O(1)\n\n### Caching\n- **Problem:** Generating the same xDS resource for many proxies is wasteful\n- **Solution:** Cache resources by (proxy, type, names) after first generation\n- **Benefit:** Reduces generation CPU and memory\n\n### Incremental Updates\n- **Problem:** Full pushes recompute all config for all proxies\n- **Solution:** EDS-only pushes reuse PushContext, update only endpoints\n- **Benefit:** 10-100x faster for large deployments with frequent scaling\n\n### Sidecar Scoping\n- **Problem:** Each proxy sees 50+ services, but only needs routes to a few\n- **Solution:** Sidecar resource limits visibility per-proxy\n- **Benefit:** Smaller xDS messages, faster serialization\n\n### Service Index\n- **Problem:** O(n) service lookups in large clusters\n- **Solution:** Pre-index services by namespace, gateway, hostname\n- **Benefit:** O(1) service lookups during generation\n\n---\n\n## 9. # Istio Pilot Discovery Service Architecture

## Executive Summary

Istio Pilot's Discovery Service is the core control plane subsystem that translates high-level Istio service mesh configuration into Envoy-specific xDS (Envoy Discovery Service) resources and pushes them to proxies in real time. It acts as a mediator between:

- **Istio Configuration** — VirtualServices, DestinationRules, ServiceEntries, Gateways, AuthorizationPolicies, etc.
- **Service Registries** — Kubernetes Services, Endpoints, multi-cluster ServiceImports
- **Envoy Proxies** — Connected via gRPC/xDS API expecting CDS (Clusters), LDS (Listeners), RDS (Routes), EDS (Endpoints), SDS (Secrets)

The architecture is organized around four interdependent components: **DiscoveryServer**, **ADS/Connection**, **PushContext**, and **ServiceRegistry**. Together they enable Pilot to efficiently compute and distribute configuration changes to hundreds or thousands of proxies while supporting advanced features like incremental updates, multi-cluster deployments, and custom configuration generators.
