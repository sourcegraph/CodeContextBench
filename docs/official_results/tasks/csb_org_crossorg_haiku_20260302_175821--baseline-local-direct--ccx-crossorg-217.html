**Django's Middleware Chain Implementation:**

1. **Main Implementation File**: `django/core/handlers/base.py`
   - The `BaseHandler.load_middleware()` method loads middleware from `settings.MIDDLEWARE` in **reversed order** (line 41)
   - Each middleware is wrapped with exception handling to prevent exceptions from leaking
   - The middleware chain is stored in `_middleware_chain` attribute

2. **Middleware Base Class**: `django/utils/deprecation.py` - `MiddlewareMixin`
   - Provides the standard middleware interface
   - `__call__()` handles sync requests: process_request → get_response → process_response
   - `__acall__()` handles async requests with the same pattern

3. **Different Processing Orders**:
   - **process_view**: Inserted at position 0 (processed in reverse of definition order)
   - **process_template_response**: Appended (processed in definition order)
   - **process_exception**: Appended (processed in definition order)

4. **WSGI vs Django Middleware Comparison**:
   - Unlike raw WSGI middleware stacking, Django explicitly reverses the middleware list
   - Django separates different middleware concerns (request, view, exception, template response)
   - Each middleware layer is wrapped with `convert_exception_to_response()` to prevent exception leakage
   - The middleware chain is constructed as nested callables at startup, not dynamically per-request

5. **Handler Integration**:
   - `WSGIHandler` (wsgi.py): Initializes middleware in `__init__()`, executes via `get_response()`
   - `ASGIHandler` (asgi.py): Uses async-capable middleware loading with `is_async=True`
