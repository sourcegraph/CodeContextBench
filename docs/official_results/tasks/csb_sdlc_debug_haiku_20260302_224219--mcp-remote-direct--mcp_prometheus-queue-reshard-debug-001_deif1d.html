## Key Findings Summary

**Root Cause**: A race condition during queue resharding where incoming samples continuously retry enqueue operations (holding `RLock`) while the resharding process times out waiting for old shards to flush, triggering a hard shutdown that silently drops all pending data.

**Why It Happens**:
1. Target discovery changes trigger resharding
2. `Append()` retries enqueue in a tight loop, repeatedly holding `RLock`
3. `stop()` waits for these `RLock` holders to finish so it can acquire the exclusive `Lock`
4. If samples keep arriving faster than the old shards can flush, `stop()` times out after `flushDeadline`
5. Hard shutdown silently drops all pending samples and marks them as "failed"

**Why It's Intermittent**:
- Depends on target discovery rate, sample arrival rate, and network conditions
- Exacerbated when recoverable errors (429/503) occur, triggering aggressive resharding attempts

**Evidence**: Multiple code locations in `storage/remote/queue_manager.go` including:
- The retry loop (lines 731-755) that causes lock contention
- The hard shutdown trigger (lines 1286-1294)
- Sample drop mechanism (lines 1563-1578)
- Mitigation code that disables resharding during recoverable errors (lines 2014-2029)

The report includes diagnostic steps, affected components, and recommended fixes for both immediate mitigations and long-term improvements.
