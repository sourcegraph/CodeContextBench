Failure Modes\n\n### eBPF Program Loading Failures\n\n#### Verifier Failures\n\n**Symptom**: Error message with \"verifier log\" in logs\n**Root Causes**:\n- Invalid memory access patterns\n- Unbounded loops or complexity\n- Use of unsupported BPF helpers on kernel\n- Stack overflows (eBPF stack is limited to 512 bytes)\n\n**Handling**: `pkg/datapath/loader/netlink.go:87-92`\n```go\nif errors.As(err, &ve) {\n    // Print verifier error for debugging\n    fmt.Fprintf(os.Stderr, \"Verifier error: %s\\nVerifier log: %+v\\n\", err, ve)\n}\n```\n\n**Recovery**: Must recompile with simpler logic or requires kernel upgrade\n\n#### Map Incompatibility\n\n**Symptom**: `ErrMapIncompatible` when loading\n**Cause**: Pinned map from previous run has different structure (type, key/value size, etc.)\n**Handling**: `pkg/datapath/loader/netlink.go:253-256` - Maps are removed if incompatible before reload\n\n#### Map Full / Allocation Failures\n\n**Symptom**: `BPF_MAP_CREATE` fails\n**Causes**:\n- Insufficient kernel memory\n- `/sys/fs/bpf/` filesystem limits\n- Memory pressure from other applications\n\n**Diagnosis Tools**:\n```bash\nbpftool map list              # See all maps and their usage\nbpftool map show id <id>      # Details of specific map\ncat /proc/meminfo             # Kernel memory pressure\n```\n\n### Kernel Compatibility Issues\n\n#### Unsupported Feature Detection\n\n**System**: `pkg/datapath/linux/probes/probes.go`\n\n**Detection Pattern**:\n```go\n// Test if kernel supports a feature by trying to load it\nfunc HaveProgramHelper(progType ebpf.ProgramType, helper asm.BuiltinFunc) error {\n    // Attempt to create minimal program using the helper\n    // If it fails with ErrNotSupported, feature is missing\n}\n```\n\n**Examples**:\n- `probes.HaveFibLookup()` - Required for NodePort\n- `probes.HaveKtimeGetBootNs()` - Required for packet recording\n- `probes.HaveRedirectNeigh()` - Required for host routing\n\n**Configuration Validation**: `daemon/cmd/kube_proxy_replacement.go:245-384`\n- Validates kernel supports required helpers before enabling features\n- Returns errors if configuration impossible on this kernel\n\n### Configuration Errors\n\n#### Invalid Program Symbols\n\n**Cause**: eBPF source doesn't define expected symbol\n**Detection**: eBPF collection loading fails with \"program not found\"\n**Prevention**: Symbol constants verified in compile and load stages\n\n#### Missing or Malformed Headers\n\n**Cause**: Configuration headers (node_config.h, netdev_config.h) generation fails\n**Symptom**: Compilation errors mentioning undefined macros\n**Prevention**: `WriteNodeConfigHeader()` validates all required options\n\n#### Template Cache Invalidation\n\n**Mechanism**: Hash computed on configuration changes (`loader.go:621-627`)\n```go\nspec, hash, err := l.templateCache.fetchOrCompile(ctx, cfg, ep, &dirs, stats)\n```\n- If datapath configuration changes, old templates invalidated\n- Forces recompilation\n- Cache prevents redundant compilations\n\n### Network Interface Issues\n\n#### Device Not Found\n\n**Symptom**: \"retrieving device\" error in logs\n**Cause**: Interface doesn't exist when loader tries to attach programs\n**Handling**: `loader.go:382-385` - Logs warning and continues\n\n#### Attachment Failures\n\n**TC Qdisc Errors**: `attachTCProgram()` failures\n**Cause**:\n- qdisc already exists\n- Permission denied\n- Kernel TC subsystem issues\n\n**Fallback Chain** (attachSKBProgram):\n1. Try TCX (modern, Linux 6.6+)\n2. Try netkit (specialized for netkit devices)\n3. Fall back to legacy TC\n4. If all fail, error returned\n\n### Recovery Mechanisms\n\n#### Automatic Cleanup\n\n**On Failure**: `reloadDatapath()` returns error but partial state may remain\n- Pinned BPF links in `/sys/fs/bpf/` may remain\n- Maps may be orphaned\n\n**Manual Recovery**:\n```bash\n# Remove all Cilium BPF objects\nrm -rf /sys/fs/bpf/tc/\nrm -rf /sys/fs/bpf/cilium/\n\n# Restart Cilium - will reinitialize everything\nsystemctl restart cilium\n```\n\n#### State Preservation\n\n- Pinned maps preserved across restarts for connection tracking continuity\n- Obsolete programs removed by `removeObsoleteNetdevPrograms()`\n\n#### Configuration Rollback\n\n- Previous node configuration preserved in templates\n- If new compilation fails, old programs can still run (if pinned)\n\n---\n\n## 5. ## 1. Purpose

### What Problem Does the eBPF Datapath Solve?

The eBPF (extended Berkeley Packet Filter) datapath is the foundation of Cilium's networking and security enforcement. Instead of relying on traditional iptables rules or userspace packet processing, Cilium uses eBPF programs running in the Linux kernel to:

1. **High-performance packet processing** - eBPF programs execute in kernel space with minimal overhead
2. **Fine-grained network policy enforcement** - Policies are enforced at kernel level before packets reach applications
3. **Dynamic behavior without kernel changes** - New networking features can be added by loading new eBPF programs without modifying the kernel
4. **Visibility and observability** - eBPF programs can generate events for network monitoring

### Why eBPF Instead of iptables or Userspace Networking?

- **Performance**: eBPF programs execute directly in the kernel JIT-compiled to CPU instructions, avoiding context switches
- **Flexibility**: eBPF allows dynamic packet processing logic that can be updated without recompiling the kernel
- **Safety**: The in-kernel verifier ensures eBPF programs are safe and cannot crash the kernel
- **Real-time capabilities**: Enables features like TCP retransmission optimization and connection tracking that iptables cannot provide
- **Container-awareness**: eBPF can access kernel data structures to make decisions based on container/pod identity
