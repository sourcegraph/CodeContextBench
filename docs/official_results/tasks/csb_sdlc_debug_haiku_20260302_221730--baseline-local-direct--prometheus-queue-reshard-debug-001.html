### **Root Cause: Race Condition in Queue Shutdown**

The issue is a **critical race condition** in `storage/remote/queue_manager.go` where:

1. **Location:** `FlushAndShutdown()` function (lines 1447-1455)

2. **The Problem:** When resharding occurs and triggers a hard shutdown, the partial batch of samples waiting to be sent is **discarded with `q.batch = nil`** without properly decrementing the `prometheus_remote_storage_samples_pending` metric.

3. **Why Samples Get Stuck:**
   - Samples are enqueued and counted in `pendingSamples` metric
   - When partial batches are waiting to flush during resharding
   - Hard shutdown forces closure before the partial batch can be flushed
   - The partial batch is deleted without updating metrics
   - Result: `prometheus_remote_storage_samples_pending` shows stuck samples

### **Why It's Intermittent**

The race condition only occurs when ALL these conditions align:
- **Remote endpoint is slow** (blocks `sendSamplesWithBackoff`)
- **Partial batch exists** (samples below `MaxSamplesPerSend` threshold)
- **flushDeadline times out** (graceful shutdown expires)
- **Resharding is triggered** (target discovery changes during this window)

This timing dependency explains why it's intermittent—it depends on the exact state of the remote endpoint and sample arrival patterns.

### **Verification Metrics**

The report recommends checking:
- `prometheus_remote_storage_samples_pending` > 0 with `sent_batches_total` rate = 0
- Correlation of resharding logs with metric anomalies
- Spikes in `retried_samples_total` during resharding
