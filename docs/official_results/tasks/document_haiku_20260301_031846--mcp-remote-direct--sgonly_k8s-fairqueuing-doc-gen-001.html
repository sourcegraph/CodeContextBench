Fair Queuing Algorithm and Virtual Time Mechanism\n\n### 4.1 Algorithm Foundations\n\nThe implementation is based on **Weighted Fair Queuing (WFQ)** adapted for server requests, documented in the original research paper on fair queuing and explained in depth in `doc.go`.\n\n**Key adaptation**: Instead of transmitting one packet at a time from one queue, the algorithm:\n1. Dispatches multiple requests concurrently (up to `ConcurrencyLimit`)\n2. Weights by request \"width\" (seats) instead of packet size\n3. Estimates service time rather than measuring packet transmission\n\n### 4.2 Virtual Time Formula\n\nIn traditional fair queuing, virtual time advances as:\n\n```\ndR/dt = 1 / NEQ(t)\n\nwhere NEQ(t) = number of non-empty queues\n```\n\nFor concurrent server requests with C concurrency limit:\n\n```\ndR/dt = min(C, sum of reqs(q, t)) / NEQ(t)\n\nwhere reqs(q, t) = number of requests executing from queue q\n```\n\n**Implementation** (`getVirtualTimeRatioLocked`):\n\n```go\nreturn math.Min(float64(seatsRequested), float64(qs.dCfg.ConcurrencyLimit)) / float64(activeQueues)\n```\n\n### 4.3 Virtual Finish Time\n\nVirtual finish time for a request in queue q:\n\n```\nVFT_i = queue.nextDispatchR + (work of request i)\n\nwhere work = InitialSeats \u00d7 EstimatedServiceDuration + FinalSeats \u00d7 AdditionalLatency\n```\n\nThe request with the **minimum virtual finish time** across all queues is dispatched next.\n\n**Invariant**: This ensures each queue makes proportional progress, and requests are served fairly.\n\n### 4.4 Virtual Start Time and Anti-Windup\n\nTo avoid expensive computation of virtual finish times for each request, the implementation uses virtual start time:\n\n```\nqueue.nextDispatchR = virtual start time of next request\n\nRequest i's virtual finish time = queue.nextDispatchR + request[i].work\n```\n\nWhen a request finishes:\n\n```\nqueue.nextDispatchR += request.work  (at dispatch time)\n```\n\n**Anti-windup adjustment** (`boundNextDispatchLocked`, line 945-964):\n\nSome queues may be unable to use all allocated virtual time (e.g., queue with small requests competing for limited max-seats). Without correction, such queues would accumulate progressively earlier virtual start times, starving other queues.\n\n```go\nfunc (qs *queueSet) boundNextDispatchLocked(queue *queue) {\n    oldestReq, _ := queue.requestsWaiting.Peek()\n    if oldestReq == nil {\n        return\n    }\n    // Ensure virtual time doesn't precede arrival of next request\n    virtualStartBound = oldestReq.arrivalR\n    if queue.nextDispatchR < virtualStartBound {\n        queue.nextDispatchR = virtualStartBound\n    }\n}\n```\n\nThis hack ensures dispatch in virtual time never precedes arrival in virtual time, preventing unfair accumulation of virtual time debt.\n\n### 4.5 Epoch Advancement (Overflow Handling)\n\n**Problem**: After long running, `currentR` can exceed the maximum SeatSeconds value.\n\n**Solution**: Periodically wind back all R values by `rDecrement = MaxSeatSeconds / 2` (line 483-522).\n\n```go\nconst rDecrement = fqrequest.MaxSeatSeconds / 2\nconst highR = rDecrement + rDecrement/2\n\nfunc (qs *queueSet) advanceEpoch(ctx context.Context, now time.Time, incrR fqrequest.SeatSeconds) {\n    qs.currentR -= rDecrement\n\n    for qIdx, queue := range qs.queues {\n        // Empty queues reset to zero\n        if queue.requestsWaiting.Length() == 0 && queue.requestsExecuting.Len() == 0 {\n            queue.nextDispatchR = 0\n            continue\n        }\n        // Non-empty queues decrement by rDecrement\n        queue.nextDispatchR -= rDecrement\n\n        // Walk all waiting requests and decrement their arrivalR\n        queue.requestsWaiting.Walk(func(req *request) bool {\n            req.arrivalR -= rDecrement\n            return true\n        })\n    }\n}\n```\n\n---\n\n## 5. # APF QueueSet Subsystem: Deep-Dive Analysis

## Executive Summary

The Advanced Priority and Fairness (APF) QueueSet subsystem implements fair queuing for server requests in the Kubernetes API server. It provides isolation and prioritization across multiple request flows by managing queues with virtual time-based scheduling and shuffle-sharded queue selection. This document analyzes the architecture, algorithms, data flow, and operational characteristics of this critical subsystem.

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

The APF QueueSet subsystem ensures fair and prioritized execution of API server requests by:

1. **Preventing resource starvation**: Multiple flow classes (defined by flow schemas and flow distinguishers) share concurrency fairly
2. **Enforcing concurrency limits**: Ensuring total executing requests don't exceed server capacity
3. **Fair queueing**: When requests must wait, they're dispatched in a fair manner proportional to flow class share
4. **Queue isolation**: Using shuffle sharding to reduce head-of-line blocking across unrelated flows
