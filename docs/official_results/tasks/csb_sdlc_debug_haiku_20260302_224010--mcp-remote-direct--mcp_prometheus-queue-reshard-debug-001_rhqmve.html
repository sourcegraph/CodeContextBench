## Key Findings

**Root Cause:** A **race condition in the metrics counter management** during queue resharding. When new shards are started, the `pendingSamples` counter is immediately reset to 0 (via `Set(0)` at line 1241), but old shards are still running in background goroutines and decrementing this same counter. This creates a critical race condition.

**The Mechanism:**
1. Old shards begin shutting down via `stop()` which closes the `softShutdown` channel
2. Concurrently, `start()` is called which resets `pendingSamples` to 0 while new queues are created
3. If old shards hard-shutdown (exceed `flushDeadline`), they drop remaining samples and decrement the already-reset counter
4. New samples enqueued to new shards increment the counter, but the metric becomes unreliable

**Why It's Intermittent:**
- Only manifests when target discovery changes trigger resharding during active ingestion
- More likely with high sample rates, slow remote endpoints, or frequent resharding
- Timing-dependent on old shard flush duration relative to `flushDeadline`

**Evidence Points:**
- Lines 1237-1266: `start()` method resets counter too early
- Lines 1184-1199: `reshardLoop()` doesn't synchronize counter state
- Lines 1071-1074: `numShards` updated before actual resharding completes
- Lines 1563-1578: Old `runShard()` goroutines may decrement counter after hard shutdown

# Investigation Report: Remote-Write Queue Resharding Failure

## Summary

The intermittent stalling of remote-write shards after resharding is caused by a **race condition in the metrics counter reset** combined with **asynchronous queue shutdown timing**. When `start()` is called on new shards, it immediately resets `pendingSamples` to zero, but old shards are still in the background flushing data and decrementing this same counter. Additionally, samples enqueued during the transition window between `stop()` and `start()` may end up in queues that are being hard-shutdown before they can be sent.

## Root Cause

**Location:** `storage/remote/queue_manager.go:1237-1266` (`start()` method) and lines 1184-1199 (`reshardLoop()`)

**Mechanism:**

The resharding process has a critical flaw in how it manages the pending samples metric during queue transitions:

1. **Premature Counter Reset** (line 1241):
   ```go
   func (s *shards) start(n int) {
       s.mtx.Lock()
       defer s.mtx.Unlock()
       s.qm.metrics.pendingSamples.Set(0)  // <-- RESETS COUNTER IMMEDIATELY
       ...
   }
   ```
   This resets the counter to 0 immediately when new shards start.

2. **Race with Old Shards Cleanup**:
   ```go
   func (t *QueueManager) reshardLoop() {
       for {
           select {
           case numShards := <-t.reshardChan:
               t.shards.stop()         // Old shards close softShutdown, start flushing
               t.shards.start(numShards)  // New shards created, counter reset to 0
           ...
           }
       }
   }
   ```

   Between `stop()` and `start()`:
   - Old shards close `softShutdown` channel to block new enqueues
   - Old shards begin `FlushAndShutdown()` on their queues
   - If flush takes longer than expected, old shards hard-shutdown via context
