# Bug Investigation: Authentication Fails for Browser Cookie-Based Tokens

**Repository:** flipt-io/flipt
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

The authentication system in Flipt only accepts client tokens via the `Authorization` header using Bearer format. This prevents browser-based sessions from working, since browsers typically store authentication tokens in HTTP cookies.

Specifically:

1. **Cookie tokens rejected**: When a valid client token is stored in an HTTP cookie (under the key `flipt_client_token`), the authentication middleware does not extract or validate it. Requests that carry a valid token in a cookie but not in the Authorization header are rejected as unauthenticated.

2. **No server-level authentication bypass**: Certain internal servers (such as those handling delegated authentication flows) need to operate without authentication checks. There is currently no mechanism to configure the middleware to skip authentication for specific server instances, forcing all registered servers through the same authentication path regardless of their role.

## Investigation Summary

I've successfully created a regression test that demonstrates the two authentication bugs in the Flipt codebase:

### Bug 1: Cookie-based tokens are not extracted
**Location**: `/app/internal/server/auth/middleware.go` (lines 51-55)

The authentication middleware only checks the `authorization` header for Bearer tokens. It completely ignores HTTP cookies, which browsers naturally use for storing authentication tokens. This prevents browser-based sessions from working.

**Evidence**:
- Lines 16: The middleware only looks for the constant `authenticationHeaderKey = "authorization"`
- Lines 51-55: If the authorization header is missing/empty, authentication fails immediately without checking cookies
- The test demonstrates this by showing the same token works with Bearer header but fails when in a cookie

### Bug 2: No per-service authentication bypass
**Location**: `/app/cmd/flipt/main.go` (line 486)

The authentication interceptor is applied globally to ALL gRPC services with no way to configure exceptions. This means:
- Bootstrap/delegation services that need to operate without authentication can't
- There's no per-service configuration option to skip auth
- All services registered with the server go through the same interceptor chain

**Evidence**:
- The auth interceptor is registered once at the server level (line 486)
- All services (Flipt, Auth, TokenMethod) are registered on the same gRPC server that already has the interceptor
- There's no mechanism to selectively skip authentication for specific services
