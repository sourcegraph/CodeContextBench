## Summary of Findings

I've identified the key components of Django's middleware processing:

**Core Files:**
1. **django/core/handlers/base.py** - Contains `BaseHandler.load_middleware()` which reads `settings.MIDDLEWARE` in **reversed order** and builds the middleware stack
2. **django/utils/deprecation.py** - Contains `MiddlewareMixin` with `__call__()` and `__acall__()` methods that handle process_request/process_response
3. **django/core/handlers/wsgi.py** and **asgi.py** - WSGI and ASGI handler implementations
4. **django/core/wsgi.py** and **asgi.py** - Entry points that instantiate handlers

**Key Insights:**

1. **Middleware Stack Construction**: Django reverses the MIDDLEWARE setting and processes it, where each middleware wraps the next, creating an onion-like structure with the view resolution at the core.

2. **Process Request/Response Order**:
   - `process_request`: Called in **forward order** (outermost → innermost)
   - `process_response`: Called in **reverse order** (innermost → outermost)

3. **Django vs WSGI Comparison**: Unlike traditional WSGI middleware where all processing happens on entry/exit, Django separates hooks for process_request, process_view, process_template_response, and process_exception, allowing fine-grained control at different stages of request/response processing.
