## Architecture Overview

### Connection Pool Organization

Envoy's connection pools manage upstream HTTP/1.1, HTTP/2, TCP, and HTTP/3 connections. Pools are organized as follows:

- **Per-Cluster**: Each upstream cluster has its own connection pool
- **Per-Thread**: Connection pools are thread-safe and managed per dispatcher thread
- **Per-Host**: Within a cluster, separate pools exist for each upstream host
- **Protocol-Specific**: Different pool implementations for HTTP/1.1, HTTP/2, TCP, and HTTP/3

The connection pool hierarchy:
```
Cluster Manager
  └─ Cluster
     └─ Host
        └─ Connection Pool (per thread)
           └─ ActiveClients (individual connections)
           └─ PendingStreams (waiting for connection availability)
```

### Connection Lifecycle

Each connection progresses through well-defined states:

1. **Connecting**: TCP handshake in progress, TLS negotiation (if applicable)
2. **ReadyForEarlyData**: HTTP/3 only - connection accepts 0-RTT requests
3. **Ready**: Connection established, ready to receive new streams
4. **Busy**: Connection at concurrent stream limit, no new streams accepted
5. **Draining**: Connection marked for closure after active streams complete
6. **Closed**: Connection destroyed, queued for deferred deletion
