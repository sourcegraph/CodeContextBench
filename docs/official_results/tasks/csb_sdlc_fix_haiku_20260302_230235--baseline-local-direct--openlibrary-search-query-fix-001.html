# Work Search Query Processing Fix - Implementation Guide

## Executive Summary

This document describes the fixes applied to the OpenLibrary work search query processing to handle edge-case inputs that caused parse errors or unintended query behavior.

**Status**: ✅ All fixes implemented and verified
**Tests**: ✅ All 26 existing tests pass
**Edge Cases**: ✅ All tested edge cases handled correctly

## Problem Statement

The OpenLibrary work search feature was failing to properly normalize and escape raw user queries before sending them to Solr. This caused issues with:

1. **Trailing dashes**: Queries like `Horror-` would fail because the dash was interpreted as a Lucene negation operator
2. **Operator-like tokens**: Queries ending with `AND` or `OR` would fail parsing
3. **Quoted phrases**: Phrases containing special characters were not properly escaped
4. **ISBN strings**: ISBN-like inputs with dashes needed proper handling

## Root Causes

### Issue 1: Regex Pattern Bug in `fully_escape_query()`

**File**: `/app/openlibrary/solr/query_utils.py`
**Line**: 124

**Problem**: The character class in the regex was incorrectly written:
```python
escaped = re.sub(r'[\[\]\(\)\{\}:"-+?~^/\\,]', r'\\\g<0>', escaped)
```

In this pattern, the sequence `"-+` is interpreted as a range from `"` (ASCII 34) to `+` (ASCII 43), which does NOT include the dash character `-` (ASCII 45). This meant dashes were never being escaped.

**Solution**: Move the dash to the beginning of the character class where it's treated as a literal:
```python
escaped = re.sub(r'[-\[\]\(\)\{\}:"+?~^/\\,*&|]', r'\\\g<0>', escaped)
```

# Work Search Query Processing Fix - Solution Summary

## Problem Analysis

The OpenLibrary work search functionality had issues with edge-case user inputs that caused:
1. Parse errors when queries had trailing dashes (e.g., "Horror-")
2. Misinterpretation of special characters by the Solr query parser
3. Unintended query behavior due to operator-like tokens

## Root Causes Identified

### Issue 1: Dash Character Not Properly Escaped
In `/app/openlibrary/solr/query_utils.py`, the regex pattern in `fully_escape_query()` was incorrectly written:
```python
escaped = re.sub(r'[\[\]\(\)\{\}:"-+?~^/\\,]', r'\\\g<0>', escaped)
```

The dash `-` was being interpreted as a range operator from `"` (quote) to `+` (plus), rather than as a literal dash character. This meant queries like "Horror- AND movies" would fail because the trailing dash was interpreted as a Lucene negation operator, making the query invalid.

### Issue 2: Missing Preprocessing in process_user_query
In `/app/openlibrary/plugins/worksearch/code.py`, the `process_user_query()` function was not escaping special characters like `-`, `*`, `+`, `&`, and `|` before parsing, which could cause these to be misinterpreted as Lucene operators.
