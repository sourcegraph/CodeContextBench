## Summary

I found that Firefox's CSS calc() overflow issue involves two main layers:

**Servo (Rust) CSS Engine:**
1. **`servo/components/style/values/specified/calc.rs`** - Parses CSS calc() expressions, explicitly allowing `infinity`, `-infinity`, and `nan` values (lines 681-683)
2. **`servo/components/style/values/generics/calc.rs`** - The core calc node resolution engine where arithmetic operations (+, *, min, max, clamp) can produce overflow/infinity
3. **`servo/components/style/values/computed/length_percentage.rs`** - Resolves calc length-percentage values where multiplication `basis.px() * percentage` can produce infinity
4. **`servo/components/style_traits/values.rs`** - Contains the `AllowedNumericType::clamp()` function that fails to properly handle infinity values

**Gecko (C++) Integration:**
5. **`layout/style/ServoStyleConstsInlines.h`** - The FFI boundary where Gecko receives potentially infinite values from Servo via `StyleCalcLengthPercentage::ResolveToCSSPixels()` and `Resolve()`

The crash occurs because extreme values (infinity) from calc() expressions propagate through the arithmetic operations without proper validation, and the normalization/clamping logic doesn't properly constrain these values before they're used in layout calculations.
