### Key Findings:

**Root Cause:** A race condition in `storage/remote/queue_manager.go` where shard assignments change during resharding, causing samples from the same series to be routed to different shards.

**The Mechanism:**
1. Samples are assigned to shards using modulo arithmetic: `shard = ref % numShards` (line 1315)
2. When resharding occurs, `reshardLoop()` calls `stop()` then `start()` (lines 1193-1194)
3. Samples in retry backoff loops get `return false` from `enqueue()` when old shards close
4. While they sleep, new shards are created with different shard counts
5. On retry, the same series ref calculates to a **different shard number**
6. This violates ordering guarantees and causes shards to stall

**Why It's Intermittent:**
- Only manifests when resharding happens while samples are actively being retried
- Depends on timing alignment between the retry backoff sleep and the resharding window
- Different shard count changes trigger it (4→6 shards: ref=10 changes from shard 2 to shard 4)
