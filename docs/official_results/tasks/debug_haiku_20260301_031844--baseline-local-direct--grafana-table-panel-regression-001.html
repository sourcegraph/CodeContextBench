### Key Finding

**The bug is in the conditional logic of `processPanelsV38()` function** (line 120-123 of `/workspace/apps/dashboard/pkg/migration/schemaversion/v38.go`).

The code has a premature `continue` statement that skips the `migrateOverrides()` call whenever `fieldConfig.defaults.custom` doesn't exist. This prevents field configuration overrides from being migrated to the new `cellOptions` format.

### Core Issue

The backend migration incorrectly couples override migration to defaults migration:
- Dashboards WITH explicit `defaults.custom` → overrides are migrated ✓
- Dashboards WITHOUT explicit `defaults.custom` → overrides are skipped ✗

But overrides can exist independently. The frontend migration (DashboardMigrator.ts) correctly handles this by processing overrides with a separate conditional check.

### Why It Broke Dashboards

1. Dashboard imported from v10.3 without explicit `defaults.custom`
2. Backend v38 migration skips override processing
3. Dashboard saved as schema v38 with unmigrated overrides still using `custom.displayMode`
4. Frontend expects `custom.cellOptions` but finds deprecated `custom.displayMode`
5. Table formatting fails (column widths, text alignment, etc. don't apply)

### Missing Test Coverage

The test suite only covers dashboards with explicit `defaults.custom`. There are **no test cases** for the regression scenario (missing `defaults.custom` with overrides present).

The complete investigation report is ready for review at `/logs/agent/investigation.md`.

The V38 schema migration for table panels has a critical bug where field configuration override migrations are skipped when `fieldConfig.defaults.custom` is not explicitly set in saved dashboard JSON. This causes field overrides referencing `custom.displayMode` to remain unmigrated while the frontend expects `custom.cellOptions`, resulting in broken table formatting.

**File:** `/workspace/apps/dashboard/pkg/migration/schemaversion/v38.go`
**Function:** `processPanelsV38()` (lines 90-137)
**Buggy Code:** Lines 120-123
