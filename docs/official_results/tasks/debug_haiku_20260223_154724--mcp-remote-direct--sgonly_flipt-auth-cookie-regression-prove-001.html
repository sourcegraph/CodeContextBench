## Reported Issue

The authentication system in Flipt only accepts client tokens via the `Authorization` header using Bearer format. This prevents browser-based sessions from working, since browsers typically store authentication tokens in HTTP cookies.

Specifically:

1. **Cookie tokens rejected**: When a valid client token is stored in an HTTP cookie (under the key `flipt_client_token`), the authentication middleware does not extract or validate it. Requests that carry a valid token in a cookie but not in the Authorization header are rejected as unauthenticated.

2. **No server-level authentication bypass**: Certain internal servers (such as those handling delegated authentication flows) need to operate without authentication checks. There is currently no mechanism to configure the middleware to skip authentication for specific server instances, forcing all registered servers through the same authentication path regardless of their role.

## Investigation Summary

I found **two authentication issues** in the Flipt codebase (github.com/sg-benchmarks/flipt--3d5a345f):

### Issue 1: Cookie Token Extraction Bug
**Location:** `internal/server/authn/method/http.go`, lines 45-54

The `ForwardCookies` function has a bug where it **always stores cookies under the wrong metadata key**:
- It loops through both `stateCookieKey` and `tokenCookieKey`
- But **always stores** under `md[stateCookieKey]`, regardless of which cookie is being processed
- This means the `flipt_client_token` cookie is never added to metadata with the correct key
- Result: Browser-based tokens in HTTP cookies cannot be authenticated

**The bug:**
```go
md[stateCookieKey] = []string{cookie.Value}  // Wrong - always uses state key
```

**Should be:**
```go
md[key] = []string{cookie.Value}  // Correct - use the loop variable
```

### Issue 2: Server Skip Authentication Mechanism
**Location:** `internal/server/authn/middleware/grpc/middleware.go`

While the infrastructure exists for servers to skip authentication (via `SkipsAuthenticationServer` interface and `WithServerSkipsAuthentication` option), servers must be **explicitly configured** or they will require authentication for all requests. This could break internal services like OIDC callbacks.

# Flipt Authentication Bug Investigation

## Executive Summary

This investigation identified two critical authentication issues in Flipt's authentication middleware:

1. **Cookie-Based Token Extraction Bug** - HTTP cookies carrying authentication tokens are not properly forwarded from HTTP requests to gRPC metadata due to a bug in the `ForwardCookies` function.

2. **Server-Level Authentication Skip Mechanism** - While infrastructure exists to allow servers to skip authentication, the mechanism may not be consistently applied across all internal services that require it.

## Issue 1: Cookie Token Not Extracted from HTTP Cookies

### Root Cause
**File:** `internal/server/authn/method/http.go`, lines 45-54
**Function:** `ForwardCookies`

The `ForwardCookies` function contains a bug where it always stores cookie values under the `stateCookieKey` metadata key, regardless of which cookie is being processed:

```go
func ForwardCookies(ctx context.Context, req *http.Request) metadata.MD {
    md := metadata.MD{}
    for _, key := range []string{stateCookieKey, tokenCookieKey} {
        if cookie, err := req.Cookie(key); err == nil {
            md[stateCookieKey] = []string{cookie.Value}  // BUG: Always uses stateCookieKey
        }
    }
    return md
}
```

### Impact
- Browser-based clients that send authentication tokens via HTTP cookies cannot authenticate
- The `flipt_client_token` cookie is stored under the wrong metadata key (`flipt_client_state`)
- gRPC middleware cannot extract the token because it looks for the correct key

### Expected Behavior
When iterating through cookie keys, the function should use the current loop variable:
```go
md[key] = []string{cookie.Value}  // Correct: uses the actual key
```
