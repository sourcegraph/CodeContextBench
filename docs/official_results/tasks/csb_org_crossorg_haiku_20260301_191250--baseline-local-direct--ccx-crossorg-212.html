## Summary of Findings

**Envoy Proxy (C++):**
- **Local Rate Limiting**: Token bucket implementation in `source/extensions/filters/http/local_ratelimit/` with the `FilterConfig` class managing fill intervals and max tokens
- **Global Rate Limiting**: gRPC service client (`GrpcClientImpl`) in `source/extensions/filters/common/ratelimit/ratelimit_impl.cc` that communicates with external RateLimitService

**Kubernetes (Go):**
- **Token Bucket**: Client-side rate limiter in `staging/src/k8s.io/client-go/util/flowcontrol/throttle.go` with the `tokenBucketPassiveRateLimiter` wrapping golang's `rate.Limiter`
- **APF Controller**: Manages PriorityLevel resources and lifecycle in `staging/src/k8s.io/apiserver/pkg/util/flowcontrol/apf_controller.go`
- **APF Filter**: Main request handling interface in `staging/src/k8s.io/apiserver/pkg/util/flowcontrol/apf_filter.go`
- **Fair Queuing**: Queue management in `staging/src/k8s.io/apiserver/pkg/util/flowcontrol/fairqueuing/queueset/queueset.go`

Both systems use token bucket algorithms as their core rate limiting mechanism, with Envoy applying it at the gateway/filter level and Kubernetes applying it at the API server level with sophisticated priority and fairness handling.
