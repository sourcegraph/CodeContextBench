## Key Findings

### Factory Registration
- Filters implement **NamedHttpFilterConfigFactory** (defined in `envoy/server/filter_config.h`)
- Registered globally via `REGISTER_FACTORY` macro
- Each factory provides `createFilterFactoryFromProto()` that returns a `FilterFactoryCb` callback

### Configuration Processing
- **FilterChainHelper** in `source/common/http/filter_chain_helper.h` processes HTTP filter configs during HCM initialization
- Retrieves factories from the registry and calls `createFilterFactoryFromProto()`
- Stores the resulting callbacks in `FilterFactoriesList`

### Filter Chain Creation
- **FilterChainUtility::createFilterChainForFactories()** applies stored callbacks via `FilterChainManager::applyFilterFactoryCb()`
- Called when HCM is configured with filters

### Per-Request Instantiation
- **FilterManager::applyFilterFactoryCb()** (line 551 in filter_manager.cc) is called per stream
- Creates **FilterChainFactoryCallbacksImpl** which invokes the stored factory callback
- Callbacks create actual filter instances via:
  - `addStreamDecoderFilter()` - decoder-only filters
  - `addStreamEncoderFilter()` - encoder-only filters
  - `addStreamFilter()` - dual encoder/decoder filters
- Filter instances are wrapped in **ActiveStreamDecoderFilter** and **ActiveStreamEncoderFilter** objects for lifecycle management
