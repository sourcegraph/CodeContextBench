Source: `transform_destroy_edge.go`\n\n7. **CreateBeforeDestroyTransformer** - Handles create-before-destroy lifecycle\n   - Modifies graph for resources with `create_before_destroy = true`\n   - Ensures new resource is created before old is destroyed\n   - Source: `transform_destroy_cbd.go`\n\n8. **TargetTransformer** - Implements resource targeting\n   - Removes nodes not targeted with `-target` flag\n   - Keeps dependencies of targeted resources\n   - Source: `transform_targets.go`\n\n9. **DiffTransformer** - Applies plan changes as constraints\n   - Only used during apply\n   - Ensures only planned changes are applied\n   - Source: `transform_diff.go`\n\n10. **ExpandTransformer** - Expands nodes with multiple instances\n    - Handles count and for_each expansion\n    - Creates individual nodes per instance\n    - Source: `transform_expand.go`\n\n### Dynamic Expansion\n\nResources with `count` or `for_each` undergo dynamic expansion:\n\n1. **Graph Construction Phase:** Initial node represents the entire resource\n   - `nodeExpandPlannableResource` for plan\n   - `nodeExpandApplyableResource` for apply\n\n2. **DynamicExpand Phase:** During graph walk, expansion node evaluates count/for_each\n   - Calls `instances.Expander.ExpandResource()`\n   - Creates sub-graph with individual instance nodes\n   - One node per concrete instance\n\n3. **Instance Evaluation:** Individual instance nodes execute\n   - Each instance is treated as independent resource\n   - Instance-specific state is accessed via address (e.g., `aws_instance.foo[0]`)\n\nThe `instances.Expander` tracks:\n- Module instances (from module count/for_each)\n- Resource instances (from resource count/for_each)\n- Partially-expanded references (when expansion is unknown due to dynamic expressions)\n\n### BasicGraphBuilder (`internal/terraform/graph_builder.go`)\n\nAll graph builders inherit from `BasicGraphBuilder`, which applies a sequence of transforms:\n\n```go\ntype BasicGraphBuilder struct {\n    Steps []GraphTransformer  // Sequence of transforms to apply\n    Name  string              // Debug name\n}\n\nfunc (b *BasicGraphBuilder) Build(path addrs.ModuleInstance) (*Graph, tfdiags.Diagnostics) {\n    g := &Graph{Path: path}\n\n    for _, step := range b.Steps {\n        if err := step.Transform(g); err != nil {\n            return nil, diags.Append(err)\n        }\n    }\n\n    if err := g.Validate(); err != nil {\n        return nil, diags.Append(err)\n    }\n\n    return g, diags\n}\n```\n\nThe builder validates that the resulting graph is acyclic and well-formed before returning.\n\n---\n\n## Provider Interface and Lifecycle\n\n### Provider Architecture\n\nTerraform delegates all infrastructure operations to **providers**, which are plugins loaded via the gRPC-based plugin interface. The provider interface abstracts the details of interacting with specific cloud platforms, APIs, or systems.\n\n### Provider Initialization Pipeline\n\nProviders go through the following lifecycle during execution:\n\n#### 1. Plugin Loading (`internal/terraform/context_plugins.go`)\n\nThe `contextPlugins` structure manages provider and provisioner plugin discovery and loading:\n\n```go\ntype contextPlugins struct {\n    providers map[addrs.Provider]providers.Factory\n    // ...\n}\n```\n\n**Provider Factory** is a function that creates a provider instance:\n```go\ntype Factory func() (Interface, error)\n```\n\n#### 2. Schema Fetching\n\nBefore any operation, Terraform fetches the provider schema, which describes:\n- Available resources and their arguments/attributes\n- Available data sources\n- Provider configuration arguments\n\nSchema is cached in the `ContextGraphWalker` to avoid redundant gRPC calls:\n```go\nproviderSchemas map[string]providers.ProviderSchema  // Cached schemas\n```\n\n#### 3. Provider Initialization (`internal/terraform/eval_provider.go`)\n\nDuring graph execution, provider nodes (`NodeApplyableProvider`) trigger initialization:\n\n```go\n// InitProvider creates a new provider instance\ntype NodeAbstractProvider struct {\n    Addr addrs.AbsProviderConfig\n    // ...\n}\n```\n\nThe process:\n1. Factory function is called to create provider instance\n2. `ConfigureProvider` is called with configuration values\n3. Provider is cached in the `BuiltinEvalContext.ProviderCache`\n4. Subsequent resources using this provider retrieve the cached instance\n\n#### 4. Provider Configuration\n\nProvider configuration evaluation happens in two phases:\n\n**Phase 1: Configuration Evaluation** - Expressions in the provider block are evaluated\n- Depends on variables, locals, and data sources\n- Can use interpolation syntax\n\n**Phase 2: Provider Configuration** - Evaluated values are sent to provider\n```go\nfunc (ctx *BuiltinEvalContext) ConfigureProvider(\n    addr addrs.AbsProviderConfig,\n    config cty.Value,\n) tfdiags.Diagnostics {\n    // Send configuration to provider instance\n    provider := ctx.Provider(addr)\n    return provider.Configure(ctx.StopCtx(), &providers.ConfigureProviderRequest{\n        Config: config,\n    }).Diagnostics\n}\n```\n\n### Resource Operations Delegation\n\nWhen a resource node executes, it delegates to the provider:\n\n#### Plan Phase\n```\nResource Node (Plan)\n    \u2193\nEvaluate resource arguments\n    \u2193\nCall provider.PlanResourceChange()\n    \u2193\nProvider returns proposed new state\n    \u2193\nStore in plans.Change\n```\n\nThe provider's `PlanResourceChange` method:\n- Receives prior state and proposed new state\n- Returns the planned action (Create, Update, Delete, NoOp)\n- Can suggest changes beyond the configuration (e.g., computed fields)\n\n#### Apply Phase\n```\nResource Node (Apply)\n    \u2193\nRetrieve planned change from plan\n    \u2193\nCall provider.ApplyResourceChange()\n    \u2193\nProvider applies change to real infrastructure\n    \u2193\nReturn new state\n    \u2193\nUpdate state in SyncState\n```\n\nThe provider's `ApplyResourceChange` method:\n- Actually creates/modifies/deletes infrastructure\n- Returns new state after successful operation\n- Throws errors if operation fails\n\n### Provider Instance Management\n\nProvider instances are cached per module path and provider address to avoid redundant initialization:\n\n```go\ntype ContextGraphWalker struct {\n    providerCache map[string]providers.Interface\n    providerLock  sync.Mutex  // Protects concurrent access\n    // ...\n}\n```\n\n**Key properties:**\n- **Keyed by address**: `<provider>.<alias>` or just `<provider>` for default\n- **Scope-aware**: Each module instance scope can have its own provider instances\n- **Thread-safe**: Lock protects concurrent provider initialization\n- **Lifecycle-managed**: Providers are closed after graph walk completes\n\n### Provider Configuration Inheritance\n\nChild modules can inherit provider configurations from parents:\n\n1. **Configuration block declaration**: Child module declares `provider` in `required_providers`\n2. **Passing configuration**: Parent module passes provider instance to child\n3. **Resolution**: Child uses passed provider or declares its own\n\nThis enables:\n- Shared provider instances across modules\n- Alias handling (multiple configurations of same provider type)\n- Override capability in root module\n\n---\n\n## State Management\n\n### State Structure (`internal/states/`)\n\nTerraform's state is organized hierarchically:\n\n```\nState\n  \u251c\u2500\u2500 Modules\n  \u2502   \u251c\u2500\u2500 Module (root)\n  \u2502   \u2502   \u251c\u2500\u2500 Resources\n  \u2502   \u2502   \u2502   \u251c\u2500\u2500 Resource\n  \u2502   \u2502   \u2502   \u2502   \u2514\u2500\u2500 Instances\n  \u2502   \u2502   \u2502   \u2502       \u251c\u2500\u2500 Current Object\n  \u2502   \u2502   \u2502   \u2502       \u2514\u2500\u2500 Deposed Objects (for replace)\n  \u2502   \u2502   \u2502   \u2514\u2500\u2500 Resource\n  \u2502   \u2502   \u251c\u2500\u2500 Outputs\n  \u2502   \u2502   \u2514\u2500\u2500 Child Modules\n  \u2502   \u2514\u2500\u2500 Module\n  \u2514\u2500\u2500 CheckResults\n```\n\n**Key types:**\n- `states.State` - Top-level state container\n- `states.Module` - Module-level state\n- `states.Resource` - Resource definition\n- `states.ResourceInstance` - Individual instance (after count/for_each expansion)\n- `states.ResourceInstanceObject` - Actual resource data\n\n### Concurrent State Access (`internal/states/sync.go`)\n\nDuring graph execution, multiple nodes may update state concurrently. The `SyncState` wrapper provides thread-safe access:\n\n```go\ntype SyncState struct {\n    state    *State\n    writable bool\n    lock     sync.RWMutex\n}\n```\n\n**Design pattern:**\n- **Read operations**: Copy data out while holding read lock\n- **Write operations**: Atomic updates to minimize lock contention\n- **Deep copying**: Returned values are copies, preventing external mutations\n\n**Common operations:**\n```go\n// Read with automatic deep copy\ninstance := syncState.ResourceInstance(addr)\n\n// Write with atomic update\nsyncState.SetResourceInstanceCurrent(addr, obj)\n\n// Remove completely\nsyncState.RemoveResourceInstanceObject(addr)\n```\n\n### State Snapshots\n\nTerraform maintains multiple state snapshots during execution to enable correct dependency resolution:\n\n1. **Input State** (`PrevRunState`)\n   - State from previous execution\n   - Read-only, used only as reference\n   - Preserved for outputs and diagnostics\n\n2. **Refresh State** (`RefreshState`)\n   - Created by copying input state\n   - Updated during refresh phase\n   - Reflects current real-world state\n   - Used to detect drift\n\n3. **Working State** (`State`)\n   - Deep copy of input state\n   - Modified during plan/apply\n   - Final state returned to caller\n   - Persisted to state file\n\nThis multi-snapshot approach enables:\n- Detecting state drift (refresh vs input)\n- Proper diagnostics (what changed vs what was planned)\n- Rollback if operation fails mid-way\n\n### State Persistence\n\nState is persisted via **state managers** (`internal/states/statemgr/`):\n\n```go\n// State managers implement this interface\ntype Full interface {\n    State() (*states.State, error)\n    WriteState(*states.State) error\n    DeleteState() error\n    Lock(info *LockInfo) (string, error)\n    Unlock(id string) error\n    // ...\n}\n```\n\n**Implementations:**\n- `Filesystem` - Local `terraform.tfstate` file\n- `Remote` - Remote backends (S3, Consul, etc.)\n- `Cloud` - HCP Terraform/Terraform Cloud\n\n**Key operations:**\n1. **Read** - Load state from backend before plan/apply\n2. **Validate** - Check for lock (prevent concurrent operations)\n3. **Lock** - Acquire lock to prevent concurrent modifications\n4. **Write** - Persist updated state after apply\n5. **Unlock** - Release lock to allow other operations\n\n### State Locking\n\nFor remote backends, state locking prevents concurrent modifications:\n\n```go\n// Acquire lock before modifying state\nlockID, err := stateManager.Lock(&statemgr.LockInfo{\n    Operation:  \"apply\",\n    Info:       \"terraform apply\",\n    Who:        \"user@host\",\n    Version:    version.String(),\n    Created:    time.Now(),\n})\n\n// Release after operation completes\ndefer stateManager.Unlock(lockID)\n```\n\nBackends implement locking via their native mechanisms (DynamoDB for S3, etc.), preventing multiple users from running apply simultaneously.\n\n### Check Results\n\nState also stores check results from `check` blocks:\n\n```go\ntype State struct {\n    CheckResults *checks.State\n}\n```\n\nThese persist between operations to:\n- Track check status changes\n- Enable pre-destroy checks\n- Support check reporting in outputs\n\n---\n\n## Execution Flow and Hook System\n\n### High-Level Execution Flow\n\n#### Plan Operation (`Context.Plan()`)\n\n```\n1. Validate external provider configurations\n2. Build PlanGraphBuilder\n3. Apply transformers to build graph\n4. Create graphWalkOpts with:\n   - InputState (prior state)\n   - Config (desired configuration)\n   - Changes (empty, to be filled)\n5. Walk graph with ContextGraphWalker\n   For each node in parallel:\n   - Evaluate expressions\n   - Refresh state (unless skip_refresh)\n   - Call provider.PlanResourceChange()\n   - Record planned change\n6. Return plan containing changes\n```\n\n#### Apply Operation (`Context.Apply()`)\n\n```\n1. Validate plan integrity\n2. Build ApplyGraphBuilder from plan changes\n3. Apply transformers to build graph\n4. Create graphWalkOpts with:\n   - InputState (prior state)\n   - Config (configuration from plan)\n   - Changes (from plan, to be applied)\n5. Walk graph with ContextGraphWalker\n   For each node in parallel:\n   - Evaluate expressions\n   - Retrieve planned change\n   - Call provider.ApplyResourceChange()\n   - Update state with new values\n6. Return updated state\n```\n\n### Graph Walking (`internal/terraform/graph_walk.go`)\n\nThe graph walk is the core execution engine. The `Graph.Walk()` method traverses the DAG:\n\n```go\nfunc (g *Graph) Walk(walker GraphWalker) tfdiags.Diagnostics {\n    // Get evaluation context\n    ctx := walker.EvalContext()\n\n    // Walk function called for each vertex\n    walkFn := func(v dag.Vertex) tfdiags.Diagnostics {\n        // Call walker to execute node\n        return walker.Execute(ctx, v.(GraphNodeExecutable))\n    }\n\n    // Use DAG walk (topological sort, parallel where possible)\n    return g.AcyclicGraph.Walk(walkFn)\n}\n```\n\n**Execution characteristics:**\n- **Dependency order**: Nodes with no unmet dependencies execute first\n- **Parallelism**: Nodes with no dependencies on each other run in parallel\n- **Resource limit**: Parallelism is limited by `parallelism` context option (default 10)\n- **Error handling**: Errors are collected; dependent nodes wait before executing\n\n### ContextGraphWalker (`internal/terraform/graph_walk_context.go`)\n\nThe walker orchestrates the execution environment for each node:\n\n```go\ntype ContextGraphWalker struct {\n    Context                 *Context\n    State                   *states.SyncState\n    RefreshState            *states.SyncState\n    PrevRunState            *states.SyncState\n    Changes                 *plans.ChangesSync\n    Checks                  *checks.State\n    NamedValues             *namedvals.State\n    InstanceExpander        *instances.Expander\n    Deferrals               *deferring.Deferred\n    EphemeralResources      *ephemeral.Resources\n    Operation               walkOperation\n    // ...\n}\n```\n\n**Key methods:**\n\n1. **EvalContext()** - Returns evaluation context for nodes\n   ```go\n   func (w *ContextGraphWalker) EvalContext() EvalContext {\n       // Create BuiltinEvalContext with:\n       // - Hook callbacks\n       // - Provider/provisioner cache\n       // - State snapshots\n       // - Expression evaluator\n   }\n   ```\n\n2. **enterScope(scope)** - Returns context for specific module/scope\n   - Caches contexts per scope for reuse\n   - Enables module-level isolation\n\n3. **Execute(ctx, node)** - Executes a single graph node\n   ```go\n   func (w *ContextGraphWalker) Execute(\n       ctx EvalContext,\n       n GraphNodeExecutable,\n   ) tfdiags.Diagnostics {\n       // Acquire semaphore (parallelism limit)\n       w.Context.parallelSem.Acquire()\n       defer w.Context.parallelSem.Release()\n\n       // Delegate to node\n       return n.Execute(ctx, w.Operation)\n   }\n   ```\n\n### Node Execution\n\nEach graph node implements the `GraphNodeExecutable` interface:\n\n```go\ntype GraphNodeExecutable interface {\n    Execute(EvalContext, walkOperation) tfdiags.Diagnostics\n}\n```\n\nDifferent node types execute differently based on operation type:\n\n#### Resource Nodes\n\n**Plan Phase:**\n- `nodeResourcePlanInstance` evaluates resource arguments\n- Calls `provider.PlanResourceChange()`\n- Records change in `plans.Changes`\n\n**Apply Phase:**\n- `nodeResourceApplyInstance` retrieves planned change\n- Calls `provider.ApplyResourceChange()`\n- Updates state\n\n**Destroy Phase:**\n- `nodeResourceDestroyInstance` creates destroy action\n- Calls `provider.ApplyResourceChange()` with null state\n- Removes from state\n\n#### Provider Nodes\n\n- `NodeApplyableProvider` initializes provider\n- Evaluates provider configuration\n- Calls `provider.Configure()`\n- Caches provider instance\n\n#### Module/Variable Nodes\n\n- `NodeRootVariable` evaluates root module variables\n- `NodeModuleVariable` evaluates module input variables\n- `NodeOutput` evaluates output expressions\n- `NodeLocal` evaluates local value expressions\n\n### EvalContext Interface (`internal/terraform/eval_context.go`)\n\nThe `EvalContext` is the execution environment provided to each node. It provides access to:\n\n```go\ntype EvalContext interface {\n    // Stop context for cancellation\n    StopCtx() context.Context\n\n    // Current module path\n    Path() addrs.ModuleInstance\n\n    // Hook callbacks\n    Hook(func(Hook) (HookAction, error)) error\n\n    // Provider operations\n    InitProvider(addrs.AbsProviderConfig, *configs.Provider) (providers.Interface, error)\n    Provider(addrs.AbsProviderConfig) providers.Interface\n    ConfigureProvider(addrs.AbsProviderConfig, cty.Value) tfdiags.Diagnostics\n    ProviderSchema(addrs.AbsProviderConfig) (providers.ProviderSchema, error)\n\n    // Evaluation context\n    Evaluator() *Evaluator\n    InstanceExpander() *instances.Expander\n    State() *states.SyncState\n    RefreshState() *states.SyncState\n\n    // Output access\n    Changes() *plans.ChangesSync\n    Checks() *checks.State\n    // ...\n}\n```\n\n### Hook System\n\nHooks provide extension points throughout execution. They're callbacks invoked at key moments:\n\n```go\ntype Hook interface {\n    // Pre/post lifecycle hooks\n    PreApply(id HookResourceIdentity, dk addrs.DeposedKey, ...) (HookAction, error)\n    PostApply(id HookResourceIdentity, dk addrs.DeposedKey, ...) (HookAction, error)\n\n    PreDiff(id HookResourceIdentity, dk addrs.DeposedKey, ...) (HookAction, error)\n    PostDiff(id HookResourceIdentity, dk addrs.DeposedKey, ...) (HookAction, error)\n\n    PreRefresh(id HookResourceIdentity, dk addrs.DeposedKey, ...) don't modify\n- **Apply**: Execute changes; modify state and infrastructure\n- **Destroy**: Remove resources; delete from state\n- **Refresh**: Update state from real infrastructure\n\n### Deferred Actions\n\nTerraform supports deferring changes when some dependencies cannot be determined:\n\n```go\ntype Deferred struct {\n    // Tracks deferred resource changes\n    resources map[addrs.AbsResourceInstance]*DeferredResourceInstanceChangeSrc\n}\n```\n\n**When deferring occurs:**\n- Dynamic expressions in dependencies can't be evaluated\n- Child stack instances can't be fully determined\n- Sensitive data might be exposed\n\n**Behavior:**\n- Deferred nodes skip execution\n- Plan marks actions as deferred\n- Apply skips deferred actions\n- User manually applies when dependencies resolve\n\n---\n\n## Extension Points\n\n### Custom Hooks\n\nImplement the `Hook` interface to integrate with execution:\n\n```go\ntype CustomHook struct {\n    // Implement Hook methods for specific events\n}\n\nfunc (h *CustomHook) PostApply(...) Calculate minimum set of operations\n- **Parallelism**: Execute independent operations simultaneously\n- **Determinism**: Reproducible execution order across runs\n- **Validation**: Detect cycles and impossible orderings\n- **Flexibility**: Support various operations (plan, apply, destroy, refresh)\n\n### Why Multiple State Snapshots\n\nMultiple snapshots enable:\n- **Drift detection**: Compare prior vs refreshed state\n- **Diagnostics**: Show before/after in messages\n- **Atomicity**: Consistent view across operation\n- **Rollback: Revert to input if operation fails mid-way\n\n### Why ## Architecture Overview

Terraform's plan/apply pipeline follows a graph-based execution model where:

1. **Configuration** is parsed into a static model
2. **Graph Builder** constructs a directed acyclic graph (DAG) representing dependencies
3. **Graph Transformers** modify the graph to add additional logic and constraints
4. **Graph Walker** traverses the graph in dependency order, executing nodes in parallel where possible
5. **State** is updated atomically as resources are created, modified, or destroyed

The main entry point is the `Context` struct (`internal/terraform/context.go`), which orchestrates the entire pipeline. Context methods like `Plan()` and `Apply()` construct a graph using the appropriate graph builder and then walk that graph to execute the operation.
