## Summary of Findings

I identified the **Startable/Stoppable service lifecycle pattern** in Apache Kafka across two major areas:

### **Scala-based Server Infrastructure** 
The core lifecycle is defined by the **Server trait** (in `kafka/server/Server.scala`) with three methods:
- `startup()` - initiate startup
- `shutdown()` - initiate shutdown  
- `awaitShutdown()` - wait for completion

**KafkaRaftServer** orchestrates dependent services with proper startup/shutdown ordering:
- **Startup**: Controller first, then Broker
- **Shutdown**: Broker first (reverse order), then Controller

### **Detailed Implementation in BrokerServer**
The **BrokerServer** contains an extensive lifecycle with components started in dependency order and stopped in reverse order, including:
- Scheduler, Log Manager, Socket Server
- Replica Manager, Coordinators (group, transaction, share)
- Various managers and metadata publishers

### **Connect Framework Pattern**
The **Herder interface** and **AbstractHerder** class implement coordinated startup/shutdown of dependent services:
- **startServices()**: Worker → StatusBackingStore → ConfigBackingStore
- **stopServices()**: StatusBackingStore → ConfigBackingStore → Worker (reverse order)

### **Key Patterns**
1. ✅ Dependent services start after dependencies, stop before them
2. ✅ State machine (SHUTDOWN → STARTING → STARTED → SHUTTING_DOWN)
3. ✅ Reverse-order shutdown ensures invariants are maintained
4. ✅ Fault-tolerant cleanup (failures don't cascade)
5. ✅ Future-based coordination for async milestones
