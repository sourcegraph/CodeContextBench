added/updated/deleted\n```\n\nThe state manager tracks changes:\n- **Addition**: Resource created with new attributes\n- **Modification**: Attributes changed (deep merge with existing)\n- # Terraform Plan/Apply Pipeline Architecture

## Overview

Terraform's plan/apply pipeline is built around a sophisticated directed acyclic graph (DAG) execution model. The core architecture separates concerns into several key systems: graph construction, provider management, state handling, and execution orchestration. This document provides a technical deep-dive into how these components interact to deliver Terraform's declarative infrastructure-as-code paradigm.

The pipeline has three primary phases:
1. **Configuration & Graph Construction**: Parse configuration and build an execution graph
2. **Plan Phase**: Analyze differences between desired and current state without making changes
3. **Apply Phase**: Execute graph nodes to realize the desired state

---

## 1. Graph Builder System

### 1.1 Architecture Overview

The graph builder constructs a Directed Acyclic Graph (DAG) that represents the complete dependency structure of a Terraform configuration. Each node in the graph represents an actionable unit (resource, data source, provider, etc.), and edges represent dependencies.

**Key Design Principles:**
- All operations execute as graph nodes with explicit dependencies
- The graph enables parallelization of independent operations
- Transformers add implicit nodes (providers, dependencies) after initial construction
- Graph validation prevents circular dependencies before execution
