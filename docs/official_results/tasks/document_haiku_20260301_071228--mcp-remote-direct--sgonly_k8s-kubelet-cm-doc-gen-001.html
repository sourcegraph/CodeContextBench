Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n| Scenario | Impact | Mitigation |\n|----------|--------|-----------|\n| **Cgroup subsystem not mounted** | Container manager fails to initialize | Kubelet startup error; operator must enable cgroups |\n| **Swap enabled on node** | Memory limits not enforced; pod eviction may fail | `--fail-swap-on=true` (default) prevents kubelet start, or warning issued |\n| **CPU quota period unsupported** | CPU hardcapping unavailable | Status.SoftRequirements reports error; kubelet still starts |\n| **Device plugin crashes** | Containers can't allocate devices | Device allocation fails; pod admission rejected if devices required |\n| **NUMA topology detection fails** | Topology manager can't provide hints | Topology manager reverts to \"none\" policy; pod admission succeeds without alignment |\n| **Cgroup v1 to v2 migration** | Old state not readable in new hierarchy | CPUManager/MemoryManager state files incompatible; state must be cleared |\n| **Container removed without notification** | Resource leak (CPUs, memory, devices) | Periodic reconciliation detects orphaned resources and cleans up |\n| **Kubelet restart while pod running** | State loss; resources may be re-allocated | Checkpoints persisted to disk; restored on startup |\n\n### 4.2 Resource Limits and Constraints\n\n| Limit | Default | Tunable | Impact |\n|-------|---------|---------|--------|\n| **Pod PIDs limit** | Unbounded | `--pod-pids-limit` | Prevents fork bombs at pod level |\n| **CPU CFS quota period** | 100ms | `--cpu-cfs-quota-period` | Too low: scheduler overhead; too high: coarse granularity |\n| **CPU Manager reconcile period** | 200ms | `--cpu-manager-reconcile-period` | Tradeoff between responsiveness and overhead |\n| **Topology Manager scope** | container | `--topology-manager-scope=pod` | Pod scope: stricter alignment but higher rejection rate |\n| **Device manager max devices per pod** | No hard limit | Plugin-dependent | Device availability is soft limit |\n| **Memory allocatable reservation** | Computed from allocatable | Per-NUMA manual reservation | Prevents overallocation on NUMA nodes |\n\n### 4.3 Design Tradeoffs\n\n#### 3.3.1 Stateful Resource Allocation vs. Runtime State\n\n**Tradeoff**: CPU and Memory managers maintain state on disk (checkpoint files).\n- **Advantage**: Survive kubelet restart; container resources not lost\n- **Disadvantage**: State can become stale; requires garbage collection on pod cleanup\n- **Evidence**: `cpu_manager_state`, `memory_manager_state` files in kubelet root directory\n\n#### 3.3.2 Topology Hints vs. Hard NUMA Binding\n\n**Tradeoff**: Topology manager collects hints; final NUMA decision made during runtime.UpdateContainerResources().\n- **Advantage**: Resource managers can express preferences; runtime (e.g., containerd) applies best policy\n- **Disadvantage**: Hints can be ignored by runtime; containers may not get optimal NUMA locality\n- **Evidence**: `topologymanager/policy_*.go` provide hints; CPU/Memory managers don't directly set NUMA masks\n\n#### 3.3.3 Per-Pod vs. Per-Container Resource Management\n\n**Tradeoff**: Kubelet enforces pod-level cgroup limits; per-container limits via cgroup limits.\n- **Advantage**: Pod-level QoS applies across all containers; simpler admission\n- **Disadvantage**: Container-level oversubscription possible (e.g., 2 containers each requesting 500m CPU in 1CPU pod)\n- **Evidence**: `podContainerManagerImpl.EnsureExists()` creates pod cgroup; CPUs allocated per-container\n\n#### 3.3.4 Exclusive CPU Allocation and Shared Pool\n\n**Tradeoff**: Static CPU policy allocates exclusive CPUs to some containers; others share.\n- **Advantage**: Deterministic latency for latency-sensitive workloads (exclusive)\n- **Disadvantage**: CPU fragmentation; shared pool may be underutilized or overloaded\n- **Evidence**: `cpumanager/policy_static.go` tracks exclusive set; others use shared pool\n\n#### 3.3.5 Device Plugin Protocol vs. Built-in Device Support\n\n**Tradeoff**: Devices managed via out-of-process plugins (Unix sockets).\n- **Advantage**: Extensibility; new device types without kubelet changes\n- **Disadvantage**: Plugin crashes impact availability; state not persisted in kubelet\n- **Evidence**: Device manager scans `/var/lib/kubelet/device-plugins/` for sockets; allocations stored in memory only\n\n## 5. Operational Implications\n\n### 7.1 Performance Considerations\n\n#### CPU Manager Overhead\n- Reconciliation runs every `cpu-manager-reconcile-period` (default 200ms)\n- Cost: Iterating active pods, reading cgroup state, CRI UpdateContainerResources() calls\n- Mitigation: Set reconciliation period based on workload sensitivity (higher period for batch workloads)\n\n#### Topology Manager Admission Latency\n- Pod admission time increases with number of containers and NUMA nodes\n- Complexity grows in `restricted` and `single-numa-node` policies\n- Mitigation: Use `best-effort` for large clusters; reserve topology-sensitive pods for dedicated nodes\n\n#### Device Manager Scalability\n- Device plugins communicate synchronously with kubelet\n- Large number of devices per container \u2192 longer Allocate() RPC time\n- Mitigation: Device plugins should implement PreferredAllocation() to minimize allocation rounds\n\n### 7.2 Reliability and Recovery\n\n#### Checkpoint Persistence\n- CPU and Memory managers write state to disk after each change\n- **Location**: `${kubelet-root-dir}/cpu_manager_state`, `memory_manager_state`\n- **Recovery**: On kubelet restart, state is restored from checkpoint\n- **Risk**: If checkpoint corrupted, manually delete and kubelet will re-allocate from runtime state\n\n#### Orphaned Resource Cleanup\n- If container dies without notifying kubelet, resources may be allocated in cgroup but not tracked\n- **Detection**: Periodic reconciliation compares runtime pods with cgroup state\n- **Cleanup**: Garbage collection removes resources from orphaned cgroups\n\n#### Device Plugin Tolerance\n- If device plugin crashes, existing pod allocations persisted in kubelet state\n- New pods cannot allocate from crashed plugin\n- **Recovery**: Restart device plugin; kubelet re-establishes connection\n\n### 7.3 Monitoring and Observability\n\n#### Key Metrics\n- `kubelet_cgroup_manager_duration_seconds`: Time to create/update cgroups\n- `kubelet_pod_worker_duration_seconds`: Pod synchronization time (includes resource allocation)\n- `kubelet_device_manager_alloc_duration_seconds`: Device allocation latency\n- `kubelet_cpu_manager_reconcile_duration_seconds`: CPU manager reconciliation time\n\n#### Logs to Monitor\n- Look for \"Failed to allocate CPU\" \u2192 CPU exhaustion or allocation errors\n- Look for \"topology manager policy violation\" \u2192 Pod rejected due to NUMA constraints\n- Look for \"device plugin endpoint not found\" \u2192 Device plugin startup issues\n\n### 7.4 Configuration Best Practices\n\n| Setting | Recommendation | Rationale |\n|---------|-----------------|-----------|\n| `--cpu-manager-policy=static` | Only for latency-sensitive workloads | Overhead is minimal; helps predictable latency |\n| `--memory-manager-policy=static` | Only if NUMA allocation critical | Adds memory pinning overhead |\n| `--topology-manager-policy=restricted` | For co-located workloads needing alignment | Restrictive; use `best-effort` for flexibility |\n| `--pod-pids-limit=4096` | Set per node based on workload | Prevents fork bombs; reduce for untrusted multi-tenant |\n| `--cgroups-per-qos=true` | Recommended for multi-tenant clusters | Enables QoS isolation; # Kubelet Container Manager Architecture Guide

## 1. Subsystem Purpose and Boundaries

The container manager is the core subsystem responsible for managing compute resource allocation, enforcement, and lifecycle on a node. It acts as the orchestration layer between the kubelet's pod synchronization logic and the underlying kernel resource control mechanisms (cgroups), container runtime (CRI), and specialized resource controllers.

### Core Responsibilities
- **Resource enforcement**: Apply CPU, memory, and device quotas to containers via cgroups
- **Hierarchical QoS management**: Organize pods into Guaranteed, Burstable, and BestEffort cgroups
- **Node allocatable enforcement**: Reserve resources for system and kubelet components
- **Resource coordination**: Align CPU, memory, and device allocations with NUMA topology
- **Plugin integration**: Manage device plugins and device memory allocation
- **Dynamic resource lifecycle**: Support runtime preparation/unpreperation of extended resources (DRA)
