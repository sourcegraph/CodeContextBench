## Django QuerySet Lazy Evaluation and Query Execution

**Lazy QuerySet SQL Compilation:**
- **QuerySet** (`django/db/models/query.py:303`) - The main ORM class that implements lazy evaluation by storing a Query object without executing it immediately
- **Query** (`django/db/models/sql/query.py:226`) - Encapsulates all SQL construction logic and provides `get_compiler()` to obtain a compiler

**Database Query Execution (Where OperationalError Occurs):**
- **QuerySet.__iter__()** (line 396) - Entry point triggered when iterating the queryset; calls `_fetch_all()`
- **QuerySet._fetch_all()** (line 2093) - Instantiates `ModelIterable` which forces query execution
- **ModelIterable** (line 88) - The iterable class responsible for executing the database query
- **ModelIterable.__iter__()** (line 91) - Calls `compiler.execute_sql()` to run the compiled SQL
- **SQLCompiler** (`django/db/models/sql/compiler.py:40`) - The compiler that builds and executes SQL
- **SQLCompiler.as_sql()** (line 763) - Compiles the Query into SQL string and parameters
- **SQLCompiler.execute_sql()** (line 1601) - Executes the compiled SQL via `cursor.execute()` at **line 1631** - THIS IS WHERE THE **OperationalError IS RAISED**

The execution chain demonstrates that Django defers SQL compilation and execution until the queryset is actually iterated, implementing true lazy evaluation. The OperationalError occurs at the boundary between Python and the database when `cursor.execute()` is called with the compiled SQL.
