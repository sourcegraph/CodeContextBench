## Summary of Findings

The regression is caused by a **conditional logic bug in the v38 schema migration** (`apps/dashboard/pkg/migration/schemaversion/v38.go`, lines 116-127):

### Root Cause
The V38 migration only processes `fieldConfig.defaults.custom` **if it already exists** in the dashboard JSON. For dashboards saved from v10.3 that don't explicitly include `fieldConfig.defaults.custom` in their JSON, the entire migration logic is silently skipped.

### Why This Matters
- **Dashboards WITH explicit `custom`**: ✓ Migration processes successfully
- **Dashboards WITHOUT explicit `custom`**: ✗ Migration skipped, field overrides potentially orphaned
- The `migrateOverrides()` function is called regardless (line 131), showing the code intended to handle both cases

### Key Evidence
1. **V38.go lines 116-127** shows the bug - the nested conditional fails when custom doesn't exist
2. **V38 test cases** only test scenarios with explicit `fieldConfig.defaults.custom` - no regression coverage
3. **Frontend code** (`migrations.ts:329`) assumes `defaults.custom` exists: `panel.fieldConfig.defaults.custom = panel.fieldConfig.defaults.custom ?? {}`

### Fix Strategy
The v38 migration should **ensure `fieldConfig.defaults.custom` is initialized** (create it if missing) before attempting any migrations, allowing all downstream logic to work consistently.
