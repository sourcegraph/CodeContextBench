For performance, it can request `as=PartialObjectMetadata;g=meta.k8s.io;v=v1` to receive only metadata without the full spec.\n\n**Rate limiting:** QPS and Burst apply client-side token bucket limiting via `rest/client.go` (uses `util/flowcontrol/`). Server-side throttling (HTTP 429) is automatically handled by reading `Retry-After` headers and backing off.\n\n### 2. Authentication and Transport (`transport/`, `rest/config.go`)\n\n**Responsibility:** Build an `http.RoundTripper` chain that adds authentication and observability.\n\nThe Config.TransportConfig() method (rest/config.go:84) converts client Config to a transport.Config, which is then wrapped in authentication middleware:\n\n- Certificates (TLS client certs from CAFile, CertFile, KeyFile)\n- Bearer tokens (static or from file)\n- Exec-based auth (external credential providers)\n- Basic auth (username/password)\n- OIDC (via auth plugins)\n- Impersonation headers\n\nEach middleware is a `http.RoundTripper` that wraps the next one, allowing pluggable auth strategies.\n\n### 3. Clientset (Typed Client) (`kubernetes/`)\n\n**Responsibility:** Provide compile-time-safe, strongly-typed access to core Kubernetes API groups.\n\n**Key file:**\n- `kubernetes/clientset.go` \u2014 Generated factory producing typed clients for each API version\n- `kubernetes/typed/` \u2014 Auto-generated concrete client implementations (e.g., `typed/core/v1/`, `typed/apps/v1/`)\n\nThe Clientset is code-generated by `k8s.io/code-generator`. It exposes methods like:\n\n```go\ntype Interface interface {\n    Discovery() discovery.DiscoveryInterface\n    AppsV1() appsv1.AppsV1Interface  // For Deployments, StatefulSets, etc.\n    CoreV1() corev1.CoreV1Interface   // For Pods, Services, etc.\n    // ... one method per API group/version\n}\n```\n\nEach group accessor returns an interface with resource-specific clients:\n\n```go\ntype AppsV1Interface interface {\n    Deployments(namespace string) DeploymentInterface\n    StatefulSets(namespace string) StatefulSetInterface\n    // ...\n}\n```\n\nEach resource interface provides CRUD operations:\n\n```go\ntype DeploymentInterface interface {\n    Create(ctx context.Context, obj *appsv1.Deployment, opts metav1.CreateOptions) error\n    Update(ctx context.Context, obj *appsv1.Deployment, opts metav1.UpdateOptions) error\n    Delete(ctx context.Context, name string, opts metav1.DeleteOptions) error\n    Get(ctx context.Context, name string, opts metav1.GetOptions) (*appsv1.Deployment, error)\n    List(ctx context.Context, opts metav1.ListOptions) (*appsv1.DeploymentList, error)\n    Watch(ctx context.Context, opts metav1.ListOptions) (watch.Interface, error)\n    // ...\n}\n```\n\n**Design advantage:** Compile-time type checking prevents API version mismatches and typos. The generated code is checked into the repository to avoid build-time code generation.\n\n### 4. Dynamic Client (`dynamic/`)\n\n**Responsibility:** Provide runtime-flexible access to any Kubernetes resource, including Custom Resource Definitions (CRDs).\n\n**Key file:**\n- `dynamic/simple.go` \u2014 DynamicClient implementation\n\nThe DynamicClient wraps a REST client but represents all resources as `unstructured.Unstructured` (from `k8s.io/apimachinery`):\n\n```go\ntype DynamicClient struct {\n    client rest.Interface  // Underlying REST client\n}\n\n// Returns a resource client that works with any GroupVersionResource\nfunc (c *DynamicClient) Resource(gvr schema.GroupVersionResource) NamespaceableResourceInterface\n```\n\n**Advantage:** Can work with any resource without code generation. **Tradeoff:** Type safety is lost; runtime errors for typos or schema mismatches are common.\n\n**Discovery dependency:** The dynamic client often partners with the discovery client to enumerate available resources at runtime.\n\n### 5. Discovery Client (`discovery/`)\n\n**Responsibility:** Discover what API resources exist on the server and their schemas.\n\n**Key file:**\n- `discovery/discovery_client.go` \u2014 Implementation of DiscoveryInterface\n\nThe discovery client queries:\n- `/api` and `/apis` \u2014 Lists of API groups and their versions\n- `/api/v1/...` \u2014 Lists of resources in a group/version\n- `/openapi/v3` \u2014 OpenAPI 3.0 schemas for all resources (used by dynamic clients)\n\n**Caching:** The `discovery/cached/` subdirectory provides CachedDiscoveryClient, which writes results to `~/.kube/cache/` to avoid repeated server queries during CLI operations.\n\n**Use in dynamic clients:** Before operating on a CRD, a controller often queries discovery to confirm the resource exists.\n\n### 6. Informer and Caching System (`tools/cache/`)\n\n**Responsibility:** Maintain a consistent local cache of Kubernetes resources by watching changes from the API server.\n\nThis is the heart of the controller system. The architecture is:\n\n```\nAPI Server\n    \u2193 (LIST + WATCH)\nReflector (tools/cache/reflector.go)\n    \u2193\nDeltaFIFO (tools/cache/delta_fifo.go) \u2014 Accumulates change deltas\n    \u2193\nIndexer/Store (tools/cache/store.go, thread_safe_store.go) \u2014 In-memory cache\n    \u2193\nSharedInformer (tools/cache/shared_informer.go) \u2014 Event broadcaster\n    \u2193\nEvent Handlers (user code) \u2014 React to changes\n    \u2193\nWork Queue (util/workqueue/)\n```\n\n#### Reflector (`tools/cache/reflector.go`)\n\nPerforms a continuous sync loop:\n\n1. **Initial LIST** \u2014 Get a consistent snapshot of all objects and their `resourceVersion`\n2. **WATCH from resourceVersion** \u2014 Stream subsequent changes (ADDED, MODIFIED, DELETED)\n3. **Relist on \"too old\" error** \u2014 If the resourceVersion is no longer available, LIST again\n\nThe Reflector feeds objects into a DeltaFIFO:\n\n```go\ntype Reflector struct {\n    name string\n    listerWatcher ListerWatcher  // Provides LIST and WATCH\n    store ReflectorStore          // DeltaFIFO\n    expectedType reflect.Type\n}\n```\n\n**Watch bookmarks:** The server periodically sends BOOKMARK events with a fresh `resourceVersion` so the Reflector can resume from a recent point without full resync.\n\n#### DeltaFIFO (`tools/cache/delta_fifo.go`)\n\nA queue that accumulates changes as Deltas (a slice of operations on the same object key). Instead of storing the latest state, it stores a history of operations:\n\n```go\ntype Delta struct {\n    Type   DeltaType  // Added, Updated, Deleted, Sync, Replaced\n    Object interface{}\n}\n\ntype Deltas []Delta\n```\n\n**Why Deltas?** If an object is added and then deleted before the controller can process it, the controller needs to know both events. Raw state-based caches (only storing the latest value) would lose the deletion.\n\n#### Indexer and Store (`tools/cache/store.go`, `thread_safe_store.go`)\n\nA thread-safe in-memory store indexed by object key (namespace/name). The ThreadSafeStore wraps a basic map-based store with RWMutex locking:\n\n```go\ntype ThreadSafeStore interface {\n    Add(key string, obj interface{}) error\n    Update(key string, obj interface{}) error\n    Delete(key string) error\n    Get(key string) (interface{}, bool, error)\n    ListKeys() []string\n    List() []interface{}\n    Index(indexName string, obj interface{}) ([]interface{}, error)\n    // ... indexing methods\n}\n```\n\n**Indexing:** Supports custom secondary indexes. For example, a Pod informer might have an index by node name, allowing fast queries like \"all pods on this node.\"\n\n#### SharedInformer (`tools/cache/shared_informer.go`)\n\nThe user-facing interface to the cache. Multiple consumers can register event handlers on the same SharedInformer without duplicating the LIST/WATCH from the server:\n\n```go\ntype SharedInformer interface {\n    AddEventHandler(handler ResourceEventHandler) (ResourceEventHandlerRegistration, error)\n    GetStore() Store\n    GetIndexer() Indexer\n    Run(stopCh <-chan struct{})\n    // ...\n}\n```\n\n**Shared state:** All handlers receive events from the same Reflector \u2192 DeltaFIFO \u2192 Indexer pipeline. If you have 10 controllers interested in Pods, only one Reflector watches them.\n\n#### Lister (`tools/cache/listers.go`)\n\nA read-only, thread-safe interface for accessing the cache. Generated listers are typically:\n\n```go\ntype PodLister interface {\n    List(selector labels.Selector) ([]*v1.Pod, error)\n    Pods(namespace string) PodNamespaceLister\n}\n\ntype PodNamespaceLister interface {\n    List(selector labels.Selector) ([]*v1.Pod, error)\n    Get(name string) (*v1.Pod, error)\n}\n```\n\n**Design advantage:** Controllers interact with Listers, not the raw Indexer. This enforces a read-only view and makes it obvious that controller logic is not directly modifying the cache.\n\n### 7. Work Queue (`util/workqueue/`)\n\n**Responsibility:** Buffer work items with deduplication, retry, and rate limiting.\n\nThe work queue sits between event handlers and controller reconciliation logic. It decouples:\n\n- **Informer event rate** (potentially bursty, tied to API server load)\n- **Controller processing rate** (steady, with backoff)\n\n**Core interface:**\n\n```go\ntype Interface interface {\n    Add(item interface{})\n    Get() (item interface{}, shutdown bool)\n    Done(item interface{})\n    ShutDown()\n}\n```\n\n**Deduplication:** If a Pod is updated three times in quick succession, `Add()` three times adds only one item to the queue (the key, not the object). When the controller gets the item via `Get()`, it fetches the latest state from the Lister.\n\n**Retries:** If a controller returns an error processing an item, it calls `AddRateLimited()` instead of `Done()`, which re-queues the item with an exponential backoff delay.\n\n**Rate limiting:** The `RateLimitingInterface` applies user-configured delays:\n- `DefaultRateLimiter` \u2014 Token bucket (QPS-based)\n- `BucketRateLimiter` \u2014 Token bucket with max queue depth\n- `ItemExponentialFailureRateLimiter` \u2014 Per-item exponential backoff\n- Composable: combine multiple limiters with `NewMaxOfRateLimiter()`\n\n### 8. Informer Factory (`informers/factory.go`, `informers/generic.go`)\n\n**Responsibility:** Create and manage multiple SharedInformers from a single Clientset, sharing LIST/WATCH connections.\n\n```go\ntype SharedInformerFactory interface {\n    Start(stopCh <-chan struct{})\n    WaitForCacheSync(stopCh <-chan struct{}) map[reflect.Type]bool\n\n    Core() corev1.Interface          // Pod, Service, ConfigMap informers\n    Apps() appsv1.Interface           // Deployment, StatefulSet informers\n    // ... one method per API group\n}\n```\n\nOptions:\n- `WithNamespace()` \u2014 Limit informers to a single namespace (reduces memory and network)\n- `WithCustomResyncConfig()` \u2014 Per-resource resync periods\n- `WithTweakListOptions()` \u2014 Add label/field selectors to all LISTs\n\n**Resync period:** Periodically forces a full re-process of all cached objects via a Sync delta (no API call). Default is 15 minutes. Used to catch API server state changes that were missed (watch timeout, etc.).\n\n### 9. Leader Election (`tools/leaderelection/`)\n\n**Responsibility:** Ensure only one replica of a controller runs at a time (for stateful operations).\n\n```go\ntype LeaderElectionConfig struct {\n    Client ClientInterface          // NamespacedResourceLock interface\n    LeaseLockName string\n    LeaseLockNamespace string\n    LeaseDuration time.Duration     // How long the lease lasts (e.g., 60s)\n    RenewDeadline time.Duration     // Must renew before this (e.g., 40s)\n    RetryPeriod time.Duration       // How often to retry acquiring (e.g., 2s)\n}\n\n// OnStartedLeading and OnStoppedLeading callbacks run when leadership changes\n```\n\nUses a Lease object (API group `coordination.k8s.io`) as a distributed lock. One replica acquires the lease by setting `spec.holderIdentity` to its identity and a `spec.renewTime` timestamp. Others periodically check; when the lease expires, a replica can take over.\n\n**Failure recovery:** If the leader crashes, non-leaders detect the expired lease (after LeaseDuration) and one acquires it. If the leader loses network (can't renew the lease), it detects this and calls OnStoppedLeading to gracefully shut down.\n\n### 10. Configuration and Authentication (`tools/clientcmd/`)\n\n**Responsibility:** Parse kubeconfig files and produce a Config suitable for constructing clients.\n\nKey files:\n- `tools/clientcmd/loader.go` \u2014 Loads `~/.kube/config` (or KUBECONFIG env)\n- `tools/clientcmd/config.go` \u2014 Merges contexts, users, and clusters\n\nHandles:\n- Multiple clusters in one kubeconfig\n- Switching contexts\n- External credential providers (e.g., cloud IAM providers via exec plugins)\n- Certificate and key files\n\nOutput: A `rest.Config` ready to be passed to NewForConfig() or a Clientset.\n\n---\n\n## End-to-End Control Flow\n\n### Scenario: A Simple Pod Watcher Controller\n\nLet's trace code execution when a controller watches Pods, detects changes, and processes them.\n\n#### Setup\n\n```go\n// 1. Load kubeconfig (tools/clientcmd/)\nconfig, _ := clientcmd.BuildConfigFromFlags(\"\", os.ExpandEnv(\"$HOME/.kube/config\"))\n\n// 2. Create a Clientset (kubernetes/clientset.go)\nclientset, _ := kubernetes.NewForConfig(config)\n\n// 3. Create an informer factory (informers/factory.go)\nfactory := informers.NewSharedInformerFactory(clientset, 15*time.Minute)\n\n// 4. Get a Pod informer (auto-generated via code-generator)\npodInformer := factory.Core().V1().Pods()\n\n// 5. Register an event handler\npodInformer.Informer().AddEventHandler(cache.ResourceEventHandlerFuncs{\n    AddFunc: func(obj interface{}) {\n        pod := obj.(*corev1.Pod)\n        queue.Add(pod.Namespace + \"/\" + pod.Name)\n    },\n    UpdateFunc: func(oldObj, newObj interface{}) {\n        queue.Add(newObj.(*corev1.Pod).Namespace + \"/\" + newObj.(*corev1.Pod).Name)\n    },\n})\n\n// 6. Start the factory (starts all informers, including the Reflector)\nstopCh := make(chan struct{})\nfactory.Start(stopCh)\n\n// 7. Wait for initial sync\nfactory.WaitForCacheSync(stopCh)\n\n// 8. Main controller loop\nfor {\n    key, _ := queue.Get()\n    pod, exists, _ := podInformer.Lister().Pods(namespace).Get(name)\n    // Process pod...\n    queue.Done(key)\n}\n```\n\n#### Request/Response Trace\n\n**Phase 1: Config Loading** (tools/clientcmd/)\n\n1. `BuildConfigFromFlags()` reads `~/.kube/config` (or KUBECONFIG paths)\n2. Parses YAML to extract:\n   - API server URL (e.g., `https://kubernetes.default.svc`)\n   - Current context name\n   - User auth info (certificates, token, exec config)\n   - Cluster CA certificate\n3. Returns a `rest.Config` with:\n   - `Host = \"https://kubernetes.default.svc\"`\n   - `BearerToken` or client cert paths\n   - `CAData` (cluster root CA)\n   - `QPS = 5, Burst = 10` (defaults from rest/config.go:47-48)\n\n**Phase 2: Client Creation** (rest/transport.go, rest/config.go)\n\n1. `NewForConfig(config)` calls `TransportFor(config)` (rest/transport.go:63)\n2. `TransportFor()` converts rest.Config to transport.Config\n3. `transport.New(cfg)` wraps the HTTP transport in authentication middleware:\n   - TLS setup from CAData/CertFile/KeyFile\n   - Bearer token middleware (injects `Authorization: Bearer <token>` header)\n   - Rate limiter (token bucket with QPS=5)\n   - Retry logic with exponential backoff\n4. Creates an RESTClient wrapping this transport\n\n**Phase 3: Informer Startup** (tools/cache/, informers/)\n\n1. `factory.Start(stopCh)` iterates all registered informers and calls their `Run()` method\n2. For each informer's Reflector:\n   - Calls `listerWatcher.List()` (a REST call using the RESTClient)\n   - Makes an HTTP GET to `/api/v1/pods` (or `/api/v1/namespaces/{namespace}/pods` if scoped)\n   - REST request builder constructs: `GET /api/v1/namespaces/default/pods?limit=500`\n   - Server responds with a PodList and a `resourceVersion` (e.g., \"12345\")\n   - Reflector stores objects in DeltaFIFO via `Add()` deltas\n   - Objects flow to Indexer (the in-memory cache)\n   - Indexer calls event handlers, which queue keys\n\n3. After LIST, Reflector calls `listerWatcher.Watch()` with `resourceVersion=12345`\n   - Makes an HTTP GET with `watch=true&resourceVersion=12345`\n   - Server streams watch.Event objects (JSON lines)\n   - For each event (ADDED, MODIFIED, DELETED):\n     - Reflector puts a Delta in DeltaFIFO\n     - DeltaFIFO\u2192Indexer\u2192Event handlers\u2192Work queue\n\n4. `factory.WaitForCacheSync(stopCh)` blocks until the Reflector has listed and the initial handler calls complete\n\n**Phase 4: Processing** (util/workqueue/)\n\n1. Main controller loop calls `queue.Get()` (blocking call)\n2. Returns a queued key (e.g., \"default/my-pod\")\n3. Controller calls `podInformer.Lister().Pods(\"default\").Get(\"my-pod\")`\n   - This reads from the in-memory Indexer (no API call)\n   - Returns the cached Pod object or \"not found\"\n4. Controller business logic runs (reconciliation)\n5. If error, controller calls `queue.AddRateLimited(key)` which re-queues with delay\n6. If success, controller calls `queue.Done(key)`\n\n**Phase 5: Handling Updates**\n\nWhen the user edits a Pod (via kubectl or another client):\n\n1. API server receives the PATCH request\n2. Increments the Pod's `resourceVersion` (e.g., from \"12345\" to \"12346\")\n3. Sends a MODIFIED watch event with the new object to all watchers\n4. Reflector receives the event, creates a Delta, adds to DeltaFIFO\n5. Controller processes the new state from the Lister\n\n**Failure Scenarios:**\n\n- **Watch timeout (e.g., resourceVersion too old):** Reflector detects `errors.IsResourceExpired()`, calls `List()` again, starts a new watch\n- **API server down (connection refused):** Transport retries with exponential backoff (default max 10 retries, ~30 seconds total)\n- **429 (API Priority and Fairness throttling):** Client reads `Retry-After` header, backs off, retries (rest/with_retry.go)\n- **Controller crashes:** Lease expires (after LeaseDuration), another replica takes over and continues from the latest resourceVersion\n\n---\n\n## Failure Modes, Limits, and Design Tradeoffs\n\n### Failure Mode 1: Watch Timeout (ResourceVersion Too Old)\n\n**Symptom:** Logs show `\"ResourceVersion too old\"` error.\n\n**Root cause:** Watch request lasted longer than the API server's watch history window. Default is 5 minutes; if a watch runs for > 5 minutes without reconnecting, the resourceVersion becomes invalid.\n\n**Impact:** No objects are lost, but there is a gap. The Reflector must re-LIST and restart the watch.\n\n**Design tradeoff:**\n- **Pro:** Bounds the memory used by the API server to track individual watches\n- **Con:** Controllers with slow processing might miss events if the queue backs up for > 5 minutes\n\n**Mitigation:**\n- Increase resync period (tools/cache/ runs a Sync delta periodically to catch any missed events)\n- Optimize controller processing (don't block on external I/O)\n- Use multiple replicas (if one falls behind, leadership can transfer)\n\n### Failure Mode 2: Duplicate Events in Queue\n\n**Symptom:** Controller receives the same key twice in quick succession.\n\n**Root cause:** The Reflector's relist loop might send Sync deltas for every object in the cache. If this races with a fresh WATCH event, the object's key can be queued twice.\n\n**Design tradeoff:**\n- **Pro:** Ensures controllers see eventual consistency; even if a WATCH is missed, the next resync will trigger\n- **Con:** Not at-most-once; controllers must be idempotent\n\n**Mitigation:** Ensure your reconciliation is idempotent (i.e., running it twice produces the same result as once).\n\n### Failure Mode 3: Cache Inconsistency (Stale Data)\n\n**Symptom:** Controller reads an old Pod from the Lister, even though it was just updated on the server.\n\n**Root cause:**\n1. User updates a Pod (new resourceVersion on server)\n2. WATCH event in flight\n3. Controller calls Lister.Get() before the event is processed\n4. Lister returns the cached (old) object\n\n**Impact:** Controller's next reconciliation will see fresh data (from the WATCH event in flight). Usually not a correctness issue due to resync.\n\n**Design tradeoff:**\n- **Pro:** Informers provide \"eventually consistent\" views, not strongly consistent. This scales better.\n- **Con:** Not suitable for applications requiring strict consistency (e.g., distributed consensus algorithms)\n\n**Mitigation:**\n- If strict consistency is needed, re-fetch from the API server (not the cache)\n- Or use a server-side apply pattern with conflict detection (applyconfigurations/)\n\n### Failure Mode 4: Memory Bloat from Large Caches\n\n**Symptom:** Client process memory grows to gigabytes.\n\n**Root cause:**\n- Namespace not specified: informer caches all Pods across all namespaces\n- Resync period too frequent: high-frequency Sync deltas even if nothing changed\n- No resource selectors: caching all objects of a type when only a subset matters\n\n**Impact:** Memory leak in long-running controllers; eventual OOM.\n\n**Design tradeoff:**\n- **Pro:** Caching everything allows fast queries without API calls\n- **Con:** Not suitable for large clusters (e.g., 100k Pods)\n\n**Mitigation:**\n- Use `WithNamespace()` when creating the factory: only caches Pods in one namespace\n- Add label selectors via `WithTweakListOptions()`\n- Increase resync period or disable resync for lightweight caches\n- Consider pagination and off-loading queries to an external database for large scales\n\n### Failure Mode 5: Rate Limiting and Backoff Interaction\n\n**Symptom:** API calls are delayed more than expected.\n\n**Root cause:** Multiple layers of rate limiting:\n1. Client-side work queue rate limiter\n2. REST client QPS limiter (rest/config.go:47-48)\n3. Server-side API Priority and Fairness (429 responses)\n\nAll three interact; a controller hitting the QPS limit will see requests queued.\n\n**Design tradeoff:**\n- **Pro:** Prevents a single controller from overwhelming the API server\n- **Con:** Harder to predict latency; requires tuning\n\n**Mitigation:**\n- Monitor metrics: `client_go_request_duration_seconds` (rest client metrics)\n- Adjust Config.QPS and Burst based on workload\n- Use `workqueue.RateLimitingInterface` to tune per-type backoff\n\n### Failure Mode 6: Circular Dependencies in Finalizers\n\n**Symptom:** Object deletion hangs.\n\n**Root cause:** Two resources have finalizers that depend on each other being deleted first. The Reflector and work queue don't have special logic for circular dependencies.\n\n**Design tradeoff:**\n- **Pro:** Controllers are simple; no built-in deadlock detection complexity\n- **Con:** Controllers must coordinate to avoid deadlocks\n\n**Mitigation:**\n- Carefully design finalizer dependencies (document which resources can depend on which)\n- Consider using owner references and cascade delete (server-side deletion logic)\n\n---\n\n## Extension Points, Customization Hooks, and Associated Risks\n\n### 1. Custom REST Transport (`rest/config.go`, `transport/`)\n\n**Hook:** `Config.Transport` (custom `http.RoundTripper`), `Config.WrapTransport`\n\n**Use case:** Inject custom headers, logging, metrics, or authentication schemes.\n\n**Example:**\n```go\nconfig.WrapTransport = func(rt http.RoundTripper) http.RoundTripper {\n    return &LoggingTripper{inner: rt}\n}\n\ntype LoggingTripper struct {\n    inner http.RoundTripper\n}\n\nfunc (t *LoggingTripper) RoundTrip(req *http.Request) (*http.Response, error) {\n    log.Printf(\"Request: %s %s\", req.Method, req.URL)\n    resp, err := t.inner.RoundTrip(req)\n    if err != nil {\n        log.Printf(\"Error: %v\", err)\n    }\n    return resp, err\n}\n```\n\n**Risk:** If the transport blocks indefinitely, all API calls hang. Ensure timeouts are set and goroutines can't accumulate unboundedly.\n\n### 2. Custom Retry Logic (`rest/with_retry.go`)\n\n**Hook:** `Request.WarningHandler()`, `Request.WithRetry()`\n\n**Use case:** Implement custom retry strategies for specific error types.\n\n**Example:**\n```go\nreq := client.CoreV1().Pods(\"default\").Create(ctx, pod, metav1.CreateOptions{})\n// Automatically retries on 5xx and 429\n```\n\n**Risk:** Over-aggressive retries can mask transient issues or exhaust server resources. Ensure exponential backoff is used.\n\n### 3. Custom Lister/Indexer Storage (`tools/cache/store.go`)\n\n**Hook:** `cache.NewStore(keyFunc, indexers)` \u2014 provide a custom Store implementation\n\n**Use case:** Persist cache to disk, use an external database, or implement custom indexing.\n\n**Example:**\n```go\ncustomStore := &MyCustomStore{/* ... */}\nreflector := cache.NewReflector(listerWatcher, &corev1.Pod{}, customStore, resyncPeriod)\n```\n\n**Risk:** The custom store must be thread-safe. If it's slow, the Reflector's main loop blocks and watch events queue up.\n\n### 4. Custom Event Handlers (`tools/cache/shared_informer.go`)\n\n**Hook:** `informer.AddEventHandler(handler)` or `AddEventHandlerWithResyncPeriod()`\n\n**Use case:** React to object changes with business logic.\n\n**Example:**\n```go\nhandler := cache.ResourceEventHandlerFuncs{\n    AddFunc: func(obj interface{}) {\n        log.Printf(\"Pod added: %v\", obj.(*corev1.Pod).Name)\n    },\n    UpdateFunc: func(oldObj, newObj interface{}) {\n        // Only queue if spec changed, not just metadata\n    },\n}\ninformer.AddEventHandler(handler)\n```\n\n**Risk:** Handler execution blocks the Reflector's event broadcasting loop. If a handler is slow, watch events pile up and the Reflector can't keep up with the server. **Always use a work queue to offload processing.**\n\n### 5. Custom Work Queue Rate Limiters (`util/workqueue/`)\n\n**Hook:** `workqueue.NewRateLimitingQueue(limiter)` \u2014 provide a custom `RateLimiter`\n\n**Use case:** Implement domain-specific backoff strategies (e.g., exponential with a cap).\n\n**Example:**\n```go\nlimiter := workqueue.NewMaxOfRateLimiter(\n    workqueue.NewItemExponentialFailureRateLimiter(5*time.Millisecond, 5*time.Second),\n    &workqueue.BucketRateLimiter{Limiter: rate.NewLimiter(rate.Limit(10), 100)},\n)\nqueue := workqueue.NewRateLimitingQueue(limiter)\n```\n\n**Risk:** A limiter that doesn't back off can thrash (busy-loop) and consume CPU. Ensure exponential backoff is used.\n\n### 6. Custom Informer Factory (`informers/factory.go`)\n\n**Hook:** `WithCustomResyncConfig()`, `WithTweakListOptions()`, `WithNamespace()`\n\n**Use case:** Tune the factory for specific workloads (e.g., only watch Pods in a single namespace).\n\n**Example:**\n```go\nfactory := informers.NewSharedInformerFactory(\n    clientset,\n    15*time.Minute,\n    informers.WithNamespace(\"default\"),\n    informers.WithTweakListOptions(func(opts *metav1.ListOptions) {\n        opts.LabelSelector = \"app=myapp\"\n    }),\n)\n```\n\n**Risk:** Narrowing the informer scope means controllers miss objects outside the scope. E.g., if you only watch \"default\" namespace, you won't see cross-namespace references.\n\n### 7. Custom Leader Election (`tools/leaderelection/`)\n\n**Hook:** `leaderelection.RunOrDie()` with custom `LeaderElectionConfig`\n\n**Use case:** Coordinate multiple replicas of a controller.\n\n**Example:**\n```go\ncfg := leaderelection.LeaderElectionConfig{\n    Lock:            lock,\n    LeaseDuration:   60 * time.Second,\n    RenewDeadline:   40 * time.Second,\n    RetryPeriod:     2 * time.Second,\n    OnStartedLeading: func(ctx context.Context) {\n        log.Println(\"Became leader, starting controller\")\n        startController(ctx)\n    },\n    OnStoppedLeading: func() {\n        log.Println(\"Lost leadership, stopping controller\")\n    },\n}\nleaderelection.RunOrDie(context.Background(), cfg)\n```\n\n**Risk:** If `OnStoppedLeading` doesn't gracefully shut down, two replicas might both be running (split-brain). # Kubernetes Client-Go Systems Guide

## Overview

**client-go** is the official Go client library for the Kubernetes API. It provides the foundation for all Kubernetes Go tools, operators, and controllers by offering:

1. **REST transport abstraction** — HTTP connectivity, authentication, and request/response handling
2. **Typed and dynamic API clients** — Compile-time-safe and runtime-flexible access to Kubernetes resources
3. **Caching and event synchronization** — An event-driven informer system that keeps local caches consistent with the API server
4. **Controller infrastructure** — Work queues, leader election, and metrics for building reliable control loops

The subsystem is located in `staging/src/k8s.io/client-go/`.

---

## Subsystem Purpose, Boundaries, and Dependencies

### Purpose

client-go bridges Go applications to the Kubernetes API by:
- Translating high-level Go function calls into HTTP REST requests against `/api`, `/apis`, and discovery endpoints
- Managing persistent data coherence through a local cache synchronized via watches
- Providing structured error handling and retry semantics
- Supporting both strongly-typed clients (for built-in resources) and dynamic clients (for custom resources)

### Boundaries

**Upstream dependencies:**
- `k8s.io/apimachinery` — Provides the core runtime types (Object, Serializer, Scheme), error types, and watch events
- `k8s.io/api` — Defines the API types (Pod, Service, Deployment, etc.)
- `k8s.io/code-generator` — Generates typed clients, listers, and informers from API definitions
- Standard Go `net/http` and TLS libraries

**Downstream consumers:**
- `kubectl` — CLI tool using dynamic client and discovery
- `kube-controller-manager`, `kubelet`, `kube-scheduler` — Built-in Kubernetes controllers
- Custom operator
