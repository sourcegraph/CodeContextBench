## Bug Report

**Bug ID**: 53441
**Summary**: hda_intel: sound stops working after suspend to ram
**Regression**: Yes (works on v3.6.11, broken since v3.7.1)

On a fresh boot the sound card works fine. After a few suspends to ram (it doesn't necessarily happen after just one) the sound card stops working well. The sound comes out choppy, as a series of clicks separated by 2 second or so silences.

I see these messages in the kernel log right after resume:
```
hda-intel: azx_get_response timeout, switching to polling mode: last cmd=0x00170503
hda-intel: No response from codec, disabling MSI: last cmd=0x00170503
hda_intel: azx_get_response timeout, switching to single_cmd mode: last cmd=0x00170503
```

The only way to fix the sound seems to be a reboot. I don't have this problem with v3.6.11, it appeared with kernel v3.7.1 (I have not tested v3.7). Right now I'm running 3.7.6 and the problem is still there.

## Fault Localization Summary

### Bug Location
- **File**: `sound/pci/hda/hda_intel.c`
- **Function**: `azx_resume` (line 2539-2566)
- **Confidence**: 95%

### Root Cause
The `azx_resume` function is **missing the `synchronize_irq()` call** after acquiring the IRQ during resume from suspend. This is the key difference when compared to `azx_first_init`, which properly calls `synchronize_irq(chip->irq)` at line 3159.

### Why This Causes the Bug

1. **IRQ Synchronization Issue**: During suspend, the IRQ is freed and MSI is disabled. During resume:
   - The PCI device is re-enabled
   - MSI is re-enabled  
   - A new IRQ is requested via `azx_acquire_irq()`
   - **BUT**: The `synchronize_irq()` call that synchronizes the IRQ with hardware state is missing

2. **Race Condition**: Without `synchronize_irq()`, there's a window where:
   - The kernel's IRQ handler registration is out of sync with hardware state
   - Any pending interrupts from before suspend aren't properly cleared
   - The codec communication becomes unreliable

3. **Stale State Flags**: Additionally, the `chip->polling_mode` and `chip->single_cmd` flags are not reset during resume, causing the system to remain in degraded modes if they were set before suspend.
