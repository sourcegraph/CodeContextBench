## 1. Code & Architecture Review

### 1.1 CRITICAL: Set Comparison Operator Semantics Error in IR Metrics

`scripts/csb_metrics/ir_metrics.py` line 749:

```python
if gt_found >= gt_normalized:
    tt_all_r = ts - start_ts
```

The `>=` operator on sets means "is superset of." This condition checks
whether the set of found ground-truth items is a *superset* of the
normalized ground-truth set -- which is the correct direction. However,
`tt_all_r` (time-to-all-relevant) is assigned inside a loop that iterates
over retrieval events chronologically. The variable is reassigned on every
iteration where the condition holds, meaning it records the **last** time
all items were found, not the **first** time. Once all GT items are found,
every subsequent event also satisfies the superset check, continuously
overwriting `tt_all_r` with later timestamps.

**Impact**: The `tt_all_r` metric systematically overestimates how long it
takes the agent to find all relevant items. For agents that do extensive
post-retrieval work after finding everything, this inflates timing metrics.

**Fix**: Add a `break` after the assignment, or guard with
`if tt_all_r is None:` to capture only the first occurrence.

### 1.2 CRITICAL: Judge Output Violates Its Own JSON Schema

`scripts/csb_metrics/judge/models.py` lines 85-90 emit four fields that
are not declared in the schema:

```python
if self.criteria_scores:
    result["criteria_scores"] = self.criteria_scores
    result["rubric_score"] = self.rubric_score
if self.hybrid_composite is not None:
    result["hybrid_composite"] = self.hybrid_composite
    result["verifier_weight"] = self.verifier_weight
```

`schemas/judge_result.schema.json` line 109 declares
`"additionalProperties": false`. Any judge result with hybrid scoring
enabled will fail strict schema validation. Tools that validate judge
output against the schema will reject valid results.

**Fix**: Add `criteria_scores`, `rubric_score`, `hybrid_composite`, and
`verifier_weight` to the schema's `properties` section, or remove the
`additionalProperties: false` constraint.

### 1.3 CRITICAL: 66 Org Tasks Missing Oracle `text` Field

66 of 144 org-scale tasks have `oracle_answer.json` files that contain
`files` and `symbols` arrays but are missing the `text` field entirely.
The `text` field provides the natural-language description of the
cross-repository dependency chain and is used by the judge system as
ground-truth context.

| Suite | Missing | Total | Gap |
|-------|--------:|------:|----:|
| `csb_org_crossrepo` | 10 | 11 | 91% |
| `csb_org_compliance` | 8 | 13 | 62% |
| `csb_org_platform` | 8 | 13 | 62% |
| `csb_org_migration` | 13 | 25 | 52% |
| `csb_org_crossorg` | 6 | 12 | 50% |
| `csb_org_domain` | 5 | 11 | 45% |
| `csb_org_org` | 5 | 11 | 45% |
| `csb_org_incident` | 4 | 13 | 31% |
| `csb_org_security` | 4 | 13 | 31% |
| `csb_org_crossrepo_tracing` | 3 | 11 | 27% |

**Impact**: The oracle discovery system in `scripts/csb_metrics/judge/oracle.py`
falls back to file/symbol lists when `text` is absent, but the judge prompt
loses descriptive context. This degrades judge accuracy for nearly half of
all org-scale tasks. The `csb_org_crossrepo` suite is almost entirely
affected (91%).

### 1.4 HIGH: KeyError Risk in Retrieval Metrics -- Missing Type Guards

`scripts/csb_metrics/retrieval.py` lines 240-242 use direct dict key access
on `oracle_item` without checking key existence:

```python
return (
    _repo_matches(arg_repo, oracle_item["repo"])    # KeyError if missing
    and _path_matches(oracle_item["path"], arg_path) # KeyError if missing
)
```

The same pattern recurs at lines 256-257, 286, and 294. Meanwhile, line 251
uses the safe `.get("repo", "")` pattern on the same data type. If any
oracle item lacks a `repo` or `path` key (which is plausible given the 66
incomplete oracle files in Section 1.3), retrieval metric extraction crashes
for that task.

**Fix**: Use `oracle_item.get("repo", "")` and `oracle_item.get("path", "")`
consistently.

### 1.5 HIGH: Inconsistent Token Data Extraction Between Sources

`scripts/csb_metrics/extractors.py` line 311 vs `discovery.py` lines 222-223:

- `result.json` provides a single `n_cache_tokens` field (conflating
  creation and read tokens)
- Transcript data provides separate `cache_creation_input_tokens` and
  `cache_read_input_tokens`

When transcript data is unavailable (e.g., the run crashed before
transcript post-processing), only the `result.json` value is used, but it
gets assigned to `cache_creation_tokens` only. The `cache_read_tokens`
field remains `None`, making cache hit rate calculations
(`cache_read_tokens / total_cache_tokens` at line 305-306) produce `None`
instead of a meaningful value derived from the combined field.

**Impact**: Cost reports and cache efficiency metrics silently degrade to
`None` for any run that lacks transcript data, rather than using the
available combined value as a reasonable approximation.

### 1.6 HIGH: Four Harness Wrappers Missing Credential Loading

`configs/codex_2config.sh`, `configs/copilot_2config.sh`,
`configs/cursor_2config.sh`, and `configs/gemini_2config.sh` all source
`_common.sh` but do not call `load_credentials` afterward. In contrast:

- `openhands_2config.sh` calls `load_credentials` at line 33
- `sdlc_suite_2config.sh` calls `load_credentials` at line 18
- `run_selected_tasks.sh` calls `load_credentials` at line 178

Without `load_credentials`, environment variables from `.env.local` (like
`SOURCEGRAPH_ACCESS_TOKEN`) are not loaded. MCP runs via these four
harnesses will fail if the token comes from `.env.local` rather than the
operator's shell environment.

### 1.7 HIGH: `run_selected_tasks.sh` Does Not Filter Excluded Tasks

`configs/selected_benchmark_tasks.json` contains 11 tasks marked with
`"excluded": true`. All five alternative harness wrappers (codex, copilot,
cursor, gemini, openhands) filter these out during task extraction:

```python
if task.get("excluded", False):
    continue
```

However, `configs/run_selected_tasks.sh` lines 221-247 (`extract_tasks`
function) have no check for the `excluded` field. The main runner will
attempt to run all 11 excluded tasks, which are typically
Daytona-incompatible and will fail, wasting sandbox allocation time and
cost.

### 1.8 MEDIUM: No Timeout Protection for Judge LLM Calls

`scripts/csb_metrics/judge/backends.py` lines 112 and 158:

```python
# SDK backend (line 112) -- no timeout parameter
response = client.messages.create(
    model=self.model, max_tokens=self.max_tokens, ...
)

# urllib backend (line 158) -- no timeout parameter
with urllib.request.urlopen(req) as resp:
```

Neither backend implements request timeouts. In batch judge runs across
275 tasks with 3-round voting, a single hung LLM call blocks the entire
pipeline indefinitely. The Anthropic SDK supports a `timeout` parameter,
and `urllib.request.urlopen` accepts a `timeout` keyword argument.

### 1.9 MEDIUM: Task ID Casing Mismatch Across 144 Org Tasks

All 144 org-scale tasks have a directory name using lowercase (`ccx-*`)
but `task.toml` declares the ID with uppercase prefix (`CCX-*`):

- Directory: `benchmarks/csb_org_domain/ccx-domain-073/`
- `task.toml` line 9: `id = "CCX-domain-073"`

Any code that matches task IDs using case-sensitive string comparison
against directory names will fail to find these tasks. The `sync_task_metadata.py`
script and `validate_tasks_preflight.py` both use case-sensitive matching.

### 1.10 MEDIUM: Inconsistent Task ID Normalization Across Modules

Two modules in `scripts/csb_metrics/` normalize task IDs differently:

| Module | Function | Strategy |
|--------|----------|----------|
| `task_selection.py:21-31` | `_normalize_task_id()` | Strips `mcp_` prefix and 6-char hash suffix |
| `discovery.py:158-168` | `_extract_task_id()` | Splits on `__` separator and removes trailing hash |

Edge-case task IDs that contain both `mcp_` prefixes and `__` separators
could produce different normalized forms depending on which module
processes them. This affects task matching between the discovery and
selection subsystems.

### 1.11 MEDIUM: 13 SDLC Tasks Have Malformed `task.toml` Structure

**Missing `[metadata]` section** (8 tasks in `csb_sdlc_debug`):
All 8 regression-prove tasks (e.g., `ansible-galaxy-tar-regression-prove-001`,
`flipt-auth-cookie-regression-prove-001`, etc.) start with `[task]` instead
of the standard `[metadata]` section. Scripts that parse `task.toml` by
looking for `[metadata]` keys will fail to extract suite, difficulty, and
language metadata for these tasks.

**Missing `[task]` section** (5 tasks across SDLC suites):
`envoy-grpc-server-impl-001`, `etcd-grpc-api-upgrade-001`,
`k8s-runtime-object-impl-001`, `python-http-class-naming-refac-001`, and
`numpy-dtype-localize-001` use `[metadata]` with a `task_id` field instead
of the standard `[task]` section with an `id` field. Any tool expecting
the standard structure will miss these tasks' identifiers.

### 1.12 MEDIUM: Unsafe Attribute Access in Discovery Module

`scripts/csb_metrics/discovery.py` lines 179-183:

```python
agents = data.get("agents") or []
if agents and isinstance(agents, list):
    model = agents[0].get("model_name")  # agents[0] could be non-dict
```

The code validates that `agents` is a list but does not validate that
`agents[0]` is a dict. If the first element is a string, integer, or
other non-dict type, `.get()` raises `AttributeError`. The same pattern
appears in `scripts/csb_metrics/extractors.py` lines 651-654.

### 1.13 LOW: Missing Content-Type Header in Auth Proxy GET Error Response

`agents/harnesses/openhands/sg_auth_proxy.py` lines 172-175:

The `do_GET()` error handler omits the `Content-Type` header, while
`do_POST()` at lines 147-150 correctly sends `Content-Type: application/json`.
Clients receiving error responses from GET requests may misinterpret the
body format.

### 1.14 LOW: 7 Oracle Files Missing `symbols` Field

7 tasks across 5 org suites have `oracle_answer.json` files with `files`
and `text` but no `symbols` array: `ccx-incident-034`, `ccx-incident-037`,
`ccx-platform-094`, `ccx-platform-100`, `ccx-dep-trace-001`,
`ccx-agentic-081`, `ccx-vuln-remed-014`. The retrieval metrics system uses
`symbols` for symbol-level precision/recall calculations. Missing symbols
silently produce `precision=None, recall=None` for these tasks rather than
explicit zeros.

## 2. Feature & UX Improvements

### 2.1 Judge `expected_approach` Field Is Extracted but Never Used

`scripts/csb_metrics/judge/models.py` line 24 defines
`oracle_expected_approach: str` on `JudgeInput`. The oracle discovery
system (`run_judge.py` line 499) populates it from oracle data. But the
field is never referenced in `engine.py` or `prompts.py` -- the judge
prompt never includes methodological guidance from the oracle.

**Improvement**: Inject `expected_approach` into the judge prompt as a
"preferred methodology" hint. This would help the judge distinguish
between correct results achieved via the wrong method (e.g., hardcoded
values vs. actual code analysis) and genuine solutions.

### 2.2 No LLM Response Completeness Validation in Judge

`scripts/csb_metrics/judge/engine.py` lines 230-234:

```python
response = self._backend.call(system_prompt, user_prompt)
dim_scores = _parse_dimension_scores(response, self.dimensions)
```

If the LLM returns `{"reasoning": "..."}` but omits `dimension_scores`
entirely, `_parse_dimension_scores` silently defaults every dimension to
0.0 via `entry.get("score", 0.0)` at line 160. A malformed LLM response
is indistinguishable from a legitimately scored 0.0 result.

**Improvement**: Add a completeness check after parsing. If
`dimension_scores` is absent from the response dict, log a warning and
either retry the LLM call or mark the result with a `"parse_error": true`
flag so downstream consumers can filter it.

### 2.3 Agent Harness Code Has Zero Test Coverage

The agent execution layer (3,115 LOC across 4 files) has no unit tests:

| File | Lines | Untested Functionality |
|------|------:|----------------------|
| `agents/claude_baseline_agent.py` | 2,112 | MCP config generation, token refresh, environment setup |
| `agents/harnesses/openhands/agent.py` | 455 | Launcher script generation, proxy setup, FD management |
| `agents/harnesses/base.py` | 342 | Task path resolution, config mode selection |
| `agents/harnesses/openhands/sg_auth_proxy.py` | 206 | Request forwarding, tool blocking, error handling |

The March 6 report noted testing infrastructure is broadly absent (4.2%
of scripts have tests). This narrows the finding: the agent harness is
the single most critical untested subsystem because it controls benchmark
execution, credential handling, and MCP tool access -- all areas where
bugs silently corrupt results rather than crashing visibly.

**Improvement**: Prioritize tests for MCP config generation (4 code paths)
and token field extraction (the `dict.get()` bugs from March 8 live here).

### 2.4 Fragile Markdown Code Block Extraction in Judge Backend

`scripts/csb_metrics/judge/backends.py` line 35:

```python
m = re.search(r"```(?:json)?\s*\n?(.*?)\n?\s*```", text, re.DOTALL)
```

This regex assumes exactly 3 backticks. Some LLMs use 4+ backticks
(especially in responses that discuss code containing triple backticks).
Additionally, the regex does not handle responses where JSON is returned
without any code fence (some models return raw JSON when instructed to).

**Improvement**: Add a fallback chain: try the fence regex first, then
attempt direct `json.loads()` on the raw response, then try stripping
leading/trailing non-JSON text with `text[text.index('{'):text.rindex('}')+1]`.

## 3. Research Recommendations

### 3.1 Implement Oracle Data Validation as a Pre-Flight Check

The 66 tasks missing `text` fields and 7 missing `symbols` fields
(Sections 1.3, 1.14) indicate oracle data quality was not validated
during task authoring. Add a check to `validate_tasks_preflight.py`
that verifies oracle_answer.json completeness:

```python
REQUIRED_ORACLE_FIELDS = {"text", "files", "symbols"}

def check_oracle_completeness(task_dir: Path) -> list[str]:
    oracle_path = task_dir / "tests" / "oracle_answer.json"
    if not oracle_path.exists():
        return [f"Missing oracle_answer.json in {task_dir}"]
    oracle = json.loads(oracle_path.read_text())
    missing = REQUIRED_ORACLE_FIELDS - set(oracle.keys())
    if missing:
        return [f"Oracle missing fields {missing} in {task_dir}"]
    return []
```

This catches authoring gaps before they affect evaluation runs.

### 3.2 Add Request Timeouts to Judge LLM Backends

Both the Anthropic SDK and urllib backends should specify explicit
timeouts. The Anthropic SDK accepts a `timeout` parameter (in seconds)
on `client.messages.create()`. For urllib, pass `timeout=120` to
`urlopen()`. A 2-minute timeout is generous for judge calls (which
typically return in 5-15 seconds) while preventing indefinite hangs.

Additionally, implement a retry-with-backoff wrapper around the backend
`call()` method. Rate limits and transient errors are common during
batch judge runs. The retry logic already exists in the auth proxy
(`sg_auth_proxy.py` lines 95-101) and could be extracted into a shared
utility.

### 3.3 Standardize `task.toml` Structure with Schema Validation

The 13 malformed `task.toml` files (Section 1.11) and 144 casing
mismatches (Section 1.9) indicate structural drift in task authoring.
Create a `schemas/task_toml.schema.json` that encodes the required
sections (`[metadata]`, `[task]`, `[verifier]`) and field formats. Add
validation to `validate_tasks_preflight.py` using `tomllib` (Python 3.11+
stdlib) for parsing and the existing `jsonschema` dependency for
validation.

### 3.4 Unify Task ID Normalization

The two competing normalization strategies (Section 1.10) should be
consolidated into a single `normalize_task_id()` function in
`scripts/csb_metrics/models.py` (where `TaskMetrics` is defined). Both
`task_selection.py` and `discovery.py` should import this function.
The unified normalizer should handle:
- `mcp_` prefix stripping
- Hash suffix removal (both `_XXXXXX` and `__XXXXXXX` patterns)
- Case normalization for `CCX-` vs `ccx-` task IDs

### 3.5 Add Structured Error Classification to Judge Results

Currently, a judge score of 0.0 is ambiguous: it could mean the agent's
answer was genuinely incorrect, or the LLM response was malformed (missing
`dimension_scores`), or the oracle data was incomplete (missing `text`
field). Adding a `result_quality` field to `JudgeResult` with values like
`"full"`, `"partial_oracle"`, `"parse_error"` would enable downstream
analysis to distinguish evaluation failures from genuine agent failures.

## 4. Recommended Next Feature

### Oracle and Task Data Integrity Validation System

**The single most impactful feature** is a comprehensive validation layer
for oracle and task definition data. The previous three reports identified
infrastructure gaps (CI testing, operational tooling, evaluation code bugs).
This report reveals a deeper problem: **the benchmark data itself has
systematic quality issues** that silently degrade evaluation accuracy.

The scope: 66 oracle files missing `text` (46% of org tasks), 7 missing
`symbols`, 13 malformed `task.toml` files, 144 case-mismatched task IDs,
4 direct dict access sites that crash on incomplete oracle items, and
inconsistent token extraction that loses data when transcripts are
unavailable. These issues compound: incomplete oracle data feeds into
the judge system which produces degraded scores, which flow into
statistical comparisons that inform the project's conclusions about
MCP effectiveness.

**Part A -- Oracle completeness validation** (1 hour)

1. Add oracle field validation to `scripts/validate_tasks_preflight.py`:
   check that every `oracle_answer.json` contains `text`, `files`, and
   `symbols` fields
2. Add a `--fix-oracle` flag that generates stub `text` fields from
   existing `files`/`symbols` data for the 66 incomplete oracle files
3. Run the fix and commit the completed oracle data

**Part B -- Task.toml structural validation** (1 hour)

1. Add `task.toml` structure validation to `validate_tasks_preflight.py`:
   verify required sections (`[metadata]`, `[task]`), required fields
   (`id`, `suite`, `difficulty`), and ID casing consistency with directory
   names
2. Fix the 8 regression-prove tasks missing `[metadata]` and the 5 tasks
   using non-standard `[metadata].task_id` instead of `[task].id`
3. Normalize the 144 `CCX-` IDs to match directory casing, or update
   validation/matching code to be case-insensitive

**Part C -- Defensive dict access in retrieval metrics** (30 minutes)

1. Replace all 4 instances of `oracle_item["key"]` with
   `oracle_item.get("key", "")` in `scripts/csb_metrics/retrieval.py`
2. Add `isinstance(x, dict)` guards in `discovery.py` line 181 and
   `extractors.py` line 651 before calling `.get()` on list elements
3. Fix the `tt_all_r` overwrite bug in `ir_metrics.py` line 749 by
   adding a guard to capture only the first occurrence

**Part D -- Integration tests for data integrity** (1 hour)

1. Create `tests/test_oracle_integrity.py` that validates all 275 tasks'
   oracle files against expected schema
2. Create `tests/test_task_toml_integrity.py` that validates all
   `task.toml` files for structural consistency
3. Add both to the CI workflow (when created per March 6 recommendation)

**Why this is highest impact**: The March 6 report recommended CI testing
(foundational). March 7 recommended operational recovery (unblocking runs).
March 8 recommended evaluation code integrity (fixing bugs in metric
extraction). This report recommends **benchmark data integrity** because
it is upstream of all three: no amount of correct code can produce valid
results from incomplete or inconsistent input data. The 66 incomplete
oracle files affect 46% of org-scale evaluations. Fixing this data layer
directly improves the accuracy of every future evaluation run without
changing any pipeline code.

**PRD-ready description**: "Implement a benchmark data integrity validation
system consisting of: (1) oracle completeness checks in
`validate_tasks_preflight.py` that verify every `oracle_answer.json`

to generate stub text from existing data; (2) `task.toml` structural

`ir_metrics.py` to capture first occurrence rather than last; and (5)
integration test suites for oracle and task.toml data integrity covering
all 275 canonical tasks. Success criteria: `validate_tasks_preflight.py
