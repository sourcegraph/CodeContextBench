# Envoy Contributor Guide

## 1. Build Prerequisites

### System Dependencies

Before building Envoy, you must install platform-specific dependencies:

#### Ubuntu/Debian
```bash
sudo apt-get install \
   autoconf \
   curl \
   libtool \
   patch \
   python3-pip \
   unzip \
   virtualenv
```

#### Fedora/RHEL
```bash
dnf install \
    aspell-en \
    libatomic \
    libstdc++ \
    libstdc++-static \
    libtool \
    lld \
    patch \
    python3-pip
```

#### macOS
```bash
brew install coreutils wget libtool go bazelisk clang-format autoconf aspell
```
- Requires full Xcode (not just Command Line Tools)
- Tested with Apple clang version 11.0.0 or later

#### Windows
- Windows 10 SDK version 1903 (10.0.18362.1) or later
- Visual Studio 2019 Build Tools with VC++ workload
- Python 3 (from python.org, not MSYS2)
- Git
- MSYS2

### Compiler Requirements

- **Clang**: Version 9 or later (CI uses Clang 14)
- **GCC**: Version 9 or later
- **C++17 support**: Required

### Build Tools

1. **Bazelisk** (recommended instead of Bazel to avoid version conflicts)
   - Linux: `sudo wget -O /usr/local/bin/bazel https://github.com/bazelbuild/bazelisk/releases/latest/download/bazelisk-linux-$([ $(uname -m) = "aarch64" ] && echo "arm64" || echo "amd64") && sudo chmod +x /usr/local/bin/bazel`
   - macOS: `brew install bazelisk`
   - Windows: Download from GitHub releases and add to PATH

2. **Go** (version 1.17 or later) - Required for BoringSSL and Buildifier

3. **Buildifier and Buildozer** - For formatting Bazel BUILD files
   ```bash
   go install github.com/bazelbuild/buildtools/buildifier@latest
   go install github.com/bazelbuild/buildtools/buildozer@latest
   ```

### Optional Setup: Development Support Toolchain

Install git hooks for automatic DCO signoff and pre-commit/pre-push checks:
```bash
./support/bootstrap
```

This sets up:
- Automatic DCO (Developer Certificate of Origin) signoff on commits
- Format checking on push
- Can be bypassed with `git commit --no-verify` if needed

**Documentation**: See `support/README.md`

## 2. Build System

Envoy uses **Bazel** as its build system. Bazel is a fast, scalable build system that supports building and testing large codebases efficiently.

### Key Build Commands

#### Basic Build (debug/development)
```bash
# Build main Envoy binary
bazel build //source/exe:envoy

# Build with fastbuild (unoptimized, faster builds)
bazel build //source/exe:envoy -c fastbuild

# Build with optimization (release)
bazel build -c opt //source/exe:envoy

# Build with debug symbols
bazel build -c dbg //source/exe:envoy

# Build entire Envoy (including all extensions)
bazel build //source/exe:envoy-static
```

#### Building Specific Components
```bash
# Build only a specific filter
bazel build //source/extensions/filters/http/router:config_lib

# Build a library for a subsystem
bazel build //source/common/http:...

# Use ... to build all targets in a directory
bazel build //source/common/...
```

#### Compiler Configuration
```bash
# Build with libc++ (recommended for official releases)
bazel build --config=libc++ //source/exe:envoy

# Build with libstdc++
bazel build --config=clang //source/exe:envoy

# Build with GCC
bazel build //source/exe:envoy  # default without --config=clang

# Use Clang specifically
bazel build --config=clang //source/exe:envoy
```

#### Using Docker (Recommended for First-Time Setup)
Avoids environment issues and ensures consistency with CI:
```bash
# Build development version using Docker
./ci/run_envoy_docker.sh './ci/do_ci.sh dev'

# Build release version
./ci/run_envoy_docker.sh './ci/do_ci.sh release.server_only'

# Build debug version
./ci/run_envoy_docker.sh './ci/do_ci.sh debug.server_only'
```

Output locations:
- Dev build: `/tmp/envoy-docker-build/envoy/source/exe/envoy-fastbuild`
- Release: `/tmp/envoy-docker-build/envoy/source/exe/envoy`
- Debug: `/tmp/envoy-docker-build/envoy/source/exe/envoy-debug`

You can change output directory: `ENVOY_DOCKER_BUILD_DIR=~/build ./ci/run_envoy_docker.sh './ci/do_ci.sh dev'`

### Build Profiles

- **fastbuild** (`-c fastbuild`): Unoptimized, minimal symbols, fastest compilation
- **opt** (`-c opt`): Full optimization, minimal debug info, suitable for release
- **dbg** (`-c dbg`): Unoptimized, full debug symbols, suitable for debugging with GDB

### Getting Build Info
```bash
# Find the built binary path
bazel info bazel-genfiles

# Run the built binary
$(bazel info bazel-genfiles)/source/exe/envoy --config-path /path/to/config.yaml
```

### Bazel Configuration Files

- **Main config**: `.bazelrc` in repository root
- **User overrides**: `user.bazelrc` in repository root (not tracked by git)
- **Clang setup**: `clang.bazelrc` (generated by `bazel/setup_clang.sh`)

## 3. Running Tests

Envoy uses the **Google Test** (gtest) framework with Google Mock for unit tests and a custom integration test framework.

### Test Structure

- **Unit tests**: Located in `test/` directories matching source structure
  - Example: source code in `source/common/http/` has tests in `test/common/http/`
  - Use Google Test framework with custom matchers
- **Integration tests**: Located in `test/integration/`
  - End-to-end testing with fake downstream/upstream servers
  - Test real Envoy server code behavior
- **Test frameworks**:
  - Google Test + Google Mock for unit tests
  - Custom `HttpIntegrationTest` framework for integration tests

### Running Tests

#### All Tests
```bash
# Run all tests
bazel test //test/...

# Run with verbose output
bazel test --test_output=streamed //test/...
```

#### Specific Test Targets
```bash
# Run a single test file
bazel test //test/common/http:async_client_impl_test

# Run tests matching a pattern
bazel test //test/common/http/...

# Run tests in specific subsystem
bazel test //test/extensions/filters/http/router:...
```

#### With Additional Options
```bash
# Verbose test output
bazel test --test_output=streamed //test/common/http:async_client_impl_test

# Pass arguments to test binary
bazel test --test_arg="-l trace" //test/common/http:async_client_impl_test

# Skip cache, rerun even if passed before
bazel test --cache_test_results=no //test/common/http:async_client_impl_test

# Test with IPv4 only
bazel test //test/... --test_env=ENVOY_IP_TEST_VERSIONS=v4only

# Test with IPv6 only
bazel test //test/... --test_env=ENVOY_IP_TEST_VERSIONS=v6only
```

#### Coverage Builds
```bash
# Generate coverage report (requires specific setup)
bazel test --config=coverage //test/...

# Coverage build must achieve 100% for new code (validated in CI)
```

### Testing with Docker
```bash
# Run all tests in Docker (matches CI environment)
./ci/run_envoy_docker.sh './ci/do_ci.sh test'
```

### Heap Checking
By default, tests run with gperftools heap checker in "normal" mode:
```bash
# Disable heap checker
bazel test //test/... --test_env=HEAPCHECK=

# Use minimal mode instead
bazel test //test/... --test_env=HEAPCHECK=minimal
```

### Integration Test Helpers

**Location**: `test/integration/`

Key classes:
- `HttpIntegrationTest`: Base class for HTTP integration tests
- `AutonomousUpstream`: Automatically responds to requests (useful for simple tests)
- `ConfigHelper`: Utilities for modifying Envoy configuration in tests

Example test structure:
```cpp
// Initialize Envoy with default config
initialize();

// Create client connection
codec_client_ = makeHttpConnection(makeClientConnection((lookupPort("http"))));

// Create request and send it
Http::TestRequestHeaderMapImpl request_headers{...};
auto response = sendRequestAndWaitForResponse(request_headers, 0, ...);

// Verify results
EXPECT_TRUE(response->complete());
```

See `test/integration/README.md` for detailed integration test documentation.

### Debugging Tests
```bash
# Build test binary with debug symbols
bazel build -c dbg //test/common/http:async_client_impl_test
bazel build -c dbg //test/common/http:async_client_impl_test.dwp

# Debug with GDB
gdb bazel-bin/test/common/http/async_client_impl_test
```

### Custom Test Matchers

Envoy provides custom Google Mock matchers:
- `HeaderValueOf(key, value)`: Check header values
- `HttpStatusIs(status)`: Check HTTP status
- `ProtoEq(proto)`: Compare protobuf messages
- `HeaderMapEqual` / `HeaderMapEqualRef`: Compare header maps

## 4. CI Pipeline

Envoy uses multiple CI systems that automatically test all pull requests:

### CI Platforms

1. **Azure Pipelines** (Primary)
   - Main CI platform
   - Runs on all PR submissions
   - **URL**: https://dev.azure.com/cncf/envoy/

2. **BazelCI**
   - Runs remote execution tests
   - Tests on Ubuntu 2004
   - Tests: `//test/common/common/...`, `//test/integration/...`, `//test/exe/...`

3. **Fuzz Testing**
   - OSS-Fuzz continuous fuzzing
   - **Badge**: Shown in README.md

4. **Architecture-Specific CI**
   - ppc64le builds (Jenkins: powerci.osuosl.org)
   - s390x builds (Jenkins: ibmz-ci.osuosl.org)

### CI Configuration Files

- **Main CI config**: `.azure-pipelines/` directory
- **BazelCI config**: `.bazelci/presubmit.yml`
- **Script runners**: `ci/do_ci.sh`, `ci/run_envoy_docker.sh`
- **Build setup**: `ci/build_setup.sh`
- **Docker images**: See `ci/README.md`

### Checks That Run on PRs

The CI pipeline validates:
1. **Build**: Compilation with various configurations (Clang, GCC)
2. **Unit Tests**: All Google Test tests pass
3. **Integration Tests**: End-to-end tests pass
4. **Code Format**: `clang-format` compliance (enforced by `.clang-tidy`)
5. **Code Analysis**: `clang-tidy` linter checks
6. **Coverage**: New code must have 100% test coverage
7. **Documentation**: Markdown and RST validation
8. **API**: Protocol buffer changes validated against API checklist

### Typical CI Run Duration

- Standard PR tests: **1-3 hours**
- Full coverage build: **2-4 hours**
- Release builds: **Varies by configuration**

### Docker Build Images

- **Ubuntu**: `envoyproxy/envoy-build-ubuntu` (Ubuntu 20.04, GCC 9, Clang 14)
- **Alpine**: `envoyproxy/envoy-build-alpine` (minimal image)
- **Windows**: `envoyproxy/envoy-build-windows2019` (Note: Windows support ended Aug 2023, patches accepted)

Get image hash: `cat ci/envoy_build_sha.sh`

### Rerunning CI Without Code Changes

```bash
# Rerun failed tasks only
# Comment on PR with:
/retest

# Rerun all checks (empty commit push)
git commit -s --allow-empty -m 'Kick CI'
git push

# Or use git alias:
git config --add alias.kick-ci '!git commit -s --allow-empty -m "Kick CI" && git push'
git kick-ci
```

### CI Environment Variables

- `ENVOY_RBE`: Use remote execution
- `ENVOY_STDLIB`: Choose `libstdc++` or `libc++` (default: `libc++`)
- `IMAGE_NAME`: Choose Docker image (default: `envoyproxy/envoy-build-ubuntu`)
- `ENVOY_DOCKER_PULL=true`: Force Docker image refresh

## 5. Code Review Process

### Before Starting Work

1. **Check GitHub Issues**
   - Search for existing issues related to your change
   - Check if someone is already working on it

2. **For Major Features**
   - Create a GitHub issue for discussion
   - **Major** = > 100 lines of code (excluding tests) OR changes user-facing behavior
   - Discuss design and get maintainer agreement before implementing
   - Prevents wasted effort and ensures architectural alignment

3. **For Small Bugs and Fixes**
   - No prior communication needed
   - Just submit the PR

4. **For Extensions**
   - Read **EXTENSION_POLICY.md** for extension addition policy
   - Review **GOVERNANCE.md** for extension guidelines

**Documentation**: `CONTRIBUTING.md`, `PULL_REQUESTS.md`, `EXTENSION_POLICY.md`

### Contribution Requirements

Every PR must satisfy:

1. **100% Test Coverage**
   - All new code must be covered by tests
   - Can verify with coverage build
   - If coverage impossible, explain why in PR description

2. **Documentation**
   - Changes to user-facing behavior require docs in `docs/` directory
   - API changes documented inline with protos
   - Release notes added to `changelogs/current.yaml` for user-impacting changes

3. **DCO Sign-off**
   - Every commit must be signed off: `Signed-off-by: Your Name <email@example.com>`
   - Set up automatic signoff: `./support/bootstrap`
   - Or use `git commit -s`

4. **Code Style**
   - Follow **STYLE.md** guidelines
   - Code formatted with `clang-format` (automatically checked by CI)
   - Follow Google C++ style guide with Envoy deviations:
     - Function names: `camelCase` starting with lowercase (e.g., `doFoo()`)
     - Member variables: postfix `_` (e.g., `foo_`)
     - Enums: PascalCase (e.g., `RoundRobin`)
     - Line limit: 100 columns
     - Include guards: `#pragma once`

5. **Inclusive Language**
   - No "whitelist" → use "allowlist"
   - No "blacklist" → use "denylist" or "blocklist"
   - No "master" → use "primary" or "main"
   - No "slave" → use "secondary" or "replica"

6. **Tests Must Pass**
   - All existing tests must continue to pass
   - New tests must pass
   - CI must be green before merge

### Submitting a PR

1. **Fork the repository** on GitHub

2. **Create a feature branch**
   ```bash
   git checkout -b my-feature
   ```

3. **Install git hooks**
   ```bash
   ./support/bootstrap
   ```
   This provides:
   - Automatic DCO signoff
   - Pre-push format checking

4. **Implement your changes**
   - Add tests
   - Run tests locally: `bazel test //test/...`
   - Run format checks: `bazel run //tools/code_format:check_format -- fix`

5. **Commit with DCO signoff**
   ```bash
   git commit -s -m "subsystem: brief description"
   ```
   Example messages:
   - `"docs: fix grammar error"`
   - `"http conn man: add new feature"`
   - `"router: fix bug in retry logic"`

6. **Create Pull Request**
   - Use descriptive title (starts with subsystem)
   - Provide detailed commit message explaining what and why
   - Include PR description with sections:
     - **Risk**: Low | Medium | High
     - **Testing**: What testing was done
     - **Documentation**: Changes to docs/API or "N/A"
     - **Release Notes**: Add to `changelogs/current.yaml` if user-impacting
     - **Fixes**: Link to issues (e.g., "Fixes #1234")
   - Use PR template: `PULL_REQUEST_TEMPLATE.md`

7. **Address Review Feedback**
   - **DO NOT rebase** after review starts (forces reviewer to re-review)
   - **DO NOT force-push** (use `git merge main` instead to pull latest changes)
   - Push new commits to address feedback
   - Re-request review after addressing comments

### Code Review Expectations

- **Review turnaround**: Typically within 1 business day
- **Reviewers**:
  - Senior maintainer for core code
  - Domain expert for specific areas
  - At least one approver from different organization for new extensions
- **Merge process**: Maintainer will squash-rebase and merge when approved
  - Final commit message cleaned up before merge
  - DCO signoff preserved

### PR Fields to Complete

**Title**: `subsystem: brief description`
- Lowercase for both subsystem and description
- Examples: `"http conn man: add new feature"`, `"router: fix x-envoy-overloaded header"`

**Commit Message**: Explains what and why
- Used as final commit message when merged
- Document behavior before/after if applicable
- Note configuration options if any

**Additional Description**: Context for reviewers
- Can include information beyond commit message scope
- References to related discussions or issues

**Risk Level**:
- **Low**: Small bug fix or small optional feature
- **Medium**: New features, small-medium changes to existing components
- **High**: Complex changes, flow control, critical component rewrites

**Testing**: Describe testing performed
- Unit tests, integration tests, manual testing
- Higher risk = more comprehensive testing expected

**Documentation**: Describe doc changes
- Write "N/A" if no docs changes
- Include inline proto docs for API changes
- Update "Life of a Request" if structural dataplane changes

**Release Notes**: Add entry to `changelogs/current.yaml`
- Required if user-impacting OR extension-impacting
- Include relevant documentation links
- Format: See existing examples in changelog

**Platform-Specific Features**: If applicable
- Explain OS-specific socket options
- Create tracking issues for platforms not supported
- Avoid `#ifdef OSNAME`, use feature guards

**Runtime Guard**: If needed
- Add for user-visible behavioral changes
- Document in release notes and PR description
- See CONTRIBUTING.md runtime guarding section

### Special Cases

**Deprecations**:
- Update `changelogs/current.yaml` deprecated section
- Deprecation script will create cleanup issue in next release
- Note specific deprecation in commit message

**API Changes**:
- Review API Review Checklist: `api/review_checklist.md`
- Document API considerations in PR
- Changes to `api/` tree require special review

**Runtime Guards**:
- For high-risk behavioral changes
- Keep both old and new code paths
- Default to new behavior after testing period
- Remove old path after deprecation window

### Repository Etiquette

- Push to personal fork, not main Envoy repo (keeps it lean)
- Use branches in fork for feature branches
- Avoid rebasing after review starts

## 6. Developer Workflow Example

This example walks through implementing a bug fix to the HTTP connection manager filter.

### Step 1: Prepare Your Environment

```bash
# Clone the repository
git clone https://github.com/YOUR_USERNAME/envoy.git
cd envoy

# Set up git hooks for DCO and format checking
./support/bootstrap

# Install Bazel/Bazelisk if not already installed
sudo wget -O /usr/local/bin/bazel https://github.com/bazelbuild/bazelisk/releases/latest/download/bazelisk-linux-amd64
sudo chmod +x /usr/local/bin/bazel
```

### Step 2: Identify the Issue

Suppose you want to fix a bug in the HTTP connection manager filter that's causing it to incorrectly handle timeout values.

```bash
# Find related code
grep -r "timeout" source/extensions/filters/network/http_connection_manager/ | grep -i bug

# Look at the filter structure
ls -la source/extensions/filters/network/http_connection_manager/
```

### Step 3: Create a Feature Branch

```bash
git checkout -b fix-http-conn-man-timeout
```

### Step 4: Write Tests First

Create a test file to demonstrate the bug:
```bash
# Find existing test file for the component
ls test/extensions/filters/network/http_connection_manager/

# Open the test file and add a test case that fails with current code
vim test/extensions/filters/network/http_connection_manager/http_connection_manager_test.cc
```

Example test structure:
```cpp
// test/extensions/filters/network/http_connection_manager/http_connection_manager_test.cc
TEST_F(HttpConnectionManagerTest, TimeoutBugFix) {
  // Setup config with timeout value
  setupBasicConfig();
  config_helper_.addRuntimeOverride("envoy.http.timeout", "5000");

  initialize();

  // Create connection and verify timeout behavior
  codec_client_ = makeHttpConnection(...);

  // Verify fix: timeout should be applied correctly
  EXPECT_EQ(5000U, conn_manager_->getTimeout());
}
```

### Step 5: Run the Failing Test

```bash
# Build and run your new test
bazel test //test/extensions/filters/network/http_connection_manager:http_connection_manager_test \
  --test_arg="--gtest_filter=HttpConnectionManagerTest.TimeoutBugFix" \
  --test_output=streamed

# Should fail with current code
```

### Step 6: Implement the Fix

Locate and fix the bug in the implementation:
```bash
# Find the implementation file
vim source/extensions/filters/network/http_connection_manager/conn_manager.cc

# Make the fix to handle timeout values correctly
```

### Step 7: Verify the Fix

```bash
# Run just the test you fixed
bazel test //test/extensions/filters/network/http_connection_manager:http_connection_manager_test \
  --test_arg="--gtest_filter=HttpConnectionManagerTest.TimeoutBugFix" \
  --test_output=streamed

# Should now pass

# Run all tests for the http_connection_manager to ensure no regressions
bazel test //test/extensions/filters/network/http_connection_manager:...

# Run a broader set of integration tests
bazel test //test/integration:... -k "http"
```

### Step 8: Verify Code Style and Format

```bash
# Fix code formatting automatically
bazel run //tools/code_format:check_format -- fix

# Run clang-tidy checks
bazel run //tools/code:check -- fix -s main

# Or in Docker if you don't have the tools installed locally
./ci/run_envoy_docker.sh './ci/do_ci.sh format'
```

### Step 9: Add Documentation and Release Notes

```bash
# If this bug fix affects user-facing behavior, add release notes
vim changelogs/current.yaml
```

Example entry:
```yaml
- area: http_conn_man
  change: |
      fixed: Corrected timeout handling in HTTP connection manager filter to properly apply
      configured timeout values. Previously timeout values were incorrectly parsed and could
      cause connections to timeout prematurely. See #1234.
```

```bash
# If there's user-facing documentation, update it
vim docs/root/configuration/http/http_filters/http_connection_manager.rst
```

### Step 10: Commit Your Changes

```bash
# Stage your changes
git add source/extensions/filters/network/http_connection_manager/conn_manager.cc
git add test/extensions/filters/network/http_connection_manager/http_connection_manager_test.cc
git add changelogs/current.yaml
git add docs/root/configuration/...

# Commit with DCO signoff (./support/bootstrap adds the hook for this)
git commit -s -m "http conn man: fix timeout value handling

This fixes a bug where timeout values in the HTTP connection manager filter
were not being correctly parsed, causing connections to timeout prematurely.

Previously:
- Timeout values were passed directly without validation
- Integer overflow could occur on edge cases

Now:
- Timeout values are properly validated and sanitized
- Edge cases are handled correctly

Fixes #1234"

# Git hook will automatically add your DCO signoff
```

### Step 11: Create a Pull Request

```bash
# Push your branch to your fork
git push origin fix-http-conn-man-timeout
```

Then on GitHub:
1. Click "Create Pull Request"
2. Fill in the PR template:
   - **Title**: `"http conn man: fix timeout value handling"`
   - **Risk Level**: `Low` (bug fix)
   - **Testing**: `"Added unit test TestTimeoutBugFix to verify timeout values are properly handled. Ran full http_connection_manager test suite. Verified no regressions with integration tests."`
   - **Documentation**: `"Updated changelogs/current.yaml with fix description. No API changes."`
   - **Release Notes**: Added to `changelogs/current.yaml`
   - **Fixes**: `"Fixes #1234"`

### Step 12: Address Review Feedback

When a maintainer reviews your PR:

```bash
# Make requested changes
vim source/extensions/filters/network/http_connection_manager/conn_manager.cc

# Verify tests still pass
bazel test //test/extensions/filters/network/http_connection_manager:...

# Commit the fix (don't rebase!)
git commit -s -m "Address review feedback: improve error message clarity"

# Push the new commit
git push origin fix-http-conn-man-timeout

# The maintainer will see just the new changes, not re-review the whole PR
```

### Step 13: Merge

Once all checks pass (CI tests turn green) and a maintainer approves:
1. Maintainer will squash-merge your PR
2. Your commits are squashed into one
3. Commit message is cleaned up if needed
4. Your DCO signoff is preserved
5. Change is now in main branch

### Step 14: Verify on Main

```bash
# Update your local repo
git checkout main
git pull

# Verify your change is there
git log --oneline | grep "timeout value handling"
```

## Troubleshooting

### Build Issues

**Clang not found**:
- Install from `bazel/setup_clang.sh` or specify compiler: `bazel build --config=clang //source/exe:envoy`

**Long paths error on Windows**:
- Set `startup --output_base=C:/_eb` in `.bazelrc` to shorten Bazel paths

**Out of disk space**:
- Bazel can use significant disk space. Clean with `bazel clean`

### Test Failures

**Flaky tests**:
- Rerun with `--cache_test_results=no`
- Use simulated time in tests instead of real time

**Heap check failures**:
- Disable with `--test_env=HEAPCHECK=` or switch mode: `--test_env=HEAPCHECK=minimal`

**Tests timing out**:
- Run with more verbose output to debug: `--test_output=streamed`
- Check system load and resources

### Git/DCO Issues

**DCO signature missing**:
- Re-run `./support/bootstrap` to set up hooks
- Or manually add: `git commit -s` or `git commit --amend -s`

**Can't push (pre-push hook failing)**:
- Check format with `bazel run //tools/code_format:check_format -- fix`
- Or bypass with `git push --no-verify` (not recommended)
