Store planned change in plan file\n```\n\n**Call Sequence During Apply:**\n\n```\nFor each resource node in execution order:\n  1. Provider.ApplyResourceChange(planned_state, config)\n  2. Update state with new actual state\n  3. If error: handle according to lifecycle rules\n```\n\n### 2.4 Error Handling and Recovery\n\n**Provider-Level Errors:**\n\n- **Transient Errors**: Network timeouts, rate limiting\n  - Provider may retry automatically\n  - Terraform may suggest manual retry\n\n- **Validation Errors**: Invalid configuration, unsupported attributes\n  - Reported during validation phase\n  - Prevent plan from completing\n\n- **Operational Errors**: API failures, permission issues\n  - During apply, recorded but don't necessarily stop execution\n  - Can trigger lifecycle actions (e.g., destroy on error)\n\n**Provider Crash Handling:**\n\n- Provider crashes are detected via gRPC connection loss\n- Terraform propagates crash as plan/apply error\n- State is preserved to enable recovery\n- User can retry or investigate provider logs\n\n### 2.5 Provider Configuration Evaluation\n\nProvider blocks are evaluated as part of graph execution:\n\n```hcl\nprovider \"aws\" {\n  region = var.aws_region\n\n  assume_role {\n    role_arn = aws_iam_role.terraform_role.arn\n  }\n}\n```\n\n**Evaluation Process:**\n\n1. Provider block configuration is identified\n2. All variable references are resolved\n3. All resource/data source references are evaluated (if provider depends on resources)\n4. Final configuration is passed to provider plugin\n5. Provider validates configuration and stores for use with resources\n\n**Dynamic Credentials:**\n\n- Providers may assume roles, fetch tokens, or perform other dynamic operations\n- Provider configuration can reference resources (e.g., assume role from created resource)\n- Creates potential dependency loops - Terraform detects and reports these\n\n---\n\n## 3. State Management\n\nTerraform state is the source of truth for actual infrastructure. State management ensures consistency between configuration and infrastructure across multiple operations and users.\n\n### 3.1 State Structure and Content\n\n**State File Format:**\n\nTerraform state is stored as JSON in `.terraform/tfstate`:\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.x.x\",\n  \"serial\": 42,\n  \"lineage\": \"uuid-string\",\n  \"outputs\": {\n    \"instance_ip\": {\n      \"value\": \"10.0.0.1\",\n      \"type\": \"string\"\n    }\n  },\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"web\",\n      \"instances\": [\n        {\n          \"index_key\": 0,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"ami\": \"ami-0c55b159cbfafe1f0\",\n            \"instance_type\": \"t2.micro\",\n            \"private_ip\": \"10.0.0.1\"\n          },\n          \"sensitive_attributes\": [],\n          \"private\": \"base64-encoded-data\"\n        }\n      ]\n    }\n  ]\n}\n```\n\n**Key Elements:**\n\n- **version**: State format version (currently 4)\n- **terraform_version**: Version of Terraform that wrote state\n- **serial**: Monotonically increasing counter for change detection\n- **lineage**: UUID maintained across state versions, detects state replacement\n- **outputs**: Computed output values\n- **resources**: Array of managed resources with instances\n- **attributes**: Resource attributes as returned by provider\n- **sensitive_attributes**: Paths to attributes containing sensitive values\n- **private**: Provider-specific data (encrypted in remote state)\n\n**State Metadata:**\n\nEach resource instance includes:\n- Resource type and name\n- Instance index (for count/for_each)\n- All attributes returned by provider\n- Sensitive attribute paths\n- Provider-specific data\n- Dependency information (implicit from graph)\n\n### 3.2 State Reading and Modification\n\n**State Loading:**\n\n1. **Local State**: Loaded from `.terraform/tfstate`\n   - Parsed as JSON\n   - Validated for consistency\n   - In-memory representation created\n\n2. **Remote State**: Loaded from configured backend\n   - HTTP request to backend API\n   - State decrypted and decompressed\n   - Merged with local state cache if `backend` block specifies it\n   - Enables team collaboration and state safety\n\n3. **State Locking**:\n   - Before modification, lock is acquired\n   - Prevents concurrent operations\n   - Lock timeout and force-unlock available for recovery\n   - Different backends implement locking differently (DynamoDB for S3, etc.)\n\n**State Modification:**\n\nState is modified through a **delta application** process:\n\n1. **Planned Changes**: Plan file contains:\n   - Resources to be created, updated, deleted, or replaced\n   - Proposed attribute values\n   - Metadata about change operations\n\n2. **Atomic Application**: During apply:\n   - State is locked to prevent concurrent modifications\n   - For each resource in planned order:\n     - Provider executes change\n     - Returned actual state is written to state\n     - State serial increments\n   - Final state is persisted\n\n3. **Partial Application Recovery**:\n   - If apply fails midway, state reflects completed operations\n   - Incomplete resource operations leave state unchanged\n   - User can retry apply (may require manual intervention)\n\n### 3.3 State Snapshots and Concurrent Evaluation\n\n**State Snapshots for Graph Traversal:**\n\nThe state snapshot is a critical mechanism enabling safe concurrent evaluation:\n\n```\nState Snapshot (Point-in-time)\n    ↓\n[Passed to all graph node evaluations]\n    ↓\n[Enables safe planning without locking for the entire plan]\n```\n\n**How Snapshots Enable Concurrency:**\n\n1. **Read Lock Acquired**: Full state is read and locked briefly\n2. **Snapshot Created**: In-memory copy of entire state taken\n3. **Lock Released**: State lock is released immediately\n4. **Snapshot Used**: All graph nodes use snapshot for reference dependencies\n   - No need to re-read from state during planning\n   - Reference resolution is consistent (same snapshot)\n   - No conflicting updates possible\n\n5. **Validation**: Before apply, snapshot is compared with current state\n   - Detects concurrent modifications by other users\n   - Prevents applying stale plan to modified state\n   - Forces user to re-plan if state changed\n\n**Benefits:**\n\n- **Planning Lock Duration**: Locks held only for milliseconds during snapshot creation\n- **Concurrent Operations**: Multiple users can plan simultaneously\n- **Consistency**: All references within a plan see consistent state\n- **Merge Conflict Detection**: Concurrent state changes are detected before apply\n\n### 3.4 State Synchronization\n\n**Refresh Operation:**\n\nBefore planning, Terraform can refresh state to match actual infrastructure:\n\n```\nFor each resource in state:\n  1. Call provider ReadState\n  2. Compare actual state with state file\n  3. Update state file with actual values\n  4. Mark if actual state differs from configuration\n```\n\n**Scenarios:**\n\n- Out-of-band changes: Manual API calls modifying infrastructure\n- Attribute drift: Resources modified outside Terraform\n- Computed value changes: Time-sensitive attributes (certificates, etc.)\n\n**Refresh Modes:**\n\n- `-refresh=true` (default for plan): Query all resources\n- `-refresh=false`: Skip refresh, use cached state\n- Useful for speed when state is known current\n- Risk: Plans based on stale state\n\n**State Refreshing During Plan:**\n\n1. **Implicit Refresh**: Plan phase refreshes state (configurable)\n2. **Selective Refresh**: Only non-targeted resources refreshed\n3. **Output of Refresh**: Plan shows drift before any changes\n\n### 3.5 Remote State and Backends\n\n**Backend Configuration:**\n\n```hcl\nterraform {\n  backend \"s3\" {\n    bucket         = \"terraform-state\"\n    key            = \"prod/terraform.tfstate\"\n    region         = \"us-east-1\"\n    encrypt        = true\n    dynamodb_table = \"terraform-locks\"\n  }\n}\n```\n\n**Backend Responsibilities:**\n\n1. **Storage**: Persist state file\n2. **Retrieval**: Provide state on demand\n3. **Locking**: Coordinate concurrent access (if supported)\n4. **Versioning**: Maintain state history (if supported)\n5. **Encryption**: Protect sensitive state data\n\n**Common Backends:**\n\n- **S3**: AWS S3 with DynamoDB locking\n- **Azure Blob Storage**: Azure Storage with blob lease locking\n- **Google Cloud Storage**: GCS with state versioning\n- **Terraform Cloud**: Managed state with built-in locking\n- **Consul**: Distributed state with Consul locking\n- **etcd**: Kubernetes-native state backend\n- **HTTP**: Generic HTTP endpoint for custom backends\n\n**Local vs. Remote State Trade-offs:**\n\n| Aspect | Local | Remote |\n|--------|-------|--------|\n| Team Collaboration | Manual sync/merge | Automatic |\n| Locking | File system | Backend-dependent |\n| Versioning | None | Backend-dependent |\n| Encryption | Unencrypted on disk | Backend-dependent |\n| Latency | Local access | Network dependent |\n\n### 3.6 State Operations and Safety\n\n**State Listing:**\n\n```\nterraform state list\n  ↓\nOutputs: All managed resource addresses\n  ↓\nFormat: aws_instance.web, aws_instance.web[0], module.vpc.aws_subnet.public\n```\n\n**State Inspection:**\n\n```\nterraform state show aws_instance.web[0]\n  ↓\nOutputs: All attributes for specific resource\n  ↓\nShows actual resource state from state file\n```\n\n**State Manipulation (Advanced):**\n\n- `terraform state rm`: Remove resource from state (without destroying)\n- `terraform state mv`: Move resource address (rename)\n- `terraform state replace-provider`: Change provider for resource\n- **Use with caution**: These operations bypass safety checks\n\n---\n\n## 4. Execution Flow and Hook System\n\nThe execution flow orchestrates graph traversal, node evaluation, and hook invocation to execute plan or apply operations.\n\n### 4.1 Overall Command Flow\n\n**Complete Terraform Execution Lifecycle:**\n\n```\nterraform plan [options] [dir]\n    ↓\nLoad Configuration & Parse\n    ↓\nInitialize Providers\n    ↓\nBuild Resource Graph\n    ↓\nValidate Graph\n    ↓\nRefresh State (optional)\n    ↓\nPlan Graph Evaluation\n    ↓\nDisplay Plan to User\n    ↓\nWrite Plan File\n    ↓\nPlan Complete\n\n\nterraform apply [options] [plan-file]\n    ↓\nLoad Configuration & Plan File\n    ↓\nInitialize Providers (if not from plan)\n    ↓\nRequest User Confirmation\n    ↓\nApply Graph Evaluation\n    ↓\nUpdate State File\n    ↓\nPersist Remote State\n    ↓\nApply Complete\n```\n\n### 4.2 Graph Node Evaluation\n\n**Execution Semantics:**\n\nNodes are evaluated respecting:\n1. **Dependency Order**: Edges create ordering constraints\n2. **Parallelization**: Nodes with no blocking dependencies run concurrently\n3. **Determinism**: Same configuration produces same plan (modulo external state)\n4. **Idempotence**: Repeated applies on unchanged resources are no-ops\n\n**Planning Phase Evaluation:**\n\nFor each resource node:\n\n```go\n1. Refresh (if enabled):\n   provider.ReadResourceState(state, config)\n   → updates state with actual values\n\n2. Validate:\n   provider.ValidateResourceConfig(config)\n   → errors prevent plan\n\n3. Plan:\n   provider.PlanResourceChange(prior_state, proposed_state, config)\n   → determines change type and planned state\n\n4. Store Planned Change:\n   plan_file[resource] = planned_change\n```\n\n**Change Types:**\n\n- **NoOp**: Resource unchanged\n- **Create**: Resource doesn't exist in state, configuration has it\n- **Update**: Resource exists, configuration specifies different values\n- **Delete**: Resource in state, not in configuration\n- **Replace**: Update requires resource destruction and recreation\n- **Read**: Data source read (no state change)\n\n**Apply Phase Evaluation:**\n\nFor each resource node in plan order:\n\n```go\n1. Verify Plan Still Valid:\n   if current_state != plan.prior_state:\n       error \"state changed since plan\"\n\n2. Apply Change:\n   if change == Create:\n       new_state = provider.ApplyResourceChange(null, planned_state, config)\n   else if change == Update:\n       new_state = provider.ApplyResourceChange(prior_state, planned_state, config)\n   else if change == Delete:\n       provider.ApplyResourceChange(prior_state, null, config)\n       remove from state\n   else if change == NoOp:\n       skip provider call, state unchanged\n\n3. Update State:\n   state[resource] = new_state\n   persist state\n\n4. Handle Errors:\n   if provider.error:\n       check lifecycle.on_failure\n       potentially delete resource\n       report error\n```\n\n### 4.3 Walker Pattern for Graph Traversal\n\nThe **Walker** is the core execution engine that traverses the dependency graph.\n\n**Walker Responsibilities:**\n\n1. **Dependency Tracking**: Maintains which nodes are ready to execute\n2. **Concurrency Control**: Limits parallel execution (default: 10 concurrent)\n3. **Ordering**: Executes nodes respecting dependency constraints\n4. **Hook Invocation**: Calls hooks at appropriate execution points\n5. **Error Handling**: Handles and reports errors from node evaluation\n\n**Walker Algorithm:**\n\n```\n1. Initialize:\n   pending_nodes = all graph nodes\n   ready_nodes = nodes with no dependencies\n   running_nodes = empty set\n   completed_nodes = empty set\n\n2. Loop while pending_nodes not empty or running_nodes not empty:\n\n   a. Fill available slots (up to concurrency limit):\n      while len(running_nodes) < parallelism:\n          if ready_nodes not empty:\n              node = ready_nodes.pop()\n              start_async_evaluation(node)\n              running_nodes.add(node)\n          else:\n              break\n\n   b. Wait for any node to complete:\n      completed_node = wait_for_any(running_nodes)\n\n   c. Process completion:\n      if completed_node.error:\n          mark as failed\n          invoke error hooks\n          propagate error to dependents\n      else:\n          mark as completed\n          invoke success hooks\n          calculate newly ready nodes\n\n      remove from running_nodes\n      add dependents to ready_nodes\n\n3. Report results:\n   if any failed:\n       report failures\n   else:\n       report success\n```\n\n**Parallelization:**\n\n- Default concurrency: 10 nodes\n- Configurable via `-parallelism=N`\n- Only nodes with no blocking dependencies run concurrently\n- Heavy I/O operations benefit from parallelism\n- Resource creation typically parallelizable if no explicit dependencies\n\n**Example Parallelization:**\n\n```hcl\n# These three resources can run in parallel (no dependencies)\nresource \"aws_instance\" \"web_1\" { ... }\nresource \"aws_instance\" \"web_2\" { ... }\nresource \"aws_instance\" \"web_3\" { ... }\n\n# This depends on all three above, must wait\nresource \"aws_lb\" \"main\" {\n  depends_on = [\n    aws_instance.web_1,\n    aws_instance.web_2,\n    aws_instance.web_3,\n  ]\n}\n```\n\n### 4.4 Hook System for Extending Terraform\n\nHooks are extension points where external code can integrate with Terraform execution.\n\n**Hook Types and Invocation Points:**\n\n**1. Pre/Post Hooks:**\n\n```go\ntype Hook interface {\n    // Pre-operation hooks\n    PreRefresh(node *Node, state *InstanceState) (HookAction, error)\n    PreImportState(addr *ResourceAddress, id string) (HookAction, error)\n    PrePlan(node *Node) (HookAction, error)\n    PreApply(addr *ResourceAddress, state *InstanceState) (HookAction, error)\n    PreProvisionerType(name string) (HookAction, error)\n\n    // Post-operation hooks\n    PostRefresh(node *Node, state *InstanceState) (HookAction, error)\n    PostImportState(addr *ResourceAddress, instance *ResourceInstance) (HookAction, error)\n    PostPlan(node *Node, action ActionType, plannedState *InstanceState) (HookAction, error)\n    PostApply(addr *ResourceAddress, state *InstanceState, err error) (HookAction, error)\n    PostProvisionerType(name string) (HookAction, error)\n\n    // Output hooks\n    Output(msg OutputMessage) (HookAction, error)\n}\n```\n\n**2. Hook Actions:**\n\n- **Continue**: Proceed with normal execution\n- **Skip**: Skip this operation (for pre-hooks)\n- **Halt**: Stop all execution (error state)\n\n**Hook Invocation Sequence (Apply):**\n\n```\nFor each resource in order:\n    PreApply Hook\n        ↓\n    Provider.ApplyResourceChange\n        ↓\n    [if error]\n    Error Hook\n        ↓\n    PostApply Hook (with error)\n        ↓\n    [else]\n    PostApply Hook (success)\n        ↓\n    [Continue to next resource]\n```\n\n**Common Hook Implementations:**\n\n- **UI Hook**: Displays plan/apply progress to user\n- **Logging Hook**: Records all operations to logs\n- **Metrics Hook**: Emits metrics to monitoring system\n- **Audit Hook**: Records changes for compliance\n- **State Hook**: Validates state transitions\n- **Safety Hook**: Enforces policies (required tags, prohibited regions, etc.)\n\n### 4.5 Dynamic Expansion and Lazy Evaluation\n\n**Count/For_Each Resource Expansion:**\n\nResources using `count` or `for_each` undergo dynamic expansion during planning.\n\n**Count Expansion:**\n\n```hcl\nresource \"aws_instance\" \"web\" {\n  count = var.instance_count  # e.g., 3\n  ...\n}\n\n# Results in three graph nodes:\n# - aws_instance.web[0]\n# - aws_instance.web[1]\n# - aws_instance.web[2]\n```\n\n**Expansion Process:**\n\n1. **Expression Evaluation**: Count expression is evaluated\n   - `var.instance_count` → 3\n   - For_each key expression is evaluated\n   - Range is determined (0..count-1)\n\n2. **Node Multiplication**: For each iteration:\n   - Create graph node with index/key\n   - Evaluate resource configuration in iteration context\n   - References like `aws_instance.web[0].id` resolve correctly\n   - Create dependencies on referenced instances\n\n3. **Splat Expression Handling**:\n   ```hcl\n   resource \"aws_lb\" \"main\" {\n     targets = aws_instance.web[*].id\n     # Depends on all instances (aws_instance.web[0], web[1], web[2])\n   }\n   ```\n\n**Lazy Evaluation Benefits:**\n\n- Computing count from dynamic data doesn't require separate apply\n- Count changes are planned and applied in single operation\n- Splat references handle any count value automatically\n- Enables infrastructure scaling without code changes\n\n**Challenges:**\n\n- Count value must be deterministic before planning\n- Depends-on relationships must be explicit if dependent on count\n- Changing count value may cause recreation of resources\n\n### 4.6 Error Handling During Execution\n\n**Error Categories and Handling:**\n\n**1. Configuration Errors:**\n- Invalid syntax, missing required values\n- Detected during parsing and validation\n- Prevents plan from completing\n- User must fix configuration and retry\n\n**2. Validation Errors:**\n- Provider schema violations\n- Invalid attribute values\n- Detected during validate or plan phase\n- Prevent plan from completing\n\n**3. Planning Errors:**\n- Provider communication failures\n- Invalid references\n- Count/for_each evaluation errors\n- Stop planning, require retry or investigation\n\n**4. Operational Errors (Apply Phase):**\n- API failures (permissions, rate limits, service unavailable)\n- Resource conflicts\n- Network errors\n- May be transient or permanent\n\n**Error Propagation:**\n\n```\nProvider Error\n    ↓\nHook.PostApply(error=err) called\n    ↓\nEvaluate Lifecycle Rules:\n    |\n    ├─ on_failure = continue (default)\n    │  ↓ Resource marked failed, continue with next\n    │\n    └─ on_failure = fail_on_error\n       ↓ Stop execution, propagate error\n\n    ↓\nState Update:\n    ├─ Error state recorded (if supported)\n    ├─ Tainted resource marked for replacement (if applicable)\n    └─ State persisted\n\n    ↓\nUser-facing Error Message\n    ↓\nExit with error code\n```\n\n**Failure Modes:**\n\n- **Partial Apply**: Some resources created, others failed\n  - State reflects completed operations\n  - Failed operations can be retried\n  - May require manual cleanup\n\n- **Tainted Resources**: Resources marked for replacement on next apply\n  - Destroyed then recreated\n- **Locked Resources**: State lock held during failure\n  - Force-unlock available but dangerous\n  - Should resolve underlying error first\n\n### 4.7 State Consistency During Execution\n\n**Serialization Guarantees:**\n\n- Only one apply operation can modify state at a time (state lock)\n- Reading state during apply prevents concurrent plan operations\n- Snapshot mechanism enables planning while others apply\n\n**Crash Recovery:**\n\nIf Terraform crashes during apply:\n\n1. **Lock Remains**: State lock still held\n2. **Incomplete Changes**: Some resources may be created\n3. **Recovery**:\n   - Force-unlock if crash is confirmed stopped\n   - Run apply again to complete remaining resources\n   - Or manually fix state if needed\n\n**Orphaned Resources:**\n\nResources created before crash but not in state:\n\n- Can be imported via `terraform import`\n- Will be deleted on next apply if not configured (depending on mode)\n- Indicates need for state recovery procedure\n\n---\n\n## 5. Integration Points and Extension Mechanisms\n\n### 5.1 Configuration Validation and Analysis\n\nTerraform validates configuration at multiple points:\n\n**1. Syntax Validation:**\n- HCL parsing\n- Block structure validation\n- Type checking within blocks\n\n**2. Semantic Validation:**\n- Reference resolution\n- Provider availability\n- Attribute requirement checking\n\n**3. Provider Schema Validation:**\n- Attribute type matching\n- Required field validation\n- Constraint validation (min/max, patterns)\n\n**4. Custom Validation Blocks:**\n\n```hcl\nvariable \"instance_count\" {\n  type = number\n\n  validation {\n    condition     = var.instance_count > 0 && var.instance_count <= 100\n    error_message = \"Instance count must be 1-100.\"\n  }\n}\n\nresource \"aws_instance\" \"example\" {\n  # ...\n\n  lifecycle {\n    postcondition {\n      condition     = self.tags[\"Environment\"] == \"prod\"\n      error_message = \"Production instances must be tagged.\"\n    }\n  }\n}\n```\n\n### 5.2 Lifecycle Rules and Customization\n\n**Lifecycle Meta-arguments:**\n\n```hcl\nresource \"aws_instance\" \"web\" {\n  # ...\n\n  lifecycle {\n    # Prevent resource deletion\n    prevent_destroy = true\n\n    # Create new before destroying old (rolling updates)\n    create_before_destroy = true\n\n    # Don't update specified attributes\n    ignore_changes = [tags]\n\n    # Custom replacement triggers\n    replace_triggered_by = [aws_security_group.web.id]\n\n    # Provisioner execution policy\n    on_failure = continue  # or fail_on_error\n  }\n\n  # Provisioners run after creation/destruction\n  provisioner \"local-exec\" {\n    command = \"echo ${self.id} >> instances.txt\"\n  }\n}\n```\n\n**Lifecycle Rule Execution:**\n\n- **prevent_destroy**: Blocks destroy operations (errors before execution)\n- **create_before_destroy**: During replacement, creates new before destroying old\n  - Enables zero-downtime updates\n  - Requires temporary resource doubling\n- **ignore_changes**: Skips diff detection for specified attributes\n  - Useful for attributes modified outside Terraform\n  - Enables management by other tools\n- **replace_triggered_by**: Replaces resource when dependent resource changes\n  - Useful for configuration-dependent resources\n\n### 5.3 Provisioners and External Execution\n\n**Provisioner Types:**\n\n1. **local-exec**: Execute command on machine running Terraform\n   ```hcl\n   provisioner \"local-exec\" {\n     command = \"bash install.sh\"\n   }\n   ```\n\n2. **remote-exec**: Execute commands on remote resource\n   ```hcl\n   provisioner \"remote-exec\" {\n     inline = [\n       \"apt-get update\",\n       \"apt-get install -y nginx\"\n     ]\n   }\n   ```\n\n3. **file**: Copy files to remote resource\n   ```hcl\n   provisioner \"file\" {\n     source      = \"local/path\"\n     destination = \"/tmp/remote\"\n   }\n   ```\n\n**Provisioner Execution:**\n\n- Run after resource is created or destroyed (configurable)\n- Failures trigger `on_failure` lifecycle rule (continue or fail_on_error)\n- Marked as \"tainted\" if provisioner fails (will be replaced next apply)\n- Available: `self.*` (resource attributes), `var.*` (variables), `local.*` (locals)\n\n**Warning - Anti-pattern:**\n\nModern best practices discourage provisioners in favor of:\n- Cloud-init user data for EC2 instances\n- Configuration management tools (Ansible, Puppet, Chef)\n- Custom provider resources\n- Containerization (Docker, Kubernetes)\n\n### 5.4 Terraform Cloud/Enterprise Integration\n\n**Remote Operations:**\n\nTerraform Cloud provides hosted plan/apply with additional features:\n\n1. **Remote Execution**: Plan/apply runs on Terraform Cloud infrastructure\n   - Centralized audit trail\n   - No local state management\n   - Consistent execution environment\n\n2. **Run Triggers**: Automatic planning on:\n   - VCS repository commits\n   - Configuration changes\n   - Variable updates\n   - Upstream workspace changes\n\n3. **Policy Enforcement**: Sentinel policies enforce governance\n   ```\n   policy \"require_tags\" {\n     rule { require_tags(\"Environment\") }\n   }\n   ```\n\n4. **Cost Estimation**: Estimate cost of changes before apply\n\n5. **Team Management**: Role-based access control to workspaces\n\n### 5.5 Custom Provider Development\n\nProvider developers implement gRPC server that Terraform core calls.\n\n**Schema Definition:**\n\n```go\ntype ProviderSchema struct {\n  Resources: map[string]schema.Resource,\n  DataSources: map[string]schema.DataSource,\n  Functions: map[string]function.Function,\n}\n\ntype ResourceSchema struct {\n  Attributes: map[string]schema.Attribute,\n  Blocks: map[string]schema.Block,\n  Validators: []...Validator,\n  Computed: []string,\n  Required: []string,\n}\n```\n\n**Resource Implementation:**\n\nProviders must implement:\n- `ValidateResourceConfig`: Validate configuration\n- `ReadResourceState`: Read current resource state\n- `PlanResourceChange`: Plan changes\n- `ApplyResourceChange`: Apply changes\n- `ImportResourceState`: Import from existing resource\n- `ReadModifyWrite`: Transaction-like pattern for complex changes\n\n---\n\n## 6. Terraform Modules and Composition\n\n### 6.1 Module Semantics\n\nModules are reusable abstractions over configuration:\n\n```hcl\nmodule \"vpc\" {\n  source = \"./modules/vpc\"\n\n  cidr_block        = \"10.0.0.0/16\"\n  availability_zones = var.azs\n\n  tags = local.common_tags\n}\n\noutput \"vpc_id\" {\n  value = module.vpc.vpc_id\n}\n```\n\n**Module Execution Model:**\n\n1. **Module Discovery**: Source is resolved (local path, registry, VCS, HTTP)\n2. **Module Configuration**: Input variables are evaluated\n3. **Subgraph Creation**: Module's resources are added to execution graph\n4. **Output Evaluation**: Module outputs are computed after module resources complete\n5. **State Tracking**: Module state is recorded within resource state\n\n### 6.2 Module Graph Integration\n\nModules appear in the execution graph as special nodes:\n\n```\nModuleCallNode\n  ├─ Input Variables (dependencies)\n  ├─ Module Resources (subgraph)\n  └─ Output Values (computed from subgraph)\n\nGraph Edges:\n  - Module depends on all variable references\n  - Resources in module depend on module inputs\n  - Module outputs depend on module resources\n  - External resources depending on module.output depend on ModuleCallNode\n```\n\n---\n\n## 7. Advanced Execution Topics\n\n### 7.1 Destroy Operations\n\nDestroy operations reverse the dependency graph:\n\n**Destroy Graph:**\n\n```\nOriginal Graph:\n  B depends on A\n  A → B\n\nDestroy Graph:\n  A depends on B (reversed)\n  B → A (reversed)\n\nResult: B destroyed first, A destroyed second\n```\n\n**Destroy Plan:**\n\n- Identified by `-destroy` flag or `destroy` operation on specific resources\n- All non-deleted resources are planned for deletion\n- Providers are called with `null` for new state\n\n### 7.2 Targeted Operations\n\nTargeting enables surgical infrastructure changes:\n\n```bash\nterraform plan -target=aws_instance.web[0]\n```\n\n**Targeting Algorithm:**\n\n1. Identifies specified resource(s)\n2. Includes all transitive dependents (must update if target changes)\n3. Removes all other resources from graph\n4. Provider Interface and Lifecycle\n\nProviders are plugins that implement resource operations. Reduces startup time and resource consumption\n  - Enables disconnected operations (plan without provider access)\n\n### 2.3 Resource Operation Delegation\n\n**Operation Types:**\n\nTerraform delegates four primary operations to providers:\n\n1. **ValidateResourceConfig**: Validates resource configuration\n   - Verify Plan Still Valid:\n   if current_state != plan.prior_state:\n       error \"state changed since plan\"\n\n2. Apply Change:\n   if change == Create:\n       new_state = provider.ApplyResourceChange(null, planned_state, config)\n   else if change == Update:\n       new_state = provider.ApplyResourceChange(prior_state, planned_state, config)\n   else if change == Delete:\n       provider.ApplyResourceChange(prior_state, null, config)\n       remove from state\n   else if change == NoOp:\n       skip provider call, state unchanged\n\n3. Update State:\n   state[resource] = new_state\n   persist state\n\n4. Handle Errors:\n   if provider.error:\n       check lifecycle.on_failure\n       potentially delete resource\n       report error\n```\n\n### 4.3 Walker Pattern for Graph Traversal\n\nThe **Walker** is the core execution engine that traverses the dependency graph.\n\n**Walker Responsibilities:**\n\n1. **Dependency Tracking**: Maintains which nodes are ready to execute\n2. **Concurrency Control**: Limits parallel execution (default: 10 concurrent)\n3. **Ordering**: Executes nodes respecting dependency constraints\n4. **Hook Invocation**: Calls hooks at appropriate execution points\n5. **Error Handling**: Handles and reports errors from node evaluation\n\n**Walker Algorithm:**\n\n```\n1. Initialize:\n   pending_nodes = all graph nodes\n   ready_nodes = nodes with no dependencies\n   running_nodes = empty set\n   completed_nodes = empty set\n\n2. Loop while pending_nodes not empty or running_nodes not empty:\n\n   a. Fill available slots (up to concurrency limit):\n      while len(running_nodes) < parallelism:\n          if ready_nodes not empty:\n              node = ready_nodes.pop()\n              start_async_evaluation(node)\n              running_nodes.add(node)\n          else:\n              break\n\n   b. Wait for any node to complete:\n      completed_node = wait_for_any(running_nodes)\n\n   c. Process completion:\n      if completed_node.error:\n          mark as failed\n          invoke error hooks\n          propagate error to dependents\n      else:\n          mark as completed\n          invoke success hooks\n          calculate newly ready nodes\n\n      remove from running_nodes\n      add dependents to ready_nodes\n\n3. Report results:\n   if any failed:\n       report failures\n   else:\n       report success\n```\n\n**Parallelization:**\n\n- Default concurrency: 10 nodes\n- Configurable via `-parallelism=N`\n- Only nodes with no blocking dependencies run concurrently\n- Heavy I/O operations benefit from parallelism\n- Resource creation typically parallelizable if no explicit dependencies\n\n**Example Parallelization:**\n\n```hcl\n# These three resources can run in parallel (no dependencies)\nresource \"aws_instance\" \"web_1\" { ... }\nresource \"aws_instance\" \"web_2\" { ... }\nresource \"aws_instance\" \"web_3\" { ... }\n\n# This depends on all three above, must wait\nresource \"aws_lb\" \"main\" {\n  depends_on = [\n    aws_instance.web_1,\n    aws_instance.web_2,\n    aws_instance.web_3,\n  ]\n}\n```\n\n### 4.4 Hook System for Extending Terraform\n\nHooks are extension points where external code can integrate with Terraform execution.\n\n**Hook Types and Invocation Points:**\n\n**1. Pre/Post Hooks:**\n\n```go\ntype Hook interface {\n    // Pre-operation hooks\n    PreRefresh(node *Node, state *InstanceState) (HookAction, error)\n    PreImportState(addr *ResourceAddress, id string) (HookAction, error)\n    PrePlan(node *Node) (HookAction, error)\n    PreApply(addr *ResourceAddress, state *InstanceState) (HookAction, error)\n    PreProvisionerType(name string) (HookAction, error)\n\n    // Post-operation hooks\n    PostRefresh(node *Node, state *InstanceState) (HookAction, error)\n    PostImportState(addr *ResourceAddress, instance *ResourceInstance) (HookAction, error)\n    PostPlan(node *Node, action ActionType, plannedState *InstanceState) (HookAction, error)\n    PostApply(addr *ResourceAddress, state *InstanceState, err error) (HookAction, error)\n    PostProvisionerType(name string) (HookAction, error)\n\n    // Output hooks\n    Output(msg OutputMessage) (HookAction, error)\n}\n```\n\n**2. Hook Actions:**\n\n- **Continue**: Proceed with normal execution\n- **Skip**: Skip this operation (for pre-hooks)\n- **Halt**: Stop all execution (error state)\n\n**Hook Invocation Sequence (Apply):**\n\n```\nFor each resource in order:\n    PreApply Hook\n        ↓\n    Provider.ApplyResourceChange\n        ↓\n    [if error]\n    Error Hook\n        ↓\n    PostApply Hook (with error)\n        ↓\n    [else]\n    PostApply Hook (success)\n        ↓\n    [Continue to next resource]\n```\n\n**Common Hook Implementations:**\n\n- **UI Hook**: Displays plan/apply progress to user\n- **Logging Hook**: Records all operations to logs\n- **Metrics Hook**: Emits metrics to monitoring system\n- **Audit Hook**: Records changes for compliance\n- **State Hook**: Validates state transitions\n- **Safety Hook**: Enforces policies (required tags, prohibited regions, etc.)\n\n### 4.5 Dynamic Expansion and Lazy Evaluation\n\n**Count/For_Each Resource Expansion:**\n\nResources using `count` or `for_each` undergo dynamic expansion during planning.\n\n**Count Expansion:**\n\n```hcl\nresource \"aws_instance\" \"web\" {\n  count = var.instance_count  # e.g., 3\n  ...\n}\n\n# Results in three graph nodes:\n# -
