Defaults to `DiagnosticSeverity.Error`\n\n**Properties:**\n\n| Property | Type | Accessibility | Description |\n|----------|------|----------------|-------------|\n| **range** | `Range` | read/write | The range in the document to which this diagnostic applies. The diagnostic will be displayed starting at the beginning of this range. |\n| **message** | `string` | read/write | The human-readable message. Should be concise and describe the issue clearly. |\n| **severity** | `DiagnosticSeverity` | read/write | The severity level. Defaults to `DiagnosticSeverity.Error`. |\n| **source** | `string \\| undefined` | read/write | A human-readable string describing the source of this diagnostic (e.g., `'typescript'`, `'eslint'`, `'super lint'`). Displayed in the Problems panel to identify the diagnostic source. |\n| **code** | `string \\| number \\| {value, target} \\| undefined` | read/write | A code or identifier for this diagnostic. Used to associate diagnostics with code actions. Can be a simple value or an object with a `value` and a `target` URI for more information about the error. |\n| **relatedInformation** | `DiagnosticRelatedInformation[] \\| undefined` | read/write | An array of related diagnostic information. Used to point to code locations that cause or are related to this diagnostic (e.g., when duplicating a symbol in a scope). |\n| **tags** | `DiagnosticTag[] \\| undefined` | read/write | Additional metadata tags that provide hints about the diagnostic type and affect rendering. |\n\n**Behavior notes:**\n- Diagnostics are scoped to a specific file (represented by a URI in the containing DiagnosticCollection)\n- A range must be valid within the document (zero or positive positions)\n- The `code` property should be used for later processing, such as when providing code actions\n- Related information is useful for showing the root cause or related locations of an issue\n- Multiple code actions can be triggered for the same diagnostic code\n\n**Example:**\n```typescript\nconst range = new vscode.Range(new vscode.Position(5, 0), new vscode.Position(5, 10));\nconst diagnostic = new vscode.Diagnostic(\n\trange,\n\t'Unused variable',\n\tvscode.DiagnosticSeverity.Warning\n);\ndiagnostic.source = 'eslint';\ndiagnostic.code = 'no-unused-vars';\ndiagnostic.tags = [vscode.DiagnosticTag.Unnecessary];\n```\n\n### DiagnosticRelatedInformation\n\n**Class**: Represents additional context information related to a diagnostic.\n\n**Constructor:**\n```typescript\nconstructor(location: Location, message: string)\n```\n\n**Parameters:**\n- `location` (required): The location of the related issue (a `Location` object containing a URI and a Range)\n- `message` (required): The message describing the relationship to the main diagnostic\n\n**Properties:**\n\n| Property | Type | Accessibility | Description |\n|----------|------|----------------|-------------|\n| **location** | `Location` | read/write | The location where the related information points to. Contains a URI (file path) and a Range (position in the file). |\n| **message** | `string` | read/write | The message explaining the relationship. For example, \"Symbol is also defined here\" or \"First occurrence\". |\n\n**Behavior notes:**\n- Used to point to other locations in the codebase that are related to the main diagnostic\n- Particularly useful for showing symbol conflicts, related errors, or the original definition\n- The location can point to a different file from the main diagnostic\n- Related information is displayed in the Problems panel when hovering over the diagnostic\n\n**Example:**\n```typescript\nconst mainDiagnostic = new vscode.Diagnostic(\n\tnew vscode.Range(0, 0, 0, 5),\n\t'Duplicate symbol declaration',\n\tvscode.DiagnosticSeverity.Error\n);\n\nmainDiagnostic.relatedInformation = [\n\tnew vscode.DiagnosticRelatedInformation(\n\t\tnew vscode.Location(\n\t\t\tvscode.Uri.file('/path/to/other.ts'),\n\t\t\tnew vscode.Range(10, 0, 10, 5)\n\t\t),\n\t\t'Symbol is also defined here'\n\t)\n];\n```\n\n### DiagnosticChangeEvent\n\n**Interface**: Event data fired when diagnostics change globally.\n\n**Properties:**\n\n| Property | Type | Description |\n|----------|------|-------------|\n| **uris** | `readonly Uri[]` | An array of URIs (file paths) for which diagnostics have changed. This represents all files affected by the diagnostic change. |\n\n**Behavior notes:**\n- This event provides the list of resources that had diagnostics added, removed, or modified\n- The event does not specify which diagnostics were added or removed, only which resources were affected\n- Extensions should query the specific diagnostics using `getDiagnostics(uri)` if they need details\n\n### DiagnosticCollection\n\n**Interface**: A container that manages a set of diagnostics for multiple files.\n\n**Properties:**\n\n| Property | Type | Accessibility | Description |\n|----------|------|----------------|-------------|\n| **name** | `string` | read-only | The name of the collection (e.g., `'typescript'`, `'eslint'`). Used to identify the diagnostic source in the Problems panel and for task problem matchers. Every diagnostic from this collection is associated with this name. |\n\n**Methods:**\n\n#### `set(uri: Uri, diagnostics: readonly Diagnostic[] | undefined): void`\n\nAssign diagnostics for a given resource, replacing any existing diagnostics for that file.\n\n**Parameters:**\n- `uri` (required): The file URI to associate with the diagnostics\n- `diagnostics` (required): Array of diagnostics or `undefined` to clear all diagnostics for this URI\n\n**Behavior:**\n- If `diagnostics` is `undefined`, all previously set diagnostics for this URI are removed\n- If `diagnostics` is an empty array, the same clearing behavior applies\n- Setting diagnostics for the same URI multiple times replaces previous diagnostics\n- All diagnostics in the array must have ranges that fit within the document\n\n**Example:**\n```typescript\nconst uri = vscode.Uri.file('/path/to/file.ts');\nconst diagnostics = [\n\tnew vscode.Diagnostic(new vscode.Range(0, 0, 0, 5), 'Error message'),\n\tnew vscode.Diagnostic(new vscode.Range(1, 10, 1, 15), 'Warning message')\n];\ncollection.set(uri, diagnostics);\n\n// Clear diagnostics\ncollection.set(uri, undefined);\n```\n\n#### `set(entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>): void`\n\nReplace diagnostics for multiple resources in this collection.\n\n**Parameters:**\n- `entries` (required): An array of tuples `[Uri, Diagnostics]` representing file URIs and their associated diagnostics\n\n**Behavior:**\n- Allows batch updating of diagnostics for multiple files in a single operation\n- Multiple tuples with the same URI are merged (not overwritten)\n- `undefined` as the diagnostics for a URI clears all previous diagnostics for that file but doesn't prevent subsequent entries with the same URI\n- If a diagnostics item is `undefined` as in `[file1, undefined]`, all previous diagnostics are removed\n- This method is more efficient than calling `set()` multiple times\n\n**Example:**\n```typescript\ncollection.set([\n\t[uri1, [diagnostic1, diagnostic2]],\n\t[uri2, [diagnostic3]],\n\t[uri3, undefined]  // Clear all diagnostics for uri3\n]);\n```\n\n#### `delete(uri: Uri): void`\n\nRemove all diagnostics from this collection for the provided URI.\n\n**Parameters:**\n- `uri` (required): The file URI to clear diagnostics from\n\n**Behavior:**\n- Equivalent to calling `set(uri, undefined)`\n- Safe to call even if the URI has no diagnostics\n- Does not affect diagnostics in other collections\n\n**Example:**\n```typescript\ncollection.delete(uri);\n```\n\n#### `clear(): void`\n\nRemove all diagnostics from this collection.\n\n**Behavior:**\n- Clears diagnostics for all files in the collection\n- Equivalent to calling `set(undefined)` (though `set()` with no arguments is not available)\n- After clearing, the collection remains valid for setting new diagnostics\n\n**Example:**\n```typescript\ncollection.clear();\n```\n\n#### `forEach(callback: (uri: Uri, diagnostics: readonly Diagnostic[], collection: DiagnosticCollection) => any, thisArg?: any): void`\n\nIterate over each entry in the collection.\n\n**Parameters:**\n- `callback` (required): Function called for each URI with diagnostics\n  - `uri`: The file URI\n  - `diagnostics`: The array of diagnostics for this URI (read-only)\n  - `collection`: Reference to the DiagnosticCollection itself\n- `thisArg` (optional): The `this` context to use when invoking the callback\n\n**Behavior:**\n- Iterates in insertion order (though order is not guaranteed across extensions)\n- The diagnostics array is immutable (read-only)\n- Modifying the collection during iteration may have unexpected results\n- Callback may return any value (return value is discarded)\n\n**Example:**\n```typescript\ncollection.forEach((uri, diagnostics, collection) => {\n\tconsole.log(`${uri.fsPath}: ${diagnostics.length} diagnostics`);\n});\n```\n\n#### `get(uri: Uri): readonly Diagnostic[] | undefined`\n\nGet the diagnostics for a given resource.\n\n**Parameters:**\n- `uri` (required): The file URI to query\n\n**Returns:**\n- An immutable array of diagnostics for the URI, or `undefined` if no diagnostics are set\n\n**Behavior:**\n- Returns a read-only array that cannot be modified\n- Trying to modify the returned array will result in a runtime error\n- Returns `undefined` (not an empty array) if no diagnostics have been set\n- The returned array includes all diagnostics set via `set()` for this URI\n\n**Example:**\n```typescript\nconst diagnostics = collection.get(uri);\nif (diagnostics) {\n\tdiagnostics.forEach(diag => console.log(diag.message));\n}\n```\n\n#### `has(uri: Uri): boolean`\n\nCheck if this collection contains diagnostics for a given resource.\n\n**Parameters:**\n- `uri` (required): The file URI to check\n\n**Returns:**\n- `true` if the collection has diagnostics for the URI, `false` otherwise\n\n**Behavior:**\n- Efficient way to check for presence without retrieving the diagnostics\n- Returns `false` for URIs that have been cleared or never set\n\n**Example:**\n```typescript\nif (collection.has(uri)) {\n\tconst diagnostics = collection.get(uri)!;\n\t// Process diagnostics\n}\n```\n\n#### `dispose(): void`\n\nDispose and free associated resources.\n\n**Behavior:**\n- Automatically calls `clear()` before disposing\n- After disposal, the collection cannot be used\n- Attempting to use a disposed collection will throw an error\n- Should be called when the extension is deactivated or no longer needs the collection\n- Collections created with `createDiagnosticCollection()` should be disposed in the extension's `deactivate()` function\n\n**Example:**\n```typescript\nextension.subscriptions.push(collection);  // Auto-dispose when extension deactivates\n// OR\ncontext.subscriptions.push(collection);    // VS Code will dispose when needed\n// OR explicit disposal\ncollection.dispose();\n```\n\n**Lifecycle notes:**\n- Diagnostic collections are tied to the extension lifecycle\n- Creating many diagnostic collections without disposing them can cause memory leaks\n- VS Code automatically disposes collections when the extension is deactivated\n- Extensions should store references to collections they create and dispose them properly\n\n---\n\n## Functions\n\n### createDiagnosticCollection\n\n**Signature:**\n```typescript\nexport function createDiagnosticCollection(name?: string): DiagnosticCollection\n```\n\n**Parameters:**\n- `name` (optional): A unique name for the collection (e.g., `'typescript'`, `'eslint'`). If not provided, a random UUID is generated. The name is displayed in the Problems panel to identify the diagnostic source.\n\n**Returns:**\n- A new `DiagnosticCollection` instance\n\n**Behavior:**\n- Creates an empty diagnostic collection that can hold diagnostics for multiple files\n- If `name` is provided, it appears in the Problems panel to identify this source of diagnostics\n- Multiple extensions can create collections with the same name (they will be merged in the UI)\n- Each call creates a new, independent collection\n- The collection is automatically disposed when the extension deactivates if stored in 1):**\n```typescript\nexport function getDiagnostics(resource: Uri): Diagnostic[]\n```\n\n**Parameters:**\n- `resource` (required): A URI representing a specific file\n\n**Returns:**\n- An array of all diagnostics for the given file across all diagnostic collections, or an empty array if none exist\n\n**Behavior:**\n- Retrieves diagnostics from all diagnostic collections (not just one)\n- Returns an empty array (not `undefined`) if no diagnostics are found\n- The returned array includes diagnostics from all sources in the workspace\n- The array is mutable (can be modified without affecting the stored diagnostics)\n\n**Example:**\n```typescript\nconst uri = vscode.Uri.file('/path/to/file.ts');\nconst allDiagnostics = vscode.languages.getDiagnostics(uri);\nconsole.log(`${allDiagnostics.length} total diagnostics in file`);\n```\n\n---\n\n**Signature (overload 2):**\n```typescript\nexport function getDiagnostics(): [Uri, Diagnostic[]][]\n```\n\n**Parameters:**\n- None\n\n**Returns:**\n- An array of tuples, where each tuple contains a URI and an array of diagnostics for that file\n\n**Behavior:**\n- Retrieves all diagnostics in the workspace across all collections and files\n- Returns an empty array if no diagnostics exist anywhere in the workspace\n- Each tuple represents one file and all its diagnostics\n- Each file appears only once, even if diagnostics are from different collections\n- The returned array is mutable\n\n**Behavior notes:**\n- This is an expensive operation if many diagnostics exist across the workspace\n- Should be used sparingly, typically only for global diagnostic queries\n- In most cases, you should query specific files using the single-parameter overload\n\n**Example:**\n```typescript\nconst allDiagnostics = vscode.languages.getDiagnostics();\nconsole.log(`Total files with diagnostics: ${allDiagnostics.length}`);\n\nfor (const [uri, diagnostics] of allDiagnostics) {\n\tconsole.log(`${uri.fsPath}: ${diagnostics.length} issues`);\n}\n```\n\n---\n\n## Events\n\n### onDidChangeDiagnostics\n\n**Signature:**\n```typescript\nexport const onDidChangeDiagnostics: Event<DiagnosticChangeEvent>\n```\n\n**Event data:**\n- `DiagnosticChangeEvent` with a property `uris: readonly Uri[]`\n\n**Behavior:**\n- Fires when diagnostics change in any collection\n- Provides a list of all URIs that had diagnostics added, removed, or modified\n- Debounced internally (changes are batched and fired after a short delay, typically 50ms)\n- Does not provide details about what changed, only which files were affected\n\n**When the event fires:**\n- When diagnostics are set via `set()`\n- When diagnostics are cleared via `delete()` or `clear()`\n- When a collection is disposed (clearing its diagnostics)\n- NOT fired when you subscribe to the event (only for subsequent changes)\n\n**Usage patterns:**\n- Query `getDiagnostics(uri)` in the event handler to get details\n- Use to update UI, re-validate documents, or sync with external systems\n- Remember that changes are debounced, so a single set operation may fire only once\n\n**Error cases:**\n- No errors are thrown; the event is safe to subscribe to\n\n**Example:**\n```typescript\nconst subscription = vscode.languages.onDidChangeDiagnostics(event => {\n\tconsole.log(`Diagnostics changed for ${event.uris.length} files:`);\n\n\tfor (const uri of event.uris) {\n\t\tconst diagnostics = vscode.languages.getDiagnostics(uri);\n\t\tconsole.log(`  ${uri.fsPath}: ${diagnostics.length} total issues`);\n\t}\n});\n\ncontext.subscriptions.push(subscription);\n```\n\n---\n\n## Usage Examples\n\n### Basic Example: Creating and Publishing
