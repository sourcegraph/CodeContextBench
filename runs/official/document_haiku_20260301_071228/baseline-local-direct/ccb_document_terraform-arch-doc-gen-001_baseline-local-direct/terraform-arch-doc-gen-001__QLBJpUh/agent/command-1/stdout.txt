Create placeholder nodes for dynamic resources\n- Identify resources that depend on expanded resources\n\n**Pass 2: Expansion**\n- Evaluate count/for_each expressions using available construction, allowing:\n- Access to data sources for count expressions\n- Dependency on resource outputs for for_each\n- Correct ordering of resource creation\n\n---\n\n## 2. Provider Interface and Lifecycle\n\n### 2.1 Provider Plugin Architecture\n\nTerraform uses a **plugin-based provider architecture** to decouple core from resource implementations. Providers run as separate processes communicating via gRPC.\n\n```\n┌─────────────────────┐\n│  Terraform Core     │\n│  (Main Process)     │\n└──────────┬──────────┘\n           │ gRPC\n           │ (proto3)\n┌──────────▼──────────┐\n│  Provider Process   │\n│  (Plugin Process)   │\n│  - aws              │\n│  - azurerm          │\n│  - kubernetes       │\n└─────────────────────┘\n```\n\n**Communication Protocol**:\n- Uses gRPC with Protocol Buffers for message serialization\n- Each provider instance is a separate process\n- Core spawns provider processes on demand\n- IPC handles marshaling of complex types (objects, lists, maps)\n\n### 2.2 Provider Initialization and Lifecycle\n\nProvider lifecycle follows a distinct sequence:\n\n#### Phase 1: Plugin Discovery\n1. Scan plugin directories for executable matching provider naming convention\n2. Hash plugin binary and verify it matches lock file (if using provider lock)\n3. Execute plugin with version flag to confirm compatibility\n\n#### Phase 2: Configuration\n1. Collect provider configuration from `terraform` and `provider` blocks\n2. Merge configurations with provider aliases\n3. Validate configuration attributes against provider schema\n4. Prepare configuration values (evaluate expressions)\n\n#### Phase 3: Instantiation\n1. Spawn provider process with plugin path\n2. Call `Configure` RPC with configuration values\n3. Provider initializes internal state and connections\n4. Provider reports back with schema for all resource types\n\n#### Phase 4: Graph Execution\n1. Core calls provider methods during resource operation\n2. Provider maintains stateful connection (database, API client, etc.)\n3. Multiple resource operations may use single provider instance\n\n#### Phase 5: Cleanup\n1. Explicit provider shutdown (usually at program end)\n2. Provider process terminates\n3. Cleanup any open connections or temporary resources\n\n### 2.3 Resource Operation Delegation\n\nResource operations are delegated to providers through well-defined RPC interfaces:\n\n#### Resource Read Operation\n```\nResource Node (Plan)\n    ↓\nCall Provider.ReadResource()\n    ├─ Current: State from terraform.tfstate\n    ├─ Config: Resource configuration\n    └─ Return: Refreshed state\n    ↓\nUpdate resource state in-memory\n```\n\n**Used for**:\n- Refreshing state during plan phase\n- Detecting external modifications (drift detection)\n- Reading computed attributes after apply\n\n#### Resource Plan Diff Operation\n```\nResource Node (Plan)\n    ↓\nCall Provider.PlanResourceChange()\n    ├─ Prior: Current state\n    ├─ Proposed: Configuration as would be applied\n    └─ Return: Planned changes (add/update/delete)\n    ↓\nStore planned changes in plan file\n```\n\n**Determines**:\n- Which attributes will change\n- Whether the change requires resource replacement\n- Which attributes are computed (calculated by provider)\n\n#### Resource Apply Operation\n```\nResource Node (Apply)\n    ↓\nCall Provider.ApplyResourceChange()\n    ├─ Planned: Changes from plan phase\n    ├─ Config: Current configuration\n    └─ Return: Final state after apply\n    ↓\nUpdate state file with new state\n```\n\n**Execution semantics**:\n- Called sequentially for each resource (respecting graph ordering)\n- May create new resource or update/destroy existing\n- Returns final state that's persisted to state file\n\n#### Data Source Read Operation\n```\nData Source Node (Plan/Apply)\n    ↓\nCall Provider.ReadDataSource()\n    ├─ Config: Data source configuration\n    └─ Return: Data source state\n    ↓\nCache result in context\n```\n\n**Characteristics**:\n- Executes in both plan and apply phases\n- Results available for interpolation\n- Never modifies external state\n\n### 2.4 Provider Instance Management\n\nProviders are managed with attention to state, performance, and concurrency:\n\n#### Caching Strategy\n- Single provider instance per unique configuration\n- Configuration identity determined by merged attribute values\n- Instance reused across all resources using that provider\n- Reduces provider process creation overhead\n\n#### Concurrent Access\n- Provider instances are thread-safe\n- Core can call provider concurrently for independent resources\n- Graph determines safe parallelization points\n- Some providers limit concurrent operations (configured via provider config)\n\n#### Instance Scoping\n- Provider lifecycle scoped to terraform command execution\n- Separate instance for each `terraform` execution\n- Separate instances for different workspaces\n- Allow state-specific provider configurations\n\n---\n\n## 3. State Management\n\n### 3.1 State File Structure and Semantics\n\nThe Terraform state file serves as the single source of truth for infrastructure state:\n\n```hcl\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.0.0\",\n  \"serial\": 42,\n  \"lineage\": \"uuid\",\n  \"remote\": {...},  // Remote backend config\n  \"outputs\": {...}, // Output values\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"web\",\n      \"instances\": [\n        {\n          \"schema_version\": 0,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"private_ip\": \"10.0.0.1\",\n            ...\n          },\n          \"sensitive_attributes\": {},\n          \"private\": \"...\",  // Encrypted private data\n          \"dependencies\": [\n            \"aws_security_group.main\"\n          ]\n        }\n      ]\n    }\n  ]\n}\n```\n\n**Key components:**\n- **Serial**: Monotonically increasing counter tracking state version\n- **Lineage**: UUID tracking state identity across resets\n- **Resources**: Array of resource instances and their attributes\n- **Outputs**: Calculated values exported from the root module\n- **Dependencies**: Explicit dependency references (for drift detection)\n\n### 3.2 State Reading and Initialization\n\nState initialization occurs at the start of plan/apply:\n\n```\nState Backend → Acquire Lock → Verify Lineage → Parse JSON\n                    ↓\n            Create State Snapshot\n                    ↓\n            Validate Instances Match\n            Current Configuration\n```\n\n**Lock Acquisition**:\n- Remote backends (S3, Terraform Cloud) acquire distributed lock\n- Lock timeout prevents indefinite blocking\n- Lock metadata includes operator identity and timestamp\n- Force unlock available for emergency recovery\n\n**Lineage Verification**:\n- Confirms state matches expected lineage UUID\n- Prevents mixing unrelated states (e.g., different projects)\n- Detectable misconfiguration during init\n\n**State Snapshot**:\n- Create immutable copy of state at read time\n- Used for drift detection (comparing snapshot to current)\n- Enables concurrent graph evaluation with consistent view\n\n### 3.3 State Modification During Execution\n\nState is modified in-memory during plan/apply with write-back at completion:\n\n#### State Delta Tracking\n```\nInitial State → Node Evaluation → State Changes Recorded → Final State\n                     ↓\n                Resource attributes\n                added/updated/deleted\n```\n\nThe state manager tracks changes:\n- **Addition**: Resource created with new attributes\n- **Modification**: Attributes changed (deep merge with existing)\n- **Deletion**: Resource removed entirely\n- **Replacement**: Resource destroyed and recreated (terraform tracks this)\n\n#### Merge Strategy\n```\nBase State (from backend)\n    ↓\nResource Node Creates/Updates Instance\n    ↓\nChanges Merged with Base\n    ↓\nUpdated State Ready for Next Operation\n```\n\nWhen a resource applies:\n1. Resource instance in state is located\n2. Operation result (new attributes) merged into instance\n3. Metadata updated (ID, timestamps, etc.)\n4. State object marked dirty for persistence\n\n#### Computed Attribute Handling\n```\nConfiguration: { name = \"instance\" }\nProvider Returns: { id = \"i-123\", created_at = \"2024-01-01\" }\n                       ↓\nFinal State: { name = \"instance\", id = \"i-123\", created_at = \"2024-01-01\" }\n```\n\nComputed attributes come from provider responses:\n- Not in configuration, only in state\n- Updated during each apply operation\n- Enable dependent resources to reference computed values\n\n### 3.4 State Backend and Persistence\n\nState is abstracted through a backend interface enabling multiple storage strategies:\n\n#### Local Backend\n- File-based storage on local filesystem\n- Default for local development\n- Locking via filesystem operations (or DB backend)\n- No remote access capability\n\n#### Remote Backends\n**Terraform Cloud/Enterprise**:\n- Network-accessed state management\n- Built-in locking and versioning\n- Cost billing based on runs\n- Team management and policy enforcement\n\n**S3 Backend**:\n- Stores state in S3 object\n- Locking via DynamoDB\n- Versioning via S3 object versioning\n- Cost-effective for small teams\n\n**Other Backends** (Azure, GCS, etc.):\n- Provider-specific implementations\n- Similar locking and consistency guarantees\n- Varying feature sets\n\n#### Backend Interface\n```\nBackend Interface:\n  - State()              → Load current state\n  - WriteState(state)    → Persist modified state\n  - Delete()             → Remove state\n  - Lock(info)           → Acquire distributed lock\n  - Unlock(id)           → Release distributed lock\n```\n\n### 3.5 State Locking and Concurrency Control\n\nState locking prevents concurrent modifications:\n\n#### Locking Flow\n```\nterraform plan/apply\n    ↓\nAcquire Lock\n    ├─ Lock ID: UUID\n    ├─ Info: operator, timestamp, workspace\n    └─ Timeout: typically 10 minutes\n    ↓\nRead State\n    ↓\nExecute Graph\n    ↓\nWrite State\n    ↓\nRelease Lock\n```\n\n#### Lock Implementation Strategies\n\n**Filesystem Locks** (local):\n- Exclusive file lock on state file\n- Relies on OS file locking primitives\n- Not suitable for remote access\n\n**DynamoDB Locks** (S3 backend):\n- DynamoDB item represents lock\n- Conditional writes prevent race conditions\n- Automatic expiration via TTL\n- Requires separate table: `terraform-locks`\n\n**Cloud Locks** (Terraform Cloud):\n- Backend manages all locking\n- Integrated with run queue\n- Prevents runs on locked state\n\n#### Stale Lock Handling\n- Manual unlock with `terraform force-unlock` removes lock file entry\n- Safe only when confident no other process is modifying state\n- Lock timeout provides eventual recovery\n\n### 3.6 State Synchronization and Consistency\n\nState consistency is maintained through several mechanisms:\n\n#### Serial Number Semantics\n```\nRead: state.json has serial = 42\nModify: Create new state with serial = 43\nWrite: Conditional write only if remote serial == 42\nSuccess: Remote serial updated to 43\nFailure: Someone else modified state (serial != 42)\n```\n\nSerial numbers detect conflicts:\n- Each state write increments serial\n- Before write, verify remote serial unchanged\n- Reject if conflicting modification detected\n- User must manually reconcile\n\n#### Lineage Invariant\n```\nterraform init: Initialize new lineage UUID\nAll future operations: Verify lineage unchanged\nterraform state pull: Create new lineage if state reset\nProtection: Prevents accidental state mix-up\n```\n\nLineage prevents mixing unrelated states:\n- Each state file has unique lineage ID\n- Operations fail if lineage changes unexpectedly\n- Manual override available with `terraform init -reconfigure`\n\n#### State Validation\nBefore persisting modified state:\n1. Validate all resource instances have required attributes\n2. Confirm resource types match schema definitions\n3. Verify no dangling references exist\n4. Check lineage and serial consistency\n\n---\n\n## 4. Execution Flow and Hook System\n\n### 4.1 Overall Pipeline Execution\n\nThe complete execution flow from command invocation to completion:\n\n```\nterraform plan/apply [args]\n    ↓\n[1] Load Configuration\n    ├─ Parse HCL files\n    ├─ Validate syntax\n    └─ Build config AST\n    ↓\n[2] Load Variables & Inputs\n    ├─ Merge .tfvars files\n    ├─ CLI-provided values\n    ├─ Environment variables\n    └─ Prompt for missing\n    ↓\n[3] Load State & Workspace\n    ├─ Connect to backend\n    ├─ Acquire lock\n    └─ Read current state\n    ↓\n[4] Initialize Providers\n    ├─ Discover plugins\n    ├─ Parse provider configs\n    ├─ Spawn provider processes\n    └─ Call Configure RPC\n    ↓\n[5] Build Execution Graph\n    ├─ Create resource nodes\n    ├─ Apply transformers\n    ├─ Validate graph\n    └─ Expand dynamic resources\n    ↓\n[6] PLAN PHASE (read-only analysis)\n    ├─ Refresh resource state (Read RPC)\n    ├─ Compute planned changes (PlanResourceChange RPC)\n    ├─ Analyze resource drift\n    └─ Generate plan file\n    ↓\n[7a] APPLY PHASE (if approved)\n    ├─ Execute graph with Apply RPC\n    ├─ Update state for each resource\n    ├─ Persist state to backend\n    └─ Write outputs\n    ↓\n[8] Cleanup & Report\n    ├─ Release lock\n    ├─ Print results\n    └─ Close connections\n```\n\n### 4.2 Plan Phase Execution\n\nThe plan phase analyzes desired vs. actual state without making changes:\n\n#### Phase Structure\n```\nPlan Phase\n    ├─ State Refresh (Read current)\n    ├─ Plan Calculation (Diff desired vs actual)\n    └─ Plan Serialization (Write plan file)\n```\n\n#### State Refresh\nFor each resource in the dependency graph:\n1. Call `ReadResource` RPC with current state\n2. Provider returns refreshed state from remote system\n3. Compare returned state with terraform state\n4. Detect external modifications (drift)\n5. Update in-memory state snapshot\n\n```\ntf state:    {id: \"i-1\", tags: {env: \"prod\"}}\nprovider:    {id: \"i-1\", tags: {env: \"dev\"}}  # Changed externally!\n             ↓\nReport: drift detected on tags\n```\n\n#### Plan Calculation\nFor each resource with configuration:\n1. Get prior state from snapshot\n2. Get proposed state from configuration (with variables evaluated)\n3. Call `PlanResourceChange` RPC\n4. Provider returns planned actions:\n   - `No-op`: No change needed\n   - `Update`: In-place modification\n   - `Replace`: Destroy and recreate\n   - `Delete`: Destroy only (orphaned resources)\n\n```\nPlanResourceChange Input:\n  - prior:    Current state\n  - proposed: Desired state from config\n\nProvider Output:\n  - requires_replace: [list of attributes causing replacement]\n  - sensitive_attrs: [attributes with sensitive values]\n```\n\n#### Plan File Serialization\nPlan is serialized to binary file containing:\n- Complete resource graph with node dependencies\n- Planned changes for each resource\n- Configuration values (sensitive values encrypted)\n- State snapshot taken during planning\n- Hash of configuration for consistency check\n\n### 4.3 Apply Phase Execution\n\nThe apply phase realizes the desired state by executing the dependency graph:\n\n#### Graph Execution Strategy\n```\nWalk Graph in Dependency Order\n    ├─ Parallel execution where safe (no edges between nodes)\n    ├─ Sequential execution for dependent nodes\n    └─ Backtrack on error (apply already-applied nodes)\n```\n\n#### Walker Pattern\n\nThe core uses a **walker pattern** to traverse and execute the graph:\n\n```go\ntype Walker struct {\n    Graph      *Graph\n    Vertices   []*Vertex  // Ordered by dependency\n    Edges      []*Edge\n    Results    map[Vertex]Result\n    Lock       sync.Mutex // Serialize state writes\n}\n\nfunc (w *Walker) Walk(ctx Context) {\n    for each Vertex in topological order {\n        // Wait for dependencies\n        dependencies := w.dependenciesOf(vertex)\n        for each dependency {\n            w.awaitResult(dependency)\n            if dependency failed {\n                w.skipVertex(vertex)\n                continue\n            }\n        }\n\n        // Evaluate vertex\n        result := w.evaluateVertex(vertex)\n        w.recordResult(vertex, result)\n    }\n}\n```\n\n**Key characteristics:**\n- Topological sort ensures dependencies evaluated first\n- Futures/promises for dependency awaiting\n- Parallel execution where dependency graph allows\n- Atomic state updates (critical section protected)\n\n#### Resource Application Sequence\n```\nFor each resource node in execution order:\n    1. Verify dependencies completed\n    2. Determine action (create/update/delete/no-op)\n    3. Call appropriate provider RPC\n    4. Update state with result\n    5. Record outputs for dependent resources\n    6. Execute provisioners (if any)\n    7. Mark node complete\n```\n\n#### State Update Atomicity\n```\nBefore RPC Call:\n    - Lock state mutex\n    - Read current state\n    - Prepare update\n    - Call provider\n\nAfter RPC Call:\n    - Merge result with state\n    - Increment serial\n    - Write to backend (with conditional check)\n    - Unlock state mutex\n```\n\nState writes are atomic:\n- Only one resource update at a time (serialized by lock)\n- If write fails, apply fails (prevent inconsistency)\n- Serial number prevents lost-update anomalies\n\n### 4.4 Hook System\n\nThe hook system provides extension points for observing and potentially modifying execution:\n\n#### Hook Interface\n```go\ntype Hook interface {\n    // Pre/post hooks for each operation\n    PreApply(addr string, gen GraphNodeDynamicExpandable) error\n    PostApply(addr string, gen GraphNodeDynamicExpandable, err error)\n\n    PrePlan(addr string, gen GraphNodeDynamicExpandable) error\n    PostPlan(addr string, gen GraphNodeDynamicExpandable) error\n\n    PreRefresh(addr string, gen GraphNodeDynamicExpandable) error\n    PostRefresh(addr string, gen GraphNodeDynamicExpandable, newState State) error\n\n    PreProvisionerRun(addr string, provisioner string) error\n    PostProvisionerRun(addr string, provisioner string, err error)\n\n    ProvisionOutput(addr string, provisioner string, msg string)\n\n    Diagnostics(diags Diagnostics) // Collect messages\n}\n```\n\n#### Hook Registration\n```\nUI ─→ Hook Registry ←── Providers\n      │\n      └─→ Walker\n            └─→ Fire hooks during execution\n```\n\nHooks are registered with the context and fired by the walker at key points.\n\n#### Hook Examples\n\n**UI Hooks** (built-in):\n- Progress indicator: Show \"Processing resource...\"\n- Output buffering: Collect provisioner output\n- Diagnostic reporting: Display warnings and errors\n\n**Custom Hooks** (extension):\n- Compliance verification: Ensure changes meet policy\n- Cost estimation: Calculate estimated cloud costs\n- Change notifications: Send Slack messages on major changes\n- Audit logging: Record all modifications\n\n#### Hook Execution Guarantees\n- Pre-hooks fired before operation (can return error to skip)\n- Post-hooks fired after operation (cannot prevent completion)\n- Hooks receive operation status (success/failure)\n- Hook panics do not crash core (isolated execution)\n\n### 4.5 Dynamic Expansion and Count/For_Each\n\nDynamic expansion allows resources to be instantiated based on runtime data:\n\n#### Count Expansion\n```hcl\nresource \"aws_instance\" \"web\" {\n  count = var.instance_count\n  ami   = \"...\"\n}\n\n# Results in:\n# aws_instance.web[0]\n# aws_instance.web[1]\n# aws_instance.web[2]\n```\n\n**Expansion process:**\n1. Count expression evaluated (e.g., `length(var.instances)`)\n2. Resource node duplicated N times\n3. Index variable available inside resource block\n4. Dependencies on resource reference all instances\n\n#### For_Each Expansion\n```hcl\nresource \"aws_instance\" \"web\" {\n  for_each = var.instances\n  ami      = \"...\"\n}\n\n# Results in:\n# aws_instance.web[\"prod\"]\n# aws_instance.web[\"staging\"]\n```\n\n**For each particulars:**\n- Each reference creates `for_each` variable (map of key→value)\n- Splat operations create implicit dependencies\n- Conditional expressions determine which instances created\n\n#### Expansion Dependencies\n```\nResource A count = length(var.items)\n    ↓\nDepends on: var.items\n    ↓\nResource B depends on: aws_instance.a[*].id\n    ↓\nCreates dependencies on ALL expanded instances of A\n```\n\nWhen `for_each` or `count` is modified:\n- New instances created, old destroyed\n- Terraform tracks instance identity through state\n- Replacing expression destroys and recreates all (careful!)\n\n#### Dynamic Blocks\n```hcl\nresource \"aws_security_group\" \"example\" {\n  dynamic \"ingress\" {\n    for_each = var.ports\n    content {\n      from_port = ingress.value\n    }\n  }\n}\n```\n\nDynamic blocks:\n- Evaluate at plan time\n- Expand to multiple configuration blocks\n- Useful for variable-length nested structures\n- Does NOT create separate resource nodes (inline expansion)\n\n### 4.6 Error Handling and Recovery\n\nError handling across the pipeline:\n\n#### Error Categories\n\n**Configuration Errors**:\n- Invalid HCL syntax\n- Type mismatches in interpolations\n- Missing required arguments\n- **Handling**: Reported immediately, plan/apply prevented\n\n**Planning Errors**:\n- Provider RPC failures during plan\n- Missing data sources\n- Impossible configuration\n- **Handling**: Plan fails, state unchanged, retry allowed\n\n**Apply Errors**:\n- Provider RPC failures during apply\n- Resource creation/update failures\n- Insufficient permissions\n- **Handling**: Partial apply possible (some resources created), state updated, retry continues from last successful\n\n**State Errors**:\n- Lock timeout (another process locked state)\n- Serial number mismatch (concurrent modification)\n- Corrupted state file\n- **Handling**: Error reported, manual recovery required\n\n#### Error Recovery\n```\nApply Fails on Resource 3 (of 5 in sequence)\n    ↓\nResources 1, 2: Successfully applied (state updated)\nResource 3: Failed (state NOT updated)\nResources 4, 5: Skipped\n    ↓\nUser fixes issue and runs apply again\n    ↓\nPlan detects resources 1, 2 not in plan (no-op)\nResource 3: Retried with same plan\n```\n\nPartial application allows:\n- Idempotent retry of failed operations\n- Rerun with same or modified plan\n- Avoiding wasted work on independent resources\n\n#### Error Propagation\n```\nProvider RPC Error\n    ↓\nWrapped with context (resource address, operation)\n    ↓\nWalk aborts subsequent dependent operations\n    ↓\nError collected in results\n    ↓\nFinal report shows all errors\n```\n\n---\n\n## 5. Key Extension Points and Customization\n\nThe architecture provides several extension mechanisms:\n\n### 5.1 Hooks\n- Observe any operation pre/post\n- Integrate with external systems\n- Collect diagnostic information\n- Cannot prevent operations (post-hooks)\n\n### 5.2 Providers\n- Implement resources, data sources, functions\n- Custom logic for any infrastructure type\n- Plugin architecture enables third-party implementations\n\n### 5.3 State Backends\n- Custom backend implementations\n- Alternative storage systems\n- Specialized locking strategies\n\n### 5.4 Graph Transformers\n- Add implicit dependencies\n- Modify graph structure\n- Can be registered to customize execution order\n\n### 5.5 Provisioners\n- Post-resource-creation operations\n- Execute arbitrary commands\n- Local and remote execution\n\n---\n\n## 6. Data Flow During Plan/Apply\n\n### 6.1 Complete Data Flow Diagram\n\n```\nConfiguration Files (.tf)\n    ↓ [Parse]\nConfiguration AST\n    ↓ [Interpolation Values]\nVariables, Locals, Outputs\n    ↓ [Merge]\nFinal Configuration Map\n    ↓ [Build Graph]\nDependency Graph\n    ├─ Resource Nodes\n    ├─ Provider Nodes\n    └─ Edges (dependencies)\n    ↓\nState Backend\n    ↓ [Load]\nCurrent State Snapshot\n    ├─ Resource instances\n    ├─ Outputs\n    └─ Metadata (serial, lineage)\n    ↓ [PLAN PHASE]\n    │\n    ├─ Provider.ReadResource() for each resource\n    │   ↓\n    │   Refreshed State\n    │   ↓ [Compare with snapshot]\n    │   Drift Detection\n    │\n    ├─ Provider.PlanResourceChange() for each resource\n    │   ├─ Prior: current state\n    │   ├─ Proposed: desired configuration\n    │   ↓\n    │   Planned Changes (add/update/delete/no-op)\n    │\n    └─ Serialize to Plan File\n        ├─ Graph structure\n        ├─ All planned changes\n        ├─ Configuration snapshot (encrypted)\n        └─ State snapshot (encrypted)\n\n    ↓ [APPLY PHASE]\n    │\n    ├─ Load Plan File\n    │   ↓\n    │   Graph + Planned Changes\n    │\n    ├─ For each node in dependency order:\n    │   │\n    │   ├─ Apply Planned Change\n    │   │   ├─ CREATE: Provider.ApplyResourceChange(proposed)\n    │   │   ├─ UPDATE: Provider.ApplyResourceChange(proposed)\n    │   │   └─ DELETE: Provider.ApplyResourceChange(null)\n    │   │\n    │   └─ Update State\n    │       ├─ Merge result attributes\n    │       ├─ Increment serial\n    │       └─ Write to backend\n    │\n    └─ Finalize State\n        ├─ Calculate outputs\n        ├─ Final write to backend\n        ├─ Release lock\n        └─ Print summary\n\nOutput: Updated state file + user output\n```\n\n### 6.2 State Attribute Flow\n\n```\nConfiguration:\n    {\n      name = \"web-server\"\n      instance_type = \"t3.medium\"\n    }\n\n↓ [Evaluated with variables/functions]\n\nProposed State:\n    {\n      name = \"web-server\"\n      instance_type = \"t3.medium\"\n    }\n\n↓ [Provider computes additional attributes]\n\nApply Result:\n    {\n      id = \"i-1234567890abcdef0\"        (computed by provider)\n      name = \"web-server\"\n      instance_type = \"t3.medium\"\n      private_ip = \"10.0.0.5\"           (computed by provider)\n      created_at = \"2024-01-01T12:00Z\"  (computed by provider)\n    }\n\n↓ [Merged into state]\n\nPersisted State:\n    {\n      id = \"i-1234567890abcdef0\"\n      name = \"web-server\"\n      instance_type = \"t3.medium\"\n      private_ip = \"10.0.0.5\"\n
