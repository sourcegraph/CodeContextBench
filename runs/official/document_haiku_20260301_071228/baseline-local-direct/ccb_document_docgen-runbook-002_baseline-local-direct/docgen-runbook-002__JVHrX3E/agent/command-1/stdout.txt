Common causes:\n1. Upstream timeout (nginx, application timeout)\n2. Upstream memory/resource exhaustion\n3. Load balancer or proxy between Envoy and upstream\n4. Explicit upstream server restart/drain\n\n**Code Path:**\n- `source/common/http/conn_pool_base.cc` - `onGoAway()` handler (HTTP/2)\n- `source/common/http/http1/conn_pool.cc` - `decodeHeaders()` detects \"Connection: close\"\n- `source/common/tcp/conn_pool.cc` - `onEvent(RemoteClose)` handler\n- Metrics: `upstream_cx_close_notify`, `upstream_rq_pending_failure_eject`\n\n**Diagnostic Steps:**\n\n```bash\n# 1. Check remote close metrics\ncurl -s http://localhost:9901/stats | grep -E \"upstream_cx_destroy|upstream_cx_close_notify\"\n\n# Expected pattern:\n# cluster.your_cluster.upstream_cx_destroy_remote: X\n# cluster.your_cluster.upstream_cx_destroy_remote_with_active_rq: Y  (if Y > 0, this is the issue)\n# cluster.your_cluster.upstream_cx_close_notify: Z  (HTTP GOAWAY/Connection: close)\n\n# 2. Check for stream reset metrics\ncurl -s http://localhost:9901/stats | grep upstream_rq.*reset\n\n# 3. Check connection timing\ncurl -s http://localhost:9901/stats | grep upstream_cx_connect_ms  # Connection establishment time\n\n# 4. Enable debug logging to see close reason\n# In config, set log_level: debug for pool component\n# Watch logs for: \"remote close\", \"goaway\", \"connection closed\"\n```\n\n**Fix:**\n\n**Option A: Increase Connection Timeout (if upstream is slow)**\n```yaml\ncluster:\n  name: my_cluster\n  connect_timeout: 5s  # Increase from default 1s\n  request_timeout: 30s  # Timeout for individual requests\n```\n\n**Option B: Connection Duration Limit (if upstream kills long-lived connections)**\n```yaml\ncluster:\n  name: my_cluster\n  max_connection_duration: 5m  # Upstream closes after this, so pre-drain\n```\n\n**Option C: Per-Connection Buffer Limits (if upstream buffers are small)**\n```yaml\ncluster:\n  name: my_cluster\n  per_connection_buffer_limit_bytes: 32768  # Reduce Envoy buffer size\n```\n\n**Option D: Enable Connection Drain on Upstream Goaway**\n```yaml\ncluster:\n  name: my_cluster\n  # Envoy automatically transitions to draining state on HTTP/2 GOAWAY\n  # Ensure idle connections are cleaned up:\n```\n\n**Option E: Investigate Upstream (best)**\n- Check upstream logs for \"connection reset\" or timeout messages\n- Verify upstream resource availability (CPU, memory, file descriptors)\n- Check for upstream load balancer health checks causing resets\n- Increase upstream `keepalive_timeout` if applicable\n\n---\n\n### Scenario 3: Circuit Breaker Triggering Unexpectedly\n\n**Symptom:**\n- Requests fail with 503 despite healthy upstreams\n- Metrics show:\n  - `cluster.circuit_breakers.<priority>.cx_open` = 1 (connection circuit breaker open)\n  - OR `cluster.circuit_breakers.<priority>.rq_open` = 1 (request circuit breaker open)\n  - `cluster.upstream_cx_overflow` or `cluster.upstream_rq_pending_overflow` incrementing\n- Cluster shows as \"all degraded\" or \"all draining\" in `/clusters` output\n\n**Root Cause:**\n\nCircuit breaker thresholds exceeded. Envoy has several circuit breakers:\n- **Connection CB:** Limits concurrent connections (`max_connections`)\n- **Pending request CB:** Limits queued requests (`max_pending_requests`)\n- **Request CB:** Limits concurrent requests (`max_requests`)\n- **Retry CB:** Limits retries (`retry_budget`)\n\nExceeded due to:\n1. Upstream slow/overloaded (requests back up)\n2. Circuit breaker thresholds too aggressive for actual load\n3. Cascading failures from other services\n\n**Code Path:**\n- `source/common/upstream/cluster_manager_impl.cc` - CircuitBreaker implementation\n- Check functions in `conn_pool_base.cc:shouldCreateNewConnection()`\n- Metrics: All `circuit_breaker` stats and `_overflow` counters\n\n**Diagnostic Steps:**\n\n```bash\n# 1. Identify which circuit breaker is open\ncurl -s http://localhost:9901/stats | grep circuit_breaker | grep your_cluster\n\n# Output format:\n# cluster.your_cluster.circuit_breakers.default.cx_open: 0 or 1\n# cluster.your_cluster.circuit_breakers.default.cx_pool_open: 0 or 1\n# cluster.your_cluster.circuit_breakers.default.rq_pending_open: 0 or 1\n# cluster.your_cluster.circuit_breakers.default.rq_open: 0 or 1\n\n# 2. Check current utilization vs limits\ncurl -s http://localhost:9901/stats | grep -E \"circuit_breaker.*remaining|upstream_cx_active|upstream_rq_active\"\n\n# 3. Check which overflow type\ncurl -s http://localhost:9901/stats | grep overflow | grep your_cluster\n\n# 4. Check if it's upstream slow (high latency)\ncurl -s http://localhost:9901/stats | grep upstream_rq_time | grep your_cluster\n\n# 5. Examine cluster membership\ncurl -s http://localhost:9901/clusters | grep -A 30 \"cluster_name::your_cluster\"\n# Look for: health status, degraded hosts, update frequency\n```\n\n**Fix:**\n\n**Option A: Relax Aggressive Circuit Breaker Thresholds**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_connections: 200       # Increase from 100\n        max_pending_requests: 256   # Increase from 128\n        max_requests: 200           # Increase from 100\n```\n\n**Option B: Track Remaining Resources (monitoring)**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_requests: 100\n        track_remaining: true  # Enables circuit_breaker.remaining_* metrics\n```\n\n**Option C: Separate High/Low Priority Traffic**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_requests: 100\n      - priority: HIGH\n        max_requests: 50  # Higher priority traffic gets less\n```\n\n**Option D: Diagnose Upstream Issue (best)**\n- Monitor upstream response times with `upstream_rq_time` histogram\n- If latency is high, upstream is bottleneck, not circuit breaker configuration\n- Scale upstream horizontally or optimize application\n\n---\n\n### Scenario 4: HTTP/2 GOAWAY Causing Request Failures\n\n**Symptom:**\n- Requests fail with \"stream reset by remote\" in access logs\n- Metrics show:\n  - `cluster.upstream_cx_close_notify` increasing rapidly\n  - `cluster.upstream_rq_pending_failure_eject` spiking\n  - Connection churn: `upstream_cx_total` and `upstream_cx_destroy` both increasing\n- Affected only on HTTP/2 clusters\n\n**Root Cause:**\n\nUpstream HTTP/2 server sends GOAWAY frame, closing connection. Causes:\n1. **Upstream graceful shutdown:** Server initiating drain\n2. **Max request limit:** Server closes after `max_requests_per_connection` requests\n3. **Upstream connection timeout:** Server configured idle timeout\n4. **Resource exhaustion:** Upstream OOM kills connections\n5. **Load balancer drain:** Kubernetes `preStop` hook or similar\n\n**Code Path:**\n- `source/common/http/conn_pool_base.cc` - `MultiplexedActiveClientBase::onGoAway()`\n- Transitions connection to **Draining** state if streams active\n- Or closes immediately if idle\n- Metrics: `upstream_cx_close_notify`, `upstream_cx_destroy_remote_with_active_rq`\n\n**Diagnostic Steps:**\n\n```bash\n# 1. Confirm GOAWAY pattern\ncurl -s http://localhost:9901/stats | grep upstream_cx_close_notify\n\n# 2. Check connection lifetime\ncurl -s http://localhost:9901/stats | grep upstream_cx_length_ms | grep your_cluster\n\n# Expected: If GOAWAY frequent, max connection duration very short (e.g., 5-10s)\n\n# 3. Check if it's max_requests being hit\ncurl -s http://localhost:9901/stats | grep upstream_cx_max_requests | grep your_cluster\n\n# 4. Compare HTTP/1 vs HTTP/2 behavior\ncurl -s http://localhost:9901/stats | grep -E \"upstream_cx_http[12]_total\"\n\n# If HTTP/1 stable but HTTP/2 churning, upstream GOAWAY issue\n\n# 5. Check connection count trends\n# High upstream_cx_destroy combined with high upstream_cx_close_notify = GOAWAY pattern\n```\n\n**Fix:**\n\n**Option A: Align with Upstream Max Requests Limit**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_requests: 50  # Match upstream max_requests_per_connection\n```\n\n**Option B: Increase Max Concurrent Streams (reuse connections better)**\n```yaml\ncluster:\n  name: my_cluster\n  http2_protocol_options:\n    max_concurrent_streams: 100  # More streams per connection before new connection\n```\n\n**Option C: Increase Connection Reuse with Higher Request Limit**\n```yaml\ncluster:\n  name: my_cluster\n  # Assume upstream has requests_per_connection = 1000\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_requests: 1000  # Tune to match upstream\n```\n\n**Option D: Enable Connection Pooling with Keepalive (HTTP/2 level)**\n```yaml\ncluster:\n  name: my_cluster\n  http2_protocol_options:\n    connection_keepalive:\n      interval: 30s    # Ping interval to detect dead connections early\n      timeout: 10s     # Time to wait for pong\n```\n\n**Option E: Investigate Upstream Drain (best)**\n- Check upstream logs for GOAWAY reason code\n- Monitor upstream process restart frequency\n- If intended (graceful drain), increase `max_requests_per_connection` expectation\n- If unintended, investigate resource exhaustion on upstream\n\n**Error Code Mapping:**\n- GOAWAY code 0 = Normal shutdown\n- GOAWAY code 1 = Protocol error\n- GOAWAY code 8 = Cancel (load balancer drain)\n\n---\n\n## 3. Configuration Fixes\n\n### Connection Pool Limits Tuning\n\n**`max_connections`** (Circuit Breaker)\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_connections: 100\n```\n- **Default:** 100\n- **Effect:** Maximum concurrent connections per Envoy worker thread\n- **When to increase:**\n  - If `upstream_cx_active` consistently near limit\n  - If `upstream_cx_overflow` metric incrementing\n- **When to decrease:**\n  - If upstream has resource constraints\n  - To force connection reuse (especially for HTTP/1.1)\n\n**`max_pending_requests`** (Circuit Breaker)\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_pending_requests: 128\n```\n- **Default:** 128\n- **Effect:** Maximum queued requests waiting for connection\n- **When to increase:**\n  - If `upstream_rq_pending_overflow` incrementing\n  - If `rq_pending_open` circuit breaker frequently opens\n- **When to decrease:**\n  - To fail fast instead of queuing (fail-fast pattern)\n\n**`max_requests`** (Circuit Breaker)\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_requests: 100\n```\n- **Default:** 100\n- **Effect:** Maximum concurrent in-flight requests\n- **When to increase:**\n  - If upstream latency is high\n  - If `upstream_rq_active` near limit with pending requests\n- **When to decrease:**\n  - To protect overloaded upstream\n\n### Connection Timeout Configuration\n\n**`connect_timeout`** - Time to establish TCP connection\n```yaml\ncluster:\n  name: my_cluster\n  connect_timeout: 5s  # Default 1s\n```\n- **Effect:** How long to wait for TCP three-way handshake\n- **Increase if:**\n  - Network latency high (GCP to on-prem, cross-region)\n  - Upstream slow to accept connections\n- **Metrics:** `upstream_cx_connect_timeout` increasing = too short\n\n**`request_timeout`** - Time to complete entire HTTP request/response cycle\n```yaml\nroutes:\n  - route:\n      timeout: 30s  # Default no timeout\n```\n- **Effect:** Total time for request to complete\n- **Interaction:** `connect_timeout` is included in this\n- **Increase if:** Upstream processing is slow\n\n**`max_connection_duration`** - Force-close connections after time\n```yaml\ncluster:\n  name: my_cluster\n  max_connection_duration: 5m\n```\n- **Effect:** Close and drain connection after duration\n- **Use case:** Upstream has internal max duration (e.g., nginx proxy)\n- **Set to:** Slightly less than upstream timeout to pre-drain gracefully\n\n### Per-Connection Buffer Configuration\n\n**`per_connection_buffer_limit_bytes`** - Max buffered data per connection\n```yaml\ncluster:\n  name: my_cluster\n  per_connection_buffer_limit_bytes: 32768  # Default 1MB\n```\n- **Effect:** Limits memory usage for buffering\n- **Decrease if:**\n  - Memory pressure on Envoy hosts\n  - Many connections with variable data rates\n- **Increase if:**\n  - Upstream sends bursts of data\n  - Seeing buffer limit exceeded errors\n\n### HTTP/2 Specific Configuration\n\n**`max_concurrent_streams`** - HTTP/2 streams per connection\n```yaml\ncluster:\n  name: my_cluster\n  http2_protocol_options:\n    max_concurrent_streams: 100  # Default depends on upstream SETTINGS\n```\n- **Effect:** Envoy limit on streams before new connection needed\n- **Increase if:**\n  - Many concurrent requests and few upstream connections desired\n  - Upstream advertises higher `MAX_CONCURRENT_STREAMS`\n- **Monitor:** `upstream_rq_per_cx` histogram - requests per connection\n\n**`connection_keepalive`** - Send keepalive pings\n```yaml\ncluster:\n  name: my_cluster\n  http2_protocol_options:\n    connection_keepalive:\n      interval: 30s\n      timeout: 10s\n```\n- **Effect:** Detects dead connections early\n- **Use case:** Long-lived connections with idle periods\n- **Metrics:** Helps with `upstream_cx_destroy_remote` detection\n\n### HTTP/1.1 Connection Behavior\n\n**`max_requests_per_connection`** - Via circuit breaker\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_requests: 100  # HTTP/1.1 reuses same connection\n```\n- **Effect:** Higher limit means more requests per connection (better reuse)\n- **HTTP/1.1 specific:** Each connection sequential (concurrent streams = 1)\n- **Monitor:** `upstream_cx_http1_total` and `upstream_rq_per_cx`\n\n### Circuit Breaker Thresholds Examples\n\n**Aggressive (Fail Fast):**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_connections: 50\n        max_pending_requests: 50\n        max_requests: 50\n```\nBest for: Frontend services, protecting overloaded upstreams\n\n**Relaxed (High Throughput):**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_connections: 1000\n        max_pending_requests: 5000\n        max_requests: 5000\n```\nBest for: Internal services, high-volume data processing\n\n**Balanced (Default):**\n```yaml\ncluster:\n  name: my_cluster\n  circuit_breakers:\n    thresholds:\n      - priority: DEFAULT\n        max_connections: 100\n        max_pending_requests: 128\n        max_requests: 100\n```\n\n---\n\n## 5.
