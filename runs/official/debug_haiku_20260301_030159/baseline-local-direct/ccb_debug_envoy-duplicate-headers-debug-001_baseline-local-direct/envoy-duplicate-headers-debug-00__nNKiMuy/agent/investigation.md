# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` in route configuration are being duplicated in local replies (responses generated by the router filter itself, such as upstream timeouts or connection failures) because `finalizeResponseHeaders()` is invoked twice in the response header processing pipeline: once in the filter manager's local reply lambda and again in the router filter's `modify_headers_` callback.

## Root Cause

The root cause is a **double invocation of `finalizeResponseHeaders()`** on the same response header map during local reply processing.

### Technical Details

When a local reply is generated (e.g., cluster not found, upstream timeout), the flow is:

1. **Filter Manager's Local Reply Lambda** (`source/common/http/filter_manager.cc:1123-1129` and `1156-1162`)
   - Calls `streamInfo().route()->routeEntry()->finalizeResponseHeaders()` at line 1125/1158
   - Then invokes the `modify_headers` callback (router's lambda)

2. **Router's modify_headers_ Lambda** (`source/common/router/router.cc:444-461`)
   - Calls `route_entry_->finalizeResponseHeaders()` at line 453

This results in the same response headers being processed twice by `finalizeResponseHeaders()`, which invokes the header parser twice.

3. **Header Parser** (`source/common/router/header_parser.cc:145-213`)
   - For headers with `APPEND_IF_EXISTS_OR_ADD` action (default), uses `headers.addReferenceKey()` (line 211)
   - This appends the header value
   - When called twice, the header is appended twice, resulting in duplication

## Affected Components

1. **`source/common/router/router.cc`** - Router filter that sets up `modify_headers_` callback with `finalizeResponseHeaders()` call
2. **`source/common/http/filter_manager.cc`** - DownstreamFilterManager that calls `finalizeResponseHeaders()` in its local reply lambda before invoking the router's `modify_headers_` callback
3. **`source/common/router/header_parser.cc`** - HeaderParser that applies headers via append/overwrite operations
4. **`source/common/http/utility.cc`** - HTTP Utility that processes the EncodeFunctions and invokes the modify_headers callback
5. **`api/envoy/config/core/v3/base.proto`** - Proto definitions for HeaderValueOption with append_action field

## Causal Chain

1. **Symptom**: Response headers appear twice in HTTP responses for local replies (observed in access logs or client responses)
   - Example: `x-custom-trace: abc123` appears twice
   - Observed in responses where `response_code_details=upstream_response_timeout` or similar local reply details

2. **Intermediate Hop 1**: Router filter sets up a `modify_headers_` callback during request decoding
   - `router.cc:444-461` - Lambda is created that will call `route_entry_->finalizeResponseHeaders()`

3. **Intermediate Hop 2**: Local reply generation occurs (e.g., cluster not found)
   - `router.cc:522` and other sites - `sendLocalReply()` is called with the `modify_headers_` callback

4. **Intermediate Hop 3**: Filter manager receives the sendLocalReply call
   - `filter_manager.cc:982-1053` - `DownstreamFilterManager::sendLocalReply()` is invoked
   - Determines to send the reply via the filter chain

5. **Intermediate Hop 4**: Filter manager creates a response headers lambda
   - `filter_manager.cc:1123-1129` - Creates an `EncodeFunctions` struct with a lambda that:
     - Calls `finalizeResponseHeaders()` (FIRST TIME)
     - Calls the router's `modify_headers_` callback (which calls `finalizeResponseHeaders()` SECOND TIME)

6. **Intermediate Hop 5**: Utility prepares the local reply
   - `utility.cc:719` - `Utility::prepareLocalReply()` invokes the `modify_headers_` callback
   - This is the filter manager's lambda, which triggers the double invocation

7. **Intermediate Hop 6**: Header parser processes the headers
   - `header_parser.cc:173-212` - `evaluateHeaders()` runs and applies `response_headers_to_add`
   - For `APPEND_IF_EXISTS_OR_ADD`, uses `addReferenceKey()` which appends

8. **Root Cause**: Double header application due to `finalizeResponseHeaders()` being called twice
   - First call: Headers are added/appended
   - Second call: Headers are added/appended again
   - With `APPEND_IF_EXISTS_OR_ADD` action, results in duplicate header values

## Recommendation

### Fix Strategy

The fix should eliminate the double invocation of `finalizeResponseHeaders()` during local reply processing. There are two potential approaches:

**Option A (Recommended)**: Remove the `finalizeResponseHeaders()` call from the filter manager's local reply lambda
- Keep the call only in the router's `modify_headers_` callback
- Modify `filter_manager.cc:1123-1129` and `1156-1162` to not call `finalizeResponseHeaders()` directly
- This way, route-specific response headers are only finalized once, by the router filter

**Option B**: Modify the router filter to not call `finalizeResponseHeaders()` in its `modify_headers_` callback when used for local replies
- This would require tracking whether `modify_headers_` is being invoked for a local reply or an upstream response
- More complex but maintains current filter manager behavior

### Diagnostic Steps

To verify the issue:

1. **Check Access Logs**: Look for response headers appearing twice in 5xx responses (local replies)
   - Filter for `response_code_details=upstream_response_timeout` or similar local reply codes
   - Compare with 200 responses to see that duplicates don't appear there

2. **Count Header Occurrences**: Use a test with specific route configuration
   ```yaml
   response_headers_to_add:
     - header:
         key: "x-test-header"
         value: "test-value"
       append_action: OVERWRITE_IF_EXISTS_OR_ADD
   ```
   - For `OVERWRITE_IF_EXISTS_OR_ADD`, the header should appear once (double-write makes it a single header)
   - Test scenario: Trigger cluster not found error
   - Verify header count matches expectation

3. **Enable Debug Logging**: Add logging in `HeaderParser::evaluateHeaders()` to see when it's called
   - Confirm it's called twice for local reply cases

4. **Test Upstream vs Local Replies**:
   - Configure a route with `response_headers_to_add`
   - Send request that gets upstream response: header appears once (expected)
   - Send request that triggers local reply: header may appear twice (bug)
