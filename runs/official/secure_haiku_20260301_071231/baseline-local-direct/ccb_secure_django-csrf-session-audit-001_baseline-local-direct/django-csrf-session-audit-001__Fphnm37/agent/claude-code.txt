CSRF Token Masking Scheme\n\n**Threat Model:** BREACH attack (compressing CSRF token in response body)\n\n**Mechanism:**\n- Secret generated as 32 random alphanumeric chars\n- Mask generated separately (32 chars)\n- Cipher = (secret[i] + mask[i]) mod 62 for each position\n- Token = mask + cipher (64 chars)\n- Each request: new mask, new token, same secret\n\n**Security Properties:**\n✅ **COMPRESSION RESISTANCE:** Attacker sees 64-char token that changes per request\n   - Compression algorithm can't derive static secret from token\n   - Token entropy = 32 chars (per 62-char alphabet) ≈ 190 bits\n   - ✓ Exceeds compression overhead, defeating BREACH\n\n✅ **RANDOMNESS:** Uses `secrets.choice()` (OS entropy)\n   - Cryptographically secure\n   - No pseudo-random number generator bias\n\n✅ **MATHEMATICAL SOUNDNESS:**\n   - Operation is reversible: (secret + mask) mod 62\n   - No collision risk: 62^32 possible secrets, 62^32 possible masks\n   - Adversary knowing mask + cipher can compute secret (intended)\n\n⚠️ **COOKIE LEAKAGE:** Unmasked secret stored in CSRF_COOKIE\n   - If cookie exposed (XSS, MITM), attacker has secret directly\n   - Attacker still needs matching token from form/header (but can request page to get token)\n   - Masked token on client still provides protection against compression attacks\n\n---\n\n### 2. Constant-Time Token Comparison\n\n**Code:** `django/utils/crypto.py:65-67`\n```python\ndef constant_time_compare(val1, val2):\n    return secrets.compare_digest(force_bytes(val1), force_bytes(val2))\n```\n\n**Threat Model:** Timing attacks on token verification\n\n**Security Properties:**\n✅ **TIMING-SAFE IMPLEMENTATION:**\n   - Uses Python's `secrets.compare_digest()` (wraps OpenSSL's CRYPTO_memcmp)\n   - All comparisons take same CPU cycles regardless of mismatch position\n   - Prevents statistical analysis of response times to infer token bits\n\n✅ **ALGORITHM SOUNDNESS:**\n   - Byte-by-byte comparison with early-exit prevention\n   - No shortcut returns on first mismatch\n   - Resilient against cache timing attacks (to extent of Python GIL)\n\n⚠️ **Python GIL CAVEAT:**\n   - Python's GIL may introduce jitter\n   - Under high concurrency, timing still measurable but harder to exploit\n   - Modern Python versions minimize GIL contention\n\n---\n\n### 3. HMAC-Based Session Signing\n\n**Chain:** `salted_hmac()` → `Signer.signature()` → `signing.loads()`\n\n**Security Properties:**\n✅ **MESSAGE AUTHENTICATION:**\n   - HMAC-SHA256 provides 256-bit security\n   - Impossible to forge signature without SECRET_KEY\n   - Tampered session data detected 100% of time\n\n✅ **KEY DERIVATION:**\n   - `key = hash(salt + secret)`\n   - Salt ensures different HMAC key for different contexts\n   - Multiple salts prevent cross-context attacks (e.g., signed_cookies vs db backend)\n\n✅ **FALLBACK KEY SUPPORT:**\n   - Tries current key first, then fallback_keys in order\n   - Enables key rotation without session invalidation\n   - Secure key transition: old key validated for compatibility period\n\n⚠️ **KEY DERIVATION NONSTANDARD:**\n   - Method: `hash(salt + secret)` rather than HKDF\n   - Is it secure? Yes, for one-time KDF\n   - Better practice: HKDF-Expand with application-specific context\n   - Current approach is \"good enough\" but not industry best-practice\n\n⚠️ **TIMESTAMP ONLY 32-BIT (BASE62):**\n   - `max_age` session expiry checked against timestamp\n   - Timestamp resolution: seconds\n   - Y2038 problem avoidable since encoded as base62 (supports larger numbers)\n   - No cryptographic weakness, but design note\n\n---\n\n### 4. Cryptographic Weaknesses and Mitigations\n\n**Weakness 1: Non-Standard KDF in salted_hmac()**\n```python\nkey = hasher(key_salt + secret).digest()\n```\n**Assessment:** Adequate for one-time use, but not HKDF-standard\n**Impact:** Minimal; pre-image resistance is 2^256 for SHA256\n**Mitigation:** None needed; design is acceptable\n\n---\n\n**Weakness 2: No Nonce/IV in Session Encoding**\n```python\nsession_data = signing.dumps(data, salt=..., compress=True)\n# No cipher IV; HMAC provides authentication only\n```\n**Assessment:** This is not encryption; it's signing + compression\n**Impact:** None; no confidentiality claim (data is readable if SECRET_KEY known)\n**Note:** Session data is not encrypted; confidentiality depends on HTTPS\n\n---\n\n**Weakness 3: Compression Before Signing (CRIME/BREACH Risk)**\n```python\ncompressed = zlib.compress(json_bytes)\nif len(compressed) < len(json_bytes) - 1:\n    data = compressed\nbase64d = b64_encode(data)\nsignature = hmac(base64d)\n```\n**Assessment:** Compression is applied before signing; if attacker controls session data, timing attacks possible\n**Impact:** Low in practice; session data is not user-controlled\n**Mitigation:** Compression is optional (`compress=False` avoids); signed_cookies use compression by default\n\n---\n\n**Weakness 4: Insecure Default Cookie Settings**\n```python\nsecure=settings.CSRF_COOKIE_SECURE,     # Default: False\nhttponly=settings.CSRF_COOKIE_HTTPONLY, # Default: False\n```\n**Assessment:** Defaults are insecure; assumes developer will configure correctly\n**Impact:** **CRITICAL for production**; allows XSS + MITM attacks\n**Mitigation:** **Developer must set in settings.py:**\n```python\nCSRF_COOKIE_SECURE = True\nCSRF_COOKIE_HTTPONLY = True\nSESSION_COOKIE_SECURE = True\nSESSION_COOKIE_HTTPONLY = True\nSECURE_SSL_REDIRECT = True\n```\n\n---\n\n### 7.
