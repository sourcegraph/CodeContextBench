# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` in route configuration are duplicated in local replies generated by the router filter (e.g., upstream timeout, connection failure, 404 errors) due to double invocation of `finalizeResponseHeaders()`. This occurs because the router's `sendLocalReply()` callback explicitly calls `finalizeResponseHeaders()`, and the FilterManager's local reply path unconditionally wraps the callback to call `finalizeResponseHeaders()` again before invoking the user-supplied modifier function.

## Root Cause

**File:** `source/common/http/filter_manager.cc` lines 1072-1078, 1123-1129, 1156-1162
**Mechanism:** The FilterManager's three local reply methods (`prepareLocalReplyViaFilterChain`, `sendLocalReplyViaFilterChain`, `sendDirectLocalReply`) create a wrapper lambda that unconditionally calls `finalizeResponseHeaders()` before executing the caller-supplied `modify_headers` callback.

When the router's `sendLocalReply()` is invoked with a callback that also calls `finalizeResponseHeaders()` (e.g., for direct responses), this results in the function being called twice in the same response header processing pipeline.

4. **`source/common/router/config_impl.cc`** — Route entry configuration and header parsing
5. **`source/common/router/header_parser.cc`** — Header parser implementation with `append_action` handling

## Causal Chain

1. **Symptom:** Response header `x-custom-trace` appears twice in access logs for local replies (e.g., 504 Upstream Timeout)

2. **Upstream Cause #1 (Direct Response Path):**
   - Router detects direct response entry at route/vhost level
   - Calls `callbacks_->sendLocalReply()` with lambda containing `finalizeResponseHeaders()` call
   - First invocation of `finalizeResponseHeaders()` occurs INSIDE the router's callback

3. **Intermediate Layer (FilterManager Wrapping):**
   - Router's callback reaches `FilterManager::sendLocalReply()`
   - FilterManager does NOT know that the router's callback already calls `finalizeResponseHeaders()`
   - FilterManager unconditionally wraps ALL `modify_headers` callbacks to call `finalizeResponseHeaders()` first

4. **Upstream Cause #2 (Double Invocation):**
   - Wrapped lambda is passed to `Utility::prepareLocalReply()`
   - The wrapped lambda executes:
     - Call #1 to `finalizeResponseHeaders()` (from FilterManager's wrapper)
     - Call #2 to `finalizeResponseHeaders()` (from router's original callback)

5. **Header Mutation Details:**
   - `finalizeResponseHeaders()` calls `header_parser->evaluateHeaders()` for each header parser
   - `evaluateHeaders()` iterates through configured headers and applies `HeaderValueOption::append_action` logic
   - With `OVERWRITE_IF_EXISTS_OR_ADD` (or default `APPEND_IF_EXISTS_OR_ADD`), headers are added via `headers.addReferenceKey()`
   - Two consecutive calls result in headers being added to the map twice

6. **Root Cause:** The architectural change that moved header finalization responsibility to FilterManager's local reply path created an unintended double-invocation when the router filter also finalizes headers in its callbacks.

## Filter Manager Encode Path

**File:** `source/common/http/filter_manager.cc` lines 1244-1330

The `FilterManager::encodeHeaders()` method iterates through encoder filters in REVERSE order (line 1253-1305), allowing each filter to modify response headers. However, local replies bypass this iteration:

1. When a local reply is triggered during request decoding, FilterManager prepares the response through `prepareLocalReplyViaFilterChain()`
2. During preparation, the wrapped lambda is called, which applies `finalizeResponseHeaders()` once
3. Then the user's modifier (which may include ANOTHER `finalizeResponseHeaders()` call) is invoked
4. Result: Headers are finalized twice before being sent to the encoder filters

## Header Configuration Details

**File:** `source/common/router/header_parser.cc` lines 53-76

The `HeadersToAddEntry` constructor handles the deprecated `append` BoolValue field:

```cpp
if (header_value_option.has_append()) {
  // If 'append' is set, convert to append_action enum
  append_action_ = header_value_option.append().value()
                       ? HeaderValueOption::APPEND_IF_EXISTS_OR_ADD
                       : HeaderValueOption::OVERWRITE_IF_EXISTS_OR_ADD;
} else {
  // Use the append_action field directly
  append_action_ = header_value_option.append_action();
}
```

**File:** `api/envoy/config/core/v3/base.proto` line 475-476

The default value for `append_action` is `APPEND_IF_EXISTS_OR_ADD`, which causes headers to be appended rather than replaced. When `finalizeResponseHeaders()` is called twice, both invocations use `addReferenceKey()` (line 211 in header_parser.cc), resulting in duplicate header values in the response.

## Complete Causal Chain

```
User Configures Route with response_headers_to_add
         ↓
[Request processed by Router filter]
         ↓
[Router detects condition: timeout/connection failure/no cluster/direct response]
         ↓
Router calls callbacks_->sendLocalReply(..., modify_headers_callback, ...)
         ↓
[For DIRECT RESPONSE case] modify_headers_callback includes finalizeResponseHeaders() call
         ↓
[FilterManager::sendLocalReply receives the callback]
         ↓
FilterManager wraps callback:
  wrapped_lambda() {
    finalizeResponseHeaders()  // Call #1 (FilterManager adds this)
    modify_headers(headers)    // Calls router's callback
  }
         ↓
Utility::prepareLocalReply calls wrapped_lambda()
         ↓
[Inside wrapped_lambda]
Call #1: FilterManager's finalizeResponseHeaders() executes
         → header_parser->evaluateHeaders() processes response_headers_to_add
         → For each header: headers.addReferenceKey() adds header to map
         ↓
Call #2: Router's modify_headers() callback executes
         → Which also contains direct_response->finalizeResponseHeaders()
         → header_parser->evaluateHeaders() processes response_headers_to_add AGAIN
         → For each header: headers.addReferenceKey() adds header to map AGAIN
         ↓
Result: Each configured header appears TWICE in the response
         ↓
Access log displays: "x-custom-trace: abc123" appears twice in response headers
```

## Recommendation

**Fix Strategy:**

The FilterManager should NOT unconditionally call `finalizeResponseHeaders()` in its local reply path when the caller's `modify_headers` callback will do so. Options include:

1. **Option A (Cleanest):** Remove the `finalizeResponseHeaders()` call from FilterManager's local reply wrapper lambda. Let the caller (router) be responsible for calling `finalizeResponseHeaders()` through its own callback. Update the "no cluster found" path in router to also include `finalizeResponseHeaders()` in its callback.

2. **Option B:** Create a separate callback mechanism for "already finalized" headers vs "needs finalization" headers, allowing FilterManager to distinguish when finalization has already been performed.

3. **Option C:** Modify the router's direct response path to NOT include `finalizeResponseHeaders()` in its callback, and let FilterManager handle all finalization. This requires ensuring the "no cluster found" and other local reply paths also pass callbacks WITHOUT embedded finalization calls.

**Diagnostic Steps:**

1. Add conditional logging to `finalizeResponseHeaders()` to track call stack and identify double invocations:
   ```
   ENVOY_STREAM_LOG(warn, "finalizeResponseHeaders() called, stack: {}", *this, stack_trace());
   ```

2. Count header additions using `headers.addReferenceKey()` to verify duplicate insertions

3. Compare response headers before/after each local reply path to identify which code paths produce duplicates

4. Cross-reference with route configuration to verify `response_headers_to_add` values match the duplicated headers

**Affected Versions:** Based on code structure, this affects versions where:
- `sendLocalReply()` wrapping was introduced in FilterManager's local reply handling
- Router filter generates local replies with explicit `finalizeResponseHeaders()` calls in callbacks
- A specific architectural change unified local reply handling in FilterManager
